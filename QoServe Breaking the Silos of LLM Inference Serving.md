    1、论文工作：
        - 论文要解决的核心问题：生产级 LLM serving 通常把请求粗粒度拆成 interactive 和 batch 两类，并放在彼此隔离的 GPU 集群或 replica 上服务。interactive 为满足 TTFT/TBT 往往使用小 prefill chunk，吞吐低、成本高；batch 可用大 chunk 提高吞吐，但独占资源且延迟约束宽松。随着多应用、多 QoS tier 和突发负载出现，这种 siloed deployment 会导致 GPU underutilization、过度 provisioning、难以支持细粒度 QoS，并在过载时产生队列堆积和级联 SLO violation。
        - 论文的主要贡献：提出 QoServe，一个 QoS-driven LLM inference serving system，在共享基础设施上 co-schedule 多类 QoS 请求。它允许应用为请求指定 TTFT、TBT、TTLT 和 priority；通过 dynamic chunking 利用 deadline slack 放大 prefill chunk；通过 hybrid prioritization 在 EDF 的 deadline-aware 和 SRPF/SJF 的短作业优先之间插值；通过 eager relegation 在过载时优先降级低优先级或已不可满足 deadline 的请求，以保护大多数重要请求的 SLO。
        - 论文所处背景：该工作位于 LLM serving 调度、多 SLO/QoS-aware serving、chunked prefill、prefill/decode co-location 与资源效率优化方向。它建立在 Sarathi-Serve/vLLM 的 chunked prefill 和 PagedAttention 机制之上，核心不是提出新的 attention 算子，而是把 QoS deadline、动态 chunk size、请求优先级和过载降级策略放进 serving scheduler，使一个共享 serving pool 能承载不同延迟目标的应用。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：Sarathi-Silo 这类 SOTA siloed deployment 为每个 QoS bucket 分配独立 replica，严格 TBT bucket 被迫长期使用小 chunk size，例如 256，从而牺牲吞吐；宽松 bucket 虽可使用大 chunk，例如 2K，但其 slack 不能被其他 bucket 借用。Sarathi-FCFS 在共享集群中不感知 deadline，长请求或宽松请求会阻塞急迫请求，形成 head-of-line blocking。Sarathi-EDF 在低负载下 deadline violation 少，但负载略超容量后队列迅速堆积，violation 可接近 100%。Sarathi-SRPF/SJF 在过载时偏向短 prefill，能降低 median latency，却会不公平地牺牲长请求，甚至在低负载下也可能让长请求错过 deadline。简单 rate limiting 或丢弃新请求也不考虑请求 QoS 和业务优先级。
        - 论文的设计方法：QoServe 把请求分为 interactive 和 non-interactive 两类，但允许每个应用设置具体 SLO。interactive 请求使用 TTFT 和 TBT：第一个 token deadline 是 arrival + TTFT，后续第 n 个 token deadline 是 arrival + TTFT + (n-1) * TBT；non-interactive 请求使用 TTLT deadline。每一轮 scheduler 从 decode queue 取所有 decode 请求，再从 prefill queue 选择一个或多个 prefill chunk 组成 mixed batch。dynamic chunking 根据 decode 请求数、context length、最小 slack 等状态，用轻量 random forest predictor 估计 batch latency，并选择在不违反 deadline 前提下吞吐最高的 prefill budget。hybrid prioritization 对 prefill 请求计算由 deadline 和剩余 prefill work 线性组合的 priority，参数 alpha 控制偏 EDF 还是偏 SRPF。eager relegation 把已经或即将违反 TTFT/TTLT 的请求移动到 relegated queue；在多租户场景中，priority hint 使低优先级请求先被降级。selective preemption 只允许 prefill queue 中的请求被更高优先级请求抢占，decode queue 不抢占，以保护严格 TBT。
        - 方法如何对冲 Baseline 缺陷：相对 siloed deployment，QoServe 将不同 QoS 请求放入共享 replica，利用宽松请求或低负载期间积累的 slack 提高 chunk size，让严格 QoS 类别不必永久承受最小 chunk 的吞吐损失。相对 FCFS/EDF，QoServe 不只看 arrival order 或 earliest deadline，还显式考虑剩余 prefill work、系统负载和 deadline slack，减少长请求造成的队头阻塞和过载时的级联 violation。相对 SJF/SRPF，它只在高负载时通过 alpha 和 relegation 引入短作业优先语义，正常负载下仍保持 deadline fairness。相对直接拒绝或 rate limiting，relegation 是 application-aware 的 graceful degradation：被降级请求不会永久拒绝，而是在低负载时 opportunistically service。
        - 关键 trade-off：QoServe 用更复杂的在线 scheduler 换取资源效率和 SLO 稳定性；dynamic chunk predictor 需要为每个 model、hardware、parallelism configuration 收集 profile 并训练，论文也指出 GPU clock locking 对复现实验很重要；为避免 latency over-prediction，predictor 偏向低估 chunk size，可能牺牲少量吞吐上限；alpha 需要调参，过大更像 SRPF，median latency 更低但长请求 fairness 变差，过小更像 EDF，低负载公平性好但过载稳定性弱；eager relegation 本质上要求产品侧接受 priority hint 和服务降级语义，否则公平性会从系统问题转化为业务策略问题。

    3、论文实现：
        - Baseline 如何实现：论文以 Sarathi-Serve 为基础实现和比较，Sarathi-Serve 本身扩展自 vLLM。主要 baseline 包括 Sarathi-Silo，即每个 QoS bucket 独立 GPU cluster/replica，Q1 严格 TBT 用小 chunk，Q2/Q3 用大 chunk；Sarathi-FCFS，即共享集群但按 FCFS 调度；Sarathi-EDF，即共享集群并用 earliest deadline first；Sarathi-SRPF，即按剩余 prefill token 最少优先；在 prefill/decode disaggregation 设置下还比较 Disagg-FCFS、Disagg-EDF、Disagg-QoServe。论文还以 Medha、PolyServe、SLOs-Serve 等 concurrent work 做 best-effort 或定性比较。
        - 新设计如何实现：QoServe 扩展 Sarathi scheduler，同时保留 vLLM 的 tensor parallelism 和 PagedAttention。它扩展 vLLM API，使每个 inference request 携带 QoS 要求和 priority。scheduler 内部维护 prefill queue、decode queue、relegated queue；hybrid prioritization 用 priority queue 实现；non-interactive 请求的未知 decode length 由每个应用的历史 token generation pattern 估计，并用均值加两倍标准差做保守近似；dynamic chunking predictor 是运行在 CPU 上的 random forest，输入包括 batch 请求数、context length、chunk size 等，训练数据来自 Vidur simulator profiling harness 对 MLP 和 attention latency 的 profile。论文称 chunk size prediction 误差小于 10%，运行时改变 chunk size 只需从 pending prefill queue 取相应 token 数，没有额外 GPU kernel 成本。GitHub artifact README 进一步指向 `sarathi/core/scheduler/deadline_scheduler.py` 中的 scheduling loop、chunk size predictor 调用和 eager relegation，以及 `sarathi/core/datatypes/sequence.py` 中的 SLA tier、hybrid priority 常量和 comparator。
        - 实验 / 实现平台：模型和硬件覆盖 Llama3-8B on A100-80GB TP1、Qwen-7B on A100-80GB TP2、Llama3-70B on H100-80GB TP4；数据集包括 ShareGPT、Azure Conversation trace、Azure Code trace。请求到达默认按 Poisson distribution 生成，并保留 trace 中的 prefill/decode token 数。默认 QoS 组合为三等分：Q1 interactive，TTFT=6s、TBT=50ms；Q2 non-interactive，TTLT=600s；Q3 non-interactive，TTLT=1800s。cluster-scale 实验使用 16 张 A100 GPU，4 nodes，每节点 4 GPUs，80GB，pairwise NVLink，服务 Azure Code 35 QPS、360K requests。artifact 环境要求 Ubuntu 20.04+、CUDA 12.1、Python 3.10、PyTorch 2.3.0、FlashInfer 0.1.1；推荐 4xA100 80GB 复现 Llama3-8B/Qwen-7B，Llama3-70B 需要 4xH100 80GB NVLink。
        - 关键实验设置与指标：核心指标包括 TTFT、TBT、TTLT、goodput(QPS)、deadline violation percentage、p50/p95/p99 latency。cluster-scale 中，Sarathi-Silo 需要 13 GPUs，以 Q1/Q2/Q3 = 7/3/3 replicas 达到 0.24% overall violation；QoServe 使用 10 mixed-workload replicas 达到 0% violation，节省 23% GPUs。若 silo 也压到 10 GPUs，6/2/2 配置 violation 上升到 60.4%，4/3/3 配置约 95% Q1 violation。PD colocation 单 replica 实验中，QoServe across models/datasets 相比 Sarathi-FCFS 获得 1.5x 到 2.4x goodput，相比 Sarathi-EDF 高 20-40%。dynamic chunking 单独带来约 20% throughput improvement；在 Azure Conv + Llama3-8B 上，chunk size 可升到约 2500，相比严格 TBT 下默认 256 chunk size 约 2x 吞吐。过载实验中 QoServe 可比 baseline 多承受约 40% load，并在比下一最好方案高约 30% 的 load 前维持 0 deadline violation；transient load 中，Sarathi-FCFS/EDF 出现大面积崩溃，而 QoServe 对 important requests 为 0% violation、overall violation 为 8.64%。

    4、pipeline/kernel解析：
        - 新 pipeline/kernel 是什么：论文没有提出新的底层 GPU kernel 或 attention kernel；最接近的新执行路径是 QoServe 的 QoS-aware dynamic batch creation pipeline，也就是把 deadline-aware request selection、dynamic chunking、hybrid prioritization、eager relegation 和 selective preemption 放进 Sarathi/vLLM 的 mixed prefill-decode scheduling loop。底层仍使用 vLLM/Sarathi 的 tensor parallelism、PagedAttention 和 fused prefill-decode execution；创新主要在 runtime scheduler 和 batch construction，而不是 CUDA kernel 本身。
        - 新 pipeline/kernel 的执行流例子：一个请求到达后，系统根据应用给出的 SLO 和 priority 将其放入 prefill queue。如果它是 interactive 请求，scheduler 计算其 TTFT deadline 和后续每个 token 的 TBT deadline；如果它是 non-interactive 请求，则计算 TTLT deadline，并用应用历史输出长度估计剩余 decode work。每轮调度时，QoServe 先取出 decode queue 中所有正在生成下一 token 的请求，计算这些 decode 请求的最小 slack、context length 和 batch composition；随后 CPU 上的 predictor 给出本轮可承受的 prefill budget。prefill selector 用 hybrid priority 从 prefill queue 取候选请求：如果候选请求已经或将在本轮违反 TTFT/TTLT，则 violation checker 将其放入 relegated queue，低优先级请求会更早被 relegated；如果不违反，则取不超过 prefill budget 的 prompt token 作为 prefill chunk。选出的 prefill chunk 与所有 decode token 组成 mixed batch 送入 GPU execution engine。GPU 完成本轮后，prefill 完成的请求进入 decode queue，未完成 prefill 的请求保留剩余 prompt tokens 等待后续轮次；decode 请求生成一个 token 后继续等待下一轮，直到完成。若某个 interactive 请求进入 decode phase，QoServe 会缩小或限制 prefill chunk 来满足其严格 TBT；当该请求完成或积累足够 slack 后，scheduler 又可增大 prefill chunk，以吞吐换取不违反 deadline 的资源利用率。
