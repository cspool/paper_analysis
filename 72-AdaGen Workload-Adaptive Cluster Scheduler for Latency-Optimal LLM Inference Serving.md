1、论文工作：
    - 论文要解决的核心问题：多实例 LLM serving 中，现有 cluster scheduler 主要按负载均衡或 KV cache / token 数量把请求分到不同实例，但 LLM 请求的 prompt / response 长度差异很大，且 prefill 与 decode 在 iteration-level batching 中的组织方式不同。结果是即使实例间 token 总数均衡，实例内部的 compute layout 仍可能让某些请求的 prefill 被长 decode 阶段挤压，导致 TTFT 变差、TTFT SLO attainment 下降。论文要优化的不是单个 kernel，而是跨实例请求分配后形成的 compute layout，从而降低请求延迟并提高同样 GPU 资源下的 SLO 满足率。
    - 论文的主要贡献：AdaGen 提出一个 workload-adaptive cluster scheduler，按一个请求集合 S_t 做联合调度，而不是逐请求独立调度。它先用 DistilBERT 预测每个请求的 decode length，再用 compute-layout simulator 近似不同调度决策下每个实例的 token iteration 布局，并通过三步调度逐步改进：长度感知的 prefill/decode 分类分配、分布感知的负载再平衡、selective distributed execution。论文还提出了跨实例 selective distributed execution 的快速选择方法：先给实例打分形成 source-destination pair，再用 DBSCAN 对相似请求阶段聚类，只评估代表请求，并用直接计算 token 起止位置的 heuristic 避免完整重模拟。
    - 论文所处背景：LLM inference 通常分为 prefill 和 decode：prefill 并行处理 prompt 并生成 KV cache，decode 自回归生成 response，每个请求每轮通常只能生成一个 decode token。vLLM、Sarathi-Serve、DistServe 等 instance scheduler 主要优化实例内部 batching、KV cache 管理或 prefill/decode 调度；Llumnix 等 cluster scheduler 关注多实例间负载或内存均衡。AdaGen 的切入点是：在真实 trace 中，prompt length、response length 及其分布会决定同一批请求在实例内形成怎样的 compute layout，而 compute layout 直接决定 TTFT / TBT。因此，仅靠 load balancing 不能保证低延迟。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：Round-robin 不考虑请求长度；Llumnix 把请求分配给当前 token 数更少的实例以均衡 KV cache 使用，但 decode length 在调度时未知，且负载均衡不等价于 latency-optimal compute layout。论文用 workload_1 / workload_2 说明：对某些 workload，Llumnix 确实优于 round-robin；但当请求总长度相同而 prefill/decode 组成不同，Llumnix 可能把长 prefill 请求和长 decode 请求放在同一实例，decode-prioritizing instance scheduler 会让 decode token 抢占 prefill slot，使长 prefill 请求 TTFT 变差。穷举实验中，最优调度相对现有 scheduler 可降低 P90 TTFT 约 1.5x-1.9x；若把最优 P90 TTFT 设为 TTFT SLO，round-robin 和 Llumnix 的 SLO attainment 只有约 25% 和 45%，说明 baseline 的关键缺陷是没有优化 compute layout。
    - 论文的设计方法：AdaGen 首先把到达请求按时间窗口组成 S_t，并用微调后的 DistilBERT 预测每个请求的 decode length，论文报告预测平均上下界后取均值在实验数据上约 76.4% accuracy。随后 AdaGen 用 simulator 根据 instance scheduler 规则生成近似 compute layout：以 chunked-prefill-based decode-prioritizing scheduler 为例，每个 iteration 先放 decode token，再用剩余 slot 放 prefill token，batch size 设为 2048，并估计动态 KV cache 占用。调度本体分三步：第一步按 P75 prefill length / decode length 区分长短，将长 prefill 与短 decode 共置、短 prefill 与长 decode 共置，必要时把同一请求的 prefill/decode 放在不同实例；第二步根据模拟出的 layout iteration 数识别 overloaded / underloaded 实例，把长 sequence 请求整体迁移以改善分布不均；第三步做 selective distributed execution，允许即使没有内存溢出，也把某些请求的 prefill 或 decode 阶段拆到另一个实例执行，以改善 source 实例和全局 TTFT。
    - 方法如何对冲 Baseline 缺陷：Baseline 的隐含假设是负载均衡后实例内延迟也会好；AdaGen 直接观察并优化实例内 compute layout。长度感知调度避免长 prefill 被长 decode 持续压住；分布感知再平衡处理某类请求集中导致的 layout iteration 膨胀；selective distributed execution 则进一步移动某个请求的 prefill 或 decode 阶段，腾出 source 实例中会影响 TTFT 的 slot。为避免跨实例迁移开销吞掉收益，AdaGen 在调度评估中显式考虑 KV cache migration 的通信 downtime，并优先选择高带宽连接如 NVLink。
    - 关键 trade-off：AdaGen 接受更复杂的在线调度逻辑，以换取更好的 TTFT SLO attainment。它依赖 decode length 预测，预测错误会导致 sub-optimal scheduling，因此需要周期性 on-demand rescheduling；它允许跨实例拆分请求阶段，但这要求迁移 KV cache，高带宽互连下收益更明显，低带宽场景中机会会受限；它用 simulator 和 heuristic 代替真实执行或穷举搜索，牺牲少量精确性以控制调度时间。论文的 simulator 在 Llama3-8B 上计算 TTFT / TBT 的 accuracy 分别为 99.01% / 98.83%，生成 layout 约 0.7s，而真实执行约 3.19s。

3、论文实现：
    - Baseline 如何实现：论文主要对比 round-robin、Llumnix++ 和穷举最优调度。Llumnix 代表已有 LLM cluster scheduler，按最少 token / KV cache 使用进行 load balancing；论文为了更强 baseline，把 Llumnix 增强为 Llumnix++，使其初始调度考虑预测 response length，并且实验中只报告 Llumnix++，因为它在实验里优于原始 Llumnix。instance scheduler 默认采用 Sarathi-Serve 风格的 chunked-prefill-based decode-prioritizing scheduling；论文还用 disaggregating instance scheduler 做泛化实验，验证 AdaGen 的方法不绑定某一种 instance scheduler。
    - 新设计如何实现：AdaGen 基于 vLLM 实现，论文称扩展了 3,407 行 Python 代码，支持 decode length prediction、compute layout simulator、length/distribution-aware scheduling、selective distributed execution scheduling 和 on-demand rescheduling。多个模型实例用 Ray actors 实例化，attention backend 使用 FlashAttention，通信 backend 使用 NCCL，用于 KV cache migration 以及 pipeline / tensor parallelism。S_t 的大小经验上取为 request rate，即约等于 1 秒窗口内平均到达请求数；为了隐藏调度开销，AdaGen 让 S_t 的调度与 S_{t-1} 的执行重叠。
    - 实验 / 实现平台：实验 testbed 是 2 台服务器、共 8 张 NVIDIA H100-80GB GPU；每台服务器内 4 张 H100 通过 NVLink 互连，服务器间带宽约 70Gbps。模型包括 Llama3-8B、Mixtral-8X7B 和 Llama3-70B：Llama3-8B 每实例 1 H100、8 个实例、BurstGPT 数据；Mixtral-8X7B 每实例 2 H100、TP2、4 个实例、Openchat 数据；Llama3-70B 每实例 4 H100、TP2-PP2、2 个实例、Azure LLM conversation 数据。每个数据集随机选 8k 请求；BurstGPT 和 Openchat 没有 timestamp，因此按 prior work 用 Poisson 分布生成到达时间，默认 request rate 分别为 60、20、8 req/s。
    - 关键实验设置与指标：主要指标是 TTFT SLO attainment 和 TBT SLO attainment，定义为 1 秒窗口内满足 SLO 的请求比例。Llama3-8B 的 TTFT / TBT SLO 为 0.2s / 0.1s；Mixtral-8X7B 为 2s / 0.15s；Llama3-70B 为 2.5s / 0.18s。端到端结果显示 AdaGen 相对 Llumnix++ 最高达到 2x TTFT SLO attainment，相对 round-robin 最高达到 3.6x，同时 TBT SLO attainment 基本相近。成本实验中，Llama3-8B 使用 4 个实例时 AdaGen 约达到 64% SLO attainment，而其他系统需要 1.7x-2x 更多实例才能达到类似水平，因此论文报告最高 2x cost-efficiency。500-instance stress test 使用 vLLM sleep 模拟真实 GPU 执行时长，AdaGen 仍比 Llumnix++ / round-robin 最高达到 1.9x / 3.6x TTFT SLO attainment。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：AdaGen 没有提出新的 GPU kernel；它提出的是一个 cluster scheduling pipeline，核心是“预测 decode length -> 模拟 compute layout -> 多步调度 refinement -> vLLM/Ray 实例执行 -> 根据真实状态周期性修正”。这个 pipeline 的新意在于把 LLM 请求的 prefill/decode 长度和分布显式映射到实例内 compute layout，再把 layout 中的 TTFT/TBT 变化反馈到跨实例调度决策。更细地说，它包含 length- and distribution-aware scheduling、selective distributed execution scheduling、compute-layout simulation、KV-cache-migration-aware evaluation 和 on-demand rescheduling。
    - 新 pipeline/kernel 的执行流例子：假设当前 1 秒窗口 S_t 中有一批 chatbot 请求，部分请求 prompt 很长但 response 较短，另一些请求 prompt 较短但 response 很长。AdaGen 先调用 DistilBERT 预测每个请求的 decode length，并记录已知 prefill length。接着第一步调度按 P75 阈值把 prefill/decode 分成长短类：长 prefill 请求倾向与短 decode 请求放在同一半实例，短 prefill 请求倾向与长 decode 请求放在另一半实例；如果某个请求 prefill 和 decode 都长或都短，AdaGen 可以把它们拆到不同实例，并优先选 NVLink 等高带宽连接以降低 KV cache migration downtime。
    - simulator 与分布再平衡过程：第一步得到初始分配后，AdaGen 不把它真实跑到 GPU 上，而是在 simulator 中按 instance scheduler 规则生成每个实例的 compute layout。对 decode-prioritizing scheduler，模拟器在每个 iteration 先填 decode slot，再填 prefill slot，并用运行中的 active token 计数估计动态 KV cache memory。若某实例的 layout iteration 数明显高于平均值，说明它 overloaded；AdaGen 与 iteration 数较低的 underloaded 实例配对，优先整体迁移 sequence length 更长的请求，因为长请求更快缓解 layout imbalance。
    - selective distributed execution 过程：完成整体再平衡后，AdaGen 给每个实例计算 source/destination score，因素包括 total prefill tokens、total decode tokens、请求最后一个 prefill iteration 中的平均 prefill token 数等。高分实例更像 source，低分实例更像 destination。对每个 source-destination pair，AdaGen 为请求的 prefill phase 和 decode phase 构造 layout 特征向量，用 DBSCAN 聚类相似阶段，只抽每个 cluster 的代表请求评估。若把代表请求某个阶段迁走能提高最终 SLO attainment，AdaGen 就选择该 distributed execution 决策。
    - heuristic 如何替代完整重模拟：评估一个请求阶段迁移时，AdaGen 先检查 destination 是否有足够 KV cache memory；然后只对被迁移阶段在 destination 中的放置进行局部重模拟。对于 source 中受影响的其他请求，AdaGen 不是从头模拟所有 token，而是直接计算 token 起止位置如何左移；prefill token 可以在 iteration 内连续填充，而 decode token 每个请求每轮只能有一个，因此 decode 的起止位置移动规则不同。之后 ENFORCE_LLM_RULES 保证 decode 起始 iteration 晚于 prefill 结束 iteration；若 prefill/decode 跨实例，还必须晚于 KV cache migration 完成时刻。最终根据新的 TTFT / TBT 计算 SLO attainment 改变量。
    - 运行时状态回写：最终调度决策交给 vLLM 实例执行，Ray actor 管理多个实例，FlashAttention 负责 attention 计算，NCCL 负责 KV cache migration 及并行通信。执行结束后，AdaGen 用真实 active token 计数校正 simulator 估计；如果估计与真实值差异超过 5%，后续步骤会按真实值重算。每隔约 100s，AdaGen 还会做 on-demand rescheduling：从 SLO attainment 低的 source 实例迁移请求到 SLO attainment 高的 destination 实例，并优先选短 sequence 请求以降低已生成 KV cache 的迁移量。
