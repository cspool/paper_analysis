论文标题：iRoute: Local Routing Table-based Workflow Management in Serverless Computing


Serverless workflow in cluster。

1、论文工作：
    - 论文要解决的核心问题：serverless workflow 的函数间通信同时包含 function-level orchestration 和 instance-level routing / data transfer。传统平台依赖 centralized orchestrator 与 gateway：orchestrator 解析 DAG 依赖并触发下游函数，gateway 再查找具体函数实例，中间数据通常经第三方 storage 转发。论文指出这种 intermediary intervention 对微秒到亚毫秒级 web service 很重，在 DeathStarBench Social Network posting 场景中，单个函数计算时间可低至 28 us 到 1.6 ms，但函数间数据传输可超过 3.3 ms，甚至超过计算时间 20x。
    - 论文的主要贡献：论文提出 iRoute，把 function dependency resolution 和 instance routing 都从全局组件下放到本地 sidecar。核心是 Local Routing Table（LRT）：每个函数实例旁边的 Local Routing Controller（LC）维护依赖实例地址、状态和通信 tunnel，并直接选择下游实例进行 1-hop transfer。系统采用 Local Routing Controllers + Centralized Coordinator 的双层架构，前者做本地路由和直接通信，后者只做 DAG 解析、扩缩容和多份 LRT 同步。论文还提出 fan-in 正确路由的 CH-EM（Consistent Hashing with Exploration Mode）、面向高动态负载的 instance partition，以及 at-most-once / at-least-once / exactly-once 执行语义支持。
    - 论文所处背景：该工作位于 serverless computing、serverless workflow、FaaS 数据平面和控制平面协同优化的交叉位置。目标不是 GPU kernel 或模型推理本身，而是低延迟 web service / serverless workflow 的函数级路由和中间数据传输。论文特别关注两类真实负载特征：Azure Durable Functions 中大量 workflow 调用稀疏，80% workflow 每天调用少于 100 次；同时云函数负载又可能出现一分钟内 33,000x 的 burst。因此，单纯维持 keep-alive connection 很难同时覆盖 sparse invocation、burst scaling 和资源效率。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：OpenFaaS 代表 centralized orchestrator + gateway + third-party forwarding，所有函数间通信都要经全局组件和外部存储，路径长、延迟高。Unum 把 function dependency resolution 下放到本地实例，但仍需要定位下游实例并依赖间接数据传输。Nightcore 用 IPC 加速同节点函数通信，但仍不是跨节点、跨扩缩容场景的通用 1-hop routing 方案。FMI 和 FUYAO 代表 keep-alive connection：先通过全局路由表交换实例地址，再建立持久直接连接；在调用频繁且拓扑稳定时可降低数据传输延迟，但首次 routing lookup / connection setup 有 5.4-100.7 ms 量级开销，burst 场景中频繁扩缩容会反复付出连接建立成本。更关键的是 keep-alive connection 会产生 binding scaling：下游函数必须随上游同步扩容来避免过载，论文在 FMI / FUYAO 上测到可带来 50% 到 177.8% 的额外实例资源浪费。
    - 论文的设计方法：iRoute 用 LRT 替代每次请求的 GRT 查询。LRT entry 包含 Instance ID、Address、State 和 Tunnel，其中 Tunnel 可为同节点 IPC、跨节点 Socket 或 RDMA；State 标记该下游实例在当前 partition 中是否 active。Chain 和 fan-out 模式下，LC 用 Round Robin 做负载均衡；fan-in 模式下，多个上游函数必须把同一请求的数据送到同一个下游实例，因此 iRoute 用 CH-EM，将下游实例 ID 和 request ID 映射到一致性哈希环，保证独立 LC 在本地视图一致时做出相同路由决策。扩缩容时，CC 让 LC 进入 exploration mode，在 LRT 同步期间通过下游完整性检查和 upstream re-routing 修复临时不一致造成的错路由。
    - 方法如何对冲 Baseline 缺陷：相对 OpenFaaS / Unum，iRoute 消除函数间数据传递中的 orchestrator / gateway / third-party storage 必经路径，让上游 LC 直接把输出传给下游实例。相对 FMI / FUYAO，iRoute 不依赖长期绑定连接，也不需要每次扩容都让上下游实例同步成组扩缩；LRT 已经预先携带可达下游实例地址，LC 只需本地查表即可建立或使用直接传输。相对全局路由，iRoute 用双层架构把高频 routing decision 留在本地，把低频一致性维护留给 CC；instance partition 进一步降低 LRT 同步 fan-out，只优先同步新实例所在 partition 的相关 LRT。
    - 关键 trade-off：iRoute 用分布式本地控制换取低延迟，因此需要处理多份 LRT 的一致性、fan-in 路由正确性、实例失败后的重调度，以及 LC 的可信边界。exploration mode 会增加 P99 latency，论文在 FINRA 上测到 1.9-11.1% 开销，fan-in 度越高，重路由检查越贵。at-least-once 需要 LC 临时 buffer 执行结果和路由目的，exactly-once 需要 Redis setnx 这类第三方存储做中心化 buffer，平均延迟会增加 0.5-18 ms。当前论文还承认动态 conditional workflow 需要运行时获取或预分发分支 LRT，完整方案留作 future work；LC 目前也属于 TCB，需要只读 GRT 权限、周期性 credential 刷新，若移出 TCB 还要加入 LC-CC 和 LC-LC 认证甚至 BFT 协议。

3、论文实现：
    - Baseline 如何实现：论文比较 OpenFaaS、Unum、Nightcore、FMI 和 FUYAO。OpenFaaS 使用独立 orchestrator 和基础 third-party forwarding。Unum 将函数依赖解析下放到本地实例，但数据仍走间接路径。Nightcore 用 IPC 加速同节点函数间通信。FMI 实现基于 Socket 的 keep-alive connection。FUYAO 支持 IPC 和 RDMA 的 keep-alive direct connection。为公平比较，论文按通信通道分组：FMI 对 iRoute-Socket，FUYAO-RDMA 对 iRoute-RDMA，Nightcore / FUYAO-IPC 对 iRoute-IPC；所有函数实例都 pre-warmed，以排除 cold start 影响。
    - 新设计如何实现：iRoute 用 Python3 构建在 OpenFaaS 之上。CC 与 LC 分别集成为 OpenFaaS 的 web service 和 runtime dependency library；系统替换 OpenFaaS alert-manager 为自定义 scaling engine，同时复用 gateway、faas-netes 等模块以减少工程成本。CC 作为 daemon process 运行，接收 JSON 格式 DAG，生成每个函数的 config.json，并调用 faas-cli 部署 workflow；Kubernetes 启动函数实例并分配 IP 后，CC 通过 Kubernetes API 查询实例 metadata，生成 routing table 信息，并通过 TCP direct connection 同步到 LC。GRT 存在 ZooKeeper 中，每个函数一个 ZooKeeper node，LC 通过 ACL 只读访问；LC 只有收到更新通知时才拉取最新 routing table。LC 接管函数实例主进程 index.py，完成中间数据接收后直接调用用户 handle function。为支持 IPC，函数实例挂载共享 tmpfs 目录，用 named pipe 和 shared memory buffer；LC 用 epoll 监听 pipes、sockets 和 RDMA completion event channel，并调用 os、socket 与 rdma-core 完成收发。exactly-once 语义使用 Redis setnx 保证每个函数调用只写入一个一致 buffer。
    - 实验 / 实现平台：评测使用 12 节点集群，每个节点为 Intel Xeon Gold 6338 @ 2.00GHz，128 CPU threads（64 physical cores），256 GB memory，2 TB SSD，Ubuntu 22.04；节点之间通过 25 Gbps full-bisection Ethernet 连接。Benchmark 包括 DeathStarBench Social Network（延迟敏感 create post）、Excamera（视频处理，函数计算占主导）和 FINRA（金融交易校验，可配置 fan-in 宽度 5/10/20/40）。负载来自 Azure Function production trace，覆盖 stable、burst 和 sporadic 三类 120-hour workload；keep-alive duration 设置为 10 分钟。主要指标为 data transfer latency、end-to-end P99 latency、maximum throughput 和达到给定 QPS 所需函数实例数；每个测试运行 5 次并报告平均值。
    - 关键实验设置与指标：连接建立方面，FMI / FUYAO 需要经 controller 交换实例地址，connection overhead 为 5.4-100.7 ms；iRoute 从 LRT 取目的地址并避免全局交互，将连接时间降到 0.2-2.3 ms。数据传输方面，建立直接连接后 iRoute、FMI、FUYAO 使用相同 1-hop 机制；传 512 MB 时 iRoute 延迟为 288 ms，相比 OpenFaaS 和 Nightcore 分别快 8.6x 和 2.3x；小数据下 IPC 最好，大于等于 1 MB 时 RDMA 更优。应用 P99 latency 上，iRoute 在 Social Network 和 FINRA-5 中稳定优于全部系统，FINRA-5 stable 下 iRoute-Socket 相比 FMI 最高 81.8x；相对 FUYAO-IPC 和 FUYAO-RDMA，P99 latency 最高分别降低 27.3x 和 10.4x。资源效率上，iRoute 为相同 throughput 所需实例数相比 OpenFaaS、Unum、Nightcore、FMI、FUYAO-RDMA、FUYAO-IPC 平均减少 1.6x、2.6x、1.1x、2.3x、1.8x、1.4x。最大吞吐上，在本地集群中 iRoute 相比 OpenFaaS、Unum、Nightcore、FMI、FUYAO-RDMA、FUYAO-IPC 分别最高提升 100x、80x、10x、5x、11.8x、2.3x。系统开销上，LC 对两个 no-op 函数的平均额外开销为 35 us，其中 scaling metrics、fault buffer、routing algorithm 分别增加约 7 us、12 us、16 us；instance partition 将 LRT 同步开销降低 1.9-5.6x。

4、pipeline/kernel解析：
    - 新pipeline/kernel是什么：论文没有提出新的计算 kernel；最核心的新执行路径是 iRoute 的 LRT-based serverless workflow routing pipeline。它由离线 DAG parsing / config distribution、每实例 LC sidecar、Local Routing Table、Round Robin / CH-EM routing、IPC / Socket / RDMA direct tunnel、exploration-mode re-routing、CC-driven scaling 与 partitioned LRT synchronization 组成。可以把它理解为把 serverless workflow 的“谁调用谁、数据发给哪个实例、走哪条通道”从中心 gateway 拆到每个函数实例旁边执行，而中心 CC 只维护低频全局一致性和扩缩容信息。
    - 新pipeline/kernel的执行流例子：以一个 FINRA fan-in workflow 请求为例。开发者提交 JSON DAG 后，CC 的 DAG parser 解析函数依赖，为每个函数生成 config.json，并部署 OpenFaaS workflow。Kubernetes 创建函数实例后，CC 收集实例 IP / port 等 metadata，把 GRT 写入 ZooKeeper，并把相关下游实例信息同步给每个 LC 的 LRT；同 partition 的下游实例被标记 active，Tunnel 根据位置设为 IPC、Socket 或 RDMA。请求进入第一个函数实例后，LC 接收输入并调用用户 handle function；函数完成后，LC 根据 DAG 找到下游函数。若是 chain 或 fan-out，LC 对每个下游函数从 LRT 过滤 active 实例并用 Round Robin 选择目的实例；若是 fan-in，多个上游 LC 对同一 request ID 和下游 instance ID 使用一致性哈希，独立选择同一个下游实例。选定目的后，LC 不再访问 orchestrator / gateway，而是按 Tunnel 直接通过 shared memory / named pipe、socket 或 RDMA 把中间数据发送过去。下游 LC 收到完整输入后继续调用本地函数代码。若此时发生扩容导致部分 LRT 暂时不同步，CC 会先让 LC 进入 exploration mode；下游 LC 在收到 critical path 数据后检查是否已收到所有上游结果，若发现缺失，就向可能持有缺失数据的上游实例发送 re-routing request。上游 LC 若本地 buffer 中有该请求输出，就把数据重新路由到正确下游实例，并通知原错误目的释放旧数据。扩缩容时，CC 根据各 LC 上报的 requests per second 和 request duration 计算函数并发需求，再用 partition algorithm 将新实例放入未填满 partition；同步优先覆盖新实例所在 partition 的相关 LRT，从而在保证每个 LRT 有 active routing destination 的同时避免向所有上游实例广播更新。
