    1、论文工作：
        - 论文要解决的核心问题：多 chiplet GPU 为了缓解 NUMA 和远端访问，通常会在每个 chiplet 内加入额外 cache 层，例如 L1.5/L2，并通过 inter-chiplet network 访问全局 LLC。这个结构提升了普通数据访问的局部性，却显著放大了显式同步操作的成本。传统 GPU 同步在 acquire 时要 invalidate 本地 cache，在 release 时要 flush dirty data，并且 atomic synchronization 通常绕过本地 cache 到 LLC 执行；放到 multi-chiplet GPU 后，额外 cache 层让 invalidate/flush 更贵，跨 chiplet atomic 又受 inter-chiplet bandwidth 限制。论文用 4-chiplet GPU 与等价但不可制造的 monolithic GPU 对比，指出带全局同步的 workload 因额外 cache-level invalidation 平均损失 22.5%，因 remote atomic access 平均损失 23.5%，整体相对 monolithic GPU 平均下降 50.5%。
        - 论文的主要贡献：第一，提出 LRM-GPU，把 lazy release consistency 引入 multi-chiplet GPU 的显式同步路径，只在 synchronization variable 的 owner 跨 chiplet 迁移时才对额外 cache 层执行 coherence action，避免每次 acquire/release 都保守地 invalidate/flush L1.5。第二，在 LLC 中加入 synchronization variable directory，只跟踪同步变量的 owner，而不是像完整 coherence protocol 那样跟踪所有数据 sharer，因此目录容量较小。第三，提出 in-network atomic merge，在 chiplet 网络中嵌入 Synchronization Atomic Merge Unit (AMU)，把同地址、可合并的跨 chiplet atomic 请求在网络内合并，减少同步 atomic traffic。第四，基于扩展 GPGPU-Sim 和 BookSim 2.0 的评估表明，相比 MCM-GPU 平均加速 1.33x，相比 HMG 平均加速 1.22x，并相对 HMG 平均减少 52% inter-chiplet traffic 和 32% energy consumption。
        - 论文所处背景：MCM-GPU / multi-chiplet GPU 用多个较小 die 组成逻辑上的单个 GPU，绕开单片大 GPU 的工艺、良率和 die-size 约束。已有工作主要优化普通内存访问的 NUMA 问题，例如 first-touch page allocation、distributed CTA scheduling、远端数据缓存、request coalescing、nearby fetching 或 LLC 组织重构。但当 GPU 运行 graph、HPC、锁、barrier、semaphore、共享数据更新等需要显式同步的 workload 时，性能瓶颈不再只是普通数据 locality，而是同步语义要求下的 cache 一致性动作和跨 chiplet atomic 流量。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：MCM-GPU baseline 沿用传统 GPU 的同步做法：同步变量不在本地 cache 中维护一致副本，acquire invalidate L1 和 L1.5，release 保证 dirty data 回到 LLC，atomic operation 直接路由到 LLC。这个方案简单，但在 multi-chiplet GPU 上会产生两个硬瓶颈。其一，额外 L1.5 cache 层让每次同步都带来更大范围的 invalidate/flush，即使连续同步都发生在同一个 chiplet 内也会重复付费。其二，atomicCAS、atomicAdd 等同步 atomic 跨 chiplet 访问同一地址时，大量失败的 lock spin 或共享变量 update 请求会穿过 inter-chiplet network，受有限带宽和网络排队影响。
        - 既有方案的不足：hLRC 通过追踪同步变量 ownership 来利用 locality，但它把同步变量缓存到多级 cache 中；当同步变量在不同 SM 间迁移时，需要把远端 cache 中的同步变量写回，期间其他线程无法注册并访问该变量，竞争激烈时会增加等待和重试流量。HMG 使用面向 multi-GPU / multi-chiplet 的 coherence protocol 和 hierarchical sharer tracking，能避免部分 L1.5 invalidation，但需要为写入数据向 sharer 发送 invalidation request，复杂度和 inter-chiplet coherence message 仍较高；论文还观察到 HMG 在 MST、pagerank 等 atomic 密集 workload 上可能因大量 write-invalidation 而性能下降。ARC、LAB、Atomic Cache 等 GPU atomic 优化主要在 SM 内、warp 内或 cache 内处理 atomic，没有利用跨 SM / 跨 chiplet 同步 atomic 的网络级合并机会。
        - 论文的设计方法：LRM-GPU 的核心机制是“同步变量 owner locality + 网络内 atomic locality”。对于 acquire/release，它把每个 synchronization variable 关联到最近访问它的 chiplet，并在 LLC 维护 sync-val directory。若下一次同步仍来自同一 chiplet，说明同步相关数据的最新副本仍在本 chiplet 的 L1.5/local path 中，LRM-GPU 就延迟额外 cache 层的 coherence action，只从 LLC 处理同步变量本身，不 invalidate/flush L1.5。只有当 owner 从一个 chiplet 转移到另一个 chiplet，才 flush 旧 owner 的 L1.5、更新 owner，并 invalidate 新 owner 的 L1.5 以保证后续读到全局最新数据。对于跨 chiplet atomic，LRM-GPU 在每个 chiplet 网络中放置 AMU，检测并合并同地址 atomic 请求，例如多个 atomicAdd(addr, 1) 合并为 atomicAdd(addr, n)，多个对同一 lock 的 atomicCAS 在比较值相同情况下只发送一个可能成功的请求，其余等待合并请求返回。
        - 方法如何对冲 Baseline 缺陷：对 MCM-GPU 的“每次同步都 invalidate/flush”问题，LRM-GPU 用 directory 记录 owner，把同步开销从“每次同步触发”改成“跨 chiplet ownership 变化才触发”，因此能利用同 chiplet 内连续同步的时间局部性。对跨 chiplet atomic 的网络压力，AMU 让多个相同地址的 atomic 请求先在网络入口合并，减少实际跨 chiplet 传输次数、LLC 端 atomic 操作数和返回广播次数。对 HMG 的完整 coherence 复杂度，LRM-GPU 只跟踪同步变量 owner，不维护所有 data cache line 的 sharer 状态，因此状态空间和目录容量更小。
        - 关键 trade-off：第一，LRM-GPU 的收益依赖同步 locality；当 chiplet 数从 4 增到 8，论文报告加速从 1.33x 降到 1.21x，因为同步变量跨 chiplet 迁移更频繁、owner locality 变弱。第二，AMU 合并需要倒计时窗口和 merge table，窗口太短合并机会少，太长可能增加单个 atomic 请求等待；论文没有在正文中给出完整窗口参数敏感性。第三，AMU 支持的是可交换/无序或可安全合并的 atomic 类型，对不能合并或比较值不同的 atomicCAS，请求仍要直接发送或单独建表项。第四，设计引入硬件面积、功耗和验证复杂度：AMU 在 40nm 下总面积 1.84 mm2、功耗 301.44 mW；sync-val directory 很小，但仍要和 LLC、network、同步语义共同验证。

    3、论文实现：
        - Baseline 如何实现：MCM-GPU baseline 在 GPGPU-Sim 中实现，包含 L1.5 cache、first-touch page allocation 和 distributed CTA scheduling。同步机制类似传统 GPU：acquire 时 invalidate L1/L1.5；release 时保证 dirty data 写回 LLC；由于论文配置中 L1 和 L1.5 为 write-through，LLC 不会保存过期数据，因此不需要额外 flush L1/L1.5。hLRC baseline 也在 GPGPU-Sim 中实现并扩展到 multi-chiplet GPU，它跟踪同步变量注册位置，并让同步变量以 write-back 方式缓存在多级 cache 中。HMG baseline 作为 state-of-the-art chiplet-based multi-GPU coherence protocol，被移植到 multi-chiplet GPU 场景；它使用 SM-side LLC、write-through LLC policy，并用 LLC coherence directory 跟踪所有数据，每个 chiplet 12K entries，每个 entry 覆盖四个 cache line。
        - 新设计如何实现：LRM-GPU 在 GPGPU-Sim 中实现了 lazy release consistency 的 coherence action 逻辑，并在 LLC 增加 synchronization variable directory。4-chiplet 系统中 directory 共 64 entries，每个 chiplet 16 entries；entry 记录同步变量地址、valid bit 和 owner chiplet。LRM-GPU 还在每个 chiplet 网络中加入 AMU，AMU 有 16 channels，merge table 有 2K entries、16 banks，每个 entry 的 SM list 最多记录 8 个 SM ID，data 字段为 32B。AMU 由 merge table、instruction decoder、ALU 和 multicast unit 组成；key 使用 CAM 查询 status/opcode/address，data 使用 SRAM 保存 SM list 和操作数，CAM/SRAM 均采用 dual-port 以支持并发读写。
        - 实验 / 实现平台：作者扩展 GPGPU-Sim 来模拟 multi-chiplet GPU，并基于 BookSim 2.0 构建 chiplet interconnection platform，支持 composable network construction、heterogeneous router/crossbar microarchitecture、network integration、modular routing 和 chiplet interface incorporation。评估系统包含 4 个 chiplet、总计 256 SM，每个 chiplet 64 SM；每个 SM 有 128KB L1 data cache，128B line、4-way、write-through；每个 chiplet 有 2MB L1.5 cache，128B line、16-way、write-through，仅缓存 remote data；全局 LLC 为 8MB、128B line、16-way、write-back；L1.5+LLC 总容量 16MB。inter-chiplet bandwidth 为 768GB/s，每 hop latency 为 32 cycles；DRAM 为 64 channels、3TB/s；page size 4KB，采用 first-touch page allocation 和 distributed CTA scheduling。
        - 关键实验设置与指标：workload 包含同步 microbenchmarks，如 atomicTreeBarr、lfTreeBarr、spinMutex、sleepMutex、faMutex、spinSem2、spinSem10、spinSem120；也包含带全局同步的 reduce、scan、histogram、pagerank、barnes-hut、hash-table、minimum spanning tree；另测不带全局同步的 b+tree、backprop、bfs、dwt2d、nn、lavaMD、VGG16 forward、GPT-2 forward、hotspot，以评估副作用。workload 用 CUDA 11.1、O3 编译，重点测需要 global synchronization 的 kernels。指标包括 normalized speedup、L1.5 invalidation count、inter-chiplet traffic、energy、area/power overhead，以及 chiplet scale 和 inter-chiplet latency sensitivity。
        - 主要实验结论：相对 MCM-GPU，LRM-GPU 在同步 microbenchmarks 上平均提升 1.19x，在全局同步 workload 上平均提升 1.33x；AMU-only 平均贡献 1.16x。相对 HMG，LRM-GPU 平均提升 1.22x。LRM-GPU 相比 MCM-GPU 平均减少 30% L1.5 cache invalidations；相比 MCM-GPU 平均减少 28% inter-chiplet traffic，其中 AMU 贡献 12% traffic reduction；相比 HMG 平均减少 52% inter-chiplet traffic。能耗方面，LRM-GPU 相比 MCM-GPU 平均降低 18%，相比 HMG 平均降低 32%。对没有 global synchronization 的程序，LRM-GPU 与 MCM-GPU 平均性能差异约 2%，说明新增同步路径对普通 workload 干扰较小。AMU 的 energy breakdown 中新增 AMU 只占系统能耗约 0.13%；AMU 在 40nm 下 merge table 功耗 185.51mW、面积 1.52mm2，其他逻辑功耗 115.93mW、面积 0.32mm2，总功耗 301.44mW、总面积 1.84mm2；sync-val directory 容量约 0.4KB，仅约为一个 L1 cache 容量的 0.3%。

    4、pipeline/kernel解析：
        - 新 pipeline/kernel 是什么：论文没有提出新的 CUDA kernel 或软件 runtime kernel；它提出的是 multi-chiplet GPU 显式同步路径上的硬件执行 pipeline，可概括为“LRM synchronization pipeline”：同步变量访问先经过 LLC 中的 sync-val directory 决定是否触发 L1.5 coherence action；跨 chiplet atomic 请求进入网络时经过 AMU 尝试合并；合并后的请求再进入 inter-chiplet network 和 LLC 执行 atomic / synchronization operation；响应由 AMU multicast 回所有被合并的 SM。
        - Acquire/release pipeline：当一个 SM 对同步变量 X 发起 acquire 时，系统先在 LLC 的 sync-val directory 查询 X。若 X 不存在且有空 entry，directory 为 X 分配 entry、记录当前 chiplet 为 owner，从 LLC 读取同步变量，并 invalidate 本地 L1.5，确保后续数据访问读到全局最新值。若 X 的 owner 是本 chiplet，acquire 直接从 LLC 读取同步变量，不对 L1.5 做 coherence action，因为最新数据仍可由本 chiplet 的 L1.5 提供。若 X 的 owner 是远端 chiplet，则先 flush 远端 owner 的 L1.5，把 dirty data 写回 LLC，再把 owner 更新为当前 chiplet，读取同步变量，并 invalidate 当前 chiplet 的 L1.5。若 directory 满，需要按 LRU 等策略 evict 一个 entry，并在必要时 flush 被驱逐 owner 的 L1.5，再为当前 X 建立 entry。release 的逻辑类似：local owner 情况下只把同步变量写入 LLC，不立即 flush/invalidate L1.5；remote owner 或 evicted 情况下才执行旧 owner flush、owner 更新和本地 invalidate。
        - Acquire/release 执行流例子：论文用三个 SM 竞争 lock 的例子说明 LRM-GPU 的收益。SM0 和 SM1 在 chiplet0，SM2 在 chiplet1，执行顺序为 SM0 -> SM1 -> SM2。SM0 首次 acquire lock 后，directory 中没有 X，于是记录 owner=chiplet0，并 invalidate chiplet0 的 L1.5；SM0 修改共享数据 A 后 release，只把 X 写回 LLC，不 flush L1.5，因为 owner 未变。随后 SM1 在同一 chiplet0 acquire X，directory 发现 owner 仍是 chiplet0，于是无需 invalidate/flush L1.5，SM1 可以继续利用 chiplet0 内的最新 A。最后 SM2 在 chiplet1 acquire X，owner 从 chiplet0 转到 chiplet1，此时才 flush chiplet0 的 L1.5，使 A=2 写回 LLC，再 invalidate chiplet1 的 L1.5，让 SM2 读取最新 A 并继续执行。相比 MCM-GPU 每次 acquire/release 都对 L1.5 付费，LRM-GPU 把 coherence action 集中到真正跨 chiplet handoff 的边界。
        - AMU pipeline：当 SM 发出的 atomic 请求进入网络时，硬件先判断它是否为跨 chiplet atomic。若不是，直接按普通路径发送；若是，则送入 AMU。AMU 查询 merge table：如果存在 status=valid、opcode/address 可匹配的 entry，就用 ALU 将新请求与 entry 中数据合并，并把发起 SM 的 ID 加入 SM list；如果不存在匹配 entry 且有空 entry，则分配新 entry，并设置 countdown timer；如果 entry 已经 reserve 或没有空位，请求直接发送。timer 到期或 SM list 满后，entry 中的合并请求被发送到 LLC，entry 状态从 valid 变为 reserve，禁止继续合并。响应返回时，AMU 用地址查询 merge table，若命中 reserve entry，就根据 SM list 通过 multicast unit 向所有对应 SM 广播响应，并释放 entry；若不命中，则直接转发给目标 SM。
        - AMU 执行流例子：假设 SM1 先发出 atomicAdd(addr0, 1)，AMU 为它建表项。随后 SM0 也发出 atomicAdd(addr0, 1)，AMU 命中同地址同 opcode entry，把请求合并为 atomicAdd(addr0, 2)，并记录 SM0、SM1。若 SM1 又发 atomicAdd(addr1, 1)，AMU 因地址不同分配新 entry。倒计时到期后，addr0 的合并请求被送出，状态变成 reserve；此时如果又来了 atomicAdd(addr0, 3)，不能再与 reserve entry 合并，只能新建 entry 或直接发送。等 addr0 的响应返回，AMU 按 SM list 把结果广播给 SM0 和 SM1。对于 atomicCAS，AMU 只有在同地址且比较数据相同时才合并；因为多个 CAS 竞争同一个 lock 时最多一个成功，其他请求可等待合并请求的返回并得到失败响应。
        - pipeline 边界与适用性：LRM-GPU 的 pipeline 位于硬件 cache coherence / inter-chiplet network / atomic synchronization 路径，而不是应用代码、CUDA kernel、compiler pass 或 LLM serving scheduler。它对有显式 global synchronization、锁、barrier、semaphore、atomic update 的 workload 最有价值；对没有 global synchronization 的普通 ML inference workload，论文测得平均性能影响很小。其限制也来自这里：若 workload 同步变量在 chiplet 间频繁迁移，owner locality 下降；若 atomic 类型不可合并，AMU 收益下降；若真实硬件的 interconnect、cache policy、directory eviction、atomic memory ordering 与模拟假设差异较大，收益需要进一步通过公开 artifact 或硅后数据验证。
