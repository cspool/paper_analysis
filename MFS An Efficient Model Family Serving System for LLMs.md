    1、论文工作：
        - 论文要解决的核心问题：现有 LLM serving 优化通常以“单个模型”为单位设计 batching、KV-cache 管理和采样加速，但真实服务商往往同时提供一个 model family，例如 Llama2-7B / 13B / 70B 这类结构相似但规模不同的模型。由于不同模型的层数、head 数、hidden size 和参数不同，传统系统很难把不同模型的请求放进同一个高效 batch；KV-cache 又绑定具体模型参数和层结构，不能在小模型和大模型之间直接共享。这导致 model-family serving 同时面临 GPU 显存重复占用、decode 阶段 GPU compute underutilization、跨模型调度成本高，以及服务商为满足不同 QoS 而过度部署多套模型的问题。
        - 论文的主要贡献：论文提出 MFS，一个面向 LLM model family 的 multi-tier serving system。核心思想是用 Knowledge Precipitation 将家族中最大模型微调成一个嵌套式 multi-tier 模型：低 tier 对应小模型，高 tier 对应大模型，前若干层 / tier 之间共享参数和中间状态。基于这个结构，MFS 提出 group batching，让不同 tier 中共享模型段的请求一起执行；提出 attention fusion，用拼接 QKV 的方式提升小 batch decode attention 的 GPU 并行度；提出 shareable KV-cache，使不同 tier 切换时能复用已有 KV-cache；并把 speculative sampling 适配到 multi-tier 模型中，让低 tier 作为 draft、高 tier 作为 target，同时复用 lower-tier KV-cache。
        - 论文所处背景：该工作属于 LLM serving、multi-model serving、KV-cache memory management、continuous batching 和 model-system co-design 的交叉方向。它假设服务场景正在从用户指定固定模型，转向用户指定 latency / quality 等需求的 model-less inference；在这种场景下，系统可以选择不同 tier 满足请求，而不必维护多套完全独立模型。论文利用两个事实：同一家族 LLM 多为 stacked decoder-only Transformer，结构具有统一性；Transformer 存在 layer / head 冗余，允许在保持质量的前提下构造低计算量 tier。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：Orca、vLLM、PagedAttention 等 serving 优化主要服务一个完整模型。Orca 的 selective batching 假设 batch 内请求共享同一模型结构，因此无法高效跨不同规模模型 batching；独立部署小模型和大模型时，模型权重和 KV-cache 都需要分别保存，显存占用随模型数增加而上升；在 speculative sampling 中，小 draft model 和大 target model 参数不一致，小模型生成后大模型无法直接复用小模型 KV-cache，切换模型会带来重复 prefill / decode 计算。构造 multi-tier 模型的直接 baseline 也有缺陷：简单 early-exit / strawman 只在中间层加输出头，低 tier 会生成无意义文本；PEFT / LoRA 版本仍不能解决生成质量；从头训练 early-exit LLM 代价极高；deep pruning 会破坏层连续性，使增量计算和 KV-cache sharing 难以成立。
        - 论文的设计方法：MFS 首先用 Knowledge Precipitation 对最大模型做 full-parameter fine-tuning。每个 tier 有独立 loss，训练目标是 L = L0 + λ1L1 + ... + λiLi，使低 tier 获得独立语言建模能力，同时由高 tier 的梯度传递知识。对 n-tier 模型，论文采用 step-by-step fine-tuning，逐层向低 tier “沉淀”知识，避免一次性把所有 tier loss 加起来导致低 tier 性能不可控。系统感知优化方面，MFS 最终选择只按 layer 划分 tier，而不是同时切 head，因为 layer 串行执行、对 latency 影响更大；attention 计算又需要所有 head 保持一致，切 head 会破坏各 tier KV-cache 一致性。tier 边界不是按参数量线性切，而是测量最大模型前若干层的实际推理 latency，使某个 tier 的 latency 对齐对应小模型。
        - 方法如何对冲 Baseline 缺陷：相对独立部署多个模型，MFS 用一个嵌套模型承载多个服务等级，直接共享低层参数，降低模型权重显存占用。相对传统跨模型 serving，multi-tier 结构让不同 tier 的公共前缀层完全相同，因此公共层请求可以 group batching；相对独立 draft / target speculative sampling，低 tier 产生的 hidden states 和 KV-cache 对高 tier 仍然兼容，高 tier 继续执行剩余层时无需重算前面层。相对 deep pruning，MFS 保持层连续性，KV-cache 不会因为删除中间层而失效。相对普通 selective batching，attention fusion 通过把不同请求的 QKV 拼接起来，牺牲少量冗余 attention 计算，换取 GPU 并行度和 decode 阶段吞吐。
        - 关键 trade-off：MFS 把离线微调成本、模型结构约束和一定冗余计算换成在线 serving 的显存与延迟收益。Knowledge Precipitation 需要对最大模型做 full-parameter fine-tuning，论文实验在 Llama2-13B / Qwen-14B 级别验证，但受 16× H800 资源限制没有实际微调 Llama2-70B，因此大模型家族规模上的成本和效果仍有外推成分。只按 layer 切分保持了 KV-cache 兼容，却限制了 tier 粒度；attention fusion 会计算跨请求 token 间本不需要的 attention，论文认为 GPU 并行能力可掩盖这部分时间，但这是 compute 换 parallelism。MFS 不规定如何为请求选择 tier，调度策略是 plug-and-play，因此质量 / latency 决策仍依赖外部策略或用户指定模型。

    3、论文实现：
        - Baseline 如何实现：质量评估中，论文用公开 Llama2-7B-chat、Llama2-13B-chat 作为原始模型 baseline，并额外把同一 fine-tuning 数据直接用于原模型得到 FT-7B / FT-13B，以排除数据集本身对指标的影响。构造方法 baseline 包括 strawman early exiting、PEFT / LoRA、从头训练 early-exit 和 deep pruning，论文主要以设计论证和局部实验说明它们不满足 MFS 的质量、成本或 KV-cache 兼容要求。系统性能 baseline 主要是 Orca：batching 实验关闭 Orca scheduler，重复注入同一 batch，以比较引擎执行时间；end-to-end 实验合成 generative request trace；KV-cache sharing 和 speculative sampling 实验中，Orca 代表独立小模型 / 大模型部署，无法共享参数和 KV-cache。
        - 新设计如何实现：MFS 的系统组件包括 front-end、request pool、tier-level scheduler、multi-tiered model 和 KV-cache manager。front-end 接收 prompt 和所需模型 / tier，request pool 保存待处理请求，tier-level scheduler 为不同 tier 维护队列并决定执行顺序，multi-tiered model 替代原始 model family 执行不同 QoS 请求，KV-cache manager 负责 multi-tier 模型的 KV-cache 保存与复用。分布式部署上，MFS 支持 intra-layer parallelism 和 inter-layer parallelism；类似 Orca，每个 worker 负责一个 inter-layer partition，worker 内多个 GPU 做 intra-layer parallelism。由于 tier 是层连续的，MFS 在做 inter-layer partition 时把同一 tier 的层放在一起，使不同 partition 可异步执行，减少同步开销。
        - 实验 / 实现平台：训练基础设施为两台服务器，每台 8 张 NVIDIA H800 SXM5 GPU、2 × 56 核 Intel Xeon Gold CPU、2TB 内存，两台服务器之间通过 8 个 400Gbps NDR InfiniBand 互连。推理服务实验使用两类环境：2× NVIDIA A100 GPU 服务器，配 2 × 48 核 Intel Xeon Gold CPU 和 512GB 内存；以及 8× NVIDIA 3090 GPU 服务器，配 80 核 Intel Xeon Gold CPU 和 256GB 内存。模型主要选择 Llama2 family；由于资源限制，fine-tuning 实验覆盖 Llama2-7B-chat 和 Llama2-13B-chat，并用 Qwen1.5 的 Qwen-14B / Qwen-7B 做泛化验证。fine-tuning 数据为 HuggingFace guanaco-llama2 约 9.85k 对话；优化器为 AdamW，学习率 2e-5，half-period cosine learning rate，weight decay 0.1，gradient clipping 0.3，8 次 gradient accumulation，等效 batch size 64，输入序列长度 4096，fine-tune 1 epoch / 2500 iterations，耗时约 24 小时。
        - 关键实验设置与指标：质量指标包括 MMLU、PIQA、OpenBookQA、HellaSWAG、BoolQ、ARC Easy、ARC Challenge、ANLI-R1/R2/R3。Llama2-7B 转两 tier 时，MFS-7B 在 10 个指标中 8 个优于 Llama2-7B；Llama2-13B 转两 tier 时，MFS-13B / MFS-7B 分别有 8 / 7 个指标优于对应 Llama2-13B / Llama2-7B。system-aware optimization 中，论文通过测量 latency 发现 Llama2-13B 的前 24 层可对齐 Llama2-7B 的 latency，并据此切 tier。三 tier 实验包括 13B / 7B / 3B 和 13B / 10B / 7B，结果显示各 tier 质量接近对应 baseline。系统指标包括 request execution time、median end-to-end latency per generated token、JCT、GPU memory footprint 和 GPU utilization。论文报告 MFS 相比 Orca 在 batching 场景中 JCT 最多提升 31.2%，per-token response latency 最多降低 56.1%；KV-cache sharing 最多降低 47.8% GPU memory footprint；speculative sampling 场景中 GPU utilization 从 Orca 的约 23.9% 提升到 MFS 的约 59.8%。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：论文没有提出新的底层 GPU attention kernel；最接近的新执行路径是 MFS multi-tier model family serving pipeline。它由离线 Knowledge Precipitation pipeline 和在线 tier-aware serving pipeline 两部分组成。离线部分把最大 LLM 变成多出口、层连续、参数共享的 multi-tier 模型；在线部分围绕这个模型做 tier-level scheduling、group batching、attention fusion、shareable KV-cache 和可选 speculative sampling。它改变的是 serving runtime 的请求组织、模型结构和 KV-cache 生命周期，而不是单个 CUDA kernel 的数学定义。
        - 新pipeline/kernel的执行流例子：假设系统中同时到达三类请求：低 latency 请求只需 tier-1，中等质量请求需要 tier-2，高质量请求需要 tier-3。front-end 把 prompt 和 tier 需求写入 request pool；tier-level scheduler 将请求放入不同 tier 队列，并按优先级、公平队列或外部策略选择下一批执行。执行 tier-1 的公共前缀层时，tier-1 请求、tier-2 请求和 tier-3 请求都需要经过同一段模型层，因此 scheduler 将它们组成 group batch 一起运行。decode attention 中，由于每个请求只生成一个 token，普通 batch 很小，MFS 把各请求的 QKV 矩阵拼接后进行 attention fusion，以提升 GPU 并行度；产生的 KV-cache 由 KV-cache manager 记录为 multi-tier 可共享状态。如果某个请求只要求 tier-1，完成 tier-1 输出后即可采样并返回；如果请求需要 tier-2 或 tier-3，系统继续执行后续层，同时复用前面层已经生成的 hidden states 和 KV-cache。若启用 speculative sampling，tier-1 先作为 draft model 快速提出若干 token，高 tier 再作为 target model 验证；由于 draft 和 target 是同一嵌套模型的不同 tier，高 tier 可以直接继承低 tier 的 KV-cache，而 Orca 式独立小模型 / 大模型组合则需要为 target model 重新计算或维护另一份 KV-cache。
