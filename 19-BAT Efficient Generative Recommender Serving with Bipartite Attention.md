    1、论文工作：
        - 论文要解决的核心问题：Generative Recommender（GR）把推荐排序阶段从传统 DLRM 的 embedding/MLP 范式推向 LLM/Transformer 范式后，表达能力更强，但在线 serving 成本急剧上升。GR ranking 需要把用户画像、100 到 200 个候选 item 和系统指令编码成长 prompt，再执行类似 LLM prefill 的长序列自注意力；论文在 A100 上测量 1B 到 7B 模型、512 到 8K 输入长度时，发现单请求计算延迟容易超过 100 到 200 ms 的推荐系统 SLO。Prefix caching 原本可以复用已构造 KV cache 来减少 prefill 计算，但在推荐场景中，朴素 User-as-prefix 只能复用同一用户多轮请求里的 user profile cache，面对 10^8 级用户、高度个性化画像、实时检索出的动态候选 item，实际 cache hit ratio 可低至 18%，总计算节省不足 11%。因此核心瓶颈是 compute-bound GR prefill 与低 KV cache 复用率叠加，而不是单个 Transformer 算子本身不够快。
        - 论文的主要贡献：第一，刻画 GR serving 与普通 LLM serving 的差异：用户 token 分布长尾、用户访问频率长尾、item 访问频率高度偏斜，且 item token 约占请求 token 的 33% 以上。第二，提出 Bipartite Attention，利用推荐 prompt 中 user 与 item 语义近似 permutation-invariant 的观察，让系统可在 User-as-prefix 与 Item-as-prefix 两种注意力组织之间切换；通过 attention mask 和 position encoding 调整，使 item KV cache 不依赖具体用户上下文，从而可跨用户复用。第三，设计 Bat serving system，用 compute-storage disaggregated KV cache pool 分别管理 user-prefix 和 item-prefix cache。第四，提出 hot-replicated cold-sharded（HRCS）item cache placement 和 hotness-aware prompt scheduling，在有限 CPU/GPU host memory 下动态决定每个请求应使用 user cache 还是 item cache。实验显示，相比传统 User-as-prefix，Bat 吞吐最高提升 1.6x；相比完全重算，cache hit rate 最高到 58%，吞吐最高提升 2.3x，并在总体计算量上最高减少 58%。
        - 论文所处背景：推荐系统通常分为 retrieval 和 ranking。Retrieval 用轻量模型从海量 item 中取出候选集合；ranking 用更复杂模型重排候选 item，对 CTR、收入等业务指标影响更大。传统 DLRM 在容量扩大后容易遇到表达能力瓶颈，工业界开始部署 GR/HSTU/LLM 式推荐排序模型，例如 Meta HSTU 报告过 12.4% topline metrics 改善，作者也在电商广告排序场景使用 Qwen2-1.5B 作为 GR 获得 CTR 改善。但 GR serving 的计算成本接近 LLM prefill，且推荐场景的请求结构不是普通 chatbot prompt：user profile 极个性化，item 由实时检索模块每次动态选择。Bat 正是在 LLM prefix caching、推荐 ranking、KV cache serving 和调度系统的交叉点上做算法-系统协同。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：论文主要比较四类 baseline。RE（Recomputation）完全不使用 prefix cache，每个请求都重算 user、item 和 instruction 的全部 token，因此计算量最大。UP（User-as-prefix）是现有 GR 工作常见组织方式，prompt 为 [U, I1, ..., In, Instr]，只缓存 user profile；问题是 inter-user sharing 被个性化阻断，而 intra-user item sharing 又被实时 item retrieval 阻断，因此只有同一用户反复访问时才能命中，面对大量 inactive user 时 compulsory miss 和 cache pollution 很严重。IP（Item-as-prefix）固定对所有请求使用 item prefix，能跨用户复用 item cache，但对活跃且 user profile 很长的用户不一定最优，而且 item cache 占用额外内存并可能引入跨节点通信。cache-agnostic scheduling 只按 user token 数和 item token 数贪心选择 prefix，不看 cache hotness，因此会把很多低频长用户画像放入 user cache，导致有限 cache 被低价值条目污染。
        - 论文的设计方法：Bipartite Attention 提供两种等价的 GR 推理视图。User-as-prefix 中输入为 [U, I1, ..., In, Instr]，缓存 U 的 KV；Item-as-prefix 中输入为 [I1, ..., In, U, Instr]，缓存 items 的 KV。为了让 item cache 独立于其他 token，论文对 attention mask 和 position encoding 做调整：不同 item 之间的 cross-attention 被 mask 掉；每个 item 的起始 position ID 对齐，使 item token 不带入 “排在第几个候选” 或 “排在用户之后” 的偏置。系统层面，Bat 由 hotness-aware prompt scheduler、inference workers、KV cache workers 和 cache meta service 构成。Scheduler 根据 cache hit/miss、用户热度、item 热度和 token 长度决定 prompt order；inference worker 基于 vLLM 和 FlashInfer 执行修改后的 attention；KV cache worker 用 PagedAttention 兼容的 page 管理 CPU/GPU memory；meta service 维护 user/item KV entry 的索引和热度，但不存放实际 KV tensor。
        - 方法如何对冲 Baseline 缺陷：相对 RE，Bat 直接通过 prefix cache 减少 prefill token 计算。相对 UP，Bipartite Attention 释放了 item cache 跨用户复用机会：热门 item 可以被成千上万甚至百万用户共享，而不是绑定到某个用户上下文；论文 trace 显示约 90% item accesses 集中在 top 10% hot items。相对固定 IP，Bat 没有简单地把所有请求改成 item prefix，而是用 hotness-aware prompt scheduling：当 user token 少于 item token 时直接选 Item-as-prefix；当 user token 更长时，scheduler 查询当前 user cache 中最低频条目的预测频率，如果 incoming user 的滑动窗口频率更高，才使用 User-as-prefix 并替换低频 user cache，否则回退 Item-as-prefix。相对全复制或全 hash item placement，HRCS 根据网络带宽、模型 prefill 时间、候选 item 数、平均 item token 数和 KV worker 数估算可接受的远程 KV 通信比例，只复制最热 item，冷 item 均匀 sharding，从而在节省内存与控制网络开销之间折中。
        - 关键 trade-off：Bat 接受 prompt 组织和位置编码变更带来的模型敏感性。论文报告 Llama3-Instruct 等 instruction-tuned 模型在 Item-as-prefix 下可能质量下降，因为模型区分 set semantics 与 sequence semantics 的能力不同；实践上需要选用支持 position encoding 调整的 base model，或在周期性 fine-tuning 时适配，必要时结合 PIC/CacheBlend 类 position-independent caching。系统上，Item-as-prefix 增加 item KV cache 内存占用；HRCS 降低全复制内存成本，但冷 item 访问可能产生网络通信。调度上，Bat 依赖滑动窗口频率预测用户短期热度，适合用户行为在分钟级窗口内有连续性的推荐流量；如果访问模式高度随机，hotness-aware 决策收益会降低。最后，论文主要评估 CPU memory 作为 KV cache 池，未完整探索更慢的磁盘/远端存储层，因为它可能引入有害访问延迟和复杂 IO 管理。

    3、论文实现：
        - Baseline 如何实现：所有 serving baseline 都基于 vLLM 和 FlashInfer 实现模型推理，并使用 CPU memory 作为 LRU KV cache。RE 不启用 prefix caching；UP 对所有请求采用 User-as-prefix attention，这是论文认为现有 GR 工作中广泛使用的方式；IP 对所有请求采用 Item-as-prefix attention；cache placement 对比包括 Bat-Replicate（所有机器全量复制 item cache，剩余内存给 user cache）和 Bat-Hash（item cache 在 4 台机器上 hash/round-robin sharding）。Prompt scheduling 对比中，cache-agnostic baseline 只比较 user token 和 item token 数，选择 token 数更多的一侧作为 prefix。
        - 新设计如何实现：Bat 的 inference worker 每个绑定一张 GPU，模型权重本地复制，基于 vLLM serving engine 修改 attention module 以支持 Bipartite Attention，并集成 FlashInfer backend。KV cache worker 管理 paged CPU/GPU memory；一个 user 或 item 的全部 prefix tokens 作为逻辑 KV entry，物理上切成与 PagedAttention 兼容的 fixed-size pages，并通过 local DMA、GPUDirect RDMA 等 transfer engine 与 inference worker 交互。Cache meta service 是逻辑集中式进程，追踪 user/item KV entry 在不同 worker/tier 上的索引、状态和热度，负责处理 scheduler 的 read/write/status 查询，并协调实际 KV page 传输。离线初始化阶段，Bat 先为 item cache 分配 pooled memory，其余空间给 user cache；所有 item 描述 token 的 KV cache 可预计算并加载，在线阶段再根据热度后台替换或加入新 item。
        - 实验 / 实现平台：主实验使用浙江大学 4 节点集群，每节点为 Intel Xeon Silver 4214 CPU（2x24 threads，2.20GHz）、200 GB memory、1 张 40 GB A100 GPU（PCIe 3.0 x16）和 100 Gbps network；每节点部署 1 个 inference worker 和 1 个 KV cache worker。扩展性实验使用 16 节点生产集群，每节点 1 张 NVIDIA H20、Intel Xeon Platinum 8469C CPU（双路，每路 48 cores，每 core 2 threads）、500 GB host memory 和 200 Gbps network。模型包括 Qwen2-1.5B、Qwen2-7B、Llama3-1B，KV cache 使用 FP16；Qwen2-1.5B 的 KV cache per token 为 28,672 bytes，Qwen2-7B 为 57,344 bytes，Llama3-1B 为 32,768 bytes。数据集包括 Amazon Games、Beauty、Books 三个公开推荐数据集，以及由真实电商广告 workload 生成的 synthetic Industry dataset；Industry 设置中 user 数 10M、item 数 1M、平均 user token 1500、平均 item token 10。
        - 关键实验设置与指标：系统吞吐用 QPS 衡量，cache hit rate 定义为复用 prefix tokens 数占 prompt 总 token 数的比例；每个请求取 100 个 candidate items 组成 prompt。公开数据集用户历史较短，因此作者扩展 profile token length 使最大 prompt length 接近 8K。总体性能中，Bat 相比 RE 最高达到 58% cache hit rate、2.3x 吞吐提升；相比 UP 最高 1.6x speedup。Accuracy 实验使用 Recall@5/10、MRR@5/10、NDCG@5/10 评估 UP 与 IP 的推荐质量，结果显示多数情况下 IP 与 UP 接近，部分模型/数据集 IP 更好，部分如 Qwen2-1.5B on Books 因 position encoding 修改略有退化；作者用类似 CacheBlend 的 PIC 将 Books 上 Qwen2-1.5B IP 的 Recall@10/MRR@10/NDCG@10 提升到 0.5634/0.1676/0.2576，缩小与 UP 的差距。HRCS 消融中，在 Books + Qwen2-1.5B、150 GB host memory per machine 条件下，Bat 相比 Bat-Replicate 在 10 Gbps 和 100 Gbps 网络下分别提升 10% 与 16% 吞吐；Bat-Hash 虽有更高 hit rate，但 10 Gbps 下约 31% inference latency 来自网络通信，吞吐只有 Bat-Replicate 的 78%。Latency 实验中，在 200 ms P99 SLO 下，Bat 可承载的请求率约比 UP 高 1.47x、比 RE 高 1.57x。扩展性实验显示，在 Industrial-1M 上节点数从 1 扩到 16 时吞吐近线性增长；在 Industrial-X 中 item corpus 从 1M 扩到 100M 时 Bat 仍优于 baselines。

    4、pipeline/kernel 解析：
        - 新 pipeline/kernel 是什么：论文没有提出传统意义上的新 GPU fused kernel 或矩阵乘 kernel；最接近的核心执行流是 “Bat hotness-aware Bipartite Attention serving pipeline”。它把推荐请求的 prefix 类型变成运行时调度对象：同一个 GR 模型可以按 User-as-prefix 或 Item-as-prefix 两种注意力路径执行，并由 scheduler、cache meta service、KV cache worker、inference worker 共同决定和完成 KV cache 读取、prompt 拼接、attention 计算与 cache 更新。底层 attention 计算仍依赖 vLLM/FlashInfer，但 attention mask、position encoding 和 KV cache entry 粒度被改造以支持 item 独立缓存与跨用户复用。
        - 新 pipeline/kernel 的执行流例子：一个 ranking request 从 retrieval stage 到达 Bat scheduler，携带 user ID 和 100 个 candidate item IDs。系统中 user profile、item description 和 system instruction 已预编码为 tokens。Scheduler 批量查询 cache meta service，获取该 user KV 是否存在、候选 item KV 是否存在、user/item 的热度和当前缓存空间状态。若 user profile token 数较短，或该 user 的短期频率不足以替换现有高价值 user cache，scheduler 选择 Item-as-prefix；否则选择 User-as-prefix。
        - 新 pipeline/kernel 的执行流例子：若选择 User-as-prefix，prompt 逻辑顺序为 [U, I1, ..., In, Instr]。Cache meta service 若发现 U 的 KV cache 命中，就协调 KV cache worker 将 U 的 paged KV 通过 DMA/RDMA 传给 inference worker；inference worker 只需对 item 与 instruction suffix 计算 Q/K/V，并让 suffix query attend 到 cached U 与新算出的 suffix KV。计算完成后，item/instruction KV 一般不作为长期可复用 prefix 保存，因为它们依赖当前用户上下文且候选 item 集合动态变化；必要时 scheduler 在后台协调 user KV 写回或更新热度。
        - 新 pipeline/kernel 的执行流例子：若选择 Item-as-prefix，prompt 逻辑顺序为 [I1, ..., In, U, Instr]。每个 item 的 KV cache 在离线阶段已按 item 粒度预计算，热门 item 在所有或多数 KV cache workers 复制，冷 item 被 sharding。Cache meta service 根据 HRCS placement 找到候选 item KV 所在位置；若是 replicated hot items，通常本地读取即可；若是 cold-sharded items，可能跨节点拉取，通信量受 HRCS 的 replication ratio 控制。Inference worker 收到 item prefix KV 后，只计算 user 与 instruction suffix，并让 suffix query attend 到 cached items 与新算出的 suffix KV。由于 attention mask 禁止不同 item 之间互相影响，且 position ID 让每个 item 的起始位置一致，item KV 不绑定某个用户或 prompt 位置，因此可跨用户复用。
        - 新 pipeline/kernel 的执行流例子：HRCS placement 在离线/周期性后台执行。系统先测量 inter-node bandwidth 并换算成 tokens/sec，再用离线 profiling 或回归模型估计在 “m 个候选 item token 作为 prefix、user token 作为 suffix” 下的 prefill time。给定可容忍通信时间比例，算法计算最大远程 KV 通信比例，然后按历史 item access frequency 从高到低扫描 CDF：足以覆盖 1 - max_communication_ratio 的 top hot items 被复制到每个 worker，剩余 long-tail items 均匀分片。在线时若出现突发热点或新增 item，后台重新计算/更新 replicated area 或 sharded area。
        - 新 pipeline/kernel 的执行流例子：Hotness-aware prompt scheduling 维护用户访问滑动窗口频率。Cache meta service 在 user cache 被访问时异步衰减/更新频率统计；由于推荐连续请求通常以秒级到达，异步更新不会明显影响请求延迟。对于新请求，如果 user token 比 item token 少，直接使用 Item-as-prefix 复用 item cache；如果 user token 更长，则比较该 user 的预测短期频率与当前 user cache 中最低频条目，只有当新 user 的单位 cache 价值更高时，才将其排为 User-as-prefix 并占用/替换 user cache。这个 pipeline 的目标不是改变推荐模型的打分数学，而是在不明显损失 Recall/MRR/NDCG 的前提下，把有限 KV cache 容量用在最可能被再次访问的 prefix 上。
