## 62-SCAR- Scheduling Multi-Model AI Workloads on Heterogeneous Multi-Chiplet Module Accelereators.pdf

- 属于芯片设计的实现是什么？实验比较什么？

## 27-CoCoTree: A Computation-Capable Architecture for Collective Communication in Scalable PIM

- 属于芯片设计的实现是什么？实验比较什么？
  - 实现：CoCoTree采用分层chip-level树形组织结构，匹配DIMM的内在层次（bank→chip→rank→DIMM）：(1) chip-level：每个PIM chip内含8个Co-Leaf（嵌入各bank PE）和7个Co-Node，通过片上连线形成完美二叉树。Co-Nodes布置在buffer chip或controller中。(2) rank-level：M个chip（M=8）通过双向SerDes链接（GRS[70]，25Gb/s/pin，reach 80mm）连接到含M-1个Co-Node的rank-level CoCoTree chip。(3) DIMM-level：多个rank通过Ribbon Cable（16Gb/s/pin，reach 500mm）组成更高层tree。这种LEGO-like composable和self-similar hierarchical organization提供可扩展性。芯片设计还考虑dual modularity：functional modularity（类RISC-V模块化ISA，支持B/T/RB/RSU/RSF/RCU/RCF功能按需配置）和structural modularity（按DIMM层级自相似扩展）。Co-Node支持数据宽度动态扩展（如16-bit reduce结果自动扩展为24-bit防止溢出）。Co-CoTree设计考虑了root bottleneck缓解——通信负载在Co-Node间保持平衡，in-network computation在每个reduction level将流量减半。物理实现基于45nm NanGate45工艺评估面积和功耗。
  - 实验比较：vs Baseline UPMEM PIM（无专用inter-PE通信芯片结构）、vs DIMM-Link（专用inter-DIMM点对点链接）、vs PIMnet（intra-channel多层级网络，超过256 DPU仍需host forwarding）。面积/功耗overhead：Co-Leaf仅0.5%面积/0.4%功耗 per bank；Co-Node integer reduction: 0.030mm²/1.36mW；含FP: 0.076mm²/3.78mW；8-PE chip tree总开销0.20mm²/9.76mW。

  - 模拟CoCoTree分层chip结构（chip-level→rank-level→DIMM-level）的集体通信性能。Verilator cycle-accurate仿真Co-Node和Co-Leaf的微架构行为（routing、FU computation、handshake flow control），C++模拟器simulate跨chip SerDes链接带宽和rank-level/DIMM-level tree通信。Yosys+iEDA评估各chip组件的面积和功耗。

## 28-MIMDRAM: An End-to-End Processing-Using-DRAM System for High-Throughput, Energy-Efficient and Programmer-Transparent Multiple-Instruction Multiple-Data Computing

- 属于芯片设计的实现是什么？实验比较什么？
  - 实现：在DRAM芯片层级引入fine-grained DRAM概念用于PUD计算。核心改动：(1) 将DRAM subarray中的mat从统一编址改为可独立寻址，mat isolation transistor切断global wordline使每个mat可单独ACT；(2) row decoder latch每mat存储局部行地址，允许不同mat激活不同行；(3) mat selector共享逻辑根据logical mat range [mat_begin,mat_end]断言对应mat的isolation transistor；(4) inter-mat interconnect（global I/O）通过global row buffer的MUX+bypass path实现跨mat 4-bit数据搬移；(5) intra-mat interconnect（local I/O）复用HFF + column select logic实现同mat内列间数据搬移；(6) chip select logic + mat identifier logic用4 comparators + 2 AND gates + 2 MUX + 3-bit chip ID register实现14-bit logical mat range到物理mat的映射；(7) mat queue缓冲预发mat信息，通过ACT-enqueue/PRE-enqueue/ACT-dequeue机制在不增加DRAM C/A pin的前提下传递mat地址；(8) pim_malloc OS allocator保证同mat label的memory objects物理上分配在同一mat范围内的连续物理地址。

  - 实现：HeterRAG的芯片级设计围绕PIM-enabled DIMM和PIM-enabled HBM两种存储芯片展开：(1) **PIM-enabled DIMM芯片**——在标准DDR4 DIMM的buffer chip上集成两类处理模块：DIMM-level Processing Module (DPM) 和Rank-level Processing Module (RPM)。每rank配备一个RPM（共2个RPM/DIMM），每个RPM面积3.368mm² (22nm)；DPM面积0.842mm²。单DIMM总PIM硬件面积开销7.817mm²，相对典型buffer chip (>100mm²) 可接受。RPM内部设计：PIM-Inst Queue (128-entry) → Inst Decoder → Vertex Cache (128KB, LRU, 分local/global) + DDR-C/A Generator (生成DDR ACT/RD/PRE命令) + Distance Computation Unit (32 FP32 Mult + 32 FP32 Adder)。DPM内部设计：PIM-Inst Queue (256-entry) → PIM-Inst Decoder → Rank ID路由 → 每rank Distance Buffer (32KB) → MUX轮询回传。DIMM采用统一内存组织存储vertex neighbors和vector data。(2) **PIM-enabled HBM芯片**——8-Hi HBM2 stack，在DRAM die上集成Bank-level Processing Module (BPM)。每两个bank共享一个BPM，BPM含2个向量内积计算单元 (32 FP16 Mult + 32 FP16 Adder) 和8Kb buffer。HBM面积开销：所有BPM总计6.016mm² on DRAM die，占53.15mm² HBM2 die的约11.31%。HBM采用统一内存组织存储weights、KV cache和其他数据。BPM输入来自bank-level row buffer (weight) 和channel-level global buffer (activation)。(3) **芯片互联**：AccelDIMM和AccelHBM通过CXL switch互联（类似CXL Type 2 devices），支持PCIe 5.0 ×8 (≈32GB/s)。CXL支持cxl.io/cxl.cache/cxl.mem三种协议，CXL switch网络支持最多4K节点。(4) **芯片扩展性**：AccelDIMM设备数量4-32，检索近乎超线性加速（因数据并行、无inter-device通信）；AccelHBM设备数量4-10支持LLaMA2-70B，2.5×设备增加→1.78× throughput提升。
  - 实验比较：(a) HBM PIM面积开销：BPM 6.016mm² vs HBM2 die 53.15mm² (11.31%)；(b) DIMM PIM面积开销：RPM+DPM 7.817mm² vs buffer chip >100mm² (可接受)；(c) HeterRAG整体 vs CPU-GPU/NaiveHBM/OnlyDIMM的throughput/latency/energy；(d) AccelDIMM和AccelHBM独立扩展性测试；(e) 与MemANNS/DRIM-ANN (UPMEM) 的ANNS加速对比、与NeuPIMs/AttAcc的LLM加速对比。

## 61-Duplex- A Device for Large Language Models with Mixture of Experts, Grouped Query Attention, and Continuous Batching.pdf

- 属于芯片设计的实现是什么？实验比较什么？
  - 实现：Duplex在HBM3 DRAM die和logic die层面引入多项芯片级微架构修改：(1) **Bank Bundle架构**：将HBM3的16 banks per rank+pseudo channel分成upper/lower两个bank bundle，每bundle 8 banks独立操作。在bank I/O和bank group I/O之间放置switches，分离xPU path和Logic-PIM path。Logic-PIM同时发送相同C/A到目标bank bundle的8 banks→同时读取512 bits (from 2 banks per bank group × 4 bank groups)→tCCD_L间隔定义数据传输节奏→通过dedicated TSVs传输到logic die。xPU path通过简单switch分离，可在Logic-PIM访问数据时同时从其他bank bundles读取。(2) **TSV布局优化**：Logic-PIM的dedicated TSVs放置在power TSV area而非data TSV area。理由是：若放置在data TSV area会增加bank group到TSV的datapath长度→更多energy消耗。放置在power TSV area缩短Logic-PIM-path I/O长度。TSV pitch从50um缩至22um [49]→4x TSV数量仅增加9% area overhead。每channel的TSV数量保守按4x HBM3 per-channel requirement扩展。(3) **Logic Die Processing Units**：利用logic process优势在logic die上放置处理单元。每Logic-PIM stack: 32 GEMM modules (512 FP16 MACs each @650MHz, 8KB buffer each = 3.02mm²)、2×1MB buffers (2.26mm²)、1 softmax unit (comparator tree+adders+exp units+adder tree+dividers+multipliers+128KB buffers = 1.64mm²)。总面积17.80mm²，占121mm² HBM3 logic die [41]的14.71%，低于prior Bank-PIM的20-27% [28][29]。(4) **Independent Active Paths**：Logic-PIM和xPU通过独立的active paths同时操作——Logic-PIM path从bank bundle读数据通过Logic-PIM TSVs到logic die处理；xPU path通过conventional HBM controller和data TSVs读写数据到xPU。(5) **DRAM Bank结构最小修改**：避免修改DRAM bank layout（增加bank prefetch size和I/O datapath width会带来77% bank size increase）→改为增加simultaneously operating banks数量。(6) **Logic-PIM Controller**：仅通过control pins连接Logic-PIM（无数据pin），发送compute request（weight/input起始地址、GEMM维度）到operation controller，最小化pin overhead。(7) **Memory Space-to-Bank Bundle映射**：4个memory spaces对应bank bundle index，expert FFNs round-robin分配以支持co-processing时的无冲突bank bundle访问。
  - 实验比较：(1) EDAP (Energy-Delay-Area Product)：Logic-PIM vs Bank-PIM vs BankGroup-PIM 在Op/B 1-32的FP16 GEMM上的对比（weight matrix 16384×4096）；(2) 面积开销：Logic-PIM 17.80mm² (14.71% of 121mm² logic die) vs Bank-PIM/BankGroup-PIM 20-27% DRAM die area；(3) 在各种LLM和batch size配置下的throughput/latency/energy对比；(4) Split Prefill/Decoding node方案性能对比。

## 8-SambaNova_SN40L_Scaling_the_AI_Memory_Wall_with_Dataflow_and_Composition_of_Experts.pdf

- 属于芯片设计的实现是什么？实验比较什么？
  - 实现：SN40L RDU采用多项芯片级设计创新：
    - **2.5D CoWoS Chiplet封装**：TSMC 5nm工艺，单socket含2个Reconfigurable Dataflow Dies (RDDs)，通过Die-to-Die (D2D) Interface直连流式传输数据，avoiding off-chip memory hops。
    - **三阶存储系统（Three-Tier Memory）**：第一阶520MB on-chip distributed SRAM（分布在1040个PMU中，提供hundreds of TBps aggregate bandwidth）；第二阶64GB co-packaged HBM（1.8TB/s per socket）；第三阶最高1.5TB off-package DDR DRAM（200GB/s，可插拔DIMMs）。DDR→HBM copy速度>1TB/s（单SN40L Node aggregation）。
    - **On-chip Interconnect (RDN)**：2D mesh of non-blocking switches，分为vector/scalar/control三个物理fabric，支持credit-based flow control、multicast、many-to-one reordering。
    - **Peer-to-Peer (P2P) Inter-RDU Network**：socket间直接流式数据传输协议，支持构建collective communication primitives (AllReduce等)，避免数据经过HBM hops。
  - 实验比较：(1) 三阶存储系统对CoE模型切换延迟的影响——SN40L Node DDR→HBM copy >1TB/s vs DGX A100 32GB/s (31× faster) vs DGX H100 64GB/s (16× faster)；(2) 单SN40L Node (8 sockets) 可容纳最多850个7B experts在TP8延迟下 vs DGX A100/H100在>150 experts时OOM；(3) 机器footprint reduction——SN40L Node减少达19× nodes vs DGX for CoE deployment；(4) 总存储容量——SN40L per-socket 64GB HBM + 1.5TB DDR (aggregate 2.5× higher than GH200 [64])。

- 模拟器名，模拟器链接（web search），或论文修改的模拟器。
  - 无模拟器，真实芯片评估。SN40L RDU为SambaNova Systems商业流片产品（TSMC 5nm，CoWoS封装，<650mm² die size per RDD）。
  - 芯片参数已公开于论文Table II：638 BF16 TFLOPS、520MB SRAM、64GB HBM、1.5TB DDR、1040 PCUs/PMUs、<2GHz clock。

  - 不适用（真实芯片）。论文量化了三阶存储系统的关键性能指标：(1) DDR→HBM带宽：>1TB/s aggregate (8 sockets)；(2) HBM带宽：1.8TB/s per socket，实测>85%利用率 vs GPU <50%；(3) 模型切换延迟：DDR→HBM copy 7B model weights的时间（experts fit in DDR, copy to HBM for execution）。

- 开源情况。基于开源文档和论文，使用例子解释模拟器如何使用？作用是什么？
  - SN40L RDU为商业闭源产品。芯片设计原理在MICRO 2024论文中公开描述，但无开源RTL或仿真器。
  - **芯片级CoE执行流程（以Samba-CoE 150 experts on 8-socket SN40L Node为例）**：
    1. **部署配置**：所有150个Llama2-7B expert weights存储在DDR (1.5TB capacity per socket, >150 × ~13GB feasible)，Router weights常驻HBM。专家模型按TP=8分布在8个RDU socket上。
    2. **请求到达**：host x86 CPU接收prompt → CoE runtime在DDR中定位所有expert weights → 预分配HBM region用于当前expert(s)。
    3. **Router Execution**：Router在HBM中执行（BS=8, TP=8）→ 确定每个prompt所需expert(s) → 多个prompt可能需要不同experts。
    4. **Expert Swapping**：CoE runtime检查目标expert是否已在HBM中（LRU cache hit）→ miss时，AGCU发起DDR→HBM DMA transfer（>1TB/s aggregate across 8 sockets）→ 7B expert weights (<13GB) 在~10ms内完成复制。
    5. **Expert Execution**：Expert weights从HBM stream到on-chip SRAM (PMUs) → PCUs执行fused decoder layer kernel（>>85% HBM bandwidth utilization via streaming dataflow）→ autoregressive decoding loop复用HBM中的weights（temporal locality）。
    6. **Multi-expert并发**：HBM可同时缓存多个LRU experts → 若下一请求的expert已在HBM中则立即执行，无switching overhead。
