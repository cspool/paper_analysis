- 属于编译框架的实现是什么？实验比较什么？
  - 实现：Concerto是一个基于PyTorch 2.0编译器栈的自动通信优化与调度编译器框架。核心设计：(1) 将通信优化抽象为资源受限项目调度问题（RCPSP），使用OR-Tools ILP求解器获得近最优拓扑排序。(2) 引入auto-decomposition自动识别critical（同步）通信并分解其上下文计算，创造overlap机会。(3) Communication fusion合并可互换的同类型通信算子。(4) Odd-even方法将计算图分块迭代求解，将时间复杂度降为多项式O(kt_b·n/b)。系统工作流：用户PyTorch函数 → torch.fx trace → fx Graph → 根据parallel_method转换为ConcertoIR（带SPMDSpec的ATen IR扩展）→ Auto-Decomposition pass（BFS探索分解上下文）→ Scheduling pass（RCPSP ILP求解 + Fusion融合 + Odd-even调优）→ 优化后的拓扑序列 → 轻量运行时（计算→默认CUDA Stream，通信→专用CUDA Stream，end-of-communication marker保证同步）。

    1. **输入**：用户提供PyTorch train_step函数 + 指定parallel_method="ptd"。
    2. **Graph Capture**：Concerto通过torch.fx trace用户函数，生成fx Graph（含前向/反向/优化器步骤的计算与通信算子）。
    3. **ConcertoIR转换**：根据parallel_method将fx Graph转换为ConcertoIR（每个算子标注SPMDSpec信息，来自EasyDist的算子数据库）。通信算子标注类型（all-reduce/all-gather/reduce-scatter）和参数。
    4. **Profiling**：对每个算子以operator name + input shape为key执行profiling，获取执行时间T_i。结果持久化缓存，相同算子跳过profiling。耗时通常几十秒。
    5. **Auto-Decomposition pass**：
       (a) 遍历所有通信算子，按前驱/后继/独立节点分类，判定是否为critical communication（独立计算节点可提供的overlap不足）。
       (b) 对每个critical comm，沿各tensor轴做BFS探索：节点N从comm的前驱/后继方向出队 → SPMDPropagate(N, predecessor_axis)查询SPMDSpec确定当前轴是否可分解 → 可分解则加入Decomposition Context并累计时间t_d → 当t_d≥T_comm或无可探索节点时停止。例如GPT Feed-Forward: LayerNorm→all-gather→MatMul1→GeLU→MatMul2，若沿batch/sequence轴分解，Context={LayerNorm, MatMul1, GeLU, MatMul2}；若沿hidden轴，LayerNorm不可分解，Context仅{MatMul1}。
       (c) 计算每种分解策略的成本（考虑分解度N、重叠效率α=1.2、前驱/后继计算时间），选择最低成本策略。若多critical comm的分解上下文有交集，通过ILP联合优化选择互不冲突的策略组合。
    6. **Scheduling pass**（RCPSP求解）：
       (a) 资源定义：R={computation:1, communication:1}，即任意时刻最多1个计算+1个通信并发。
       (b) 变量：每个任务i的区间[Si, Ei]，Ei=Si+Ti。约束：依赖保留（前驱完成后才能开始）、资源限制（每时刻resource usage≤1）。
       (c) ILP求解（OR-Tools）：目标函数 min(全局makespan)。Odd-even方法：将图按block_size分块，odd轮按原分块求解，even轮偏移half block_size再求解，迭代多轮逐渐逼近全局最优。block_size和轮数可配置。
       (d) Communication fusion（Algorithm 1）：遍历拓扑序，将可互换（交换执行序不破坏依赖）且可融合（同类型同参数）的通信算子合并为单个fused通信算子。
       (e) Decoding：按求解的start time排序任务；将通信任务提前到其执行的compute之前（因通信需少量SM launch kernel，避免被compute占满SM而延迟）。
    7. **运行时执行**：轻量Runtime遍历优化后的拓扑序列 → 计算算子dispatch到default CUDA Stream → 通信算子dispatch到专用CUDA Stream → end-of-communication marker在default stream需要通信结果时确保已完成。支持torch._custom_ops注册自定义kernel（如flash attention、Megatron-LM算子）。
    8. **输出**：编译后的优化函数，可直接用于训练循环。单步step time缩短。

    2. **Offline Profiling与参数推导**（Alg. 2 lines 2-8）：(a) 分析computation op的reduce axes和codebook switch axes（Attention K cache: reduce=all axes, codebook switch=H,C; V cache: reduce=T, codebook switch=H,C）；(b) Profile codebook entry access frequency → 计算nreg（>µ+3σ的hot entries数量）和nshared（基于Resource Budget preserved for occupancy的slack size）；(c) 计算split_factor使Traffic_Reduce = Traffic_Codebook平衡；(d) 计算nshuffle = layoutsrc/layoutdst（CQ-4 vector_size=2, mma要求layout=2, nshuffle=1 ≤ threshold=5 → 启用register-level fusion）；(e) 确定Thread_Mapping（mini-warp mapping for shuffle）
    3. **Codebook Cache初始化**：按access frequency降序reorder codebook entries → entries[0:nreg]→registers, entries[nreg:nshared]→shared memory, entries[nshared:]→global memory
    4. **Kernel生成**：CUDA template实例化。(a) Parallel_For沿codebook switch axes（H, C）按split_factor切分 → 每个thread block仅需load一个codebook；(b) Load codebook到codebook cache；(c) Dequantization: Access(CBcached, boundary, CB, index) → 按index范围查register/shared/global memory中的codebook entry；(d) Hierarchical Fusion: 若nshuffle≤5则Reg_Fusion(data, nshuffle)使用shfl_xor API在mini-warp内交换数据使layout对齐mma需求；否则Shared_Fusion通过shared memory重排；(e) 沿temporal iteration axes（非codebook switch axes）执行原始computation（mma.sync等）；(f) Reduce(partial, Reduce∩codebook.switch axes)做global reduction
    5. **输出**：一个optimized CUDA kernel（.cu文件或JIT编译的PTX），可直接替换原始attention kernel在推理pipeline中使用。

## 26-SoMa: Identifying, Exploring, and Understanding the DRAM Communication Scheduling Space for DNN Accelerators

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：SoMa是一个端到端的DNN调度编译器框架（end-to-end scheduling framework），接受硬件配置、框架配置和DNN模型描述文件作为输入，输出指令（instructions）、能耗/延迟报告和详细调度方案。核心架构包括三个组件：(1) **Buffer Allocator**：迭代式在最外层分配two-stage之间竞争的buffer预算；(2) **LFA Exploration Stage**（第一阶段）：基于模拟退火（SA）搜索Layer-Fusion相关的四个属性（Computing Order、Fine-grained Layer-fusion Cut、Tiling Number、DRAM Cut），DLSA使用经典double-buffer策略；(3) **DLSA Exploration Stage**（第二阶段）：固定LFA，基于SA搜索DRAM Load/Store相关属性（DRAM Tensor Order、Living Duration）。解析后的调度方案经IR Generator转换为中间表示，再经Instruction Generator生成实际加速器指令（load/store/compute）。优化目标为 Energy^n × Delay^m，其中n和m可调节以优先能耗或性能。
  - 实验比较：与SOTA Cocco调度框架[49]全面对比，评估性能（Computing Resources Utilization）和能耗（Normalized Energy Cost，包括Core Array Energy和DRAM Energy），并对比两阶段的逐步优化效果（Ours_1为第一阶段后、Ours_2为第二阶段后）。还展示了DSE（Design Space Exploration）下不同buffer size和DRAM带宽的延迟对比，以及CNN（ResNet-50/101、Inception-ResNet-v1、RandWire）和LLM（GPT-2-Small/XL Prefill和Decode）在batch size 1/4/16/64下的表现。

- 硬件平台是什么，配置是什么。
  - Edge平台：16 TOPS算力（参考Qualcomm Snapdragon 8 Gen 3 15 TOPS和Apple A15 15.8 TOPS），8MB Global Buffer，16GB/s DRAM Bandwidth。
  - Cloud平台：128 TOPS算力（参考NVIDIA Orin 138 TOPS和TPU V4i 136 TOPS），32MB Global Buffer，128GB/s DRAM Bandwidth。
  - 加速器模板：DRAM → Global Buffer (GBUF，所有core共享) → 多个Core（每个含PE Array（GEMM/Conv）、Vector Unit（element-wise/pooling/LayerNorm等）、Local Buffers WL0/AL0/OL0）。
  - 工艺：TSMC 12nm，频率1GHz。各操作单位能耗来自实际RTL代码综合与仿真。

- 开源编译框架是什么。修改了什么。
  - 基线Cocco [49]是SOTA layer-fusion调度框架，探索layer-fusion空间但采用主流tiling和prefetch-and-delayed-storing heuristic策略。在SoMa的Tensor-centric Notation中，Cocco仅可改变Computing Order和DRAM Cut，FLC Set = DRAM Cut Set，其余四个属性由heuristic确定。
  - SoMa的核心创新：(1) 提出Tensor-centric Notation以6个属性（分LFA和DLSA两类）全面编码DRAM Communication Scheduling Space；(2) 开发两阶段SA探索引擎（LFA+DLSA）和Buffer Allocator迭代分配buffer预算；(3) 精确的Evaluator模块从local到global评估延迟和能耗。

  - 已开源：https://github.com/SET-Scheduling-Project/SoMa-HPCA2025
  - 输入→输出全过程：(1) 输入：硬件配置（核心数/组织、DRAM带宽、buffer大小等）、框架配置（优化目标Energy^n×Delay^m、搜索超参数）、DNN模型描述文件（PyTorch等高层框架生成）；(2) Model Parser将模型解析为内部表示；(3) Buffer Allocator控制最外层迭代：首轮完成完整two-stage搜索，记录第一阶段最大buffer用量（Buffer_max）和最优cost（Cost_best）；后续迭代将第一阶段buffer上限缩减a%（论文用10%），记录Cost_temp；若Cost_temp优于Cost_best则更新最优方案；连续两次不超Cost_best时停止；(4) LFA Exploration Stage：初始方案为每层独立LG/FLG、Tiling Number=最小粒度、DLSA=double-buffer；SA Operators包括Change Computing Order（随机选layer改到另一合法位置）、Change Tiling Number（随机选FLG乘/除以2）、Add/Delete FLC、Add/Delete DRAM Cut；Cooling schedule: Tn = T0 × (1+αn/N)^(1-n/N)，总迭代N=100×layers；(5) DLSA Exploration Stage：固定LFA最优编码，SA搜索DRAM Tensor Order（随机选tensor改到另一合法位置）和Living Duration（随机选tensor改Start/End）；总迭代N=1000×DRAM_tensors，其中选择某tensor的概率正比于其大小；(6) Evaluator：对每个compute tile调用Core Array Scheduler & Evaluator（经典单层dataflow方法[32][42]）评估tile内能耗和延迟；每个DRAM tensor按数据量×单位能耗评估DRAM能耗；总延迟按三种依赖条件（前序DRAM tensor完成、Start≤当前tile、ofmaps需等生成tile完成）推导；(7) IR Generator将最优调度方案转为中间表示；(8) Instruction Generator生成实际加速器指令（load/store/compute），输出能耗/延迟报告。

## 22-FEATHER_A_Reconfigurable_Accelerator_with_Data_Reordering_Support_for_Low-Cost_On-Chip_Dataflow_Switching

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：LayoutLoop，一个基于Timeloop [41]增强的dataflow-layout协同搜索框架（dataflow-layout co-exploration）。核心增强包括：
    (1) **Physical Storage Modeling**：将on-chip buffer建模为(num_line × line_size) 2D逻辑数组，引入"conflict_depth"参数表示每个SRAM bank的line数量。将不同物理bank组织（2D/3D、多种分组方式）抽象为统一的逻辑2D array，无需处理具体物理实现。Bank port约束：每个bank最多2个read/write ports（TSMC 28nm标准）。
    (2) **Bank Conflict Assessment**：在dataflow evaluation中检查每个cycle的memory access是否跨越超过NP条lines（对应NP ports的bank）。若NL条lines被访问而bank仅有NP ports，引入max(NP/NL, 1) slowdown因子。此机制消除了传统Timeloop中"ideal bandwidth"假设导致的理论-实践性能差距（本文量化该差距可达128×）。
    (3) **Layout-aware Mapper**：修改Timeloop mapper使dataflow search同时考虑layout。Layout表示为"(Inter-line dimension order) (Intra-line dimension order interleaved with sizes)"，如`HWC_C4W8`（inter-line order: H→W→C，intra-line: 4 C elements × 8 W elements per row）、`CHW_W4`（row-major风格）。
    (4) **Dataflow-Layout Co-Search**：per-layer独立搜索。对每层：(a) exhaustive search layout space（基于已知加速器常用layout patterns，如Conv: HWC_C32, HWC_W32, HWC_H32, HWC_C4W8, HWC_C4H8, HWC_W4H8, HWC_C4W4H2；GEMM: MK_K32, MK_M32, MK_M4K8）；(b) pruned random search（exhaustive + random pruning）在dataflow space搜索，victory condition基于EDP最佳；(c) 每层选择EDP最优的(dataflow, layout) pair。
  - 实验比较：LayoutLoop用于分析性评估，比较FEATHER vs NVDLA-like（fixed dataflow, HWC_C32）、Eyeriss-like（row-stationary, HWC_C32）、SIGMA-like（多scenario: fixed layout/no reorder、off-chip reorder with 128 GB/s HBM、on-chip line rotation/Medusa、on-chip transpose/MTIA-like、on-chip transpose+row-reorder/TPU-like）。在所有scenario中FEATHER以最优EDP胜出。FPGA部署时简化为两个layout（DepthWise Conv和typical Conv各一最优layout），per-layer在两者间切换。

- 硬件平台是什么，配置是什么。
  - LayoutLoop为host端软件工具，运行在multi-core CPU (≥32 GB memory) 的Linux系统上。LayoutLoop评估的FEATHER accelerator配置：16×16 PE array，int8 datatype，256 PEs total（Layoutloop分析配置），on-chip buffer bandwidth 8000 bits/cycle，TSMC-28nm 500 MHz。实际FPGA部署配置：1296 PEs，720 bits/cycle bandwidth，int8，100 MHz on ZCU104。
  - 搜索runtime：LayoutLoop DSE (Design Space Exploration) 约需24小时（multi-threaded，含ResNet-50 + Mob-V3 + BERT全部layers）。

- 开源编译框架是什么。修改了什么。
  - 基于Timeloop（https://github.com/NVlabs/timeloop），Timeloop为state-of-the-art DNN accelerator dataflow cost modeling and search framework（使用C++，scons build system）。
  - 修改：(1) 新增layout specification representation和input parsing（支持inter-line/intra-line dimension order + sizes）；(2) 重写on-chip storage model——从抽象的"bandwidth-only"模型改为physical bank organization模型（num_line × line_size × conflict_depth）；(3) 新增bank conflict detection和slowdown calculation逻辑（在access pattern analysis中插入bank-port contention检查）；(4) 修改mapper search loop——在evaluate每个dataflow candidate时同步检查layout compatibility并计算slowdown；(5) 修改energy model——将slowdown导致的stall cycle计入energy（stall cycle仍消耗static energy）。
  - 提供pre-built Docker image（https://tinyurl.com/layoutloop）包含所有依赖（libconfig++-dev, libboost-*, libyaml-cpp-dev, cmake, scons等）。

  - 开源：https://github.com/maeri-project/FEATHER（MIT license），LayoutLoop代码位于`FEATHER/LayoutLoop/layoutloop/`，配置文件位于`FEATHER/LayoutLoop/configurations/`。
  - 使用流程（以ResNet-50某conv layer为例）：
    (1) **输入文件**：

## 21-Tandem Processor: Grappling with Emerging Operators in Neural Networks

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：Tandem Processor配套编译器，输入ONNX模型+架构配置，输出可执行二进制指令。编译流程：(1) 将ONNX节点映射到预定义操作模板；(2) 复杂算子（Softmax, GeLU, Sqrt等）翻译为整数基本原语（integer-based counterpart，参考gemmlowp和I-BERT方法）；(3) 参数替换（模板参数替换为实际tensor维度）；(4) tiling优化——选择uniform tiling方案使GEMM和非GEMM层对齐，不对GEMM的reduction维度做tiling（否则产生partial result导致Tandem Processor stall），为Depth-wise Conv等算子选择足够大的tile size（如5×5 kernel需包含完整patch避免stall）；(5) 依赖松弛——利用loop fission消除指令间依赖，利用loop interchange松弛长依赖链（如MaxPool）；(6) 指令生成——将模板语句lowering为Tandem Processor ISA（32-bit固定长度指令字）：LOAD/STORE语句→TILE_LD_ST配置指令（BASE_ADDR + LOOP_ITER/STRIDE），compute操作→LOOP指令（SET_ITER/SET_NUM_INST/SET_INDEX）+ ALU指令，涉及Output BUF的compute自动插入synchronization指令。
  - 实验比较：编译框架本身未被作为独立ablation评估。其效果体现在Tandem Processor端到端性能结果中（vs TPU+VPU的2.6×加速中部分归因于compiler-managed tiling和specialized loop execution）。

- 硬件平台是什么，配置是什么。
  - 编译器为host端软件工具，运行在host CPU上。生成的指令在NPU-Tandem硬件上执行：32×32 systolic array GEMM unit + 32-lane Tandem Processor（详见硬件架构条目）。

- 开源编译框架是什么。修改了什么。
  - 自研编译器，作为开源GeneSys项目（https://actlab-genesys.github.io/）的一部分。不是修改现有编译框架（如TVM/XLA），而是全新设计，针对Tandem Processor的scratchpad-based内存语义和Iterator Table-based寻址模式，将ONNX图映射为定制ISA指令序列。
  - 编译器基于ONNX graph representation提取所有算子，以预定义操作模板（operation templates）描述各算子的tile-level伪代码（嵌套循环+LOAD/STORE/ALU语句），编译优化（tiling策略、loop fission/interchange）后直接lowering为Tandem Processor指令。

- 开源情况。基于开源文档和论文，使用例子解释编译框架如何使用？作用是什么？至少具体到编译框架输入到输出的全过程。
  - 开源：GeneSys项目（https://actlab-genesys.github.io/）。
  - 编译流程示例（以BERT中的Add层为例，两个[M,N]张量相加）：
    - **输入**：ONNX IR中Add节点的tensor参数（a[M,N], b[M,N]→c[M,N]）+ 架构配置（SIMD lanes=32, Interim BUF=128KB, tile size M1×N1）
    - **模板映射**：选择element-wise add模板，伪代码：`for m(M/M1) { for n(N/N1) { a1=LOAD(a[n,m],[N1,M1]); b1=LOAD(b[n,m],[N1,M1]); for m1(M1) { for n1(N1) { c1=ADD(a1,b1); } } STORE(c[n,m],c1); } }`
    - **参数替换**：M→实际值，N→实际值，M1/N1→根据128KB scratchpad容量和32-lane SIMD约束计算的最优tile尺寸
    - **指令lowering**：
      - LD_CONFIG_TILE_LOOP_ITER 0, InterimBUF1, 24 → 配置tile循环迭代数
      - LD_CONFIG_TILE_LOOP_STRIDE LSB, InterimBUF1, 24, 1024 → 配置off-chip memory stride
      - LD_START 0, InterimBUF1, 31, 128 → 触发Data Access Engine加载tile
      - SET_ITER 0, 0, 14 → 配置外层循环
      - SET_INDEX InterimBUF2, 1, InterimBUF1, 0, InterimBUF2, 0 → 配置Iterator Table索引
      - SET_ITER 1, 0, 32 → 配置内层循环
      - SET_INST 1, 0, 1 → 配置循环体指令数
      - ADD InterimBUF2, 0, InterimBUF1, 0, InterimBUF2, 0 → 核心ALU指令（32-bit编码：4-bit ALU opcode + 4-bit func + 3×8-bit(dst ns+iter idx, src1 ns+iter idx, src2 ns+iter idx)）
      - ST_CONFIG_TILE_LOOP_ITER/STRIDE → ST_START → 触发Data Access Engine写回off-chip
    - **输出**：32-bit指令二进制序列，可直接加载到Tandem Processor的Instruction Buffer执行。

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：TAIDL（Tensor Accelerator ISA Definition Language），一个面向张量加速器的指令规格定义语言（DSL），基于Python库实现。TAIDL将加速器ISA语义用XLA-HLO张量算子描述，并自动生成ISA特定的测试预言（TAIDL-TO：ISA-specific Test Oracle）。TAIDL-TO的核心是将ISA定义中的指令序列通过变换算法转化为XLA-HLO计算图，再经XLA编译器编译为可在CPU/GPU上并行执行的可执行文件，用作快速功能仿真器。
  - TAIDL-TO生成流程（Figure 14-16）：(1) 架构师用TAIDL定义ISA（数据模型+指令语义）；(2) 自动生成ISA特定的Python测试预言库（TAIDL-TO库）；(3) 内核程序员用该库编写内核函数（Python decorator @kernel）；(4) 调用kernel.compile()——变换算法将指令序列转化为XLA-HLO IR（控制寄存器常量折叠、REPEAT展开、IF分支选择、tensor buffer读写转为slice/dynamic_update_slice、hlo_op保持不变）；(5) XLA编译器编译为序列化可执行文件(.pb)；(6) kernel.run(input)加载可执行文件+NumPy输入数据，在CPU或GPU上仿真。
  - 论文实例化了3种加速器的TAIDL定义：(a) Google TPUv1——256×256 systolic array，权重静止数据流，统一缓冲区、累加器、FIFO权重的数据模型；(b) Intel AMX——8个tile寄存器(16×64×i8)，AVX-512 zmm寄存器(16×f32)，指令如tdpbusd（混合精度矩阵乘加）；(c) Gemmini——可重配置数据流(输出静止/权重静止)的systolic array加速器，支持DIM参数可配置。
  - 实验比较：(a) TAIDL-TO vs Gemmini Spike——tiled matrix multiplication kernel，DIM=16/64/256/1024，kernel size从1KB到3MB变化；(b) TAIDL-TO vs Intel SDE——5个oneDNN benchmark kernels（cnn_inf_amx, rnn_inf_amx, sgemm_avx, mem_format_avx, cnn_inf_mix），#BLOCKS从4到256；(c) TAIDL-TO集成进Exo编译器测试基础设施——6个Exo编译的复杂Gemmini kernel；(d) 端到端I-BERT模型仿真——12 encoder layers，768 embedding size，sequence length=512，Gemmini ISA (DIM=256)。

- 硬件平台是什么，配置是什么。
  - GPU服务器：64核 Intel Xeon Platinum 8358 CPU + NVIDIA A100 GPU（用于TAIDL-TO评估）
  - TAIDL-TO有两个编译版本：XLA CPU后端（TAIDL-TO CPU）和XLA GPU后端（TAIDL-TO GPU）
  - Intel SDE评估依赖：Intel Sapphire Rapids机器（Intel Xeon Gold 5415+ CPU），仅支持x86_64
  - Gemmini Spike评估依赖：RISC-V ISA模拟器Spike扩展，支持amd64/x86_64

  - 开源编译框架：TAIDL本身是一个新的DSL（基于Python库），不是修改现有编译框架。TAIDL-TO生成过程依赖XLA编译器作为后端。XLA-HLO（StableHLO）算子集合被用作TAIDL指令语义的描述语言。

## 15-Data_Motion_Acceleration_Chaining_Cross-Domain_Multi_Accelerators

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：DRX Compiler，将高级数据重构kernel编译为DRX ISA指令。编译器接受两个输入：(1) 数据重构kernel的高级表示（类似OpenCL kernel描述），(2) DRX硬件架构配置文件（定义RE lane数、scratchpad大小等参数）。编译流程：首先将kernel映射到中间表示（IR），然后基于硬件配置和多维数组维度信息优化tiling并放宽依赖关系，最后从优化后的IR生成DRX ISA指令。DRX ISA包含四种指令类型：Loop（2-bit类型，配置Instruction Repeater遍历多维数组的Base/Stride/Iteration）、Compute（指定Dest/Src1/Src2的scratchpad地址和操作类型）、Off-chip Memory（配置Off-chip Data Access Engine的Base/Tile Control/Req Size）、Synchronization（标记instruction group start/done）。
  - 编译器的关键设计：(1) 利用数据重构workload的已知shape（known-shape）、预定位（pre-located）多维数组特性，用硬件loop替代分支指令；(2) 将kernel的多维数组跨RE lane分区（partition），消除pack/unpack指令开销；(3) 使用软件管理的on-chip scratchpad替代vector register file和cache hierarchy，编译器显式编排数据流；(4) 计算指令的scratchpad地址由Strided Scratchpad Address Calculator根据Instruction Repeater遍历多维数组的配置自动生成，大幅减少地址计算开销。
  - 实验比较：论文未单独对DRX compiler做ablation study，compiler的正确性体现在5个benchmark的不同数据重构kernel（涉及不同维度和操作类型）均能正确编译并在DRX硬件上执行，其效果体现在DMX vs Multi-Axl baseline的端到端性能对比中。

## 18-SmartMem- Layout Transformation Elimination and Adaptation for Efficient DNN Execution on Mobile

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：SmartMem是在DNNFusion [51]之上构建的DNN编译优化框架，核心包含三大组件：(1) Operator Classification——将DNN operator按两个维度分为四类：输入布局依赖+输出布局可变(ILD-Variable)、输入布局依赖+输出布局固定(ILD-Fixed)、输入布局独立+输出布局可变(ILI-Variable)、输入布局独立+输出布局固定(ILI-Fixed)；(2) Layout Transformation Elimination (LTE)——基于operator分类分析计算图中producer-consumer边，按Table 5决策矩阵决定每对operator的优化动作：Keep both（红）、Try fuse（蓝）、Eliminate 2nd/Eliminate 1st（黄）、Eliminate both（绿）。消除通过Index Comprehension实现——将Reshape/Transpose等layout transformation替换为索引计算（identity/split/merge三类索引依赖关系），并用Strength Reduction简化取模和除法运算（如i%Ca%Cb→i%Cb when Ca%Cb≡0）；(3) Reduction Dimension-Based Layout Selection——对ILD-Variable类型的保留operator，按consumer的reduction dimension（数据聚合发生的维度）决定producer应生成的数据layout。多consumer场景下合并前k个reduction dimension要求（k=2对应2.5D memory两维连续访问），若超过k则维护多份不同layout的数据副本。
  - 实验比较：与5个SOTA框架对比——MNN [32]、NCNN [50]、TFLite [1]、TVM [10]、DNNFusion [51]。指标：operator数量（fusion rate）、端到端延迟(GMACS)、memory access count、cache miss count。消融实验：DNNF（baseline）→ +LTE → +Layout Selecting → +Other Opt（texture memory mapping + auto-tuning）逐步叠加评估每项优化的贡献。

    5. **2.5D Texture Memory Mapping**：将选定layout映射到移动GPU的2.5D纹理内存——对含两个reduction dimension的tensor（如L0的D1和D3），将D1按4元素partition（匹配2.5D的0.5D向量长度），每个partition内沿D3方向连续存储k×|D3|元素块，使consumer沿D1或D3做SIMD load/reduction均可高效访问。
    6. **Auto-tuning**：Genetic Algorithm搜索最优GPU execution配置（block dimensions、unrolling factors、tiling shapes）。
    7. **输出**：优化后的计算图（operator数量减少21%-65%）+ 每个tensor的选定layout + 2.5D memory映射方案 + 优化后的GPU kernel配置。与DNNFusion相比，Swin Transformer的operator从135减少至30.6（减少约77%），端到端延迟149 GMACS（4.4× speedup over DNNFusion）。

## 17-MAGIS- Memory Optimization via Coordinated Graph Transformation and Scheduling for DNN

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：MAGIS是一个DNN内存优化框架，通过协同图变换（Graph Transformation）和图调度（Graph Scheduling）优化计算图的内存使用和执行延迟。MAGIS有四大组件：(1) M-Analyzer：分析计算图并基于dominator tree和memory hot-spot构建Fission Hierarchy Tree (F-Tree)，F-Tree是Fission Transformation (F-Trans)的层级搜索空间抽象，每个F-Tree节点记录(S, D, n)，n=1为候选节点（子图尚未分裂），n>1为已启用节点（子图已沿维度D分裂为n部分）；(2) M-Rules：包含F-Tree变异规则（Enabling/Lifting/Disabling/Mutating四种规则，通过修改F-Tree节点的n来控制F-Trans的应用）、TASO规则（Aggregation Transformation聚合小op提升硬件利用率，Interim Transformation基于代数等价提供变换机会）、Scheduling-based规则（Re-materialization/De-re-materialization/Swapping/De-swapping四条规则）；(3) M-Optimizer：通过贪心搜索算法（优先队列按BetterThan比较，先比内存满足约束则比延迟）和增量调度算法协调图变换与调度；(4) 代码生成后端：将优化后的图和调度生成PyTorch Python代码，使用PyTorch CUDA Stream API实现异步Store/Load。
  - 实验比较：(1) PyTorch baseline（简单拓扑排序+立即释放未来不用的tensor），(2) POFO（DP-based re-materialization + swapping），(3) DTR（动态tensor re-materialization，MegEngine eager模式），(4) XLA（DNN编译器，HLO-level贪心re-materialization），(5) TVM/Relay（DNN编译器，基本内存回收），(6) Torch-Inductor/TI（基于Triton的DNN编译器，基本内存回收）。实验维度：latency overhead < 10%/5%约束下比较peak memory ratio，memory < 80%/40% PyTorch peak约束下比较latency overhead，完整memory-latency Pareto曲线对比，与micro-batching POFO的对比（ViT），heuristic消融实验，增量调度vs全调度的speedup和质量对比。

- 硬件平台是什么，配置是什么。
  - 硬件平台：Intel工作站，配备20核Intel Xeon Silver 4210R CPU，NVIDIA GeForce RTX 3090 GPU（24GB显存），CUDA 11.6，cuDNN 8.4.0。软件：PyTorch 2.1.0，MegEngine 1.12.3，TensorFlow 2.15.0，TVM 0.14.0。

- 开源编译框架是什么。修改了什么。
  - MAGIS为自研框架，使用rustworkx（Python图库）实现图数据结构，未基于XLA/TVM/Relay等现有编译框架修改。
  - 核心设计：(1) 提出Dimension Graph (D-Graph)——将tensor的空间维度和reduce-axis建模为图中的节点，识别可沿哪些维度进行F-Trans；(2) 提出F-Tree层级抽象——不实际变换图以避免复杂度指数增长，仅记录分裂信息，执行时按需展开split parts；(3) 将re-mat和swapping分解为图变换规则+重排序——memory-performance trade-off完全移入图变换阶段，调度阶段仅做不影响总延迟的重排序（Store尽早、Load尽晚以隐藏PCIe传输延迟）；(4) 增量调度——利用前序调度和变换影响的子图范围，基于narrow waist (NW)值确定需重调度区间，GraphPartition按nw(v)≤1节点分割子图为独立部分，DpSchedule对各部分DP重排序；(5) 基于Weisfeiler-Lehman Test的图哈希去重，避免重复评估已搜索过的图结构。

  - 开源：是。代码位于 https://github.com/pku-liang/MAGIS（ASPLOS '24）。
  - 编译框架输入到输出全过程（以ViT-base batch=64训练，memory limit=80% PyTorch峰值，max-level L=4为例）：
    1. **输入**：计算图G（从PyTorch导出的图表示，含operator节点和tensor依赖边），memory limit M，F-Tree max-level L=4。
    2. **初始化**：对G做初始拓扑排序调度→模拟器评估memory和latency→初始化M-State。
    3. **M-Analyzer分析**（Algorithm 1）：遍历D-Graph各连通分量D→对每个D提取子图G'及其dominator tree T→定位memory hot-spot tensor集合H→对G'中每个节点v计算heat(v)=Σ|w|（v支配的热点tensor总size）和score(v)=(1-1/2)×heat(v)-Σ|inputs|（估算F-Trans后峰值内存减少量）→获最大score smax→将[0,1]按L=4分割为[0.25,0.5),[0.5,0.75),[0.75,1),[1,1]四个区间→选择不同score区间的dominator节点vdm→取vdm的descendants除vdm自身构成sub-graph S→生成Fission candidates (S, D', n=1)→构造F-Tree。
    4. **M-Optimizer搜索**（Algorithm 3）：优先队列初始含初始M-State→循环pop最优μ→计算图哈希（Weisfeiler-Lehman风格：节点特征迭代hash(xv=hash(v⊕XOR(xu for u in pre(v))))）→若已访问则跳过→若F-Tree需更新则调用M-Analyzer→应用M-Rules生成新M-State：F-Tree Enabling Rule选叶子节点将n从1改2（沿head-dim将Q/K/V projection分裂为2部分，各split part顺序执行后concat）→Swapping Rule对skip-connection附近的hot-spot子图插入Store/Load→TASO A-Trans融合Q/K/V matmul为batch matmul→对每个新图执行增量调度（Algorithm 2：GetRescheduleInterval用NW值扩展需重调度区间→GraphPartition按nw≤1节点分割→DpSchedule DP重排序→合并局部调度）→模拟器评估→若BetterThan满足1.1×松弛条件则push回队列继续搜索。
    5. **输出**：3分钟时间预算耗尽后输出best M-State。代码生成后端将最优图+调度转换为PyTorch Python代码：按调度顺序写PyTorch op调用，Store用torch.cuda.Stream异步memcpy device→host，Load用异步memcpy host→device，F-Trans split parts用Python for循环顺序执行各part后concat。

## 23-Mind the Gap: Attainable Data Movement and Operational Intensity Bounds for Tensor Algorithms

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：Orojenesis——一种集成在Timeloop中的方法论，用于计算张量算法的数据移动下界。核心实现包括：(1) **Snowcat代理架构**：简化的两级存储层次（unconstrained buffer + infinite backing store），单处理单元架构，无并行化维度。相比真实架构（如Eyeriss-like），Snowcat的mapspace极小（以4k×4k×4k GEMM为例，Snowcat的mapspace比Eyeriss-like小7350×）；(2) **单Einsum bound推导**：对Snowcat架构进行穷举mapspace搜索，收集每个mapping的buffer大小需求和backing store访问量，持续更新不同buffer容量下的最优访问量Pareto点，连接Pareto最优映射形成ski-slope diagram；(3) **多Einsum Fusion分析**：通过Fusion Friendly Mapping Template (FFMT)约束链中每个Einsum的intra-layer mapping，支持四种约束集（FFMT-Full, FFMT-TiledK, FFMT-TiledN, FFMT-TiledKN），确保producer-consumer数据通过tile交换。FFMT约束集根据Einsum在链中的位置动态施加（首层可用FFMT-TileKN，中间层须用FFMT-Full，末层可用FFMT-TiledN）；(4) **链分段优化**：穷举所有2^(#Einsums-1)种链分段方案，搜索不同buffer容量下的最优分段策略。
  - 实验比较：(a) 单Einsum分析——不同GEMM尺寸（2k×2k×2k到8k×8k×8k）、不同convolution配置（filter size 1×1到5×5，stride 2，dilation 2）、不同batch/grouped BMM（heads 1-128，groups 1-32）；(b) 多Einsum分析——2-GEMM fusion（32k×4k×16k + 32k×16k×4k），对比No Fusion vs Untiled Fusion vs Tiled Fusion，fusion reduction factor up to 3.7× at 256MB buffer；(c) MHA fusion strategies对比——FlashAttention-like vs FLAT-like vs No Fusion；(d) 链分段——六Einsum链（Q_proj, bmm_QK, bmm_QKV, Final_proj, mm_0, mm_1），对比unsegmented fusion vs segmented fusion vs unfused baseline；(e) GPT-3-6.7b完整building block——含所有MHA和FFN Einsum，full fusion可达5.6× data movement reduction at 320MB buffer。

## 25-Be CIM or Be Memory- A Dual-mode-aware DNN Compiler for CIM Accelerators.pdf

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：CMSwitch，一个面向双模CIM加速器的DNN编译器。核心包括三个组件：(1) **DEHA (Dual-Mode Enhanced Hardware Abstraction)**：在CIM-MLC硬件抽象基础上引入双模CIM array参数（switch_method、switch_latency、array数量/尺寸、internal/external bandwidth），将计算-内存模式切换纳入编译优化空间。(2) **DACO (Dual-Mode-Aware Compilation Optimization)**：两步分治优化——先通过DP（Dynamic Programming）将网络划分为多个segment（考虑inter-segment mode-switch overhead：数据写回、模式切换延迟、权重重载），再通过MIP（Mixed-Integer Programming）在每个segment内联合优化CIM array的compute/memory mode分配与算子调度pipeline。(3) **DMO (Dual-mode Meta-Operator)**：引入`CM.switch(TOM/TOC, arrayaddr)` meta-operator表达编译结果中的模式切换，用`parallel{}`表示segment内并行执行，编译结果可转换为不同CIM芯片的ISA。

    5. **输出（DMO）**：meta-operator流，如`parallel{ CM.switch(TOM, [0:32]); MMM(WQ, Q); ... CM.switch(TOC, [0:16]); MEMORY(read, [16:48]); ... }`，表示segment内并行执行+array模式切换。后续可由后端转换为具体CIM芯片的ISA/机器码。

## 28-MIMDRAM: An End-to-End Processing-Using-DRAM System for High-Throughput, Energy-Efficient and Programmer-Transparent Multiple-Instruction Multiple-Data Computing

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：MIMDRAM在LLVM 12.0.0中实现三个编译Pass，透明地将C/C++应用循环自动向量化并映射为PUD bbop指令。Pass 1（Code Identification）：修改LLVM loop auto-vectorization pass，每循环选择最大vectorization factor（替代原cost model），并删除向量化循环中的load/store指令（因PUD直接操作DRAM数据）。Pass 2（Code Scheduling & Data Mapping）：为Pass 1输出的bbop指令构建data-dependency graph (DDG)，用DFS沿拓扑序遍历DDG——独立bbop分配到不同DRAM mat（分配不同mat label），依赖bbop分配到同一mat，必要时插入bbop_mov数据搬运指令。Pass 3（Code Generation）：修改LLVM X86 backend，将bbop IR翻译为带ML（mat label）和VF（vectorization factor）字段的ISA汇编，插入pim_malloc/bbop_trsp_init调用，支持OpenMP outermost loop展开为SIMT风格执行。
  - 实验比较：与SIMDRAM（无编译支持的手动PUD编程）、CPU（Intel Skylake AVX-512）、GPU（NVIDIA A100）以及PIM架构DRISA/Fulcrum对比SIMD利用率、性能、能效、吞吐量、公平性和面积归一化性能。

  - GPU基线：NVIDIA A100 7nm，6912 CUDA Cores，108 SM，1.4GHz，40MB L2，40GB HBM2

  - 框架：LLVM compiler toolchain v12.0.0 + clang。修改了三处LLVM组件：(1) loop auto-vectorization pass——强制max VF替代cost model；(2) 新增DDG构建与mat label调度pass；(3) X86 backend新增bbop ISA指令生成（含ML/VF字段）。

  - 例子输入C代码：`for(i=0;i<1024;i++){C[i]=A[i]+B[i]; F[i]=D[i]*E[i]; G[i]=C[i]-F[i];}`
  - 全过程：
    1. **Pass 1 (Code Identification)**：clang将C编译为LLVM IR。auto-vectorization pass分析循环→max VF=1024→生成向量指令 `%3=add<1024 x i32> %1,%2`（load A,B被移除），`%6=mul<1024 x i32> %4,%5`，`%7=sub<1024 x i32> %3,%6`，store C/F/G被移除。输出：纯算术SIMD指令序列。
    2. **Pass 2 (Code Scheduling)**：构建DDG（add→sub, mul→sub）。DFS拓扑遍历：左子图(add,sub)分配mat label i，右子图(mul)分配mat label j。由于sub依赖mul结果（跨mat），插入 `bbop_mov` 从mat_j搬运数据到mat_i。输出带mat label的bbop指令：`bbop_add C A B 512 32 mat_i 1024`，`bbop_mul F D E 512 32 mat_j 1024`，`bbop_mov t F 0 0 512 3`，`bbop_sub G C t 512 32 mat_i 1024`。
    3. **Pass 3 (Code Generation)**：malloc→`pim_malloc(size,mat_label)`，插入 `bbop_trsp_init` 注册memory object到transposition unit。X86 backend将bbop IR翻译为带ML/VF字段的ISA汇编。输出：CPU+PUD混合指令的可执行二进制。
  - 作用：程序员无需手动提取SIMD并行、编写PUD代码或管理DRAM mat分配；编译器自动识别PUD-friendly循环，以mat label调度实现MIMD并行，透明生成PUD指令。

## 3-AMALI- An Analytical Model for Accurately Modeling LLM Inference on Modern GPUs.pdf

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：AMALI是一个基于SASS指令级分析的GPU解析性能模型（analytical model），用于精确预测LLM推理中CUDA kernel的cycle数。AMALI由六个组件构成一个分析工具链（pipeline）：
    1. **SASS Tracer**：基于NVBit GPU动态二进制插桩工具，收集SASS指令Trace，包括指令名、modifiers、寄存器使用、内存访问地址、grid/block size、shared memory/register file用量、warp ID和SM ID。
    2. **SASS Parser**：从Trace提取grid size、block size等信息，使用K-means聚类选取representative warp，按warp_id%4将warp分配到sub-core。
    3. **Cache Simulator**：基于SASS Tracer收集的内存访问地址和GPU架构配置参数模拟cache行为，计算AMAT（Average Memory Access Time）。
    4. **Interval Analyzer**：使用AMAT和warp profile将warp执行划分为离散intervals，建模selected、wait、short_scoreboard（shared memory stall）、long_scoreboard（global memory stall）。
    5. **Interval Parser**：建模math_pipe_throttle（使用Tensor Core Model）、lg_throttle（L1 D Cache contention）、mio_throttle（MDM: MSHR+NoC+DRAM contention）。
    6. **KLL comp**：通过micro-benchmark测量Kernel Launch Latency，建模constant cache和instruction cache miss造成的stall。

    三项核心创新：
    - **Tensor Core Modeling (TCM)**：使用 II_TC = FMA_count / TP_dt 计算tensor core initiation interval，取代GCoM的warp_size/FU_lanes。考虑HMMA指令的tensor size modifier（16816 vs 1688）和data type modifier（F32等）。例如A100上HMMA.16816 CPI=8（2048/256），HMMA.1688 CPI=4（1024/256）。
    - **Kernel Launch Latency (KLL)**：KLL = s·GS + k，其中s = α·BS² + β·BS + γ（A100: α=0.0036, β=0.0366, γ=1.1891），建模imc_miss和no_instructions stall。
    - **Instruction Divergence (ID)**：ID = (max_subCoreInstr - I_SC_Repr.warp) / IssueRate，建模warp间指令数差异（可达6x）。

    AMALI总cycle预测公式：C = SI + SI_P + KLL + ID，其中SI = S_selected + S_wait + S_short_scoreboard + S_long_scoreboard，SI_P = S_math_pipe + S_lg + S_mio。

  - 实验比较：AMALI vs GCoM（SOTA GPU解析模型，ISCA '22）在NVIDIA A100上对比kernel cycle预测准确率（MAPE指标），使用Llama3-8B/15B推理、DeepBench、Rodinia作为workload。消融实验：GCoM+KLL、GCoM+ID、GCoM+TCM逐步叠加。H100设计空间探索验证tensor core throughput建模准确性。

  - 服务器：2x AMD EPYC 7543 CPU，NVIDIA A100 80GB VRAM，2TB DDR4 DRAM
  - 软件：Ubuntu 22.04.5 LTS，CUDA 11.7
  - A100 GPU架构参数：1410 MHz，108 SMs（每SM 4 sub-cores），单发射warp scheduler，Tensor Core 256 fp16/bf16 FMAs/clk (fp32 accumulate)，INT #64/4 cycles，FP32 #64/4 cycles，FP64 #32/4 cycles，SFU #16/23 cycles，L1 128B/line sectored/write-through/64-way/4-bank（ld 23 cycles/st 19 cycles），shared memory 37 cycles，L2 40MB/16-way/32B/line sectored/write-back（224 cycles），DRAM 1940 GB/s/1512 MHz（290 cycles），NoC 1200 MHz

## 69-Optimizing Dynamic-Shape Neural Networks on Accelerators via On-the-Fly Micro-Kernel Polymerization.pdf

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：MikPoly是一个面向dynamic-shape tensor operator的两阶段动态shape tensor编译器，核心机制为micro-kernel polymerization：(1) **Offline阶段（Micro-Kernel Generation）**：从两阶段tiled program template Q的Q_offline（内层offline loops）提取micro-kernel template K_hat，使用TVM auto-scheduler离线生成一组高度优化的固定尺寸micro-kernels S_K~（每个micro-kernel对应特定(uM,uN,uK)尺度的tiled loop nest，专为M_local优化）。同时为每个micro-kernel建立性能模型g_predict(t,K~,H)——分段线性函数，通过在单PE上运行t从1到5120的pipelined task实验学习系数。超参n_gen=32（每维度tile size步长16）、n_syn=12（合成测试case数）、n_mik=40（保留的top micro-kernel数）。(2) **Online阶段（Micro-Kernel Polymerization）**：当operator的runtime shape已知时，MikPoly用预定义的polymerization patterns（9种pattern：Pattern I-IX，基于7-block skeleton的输出分块）重组Q_online（外层online loops）为不同的program implementations，每个implementation含若干region R_i，每region内包含一个parameterized micro-kernel。通过polymerization strategy将parameterized micro-kernel替换为offline生成的fixed-size micro-kernels，并用polymerization cost model（Cost=Σ f_wave × f_pipe）预估每种组合的性能，选最优。(3) **Multi-Level Accelerator Abstraction H=(P_multi, M_local, M_global)**：GPU上P_multi=SMs、M_local=(shared memory, local memory, register)、M_global=global memory；NPU上P_multi=DaVinci Cores、M_local=(L1 buffer/unified buffer, L0 buffer, register)、M_global=global memory。(4) **Cost Model**：f_pipe(R_i,K~_i,H)=g_predict(f_num(R_i,K~_i),K~_i,H)预测pipelined task在单PE上的执行开销；f_wave(R_i,K~_i,H)=ceil(f_parallel/|P_multi|)估算并行执行所需wave数。两因子相乘求和（Eq.2）得到总开销。(5) **Local Padding**：类似CUTLASS的padding技术处理非对齐shape边界，消除边界检查开销。
  - 实验比较：(a) GEMM在GPU上：MikPoly vs cuBLAS（avg 1.47x, max 4.82x）、vs CUTLASS（avg 3.02x）。Convolution在GPU上：MikPoly vs cuDNN（avg 1.98x, max 5.38x）、vs CUTLASS（avg 1.72x）。(b) GEMM在NPU上：MikPoly vs CANN（avg 1.10x）；Convolution在NPU上：MikPoly vs CANN（avg 1.41x）。(c) End-to-end语言模型inference（BERT/DistilBERT/RoBERTa/ALBERT）：MikPoly vs cuBLAS（avg 1.37x）；CNN模型（AlexNet/GoogLeNet/ResNet/VGG）：MikPoly vs cuDNN（avg 1.46x）。(d) vs dynamic-shape compilers：MikPoly vs DietCode（GEMM avg 2.94x）、vs Nimble（avg 7.54x, 因Nimble常产生out-of-range错误仅25%速度）。End-to-end语言模型MikPoly vs DietCode avg 1.55x。(e) LLM (Llama2-13b) operator级：MikPoly vs cuBLAS avg 1.09x-1.24x；end-to-end vs FasterTransformer avg 1.01x-1.05x。(f) Cost model有效性：MikPoly vs MikPoly-Oracle（穷举搜索）speedup 0.96x，但online搜索时间仅~2μs vs Oracle的~1.6s。(g) MikPoly解决load-imbalance：通过multi-micro-kernel polymerization（如GEMM-AB），将grid_size从128提升至352，sm_efficiency从58.90%恢复到96.06%。

## 8-SambaNova_SN40L_Scaling_the_AI_Memory_Wall_with_Dataflow_and_Composition_of_Experts.pdf

- 属于编译框架的实现是什么？实验比较什么？
  - 实现：SN40L compiler是一个面向dataflow架构的全栈编译器，核心功能包括：
    - **Automatic Operator Fusion**：在Python framework level自动将20+ operators融合为单一kernel call（vs 传统fusion 1-5 operators），通过coarse-grained pipeline将operators映射到PCU pipeline stages和PMU stage buffers。编译器自动执行place-and-route (PnR) 将逻辑dataflow graph映射到物理PCU/PMU 2D mesh。
    - **Static Memory Lifetime Analysis**：利用SN40L programming model无动态内存分配和指针别名的特性，编译器静态分析symbol lifetimes，实现编译期garbage collection（多个无overlap lifetimes的logical symbols共享同一device virtual address）。无需runtime GC开销。
    - **Heterogeneous Memory Management (HBM/DDR Spilling)**：编译器默认优先使用HBM。当HBM容量不足时：(a) 分析每个symbol的temporal locality和transfer footprint估算bandwidth requirement；(b) 按aggregate transfer size升序排序，spill bandwidth需求最小的symbol到DDR；(c) 实际中LLM weights获最高priority留在HBM，activation/intermediate results必要时spill。
    - **Static Bandwidth Model**：编译器维护application requirements和hardware characteristics的static bandwidth model，进行资源分配和traffic shaping。支持software-controlled packet throttling和PMU bank bit重映射消除bank conflicts。
    - **Multi-level Parallelism Mapping**：编译器将data/tensor/pipeline parallel dataflows统一抽象，自动映射到intra-socket (PCU mesh) 和inter-socket (P2P protocol) 资源。
  - 实验比较：(1) Fused vs Unfused kernel launch count——Llama7B prefill达到11× fewer kernel calls, FlashFFTConv达到~125× reduction；(2) Fusion速度提升——FlashFFTConv 13×, Llama7B prefill ~3×, Mistral推理 ~13×, sparseGPT训练 ~2.5×；(3) Hardware-Orchestrated vs Software-Orchestrated kernel launch——decode benchmarks 1.4×-8× speedup, prefill/training ~1.1× speedup；(4) 编译器managed DDR→HBM spilling vs 全部驻留HBM的performance comparison。

- 硬件平台是什么，配置是什么。
  - 编译目标：SN40L RDU (TSMC 5nm, 638 BF16 TFLOPS, 520MB SRAM, 64GB HBM, 1.5TB DDR, 1040 PCUs + 1040 PMUs)。实验运行在8-socket SN40L Node（除FlashFFTConv在单socket）。Host: x86 CPU via PCIe。
  - 模型：Llama2-7B/70B, Bloom-176B, Mistral-7B, Falcon-40B, Llava1.5-7B, sparseGPT-13B, FlashFFTConv, Llama3.1 8B/70B/405B。所有模型BF16 weights，mixed BF16/FP32 activations。

  - SN40L compiler为SambaNova自研闭源编译器。从Python framework level (PyTorch) 接收模型定义。不是修改现有开源编译器。

  - SN40L compiler为闭源商业编译器。论文描述了编译器架构但不提供源代码。
  - **编译全流程（以Llama2-7B decoder layer为例）**：
    1. **Input**：PyTorch model graph（decoder layer含self-attention Q/K/V projection GEMMs + attention + FFN GEMMs + element-wise ops + transpose/reshape）。Compiler接收Python-level operator graph。
    2. **Graph Analysis & Fusion Planning**：Compiler分析operator graph的data dependencies和access patterns。识别fusion opportunities——decoder layer内的Self-Attention + FFN operators按dataflow chain组合为coarse-grained pipeline stages。Transpose操作fused到PMU buffer的read/write access pattern（非独立operator）。
    3. **Resource Allocation**：估算每个pipeline stage的compute需求（FLOPs）和memory需求（tensor sizes + bandwidths）。决定PCU数量/配置（GEMM→systolic array, element-wise→SIMD），PMU数量/partitioning（capacity-driven多bank, bandwidth-driven多PMU交织）。以Figure 4为例：Gemm0/Gemm1分配更多PCUs（systolic mode），Mul用少数PCUs（SIMD mode），I0 split为2 PMUs满足带宽，S0-S3 split为4 PMUs满足容量，T00-T03/T10-T13用PMU diagonal striped format实现transpose。
    4. **Place and Route**：在1040 PCUs + 1040 PMUs的2D mesh上分配physical units。配置RDN routing tables（static flow routes for tensor data, scalar addresses, control tokens）。配置multicast trees for one-to-many fan-out paths。配置sequence ID generation for many-to-one reordering。
    5. **Memory Allocation**：Static lifetime analysis → 计算每个intermediate tensor的live range → 复用non-overlapping lifetimes到相同PMU regions。HBM优先分配weights（高temporal locality）→ 若HBM不足，按transfer bandwidth排序spill低需求symbols到DDR。Compiler输出memory map（每symbol的HBM/DDR地址 + PMU virtual address）。
    6. **Code Generation**：生成kernel binary（PCU/PMU/AGCU配置bitstream + RDN routing配置）。Kernel launch schedule——software-orchestrated (host CPU triggers each kernel via AGCU commands) 或 hardware-orchestrated (static schedule loaded到AGCU, AGCU autonomous trigger)。
    7. **Runtime**：CoE runtime加载compiled kernel binary → AGCU执行Program Load → Argument Load (weights addresses, input tensor addresses) → Kernel Execute。Streaming dataflow pipeline开始：HBM→AGCU→RDN vector fabric→PMU stage buffers→PCU compute→PMU buffers→...→final output via AGCU to HBM。
    8. **作用**：将PyTorch model自动编译为SN40L RDU上的单一fused streaming dataflow kernel，利用>85% HBM带宽（vs GPU <50%），消除kernel launch overhead（decode benchmarks 1.4×-8× speedup），自动管理HBM/DDR异构存储。
