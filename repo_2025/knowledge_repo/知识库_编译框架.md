## Branch-and-Bound Monotonic Optimization for LLM Scheduling (面向LLM调度的分支界限单调优化)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Branch-and-Bound Monotonic Optimization for LLM Scheduling是ExeGPT提出的调度优化算法，将LLM推理的constraint-aware scheduling形式化为一个单调优化问题，并使用branch-and-bound方法求解最优控制变量配置。优化问题：maximize Throughput(B_E, B_D, B_m, TP, F_E, S, P_E, P_D) subject to Latency(...) < L_Bound。其中控制变量（batch sizes, TP degree, decoder micro-batch, encoding frequency）被设计为对throughput和latency具有单调性（monotonicity）——即每个变量单独增加时，throughput和latency的变化方向是单调的（如增加batch size→throughput单调增、latency单调增）。利用此单调性，branch-and-bound算法可在每次split后通过仅检查block的两个边界点来估计该block内所有点的latency/throughput上界和下界，从而高效剪枝不满足约束的block。相比黑盒优化（如Bayesian optimization）或exhaustive search（需5小时+），ExeGPT的scheduling仅需3秒-5分钟。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
Branch-and-Bound Scheduling Algorithm (Algorithm 1 in paper) 伪代码：
```
Input: [a1, b1], [a2, b2] // variable ranges for x1, x2
       εT, εL             // tolerance for throughput, latency
       Lb                  // latency bound
Output: optimal (v1, v2)

1  B0 = [(a1, a2), (b1, b2)]  // initial block covering full search space
2  B0.lowr = perf(a1, a2)     // estimate latency/throughput at lower-left corner
3  T* = B0.lowr.throughput; T*.config = (a1, a2)
4  Q = PriorityQueue({B0})    // sorted by lower bound throughput
5  while Q ≠ ∅:
6    B = Q.popMax()
7    ptl, pbr = perf(B.topLeft), perf(B.bottomRight)
8    perf* = higher_thrput_satisfying_Lb in [ptl, pbr]
9    if perf* == ptl: B1, B2 = split B vertically
10   else: B1, B2 = split B horizontally
11   foreach B' in [B1, B2]:
12     B'.upp = perf(B'.topRight)       // upper bound (best throughput)
13     B'.lowr = perf(B'.bottomLeft)    // lower bound (best latency)
14     if B'.lowr.latency < Lb + εL: Q.add(B')
15   B' = higher_thrput_satisfying_Lb in [B1.upp, B2.upp]
16   if B'.upp.throughput > T*:
17     T* = B'.upp.throughput; T*.config = B'.topRight
18   Q = Q \ {B | B.upp.throughput + εT < T*}  // prune dominated blocks
19 return T*.config
```
关键设计：(1) Split heuristic——比较top-left和bottom-right两个边界点的throughput（满足latency约束者），选higher throughput的点的方向进行vertical/horizontal split，加速收敛；(2) Tolerance (εT, εL)——允许small non-monotonicity（实验中~96%+配置点满足单调性with 5% tolerance），tolerance越小剪枝越激进、搜索越快但可能sub-optimal；(3) Partial TP的特殊处理——TP degree固定为常数运行算法，多次运行不同TP degree值后选最优。

实现要点：(1) 需要XSimulator估计每个配置点的latency/throughput（基于profiling数据和序列长度分布）；(2) XScheduler对RRA和WAA分别运行branch-and-bound，选取throughput较高者的schedule；(3) Priority queue根据lower bound throughput排序——优先探索最有希望的block（best-first search）；(4) 算法复杂度远低于exhaustive search（O(K^n) → O(log(K)^n) level pruning），实际scheduling时间从数小时降至数分钟；(5) 单调性需事先验证——论文通过sweep实验验证了4个控制变量在96%+配置点上的单调性；(6) 适用于任何具有单调性控制变量的LLM推理优化问题，可作为通用scheduling framework。

## RCPSP for Communication Scheduling (资源受限项目调度问题用于通信调度)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
RCPSP（Resource Constrained Project Scheduling Problem）是运筹学中经典的NP-hard组合优化问题：给定一组任务（tasks），每个任务有执行时间、资源需求和前驱依赖约束，在有限资源总量下调度所有任务的开始时间，最小化总完工时间（makespan）。Concerto将其应用于分布式深度学习中的通信调度：将计算图中的每个算子视为一个任务，定义两类资源R={computation: 1, communication: 1}（即任意时刻最多1个计算任务+1个通信任务并发执行，对应GPU的SM用于计算+NIC用于通信的物理限制）。每个任务i的执行区间建模为离散时间变量[Si, Ei]（Si为开始时刻、Ei=Si+Ti为结束时刻，Ti为profiling获取的执行时间）。约束条件：(1) 依赖保留——任务i的所有前驱任务j满足Ej ≤ Si；(2) 资源限制——任意时刻t，同时活跃的任务数不超过该资源总量。目标函数为 min makespan = En（最后一个任务的结束时间）。通过ILP（整数线性规划）求解，使用Google OR-Tools作为求解器。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
Concerto将RCPSP建模用于编译器的Scheduling pass：
1. **输入**：计算图G=(V,E)，其中V为算子节点（含计算和通信），每个节点vi已通过profiling获得执行时间Ti，并标注资源类型ui∈{comp=1/comm=1}。
2. **变量编码**：对每个节点i定义Si（整数起始时刻）和Ei=Si+Ti。所有时间值归一化为整数（仅保留相对时间比例）。
3. **ILP公式化**：
   - 目标：min En（全局makespan）
   - 约束1（执行连续）：Ei = Si + Ti
   - 约束2（依赖）：∀j∈dep_i, Ej ≤ Si
   - 约束3（资源）：∀t∈{1..M}, Σ_{i:Si≤t<Ei} u_ir ≤ R_r（每时刻每类资源使用≤1）
4. **Decoding**：求解得到各任务的开始时间Si后，按Si升序生成拓扑执行序。将通信任务提前到其执行时段内的计算任务之前（因通信需要少量SM launch kernel，若被计算占满SM会延迟launch）。
5. **输出**：优化后的拓扑序列，交给运行时按序dispatch。
对比JAX/XLA的启发式latency hiding scheduler：RCPSP求解器在NVLink关闭时表现显著更优（因为启发式不适应不同硬件环境下的计算-通信比），展现出求解器驱动的自动化调度的通用性和适应性优势。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
RCPSP本身可用多种方法求解：(1) 精确算法——ILP（如OR-Tools的CP-SAT求解器）、分支定界（Branch and Bound）；(2) 启发式——优先级规则（如最短处理时间SPT、最多后继数MTS）、遗传算法、模拟退火。Concerto选择ILP求解器因为图规模可控（经过odd-even分块后每块节点数有限），求解质量高。实际使用中配合profiling缓存机制（operator name+input shape为key），仅第一次编译时需profiling（几十秒），后续同算子跳过。OR-Tools是Google开源的运筹优化工具包（https://developers.google.com/optimization），内置CP-SAT求解器，支持Python/C++/Java等接口。

## Auto-Decomposition (自动分解，Critical Communication + SPMDSpec BFS)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Auto-Decomposition是Concerto提出的一种编译pass，自动识别计算图中无法通过调度完全重叠的"关键通信"（Critical Communication），并沿各张量轴自动探索分解上下文（Decomposition Context），通过将通信前后的计算按轴分解成多个微batch，使通信与计算以更细粒度交错执行，创造overlap机会。核心理念：对于同步通信算子（如forward pass中的all-reduce），调度无法使其与前后计算重叠（因数据依赖），但若将计算拆为N个chunk，则可以做到(chunk 0 compute → chunk 0 communicate & chunk 1 compute → chunk 1 communicate & chunk 2 compute → ...)，实现pipeline overlap。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
Auto-Decomposition在Concerto编译流程中作为独立pass运行，详细流程：
1. **识别Critical Communication**：遍历图中所有通信算子Ci。对每个Ci，将图中其他节点分类为前驱节点P（必须等Ci完成才能执行）、后继节点S（必须等Ci完成后开始）、独立节点I（与Ci无依赖关系）。若Σ_{n∈I_comp} Time_n - Σ_{n∈I_comm} Time_n < Time_Ci（即独立计算节点能提供的overlap时间不足以完全隐藏Ci），则Ci被标记为critical communication，进入集合C。
2. **SPMDSpec驱动的BFS探索**：对每个Ci∈C，沿其各张量轴（如batch, sequence, hidden）分别向前驱和后继方向做BFS探索，使用EasyDist提供的SPMDSpec（ShardSpec+CombineSpec）判断每个节点是否可沿给定轴分解。SPMDSpec定义：(a) ShardSpec：对每个输入维度标注NoShardDim(N)或ShardDim(S_j)（可切分维度组）；(b) CombineSpec：每个S_j对应的结果合并函数（gather/reduce等）。函数SPMDPropagate从当前节点的前驱轴推导出其可用的分解轴——若推导成功，节点加入Decomposition Context S；若返回None（如LayerNorm不可沿hidden轴分解），则停止该方向探索。BFS终止条件：t_d（累计Context计算时间）≥ T_Ci（通信时间）或无可探索节点。
3. **策略候选集生成**：对每个Ci得到沿各轴的分解策略集合Si（包含Decomposition Context中的节点和分解轴）。
4. **成本计算**：对每个策略计算非重叠通信部分cost = max{cost1, cost2, cost3, 0}，其中cost1/cost2/cost3分别对应前驱和后继计算时间不同的三种情况，均考虑分解度N和重叠效率衰减因子α=1.2（empirical，因分解后算子硬件利用率下降~20%）。
5. **ILP联合选择**：若多个critical comm的分解上下文有交集，通过ILP联合优化选择互不冲突的策略组合（考虑不同轴冲突和同轴加倍开销的额外cost matrix M_ij）。
6. **示例**：GPT Feed-Forward中的all-gather（sequence维度），其前驱为LayerNorm，后继为MatMul1→GeLU→MatMul2。沿batch或sequence轴分解时，Context = {LayerNorm, MatMul1, GeLU, MatMul2}；沿hidden轴时，LayerNorm不可分解，Context仅{MatMul1}。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
Auto-Decomposition依赖EasyDist（https://github.com/alibaba/easydist）提供SPMDSpec——EasyDist是一个自动化并行系统，为每个ATen算子预定义了SPMDSpec（输入如何切分、输出如何合并）。Concerto通过torch._custom_ops注册自定义算子，使分解context中的节点调用分解版本的kernel（如将大MatMul拆为N个并行的子MatMul，结果按CombineSpec合并）。与Google Decomposition（仅能分解MatMul与all-reduce的固定模式）不同，Concerto的BFS探索可覆盖任意算子类型。N（分解度）的选择需权衡：N越大overlap机会越多，但硬件利用率下降（Achieved TFLOP/s降低）且HBM traffic增加（需多次读写）。

## Odd-Even Scheduling Method (奇偶调度方法)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Odd-Even Scheduling是Concerto为降低RCPSP ILP求解时间复杂度而提出的分块迭代近似方法。RCPSP本身是NP-hard，对于含数万节点的神经网络计算图直接求解不现实。Odd-even方法灵感来自奇偶排序算法（odd-even sort）：将计算图的拓扑序列划分为等大小的block（block_size=b），每轮对当前block内的节点调用ILP求解器求解局部最优，然后偏移block边界（偏移half block_size），重新分块再求解。通过多轮迭代，信息通过块边界的滑动逐步传播到远距离节点，使全局调度逐步逼近最优。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
流程：
1. **初始拓扑序**：从程序定义获取默认执行序（如PyTorch eager mode的算子定义顺序）。
2. **分块**：将当前拓扑序划分为连续的长度为b的block（block_1, block_2, ..., block_k，k=n/b）。
3. **Odd轮（首轮）**：对每个block（block边界为[0,b), [b,2b), ...），忽略跨block的依赖边（即edges至少一端在block外），仅优化block内节点排序。每个block独立送入ILP求解器（求解时间约t_b常数）。
4. **Even轮**：将block边界偏移b/2，即新block为[b/2, 3b/2), [3b/2, 5b/2), ...。对偏移后的block同样做ILP求解。偏移使得不同block之间的信息得以传递。
5. **迭代**：重复odd轮和even轮，共k轮。每轮时间O(n/b · t_b)，总时间O(k·t_b·n/b)为多项式。k和b可控制（平衡解质量和编译时间）。
6. **收敛**：实验显示ViT ZeRO-3在8 GPU上约4轮达近最优加速比；WideResNet自动并行约6轮达近最优。各轮加速比逐渐收敛。
原始拓扑序对最终结果影响很小——多轮迭代后信息充分传播。

实现上，Concerto的odd-even通过compile-time配置控制block_size和rounds数。各轮独立调用OR-Tools CP-SAT求解器。编译时间通常在几分钟内（与数天训练时间相比可忽略）。未来优化方向：开发自适应算法，根据图规模自动选择block_size和rounds数；对相似batch size的编译结果做增量更新避免完全重编译。

## Communication Fusion in Compiler (编译器中的通信融合)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Communication Fusion是Concerto编译器Scheduling pass中的一项优化技术，将计算图中可互换（interchangeable，交换执行序不破坏依赖）且可融合（fusible，同类型同参数）的多个通信算子合并为一个fused通信算子。目的是：(1) 减少GPU kernel launch overhead（每次通信调用需占用少量SM启动kernel）；(2) 提高网络带宽利用率（合并小消息为大消息，减少传输协议开销）。该优化在ZeRO等产生大量通信算子的场景尤其有效。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
Concerto的Communication Fusion算法（Algorithm 1）流程：
1. 获取当前拓扑序sched ← TopoOrder(fx_graph)
2. 遍历sched中的每个算子task：
   - 若task是可融合通信算子：若selected列表为空则加入；若task与selected中的算子可互换（交换序不破坏依赖）且可融合（同类型同参数），则加入selected。否则对selected中已收集的算子执行FuseNodes合并，清空selected，重置索引。
   - 若task不是可融合通信：继续遍历但记录retrieve_task用于回溯。
3. 遍历完所有算子后，若selected非空则最后再执行一次融合。
4. FuseNodes实现：将selected中的多个通信算子替换为单个fused通信算子（合并通信buffer、统一发起单次NCCL collective调用）。
5. 融合同步性保证：只有类型相同（如都是all-reduce或都是all-gather）且参数相同（通信域、目标GPU集合相同）的算子才能融合。
效果实验：GPT ZeRO-3在32 GPU上，融合将step time从0.614s降至0.468s（FP16），提升显著。

通信融合在NCCL层面本身支持——单次all-reduce等collective调用天然比多次小调用高效。关键是编译器层面自动识别可融合机会——人工很难在包含数千算子的图中找到所有可互换的同类型通信。Concerto通过图依赖分析+Algorithm 1自动完成。类似技术也见于PyTorch DDP的gradient bucketing（将多个参数的梯度打包后再all-reduce），但DDP是固定策略（按参数顺序分桶），Concerto是基于图结构的精准分析。

## ConcertoIR

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
ConcertoIR是Concerto编译器框架的中间表示（Intermediate Representation），在PyTorch torch.fx的ATen IR基础上扩展而来。它在fx Graph的每个算子上附加了SPMD信息（SPMDSpec，来自EasyDist）和算子级元数据（profiling时间、通信类型等），形成"计算+通信"混合图，是Concerto两个核心编译pass（Auto-Decomposition和Scheduling）的共同数据基础。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
ConcertoIR的构建和使用流程：
1. **输入**：用户PyTorch函数（如train_step）经torch.fx trace得到fx Graph（纯ATen IR）。
2. **变换**：根据parallel_method参数，将fx Graph中的并行通信算子（如all-reduce, all-gather, reduce-scatter）显式化为通信节点。通信节点的插入逻辑由parallel_method的注册函数定义（如PTD parallelism插入all-reduce in forward/backward，ZeRO插入all-gather和reduce-scatter）。
3. **标注**：对计算图中的每个算子节点（含计算和通信），通过EasyDist的算子数据库查询其SPMDSpec（ShardSpec + CombineSpec）。SPMDSpec记录算子在不同输入轴上的可分解性和结果合并方式，是Auto-Decomposition进行BFS探索的关键元信息。
4. **Profiling标注**：profiling结果（执行时间Ti）作为节点属性持久化到ConcertoIR，供Scheduling的ILP solver使用。
5. **内部数据结构**：ConcertoIR底层复用torch.fx的数据结构（Graph + Node），但逻辑上是计算+通信混合图——计算节点（MatMul, LayerNorm, GeLU等）和通信节点（all-reduce, all-gather, reduce-scatter等）在同一图中通过edge表示依赖和数据流。
6. **输出**：经过Auto-Decomposition和Scheduling两个编译pass后，ConcertoIR中的节点获得优化后的拓扑序和执行策略，供运行时dispatch。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
ConcertoIR本质上是torch.fx IR的扩展层——不定义新的IR语言或格式，而是通过Node.meta字典附加SPMDSpec、profiling结果、调度决策等信息。这种设计复用了PyTorch生态的工具链（如torch.fx的图变换API、可视化、序列化等）。EasyDist的SPMDSpec覆盖常见ATen算子（MatMul, ReLU, LayerNorm, GeLU等），通过torch._custom_ops注册的自定义算子也需提供SPMDSpec以支持auto-decomposition。

## Hardware-Mapping Co-Exploration (Co-Optimization)

术语解释
Hardware-Mapping Co-Exploration（硬件-映射协同探索）是一种将加速器硬件配置（如on-chip buffer容量）和软件映射策略（如DNN子图划分）联合优化的方法，通过统一的目标函数在一次优化流程中同时决定硬件参数和映射方案，而非分步独立决策。

术语是什么？
Hardware-Mapping Co-Exploration 是 DNN 加速器设计空间探索（DSE）的一种范式。传统方法通常分两步：先确定硬件配置（如buffer大小），再基于固定硬件做映射优化（如layer fusion/graph partition），两步之间信息不互通。Co-exploration将硬件配置和映射方案编码在同一优化变量（如genome）中，用统一cost function = Hardware_Cost + α × Mapping_Cost（其中Hardware_Cost通常是buffer面积/容量，Mapping_Cost是EMA/energy/latency等执行开销，α为偏好超参数）进行联合评估和优化。Cocco论文展示了separate buffer和shared buffer两种内存设计下的co-exploration，shared buffer因activation和weight共享同一物理空间而实现更灵活的分配，通常获得更低的总体cost。

从编译框架角度拆解：
Cocco的co-exploration在编译框架中的运作流程：
1. **输入**：DNN计算图DAG、基础硬件配置（PE阵列规格、core数、dataflow模式）、内存设计空间描述（buffer容量范围和粒度）。
2. **联合编码**：每个genome同时包含partition scheme P（每个layer → subgraph的映射）和memory configuration M（buffer容量值）。Cocco中M的范围为global buffer [128KB, 2048KB]粒度64KB、weight buffer [144KB, 2304KB]粒度72KB（separate buffer设计）。
3. **统一评估**：simulator对每个genome解码→执行consumption-centric subgraph execution flow确定各layer的tile size/offset→检查buffer约束（超限则in-situ split-subgraph保证validity）→计算子图级的EMA/energy/latency→聚合全图cost = |M| + α × ΣEnergy。
4. **联合优化**：GA的crossover混合两个parent的partition和memory配置，mutation-DSE操作专门修改memory容量（正态分布采样），使硬件参数和映射方案在同一进化过程中协同改进。
5. **输出**：最优(partition, memory)组合——既决定哪些layers融合为subgraphs，也推荐最佳buffer容量。

术语一般如何实现？如何使用？
Co-exploration通常通过以下方式实现：(1) 基于进化算法（如GA、SA）：将硬件参数和映射方案编码为个体，通过变异/交叉探索联合空间。Cocco使用GA，MAGMA[31]使用GA探索多DNN多核映射；(2) 基于强化学习：如HASCO[67]使用RL探索硬件-软件co-design空间；(3) 基于约束优化/ILP：形式化联合优化为整数规划问题。Co-exploration输出直接指导芯片设计（选择buffer大小）和编译器/部署工具（生成subgraph partition方案），避免over-provisioning或under-provisioning内存。Cocco的实验显示co-exploration相比two-step方案（RS+GA或GS+GA）具有更高的样本效率（需样本数仅为其3.3%~27%），因为two-step中各容量下的partition信息无法跨容量复用。

- 19-Cocco- Hardware-Mapping Co-Exploration towards Memory Capacity-Communication Optimization

## Genetic Algorithm for DNN Graph Partition

术语解释
Genetic Algorithm（GA）for DNN Graph Partition 是将DNN计算图划分为若干子图（subgraph）的优化方法，通过模拟自然选择（crossover, mutation, selection）在巨大的划分方案空间中搜索最优解，适用于任意拓扑的神经网络（plain/multi-branch/irregular）。

术语是什么？
DNN Graph Partition 是将神经网络计算图 G=(V,E) 划分为一系列子图的组合优化问题。目标是找到划分方案 P: V→N（将每个layer v分配到第P(v)个子图），使得总cost最小（如EMA、energy、或combined cost），且满足约束：任何layer在使用前已被计算（P(u)≤P(v) for edge (u,v)），每个子图在G中连通。GA通过维护一个population of genomes（每个genome编码一个partition方案）进行迭代优化：每代执行crossover（混合两个parent方案的特征）、mutation（随机修改方案）、evaluation（计算fitness=-cost）、selection（锦标赛选择最优进入下一代）。Cocco为该问题定制了4种mutation操作：modify-node（单节点重分配）、split-subgraph（拆分子图）、merge-subgraph（合并子图）、mutation-DSE（修改memory配置）。

从编译框架角度拆解：
Cocco的GA for graph partition在编译框架中的流程：
1. **Genome encoding**：每个genome = (partition array P[|V|], memory_config M)。P[v]为layer v的子图编号，按topo序约束随机初始化（确保P(parent)≤P(child)）。
2. **Crossover**（图9b）：从两个parent（Dad, Mom）生成offspring。按topo序遍历所有layer，每个layer随机选一parent继承其子图。若该子图中包含已assign的layer，则创建新子图（排除已assign部分）或与已assign的现有子图合并。
3. **Mutation**（图9c-e）：四种操作——(a) modify-node: 随机节点v从P(v)改到P'(v)（已有或新子图）；(b) split-subgraph: 将随机子图拆为≥2个子图；(c) merge-subgraph: 合并随机两个子图；(d) mutation-DSE: 按正态分布修改memory容量。
4. **Evaluation**（4.4.4）：fitness = -Cost。若子图超过buffer约束，evaluation阶段自动执行split-subgraph（in-situ tuning），保证genome validity并提升样本效率。
5. **Selection**（4.4.5）：锦标赛选择——每代随机选少量genome进行锦标赛，fitness最优者进入下一代。
6. **Time complexity**：相比枚举法的O(2^2N)（对不规则网络），GA通过种群基采样在合理时间内收敛。Cocco在400K样本后即找到枚举法验证的最优解（对简单网络），对大规模网络（Transformer/GPT/RandWire）枚举法无法完成而Cocco始终提供更优解。

术语一般如何实现？如何使用？
GA for partition的实现要点：(1) 编码设计需保证crossover/mutation后genome的validity（topo序约束、子图连通性）；(2) evaluation需快速（Cocco用analytical simulator而非cycle-accurate模拟）；(3) population size和generation number需平衡exploration vs exploitation。其他GA-based DNN优化工作包括GAMMA[30]（GA for HW mapping）、MAGMA[31]（GA for multi-DNN multi-core mapping）、DiGamma[32]（domain-aware GA for HW-mapping co-optimization）。GA输出的partition方案用于编译器的codegen阶段生成subgraph执行指令。

术语解释
effcc（Efficient C Compiler）是Efficient Computer公司开发的商业编译器，将C程序编译为Monaco SDA的DFG和bitstream。基于MLIR构建，支持完整C语言，包含自动并行化和NUPEA-aware PnR优化。

术语是什么？
effcc（Efficient C Compiler）是面向Monaco SDA的编译器工具链。用户通过`__effcc_rip`装饰器标记要offload到Monaco的C函数。effcc基于MLIR编译框架，将C代码lowering为标准MLIR dialects（如scf），然后执行SDA特定的lowering pass：steer insertion（将控制依赖转换为数据流steer指令）、memory ordering（保证有序数据流的访存正确性），最终生成Monaco dataflow指令的custom dialect DFG。effcc还包含自动并行化pass——将循环提升为scf parallel loop，按并行度复制，迭代增加并行度直到PnR失败（资源用尽），从而最大化fabric利用率。effcc是一款通用编译器，支持完整C语言，在2025年已通过Efficient Computer官网提供下载。

从编译框架角度拆解：
effcc的编译流程（从C到bitstream）：
1. **前端**：解析C代码，识别`__effcc_rip`标记的函数。
2. **MLIR Lowering**：将C代码lowering到MLIR标准dialects（scf、arith等）。
3. **SDA-specific Passes**：
   - Steer insertion：在控制流split/join点插入steer指令（φ⁻¹）。
   - Memory ordering：确保有序数据流模型下的访存正确性。
   - 自动并行化：将scf parallel loop按选定并行度复制。
4. **Critical Load Analysis**：分析DFG中loop recurrence上的load指令，标记为critical (a)；inner-loop memory指令标记为次critical (b)。
5. **DFG Generation**：生成Monaco dataflow指令的custom dialect DFG，lower到netlist表示。
6. **NUPEA-aware PnR**：
   - 初始placement：LS指令先放置，按域偏好排序（critical loads → D0，inner-loop → D0/D1，others → farther）。
   - Simulated Annealing优化：cost function中融入NUPEA域偏好和criticality权重。
   - Static timing analysis：评估clock divider，确保data NoC满足timing。
7. **Bitstream Generation**：输出可加载到Monaco的bitstream。

术语一般如何实现？如何使用？
effcc作为Efficient Computer商业产品，以"drop-in replacement for common compilers"定位。支持通过编译器指令（如`#pragma`）显式控制并行度。effcc的PnR使用simulated annealing（类似FPGA领域的Pathfinder和VPR），NUPEA-aware heuristic作为throughput reduction factor乘以critical path delay，形成PnR的objective function。effcc的PnR还利用Monaco 22nm sign-off timing closure数据优化fabric频率。

## Critical Load Identification

术语解释
Critical Load Identification是一种编译时静态分析技术，用于识别数据流图中位于程序关键路径上的load指令。这些load指令的延迟直接影响loop initiation interval（II）和整体吞吐。

术语是什么？
Critical Load Identification（关键Load识别）是NUPEA编译器优化的核心分析pass。目标是找出DFG中那些一旦延迟增加就会拖慢整个程序执行速度的load指令。在本论文中，effcc使用两类启发式：(a) 位于loop recurrence上的load——这些load的数据依赖链控制着循环迭代的启动间隔（II），任何额外延迟都会阻塞所有后续指令的流水；(b) inner-loop memory指令——这些指令虽不在recurrence上，但因执行频率极高，其延迟累积同样影响性能。以spmspv的stream-join为例：iAnz沿recurrence的load（LD nzIdxA[iAnz]）标记为(a)最critical，dot-product的inner-loop loads标记为(b)。这一分析与传统CPU上的static/profiling-guided load分类工作（如识别长前向依赖链loads、分支条件loads、pointer-chasing loads）一脉相承。

从编译框架角度拆解：
effcc中critical load identification的具体过程：
1. **DFG构建完成后**，对每个循环的DFG子图进行分析。
2. **Loop recurrence detection**：在DFG中识别循环依赖回路（recurrence）。使用induction variable分析和def-use chain追踪。标记recurrence上的load指令为(a)。
3. **Inner-loop classification**：标记循环体内（非recurrence）的memory指令为(b)。
4. **Priority encoding**：(a)类load优先级最高，(b)类次之，其余memory指令最低。
5. **Annotation in IR**：在MLIR IR中为load指令附加criticality attribute。
6. **PnR消费**：PnR的cost function读取criticality annotation，优先将(a)类loads放置到D0域，(b)类到D0/D1域。

术语一般如何实现？如何使用？
该技术属于编译时静态分析，不需要profiling或运行时信息。论文指出更复杂的分析是可能的（如考虑更长的依赖链、多级criticality），但简单启发式在评估的13个workload上已足够有效。在传统CPU领域，类似的分析被用于data prefetching和load scheduling优化，但在SDA的PnR上下文中是首次应用。该技术与NUMA的数据放置形成了鲜明对比：编译器识别关键load远比识别关键data容易（后者需要复杂的alias analysis和inter-procedural analysis）。

## NUPEA-aware Place-and-Route (PnR)

术语解释
NUPEA-aware PnR是SDA编译器中融入了NUPEA域感知和load criticality信息的指令放置与路由优化过程。通过修改simulated annealing的cost function和初始placement策略，将关键访存指令优先映射到快速NUPEA域。

术语是什么？
NUPEA-aware PnR（NUPEA感知的放置与路由）是effcc编译器中将DFG指令映射到Monaco fabric上的优化过程。传统SDA PnR仅优化PE间通信局部性（将频繁通信的指令放在相邻PE），NUPEA-aware PnR在此基础上额外优化PE-to-memory通信：利用NUPEA域的非均匀延迟特性，将critical loads优先放置到快速域。PnR流程基于simulated annealing（SA）——类似FPGA的Pathfinder和VPR算法。NUPEA感知通过在SA的cost function中加入域偏好项实现：偏好顺序编码为D1.c0 ≺ D0.c2 ≺ D0.c1 ≺ D0.c0（更靠近memory的列优先），LS指令按criticality排序后按此偏好分配域。初始placement阶段也专门为NUPEA定制：LS指令先放置（按域偏好），然后其余指令按breadth-first顺序通过def-use遍历放置。

从编译框架角度拆解：
NUPEA-aware PnR的完整流程：
1. **输入**：DFG的netlist表示 + Monaco fabric topology（含NUPEA域信息、PE类型、NoC routing资源）。
2. **初始Placement（NUPEA-aware seeding）**：
   - LS指令首先放置，顺序：critical loads (a) → inner-loop loads (b) → other loads。
   - 每个LS指令按域偏好顺序尝试放置：D0.c0 → D0.c1 → D0.c2 → D1.c0 → ...。
   - 算术指令随后按breadth-first traversal（通过def-use链）放置。
3. **Simulated Annealing优化**：
   - Cost function = critical_path_delay × throughput_reduction_factor。
   - Throughput reduction factor编码NUPEA域偏好：D0域load指令factor=1.0，D1域factor>1.0，D3域factor最大。
   - SA随机swap PE assignment，接受降低cost的move，概率接受增加cost的move（温度控制）。
4. **Routing**：确定data NoC中每条数据流边的物理路径。
5. **Static Timing Analysis**：评估最差路径延迟 → 确定最小clock divider。
6. **输出**：每个PE的指令分配 + Data NoC路由配置 + clock divider → bitstream。

术语一般如何实现？如何使用？
SA-based PnR是SDA和FPGA领域的经典方法。effcc的创新在于将NUPEA域偏好融入SA cost function——具体做法是将域偏好作为throughput reduction factor（乘法因子）乘以critical path delay，形成整体的throughput estimate作为objective function。这一设计使得PnR在优化PE间通信延迟和PE-to-memory延迟之间自动权衡。论文评估显示，NUPEA-aware PnR（effcc完整方案）相比Domain-Unaware baseline平均提升25%性能，其中域感知贡献16%，criticality信息贡献额外的9%。

- 10-NUPEA- Optimizing Critical Loads on Spatial Dataflow Architectures via Non-Uniform Processing-Element Access

## DRX Compiler

术语解释
DRX Compiler是将高级数据重构kernel描述编译为DRX ISA指令流的编译器工具，针对数据重构workload的已知形状多维数组特征，执行tiling优化、依赖放宽和RE lane数据划分。

术语是什么？
DRX Compiler是DRX硬件配套的编译器工具链。其输入为两个文件：(1) 数据重构kernel的高级表示（类似C/OpenCL kernel描述，指定多维数组的维度、操作序列和数据类型），(2) DRX硬件架构配置文件（定义RE lane数、scratchpad大小等）。编译器首先将kernel映射到中间表示（IR）——描述多维循环嵌套和逐元素操作序列的DAG；然后基于硬件配置和多维数组维度进行tiling优化（将大数组分解为适配scratchpad容量的tile）并放宽相邻操作间的依赖关系（允许pipeline并行）；最后从优化后的IR生成DRX ISA四类指令（Loop/Compute/Off-chip Memory/Synchronization）。DRX Compiler的设计灵感来自TVM、AutoTVM等ML编译器框架的自定义后端代码生成方法。

从编译框架角度拆解：
DRX Compiler的编译流程（以Sound Detection的mel spectrogram kernel为例）：
1. **输入**：高级kernel描述——对(8192,768) FFT频谱矩阵执行Pow→Add→Mul→Div→Log10→Cast变换 + 硬件配置(128 RE lanes, 64KB scratchpad)。
2. **IR映射**：将kernel操作映射到IR——多维循环（dim0:8192, dim1:768）+ 操作节点序列（Pow→Add→Mul→Div→Log10→Cast）+ 数据依赖边。
3. **Tiling优化**：根据64KB scratchpad容量和(8192,768)数据大小计算tile尺寸，将大数组分解为适配scratchpad的tile（如128×128 tile = 16K elements per bank）。
4. **依赖放宽**：识别相邻操作间的数据依赖，在tile边界处插入同步点但允许tile内部的操作流水化（如Mul的结果可直接forward到Div而不写回scratchpad）。
5. **RE lane划分**：将每个tile内的数据元素按data-parallel方式分配给128个RE lane——各RE处理tile的不同数据子集，DRX compiler确保划分后无lane间数据依赖（纯element-wise操作的典型特征）。
6. **ISA生成**：输出Loop指令（dim0 Base=0/Stride=768/Iter=8192, dim1 Base=0/Stride=1/Iter=768）、Off-chip Memory指令（预取配置：src=DDR4_FFT_output, tile=128×128, dst=spad[0]）、Compute指令序列（POW src1=spad[0] dst=spad[1]; ADD src1=spad[1] src2=const_eps dst=spad[1]; ...）、Synchronization指令（Start/Done标记）。

术语一般如何实现？如何使用？
DRX Compiler是自研编译器，论文未说明是否开源或基于特定编译基础设施（如MLIR/LLVM）。其设计与TVM的自定义后端代码生成流程相似：前端解析高级kernel描述 → IR优化（tiling + dependency relaxation） → 后端代码生成（DRX ISA emission）。Compiler与DRX硬件是co-designed关系——ISA指令格式直接反映compiler的代码生成pattern，compiler的tiling策略直接反映DRX scratchpad的容量约束。这种co-design使编译器的优化空间与硬件能力精确匹配，避免了通用编译器为不匹配硬件生成低效代码的问题。

## Graph Scheduling (DNN Memory Optimization)

术语解释
Graph Scheduling（图调度）是DNN内存优化的一类核心技术，通过操控计算图中tensor的生命周期——即调度每个operator/tensor何时计算（compute）、evict（驱逐）、重新计算（recompute）、offload（卸载到外部存储）和reload（从外部存储加载回来）——来降低同时驻留在内存中的tensor峰值数量，而不改变tensor的形状。

术语是什么？
Graph Scheduling是MAGIS论文框架的基础概念。它将DNN内存优化建模为在计算图上对tensor生命周期进行调度的问题，包含三种主要技术：(1) Re-materialization（重物化）：evict一些中间tensor，在后续需要时重新计算（以计算换取内存）；(2) Swapping（交换）：将暂时不用的tensor offload到外部存储（如CPU内存），后续需要时reload回GPU（以数据传输换取内存）；(3) Re-ordering（重排序）：在不改变operator集合的前提下，寻找最优的operator执行拓扑排序，使得同时活跃的tensor数量最少（不引入额外开销，仅改变执行顺序）。MAGIS的重要见解是：Re-materialization和Swapping在本质上都是改变图的结构的操作——re-materialization插入重新计算的operator/edge，swapping插入Store/Load operator——因此可以将它们表述为图变换规则，从而实现图变换与图调度的统一优化。

从编译框架角度拆解：
在MAGIS中，Graph Scheduling被分解为两个阶段：
1. **图变换阶段**（M-Rules）：通过四条Scheduling-based Rules（Re-materialization Rule、De-re-materialization Rule、Swapping Rule、De-swapping Rule）将re-materialization和swapping表述为子图替换操作。例如Swapping Rule在operator A与其user B之间插入Store(A)和Load(B)节点，将memory-performance trade-off移入图变换的搜索空间。
2. **重排序阶段**（M-Optimizer增量调度）：对变换后的图仅执行不影响总延迟的re-ordering——即寻找最优的operator拓扑排序。由于Store和Load operator的执行顺序可能影响异步数据交换的延迟隐藏效果，re-ordering策略是：将Store尽可能早放置（尽早开始swap-out），将Load尽可能晚放置（刚好在consumer需要前完成swap-in，隐藏PCIe传输延迟）。

传统上Graph Scheduling被独立应用（如Checkmate用IP求解、DTR用heuristic动态决策、POFO用DP组合re-mat和swapping），MAGIS首次将其分解为图变换统一框架。

术语一般如何实现？如何使用？
经典Graph Scheduling实现：(1) Checkmate [24]使用Integer Programming (IP)对静态图离线求解最优re-materialization方案；(2) DTR [27]使用heuristic策略在eager模式下动态决定evict/recompute哪些tensor——基于tensor的size、计算耗时和未来是否会被再次使用；(3) POFO [5]使用Dynamic Programming (DP)对线性链式结构组合优化re-materialization和swapping；(4) XLA [46]使用贪心算法在HLO-level预先规划re-materialization（近线性trade-off但在tight memory limit下性能差）；(5) Serenity [3]使用DP对re-ordering寻优。

在MAGIS中，Graph Scheduling通过M-Rules + 增量调度实现。代码生成后端将最终调度转为PyTorch Python代码：按调度顺序写PyTorch op调用，Store用torch.cuda.Stream异步memcpy(device→host)，Load用异步memcpy(host→device)。MAGIS的simulator（含operator性能cache）评估调度质量：按调度顺序模拟执行operator，跟踪每个时间步的活跃tensor集合（Ai={vj | Sj ≤ i ≤ Fj}，其中Sj=operator完成时间，Fj=max(child完成时间)），计算峰值内存Mpeak=max_i Σ|u| for u in Ai。

## Graph Transformation (DNN Computation Graph)

术语解释
Graph Transformation（图变换）是一类通过等价变换改变计算图结构而不改变语义的优化技术。在DNN领域，图变换起源于编译器领域的superoptimization [34]，通过逐步替换子图（每一步mutate一个子图）来优化图的性能或内存。与Graph Scheduling不同，图变换可以改变tensor的形状和graph structure。

术语是什么？
MAGIS对Graph Transformation进行了系统分类，将其分为三种类型：
1. **Aggregation Transformation (A-Trans)**：将多个小operator聚合为一个更大的operator（如将Q/K/V projection三个独立的MatMul融合为单个Batch MatMul），以提升硬件利用率（更大的矩阵乘法更好地利用GPU Tensor Core），代价是临时增加内存占用（因为融合后的大operator中间结果更大）。这是TASO [25]等框架主要使用的变换类型。
2. **Interim Transformation (I-Trans)**：基于代数等价（如结合律、分配律）对图进行变换，本身不直接优化性能或内存，但它为A-Trans和F-Trans提供新的机会（通过改变图结构使子图匹配变换规则的前置条件）。例如Add的重排（A+B+C → 先A+B再+C vs 先B+C再+A）可能使新的A-Trans规则可匹配。
3. **Fission Transformation (F-Trans)**：A-Trans的对偶变换——将一个大operator沿某个维度分裂为n个更小的operator，各split part顺序执行，中间tensor及时释放以降低峰值内存（降至约1/n），代价是更小的operator shape导致硬件利用率下降（GPU的CUDA core/Tensor Core利用率降低）。

MAGIS的贡献之一是将Graph Transformation从仅关注延迟优化扩展到内存优化（通过F-Trans），并将三种变换统一在M-Rules框架中与Graph Scheduling协同优化。

从编译框架角度拆解：
MAGIS中Graph Transformation的统一框架（M-Rules + M-Optimizer）：
1. **F-Trans规则**（F-Tree Mutation Rules）：
   - Enabling Rule：将F-Tree中n=1的叶子节点启用为n>1（如n=2），表示沿维度D将子图S分裂为2个split parts
   - Lifting Rule：将已启用的子节点disable，改为启用其父节点（更大的子图范围，更大的memory impact）
   - Disabling Rule：禁用一个已启用的无子节点
   - Mutating Rule：将已启用节点的n增加到下一个可整除维度长度的值
2. **TASO规则**：A-Trans和I-Trans的规则集，来自TASO [25]，用于优化延迟——通过聚合小op为大局提升GPU硬件利用率
3. **Scheduling-based规则**：Re-mat/Swapping共计4条规则，将图调度分解为图变换

所有规则被M-Optimizer的贪心搜索算法（Algorithm 3）统一探索：优先队列按BetterThan排序（先比memory，若都满足约束则比latency），每次pop optimal M-State并应用M-Rules生成新M-State，通过增量调度+simulator评估，满足松弛条件（BetterThan(μ', μbest, 1.1)）则push回队列继续搜索。

术语一般如何实现？如何使用？
图变换的经典实现包括：TASO [25]通过程序合成自动生成变换规则并验证等价性；MetaFlow [26]使用backtracking search；TenSAT [62]使用equality saturation [60]进行穷举搜索；PET [56]提出部分等价变换（partial equivalence）；Unity [54]将分布式并行优化集成到图变换中。

MAGIS的实现使用rustworkx [52]（Python高性能图库）作为图数据结构，M-Rules通过子图模式匹配（sub-graph pattern-matching）实现——当变换规则的pattern匹配到图中的子图时，执行pattern → replacement的替换。F-Trans通过F-Tree解耦：F-Tree节点记录(S, D, n)但不实际变换图（避免复杂度爆炸），代码生成阶段按n值展开split parts。图哈希基于Weisfeiler-Lehman Test [48]去重，有效过滤已评估的图结构。

## Fission Transformation (F-Trans)

术语解释
Fission Transformation（F-Trans，分裂变换）是MAGIS提出的新图变换类型，是Aggregation Transformation (A-Trans)的对偶（dual）。通过将一个凸子图沿某个维度D分裂为n个顺序执行的split parts，每个split part的中间tensor shape缩小为原来的约1/n，执行后及时释放，从而大幅降低峰值内存占用，代价是每个split part的operator shape更小导致硬件利用率降低。

术语是什么？
F-Trans在MAGIS中被形式化定义为三元组f = (S, D, n)，其中：
- S ⊆ V(G)：被分裂的凸子图（需满足convexity约束：S的输入节点不会被S外被S输出支配的节点消费，确保无冗余计算）
- D：D-Graph（Dimension Graph），表示沿哪个维度进行分裂。D中每个节点⟨v,i⟩代表tensor v的第i维
- n：分裂数量，子图被分裂为n个isomorphic split parts

约束条件：(1) G[S]弱连通；(2) G[S]凸（convex）；(3) ∀v∈S, 精确存在一个i使得⟨v,i⟩∈V(D)（每个被分裂的operator沿精确一个维度分裂）；(4) ∀(u,v)∈E(G[S]), ∃i,j使得(⟨u,i⟩,⟨v,j⟩)∈E(D)（被分裂的子图内部的数据依赖关系也要沿同一维度传播）。

F-Trans的memory effect由公式(2)量化：M0 - Mf = Σ(1-1/n)|v| for v∈H∩S - Σ|u| for u∈I\H，其中H为memory hot-spots集合、I为S的输入节点。第一项是分裂减少的hot-spot内存，第二项是S的输入节点必须常驻内存的额外开销。要使F-Trans有效，需要选择S包含更多hot-spots而I尽可能小。

从编译框架角度拆解：
MAGIS中F-Trans的搜索与应用：
1. **D-Graph分析**：对计算图G，构建D-Graph D(G)——将每个tensor的每个空间维度和reduce-axis建模为节点。例如在Transformer的self-attention中，batch-dim、head-dim、sequence-dim分别形成D-Graph的连通分量。
2. **F-Tree Construction**（Algorithm 1）：对每个D-Graph连通分量D：
   - 提取D对应的子图G'及其dominator tree T
   - 定位memory hot-spots H
   - 对G'中每个节点v计算：heat(v)=Σ|w| for w∈H∩T.des(v)（v支配的hot-spot总size），score(v)=(1-1/n)heat(v)-Σ|u| for u∈inputs\H（估算F-Trans净收益）
   - 按max-level L将score归一化区间分段，选择不同区间的dominator节点生成Fission candidates
3. **F-Tree Mutation**：通过Enabling Rule启用选定节点（n从1改为实际值），通过Mutating Rule增加n（如从2改为4），通过Lifting Rule将F-Trans范围扩大到父节点（更大的子图，更大的memory impact但更大的latency overhead）
4. **Execution**：不实际变换图，代码生成时按需展开split parts为for循环顺序执行。

术语一般如何实现？如何使用？
F-Trans在MAGIS中通过F-Tree解耦：F-Tree记录(S, D, n)信息，避免F-Trans后图复杂度指数增长。执行时，对于每个split part：
- 若u∈inps(S)且⟨u,i⟩∈V(D)（i>0），则u被slice为n份（每part一份）
- 若u∈inps(S)且⟨u,i⟩∉V(D)（如weight tensor），则u被共享给所有n个part
- 若v∈outs(S)且⟨v,i⟩∈V(D)（i>0），则v由n个part的输出concat合并
- 若v∈outs(S)且⟨v,i⟩∈V(D)（i<0，reduce-axis），则v由n个part的输出reduce合并（如add）
Split parts顺序执行：part 1的所有operator完成后释放其所有中间tensor → part 2执行 → ... → part n执行 → 合并输出。

## Dimension Graph (D-Graph)

术语解释
Dimension Graph（D-Graph，维度图）是MAGIS提出的图结构，用于识别计算图中可进行Fission Transformation的"维度"。D-Graph将tensor的每个空间维度（spatial-axis）和reduce-axis建模为图中的节点，通过追踪维度在operator间的传播关系来发现跨多个operator的"图级维度"。

术语是什么？
给定计算图G，其D-Graph D = D(G)的构造规则：
- **节点**：对每个v∈V(G)，其输出tensor有sv个维度，其计算中有rv个reduce-axis。为每个维度i=-rv,...,-2,-1,1,2,...,sv创建节点⟨v,i⟩∈V(D)（正数表示空间维度，负数表示reduce-axis）
- **边**：对每条数据依赖边(u,v)∈E(G)，若u的第i维和v的第j维对应同一个spatial-axis，则添加边(⟨u,i⟩, ⟨v,j⟩)∈E(D)；若u的第i维对应v的计算中的第j个reduce-axis，则添加边(⟨u,i⟩, ⟨v,-j⟩)∈E(D)

以MatMul c[m,n]=Σk a[m,k]×b[k,n]为例：⟨a,1⟩→⟨c,1⟩（m维空间对应），⟨a,2⟩→⟨c,-1⟩（k维是reduce-axis），⟨b,1⟩→⟨c,-1⟩（k维是reduce-axis），⟨b,2⟩→⟨c,2⟩（n维空间对应）。

D-Graph的连通分量自然地识别出了跨多个operator的"图级维度"——例如在Transformer self-attention中，batch-dim的D-Graph连通分量包含所有沿batch-dim传播的tensor，head-dim的连通分量包含Q/K/V projection及其下游等。

从编译框架角度拆解：
MAGIS使用D-Graph进行F-Trans搜索（Algorithm 1）：
1. 对D(G)的每个连通分量D（每个分量对应一个图级维度）：
   - 提取D中所有维度节点对应的tensor节点，形成子图G'⊆G（即受到该维度影响的operator集合）
   - 构建G'的dominator tree T
   - 计算G'中各节点的score并选择dominator节点生成Fission candidates
2. 不同连通分量提供不同的F-Trans维度选项。例如：
   - batch-dim连通分量 → 沿batch分裂（将batch=64分为2个batch=32的split parts）
   - head-dim连通分量 → 沿head分裂（将num_heads=12分为2个heads=6的split parts）
   - sequence-dim连通分量 → 沿序列长度分裂

D-Graph的约束：当前不考虑带sliding-window的spatial-axis（如3×3 convolution的height/width轴），论文注明将在未来工作中改进。

术语一般如何实现？如何使用？
D-Graph在MAGIS中的实现利用rustworkx有向图结构：每个节点存储(tensor_id, dim_index)对，边存储维度传播关系。D-Graph构造算法遍历G的每条边，根据operator类型推导维度对应关系。对于常见operator（MatMul, Add, Conv2d, BatchNorm等），维度映射规则如表（类似编译器的shape inference）。D-Graph分析是M-Analyzer的第一步，后续基于D-Graph连通分量构建F-Tree。

## Incremental Graph Scheduling

术语解释
Incremental Graph Scheduling（增量图调度）是MAGIS提出的调度优化技术，用于在每次图变换后高效生成新调度。核心思想是：基于前序调度结果和当前变换影响的范围，仅对受影响的子图部分进行重调度（而非对整个变换后的图从头调度），通过识别调度中的"划分点"将问题分解为独立子问题，从而大幅降低调度开销。

术语是什么？
MAGIS的增量调度算法（Algorithm 2）包含三个关键步骤：
1. **GetRescheduleInterval**：在原始调度ψ_old中定位需要重调度的operator区间。利用narrow waist (NW)值——nw(v)=|V(G)|-|anc(v)|-|des(v)|-1，即与v无关的节点数量——作为启发式寻找自然划分点。从变换影响的算子区间[minIS, maxIS]出发，使用ExtendBound向两侧扩展，直到遇到NW值足够大的节点（NW<4且单调下降）或扩展达到上限（20步），确保重调度区间的边界是"图的腰部"（narrow waist点），使得边界两侧的调度问题几乎独立。
2. **GraphPartition**：将需重调度的新图子图S_new进一步分割为独立可调度的部分。使用nw(v)≤1的节点作为划分点——当nw(v)=0时，调度问题在v处可完全分为两个独立子问题（v的所有祖先必须在v之前、所有后代必须在v之后），当nw(v)=1时问题接近可分。
3. **DpSchedule + Merge**：对各独立部分用DP算法（基于Serenity [3]的动态规划算法，枚举所有合法拓扑排序）求解最优调度，然后将局部调度merge回完整调度。

术语一般如何实现？如何使用？
增量调度在MAGIS中直接解决了"每次图变换后全量重调度太慢"的问题。论文评估显示，相比full scheduling（每次变换后对整个图从头DP调度），增量调度实现了4-30×的加速（geomean 10×），同时在94/100个测试用例中达到与full scheduling相同级别的最优性。GetRescheduleInterval的NW值启发式中的常数（20, 10, 4）是经验超参数，在实际DNN图上表现良好。

在更广泛的编译框架中，增量调度的思想可类比于增量编译（incremental compilation）——只重新编译被修改的source file及其依赖，而非全量rebuild。DpSchedule基于Serenity [3]的动态规划算法：给定子图S，枚举其所有满足数据依赖的合法拓扑排序，计算每种排序的峰值内存，选最优者——由于子图被GraphPartition分割得足够小，DP的指数复杂度可控。

## Operator Classification for Layout Optimization (四象限算子分类)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
SmartMem提出的DNN算子四象限分类体系，沿两个维度对计算图中的每个operator进行分类：(1) **输入布局依赖性（Input Layout Dependency）**——算子的计算性能是否依赖输入张量的数据布局（内存排列方式）；(2) **输出布局可定制性（Output Layout Customizability）**——算子的输出张量布局是否可以灵活选择（取决于算子实现中计算顺序的选择）。两个维度交叉产生四个象限：**ILD-Variable（输入布局依赖+输出可变）**——如Conv、MatMul，性能依赖输入布局（需良好空间局部性做复用），但输出布局可定制；**ILD-Fixed（输入布局依赖+输出固定）**——如Reshape、Transpose，性能依赖输入布局（涉及实际数据重排），但输出布局由定义确定；**ILI-Variable（输入布局独立+输出可变）**——如Add、Unary，逐元素操作使性能与输入布局无关（只需遍历即可），但输出布局可定制；**ILI-Fixed（输入布局独立+输出固定）**——如Slice、Gather，简单选择和复制使性能与输入布局无关，但输出布局由定义固定。该分类体系的形式化定义为：ILD-Variable算子数学表示为B_{σ(i),σ(j),...,σ(n)} = O^{ILD-Variable}_{π(i),π(j),...,π(m)}(A_{i,j,...,m})，其中π表示输入访问的维度排列灵活性，σ表示输出生成的维度排列灵活性。与TVM的三类分类（Layout agnostic/Lightly-layout sensitive/Heavily-layout sensitive）和DNNFusion的五类分类（One-to-One/One-to-Many/Many-to-Many/Reorganize/Shuffle）不同，SmartMem的分类专门为layout transformation消除而设计，将output layout的可定制性作为独立维度。

在SmartMem编译流程中，Operator Classification是基石——它决定了后续Layout Transformation Elimination的决策矩阵。具体流程：(1) 遍历原始计算图G中每个operator节点；(2) 解析每个op的数学语义——Conv需要沿spatial dim做滑动窗口乘加→输入layout影响cache命中率→ILD；MatMul需要沿k维做reduction→输入layout影响SIMD效率→ILD；LayerNorm的归一化操作沿特定dim→输出由定义确定→ILD-Fixed；Add逐元素操作→遍历即可→ILI；Slice简单选择→ILI-Fixed；(3) 将分类结果标注在计算图边上。分类结果驱动Table 5和Table 6的决策矩阵——例如ILD-Variable+ILD-Fixed边→Eliminate 2nd（消除固定输出op），ILI-Variable+ILD-Fixed边→Eliminate 2nd，ILD-Fixed+ILD-Fixed边→Eliminate both。所有被消除的fixed-output op（Reshape/Transpose）通过Index Comprehension替换为fused kernel内的索引计算。布局搜索仅发生在涉及ILD-Variable的operator pair上——因为只有ILD-Variable的output layout可以定制，这是search space的剪枝关键。

## Layout Transformation Elimination (LTE, 布局转换消除)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Layout Transformation Elimination (LTE)是SmartMem的核心编译优化，目标是在DNN计算图中消除显式和隐式的layout transformation操作（主要是Reshape和Transpose），而非将其作为独立kernel执行。LTE基于operator四象限分类的决策矩阵（Table 5），对计算图中每条producer-consumer边决定四种优化动作之一：(1) **Keep both**——两个ILD-Variable operator都保留，需搜索两者的最优输入输出layout；(2) **Try fuse**——尝试将两个operator融合为一个fused kernel（基于DNNFusion的融合规则判断合法性和盈利性）；(3) **Eliminate 2nd/1st**——消除consumer或producer中的ILD-Fixed或ILI-Fixed operator，保留Variable output的operator；(4) **Eliminate both**——两个Fixed output operator都被消除，通过Index Comprehension将它们的layout变换逻辑内联到后续compute kernel中。LTE的效果体现在operator数量的显著减少——SmartMem在18个模型上将operator数量减少21%-65%，相比DNNFusion额外提供1.1×-1.7×的fusion rate提升（因为消除layout op后暴露了更多compute-compute fusion机会），Transformer/Hybrid模型受益最大（layout transformation更多）。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
LTE在SmartMem编译流程中的运行：(1) 输入：已标注operator type的计算图G；(2) 遍历G中所有producer-consumer边，构建(u, v, type_u, type_v)四元组；(3) 查Table 5决策矩阵——例如Conv(ILD-Var)+Reshape(ILD-Fixed)→Eliminate 2nd（Reshape被消除），Reshape(ILD-Fixed)+Transpose(ILD-Fixed)→Eliminate both（两个layout op都消除）；(4) 对被消除的operator调用Index Comprehension模块——分析被消除op的索引变换链（identity/split/merge三类依赖），生成fused kernel内部的简化索引表达式（通过Strength Reduction去除冗余取模/除法运算）；(5) 更新计算图——将消除op的producer和consumer直接连接，fused kernel的索引计算内联到consumer的data loading/store阶段；(6) 迭代直到所有可消除的layout op被处理。以Swin中[2,256,4]→Reshape→[16,8,4,4]→Transpose→[16,4,8,4]→MatMul为例，LTE消除Reshape和Transpose后，MatMul kernel直接从Conv输出以优化后的layout读取数据（kernel内部按简化后的索引公式计算数据访问位置），省去两次global memory round-trip的Reshape/Transpose kernel。

术语一般如何实现？如何使用？
LTE的实现依赖三个关键模块：(1) 准确的operator分类——需要正确判断每个op的input layout dependency和output layout customizability，分类错误会导致非法消除（破坏计算语义）；(2) 可靠的fusion legality checking——使用DNNFusion的合法性规则（如injective mapping、same iteration space等）确保消除后的fused kernel计算语义等价于原始op序列；(3) Index Comprehension——将被消除op的索引变换关系（基于静态shape信息分析identity/split/merge依赖）转化为简化的索引计算公式。LTE也可独立于SmartMem框架集成到其他编译器（如TVM、TorchInductor）中——如论文Table 9所示的桌面GPU实验，在TorchInductor上实现了LTE+Layout Selecting并获得1.11×-1.23×加速。LTE对有大量Reshape/Transpose的模型（如Swin-like local attention Transformer、Stable Diffusion）效果最显著。

## Index Comprehension (索引理解) & Strength Reduction on Index Computation

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Index Comprehension是SmartMem实现Layout Transformation Elimination的关键技术，用于将被消除的Reshape/Transpose等layout op替换为fused kernel内部的索引计算。核心思想：连续的Reshape和Transpose操作本质上定义了一个多维索引之间的映射函数，可以离线（编译期）分析并简化为closed-form索引表达式，嵌入fused compute kernel的data loading/store阶段。具体包括两部分：(1) **Index Dependency Analysis**——基于计算图的静态shape信息，分析连续layout op之间的索引依赖关系，分为三种类型：Identity（索引直接传递，i'=i）、Split（一个输入索引分解为多个输出索引，如i'=i/8, j'=i%8）、Merge（多个输入索引合并为一个输出索引，如i'=i*8+j）；(2) **Strength Reduction**——对索引表达式中的取模（%）和除法（//）运算进行代数简化。由于GPU上取模和除法指令代价昂贵，strength reduction可大幅降低fused kernel的索引计算开销。典型简化规则：i%Ca%Cb → i%Cb（当Ca%Cb≡0）；(i*Ca+Cb)%Cc → 若Ca%Cc≡0则简化为Cb%Cc；(i/Ca)/Cb → i/(Ca*Cb)（当Ca和Cb为常量）。这些简化规则在连续Reshape/Transpose引起的维度拆分和合并中频繁触发。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
以SmartMem Figure 3的[2,256,4]→Reshape→[16,8,4,4]→Transpose→[16,4,8,4]为例的Index Comprehension流程：(1) 分析Reshape的索引映射——输入索引(i,j,k)对应线性位置pos=i*(256*4)+j*4+k，输出索引(i',j',k',l')应满足pos=i'*(8*4*4)+j'*(4*4)+k'*4+l'。但论文使用更高效的split/merge依赖分析：Reshape将输入的第二维(256)拆分为两个维度(8,4,4)，产生split依赖关系——i'=i*8+j/(4*8)（merge），j'=j%4（split），k'=j%(4*8)//4（split）；(2) 分析Transpose的索引映射——交换dims 1和2，产生identity依赖——i''=i', j''=k', k''=j', l''=l'；(3) 组合两段依赖得到从input[i,j,k]到output[i'',j'',k'',l'']的完整索引映射函数；(4) Strength Reduction——分析组合后的表达式，识别并消除冗余取模/除法，例如j'%(4*8)//4中的%(4*8)在后续使用时被//4抵消；(5) 将简化后的索引计算嵌入fused kernel——在MatMul的数据load loop中，用简化表达式直接计算从Conv output读取数据的2.5D texture坐标，无需执行独立的Reshape和Transpose kernel。

术语一般如何实现？如何使用？
Index Comprehension的实现需要静态shape信息——所有维度大小必须在编译期已知（DNN推理满足此条件）。实现步骤：(1) 构建被消除layout op链的索引传递图（Index Dependency Graph），节点为各中间tensor的维度index变量，边标注变换类型（identity/split/merge）和参数；(2) 对每个consumer所需的索引变量，回溯到源头（producer的维度index），沿途组合变换函数；(3) 对组合后的表达式树应用代数简化规则（constant folding、common subexpression elimination、modular arithmetic simplification）。这项技术与传统编译器的strength reduction和loop invariant code motion共享原理，但专为DNN layout transformation的数学特征（维度置换、reshape的模运算关系）优化。Index Comprehension使得fused kernel的索引计算开销远低于执行独立的Reshape/Transpose kernel，是LTE得以产生性能收益的关键。

## Reduction Dimension-Based Layout Selection (基于归约维度的布局选择)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Reduction Dimension-Based Layout Selection是SmartMem的启发式布局选择方法，用于为LTE后保留的ILD-Variable operator选择最优输入/输出tensor layout。核心概念：**Reduction Dimension（归约维度）**——对于给定operator的某个操作数，reduction dimension是数据元素参与聚合（aggregation）的那个维度。例如MatMul C=A·B中，A的reduction dimension是k，B的reduction dimension也是k（沿k维做dot product累加）；Conv的reduction dimension包括输入通道维和空间kernel维（沿这些维做加权求和）。启发式规则：对于每条producer-consumer边，**强制producer生成consumer的reduction dimension所偏好的数据layout**——具体来说，将tensor沿reduction dimension在内存中连续存储，使consumer可以stride=1的顺序访问方式沿reduction dim做SIMD load和reduction操作。选择优化producer的写layout（子优化输出）而非consumer的读layout的insight来自microbenchmark实验：优化读性能 vs 优化写性能在Conv/MatMul/Activation上分别实现1.7×/1.4×/1.1×的加速——优化写性能的开销低于子优化读数据。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
布局选择过程：(1) **局部选择（per-edge）**——对于每条producer-consumer边（producer为ILD-Variable且已被标注"search layout"），分析consumer的reduction dimension集合。例如MatMul(producer)→Reduce(consumer, reduction dim=D1)，Reduce的reduction dim是D1→强制producer按D1连续存储输出。若producer有多个consumer：(2) **全局选择（multi-consumer）**——收集所有consumer的reduction dimension需求，取前k个进行合并（k为可连续访问的维度数，2.5D memory的k=2）。例如Figure 4中MatMul输出被Reduce(reduction dim=D1)、Reduce'(reduction dim=D1)和FusedConv(reduction dim=D3)同时消费→合并D1和D3两个reduction dim需求，生成优化布局L0（D1和D3均映射到2.5D memory的两维连续方向）。若consumers要求的reduction dim超过k个，SmartMem维护多份不同layout的数据副本（runtime冗余副本通常<3MB以Swin和ViT为例）。

术语一般如何实现？如何使用？
实现步骤：(1) 对每个保留的ILD-Variable operator的每条出边，计算consumer的reduction dimension——使用operator语义信息（如MatMul的k dim、Conv的in_channel和kernel dims、Reduce的specified axis）；(2) 对producer的多个consumer，收集所有reduction dimension需求→按需求频率排序→取前k个生成合并layout→将合并layout映射到memory hierarchy的对应维度（在2.5D memory中映射到width和height两维，在1D buffer中映射到stride配置）；(3) 若consumer数量>k则生成多个layout变体并分配数据副本。这种启发式方法的理论基础是将NP-hard的全局layout选择问题（等价于PBQP）退化为局部贪心+多consumer合并，在可接受的最优性损失下实现高效的布局搜索（无需昂贵的profiling或ILP求解）。

## DNNFusion (DNN算子融合框架)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
DNNFusion是一个先进的DNN算子融合编译框架，发表于PLDI 2021，由William & Mary和Northeastern University等机构开发。DNNFusion的核心创新包括：(1) 将DNN算子按输入到输出映射关系分为五类——One-to-One（逐元素：Add/ReLU）、One-to-Many（扩展：Expand/Gather）、Many-to-Many（计算密集：Conv/GEMM）、Reorganize（形状变换：Reshape/Flatten）、Shuffle（数据重排：Transpose/DepthToSpace）；(2) Extended Computational Graph (ECG)——在计算图中显式标注算子类型和映射属性；(3) 基于数学性质的图重写——利用结合律、分配律、交换律等代数性质，通过45+38+66条重写规则简化计算图；(4) Mapping Type Analysis——将operator pair融合分为Green（总是合法且盈利）、Yellow（需profiling决定）、Red（非法或不盈利）三类；(5) 集成融合计划生成——结合机器无关分析和轻量profiling。DNNFusion在PLDI 2021上展示比现有框架高达8.8×更多的融合机会和9.3×的端到端加速。SmartMem构建在DNNFusion之上——复用DNNFusion的operator融合合法性和盈利性检查规则，在此之上新增了Layout Transformation Elimination（消除而不仅仅是融合layout op）和Reduction Dimension-Based Layout Selection，进一步扩展了融合优化空间。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
DNNFusion编译流程：(1) 输入：ONNX或其他格式的计算图G；(2) 将G转换为Extended Computational Graph (ECG)——对每个operator标注五类type和input-output mapping属性；(3) Mapping Type Analysis——遍历ECG中所有相邻operator pair，根据pair的分类判断融合类型：One-to-One + One-to-One → Green（无依赖冲突），Conv + Reorganize → Yellow（需profiling决定reshape前后的memory access效率），Shuffle + Many-to-Many → Red（transpose改变数据访问模式使融合后的计算效率下降）；(4) 数学性质图重写——应用结合律/分配律/交换律将多个小op重排为可融合格式（如将element-wise ops通过结合律合并到Conv的computation loop中）；(5) Profiling——对Yellow pair执行轻量profiling（测量融合vs非融合的延迟），确定是否融合；(6) 融合计划生成——综合Green/Yellow/Red分析结果生成最终融合图；(7) 代码生成——将每个fused group编译为单个GPU kernel。SmartMem在DNNFusion的step(3)和step(6)之间插入了LTE和Layout Selection，先消除layout op再让DNNFusion进行常规融合——这使得更多原本被layout op隔开的compute-compute pair成为相邻，释放额外融合机会（SmartMem相比DNNFusion额外提供1.1×-1.7× fusion rate提升）。

术语一般如何实现？如何使用？
DNNFusion已在PLDI 2021发表，有美国专利（US 20220413862），但论文中未明确提供开源仓库链接（SmartMem论文中也未给出DNNFusion代码链接）。其设计思想——基于算子分类和映射类型分析的融合策略——已被SmartMem继承和扩展。DNNFusion的输出是与框架无关的融合计划（表示为DAG），可适配多种后端（OpenCL、CUDA、TensorFlow Lite等）。在移动端推理的实际部署中，DNNFusion通常与MNN或自研runtime配合使用——融合计划由框架的高层IR表示，运行时compiler将其lower到具体硬件的kernel实现，配合auto-tuning搜索最优GPU配置参数。

## TAIDL (Tensor Accelerator ISA Definition Language)

术语解释
TAIDL 是第一个面向张量加速器的指令规格定义语言（DSL），允许架构师用高级张量算子（XLA-HLO）描述加速器ISA的语义，并自动生成快速、可扩展的测试预言（功能性仿真器）。

术语是什么？
TAIDL（Tensor Accelerator ISA Definition Language，张量加速器ISA定义语言）是UIUC ACT Compiler团队在MICRO 2025提出的领域特定语言。它解决了两大痛点：(1) 大多数学术张量加速器缺乏良好文档化的ISA规格，导致编译器回填和优化kernel编写无法系统化进行；(2) 手写指令级仿真器（如Gemmini Spike、Intel SDE）针对特定加速器，速度慢（单线程C++）、不可扩展、工程复用成本高。

TAIDL定义由两部分组成：(a) 数据模型：描述加速器的可编程存储单元——tensor buffers（多维数组，支持多维base-type和addressing）和control registers（暴露给程序员的控制状态，如信号量、配置标志、计数器）；(b) 指令语义：描述每条指令的行为，使用5种语句——tb_read（从tensor buffer切片读取）、tb_write（更新tensor buffer切片）、hlo_op（XLA-HLO张量算子）、assign（更新控制寄存器）、assert（指令有效性的前提条件），并增补IF和REPEAT块以支持控制寄存器驱动的分支和循环展开。

TAIDL选XLA-HLO算子作为计算描述语言的理由：(i) XLA-HLO提供120+算子（reshape、transpose、dot_general、reduce、convert、bitcast_convert、reduce_precision等），可精确表达复杂张量操作；(ii) XLA-HLO具有bit-precise语义（遵循IEEE 754标准，详细记录overflow/underflow/rounding行为）；(iii) XLA-HLO由Google OpenXLA社区维护，广泛集成于JAX、TensorFlow、PyTorch；(iv) XLA-HLO在理论上Turing-complete（superset of FLooP），可表达所有可计算函数。

TAIDL已实例化的ISA包括：Google TPUv1（256×256 systolic array，weight-stationary数据流）、Intel AMX（8个tile寄存器16×64×i8，含tdpbusd等混合精度指令）、Gemmini（可重配置数据流systolic array，支持输出静止/权重静止模式，DIM参数可配置）。开源在 https://github.com/act-compiler/taidl（Apache License 2.0）。

从编译框架角度拆解：
TAIDL在编译框架中扮演"ISA语义标准化层"角色：

1. 架构师输入：用TAIDL Python库编写ISA定义（数据模型 + 指令语义）。例如Intel AMX的tdpbusd指令语义用 reshape→transpose→convert→dot_general→add 等XLA-HLO算子描述。
2. 自动生成TAIDL-TO库：TAIDL根据ISA定义自动生成Python测试预言库，提供@kernel装饰器和api类（如amx.api.tdpbusd()）。
3. 内核程序员使用：编写@kernel装饰的Python函数，调用生成的API（类似JAX Pallas、AWS NKI、Triton的Python接口风格）。
4. 变换算法（transform）：kernel.compile()触发变换算法——将ISA指令流转化为XLA-HLO计算图：(i) 控制寄存器和调用属性通过常量折叠在变换时解析；(ii) REPEAT块展开；(iii) IF块根据控制寄存器值选择分支；(iv) tensor buffer读写转为XLA-HLO slice/dynamic_update_slice；(v) hlo_op保持不变直接嵌入。
5. XLA编译：XLA编译器编译XLA-HLO图→多线程CPU可执行文件或GPU kernel（.pb格式）。
6. 仿真执行：kernel.run(input)加载可执行文件+NumPy输入数据，在CPU或GPU上并行执行仿真，输出bit-accurate结果。

该流程的核心价值：将ISA指令序列这种"类似汇编的低级表示"通过XLA-HLO提升为"高级张量IR"，然后利用XLA编译器自动应用算子融合、代数化简、内存tiling、自动向量化和GPU kernel生成等优化，使测试预言比手写C++仿真器快数个数量级。

术语一般如何实现？如何使用？
TAIDL作为一个Python库实现，架构师通过API定义ISA。关键设计原则：(1) 多维tensor buffer的base-type支持（如AMX tile写为16×64×i8而非1024×i8）避免了每条指令语义中大量reshape/地址计算；(2) control registers与tensor buffers分离——前者在仿真时可静态常量折叠，后者需在XLA-HLO图中表示为动态数据流；(3) 支持标量语义（0-D tensor）和bit-vector语义（bitcast_convert）+自定义数据类型（reduce_precision/clamp/custom_call），使TAIDL向后兼容传统ISA定义格式（如Sail、Intel Intrinsics Guide的C伪代码），向前兼容未来的非标准精度格式。TAIDL定义通过Docker环境分发，实验脚本（kick-tires.sh/lite.sh/full.sh）自动化环境搭建和benchmark。TAIDL是ACT Compiler生态系统的一部分（act-compiler.github.io），下游包括ACT Backend Generator（从TAIDL自动生成编译器后端）。TAIDLv2（pre-release）已添加ANTLR4 parser和模块化架构。

## XLA-HLO as ISA Semantic IR

术语解释
XLA-HLO（含其MLIR方言StableHLO）在TAIDL中被创新性地用作张量加速器ISA指令语义的描述语言，将"类汇编的指令行为"表达为高级张量计算图。

术语是什么？
XLA-HLO（XLA High-Level Operations）是Google XLA编译器的中间表示（IR），StableHLO是其MLIR方言，提供120+种张量算子。传统上，XLA-HLO用于表达ML模型的计算图（如MatMul、Conv、ReLU等高级算子）。TAIDL的创新在于将XLA-HLO"降级"使用——把加速器ISA指令的语义（如一条tdpbusd指令的内部行为）也用XLA-HLO算子来精确描述。

这种用法的关键价值：(1) 语义精确性：XLA-HLO每个算子有详尽的bit-precise操作语义文档（遵循IEEE 754，明确overflow/underflow/rounding行为），继承到TAIDL中；(2) 编译器优化复用：用XLA-HLO表达的指令语义可被XLA编译器自动优化（算子融合、代数化简、向量化、GPU并行化），无需手写优化pass；(3) 多后端部署：XLA编译为CPU（LLVM IR）、GPU（PTX/SPIR-V）或TPU的代码，使TAIDL-TO自然支持CPU和GPU仿真；(4) 社区生态：XLA-HLO/StableHLO随OpenXLA快速演进（2025年MHLO正被废弃，全面迁移到StableHLO），支持新兴精度格式（如MXFP8、bf16_6x等）。

TAIDL中常用的XLA-HLO算子类别：(a) 数据布局变换——reshape、transpose、broadcast、slice、dynamic_update_slice；(b) 精确计算——dot_general（广义矩阵乘法，支持用户定义累加精度）、reduce（归约）；(c) 类型转换——convert（类似C++ static_cast）、bitcast_convert（bit级重解释）、reduce_precision（控制精度和舍入模式）、clamp；(d) 控制流——select（条件选择）、while（无界循环）、call/custom_call（调用外部函数）。

从编译框架角度拆解：
XLA-HLO在TAIDL编译流程中作为"语义锚点"：

1. 指令语义描述阶段：架构师在TAIDL中用XLA-HLO算子逐条描述指令行为。例如Intel AMX tdpbusd指令的7个步骤均映射为XLA-HLO算子序列：steps 1-2 tile加载→slice+reshape+bitcast_convert、step 3 src1的zig-zag布局变换→reshape+transpose、step 4 类型提升和矩阵乘法→convert(i8→i32)+dot_general、steps 5-6 累加和类型转换→reshape+add+bitcast_convert、step 7 结果写回→reshape+dynamic_update_slice。
2. 变换阶段：transform()将内核函数中的指令调用展开为XLA-HLO算子——tb_read→slice、tb_write→dynamic_update_slice、hlo_op→保持、assign→常量折叠更新状态、assert→编译时检查。
3. 优化阶段：XLA编译器对生成的XLA-HLO图自动优化。论文breakdown分析显示：实际计算（矩阵乘法）仅占~7%，内存读写（33-60%）和布局变换（16-60%）占主导——这些是XLA优化最有效的部分（融合、向量化、并行化）。
4. 代码生成：优化后的XLA-HLO→LLVM IR（CPU后端）或PTX（GPU后端）→机器码。

术语一般如何实现？如何使用？
XLA-HLO/StableHLO由OpenXLA社区维护（openxla.org），通过jaxlib库分发。StableHLO提供5年向后兼容和2年向前兼容保证，适合作为长期ISA规格文档的基础表示。2025年MHLO正被废弃，新项目应使用StableHLO。操作语义文档在openxla.org/xla/operation_semantics和openxla.org/stablehlo/spec公开。TAIDL利用jaxlib中的XLA编译器完成HLO→可执行文件的编译，无需自行实现编译pass。

## Auto-generated Scalable Test Oracle (TAIDL-TO)

术语解释
TAIDL-TO（TAIDL Test Oracle）是从TAIDL ISA定义自动生成的、可通过XLA编译器编译为CPU多线程或GPU kernel的快速功能性仿真器（测试预言），用于验证针对加速器ISA编写的程序的正确性。

术语是什么？
Test Oracle（测试预言，也称Functional Simulator/功能仿真器）是软件正确性测试中的关键组件——给定输入程序和数据，输出程序的预期正确结果。在硬件开发中，测试预言用于pre-silicon阶段的软件栈验证（编译器后端、优化kernel等），确保芯片tapeout前软件已正确。传统测试预言（如Gemmini Spike、Intel SDE）是手写C++仿真器，每条指令在嵌套循环中逐元素模拟，由通用编译器编译，通常单线程，仿真速度慢（如1024×1024矩阵乘法超60秒）。

TAIDL-TO的创新：(1) 自动生成：给定TAIDL ISA定义，自动生成Python测试预言库。变换算法将ISA指令序列→XLA-HLO计算图→XLA编译→高度并行可执行文件；(2) 可扩展：借助XLA自动并行化，TAIDL-TO在CPU上多线程执行，GPU上以kernel形式执行——TAIDL-TO (CPU)比Gemmini Spike快88-620×（DIM=16-256），TAIDL-TO (GPU)快1200-5600×；(3) 可复目标：变换算法参数化于TAIDL定义，任何TAIDL描述的加速器ISA均可自动生成对应TAIDL-TO；(4) bit-accurate：TAIDL-TO输出与RTL仿真或Intel Sapphire Rapids原生执行bit-accurate一致。

TAIDL-TO附加功能：debug（通过amx.debug()在kernel中插入打印，类似AWS NKI的device_print）、多kernel函数仿真（不同kernel可组合调用，类似Intel SDE的partial emulation）、与host代码集成（host Python代码原生执行，仅加速器指令被仿真）。

从编译框架角度拆解：
TAIDL-TO全流程（以Intel AMX矩阵乘法为例）：
1. 架构师定义TAIDL → 自动生成TAIDL-TO Python库。
2. 程序员写@kernel函数调用api（tilezero/tileloadd/tdpbusd/tilestored等）。
3. kernel.compile() → transform()将指令展开为XLA-HLO图 → XLA编译为.pb可执行文件。
4. kernel.run(A, B) → XLA Runtime加载.pb+NumPy inputs → 多线程CPU或GPU kernel执行 → 返回bit-accurate NumPy结果。

术语一般如何实现？如何使用？
TAIDL-TO基于XLA编译基础设施实现：(1) 变换算法在TAIDL Python库中实现（Figure 16的transform()函数），处理tb_read→slice、tb_write→dynamic_update_slice、hlo_op→保持、REPEAT→展开、IF→选择、assign→状态更新、assert→编译时求值等转换；(2) XLA编译通过jaxlib完成，XLA AOT编译生成protobuf格式可执行文件；(3) CPU vs GPU选择：TAIDL-TO (CPU)适合小张量（如Intel AMX 16×64 tile），TAIDL-TO (GPU)适合大张量（如Google TPU 256×256 systolic array）；(4) 已集成到Exo编译器测试基础设施中，成功仿真端到端I-BERT模型（12层, 768 emb, 512 seq, 0.8s GPU/2.4s CPU vs Gemmini Spike >50min）。

## Synchronization Instructions (for GEMM-SIMD Coordination)

术语解释
Synchronization Instructions是Tandem Processor ISA中的一类特殊指令（4-bit opcode + 4-bit func + 3-bit + 5-bit fields），用于在编译器生成的指令流中标记GEMM unit和Tandem Processor的代码区域边界，协调两者的tile粒度软件流水执行。

术语是什么？
Synchronization instructions的func字段编码为⟨GEMM/SIMD, START/END, EXEC/BUF, X⟩，四种主要用法：
1. **SIMD_START_EXEC**：标记Tandem Processor指令区域的开始。Instruction Dispatch阶段遇到此指令后，将后续非GEMM指令写入Inst.BUF供Tandem Processor执行。
2. **SIMD_END_EXEC**：标记Tandem Processor指令区域结束，Tandem Processor完成后发送Tandem_done信号给Execution FSM。
3. **GEMM_START_EXEC**：标记GEMM指令区域开始，Instruction Dispatch将GEMM配置指令发送到GEMM unit。
4. **BUF release**（func中BUF位置位）：Tandem Processor释放Output BUF ownership，发送OBUF_done信号通知GEMM unit可写入下一tile。

这些指令的关键作用：(1) **指令分发**：Instruction Dispatch unit（FSM的一个状态）解码sync指令以识别block拓扑（GEMM-only / SIMD-only / GEMM-SIMD混合），据此决定后续指令去向；(2) **同步点**：定义GEMM和Tandem Processor之间的handshaking时序——GEMM unit完成tile后才触发Tandem Processor执行（通过GEMM_tile_done信号），Tandem Processor释放OBUF后才允许GEMM写入；(3) **ownership转换**：在BUF位控制下实现Output BUF的fluid ownership管理。

从编译框架角度拆解：
编译器在指令生成阶段自动插入synchronization instructions的流程：
```
// 编译器遇到fused block: [MatMul, Add, GeLU]
// Step 1: 生成指令流骨架
// Step 2: 在GEMM和SIMD区域边界插入sync指令
// Step 3: 在SIMD内部识别Output BUF使用的scope，插入BUF release

// 生成的指令流:
Sync_GEMM_START_EXEC          // GEMM区域开始
GEMM_CONFIG MatMul ...         // GEMM配置指令
Sync_GEMM_START_EXEC_TILE     // 触发GEMM tile执行

Sync_SIMD_START_EXEC           // SIMD区域开始
// 对于使用Output BUF的SIMD指令:
TILE_LD_ST LOAD OBUF, ...     // 从OBUF加载（取得ownership）
ADD OBUF, ...                 // 直接在OBUF数据上操作
Sync_SIMD_END_BUF             // 释放OBUF（BUF release）
// 对于使用Interim BUF的后续处理:
MUL InterimBUF1, ...          // 中间计算
GeLU_decomposition ...        // GeLU分解
DATATYPE_CAST ...             // 类型转换
TILE_LD_ST STORE ...          // 写回DRAM
Sync_SIMD_END_EXEC             // SIMD区域结束
```

术语一般如何实现？如何使用？
Synchronization instructions的设计哲学：(1) **编译时确定性**：所有sync点由编译器静态分析确定——通过解析DNN graph结构识别GEMM/non-GEMM边界和Output BUF依赖范围，无需运行时动态同步；(2) **硬件极简实现**：sync指令在decode阶段由Instruction Dispatch FSM处理——仅涉及状态机的状态转换和两个handshaking信号线（OBUF_done, Tandem_done），无复杂的互锁或缓存一致性协议；(3) **与GPU barrier的区别**：GPU的__syncthreads()或CUDA stream event是运行时动态同步（支持任意线程到达顺序），而Tandem Processor的sync指令是编译时静态确定的流水同步点（确定性的producer-consumer关系）；(4) **与VLIW bundle的对比**：VLIW在指令内编码多发射槽的并行性，而sync指令在指令流中标记处理器级的执行阶段转换。

## Tandem Processor Compiler (ONNX-based Instruction Generation)

术语解释
Tandem Processor Compiler是GeneSys项目中配套的自研编译器，以ONNX模型和Tandem Processor架构配置为输入，经过模板映射、参数替换、tiling优化、依赖松弛和指令lowering，输出32-bit固定指令宽度的可执行二进制。

术语是什么？
编译器工作流程：
1. **ONNX Graph Parsing**：解析ONNX模型的计算图，识别GEMM节点（Conv/MatMul）和非GEMM节点（五大类算子）。
2. **Template Mapping**：将每个ONNX节点映射到预定义操作模板（operation template）。模板是tile-level伪代码，用嵌套循环+LOAD/STORE/ALU语句描述算子的数据访问和计算模式。复杂算子（Softmax、GeLU、Sqrt等）映射为整数基本原语模板（参考gemmlowp和I-BERT的整数近似方法）。
3. **Parameter Substitution**：将模板中的占位参数（tensor维度M/N/K、tile尺寸M1/N1等）替换为ONNX节点中的实际值和架构约束（scratchpad容量128KB、SIMD lanes=32）确定的最优值。
4. **Tiling Optimization**：选择uniform tiling方案——确保GEMM和非GEMM层的输入/输出tile划分一致，实现tile粒度软件流水。关键规则：(a) 不tile GEMM的reduction维度；(b) tile size需足够大覆盖非GEMM算子的完整邻域；(c) tile size需适配scratchpad容量。
5. **Dependency Relaxation**：利用loop fission消除指令间依赖（将独立的compute操作分离为独立循环，增加调度灵活性）；利用loop interchange松弛长依赖链（如MaxPool中沿空间维度的依赖通过交换循环层级松弛）。
6. **Instruction Lowering**：将优化后的模板语句逐一翻译为Tandem Processor ISA指令——LOAD/STORE→TILE_LD_ST配置指令序列，compute→LOOP+ALU指令组合，涉及Output BUF的compute→自动插入synchronization instructions。

从编译框架角度拆解：
编译器处理BERT中Add层的完整lowering示例（两个[M,N]张量相加，tile size M1×N1）：
```
// 优化后的模板（loop fission已应用）:
for m in range(0, M, M1):           // outer tile loop
  for n in range(0, N, N1):         // inner tile loop
    LD_CONFIG_TILE_LOOP_ITER 0, InterimBUF1, 24
    LD_CONFIG_TILE_LOOP_STRIDE LSB, InterimBUF1, 24, 1024
    LD_CONFIG_TILE_LOOP_STRIDE MSB, InterimBUF1, 24, 1024
    LD_START 0, InterimBUF1, 31, 128   // load tile a
    // ... 类似load tile b ...
    SET_ITER 0, 0, 14                  // outer loop: M1 iterations
    SET_INDEX InterimBUF2, 1, InterimBUF1, 0, InterimBUF2, 0
    SET_ITER 1, 0, 32                  // inner loop: N1 iterations
    SET_INDEX InterimBUF2, 2, InterimBUF1, 1, InterimBUF2, 1
    SET_INST 1, 0, 1                   // 1 instruction in loop body
    ADD InterimBUF2, 0, InterimBUF1, 0, InterimBUF2, 0  // core compute
    ST_CONFIG_TILE_LOOP_ITER 0, InterimBUF2, 24
    // ... 类似st stride配置 ...
    ST_START 0, InterimBUF2, 31, 128   // store result
```

术语一般如何实现？如何使用？
编译器的实现特点：(1) 不基于现有编译器框架（如TVM、XLA、MLIR），而是全新设计以匹配Tandem Processor独特的scratchpad-based内存语义和Iterator Table-based寻址模型；(2) 优化策略基于分析式方法（analytical）而非自动调优（auto-tuning）——因为非GEMM算子的循环结构在编译时完全确定，无需运行时profiling；(3) 模板方法实现了"编译一次，多模型复用"——每个非GEMM算子只需定义一次模板，不同模型的不同维度通过参数替换适配；(4) 编译器作为GeneSys多目标编译栈的一部分：同一ONNX模型可根据不同架构参数（SIMD lane数、scratchpad大小）生成不同指令流。

## LayoutLoop (Dataflow-Layout Co-Exploration Framework)

术语解释
LayoutLoop是FEATHER论文在Timeloop [41]基础上增强的dataflow-layout协同搜索框架。它在原有Timeloop的dataflow cost modeling和search功能之上，新增了物理on-chip storage建模、bank conflict评估和layout-aware mapper修改，使得加速器设计者能够对每层神经网络jointly搜索最优的(dataflow, layout) pair。

术语是什么？
LayoutLoop是一个开源DNN加速器mapping搜索和评估框架，构建于state-of-the-art Timeloop框架之上。其核心增强包括：(1) **Physical Storage Modeling**: 将on-chip buffer从Timeloop的"ideal bandwidth"模型升级为(num_line × line_size) 2D逻辑array + "conflict_depth"（每SRAM bank的physical line数）。此抽象将不同物理bank组织（2D/3D、多种grouping）统一为logical 2D array处理，保持通用性。(2) **Bank Conflict Assessment**: 在dataflow evaluation阶段，分析每个cycle的memory access pattern——若NL条lines从NP-ports的bank中被并发访问（NL > NP），引入max(NP/NL, 1) slowdown factor修正latency和energy估计。这解决了Timeloop原有模型中"即使dataflow理论上好，实践中可能因layout导致的bank conflict而表现差"的theory-practice gap（FEATHER量化此gap可达128×）。(3) **Layout-aware Mapper**: 修改Timeloop mapper使dataflow search同时考虑layout。Layout参数化为"(Inter-line dimension order) (Intra-line dimension order interleaved with sizes)"格式。每层执行dataflow-layout co-search：exhaustive search over layout space（基于已知常用layout patterns如HWC_C32, HWC_W32, HWC_C4W8等共7种） + pruned random search（exhaustive + random pruning）over dataflow space，以Energy-Delay-Product (EDP)为victory condition选择最优pair。

从编译框架角度拆解：
LayoutLoop的编译/搜索流程（以ResNet-50某conv layer为例）：
1. **输入**: (a) Architecture design YAML — PE array维度、buffer hierarchy、BIRRD connectivity和energy per operation；(b) Dataflow constraint YAML (mapspace) — 7-loop nest structure和tiling/parallelism factor range；(c) Layout specification — 从预定义layout space中选择（Conv: HWC_C32, HWC_W32, HWC_H32, HWC_C4W8, HWC_C4H8, HWC_W4H8, HWC_C4W4H2；GEMM: MK_K32, MK_M32, MK_M4K8）；(d) Workload YAML — layer dimensions；(e) Mapper config — search algorithm, victory condition (EDP), thread count。
2. **搜索过程**: Mapper枚举/采样dataflow candidates → 对每个candidate: (i) 计算给定layout下的per-cycle memory access pattern；(ii) bank conflict analysis — 检查每cycle各bank被访问的line数，若>2 ports则计算slowdown factor；(iii) energy estimation — 基于architecture定义的各memory level和compute的energy per access/op + stall cycles的static energy；(iv) cycle count estimation — theoretical cycles × slowdown × utilization efficiency。
3. **输出**: Per-layer CSV files — cycle count, pJ/compute, slowdown ratio, utilization。取各层Geomean后normalize到FEATHER baseline生成性能对比图。
4. **Runtime integration**: LayoutLoop的offline搜索结果（每层optimal (dataflow, layout) pair + BIRRD routing config）写入FEATHER的Instruction Buffer，runtime per-layer switch仅需reload配置。

术语一般如何实现？如何使用？
LayoutLoop基于Timeloop (https://github.com/NVlabs/timeloop) 修改，使用C++实现，scons build system。依赖包括libconfig++-dev, libboost-dev, libyaml-cpp-dev, cmake等。提供pre-built Docker image（https://tinyurl.com/layoutloop）。使用流程：`docker run -it feather_layoutloop` → `cd FEATHER/LayoutLoop/configurations` → `make conv_dse` / `make gemm_dse`。DSE runtime约24小时（multi-threaded CPU, ≥32 GB memory）。关键build-time flags: `ENABLE_TRANSPOSE`（激活transpose on-chip reorder建模）、`MEDUSA`（激活line rotation建模），默认假设无on-chip reorder。LayoutLoop可建模多种accelerator architecture——FEATHER（arbitrary layout choice via RIR）、SIGMA（various layout/reorder scenarios）、MTIA-like（transpose）、TPU-like（transpose+row-reorder）、Medusa-like（line rotation）、NVDLA-like/Eyeriss-like（fixed layout）。开源代码位于https://github.com/maeri-project/FEATHER (MIT license)。

## Physical Storage Modeling (in Accelerator Dataflow Evaluation Frameworks)

术语解释
Physical Storage Modeling是指在DNN加速器的dataflow/mapping evaluation framework中对on-chip SRAM buffer进行物理级建模——显式表示bank organization、port constraints和conflict behavior——以替代传统的"ideal bandwidth"抽象。这是LayoutLoop对Timeloop的核心修改。

术语是什么？
Physical Storage Modeling（物理存储建模）是LayoutLoop引入的技术，将DNN加速器的on-chip buffer从无限带宽抽象升级为物理bank-constrained模型。传统加速器mapping框架（如Timeloop、MAESTRO、Interstellar）通常将on-chip buffer建模为提供固定bandwidth（bits/cycle）的理想存储——无论数据在buffer中如何组织，只要总访问量不超过bandwidth即假设无性能损失。Physical Storage Modeling引入了：(1) **2D logical buffer representation**: (num_line × line_size)，其中line_size表示单cycle最大带宽，num_line表示buffer容量；(2) **Bank organization**: buffer物理上由多个SRAM bank实现，每个bank有NP个读写端口（典型NP=2 in TSMC 28nm HPC）；(3) **Conflict depth**: 每个bank包含的line数——同一bank内不同的lines共享该bank的读写端口；(4) **Bank conflict detection**: 分析每个cycle要访问的数据分布在几个lines上，若同一bank内并发访问的line数 > NP，则引入slowdown。模型将不同物理组织（2D bank grouping、3D stacked banks、各种grouping方式）统一抽象为logical 2D array处理，保留通用性而无需处理具体物理拓扑。

从编译框架角度拆解：
在LayoutLoop中的Physical Storage Modeling流程：
1. **Architecture specification**: 输入YAML定义各buffer level的banks数、每bank ports数(NP)、conflict depth。例如StaB定义为AW banks, 每bank 2 read + 2 write ports, conflict_depth=num_lines_per_bank。
2. **Layout-aware access pattern tracing**: 对给定(dataflow, layout) pair和给定workload，trace每个cycle的memory access。对于每个read/write request，确定其所访问数据的(buffer_line_index, bank_id) via layout mapping function。
3. **Per-bank conflict check**: 对每个cycle，aggregate对每个bank的所有并发access → 若某bank的并发access涉及NL条不同lines且NL > NP，标记为此cycle存在conflict。
4. **Slowdown calculation**: 对检测到conflict的cycle，计算slowdown factor = max(NP/NL, 1)。因为NP-ports bank只能同时服务于NP条lines的访问，多余NL-NP条lines的访问必须在额外cycle中完成。
5. **Performance修正**: 原始theoretical cycles × slowdown factor = practical cycles。Practical compute utilization = theoretical utilization × slowdown factor。Energy = 原计算energy + stall cycles的static energy。
6. **EDP修正**: 修正后的EDP用于mapper的victory condition，确保搜索选择实践中真正高效的(dataflow, layout) pair。

术语一般如何实现？如何使用？
Physical Storage Modeling在LayoutLoop中通过增强Timeloop的buffer model实现。在Timeloop架构中，每个buffer level被extended为包含`conflict_depth`和`num_ports`属性；access analysis pass被修改为记录per-bank的concurrent line count；energy model被扩展为包含stall energy contribution。这种建模也可集成到其他accelerator evaluation frameworks中（如SCALE-Sim、STONNE等）。FEATHER论文量化：忽略physical storage modeling可导致选出的"theoretically best" dataflow在实际中表现比"theoretically suboptimal but practically concordant" dataflow差达128×。

## Orojenesis

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。
Orojenesis（源自地质学术语"orogenesis"——造山运动）是一种计算张量算法数据移动下界（data movement lower bound）的方法论，集成在Timeloop加速器建模框架中（https://timeloop.csail.mit.edu/orojenesis）。其核心流程为：(1) 将给定的单Einsum或多Einsum链tensor workload映射到一个极其简化的代理架构Snowcat（单PE、两级存储：buffer + infinite backing store）上；(2) 在Snowcat极小的、无约束mapspace上进行穷举搜索——以4k×4k×4k GEMM为例，Snowcat的mapspace比Eyeriss-like架构小7350×；(3) 对每个mapping，计算其buffer size requirement（各tensor tile sizes之和）和backing store access count（内层tile size × 外层迭代次数）；(4) 持续更新不同buffer size下的最优access count，连接Pareto-optimal点形成ski-slope diagram；(5) 对于多Einsum链，通过FFMT约束intra-layer mappings以支持fusion，并穷举所有链分段策略。Orojenesis的bounds是mapping-independent的——任何mapping在任何真实架构上的数据移动均无法低于此bound。一次Orojenesis run的结果portable到任意张量加速器架构（只要workload不变）。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。
Orojenesis作为Timeloop的扩展，在编译框架层面提供mapping optimization的上限指导。以4k×4k×4k GEMM为例：
- **Step 1**: 输入Einsum B_{m,n}^{M,N} = A_{m,k}^{M,K} W_{k,n}^{K,N} (M=K=N=4096) 作为Timeloop problem specification。
- **Step 2**: 配置Snowcat架构（两级存储：可变capacity Buffer + infinite Backing Store，无PE/network）。
- **Step 3**: 穷举mapspace——遍历所有tile size组合（M=M0×M1, K=K0×K1, N=N0×N1的所有因子分解）和所有loop order permutations。每mapping: buffer_size = M0×K0 + K0×N0 + M0×N0, accesses = M1×K1×K0×N0(A) + K1×N1×K0×N0(W) + M1×N1×M0×N0(B)。
- **Step 4**: Pareto filtering提取ski-slope curve。
- **Step 5** (多Einsum可选): 施加FFMT约束→穷举fused mapping组合→提取fused Pareto curve。
单次Snowcat evaluation仅0.20ms (i7-1185G7 @ 3.0GHz)，90k mappings exhaustive约18秒，比Simba 100-design DSE (10009s) 快556×。

术语一般如何实现？如何使用？
Orojenesis作为Timeloop扩展以C++实现（scons build）。通过Zenodo artifact（DOI: 10.5281/zenodo.10850531）下载，运行install.sh安装。Jupyter notebooks（orojenesis_single.ipynb / orojenesis_multi.ipynb）引导生成论文所有关键图表。自定义workload通过orojenesis_example.ipynb修改。非GUI环境下用`jupyter nbconvert --to script`转换。Architect使用Orojenesis在早期DSE中代替algorithmic-minimum access做roofline分析：输入target workload+die area+technology，获得optimal buffer-to-compute area ratio。

## Fusion Friendly Mapping Template (FFMT)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。
Fusion Friendly Mapping Template (FFMT)是Orojenesis中约束多Einsum链中每个单Einsum的intra-layer mapping的约束集，使得相邻Einsum能通过tile粒度交换数据实现fusion，避免intermediate tensor spill。针对GEMM链定义四种模板：(1) **FFMT-Full**（仅M维度tiling）——最restrictive，存储完整M0行input/output，产生完整final sum；(2) **FFMT-TiledK**（M和K tiling）——允许tile K，产生partial sums；(3) **FFMT-TiledN**（M和N tiling）——允许tile N，产生partial output rows；(4) **FFMT-TiledKN**（全维度tiling）——最relaxed，产生partial sums of partial outputs。Chain中按位置施加：首层（Einsum 0）可用FFMT-TileKN；中间层须FFMT-Full（必须产生完整output row供下层消费）；末层可用FFMT-TiledN（输出写回backing store）。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。
以两个GEMM fusion (32k×4k×16k + 32k×16k×4k)为例：
- **Step 1**: M rank为shared dimension，成为outermost fused loop。K和N独立。
- **Step 2**: Einsum 0施加FFMT-TileKN（输入A直接来自backing store，可任意tile），produce partial sums。
- **Step 3**: Einsum 1在2-Einsum special case中可施加TiledN（与Einsum 0 K2 loop swap避免intermediate reload），produce final sums。
- **Step 4**: Loopback迭代——Einsum 0重复K2(0)×N2(0)×M1次，Einsum 1重复K2(1)×N2(1)×M1次。
- **Step 5**: Fusion mapspace遍历——各Einsum在FFMT约束子空间内穷举，valid fused mappings组成Cartesian product，提取Pareto-optimal fused bounds。
- **链分段**: 长链中FFMT过于restrictive，Orojenesis穷举2^(#Einsums-1)种分段方案构造最优segmented fusion curve。

术语一般如何实现？如何使用？
FFMT在Orojenesis中作为Timeloop mapper约束层实现。四种模板以loop nest notation描述（类似Timeloop mapping constraint语言）。支持fusion的Snowcat变体使用dual-buffer组织（InputOutputBuf + WeightBuf）。FFMT建模的fusion策略在本质上是FlashAttention（TiledN→TiledK）和FLAT（TiledK→TiledN）的通用表达——Orojenesis通过对比不同FFMT模板揭示：16MB buffer下FlashAttention-like比FLAT-like低6× backing store accesses。

## Ski-slope Diagram

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。
Ski-slope diagram（滑雪坡图）是Orojenesis的核心输出可视化——buffer size (X轴) vs backing store accesses (Y轴)的Pareto-optimal bound curve。形状类似滑雪坡：从高access低buffer的左端急剧下降（增加buffer显著减少data movement），到plateau段（继续增加buffer收益递减），最终触及algorithmic-minimum access limit的水平渐近线。描绘三个关键Gap：(1) **Gap 0**: 某buffer capacity下Orojenesis bound与algorithmic-minimum的垂直距离——表示仅靠增加buffer无法消除的access overhead；(2) **Gap 1**: Maximal effectual buffer size与total operand size的水平距离——实现理论最小access所需buffer远小于total operand size（GEMM约33%）；(3) **Slope**: marginal benefit of additional buffer capacity。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。
Ski-slope为compiler/mapper和architect提供三层分析：
- **对Compiler/Mapper**: curve是lower bound——若compiler实测accesses远高于curve，说明mapping sub-optimal或存在excess data movement。
- **对Architect**: 沿X轴标记真实buffer sizes（L1/L2/DRAM），读对应Y值得该level的attainable bound。在16k×1k×1k GEMM上可probe multi-level bounds。
- **验证使用**: GPU上不同LLC size (A2 2MB/A30 24MB/A100 40MB/H100 50MB)散点均在curve上方→bounds有效。A100/H100优化schedule接近curve→CUTLASS tuning near-optimal。
生成流程：穷举mapping→(buffer_size, access_count) pairs→Pareto filtering→piecewise linear interpolation。`GenerateBound()`函数在Orojenesis notebooks中封装此流程。

术语一般如何实现？如何使用？
通过穷举search结果进行Pareto filtering（λ1×buffer_size + λ2×accesses双目标dominance），非Pareto-optimal点discard。剩余点按buffer size升序排列，piecewise linear interpolation连接。同一ski-slope可用于所有Timeloop描述框架内的架构（Spatial Array/GPU Tensor Core/NPU等）——架构specific constraints使实测点落在curve上方但永不低于curve。

## Dual-mode-aware CIM Compilation (CMSwitch)

术语解释
Dual-mode-aware CIM Compilation 是一种面向双模CIM加速器的编译优化范式，由CMSwitch编译器首次实现。其核心思想是将CIM array的compute/memory模式切换（mode switch）作为编译优化空间中的一个维度，而非像传统CIM编译器将所有array静态视为compute-only单元。

术语是什么？
CMSwitch是首个考虑CIM array双模（compute和memory）切换能力的DNN编译器。现代CIM芯片（如DynaPlasia）的每个CIM array可通过修改输入driver信号（GIA/GIAb）在compute mode（执行MAC/bit-series运算）和memory mode（作为scratchpad buffer）之间动态切换。传统CIM编译器（PUMA、OCC、CIM-MLC）将所有array预设为compute mode，导致：(1) 低arithmetic intensity模型（如LLaMA2 AI≈2）的compute array大量闲置等待数据；(2) on-chip buffer不足时activation必须频繁off-chip搬运，增加延迟。

CMSwitch通过三个组件解决：(1) DEHA（Dual-Mode Enhanced Hardware Abstraction）将模式切换能力参数化写入硬件抽象；(2) DACO（Dual-Mode-Aware Compilation Optimization）用DP+MIP两步分治联合优化网络分段和array mode分配；(3) DMO（Dual-mode Meta-Operator）用`CM.switch(TOM/TOC, arrayaddr)`和`parallel{}`表达编译结果。相比于CIM-MLC baseline，CMSwitch平均加速1.31×（最大2.03× on OPT-13B），mode switch overhead仅占3%-5%总执行时间。

从编译框架角度拆解：
CMSwitch的编译流水线（以LLaMA2-7B attention层为例）：
1. **输入**：ONNX格式计算图 + DEHA硬件参数（96 array, 320×320, switch latency=1 cycle, internal_bw=32b/cycle）。
2. **Preprocess**：ONNX graph→flatten为拓扑排序算子序列(O1,...,Om)→CIM不支持的算子greedy拆分为子算子。
3. **DP Network Segmentation**：对i<j组合计算intra-segment latency（通过MIP求解）和inter-segment switch overhead（数据写回+模式切换+权重重载），DP递推`L[j][A]=min(L[i][A']+T_intra(A)+T_inter(A',A))`，backtrack得最优segment计划。
4. **MIP Resource Allocation**（per segment）：变量λ_min/λ_mout/λ_c(i,x,y)表示每个CIM array对每个算子的mode分配。Gurobi求解器在约束（array overlap、operator dependency、resource limit）下最小化`max(LOi)`，LOi ∝ OPOi/min(ComOi·OPcim, (MemOi·Dcim+Dmain)·AIOi)。
5. **DMO输出**：meta-operator流标记mode切换+segment内并行执行。

术语一般如何实现？如何使用？
CMSwitch基于CIM-MLC编译框架实现，使用Gurobi求解器（商用MIP solver）进行资源分配优化。编译过程是offline的一次性操作：编译时间比CIM-MLC长2.8×-6.3×（BERT ~95s, LLaMA2-7B ~169s, VGG16 ~661s），但编译结果可在运行时反复使用。DP+MIP的非枚举求解方法（impossible-case pruning + MIP solver）使编译开销随workload size近似线性增长。编译器输出为通用meta-operator流，可转换为不同CIM芯片的ISA/机器码。论文未明确说明CMSwitch代码是否开源。

## DEHA (Dual-Mode Enhanced Hardware Abstraction)

术语解释
DEHA是CMSwitch编译器中引入的双模增强硬件抽象层，在传统CIM硬件抽象的基础上增加了CIM array的双模切换能力参数，使编译器能感知并优化array的compute/memory模式分配。

术语是什么？
DEHA（Dual-Mode Enhanced Hardware Abstraction）是对CIM-MLC硬件抽象的扩展。传统CIM硬件抽象仅包含array数量、尺寸、带宽等固定参数，默认所有array为compute-only。DEHA新增以下双模相关参数：(1) `#_switch_array`：可切换模式的CIM array总数；(2) `array_size`：每个array的尺寸（如320×320）；(3) `Methodc→m/Methodm→c`：compute-to-memory和memory-to-compute的切换方式（如修改GIA/GIAb输入信号）；(4) `Lc→m/Lm→c`：模式切换延迟（如1 cycle）；(5) `Lfunc`：不同mode下array执行基本操作（compute/read/write）的延迟；(6) `internal_bw/extern_bw`：内部/外部带宽。这些参数使编译器能够量化模式切换的收益和开销，将"该array应该是compute还是memory"作为一个正式优化变量。

从编译框架角度拆解：
DEHA在CMSwitch编译流程中的角色：将双模CIM chip的架构信息抽象为key-value参数表→传递给DACO优化阶段（DP使用switch latency计算inter-segment overhead，MIP使用array数量和size作为resource constraint）→DMO输出时根据`Methodc→m/Methodm→c`生成对应的mode switch控制信号配置。DEHA的设计使得CMSwitch可以适配不同dual-mode CIM芯片（如DynaPlasia和PRIME），只需更改DEHA参数即可。

术语一般如何实现？如何使用？
DEHA以参数配置表的形式存在，用户（或芯片厂商）在部署CMSwitch前填写target CIM芯片的DEHA参数。参数直接影响编译优化结果——例如DynaPlasia的switch latency=1 cycle意味着mode switch overhead极低，DP分割时更倾向于frequent mode switching；而PRIME（ReRAM-based）写延迟更高，DP会更谨慎地选择切换点。论文展示了在DynaPlasia和PRIME两种不同DEHA配置下的编译结果，验证了DEHA的可移植性。

## Network Segmentation with DP + MIP for Dual-mode CIM (DACO)

术语解释
DACO（Dual-Mode-Aware Compilation Optimization）是CMSwitch的核心编译优化算法，采用Dynamic Programming（DP）进行时间维度的网络分段和Mixed-Integer Programming（MIP）进行空间维度的资源分配，两步分治解决dual-mode CIM编译的指数级搜索空间问题。

术语是什么？
DACO将dual-mode CIM编译优化分解为两个嵌套优化阶段：(1) DP Network Segmentation：将DNN的算子序列沿时间维度划分为多个segment，每个segment内的算子同时映射到chip上执行（pipeline），segment间串行。DP的目标是最小化`sum(T_intra(segment)) + sum(T_inter(segment_i, segment_i+1))`，其中T_inter包含三段overhead——数据写回（write-back）、模式切换（mode switch）、权重重载（weight reload）。(2) MIP Resource Allocation：在每个segment内，使用MIP联合优化CIM array的compute/memory模式分配和算子调度。决策变量λ_c/λ_min/λ_mout(i,x,y)表示CIM array(x,y)对算子Oi是compute/input-memory/output-memory。约束包括：每个array只能是一种mode、依赖算子间output memory可复用为input、非依赖算子间array不可共享、总array数不超过Ncim。目标函数：`min max(LOi)`，即最小化segment内最慢算子的延迟（pipeline瓶颈）。

从编译框架角度拆解：
DACO的DP+MIP两步的具体交互（Algorithm 1 in paper）：
1. Flatten ONNX graph为(O1,...,Om)。
2. 对每个i<j，检查Si,j所需最小array数<Ncim→如可行，调用MIP求解`T_intra(A)=MIP(S_i,j)`的optimal resource allocation，否则标记为不可行(T_intra=∞)。
3. 对相邻segment，计算`T_inter(A',A)`三段overhead。
4. DP递推`L[j][A]=min(L[i][A'] + T_intra(A) + T_inter(A',A))`，回溯得最优segmentation。
5. Post-allocation optimization（如weight duplication）进一步优化kernel mapping。

术语一般如何实现？如何使用？
DP使用标准backtracking实现，search space通过impossible-case pruning削减；MIP使用Gurobi求解器求解。DP+MIP的非枚举策略避免了2^m（m=array数量）的指数级枚举。DACO是offline编译优化，编译时间比CIM-MLC长2.8×-6.3×但线性扩展。MIP的目标函数使用simplified latency model（式10）：`LOi ∝ OPOi / min(ComOi·OPcim, (MemOi·Dcim+Dmain)·AIOi)`，其中OPcim为每个compute array提供的MAC数/cycle，Dcim为每个memory array提供的数据/cycle，AIOi为算子的arithmetic intensity。

## DRAM Communication Scheduling Space

术语解释
DRAM Communication Scheduling Space是SoMa[HPCA 2025]首次全面定义和刻画的DNN加速器DRAM通信调度优化空间，由Layer Fusion和Prefetching & Delayed Storing两种优化范式的所有可能组合构成，通过Tensor-centric Notation的6个属性编码刻画，覆盖了从层融合策略、计算粒度到每个DRAM tensor的存取时机和buffer占用生命周期的完整调度决策。

术语是什么？
在DNN加速器中，DRAM通信调度涉及决定"哪些数据何时从DRAM加载到on-chip buffer"和"哪些数据何时从on-chip buffer写回DRAM"。传统方法通常只关注单一维度（如Layer Fusion中哪些层融合），SoMa提出的DRAM Communication Scheduling Space将两个优化范式统一在一个六维空间中：(1) Layer Fusion范式——通过将多个layer的计算融合为fine-grained tile级执行序列，用on-chip buffer传递fmaps避免DRAM往返；(2) Prefetching & Delayed Storing范式——通过调整每个DRAM tensor的存取时机（提前加载权重/ifmaps、延迟写回ofmaps），利用DRAM空闲时段平滑峰值带宽压力。这两个范式存在复杂耦合：两者都通过消耗buffer换取DRAM优化（竞争buffer），且fusion选择改变DRAM tensor的种类/数量（影响prefetching/delayed storing的操作空间）。

从编译框架角度拆解：
SoMa的DRAM Communication Scheduling Space在编译框架中的运作流程：
1. **空间编码**：6个属性分为LFA（Layer-Fusion-related Attributes）和DLSA（DRAM-Load-and-Store-related Attributes）两类。LFA包括Computing Order（layer的粗粒度执行顺序）、Fine-grained Layer-fusion Cut（FLC，将layer序列切分为fine-grained layer-fusion group，组内layer以tile粒度交替计算）、Tiling Number（每个FLG的计算tile数，决定计算粒度）、DRAM Cut（FLC的子集，标记需通过DRAM交互的边界）。DLSA包括DRAM Tensor Order（所有DRAM tensor的存取顺序）和Living Duration（每个DRAM tensor的(Start, End) tile区间，控制buffer分配和load/store时机）。
2. **空间解析**：第一阶段解析LFA——给定四个LFA属性，推导出完整的compute tile序列（COMPUTE row）、所有需DRAM交互的tensor列表和on-chip复用的buffer占用时间线。第二阶段解析DLSA——给定DRAM Tensor Order和Living Duration，推导出实际的DRAM load/store时间线和各tensor在GBUF中的存活周期。
3. **空间大小**：SOTA方法Cocco仅能改变Computing Order和DRAM Cut两个属性（FLC=DRAM Cut，Tiling Number由heuristic固定，DLSA由double-buffer heuristic确定），而SoMa可探索全部6个属性，优化空间远大于Cocco。

术语一般如何实现？如何使用？
DRAM Communication Scheduling Space的实现通过SoMa编译框架完成。具体使用：输入硬件配置（核心数/组织、DRAM带宽、buffer大小）、DNN模型描述、优化目标（Energy^n×Delay^m）→SoMa通过SA探索六维空间→输出最优调度方案的指令序列。该空间不仅适用于CNN（ResNet等），也适用于LLM（GPT-2 Prefill/Decode），但Decode阶段因极低compute density（weight和KV cache加载主导延迟）导致该空间的优化潜力有限。该概念已被SoMa成功应用于商业加速器的编译器产品中。开源实现：https://github.com/SET-Scheduling-Project/SoMa-HPCA2025

涉及论文标题：
- 26-SoMa: Identifying, Exploring, and Understanding the DRAM Communication Scheduling Space for DNN Accelerators

## Tensor-centric Notation (for DRAM Communication Scheduling)

术语解释
Tensor-centric Notation是SoMa提出的一种以tensor为中心的编码系统，用6个属性将任意的DRAM通信调度方案编码为可解析的紧凑表示，并配套对应的解析方法将该编码转化为实际的硬件行为（计算序列、DRAM存取、buffer占用时间线）。

术语是什么？
Tensor-centric Notation将DRAM通信调度方案分解为两类共6个属性。**Layer-Fusion-related Attributes (LFA)**：1) Computing Order——将各layer排为串行序列，表示粗粒度执行顺序（受数据依赖约束：consumer不能在producer之前）；2) Fine-grained Layer-fusion Cut (FLC)——在layer序列中标记cut位置，将序列切分为FLG，同一FLG内的layer以tile粒度交替计算而非逐层完成；3) Tiling Number——每个FLG的计算tile数，将每层沿batch/H/W维度划分为指定份数（优先tile batch维度以避免halo overlap，再均分H/W，不切channel维度以保证fusing后续layer所需的全通道访问）；4) DRAM Cut——FLC的子集，标记需通过DRAM交互的LG边界，跨DRAM Cut的数据依赖必须经DRAM往返。**DRAM-Load-and-Store-related Attributes (DLSA)**：5) DRAM Tensor Order——所有需存取DRAM的tensor的访问顺序；6) Living Duration——每个DRAM tensor的(Start, End) tile对，控制buffer分配区间（tensor从Start到End在buffer中存活）和load/store时机（ifmaps/weights的End固定、Start可调；ofmaps的Start固定、End可调）。

从编译框架角度拆解：
Tensor-centric Notation的解析流程（以五层网络ABCDE为例）：
1. **LFA解析**：给定Computing Order=[A,B,C,E,D]、FLC Set={1,2}、Tiling Numbers={2,1,2}、DRAM Cut Set={2}→切分为三个FLG([A]×2 tiles, [B]×1 tile, [C,E,D]×2 tiles)、两个LG([A,B]和[C,E,D]，因DRAM Cut在位置2)→推导COMPUTE tile序列（A1,A2,B,C1,E1,D1,C2,E2,D2）→确定各tile的DRAM tensor需求（有weights的层→weight load请求；跨DRAM Cut的forward依赖→ifmap load请求；跨DRAM Cut的backward依赖→ofmap store请求；网络输入/输出同理）→确定on-chip复用fmaps的buffer占用区间（不跨DRAM Cut的依赖直接on-chip传递）。
2. **DLSA解析**：给定DRAM Tensor Order（如IA1,WA,IA2,WB,WE,OB,IC1,WD,IC2,OE1,OD1,OE2,OD2）和各tensor的Living Duration→按Order序列在DRAM row上排列各tensor的load/store时间线→按Living Duration在BUFFER row上标记各tensor在GBUF中的存活周期（如WE的Living Duration=(B,D2)，表示从tile B到tile D2在buffer中）。
3. **依赖驱动的时间线合并**：将DRAM row和COMPUTE row按依赖关系合并为完整timeline——DRAM tensor开始条件：前序DRAM tensor完成 且（ifmaps/weights的Start≤当前tile 或 ofmaps的生成tile已结束）；Compute tile开始条件：所有所需数据ready 且 所有End≤当前tile的DRAM tensor已完成。

术语一般如何实现？如何使用？
Tensor-centric Notation在SoMa编译框架中作为SA搜索空间的核心编码方式实现。SA operators直接操作这6个属性（如Change Computing Order、Change Tiling Number×/÷2、Add/Delete FLC、Add/Delete DRAM Cut、Change DRAM Tensor Order、Change Living Duration）。编码的合法性由依赖约束保证（如Computing Order不能违反数据依赖、Living Duration的Start必须在tensor数据的producer tile之后）。该notation已成功描述和统一了包括Cocco、DeFENIS在内的已有方法——这些方法仅覆盖6个属性中的2~3个。

## Two-Stage Simulated Annealing for DNN Scheduling

术语解释
Two-Stage Simulated Annealing是SoMa提出的用于探索DRAM Communication Scheduling Space的层次化搜索策略，将搜索分为LFA Exploration Stage（第一阶段，搜索层融合相关属性）和DLSA Exploration Stage（第二阶段，搜索DRAM存取时序属性），两阶段之间通过Buffer Allocator在最外层迭代分配buffer预算，形成top-down的协同优化结构。

术语是什么？
SoMa的Two-Stage SA采用"先粗后细"的分层搜索策略。第一阶段探索LFA（Computing Order、FLC、Tiling Number、DRAM Cut），DLSA固定为经典double-buffer heuristic；第二阶段在固定LFA最优方案的基础上探索DLSA（DRAM Tensor Order、Living Duration）。两阶段均使用Simulated Annealing算法。Buffer Allocator在最外层控制迭代——每轮执行完整的two-stage search，根据上一轮第一阶段的实际buffer用量调整当前轮第一阶段buffer上限（逐轮缩减a%），直至连续两次的最优cost不超Cost_best时停止。

从编译框架角度拆解：
Two-Stage SA在SoMa编译框架中的完整流程：
1. **Buffer Allocator初始化**：第一轮允许buffer用量≤硬件buffer容量，记录第一阶段最大buffer用量(Buffer_max)和全局最优cost(Cost_best)。
2. **第一阶段(LFA Exploration)**：初始方案为每层独立为一个LG/FLG（无fusion），Tiling Number=最小粒度（core array并行所需的最小tile size），DLSA=double-buffer。SA Operators包括Change Computing Order（随机选layer移至另一合法位置）、Change Tiling Number（随机选FLG乘/除以2）、Add/Delete FLC（增删切分点，合并时Tiling Number按层数比概率继承）、Add/Delete DRAM Cut（增删DRAM Cut，添加的必须在FLC Set中）。Cooling Schedule：Tn = T0×(1+αn/N)^(1-n/N)，总迭代N=100×layer数。接受准则：若新方案cost(c')≥旧cost(c)，以概率p=e^((c-c')/(cTn))接受；否则总是接受。
3. **第二阶段(DLSA Exploration)**：初始方案固定第一阶段的最优LFA编码，SA Operators包括Change DRAM Tensor Order（随机选tensor移至另一合法位置）、Change Living Duration（随机选tensor改Start或End），选tensor的概率正比于其大小。总迭代N=1000×DRAM tensor数。
4. **Buffer Allocator迭代**：第二轮起，第一阶段buffer上限=Buffer_max×(1-a%)（a=10%），超限方案标记为无效；若本轮Cost_temp<Cost_best则更新最优。迭代直至连续两次不改善。
5. **输出**：Buffer Allocator的最优编码→IR Generator→Instruction Generator→指令+报告。

术语一般如何实现？如何使用？
Two-Stage SA的设计理由：(1) LFA的单一变动会显著改变DRAM tensor的种类/数量/大小，使之前优化的DLSA方案可能suboptimal甚至无效——无法在连续变动LFA时同时保留好的DLSA；(2) 两阶段存在强协同效应——第一阶段倾向于最小化fmaps相关的DRAM access（因为weight占DRAM流量大头且依赖约束少、易于在第二阶段优化），更extensive的fusion减少DRAM access使compute时间更纯粹，为第二阶段提供更大优化空间。风险是第一阶段可能占用过多buffer限制第二阶段，由Buffer Allocator缓解。SoMa实验显示第一阶段平均提升1.82×性能（vs Cocco），第二阶段进一步提升1.16×（平均距理论上限仅3.1%）。该two-stage策略已在SoMa的商业加速器编译器中应用。

## LLVM Loop Auto-Vectorization Pass Modification for Non-CPU Targets (PUD)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
MIMDRAM对LLVM loop auto-vectorization pass的修改展示了一种通用技术：将CPU ISA的向量化分析用于非CPU加速器（PUD）。传统LLVM auto-vectorizer使用cost model评估"将scalar循环转换为SIMD指令是否有利"——cost model参数化于目标CPU的SIMD宽度、向量寄存器数量、各SIMD指令延迟等。MIMDRAM的修改：(1) 用max vectorization factor替代cost model——因PUD的"SIMD宽度"可变（1-128 mats × 512 cols）且latency不完全按CPU规则；(2) 删除向量化循环中的load/store指令——因PUD直接操作DRAM中数据，无需explicit memory movement；(3) 在X86 backend新增bbop ISA指令生成逻辑，将LLVM的vector IR指令（如add<1024 x i32>）翻译为bbop指令而非x86 AVX指令。这种修改模式适用于任何以"可变宽度SIMD+内存驻留数据"为特征的加速器。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
MIMDRAM编译流程（LLVM 12.0.0）：
```
C/C++ Source
    ↓ clang front-end
LLVM IR (unoptimized)
    ↓ Pass 1: Code Identification (modified LoopVectorize pass)
LLVM IR with vector instructions (VF=max possible, no load/store)
    ↓ identifying bbop candidates
bbop IR instructions (bbop_add, bbop_mul, etc.)
    ↓ Pass 2: Code Scheduling & Data Mapping
DDG construction → DFS topological traversal → mat label i/j assignment
    ↓ insert bbop_mov for cross-mat dependencies
bbop IR + mat_label metadata + bbop_mov
    ↓ Pass 3: Code Generation (modified X86 backend)
x86 binary with bbop ISA extensions (ML + VF fields)
    ↓ pim_malloc calls inserted for memory allocation
Executable binary
```

Pass 1的具体修改：
- LLVM原函数 `LoopVectorizationCostModel::selectVectorizationFactor()` → 改为返回 `computeMaxVF()`（基于loop trip count和标量操作数数量）
- LLVM原函数 `InnerLoopVectorizer::vectorizeMemoryInstruction()` → 返回null（skip load/store for bbop-marked loops）
- TargetTransformInfo中PUD的"cost"→标量不cost（PUD无法高效执行标量），向量cost=1（鼓励最大向量化）

Pass 2的核心算法（DFS on DDG）：
```
DFS_schedule(node, current_mat_label):
  node.mat_label = current_mat_label
  if node.is_leaf():
    return
  // Left path: same mat, respect dependency
  DFS_schedule(node.left_child, current_mat_label)
  // Right path: new mat, independent subtree
  new_label = allocate_new_mat_label()
  DFS_schedule(node.right_child, new_label)
  // Cross-mat dependency: insert data movement
  if node.type == 'parent_of_cross_mat_children':
    insert bbop_mov(from=new_label, to=current_mat_label)
```

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
实现和使用：
1. **LLVM Pass注册**：三个pass作为动态加载的LLVM plugin（或built-in修改），通过clang的-fplugin或直接修改LLVM源码树。按`opt -load MIMDRAM.so -mimdram-codeid -mimdram-schedule -mimdram-codegen`顺序执行。
2. **对程序员透明**：输入为unmodified C/C++应用。程序员无需添加pragma、调用PUD API或手动管理数据布局。对于使用OpenMP的应用，compiler自动将outermost loop的并行映射为mat-level MIMD。
3. **bbop ISA编码**：bbop指令作为x86 ISA扩展（利用未使用的opcode空间）。每条bbop在x86 binary中的编码包含opcode + ML (mat label) + VF (vectorization factor) 字段。
4. **开源**：https://github.com/CMU-SAFARI/MIMDRAM 包含LLVM修改源码。
5. **与CHOPPER [HPCA 2023]的区别**：CHOPPER需要程序员用bit-slicing compiler语法重写应用，MIMDRAM完全自动。CHOPPER维持全row SIMD（SIMD underutilization问题仍在），MIMDRAM支持可变宽度。
6. **通用性**：此修改模式（cost model替换 + memory instruction剥离 + custom ISA backend）可用于其他以"可变宽度SIMD+内存驻留计算"为特征的加速器（如PNM accelerators, ReRAM PIM）。

## VQ-LLM Automatic Kernel Code Generation Framework

术语解释
VQ-LLM是一个面向Vector Quantization (VQ) augmented LLM inference的自动化高性能fused kernel代码生成框架，采用CUDA template-based设计 + adaptive heuristics，自动为不同的VQ算法配置（vector size, #Entry, Residual）和不同的计算kernel类型（GeMM, GeMV, Attention）生成优化后的融合dequantization-computation CUDA kernel。

术语是什么？
VQ-LLM是一个代码生成框架（code generation framework），解决VQ算法多样性×计算kernel多样性组合空间巨大无法手工优化的挑战。框架输入为VQ配置（vector_size, #Entry, Residual, VQ algorithm type）+ 目标computation类型（GeMM/GeMV/Attention）+ 目标GPU规格，输出为优化后的fused CUDA kernel。核心架构分两层：(1) **Codebook Cache**：按offline profiling得到的codebook entry access frequency，将entries分层放置到GPU memory hierarchy（register/hot, shared memory/medium, global memory/cold），通过reorder-based static mapping实现（无tag array开销），利用GPU resource slack自适应确定各层大小。(2) **Codebook-Based Compute Engine**：采用codebook-centric dataflow（沿codebook switch axis切分并行化task消除重复codebook加载）+ codebook-centric hierarchical fusion（利用intra-warp shuffle API在register中完成数据layout重排，消除shared memory round-trip）。所有优化参数（nreg, nshared, split_factor, nshuffle, thread_mapping）通过adaptive heuristics自动确定。

从编译框架角度拆解：
VQ-LLM的kernel生成流程（Alg. 2: Complete VQ-aware computation template）：

```
// Offline Phase: 参数推导
Input: VQ_config (vector_size, n_entries, residual), compute_op (GeMM/GeMV/Attention), GPU_spec
1. Analyze compute_op:
   - All_axes = compute_op.all_axes          // 所有轴
   - Reduce_axes = compute_op.reduce_axes    // 归约轴
   - Codebook_switch_axes = identify_switch_axes(VQ_config, compute_op)
     // e.g., Attention K cache: switch=H,C; GeMM GPTVQ: switch=M,N
2. Profile codebook entry access frequency:
   - hot_entries = entries with freq > μ+3σ  // → register
   - medium_entries = entries with freq > avg // → shared memory  
   - cold_entries = remaining                 // → global memory
3. Compute resource budgets:
   - Budget_reg = free_regs_preserving_occupancy(compute_op, GPU_spec)
   - Budget_smem = free_smem_preserving_occupancy(compute_op, GPU_spec)
   - nreg = min(hot_entries, Budget_reg / entry_size)
   - nshared = min(hot_entries + medium_entries, (Budget_smem - nreg*entry_size) / entry_size)
4. Compute split_factor:
   - Solve: split_factor × Output_Size = Original_Codebook_Traffic / split_factor
   - → split_factor = sqrt(Original_Codebook_Traffic / Output_Size)
5. Compute nshuffle:
   - layout_src = VQ_config.vector_size      // dequantized data layout
   - layout_dst = compute_op.required_layout // e.g., mma requires layout=2
   - nshuffle = layout_src / layout_dst
6. If nshuffle <= 5 (threshold): enable register-level fusion
   - Thread_Mapping(compute_op.warp_tile, layout_src, layout_dst)
     // Determine mini-warp mapping offline to minimize shuffle count

// Online Phase: Kernel Template Instantiation
7. Codebook_Reorder(codebook, access_frequency_order)  // Offline reorder entries
8. Parallel_For(codebook_switch_axes, split_factor):  // Codebook-centric dataflow
     a. CB = Switch(New_Codebook_Ptr) if per-channel codebook
     b. CB_cached, boundary = Load(CB, Budget)  // Codebook Cache load
     c. For each quantized_index in thread_block's data:
          data = Access(CB_cached, boundary, CB, index)
          // if index < nreg → register; if nreg≤idx<nshared → shared; else → global
     d. If nshuffle <= 5:
          data = Reg_Fusion(data, nshuffle)  // shfl_xor in mini-warp
        Else:
          data = Shared_Fusion(data, layout_src, layout_dst)  // shared memory round-trip
     e. For temporal_iteration on All_axes - codebook_switch_axes:
          partial = compute_op(data, temporal_iteration)  // mma.sync etc.
     f. output = Reduce(partial, Reduce_axes ∩ codebook_switch_axes)  // Global reduction
9. Return optimized CUDA kernel (.cu / PTX)
```

术语一般如何实现？如何使用？
VQ-LLM的实现要点：(1) **Codebook Cache的reorder-based实现**：offline对codebook entries按access frequency降序排序并重编号索引——最频繁的entry index=0，最不频繁的index=max。Runtime dequantization通过简单index比较定位entry所在memory level（index < nreg → register; nreg ≤ index < nshared → shared memory; index ≥ nshared → global memory），无tag array或lookup table。(2) **Adaptive Heuristics**：split_factor通过平衡Traffic_Reduce（=split_factor×Output_Size）和Traffic_Codebook（=Original_Codebook_Traffic/split_factor）使两者相等来最小化总traffic；nshuffle threshold=5来自profiling结果（shared memory access latency ≈ 5× register+shuffle latency）。(3) **Register-Level Fusion via shuffle**：当dequantized data layout与computation layout不匹配时，通过shfl_xor API在mini-warp内交换数据使layout对齐mma指令需求——例如VQ<8,...>+mma layout=2时3次shuffle完成8→2重排。

VQ-LLM目前依赖手工CUDA template编写（非全自动编译器），但adaptive heuristics使同一模板可覆盖多个VQ配置。论文未明确给出独立GitHub repo链接，依赖开源VQ算法实现（AQLM: https://github.com/vahe1994/AQLM）和baseline框架（cutlass, flash-attn, qServe）。

## IANUS Compiler (NPU-PIM异构系统编译器)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
IANUS Compiler是面向NPU-PIM统一内存异构系统的离线编译器，输入为LLM模型（GPT-2/BERT, BF16），输出为ordered commands序列（含MU/VU/PIM/DMA操作指令及dependency信息）。核心功能包括：(1) FC Mapping via Algorithm 1——基于analytical model在compile time选择FC操作映射到MU或PIM；(2) Address Generation for Scratch-pad——定义AM和WM的地址以实现on-chip head splitting/merging（消除data reordering overhead）；(3) Head-wise Weight Partitioning——将Q/K/V weight分布到不同PIM channel供各NPU core独立访问；(4) Intra-layer Parallelism——column-wise weight分割减少weight data movement；(5) Synchronization Point Insertion——在4个位置插入sync commands。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
IANUS Compiler的编译流程（以GPT-2 XL, input=512, output=64为例）：
1. **Parse model graph**：解析GPT-2 decoder block → 识别FC layers（Q/K/V projection, attention output, FFN up/gate/down）、self-attention（QK^T, softmax, SV）、vector ops（layer norm, residual add, GELU）、LM head等操作。
2. **Initial command mapping**：所有FC默认映射到MU（MU_FC commands）。Vector ops→VU commands。Self-attention QK^T/SV→MU commands。DMA commands生成用于weight/activation在off-chip PIM memory和on-chip scratch-pad间的搬运。
3. **Apply Algorithm 1**：遍历ordered commands→对每个MU_FC cmd：估计MU execution time（含column-tiling, pipelined weight+compute, VU prefetching overlap）和PIM execution time（n×per-token PIM time）→若PIM更快则重映射cmd.type = PIM→若为FFN首FC则chain GELU→PIM。
4. **Generate addresses for head splitting/merging**：对于产生Q/K/V的FC命令，compiler按attention head数生成多个命令（每head一条），每条命令指定不同的output address in AM——使得Q/K/V数据在AM中直接按head分离存储（eliminating post-FC data reordering）。对于SV→merge heads的output：compiler确保各head的SV命令产生consecutive AM addresses→自动merge。
5. **Apply head-wise partitioning**：Q/K/V weight按head分解→分配到不同PIM channel memory→各core访问对应channel的weight。Intra-layer parallelism：非attention FC weight column-wise分割到各core→reduction通过sync point协调。
6. **Insert synchronization points**：4个sync点——(a) after multi-head attention（各core需完成各自head的attention）；(b)(c) after两次residual addition；(d) after GELU。
7. **Output ordered commands**：每条命令含(unit_type, operation, input/output addresses, dimensions, dependency_list)→供command scheduler运行时issue。

## GPU Analytical Model (GPU解析性能模型)

术语是什么？
GPU解析性能模型（Analytical Model）是一种基于数学公式和架构参数来预测GPU kernel执行时间（cycles）的性能建模方法，无需cycle-accurate仿真。与cycle-accurate GPU simulator（如GPGPU-Sim, Accel-Sim）相比，解析模型速度通常快数个数量级，并能提供易于理解的性能瓶颈洞察（如CPI Stack），但精度相对较低。GPU解析模型的核心方法从早期的MWP-CWP模型（Hong & Kim, ISCA '09）演进到基于interval analysis的GPUMech（MICRO '14）、MDM（MICRO '20）、GCoM（ISCA '22），以及最近的AMALI（ISCA '25）。在AI时代GPU架构快速迭代的背景下，解析模型因其速度优势在早期架构设计空间探索中尤为重要。

从编译框架角度拆解术语：
GPU解析模型可以看作一个SASS级分析"编译器"——输入为CUDA kernel的SASS指令trace，输出为预测的kernel cycles和CPI stack breakdown。AMALI的完整分析pipeline：SASS Tracer（NVBit插桩收集trace）→ SASS Parser（提取warp profile + K-means选representative warp）→ Cache Simulator（基于memory access trace + 架构参数计算AMAT）→ Interval Analyzer（基于AMAT和data dependency划分warp执行intervals，建模selected/wait/short_scoreboard/long_scoreboard）→ Interval Parser（建模math_pipe_throttle/lg_throttle/mio_throttle三类资源争用stall）→ KLL comp（建模kernel launch延迟）→ 输出C = SI + SI_P + KLL + ID。该流程与编译器的分析pass结构高度类似：trace收集→IR构建→分析→优化→输出。

术语一般如何实现？如何使用？
解析模型通过micro-benchmark预先测量目标GPU的架构参数（FU延迟、内存层次延迟、cache配置、tensor core throughput等）。运行时，用户运行目标kernel，解析模型收集SASS trace，应用interval analysis + 争用模型 + 启动延迟模型计算预测cycles。典型用途：(1) GPU架构设计空间探索——改变参数（如tensor core FMAs/cycle）预测性能变化；(2) kernel性能瓶颈定位——生成CPI stack显示各类stall占比；(3) 编译优化指导——评估不同代码生成策略的性能。代表性实现：GPUMech（开源学术项目）、GCoM、MDM、AMALI（代码未公开）。

## Interval Analysis in GPU Performance Modeling (GPU性能建模中的间隔分析)

术语是什么？
Interval Analysis（间隔分析）最初由Karkhanis & Smith（ISCA '04）为CPU性能建模提出，将线程执行划分为由stall事件边界的离散intervals。GPU领域的interval analysis由GPUMech（MICRO '14）首次适配，后续被GCoM和AMALI继承和增强。其核心思想是：遍历warp的指令序列，根据指令间数据依赖和内存访问延迟插入stall intervals，将warp执行建模为"执行intervals + 等待intervals"的交替序列。

从编译框架角度拆解术语：
在AMALI中，Interval Analyzer接收Cache Simulator的AMAT输出和SASS Parser的warp profile，逐指令遍历warp的SASS指令序列：(1) 对每条指令标记end cycle：CUDA core指令使用固定延迟（架构参数），tensor core指令使用throughput-based延迟（L_TC = FMA_count/TP_dt）；(2) global memory指令的end cycle = current cycle + AMAT；(3) shared memory指令延迟取决于load/store（如A100 ld 23 cycles, st 19 cycles）；(4) 当指令的操作数未就绪时（end cycle > current cycle），插入对应类型的stall interval（wait/compute data, short_scoreboard/shared memory, long_scoreboard/global memory）；(5) 输出四类stall的cycle计数：S_selected = I_warp/IssueRate（无hazard的执行），其余按公式S_i = Σ S_k · P(i, flag[k])分类。

术语一般如何实现？如何使用？
实现方式：需要一个SASS指令仿真器（不仿真timing，仅跟踪寄存器/内存依赖）配合一个cache simulator提供AMAT。遍历指令序列时维护每个寄存器/memory location的"就绪时间"，当后续指令等待时累加对应stall。在实际使用中，interval analysis产生的SI是kernel cycle预测的基础部分（C = SI + SI_P + KLL + ID），提供data dependency和memory access导致的stall breakdown。

- 3-AMALI- An Analytical Model for Accurately Modeling LLM Inference on Modern GPUs.pdf

## CPI Stack (Cycles Per Instruction Stack)

术语是什么？
CPI Stack（每指令周期堆栈）是GPU性能分析中按stall类型分解kernel执行时间的一种可视化/分析方法。它将kernel的总cycles按各类stall事件（如selected、wait、math_pipe_throttle、imc_miss、no_instructions等）拆分为百分比堆叠柱，使架构师能一目了然地识别性能瓶颈。在NCU中对应Scheduler Statistics的各类warp stall reason；在AMALI中对应SI + SI_P + KLL + ID各项及其子项。

从编译框架角度拆解术语：
AMALI生成CPI stack的流程：(1) 通过interval analysis + parser计算总cycle C及各stall分量；(2) 将各分量除以总cycle得到百分比；(3) 按NCU的stall分类（selected/wait/short_scoreboard/long_scoreboard/math_pipe_throttle/lg_throttle/mio_throttle/no_instructions/imc_miss）归类；(4) 输出为堆叠柱状图。在AMALI中，CPI stack不仅是最终输出（与硬件NCU测量值对比验证），也是debug工具——例如发现GCoM在VELE kernel中严重低估imc_miss和no_instructions（占CPI stack ~70%），从而驱动KLL组件的设计。

术语一般如何实现？如何使用？
CPI stack通常通过解析模型（如AMALI）计算或通过NCU的硬件性能计数器直接测量。使用场景：(1) 识别kernel的dominant bottleneck（memory-bound vs compute-bound vs cache-miss-bound）；(2) 验证解析模型的各项stall预测是否与硬件测量一致；(3) 指导优化——如发现math_pipe_throttle占比高则考虑优化GEMM tile size或使用更高tensor core utilization的kernel实现。

## SASS (Streaming Assembly)

术语是什么？
SASS（Streaming Assembly）是NVIDIA GPU的最低级指令集架构（ISA），由CUDA compiler（nvcc的ptxas阶段）将PTX（Parallel Thread Execution）中间表示编译生成。SASS是GPU硬件实际执行的指令格式，包含具体的寄存器分配、指令调度和架构特定的编码。不同GPU架构（Volta/Turing/Ampere/Hopper）有不同的SASS指令集。与PTX相比，SASS级分析精度更高，因为PTX仍然可能被编译器进一步优化变换。

从编译框架角度拆解术语：
在AMALI的编译框架pipeline中，SASS是核心分析对象：(1) SASS Tracer基于NVBit收集kernel运行时的SASS指令trace（而非PTX trace）；(2) SASS Parser提取指令名（如HMMA, IMAD, LDG, STG）、modifiers（如.16816.F32）、寄存器操作数、内存地址等信息；(3) AMALI特别关注SASS中的HMMA指令modifiers（tensor size和data type），因为modifier直接影响tensor core的throughput（CPI），这是PTX级分析无法精确捕获的；(4) AMALI还分析SASS中隐式constant memory访问（如c[0x0][0x28]寻址），这只能从SASS中识别。

术语一般如何实现？如何使用？
SASS可以通过cuobjdump（NVIDIA CUDA Binary Utilities）从编译后的cubin二进制中反汇编获得：`cuobjdump -sass <cubin_file>`。在编程层面，SASS可以用cuAssembler（https://github.com/cloudcores/CuAssembler）进行修改和重新组装，用于验证指令行为（如AMALI使用cuAssembler修改HMMA modifier验证性能差异）。SASS级分析的工具链：编译 → cubin → SASS反汇编/插桩 → trace收集 → 分析。

## PatternIR（基于循环打孔的稀疏模式中间表示）

术语是什么？
PatternIR是Fractal提出的面向结构化稀疏模式的循环级中间表示（IR），核心理念是将稀疏模式表示为对稠密tiling循环的"穿孔"（perforation）。PatternIR以嵌套循环（loop nest）作为一等公民，通过定义稀疏循环（记作`I_length^nnz`，length为迭代长度、nnz为非零元素数）来表示结构化稀疏。若nnz==length则为dense loop。每个稀疏循环附有index vector记录保留元素在原始稠密空间的索引。通过组合多层稀疏循环并定义依赖关系（如J依赖I表示J的值与I的后继循环连续存储），可表示从EW到Hybrid的任意结构化稀疏模式。

从编译框架角度拆解术语：
PatternIR定义了五种变换原语生成和变换稀疏模式：
1. **Split(loop, size_a, size_b)**：将循环分为两个子循环创建细粒度tiling，不改变存储。
2. **Reorder(loop_a, loop_b)**：交换两相邻循环依赖关系，显式转置存储和计算顺序。
3. **Join(loop_a, loop_b)**：合并两循环为一个更大循环，展平存储和索引，是Split的逆操作。
4. **Perforate(loop, nnz)**：核心原语引入稀疏性，标注循环nnz并创建稀疏index数组。穿孔外层循环跳过整列（coarse-grain），穿孔内层循环跳过单个元素（fine-grain）。
5. **Condense(loop, buffer)**：算子优化原语，将沿某循环的非连续稀疏值预聚集为稠密连续存储。

示例：PatternIR `I0^4 I1_3^8 J^1024` 表示I维分为I0(4)和I1(8)，I1穿孔为3nnz，J为dense。`I0_2^4 J0^4 I1^4 J1^4` 表示Hybrid多级稀疏（I0穿孔子J0穿孔子）。

术语一般如何实现？如何使用？
Fractal的PatternIR→SparseTIR→TVM pipeline：Dense TVM算子→GenTilingSpace生成tiling组合→GenPerforationSpace搜索穿孔nnz→PatternIR对象（含sparse loops DAG + index vectors）→Parsing转SparseTIR Sparse Axis/Buffer/Iteration→TVM TensorIR→MetaSchedule自动调优→CUDA kernel。PatternIR作为thin abstraction layer，正确性验证（loop nest边界、buffer边界）由底层SparseTIR/TVM primitives保证。

- 40-Fractal- Joint Multi-Level Sparse Pattern Tuning of Accuracy and Performance for DNN Pruning.pdf

## Fractal-ELL（多级稀疏ELL存储格式）

术语是什么？
Fractal-ELL是Fractal提出的多级结构化稀疏存储格式，在传统ELL（Ellpack）格式基础上扩展为支持多层稀疏索引。传统ELL：每行保留固定nnz元素值和列索引。Fractal-ELL：对于含n级稀疏循环的PatternIR，value矩阵大小为所有循环nnz之积，每级稀疏循环附加独立index vector，大小由所有前驱循环nnz累积决定。通过分层索引链高效存储任意复杂的多级Hybrid稀疏模式。

从编译框架角度拆解术语：
以4级Hybrid模式PatternIR `I0^4 J0_2^4 I1_4^2 J1^4`（Fig.6）为例：
1. 外层I0-J0（BW级）：J0_indices size=4×2=8（每I0行保留2个J0列）
2. 内层I1-J1（VW级）：每个保留(I0,J0) block内I1 2选2（dense）× J1 dense 4
3. Value tensor size=4×2×2×4=64（所有nnz之积）
4. I1_indices size=4×2×2=16（I0 nnz×J0 nnz×I1 length）
SparseTIR映射：`I0=dense_axis(4)`, `J0=sparse_axis(I0,(4,2),(J0_indices))`, `I1=sparse_axis(J0,(2,2),(I1_indices))`, `J1=dense_axis(4)`。

术语一般如何实现？如何使用？
在Fractal Codegen中，PatternIR的Sparse Axis声明自动决定Fractal-ELL布局：value数组按最内层dense axes连续存储；index vectors按pre-order遍历sparse loop DAG附加在value后。执行：outer sparse loop index选择保留block→加载对应inner index vector→inner sparse loop index选择block内保留vector→dense compute。相比CSR链式间接访问，内存连续性好，适合GPU SIMD并行。

## Condense Primitive（Condense原语）

术语是什么？
Condense是Fractal在PatternIR上定义的稀疏算子优化变换原语，将沿某稀疏循环的非连续值预聚集（pre-gather）为稠密连续存储，消除稀疏计算中的随机内存访问。以稀疏GEMM C[i,j]=ΣA[i,k]B[k,j]为例，A沿K轴稀疏使B的访问变成非连续的随机读（依赖A的index vector），引发GPU线程间bank conflict。Condense(B, K)将B中对应A.index位置的元素提前复制到稠密临时buffer B_Condensed，后续计算转为连续访问。

从编译框架角度拆解术语：
Condense在SparseTIR→TVM TensorIR层实现：
1. Condense前：`C[i0,i1,j] += A[i0,k_iter,i1] * B[k,j]`——B[k,j]依赖k随机索引
2. Condense(B,K)后：`B_Condensed[i0,k_iter,j]=B[k,j]`（预聚集）→ `C[i0,i1,j]+=A[i0,k_iter,i1]*B_Condensed[i0,k_iter,j]`（连续访问）
3. 额外存储：B_Condensed size=I0.length×K.nnz×J.length，按outer loop分组因不同outer iteration有不同K indices。

术语一般如何实现？如何使用？
Condense的数据搬运可融入GPU local memory caching过程（B_Condensed直接写到shared memory），实际无额外global memory traffic。Fractal中是否应用Condense及在哪个axis上由Operator Tuner（MetaSchedule）在搜索中决策——Condense的position和size作为schedule参数纳入自动搜索空间。

## SparseTIR（稀疏张量中间表示编译器）

术语是什么？
SparseTIR是面向稀疏张量计算的TVM扩展编译器（开源：https://github.com/microsoft/SparseTIR），提供可组合抽象来表示和优化稀疏算子。核心抽象：(1) Sparse Axis——带依赖关系的稀疏轴声明，指定父轴、长度和nnz；(2) Sparse Buffer——按Sparse Axes定义的稀疏存储；(3) Sparse Iteration——`with sp_iter([axes], pattern, storage)`定义稀疏循环，pattern中S=sparse（间接寻址）、R=reduction（归约）、D=dense。SparseTIR将稀疏算子转为TVM TensorIR的loop-based程序，复用TVM优化和codegen。

从编译框架角度拆解术语：
Fractal PatternIR→SparseTIR译码流水线：
1. PatternIR sparse loop→Sparse Axis：`I1=sparse_axis(I0,(8,3),(indices0))`表示I1依赖I0，长度8选3
2. Sparse Buffer：`A=match_sparse_buffer(a,[I0,I1,K0,K1,K2,I2],float16)`按Fractal-ELL布局
3. Sparse Iteration：`with sp_iter([I0,I1,K0,K1,K2,I2,J], SSRRRSS, spam) as [i0,i1,k0,k1,k2,i2,j]`生成嵌套循环，sparse axes使用index vector间接寻址
4. →TVM TensorIR：`C[i0*256+indices0[i0*3+i1]+i2,j] += A[...] * B[...]`
5. →MetaSchedule优化→CUDA kernel

术语一般如何实现？如何使用？
SparseTIR作为TVM扩展包使用：定义Sparse Axes hierarchy→声明Sparse Buffers→编写with sp_iter循环体→调用tir.SparseLowering pass→获得标准TensorIR→复用TVM build/codegen。原生支持COO/CSR/ELL等格式但不含结构化稀疏模式概念——Fractal在SparseTIR之上增加PatternIR Parsing模块，将tiling+perforation schedule自动转为SparseTIR表达式。

## Chunk-sharing Graph

术语解释
Chunk-sharing graph 是 llm.npu 提出的编译层优化技术，用于解决 mobile NPU 仅支持静态 shape 计算图与 LLM 变长 prompt 推理之间的矛盾。通过将变长 prompt 拆分为固定大小 chunk，并将 LLM 计算图分解为 static（size-independent，跨 chunk 共享）和 dynamic（size-dependent，按需创建）两类子图，消除重复的图构建和优化开销。

术语是什么？
Chunk-sharing graph 基于 decoder-only LLM 的因果性（第 i 个 token 仅依赖前 i-1 个 token），将长 prompt 模型推理等价转化为多个短 chunk 的顺序推理。每个 chunk 长度固定（如 256 tokens），预构建 NPU 计算子图。核心创新在于将每个 Transformer block 分解为两类算子：(1) Static operators（绿色）：Linear（QKV/O/FFN）、LayerNorm 等——其 tensor shape 仅依赖 chunk length，不依赖 chunk 在序列中的位置，因此同一子图可被所有 chunk 复用；(2) Dynamic operators（红色）：Attention 等——其 Q/K/V 维度随 chunk 索引变化（如第一个 chunk 的 K=32×hds，最后一个 chunk 的 K=1024×hds），需为不同 chunk 构建独立子图。

从编译框架角度拆解：
Chunk-sharing graph 的编译流程：
```
# 准备阶段（离线）
def build_chunk_sharing_graph(model, chunk_length=256, max_context=4096):
    subgraphs = []
    for layer in model.layers:
        # Static 子图：shape 不依赖 chunk index
        static_qkv = build_npu_graph(QKVLinear, input_shape=[chunk_length, hidden])
        static_ffn  = build_npu_graph(FFN, input_shape=[chunk_length, hidden])
        static_o    = build_npu_graph(OLinear, input_shape=[chunk_length, hidden])
        
        # Dynamic 子图：shape 依赖 chunk index
        for chunk_idx in range(max_context // chunk_length):
            kv_len = min((chunk_idx + 1) * chunk_length, max_context)
            dynamic_attn[chunk_idx] = build_npu_graph(
                Attention, Q_shape=[chunk_length, hidden],
                          K_shape=[kv_len, hidden],
                          V_shape=[kv_len, hidden]
            )
        
        subgraphs.append({
            'layer': layer, 
            'shared': [static_qkv, static_o, static_ffn],
            'per_chunk': dynamic_attn
        })
    
    # 优化：对 linear 层的 tensor shape 进行等效变换
    # 如 2048×2048 → 32×32×2048（更符合 NPU CNN 架构偏好，加速 1.62×）
    optimize_tensor_shapes(subgraphs, npu_profile_data)
    
    return subgraphs

# 运行时（在线）
def prefill_with_chunk_sharing(prompt_tokens, model, subgraphs, chunk_length=256):
    chunks = split_into_chunks(prompt_tokens, chunk_length)  # 1024→4 chunks
    kv_cache = {}
    for i, chunk in enumerate(chunks):
        # 所有 chunk 复用同一份 static 子图
        x = execute_static_subgraphs(subgraphs.shared, chunk)
        # 根据 chunk index 选择对应的 dynamic Attention 子图
        x = execute_dynamic_subgraph(subgraphs.per_chunk[i], x, kv_cache)
        kv_cache.update(x.k, x.v)
    return x
```
关键优化效果：(1) 图构建从 O(N_chunks) 次 → 1 次 static + N 次小 dynamic；(2) 内存占用：120/144 子图可共享 → 减少 75%（7.2GB）；(3) Tensor shape 重排：利用 NPU 对"均衡"tensor（等 height/width）的偏好，在 profiling 阶段选择最优等效 shape。

术语一般如何实现？如何使用？
Chunk-sharing graph 在 llm.npu 中基于 MLLM 和 QNN 框架实现。准备阶段使用 QNN 的图构建 API 创建每个子图（Graph::create + Graph::optimize），运行时使用 QNN 的 Graph::execute 执行。chunk_length 的选择通过 profiling 确定（Xiaomi 14 最优 chunk_length=256）。该技术适用条件：目标硬件仅支持静态 shape 推理图，且推理任务具有因果依赖关系（decoder-only transformer）。兼容任何解码引擎（llm.npu 当前使用 MLLM CPU 后端解码）。该技术的局限性：短 prompt（如 64 tokens）下 chunking 引入额外 padding 开销，且 out-of-order 调度空间缩小，加速效果下降（4.3× vs 10.5× 于 1024-token）。

## MTIA Compiler Stack: TorchDynamo → TorchInductor → Triton → Custom Silicon

术语解释
MTIA编译器栈是Meta为MTIA 2i芯片构建的软件编译路径，基于开源PyTorch 2.0生态（TorchDynamo + TorchInductor + Triton），增加MTIA Graph Compiler和MTIA Triton Compiler将Triton IR映射到MTIA的异构fixed-function硬件单元（DPE/RE/SE/MLU）。

术语是什么？
MTIA编译器栈的完整编译流程：(1) **TorchDynamo**——PyTorch 2.0的symbolic tracing组件，capture Python字节码生成fx Graph，支持dynamic shapes（通过guard条件fallback到eager）和control flow。(2) **TorchInductor**——将fx Graph lower为Triton IR，自动检测operator patterns做fusion（如FC→ReLU融合为single Triton kernel），生成Triton代码。(3) **MTIA Graph Compiler**——将Triton IR映射到MTIA异构PE架构的自定义指令集：(a) GEMM→DPE custom instructions（含tile size/blocking/streaming strategy）；(b) element-wise/activation/SIMD→SE custom instructions；(c) reduction→RE custom instructions；(d) layout transform（transpose/reshape/concat）→MLU custom instructions。(4) **MTIA Triton Compiler**——进一步优化，利用MTIA特有custom instructions（multi-context、auto-increment offset、DMA prefetch）减少instruction count和提高吞吐。(5) **Graph Optimization**——额外pass做：sibling transpose FC fusion（transpose输出被多个FC复用→融合为一个operator）、LayerNorm horizontal batching、delayed broadcast（推迟embedding broadcast减少2x memory footprint）、operator scheduling reorder（最小化activation liveness range）。(6) **Autotuning**——编译时调用autotuning框架选择optimal kernel variant/batch size/data placement/request coalescing。MTIA同时支持graph compilation路径（生产推理）和eager mode路径（训练/debugging/非graph化操作）。

从编译框架角度拆解术语：
一个推荐模型在MTIA上的完整编译流程：
```
# Step 1: Model definition (standard PyTorch)
class DLRM(torch.nn.Module):
    def forward(self, sparse_features, dense_features):
        embeddings = [self.emb_tables[i](sparse_features[i]) for i in range(N)]
        pooled = cat(embeddings)  # TBE pooling
        dense_out = self.dense_mlp(dense_features)
        interaction = pooled * dense_out  # Hadamard product
        return self.final_mlp(interaction)  # sequence of FC layers

# Step 2: TorchDynamo trace
import torch._dynamo as dynamo
fx_graph = dynamo.export(DLRM(), sparse_features, dense_features)

# Step 3: TorchInductor + MTIA Graph Compiler
# 3a. TorchInductor: pattern detection + Triton code gen
triton_modules = inductor.compile(fx_graph)
# e.g., detects FC+ReLU pattern -> fused_triton_fc_relu()

# 3b. MTIA Graph Compiler: Triton -> MTIA custom instructions
mtia_subgraph = mtia_graph_compiler.compile(triton_modules)
# Maps: tl.dot(A, B) -> DPE_GEMM(A_tile, B_tile, accum_mode=FP32)
#        tl.relu(x) -> SE_RELU(src=x_buf, dst=y_buf, mode=LUT)
#        tl.sum(x, axis=1) -> RE_REDUCE(x_buf, axis=ROW, dst=reduced_buf)
#        tl.transpose(x) -> MLU_TRANSPOSE(src=x, dst=x_t, shape=[H,W])

# 3c. MTIA Triton Compiler: custom instruction optimization
mtia_subgraph = mtia_triton_compiler.optimize(mtia_subgraph)
# Applies: multi-context instructions (reuse DPE config registers)
#          auto-increment offset (loop over tiles without re-issue)
#          DMA prefetch (prefetch next weight tile to LLC)

# 3d. Graph optimization passes
mtia_subgraph = graph_opt(mtia_subgraph)
# sibling_transpose_fc_fusion, layer_norm_horizontal_batch,
# delayed_broadcast, operator_scheduling_reorder

# Step 4: Autotuning
autotuned_cfg = autotuning_framework.search(mtia_subgraph)
# Sweeps: data_placement(LLS_size), batch_size, fc_kernel_variant,
#         request_coalescing_window
best_subgraph = apply_config(mtia_subgraph, autotuned_cfg)

# Step 5: Deployment
# Compile to firmware-compatible binary + load to MTIA 2i
# Runtime executes compiled subgraph on 64 PEs via Control Core
```

术语一般如何实现？如何使用？
MTIA编译器栈的使用方式与PyTorch 2.0的compile API一致（`torch.compile(model, backend="mtia")`），对模型开发者透明。MTIA Graph Compiler和MTIA Triton Compiler是Meta的内部闭环软件，基于开源Triton的MLIR-based编译基础设施扩展。关键创新在于：(1) 从通用Triton IR到异构fixed-function unit的映射——需要compiler理解DPE/RE/SE/MLU各自的能力和限制；(2) 与autotuning的深度集成——编译阶段同时决定kernel selection和data placement；(3) eager/compile双路径并存——对不能完全静态编译的model（如dynamic control flow、jagged tensor ops），fallback到eager mode。与其他AI芯片编译器的对比：Google TPU使用XLA编译（HLO→TPU backend，静态图）；Groq编译为确定性的dataflow schedule；MTIA的独特之处在于紧密结合PyTorch 2.0生态且支持eager fallback。

## Accelerator Autotuning Framework (Data Placement + Batch Size + Kernel Variant + Request Coalescing)

术语解释
Autotuning Framework是MTIA 2i的自动优化系统，在模型部署前自动搜索最优的硬件/软件配置参数（data placement、batch size、FC kernel variant、request coalescing参数、model sharding策略），替代人工逐个模型调优。

术语是什么？
MTIA Autotuning Framework在模型onboarding时自动确定以下配置：(1) **Data Placement**——如何将256MB SRAM分区为LLS和LLC，哪些tensor pin在LLS。启发式：LLS=activation buffer size，剩余作LLC；对比activation fit LLS vs 降batch size后fit LLS的性能择优。(2) **Batch Size**——构建多个batch size的model snapshot，通过traffic-replay test评估throughput vs latency，选满足P99 latency SLO下最大吞吐的batch size。(3) **FC Kernel Variant**——使用performance database + ANN search为每个FC shape选择最优stationary variant（input/output/weight）和DMA/CB参数。(4) **Request Coalescing**——运行实验确定最优coalescing时间窗口和并行window数量（对P99 latency高度敏感，需达到>95% requests per batch）。(5) **Model Sharding**——检查model+runtime buffers是否超出单device DRAM→自动尝试shard到多device。框架的输入是模型graph，输出是最优配置的deployable model。Autotuning已完全替代人工优化用于所有生产模型launch。

从编译框架角度拆解术语：
Autotuning的工作流程（以一个新推荐模型onboarding为例）：
```
def autotune_model(model_graph, hw_spec):
    # 1. Data placement tuning
    best_placement = None
    for lls_size in [32, 64, 96, 128, 160, 192, 224, 256]:  # MB steps
        activation_buffer = estimate_activation_size(model_graph)
        if activation_buffer <= lls_size:
            placement = (LLS = lls_size, LLC = 256 - lls_size)
            perf = traffic_replay(model_graph, placement)
            if perf > best_perf:
                best_placement = placement
    
    # Special case: activation too large for any LLS
    if best_placement is None:
        # Compare: current batch with activation in LLC
        # vs reduced batch with activation in LLS
        perf_llc = traffic_replay(current_batch, placement=(LLS=0, LLC=256))
        reduced_batch = find_max_batch_with_activations_in_lls(model_graph)
        perf_lls = traffic_replay(reduced_batch, placement_lls)
        best = max(perf_llc, perf_lls)
    
    # 2. Batch size tuning
    batch_configs = []
    for batch_size in [128, 256, 512, 1024, 2048]:
        snapshot = build_snapshot(model_graph, batch_size, best_placement)
        perf = traffic_replay(snapshot)
        if perf.p99_latency <= SLO:
            batch_configs.append((batch_size, perf.throughput))
    best_batch = max(batch_configs, key=lambda x: x[1])
    
    # 3. FC kernel variant tuning
    kernel_config = {}
    for fc_op in model_graph.get_fc_operators():
        features = extract_shape_features(fc_op)
        nearest = ann_search(perf_db, features, k=5)
        best_variant = min(nearest, key=lambda v: v.estimated_latency)
        kernel_config[fc_op.id] = best_variant
    
    # 4. Request coalescing tuning
    for window_ms in [1, 2, 5, 10, 20]:
        for parallel_windows in [1, 2, 4]:
            perf = traffic_replay(model_with_coalescing(window_ms, parallel_windows))
            if perf.fill_rate > 0.95:  # >95% requests per batch
                best_coalescing = (window_ms, parallel_windows)
                break
    
    # 5. Sharding (if needed)
    if model_size > single_device_dram:
        n_devices = ceil(model_size / single_device_dram)
        model_graph = shard_model(model_graph, n_devices)
    
    return AutoTunedConfig(best_placement, best_batch, kernel_config, 
                          best_coalescing, model_graph)
```

术语一般如何实现？如何使用？
Autotuning framework的使用：(1) Offline执行——在模型deploy到production之前在测试环境中完成，autotuning时间对FC kernel variant从exhaustive的hours→ANN search的seconds（1000x缩短）；(2) 与CI/CD集成——新模型或模型更新通过Meta的continuous deployment pipeline [12] 自动触发autotuning；autotuned配置随model一起打包deploy；(3) 生产验证——autotuned模型通过live A/B testing与GPU baseline对比Perf/TCO和model quality后正式launch。论文报告所有autotuned模型的Perf/TCO和Perf/Watt达到或超过之前人工优化的模型。当前限制：data placement tuning使用简单启发式（LLS=activation buffer），更精细的per-tensor placement正在探索中。

## Inter-Layer Scheduling for Tiled DNN Accelerators (面向Tile化DNN加速器的层间调度)

Inter-Layer Scheduling（层间调度）是tiled DNN加速器领域的核心编译优化问题：给定一个包含L层的DNN模型计算图（通常为DAG），以及一个由T个硬件tile通过NoC互联组成的加速器，层间调度的目标是决定跨层和跨tile的计算顺序、内存访问模式和资源分配，以最小化总执行代价（通常为EDP = Energy-Delay Product）。与Intra-Layer Scheduling（层内调度——决定单层如何映射到1个或多个tile，如tiling factor和loop order）不同，Inter-Layer Scheduling处理跨层协同。Crane论文将其分解为四个核心设计因素(design factors)：(1) **Execution Scheme (E)**——决定多层在时间维度和空间维度上的映射模式，三种基本模式为Sequential（逐层顺序执行，每时刻所有tile服务同一层）、Pipeline（多层流水线并发执行，tile跨层共享）、Parallel（多层同时执行，无依赖约束）；(2) **Fusion Strategy (F)**——决定前一层的输出数据是直接保留在片上SRAM供下一层使用（fusion，避免DRAM访问），还是写回DRAM再读取（no fusion）；(3) **Recomputation Scheme (R)**——决定哪些中间activation在forward pass中被丢弃（evict），在backward pass中重新计算恢复，以计算换存储；(4) **Batch Splitting Plan (B)**——决定如何将batch划分为更小的sub-batch逐块处理，以降低单次内存容量需求。现有inter-layer schedulers（SET, Tangram, TileFlow, MBS, Checkmate）均未能全面探索这四大设计因素。

从编译框架角度拆解：
Crane中inter-layer scheduling的编译流程（从模型+硬件到最优调度方案）：
1. **输入**：DNN模型计算图G=(L, E)（层集合L，层间依赖边E）+ 硬件配置（tile数T，每tile MAC数P，SRAM容量Cap_S，DRAM容量Cap_D，NoC/DRAM带宽和energy参数）+ batch size BS。
2. **Intra-Layer Exploration**：对每种sub-batch size候选B_sub，brute-force探索tiling factor组合——将sub-batch的4维(dim_1~dim_4)映射到tile grid的4因子(k_1×k_2×k_3×k_4=T)，计算tile utilization u_{i,j} = Π_q dim_q/(ceil(dim_q/k_q)×k_q)，得到单层compute latency和data traffic特征。
3. **Graph Partition + Hierarchical Block Construction**：按DAG拓扑递归分割——每个in-degree/out-degree>1的vertex（分支/合并点）触发composite block划分，连续线性segment形成basic block。Theorem 1证明该表示对任意DAG具有完备性。
4. **Per-Block MILP Optimization**：对每个block构建ScT（调度表）和MeT（内存表）的MILP——整数变量为ScT[i,j]∈[0,BS/B_sub]和MeT[i,j]^S/D∈[0,BS/B_sub]，约束为Eq.1-12（forward）/ Eq.13-15（training），目标函数min EDP=Latency×Energy。MILP solver求解最优{ScT, MeT, s_i}。
5. **Iterative Refinement**：top-level block优化→固定结果→分解到lower-level blocks→各自独立MILP优化→汇总cost→若|Cost_new-Cost_prev|<θ×Cost_prev则收敛→否则更新workload/traffic→迭代。
6. **输出**：最优schedule（每state的active layers、tile分配、sub-batch处理数和顺序、SRAM/DRAM中的存储区间），以及预估EDP/latency/energy。

术语一般如何实现？如何使用？
Inter-layer scheduling的实现方式：
- **SET** [Cai et al., ISCA 2023]：ratio-tree表示 + simulated annealing搜索。tree traversal决定execution pattern，SA随机扰动tree结构采样候选方案。仅支持E+F+B，不支持R。
- **Tangram** [Gao et al., ASPLOS 2019]：coarse-grained dataflow表示，仅支持E+F，不支持B和R。仅适用于linear chain-structured workload。
- **TileFlow** [Zheng et al., MICRO 2023]：tree-based tile-centric表示 + genetic algorithm搜索。支持E+F+B（部分），仅限linear chain拓扑，不支持training。开源：https://github.com/pku-liang/TileFlow。
- **MBS** [Lym et al., MLSys 2019]：training专用，支持F+B，但仅用sequential execution pattern。
- **Checkmate** [Jain et al., MLSys 2020]：仅支持R（recomputation），batch-level粒度，不探索E/F/B。使用integer programming确定optimal checkpoint。
- **Crane** [Gong et al., MICRO 2025]：table-format层级化表示 + MILP优化，首次全面支持E+F+R+B四种设计因素，支持任意DAG拓扑和inference+training双场景。使用MILP solver（如Gurobi/CPLEX），scheduling速度比SET快2.82×，比Tangram快156.20×。

## Scheduling Table (ScT) and Memory Table (MeT) Representation (调度表与内存表表示)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Scheduling Table (ScT) 和 Memory Table (MeT) 是Crane提出的核心编译表示创新——用两张2N-1行×N列的整数表来完整编码inter-layer scheduling的全部四个设计因素(E, F, R, B)，并将其转化为MILP可解的结构化优化问题。其中N为composite block的子block数，行索引对应pipeline-derived execution states (共2N-1个states)，列索引对应子block。ScT记录**累积处理的sub-batch数**（提供存储区间的upper bound），MeT记录**内存中存储的sub-batch lower bound**（对SRAM和DRAM分别追踪）。两张表联合定义了每state每子block在SRAM和DRAM中的sub-batch存储区间(MeT, ScT]，隐式编码了fusion/recomputation/batch splitting的决策效果。

从编译框架角度拆解：
Crane中ScT和MeT的构建和约束体系：

**ScT构造（Eq.1-6）**：
- ScT[i,j] ∈ N∪{0}：子block B_j从State-1到State-i累计处理的sub-batch数。
- Eq.1：整数非负约束。
- Eq.2：ScT[i,j]=0 for i∈[1, j-1]——在B_j首次激活前处理数为0。
- Eq.3：ScT[i,j]=BS/B_sub for i∈[N+j-1, 2N-1]——B_j执行完毕后累计为全batch。
- Eq.4：ScT[i+1,j] ≥ ScT[i,j]——单调不降。
- Eq.5：ScT[i,m] ≥ ScT[i,j] + d_{m,j}——数据依赖约束（若B_j依赖B_m，B_m处理数≥B_j处理数，防止依赖倒置）。
- Eq.6：ScT[i,j] = s_i + ScT[i-1,j]——逐state累加，s_i为State-i的state workload。

**MeT构造（Eq.7-12）**：
- MeT[i,j] = (MeT^D[i,j], MeT^S[i,j])：DRAM和SRAM中B_j输出sub-batch的lower bound（开区间）。
- Eq.7-8：MeT整数非负 + 随state单调不降（越早的sub-batch越先被淘汰）。
- Eq.9：MeT[i,j] ≤ ScT[i,j]——lower bound不能超过已处理的sub-batch数。
- Eq.10：当d_{m,j}=1时，min(MeT^S[i-1,m], MeT^D[i-1,m]) ≤ ScT[i-1,j]——B_j依赖B_m的输出时，B_m的stored data必须在上一state的ScT范围内可用。
- Eq.11-12：Σ_j (ScT[i,j]-MeT^S[i,j])×V_j ≤ Cap_S 和 Σ_j (ScT[i,j]-MeT^D[i,j])×V_j ≤ Cap_D——容量约束。

**ScT+MeT编码四因素的机制**：
- Execution Scheme (E)：由ScT的行间累计模式决定——sequential→s_1=s_{N+1}=BS/B_sub其余为0；pipeline→所有s_i>0均匀分布；parallel→s_1=s_{N+1}=BS/B_sub其余为0。
- Fusion Strategy (F)：由MeT^S区间隐式编码——若MeT^S[i,j] < ScT[i,j]（即SRAM中有B_j的输出），且下一层B_{j+1}在后续state读取该数据，则隐式实现了layer fusion。
- Recomputation (R)：前向传播MeT_FW的DRAM lower bound决定checkpoint位置；Eq.13确保前向结束时B_m的stored activation≥B_j；Step-1的ScT_BW1（Eq.14约束）消费DRAM中activation做backward；Step-2的ScT_BW2（Eq.15约束）recompute仅恢复被丢弃的sub-batch。
- Batch Splitting (B)：B_sub作为候选枚举参数；不同的B_sub改变ScT/MeT的行列规模和容量约束的松紧度。

术语一般如何实现？如何使用？
ScT和MeT作为MILP的decision variables实现：使用Gurobi、CPLEX或SCIP等commercial/open-source MILP solver求解。每个block的MILP模型包含(2N-1)×N个ScT整数变量和(2N-1)×N×2个MeT整数变量，以及Eq.1-15的线性约束。ScT和MeT的结构化特性使得约束均为线性，EDP目标函数中latency和energy分别建模为：Latency = Σ_i s_i × max_j L_comp,i,j + Σ_{i,j,m} traffic_latency；Energy = Σ compute_energy + Σ NoC_hop_energy + Σ DRAM_access_energy。MILP solver利用branch-and-bound + cutting plane高效剪枝搜索空间O(m^{4n-1} log m)，实现快速收敛。

## MILP-based Inter-Layer Scheduling Optimization (基于MILP的层间调度优化)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
MILP-based Inter-Layer Scheduling Optimization是指将DNN层间调度问题建模为混合整数线性规划(Mixed-Integer Linear Programming, MILP)，并使用成熟的MILP solver（如Gurobi、CPLEX、SCIP）求解最优调度方案。Crane首次将inter-layer scheduling的完整问题（包含Execution Scheme、Fusion Strategy、Recomputation、Batch Splitting四因素）形式化为MILP。关键创新在于：Crane的ScT和MeT表格式表示天然产生线性约束和整数变量，使得问题具有well-structured MILP形式，可用branch-and-bound + cutting plane等成熟算法高效求解，避免了既有工作中的启发式搜索（模拟退火、遗传算法）带来的收敛慢和全局最优无保证问题。

从编译框架角度拆解：
Crane中MILP formulation的完整结构：

**决策变量**：
- ScT[i,j] ∈ Z_{≥0}：子block B_j在State-i的累积处理sub-batch数，取值范围[0, BS/B_sub]。
- MeT^S[i,j], MeT^D[i,j] ∈ Z_{≥0}：SRAM和DRAM中存储的sub-batch lower bound。
- s_i ∈ R_{≥0}：State-i的workload（sub-batch数），可从ScT推导。

**约束体系**（以forward pass为例）：
```
# ScT约束
∀i∈[1,2N-1], ∀j∈[1,N]: ScT[i,j] ∈ Z_{≥0}              (Eq.1)
∀i∈[1,j-1]: ScT[i,j] = 0                                 (Eq.2)
∀i∈[N+j-1,2N-1]: ScT[i,j] = BS/B_sub                    (Eq.3)
∀i∈[j,N+j-2]: ScT[i+1,j] ≥ ScT[i,j]                     (Eq.4)
∀i∈[j,N+j-2], ∀(m,j) where d_{m,j}=1: ScT[i,m] ≥ ScT[i,j] + 1  (Eq.5)

# MeT约束
∀i,j: MeT^S[i,j], MeT^D[i,j] ∈ Z_{≥0}                   (Eq.7)
∀i: MeT^S[i,j] ≤ MeT^S[i+1,j], MeT^D[i,j] ≤ MeT^D[i+1,j]  (Eq.8)
∀i: MeT^S[i,j] ≤ ScT[i,j], MeT^D[i,j] ≤ ScT[i,j]        (Eq.9)
∀(m,j) with d_{m,j}=1: min(MeT^S[i-1,m], MeT^D[i-1,m]) ≤ ScT[i-1,j]  (Eq.10)

# 容量约束
Σ_j (ScT[i,j] - MeT^S[i,j]) × V_j ≤ Cap_S                (Eq.11)
Σ_j (ScT[i,j] - MeT^D[i,j]) × V_j ≤ Cap_D                (Eq.12)
```

**目标函数**：
min EDP = Latency × Energy，其中：
- Latency = Σ_i s_i × max_j(Workload_j / (u_{i,j} × T_{i,j} × P)) + Σ_{i,j,m} DataTraffic / BW_NoC
- Energy = Σ_i Σ_j FLOPs_j × E_comp + Σ NoC_bits × E_NoC × hops + Σ DRAM_bits × E_DRAM

**层级化MILP求解流程（Algorithm 1）**：
```
1. for each B_sub in candidate_sizes:
2.     for each block B in hierarchy (top-down):
3.         Build MILP(ScT, MeT, constraints=Eq.1-12/15, objective=EDP)
4.         Solve MILP → {ScT*, MeT*, s_i*}
5.     end for
6.     Compute total_cost (top-level block)
7.     Keep Top-K1 candidates by Cost_comp and Top-K2 by Cost_traffic
8. end for
9. Iterative refinement: top-level → lower-level → re-optimize → check convergence
10. Output optimal schedule with min EDP
```

术语一般如何实现？如何使用？
MILP-based scheduling的使用方式：(1) Crane使用C++实现（论文第7.1节），调用commercial MILP solver（论文未明确指定但通常为Gurobi或CPLEX）求解。(2) 相比heuristic方法，MILP的优势在于：thorough search（不依赖随机采样，branch-and-bound保证完备探索）和fast convergence（确定性算法vs随机收敛）。(3) 实际调度速度：Crane在AMD EPYC 7402P CPU上实现平均2.82× speedup vs SET (simulated annealing)，156.20× speedup vs Tangram（coarse-grained heuristic）。(4) 适用条件：子block数N较小时（通常N ≤ 10），MILP可直接求解；对于更深的模型，Crane通过层级化分解（将深层模型递归划分为浅层block）降低每个MILP实例的规模。(5) 与已有MILP-based scheduling的关系：Checkmate也使用IP（Integer Programming）但仅用于解决recomputation checkpoint选择问题（单一设计因素）；Crane将MILP扩展到完整四因素联合优化。

## Hierarchical Block Scheduling Representation (层级化Block调度表示)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Hierarchical Block Scheduling Representation是Crane提出的DNN模型计算图调度抽象方法：将模型的计算图（DAG）递归分解为嵌套的block结构，每个block要么是basic block（连续线性层的直接映射），要么是composite block（包含N个有序子block）。在每层composite block内部，子block按canonical pipeline pattern组织——产生2N-1个execution states。通过分配每个state的workload s_i，可表达sequential、pipeline、parallel及任意混合模式。Theorem 1形式化证明了该层级化表示对任意DAG结构模型的完备性——任何有效的inter-layer scheduling行为都可以被层级化block组合+ pipeline-derived states和关联workload完全表示。该表示的关键优势：(1) 拓扑灵活性——支持分支(fan-out)、合并(fan-in)和skip connection等非链式结构；(2) 递归可组合——basic block和composite block使用统一的scheduling语义。

从编译框架角度拆解：
层级化block表示的构建过程（以ResNet bottleneck为例）：
1. **Graph Partition**：遍历DAG G=(L,E)，找到所有in-degree>1或out-degree>1的vertex作为partition boundary。
2. **Basic Block构建**：每个连续线性段（所有vertex的in-degree=out-degree=1）封装为basic block，如Conv1→BN→ReLU为一个basic block。
3. **Composite Block构建**：在每个分支/合并vertex处，将其fan-in/fan-out的basic blocks组合为composite block。例如ResNet bottleneck：主路径([Conv1×1→Conv3×3→Conv1×1])和skip connection([Identity])组合为一个fan-in composite block。
4. **递归应用**：composite block可作为更高层composite block的子block，递归直到top-level block覆盖整个模型。
5. **Per-Block Pipeline State推导**：对每个composite block B(N个子block)，pipeline states S_B = {1,...,2N-1}，state-i的活跃子block集J_i = {B_1,...,B_i} (i≤N) 或 {B_{i-N+1},...,B_N} (i>N)。
6. **Workload分配**：为每个state分配s_i个sub-batch的workload，满足计算完备性约束：Σ_{i∈A_ℓ} s_i = BS/B_sub，其中A_ℓ是layer ℓ的involved state set。

**Fan-out/Fan-in的表达示例**：
- Fan-out（如Inception module）：sub-block B_1输出被B_2和B_3并行消费。通过设置s_1=s_{N+1}=BS/B_sub, other s_i=0，表达parallel执行。
- Fan-in（如ResNet add）：B_{N-1}和B_N的输出合并到后续block。通过设置s_{N-1}=s_{2N-1}=BS/B_sub表达merging。

术语一般如何实现？如何使用？
层级化block表示在Crane中作为C++数据结构实现：(1) Block类包含：type(basic/composite)、layer_set L_B、sub_block list C_B(仅composite)、state_set S_B、ScT、MeT。(2) Hierarchy tree从top-level block根节点递归构建。(3) 每个block独立进行MILP优化（内层block的BS、allocated tiles、SRAM capacity可从外层block的优化结果继承）。(4) 层级化refine过程来回传播workload和traffic信息：top→bottom传递tile分配和容量约束，bottom→top回传优化cost。

## Timeloop + Accelergy (DNN Accelerator Modeling Toolchain)

术语解释
Timeloop和Accelergy是MIT/Stanford开发的DNN accelerator design space exploration工具链。Timeloop建模accelerator architecture并搜索最优mapping（loop order/tiling/parallelization）；Accelergy提供architecture-level energy estimation。二者结合可对给定workload在给定architecture上评估execution time、energy和area，广泛用于accelerator architecture research。

术语是什么？
**Timeloop [41]**（https://github.com/NVlabs/timeloop）：接受三层specification——(1) Problem: workload的loop nest dimensions（Einsum-like spec，如GEMM M,N,K）；(2) Architecture: PE array topology、buffer hierarchy sizes、NoC bandwidth、compute unit capabilities；(3) Mapping: loop order constraints、tiling factors、spatial parallelization。Timeloop的mapper搜索mapping space以最小化runtime（cycle count），评估每个mapping的data movement（DRAM↔global buffer↔PE buffer↔compute）和compute cycles（含pipeline fill/drain）。输出：optimal mapping、execution cycles、data movement volume per memory level。

**Accelergy [56]**（https://github.com/nelliewu95/accelergy）：接受architecture component的energy model（per-access energy for SRAM/DRAM at given technology node, per-operation energy for MAC/exp/div/add max etc., NoC hop energy）和Timeloop的activity counts（各component的access/operation次数），计算total energy breakdown。Energy model可来自CACTI（SRAM/DRAM）、published circuit designs、或custom plug-ins。

从编译框架角度拆解术语：
FuseMax使用Timeloop+Accelergy的完整评估流程：
```
1. Architecture Specification (YAML):
   arch:
     - 2D_PE_array: { dimensions: [M0,P0], MAC/cycle: 1, exp: 6 MACs }
     - 1D_PE_array: { dimensions: [256], ops: [add, max, mul, div] }
     - GlobalBuffer: { size: X KB, bandwidth: Y GB/s }
     - DRAM: { size: ∞, bandwidth: Z GB/s }
     - NoC: { topology: mesh, hop_energy: E_pJ }

2. Einsum Specification (for each Einsum in Cascade 5):
   einsum BQK:
     iteration_space: [E, M0, P]
     inputs:
       - Q: { coordinates: [e, p], datawidth: fp16 }
       - BK: { coordinates: [e, m0], datawidth: fp16 }
     output: BQK { coordinates: [m0, p], datawidth: fp16 }
     compute: multiply-add (MACC)

3. Timeloop Mapping Search:
   - Mapper searches: loop_order, tiling_factors, spatial_parallelization
   - Objective: minimize cycles subject to buffer capacity constraints
   - Output: optimal mapping + activity counts

4. Accelergy Energy Estimation:
   - Input: architecture energy models + Timeloop activity counts
   - Output: energy breakdown (DRAM/GlobalBuffer/PE_array/NoC)

5. Full Cascade Combination (TeAAL heuristics [35]):
   - Merge per-Einsum results considering:
     a. Pipeline overlap (epoch-level software pipelining)
     b. Tile dependency stalls (e.g., RM must wait for LM drain)
     c. Shared resource contention (DRAM BW, global buffer BW)
   - Total cycles = max(compute-bound time, memory-bound time)
     after accounting for pipeline overlap
   - Total energy = Σ(per-Einsum energy) - overlap savings
```

## EDGE: Extended General Einsums Notation

术语解释
EDGE (Extended General Einsums) 是一种扩展Einsum符号系统，支持表达传统Einsum（BLAS、tensor network contraction）无法处理的复杂tensor algebra computations（如graph algorithms、attention with softmax等），通过user-defined map/reduce/populate actions提供可定制计算语义。

术语是什么？
EDGE [39]将Einsum的计算分离为三个"actions"：(1) **map**（map action）：指定two tensors的shared rank上的pair-wise computation，包含merge operator（指定iteration space中哪些points需被touched）和compute operator（指定对两个data values执行的操作）。(2) **reduce**（reduce action）：指定reduction step，同样包含merge和compute。(3) **populate**（=）：将RHS计算值放入LHS tensor的对应位置。EDGE支持user-defined merge operators（∩ intersection—touch points where both operands non-zero、∪ union—touch points where at least one operand non-zero、← left—touch points where left/target operand non-zero、1 pass-through—touch all points）和compute operators（× multiply、+ add、max maximum、÷ divide、sub-then-exp subtraction then exponentiation、σ sigmoid等）。EDGE还支持：(1) iterative/generative ranks——表达递归和迭代计算（如prefix sum, running max）；(2) filtering rank expressions——只touch tensor subset（如A_{k:k≤i}）；(3) shorthand notation——drop default reduce actions、infix notation for map、operator-specific shorthand（如max(A,B)、A/B、e^{A-B}）。

从编译框架角度拆解术语：
EDGE作为specification language表达attention kernel的完整计算（以Cascade 5的Einsum 47为例）：
```
# Full EDGE spec:
SLN_{m1,m0,p} = BQK_{m1,m0,p} · RM_{m1+1,p} :: _{m0} sub-then-exp(1)
# - map action: between BQK and RM on rank (shared rank of sub-then-exp)
# - merge: 1 (pass-through, touch all m0 points)
# - compute: sub-then-exp → first BQK - RM, then exp(result)
# - reduce: none (no reduction on m0, SLN retains m0 rank)

# Shorthand notation:
SLN_{m1,m0,p} = e^{BQK_{m1,m0,p} - RM_{m1+1,p}}
```
EDGE如何帮助FuseMax的分析：(1) Pass分析——EDGE显式表达每个Einsum的iteration space和data dependencies，使is-ﬁbertree的read-after-write关系可被机械推导；(2) Reassociation validation——利用分配律变换cascade时，EDGE的precise semantics保证functional equivalence（如从3-pass到1-pass的变换）；(3) Fusion legality check——EDGE的merge operator定义哪些points被touch，可机械检测两个Einsum的iteration space overlap和fusion safety。

术语一般如何实现？如何使用？
EDGE的reference implementation未公开（论文[39] on ArXiv 2024）。其核心思想受TeAAL [35]（sparse tensor accelerator modeling）和TACO（sparse tensor algebra compiler）影响。在FuseMax中，EDGE用作analysis和specification tool，而非compilation target——cascade经过EDGE form formalized后，manually translated to Timeloop workload specification。EDGE的表达力超越传统Einsum的限制：(1) non-linear operations（exp, sigmoid, max）可直接表达，无需decompose为multiple lower-level Einsums；(2) iterative ranks使recursive algorithms（如1-pass running softmax）可被代数式而非imperative伪代码描述。未来可能将EDGE集成到accelerator compiler pipeline作为higher-level IR，自动derive mapping constraints and fusion opportunities。

## MetaGraph (Contracted Computation Graph)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
MetaGraph是Spindle通过graph contraction（§3.1）从原始计算图G=(V,E)缩约得到的简化计算图G_M=(V_M, E_M)。G_M的节点为MetaOp（连续同构operators的融合），边保留了原图中MetaOp之间的数据流依赖关系。与原始计算图相比，MetaGraph大幅减少节点数（从|V|个operator到|V_M|个MetaOp，通常降低1-2个数量级），同时保持了完整的workload异构信息（每个MetaOp m携带operator type、input data size、L_m个operators等信息）和依赖拓扑。MetaGraph是后续所有优化步骤（scalability estimation、resource allocation、wavefront scheduling）的基础。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
MetaGraph的构建和优化流程（Spindle Execution Planner, §3.1）：
```
// Step 1: Build computation graph from user-defined tasks
G = build_computation_graph(tasks)  // PyTorch FX Tracer
// G: nodes = operators (Linear, Attention, Conv, etc.)
//    edges = data flow (tensor activations)
//    Each node has: op_type, input_shape, output_shape, parameters

// Step 2: Graph Contraction
G_M = G
for op_i in topological_order(G_M):
  for op_j in successors(op_i):
    if out_degree(op_i)==1 AND in_degree(op_j)==1
       AND op_type(op_i)==op_type(op_j)
       AND input_shape(op_i)==input_shape(op_j):
      // Contract: fuse op_i and op_j into MetaOp m
      m.operators = op_i.operators + op_j.operators  // L_m = L_i + L_j
      m.op_type = op_i.op_type
      m.input_shape = op_i.input_shape
      G_M = replace(G_M, op_i, op_j, m)

// Step 3: Assign MetaLevels via BFS
for m in BFS_order(G_M):
  m.meta_level = BFS_depth(m)  // same level → no dependencies between MetaOps
```
以图3的Multitask-CLIP为例，原始G包含20+ operators（Audio/Text/LM/Vision不同type的Transformer layers），缩约为7个MetaOp（MetaOp 1=Audio×3, MetaOp 2=Text×2, MetaOp 3=LM×3, MetaOp 4=Text×2, MetaOp 5=Vision×2, MetaOp 6=Vision×2, MetaOp 7=LM×3），分属不同MetaLevels。

## Micro-Kernel Polymerization（微内核聚合）

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Micro-Kernel Polymerization是MikPoly [ASPLOS 2024] 提出的dynamic-shape tensor operator在线编译优化的核心机制。它采用两阶段策略：**Offline阶段**从tensor operator的loop tiling模板中提取最内层offline loops作为micro-kernel template，利用现有静态auto-scheduler（如TVM）为若干固定尺寸(uM,uN,uK)生成一组高度优化的micro-kernels并编译为binary，同时在单PE上为每个micro-kernel学习分段线性性能模型g_predict(t,K~,H)。**Online阶段**在runtime获知operator具体shape后，通过预定义的polymerization patterns（基于7-block skeleton的9种输出分块模式，Pattern I-IX）将外层online loops重组为多个region R_i，每region包含一个parameterized micro-kernel，再通过polymerization strategy遍历并替换为offline生成的fixed-size micro-kernels，用polymerization cost model（Cost=Σ f_wave×f_pipe）评估每种组合，选最优方案生成最终tensor program。关键思想是将传统的"compile-time per-shape auto-tuning"解耦为"offline固定尺寸micro-kernel优化 + online按需聚合"，使编译时间从~0.33 CPU hours降至~2μs，同时保证性能接近oracle最优（96%）。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
MikPoly的micro-kernel polymerization在编译框架中按以下流程运作（以GPU上GEMM operator为例）：
1. **Two-Stage Program Template设计**：将GEMM的tiled program template Q分为Q_offline（内层offline loops: for ui.0 in uTM.0/for uj.0 in uTN.0/for uk.0 in uTK.0 → for ui.1 in uTM.1/for uj.1 in uTN.1/for uk.1 in uTK.1 → C[...]+=A[...]*B[...]）和Q_online（外层online loops: for i.0 in TM.0/.../for k.1 in TK.1）。
2. **Offline Micro-Kernel Generation**：从Q_offline提取micro-kernel template K_hat→TVM auto-scheduler在给定accelerator上搜索优化tile sizes (uTM.0, uTM.1, ...)，每个micro-kernel固定尺寸为(uM=uTM.0×uTM.1, uN=uTN.0×uTN.1, uK=uTK.0×uTK.1)→生成约n_gen³=32³=32768个候选→用n_syn=12个合成shape评估→RankAndPrune保留Top-n_mik=40个。
3. **Offline Performance Modeling**：每个micro-kernel K~在单PE上运行t从1到5120次reduction loop的pipelined task，记录执行时间→用分段线性函数g_predict(t,K~,H)学习t vs cost关系。
4. **Online Polymerization**：Runtime shape已知（如M=4096,N=1024,K=4096）→遍历9种polymerization patterns（Pattern I：单一region覆盖全部M×N→Pattern II：两个region分割M维度→...→Pattern IX：7-block全分割）→对每种pattern遍历polymerization strategies（即从S_K~中为各region R_i选具体micro-kernel K~_i）→用cost model（Eq.2: Cost=Σ f_wave(R_i,K~_i,H)×f_pipe(R_i,K~_i,H)）预估→启发式剪枝（某(R_i,K~_i)的cost超当前最优则跳过后续）→选cost最低组合。
5. **Code Instantiation**：对选中组合设置micro-kernel参数（tensor地址偏移、loop边界）→生成CUDA kernel launch→dispatch到GPU执行。整个online阶段约2μs。

Micro-kernel polymerization的实现要点：(1) Micro-kernel编译为binary文件，保持固定shape尺寸，仅tensor起始地址和loop iteration counts作为runtime参数；(2) Polymerization patterns通过7-block skeleton（将output tensor分7块）系统化生成，MikPoly最终使用9种代表性pattern（I-IX），GPU上为减少runtime overhead仅用Pattern I和II，NPU上使用全部9种；(3) Cost model的f_pipe基于offline学习的g_predict，f_wave基于parallelism计算（ceil(f_parallel/|P_multi|)），两者相乘求和，无需实际执行即可预估性能；(4) 采用local padding技术（类似CUTLASS）处理非对齐边界；(5) GPU上利用hardware scheduler动态分配thread blocks到SMs，NPU上用max-min static allocation分配到DaVinci Cores。该机制适用于任何可表示为两阶段tiled loop nest的tensor operator（如GEMM、Convolution），可扩展到不同accelerator平台（GPU/NPU）。

## Polymerization Cost Model（聚合代价模型）

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Polymerization Cost Model是MikPoly在online阶段用于快速评估不同micro-kernel组合性能的轻量代价模型。其核心公式为：Cost(S,H) = Σ_{(R_i,K~_i)∈S} f_wave(R_i,K~_i,H) × f_pipe(R_i,K~_i,H)，其中f_pipe预测单个pipelined task在单PE上的执行开销（调用offline学习的g_predict），f_wave估算所有pipelined tasks跨P_multi并行执行所需的wave数（ceil(f_parallel/|P_multi|)）。该模型同时考虑计算、内存访问和并行度三个因素，能够在~2μs内完成搜索（vs Oracle穷举搜索的~1.6s），且性能达到Oracle的96%。

## Polymerization Patterns（聚合模式）

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Polymerization Patterns是MikPoly用于将tensor operator的online loops重组为多region结构的预定义模式集合。基于一个universal 7-block output skeleton（将operator的输出划分为x-~共7个块），MikPoly系统化生成多种partition pattern，每个pattern将output划分为多个region（每个region覆盖skeleton中的一个或多个连续block）。每个region内部含有独立的parameterized micro-kernel，可在polymerization时替换为不同尺寸的fixed-size micro-kernel。从合成workload的评估中，MikPoly最终选择9种代表性pattern（I-IX）。GPU上为最小化runtime overhead仅使用Pattern I（单region）和Pattern II（双region分割M维度），NPU上使用全部9种以最大化性能。

从编译框架角度拆解术语，比如术语如何在编译框架中发挥作用，给出术语在编译框架中运转流程的具体例子。通过联网搜索让回答具体和精准。
以GEMM为例，Pattern的解释：
- **Pattern I**（单一region）：整个GEMM output (M×N)作为一个region，使用单一micro-kernel。对应7-block skeleton中全部7个block在一个region内。类似传统单-kernel tiled GEMM。
- **Pattern II**（双region沿M轴分割）：将output分为上下两个region——R1覆盖M的前x行（blocks x-~），R2覆盖M的剩余行（blocks {-|）。每region独立选择micro-kernel。Case study中GEMM-AB即用此pattern：A处理M=3072部分，B处理M=1024部分。
- **Pattern III-IX**（更复杂分割）：可能沿M、N或两者同时分割，产生3-7个regions。例如沿M和N同时分割可产生4-region layout（左上、右上、左下、右下）。
选择multi-region pattern的关键动机是解决load imbalance——当单一micro-kernel的tile尺寸导致wave数不整除时（如128 pipelined tasks在108 SMs上需2 waves, 最后一wave underutilized），通过分割为多个region用不同micro-kernel可优化总wave数分配。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
实现：(1) 7-block skeleton定义——将output的M维和N维各分两段（共3个分割点），产生7个非重叠block。(2) 对每种pattern，每个region内的online loops按skeleton边界重新计算loop ranges。(3) Pattern选择在online阶段：先遍历所有pattern，对每种pattern探索polymerization strategies，用cost model评估，选最优。(4) GPU上限于Pattern I/II是为减少搜索空间（9种pattern × 40个micro-kernels × region组合爆炸）。Pattern本身是operator无关的（基于output tensor的logical partitioning），可复用于不同operator类型。

## PIM-SYCL Programming Model (PIM-SYCL编程模型)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
PIM-SYCL是由Samsung提出的面向PIM（Processing-in-Memory）异构系统的编程模型[27]，基于SYCL（Khronos标准的跨平台异构编程框架）扩展，允许开发者在统一的编程接口下管理host CPU、GPU和PIM设备的计算和数据。在Hermes系统中，PIM-SYCL被用作GPU-NDP-DIMM异构平台的编程模型，配合CUDA Unified Memory实现GPU和NDP-DIMM之间的implicit data transfer。核心特性：(1) Unified memory space——GPU和NDP-DIMM共享同一逻辑地址空间，数据移动对程序员透明。(2) Device-specific kernel dispatch——GPU kernel通过cudaLaunchKernel触发（Tensor Cores GEMM），NDP-DIMM kernel通过memory command interface触发（MAC/softmax命令）。(3) Implicit synchronization——barrier指令自动插入以保证GPU和DIMM计算结果的一致性。

## Integer Linear Programming for Offline Neuron Mapping (离线神经元映射的整数线性规划)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
在Hermes系统中，Integer Linear Programming (ILP)被用于求解GPU-NDP-DIMM异构平台上LLM各层weight neurons的最优初始（offline）放置方案。将neuron mapping形式化为ILP问题：给定每neuron的activation frequency（从C4/Pile 128 samples offline profile获得）、每neuron的memory footprint、每processing unit的per-neuron compute latency和storage capacity，求解最小化总推理延迟的binary assignment matrix x_{jil}（neuron i in layer l是否放置于processing unit j）。Hermes使用开源PuLP solver（https://coin-or.github.io/pulp/），求解时间~110秒，offline compilation一次性完成。

从编译框架角度拆解术语，给出具体流程：
```
ILP Formulation (Offline Neuron Mapping):

Given:
  L = set of all layers
  N = set of all neurons (per layer)
  D = set of processing units {GPU, DIMM-0, ..., DIMM-7}
  f_i = activation frequency of neuron i (profiled, 0~1)
  M_i = memory space required by neuron i (bytes)
  T_l^j = time to compute one neuron in layer l on processing unit j
  Tsync = time for one synchronization (GPU↔DIMM data transfer)
  S_j = storage size of processing unit j

Find: x_{jil} ∈ {0,1} (1 = neuron i in layer l is on processing unit j)

Objective:
  min  Σ_l max( T_GPU_l , max_j(T_DIMM_j_l) )
  
  where:
  T_GPU_l = T_l^GPU · Σ_i f_i · x_GPU_il + 2·Tsync
    // GPU compute time = per-neuron time × activated neuron count + sync overhead
  T_DIMM_j_l = T_l^DIMM_j · Σ_i f_i · x_DIMM_j_il
    // Each DIMM's compute time depends on its allocated activated neurons

Constraints:
  Σ_i M_i · x_GPU_il ≤ S_GPU     ∀l  // GPU memory capacity
  Σ_i M_i · x_DIMM_j_il ≤ S_DIMM_j  ∀l,j  // DIMM memory capacity

Solution via PuLP solver:
  Input: profiled parameters → Build ILP model → Solve → Extract x_{jil}
  Output: initial neuron mapping table for each layer
  Time: ~110 seconds (one-time offline cost)
```

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
实现方式：(1) Profiling阶段——在C4和Pile数据集上跑128个样本，host CPU monitor记录每个neuron的激活频率f_i。同时在目标硬件上microbenchmark T_l^j（GPU用Nsight Compute，DIMM用Ramulator 2.0模拟）。(2) ILP建模——使用PuLP Python library（开源，基于COIN-OR solver），定义binary variables、linear objective和linear constraints。(3) 求解——PuLP调用底层MIP solver（如CBC/GLPK/CPLEX），由于neuron数量大（LLaMA-7B: ~2^1000的搜索空间），使用approximate decomposition（如per-layer独立求解+iterative refinement）。(4) 结果应用——ILP输出作为initial mapping，runtime通过online predictor和window-based scheduling动态调整。(5) 局限性：ILP求解是NP-hard问题，对大模型（LLaMA2-70B）可能需要更长的求解时间或heuristic pruning。论文未详细说明scaling到更大模型的具体方案。

## Static Offline Tiling Scheduler for Sparse Accelerators

术语解释
面向稀疏加速器的静态离线tiling调度器是HYTE提出的一种编译时优化组件，通过轻量采样分析输入稀疏张量的数据分布模式，结合硬件cost model在pruned搜索空间中定位near-optimal tiling scheme（tile shape/size/inter-tile order/buffer allocation），生成初始执行方案供硬件在线执行。

术语是什么？
HYTE的static offline scheduler运行在host CPU上（Intel Xeon Gold 6248R @3GHz），接收输入张量A和B的format、尺寸、稀疏度信息，以及硬件规格（intra-tile dataflow类型、PE吞吐、SRAM容量、DRAM带宽）。scheduler的输出是一个initial tiling scheme，包含tile shape (Ti, Tj, Tk)、inter-tile order、buffer allocation (SA, SB, SC)。

scheduler的核心挑战：与dense计算不同（计算量和数据size直接已知），稀疏场景下effectual MAC数量和output tensor size依赖于输入张量的数据分布及其复杂的相关性。HYTE使用轻量采样和估计方法推导这些统计量，而无需扫描完整输入张量。

从编译框架角度拆解：
scheduler的完整工作流程（Algorithm 1-2）：
1. **Sampling**：以sp=1/√N比例从A中采样行（SI）和从B中采样列（SJ）。若压缩格式兼容（如CSR取行、CSC取列），直接按索引提取；若不兼容（如从CSR取列），采样点集并重新分组。
2. **EstEffMAC**：遍历contracted dimension k → 对每个k，取A[SI][k]的非零数(szA)和B[k][SJ]的非零数(szB) → sum += szA × szB → effMAC = sum/sp²。
3. **EstNnzCTk**（output tensor C的tiling后访问量）：使用hash-based方法（Flajolet-Martin变体）→ 对每个非零对(i,j)计算h(i,j) = (h1(i) - h2(j)) mod 1 → 在min-heap中deduplicate → 取sk-th最小value vsk → nnzCTk_{Tk=K} = sk/(vsk×sp²)。对其他Tk值递归合并相邻heaps（优化：仅处理sk-th以内的pairs + 预建Nk,max=4096个heap逐级合并）→ 额外cost仅~10%的Tk=K case。
4. **Search Space Pruning**：Tile shape仅考虑power-of-two（log²N choices）；不tile intra-tile dataflow的最外层维度（如Gust的i）；dominated shapes剪枝（更小且所有维度≤另一个feasible shape）。Inter-tile order仅需3种（innermost loop选择）。Buffer allocation仅2种（仅intra-tile reuse vs intra+inter-tile reuse）。总搜索空间≤4×log²N（~数百schemes）。
5. **Cost Model评估**：total time = (I/Ti × J/Tj × K/Tk) × max{PE time, DRAM time, SRAM time}。其中PE time = effMAC/throughput；DRAM/SRAM time = 各张量access量/对应bandwidth，考虑buffering/streaming mode和reuse次数对hit rate的影响。Metadata访问量按fiber segments数量额外计入。
6. **输出**：cost最小的scheme（+对应的Ti/Tj/Tk, inter-tile order, SA/SB/SC）。

术语一般如何实现？如何使用？
- **运行位置**：offline在host CPU上运行（C++，g++ -O3编译），而非硬件accelerator上。offline时间占比平均16.2%（extremely sparse matrices如web-google、kkt_power更高）。
- **Hyperparameter选择**：默认sp=1/√N, sk=√N（遵循理论证明sp×sk≤1的建议[2]），平衡估计精度与计算成本。
- **采样误差与补偿**：静态估计误差平均15%，最高43%（主要发生在irregular tensors）→这些误差由硬件dynamic tile shape tuning在线补偿。
- **可扩展性**：可iteratively扩展到多级buffer hierarchy（每级独立accessor，共享global tiling controller）；对于structured sparsity pattern（N:M, block-diagonal），effMAC和nnzCTk可直接推导而非采样。
- **与dense scheduler的区别**：dense scheduler（如Timeloop[28], CoSA[16], Interstellar[39]）已知computation load和data size，仅需搜索loop order/tiling/parallelism；HYTE额外需要采样估计computation和output data量。

## Smma Instruction (Amove Custom Matrix Multiply-Accumulate Instruction)

术语是什么？
Smma是Amove为支持其fine-grained grouped vectorized data type而引入的自定义矩阵乘累加指令，是NVIDIA GPU标准MMA (Matrix Multiply-Accumulate) 指令的架构扩展。指令格式：`Smma.{M}{N}{K}.{Adtype}{Wdtype}{Sfdtype}{Accdtype}{Odtype}`。其中M/N/K指定矩阵维度；Adtype为activation矩阵数据类型；Wdtype为weight矩阵数据类型；Sfdtype为scale factor数据类型（同时编码精度和格式信息）；Accdtype为accumulation数据类型；Odtype为output数据类型。计算形式：`O_dtype[M,N] = D(A_dtype[M,K] × W_dtype[K,N], Sf_dtype) + Acc_dtype[M,N]`，D(·)为residual-based dequantization函数，执行per-cluster scale factor恢复（`S_ci = S_base - R * E_ci`）后将INT4 MAC结果乘以scale factor转为FP16输出。

从编译框架角度拆解术语：
```
// Smma instruction lowering flow:
// High-level API: output = linear(activation_amove, weight_amove, bias)

// Step 1: Compiler identifies Amove data format operands
// (marked by Sfdtype field indicating Amove scale factor encoding)

// Step 2: Lowering to Smma instruction
Smma.M16N8K32.INT4INT4AMOVE.FP16FP16
//  M=16, N=8, K=32  → matrix tile dimensions
//  Adtype=INT4      → activation elements
//  Wdtype=INT4      → weight elements
//  Sfdtype=AMOVE    → scale factor in Amove format (FP8 base + FP8 residual + 2b encodings)
//  Accdtype=FP16    → accumulate in FP16
//  Odtype=FP16      → output in FP16

// Step 3: Data transformation (compiler-managed)
// Amove format layout in memory:
//   [group 0: FP8(base)|FP8(residual)|encodings[G/C×2b]|elements[G×INT4]]
//   [group 1: ...]
// Compiler ensures alignment and packs Amove metadata with quantized data

// Step 4: Execution on hardware:
//   Preload:  unpack Amove format → register file
//   Dispatch: route metadata → scale factor decoder, INT4 → DP unit
//   Compute:  INT4 MAC (DP) || scale decode: S = base - R * E (parallel)
//   Dequant:  psum * S → FP16
//   Accumulate: acc += dequant_result
```

术语一般如何实现？如何使用？
Compiler负责管理Amove格式的量化操作数和scale factor的layout，并将高层API（如PyTorch的`torch.nn.Linear`）自动lowering为Smma指令+数据格式转换。对programmer透明：模型定义保持不变，矩阵操作通过高层API发出。Sfdtype字段编码scale factor的精度（FP8 E4M3）和格式（Amove: base+residual+encodings），使compiler能进行quantization-aware优化（如register allocation考虑scale factor decoder的operand需求）。在GPU tensor core架构中，Smma指令的硬件执行路径与标准MMA指令共享warp scheduler/operand collector/SIMT dispatch unit，差异仅在data format dispatcher将Amove metadata路由到scale factor decoder而非直接进入DP unit。

## Static Memory Lifetime Analysis for Dataflow Accelerators (数据流加速器的静态内存生命周期分析)

术语是什么？
Static Memory Lifetime Analysis是SN40L compiler实现编译期garbage collection的技术。利用RDU programming model的关键特性——**无动态内存分配（no malloc/free at runtime）、无指针别名（no pointer aliasing）**——compiler可以在编译时精确分析每个tensor symbol的live range（从定义到最后使用之间的program interval）。多个live ranges不重叠的symbols被分配到相同的device virtual addresses（HBM或DDR中的地址复用），实现无runtime overhead的garbage collection。传统dynamic GC需要频繁返回CPU reorganize memory（overhead大），static lifetime analysis在编译时完成所有分配决策——runtime仅执行compiler生成的memory map。在Llama2-7B的实际应用中，此技术使activation tensors和intermediate results能在有限的HBM capacity (64GB) 中fit（模型weights优先占据HBM）。

从编译框架角度拆解术语：
Compiler中的static lifetime analysis流程：
```
Input: Dataflow graph G = (V, E), where V = {symbol_1, ..., symbol_n}
      每个symbol有size和program order中的define/use位置

1. 构建Live Intervals:
   for each symbol s in V:
       s.define_point = program_order(first_definition_of_s)
       s.last_use_point = program_order(last_use_of_s)
       s.live_range = [s.define_point, s.last_use_point]

2. 构建Interference Graph:
   for s1, s2 in V, s1 != s2:
       if s1.live_range ∩ s2.live_range ≠ ∅:  # overlapping lifetimes
           add interference_edge(s1, s2)  # 不能复用同一地址

3. Address Assignment (类似Register Allocation的Graph Coloring):
   addresses = {}
   free_address_pool = [0, total_HBM_size]  # 或DDR
   for s in V (sorted by size descending for best fit):
       conflicting = {symbols already assigned with overlapping live ranges}
       addr = find_free_range(free_address_pool, s.size, conflicting)
       if addr == None:  # HBM不足
           mark s for DDR spill (bandwidth需求最低的symbol优先)
       else:
           addresses[s] = addr
   
4. 输出: Memory map (symbol → HBM/DDR address)
```
关键优化：compiler按aggregate transfer bandwidth requirement排序spill candidates——weights (high temporal locality, accessed every iteration) 优先留HBM，activation tensors (access once) 可spill到DDR。

术语一般如何实现？如何使用？
- 理论基础：经典的compiler register allocation（graph coloring, linear scan）直接应用到device memory management。RDU的restrictive programming model（no dynamic allocation, no aliasing）使分析sound且precise（类似早期的Fortran编译器memory management，无pointer complexity）。
- vs GPU：GPU的CUDA programming model允许dynamic memory allocation和pointer aliasing，使static lifetime analysis不sound——因此GPU通常依赖runtime memory management（PyTorch caching allocator等）或manual memory planning（如FlashAttention的hand-tuned SRAM usage）。
- 通用价值：对于任何restrictive programming model的domain-specific accelerator（无dynamic allocation、无pointer aliasing），static lifetime analysis是实现zero-overhead memory management的关键compiler技术。

## Heterogeneous Memory Spilling in AI Compilers (AI编译器中的异构存储Spilling)

术语是什么？
Heterogeneous Memory Spilling是SN40L compiler自动管理HBM和DDR两个异构存储层的编译技术。与传统的单一存储层spilling（如GPU register→shared memory→HBM）不同，RDU的spilling需决策每个symbol分配在HBM还是DDR。Compiler策略：(1) **默认优先HBM**：所有symbol分配在HBM，除非总size超出HBM capacity (64GB)。(2) **Spilling决策**：当HBM不足时，compiler分析每个symbol的temporal locality和transfer footprint，估算total bandwidth requirement over the entire application。Symbol按aggregate transfer size升序排列——bandwidth需求最小的symbol优先spill到DDR（因DDR bandwidth ~1/10 HBM bandwidth）。(3) **实践规律**：LLM模型的weights有high temporal locality（autoregressive loop中每iteration读取），获最高priority留在HBM。Activation tensors和intermediate results访问次数少（typically once），适合spill。(4) **Compiler annotations**：Compiler标注read-only symbols（如model weights）→ CoE runtime在expert eviction时跳过write-back到DDR，节省带宽。

从编译框架角度拆解术语：
Heterogeneous memory spilling的compiler决策算法：
```
Input: 所有symbols S，HBM capacity C_hbm，DDR capacity C_ddr (假设足够大)
       per symbol s: size[s], frequency[s] (该symbol在program中被访问的总次数),
                    transfer_size[s] = size[s] * frequency[s]

1. 初始分配: 所有s ∈ S分配在HBM
   total_hbm = Σ size[s]

2. if total_hbm > C_hbm:  # 需要spill
      # 计算每个symbol的bandwidth impact
      for s in S:
          bw[s] = size[s] * frequency[s]  # aggregate bytes transferred

      # 按bandwidth升序排列 (低bandwidth的spill影响小)
      sorted_S = sort(S, key=bw, ascending)

      spilled = []
      for s in sorted_S:
          spill s to DDR
          total_hbm -= size[s]
          if total_hbm <= C_hbm:
              break

3. 输出: HBM_set, DDR_set, addresses (per symbol)
```
论文提到，此算法在实践中对LLM models效果良好：weights (high frequency) → HBM，activations (low frequency, large size) → DDR。Compiler也在研究更优化的spilling策略。

术语一般如何实现？如何使用？
- 类似技术：GPU compiler (如XLA, TVM) 的memory planning（register/shared_memory/HBM分配），但通常HBM是唯一的off-chip memory tier（GPU的host DRAM via PCIe不被compiler视为分配目标）。NVIDIA's CUDA Unified Memory提供page-level migration但非compiler-controlled static allocation。
- SN40L的独特之处：(a) DDR直接attached到accelerator（bandwidth 200GB/s vs GPU host 32-64GB/s），使DDR spilling的实际overhead可接受；(b) compiler有global view of entire program（无dynamic dispatch），可做optimal static spilling decision；(c) compiler-read-only annotations + CoE runtime配合实现LRU cache的zero-writeback eviction。
- 通用价值：适用于任何拥有多个不同带宽accelerator-attached memory tier的架构——compiler static analysis可在编译时做出接近最优的数据placement决策，避免runtime profiling/migration overhead。

## Static Bandwidth Model in Dataflow Compiler (数据流编译器中的静态带宽模型)

术语是什么？
Static Bandwidth Model是SN40L compiler中用于建模application requirements和hardware characteristics的编译期分析模型。模型需要量化多个并发data streams的带宽需求：(1) Tile-level：PCU-PMU间、PMU-PMU间的RDN data streams（vector fabric上的tensor data，scalar fabric上的addresses）；(2) Memory-level：AGCU到HBM/DDR的load/store data streams；(3) Inter-socket：P2P links上的cross-socket data streams；(4) Die-to-Die：D2D Interface上的cross-die data streams。每个stream消耗对应的物理资源（RDN switch ports、HBM channels、P2P links等），model用于进行bandwidth allocation和traffic shaping，防止overprovisioning（浪费资源）或underprovisioning（造成bottleneck）。建模是static的——所有分析在compile time完成。

从编译框架角度拆解术语：
Static Bandwidth Model的compiler工作流程：
```
1. 提取Application Bandwidth Requirements:
   对每个fused kernel的数据流图：
   for each edge (producer_unit, consumer_unit, tensor_tile):
       bw_req = tensor_tile.size / pipeline_cycle_count
       add_constraint(path(producer→consumer), bw_req)

2. 建模Hardware Bandwidth Resources:
   RDN vector fabric: per-switch port capacity, per-route capacity
   HBM: controller count × per-controller bandwidth
   DDR: channel count × per-channel bandwidth
   P2P: link count × per-link bandwidth
   D2D: interface width × clock frequency

3. 求解Bandwidth Allocation:
   for each hardware resource r:
       total_demand[r] = Σ bw_req for all streams using r
       if total_demand[r] > capacity[r]:
           # 需要throttle低priority streams 或 增加resource allocation
           apply_throttle(low_priority_streams) or
           re-PnR to use alternative routes
       else:
           allocate bandwidth proportionally

4. 输出: Per-stream bandwidth allocation, throttle parameters, 
        PnR routing decisions
```
模型的secondary benefit：applications可被statically analyzed and tuned for performance to first order（编译期即能判断性能bottleneck，无需runtime profiling）。

术语一般如何实现？如何使用？
- 类似技术：TVM/MLIR的memory planning（但通常只关注memory capacity，非bandwidth allocation）。Google TPU compiler的HBM bandwidth modeling（但TPU的on-chip interconnect规整，建模更简单）。Intel FPGA OpenCL compiler的channel bandwidth allocation。
- SN40L的独特需求：RDN的2D mesh + credit-based flow control使bandwidth allocation成为multi-commodity flow问题——compiler需同时考虑route congestion和bandwidth contention。论文提到"bursty traffic can easily slow down the entire kernel if left unmanaged"——static bandwidth model提供traffic shaping指导（programmable packet throttling parameters）。
- PMU bank conflict elimination：Static bandwidth model的另一应用——识别multi-buffer PMU配置中的bank conflicts，通过programmable bank bits remapping消除。
