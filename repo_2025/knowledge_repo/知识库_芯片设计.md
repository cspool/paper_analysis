## Hybrid Bonding (HB) for 3D Chip Integration

术语解释
Hybrid Bonding (HB，混合键合) 是一种3D芯片堆叠集成技术，通过Cu-Cu (铜-铜) 直接融合键合将DRAM die垂直堆叠在logic die之上，无需传统的micro-bump或TSV中间层。

术语是什么？
Hybrid Bonding是新一代芯片堆叠互连技术。其核心原理是在两个die的接触面直接通过Cu-Cu金属扩散键合实现电气连接，同时通过氧化物-氧化物键合实现机械固定。相比2.5D集成（如HBM使用硅中介层+micro-bump+TSV），HB的关键优势包括：(1) **极高I/O密度**：3μm pitch下可达110,000 I/O/mm²，远高于micro-bump的40-50μm pitch；(2) **低寄生电容**：直接Cu-Cu连接消除了bump和TSV的寄生电容，显著降低I/O功耗（H2-LLM论文中HB I/O energy为0.88pJ/bit，远低于LPDDR5的7.0pJ/bit）；(3) **更薄的堆叠高度**：无bump/TSV中间层使整体封装更薄。HB最早由Sony在图像传感器中商业化，后由TSMC、Samsung、Intel等厂商推进到3D IC和存储器堆叠领域。在AI加速器场景中，HB使DRAM-on-Logic的NMP成为可能——在logic die上定制PEs并通过HB超高带宽访问堆叠上方的DRAM。

从芯片设计角度拆解：
在H2-LLM中，HB的芯片级设计流程：
1. **DRAM die**：包含多个DRAM banks，每个bank底部有HB I/O接口（Cu-Cu pillar阵列），数据通过HB I/O垂直传输到下方logic die。
2. **Logic die**：包含NMP PE array + HB controllers。每个PE通过其HB controller连接上方一个DRAM bank，实现PE-to-bank一对一映射。HB controller负责生成DRAM命令（ACT/RD/WR/PRE）并通过HB I/O驱动数据传输。
3. **HB I/O Architecture**：HB I/O pin数（128/256/512/1024）决定per-bank bandwidth（6.4-51.2GB/s @0.4Gbps/pin）。更多pins→更高bandwidth但controller面积更大。40nm工艺下1024 pins的controller占PE面积约40%，构成intrinsic computation-bandwidth trade-off。
4. **Cu-Cu Pillar布局**：pillar以3μm pitch排列在DRAM die底面和logic die顶面的对应位置，通过wafer-to-wafer或die-to-wafer bonding工艺对准并热压键合。
5. **Power Delivery**：logic die通过package substrate的bumps供电，DRAM die通过HB连接从logic die获取power（或部分从TSV供电）。

术语一般如何实现？如何使用？
HB在商业上主要由以下foundry提供：TSMC 3D Fabric (SoIC)、Samsung X-Cube、Intel Foveros Direct。实现流程：(1) 两片wafer分别制造DRAM和logic；(2) CMP平坦化表面暴露Cu pad；(3) 等离子激活表面；(4) 室温下对准并键合（oxide-oxide bonding）；(5) 退火使Cu-Cu形成金属键合。在AI加速器设计中，HB用于：H2-LLM的HB-NMP（DRAM-on-Logic NMP）、Graphcore IPU、Tenstorrent等。HB的area overhead主要来自：controller逻辑（15-40% logic die area）和Cu pillar的keep-out zone。论文作者通过真实芯片tape-out [55]获得了40nm下的HB面积/功耗数据。

## Row-Buffer Utilization (DRAM Row-Buffer Hit/Empty/Miss)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
DRAM Row-Buffer是每个DRAM bank内部的一行sense amplifier array（行缓冲器），保存最近激活的row（通常8K-16K bits）的全部数据。当内存控制器发出column read/write命令时，数据直接从row-buffer读取（无需再次等待row activation延迟tRCD）。row-buffer的状态分为三类：(1) **Row-buffer hit**：请求的row已在buffer中→仅需column access延迟(tCL)，最快； (2) **Row-buffer empty**：buffer未被激活→需先将目标row激活(precharge→activate, tRP+tRCD)，中等延迟； (3) **Row-buffer miss (conflict)**：buffer中已有其他row→需precharge旧row+activate新row(tRP+tRCD)，最慢。Row-buffer是DRAM中关键的performance-determining结构——它的hit/miss率直接决定了平均内存访问延迟，而hit/miss率取决于访问pattern（顺序 vs 随机）和并发负载（多核/多请求流交叉）。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
在DRAM芯片物理组织中，每个bank包含一个row-buffer（sense amplifier strip），典型的DDR4 8Gb x8 chip有8个bank，每个bank的row-buffer尺寸 = column width × number of columns（如 8-bit × 1024 = 8Kb per row-buffer, DDR4 page size通常为1KB）。Row-buffer的操作流程：(1) **Precharge**：将当前row-buffer内容写回DRAM cell array→bitlines均衡到Vdd/2，耗时tRP (≈13-18ns DDR4)；(2) **Activate**：选中wordline→charge sharing between cell capacitors and bitlines→sense amplifiers latch full row into row-buffer，耗时tRCD (≈13-18ns DDR4)；(3) **Column Access**：从row-buffer中通过column decoder选择具体列→经I/O gating输出，CAS latency tCL (≈14-16ns DDR4)。Row-buffer的hit/miss/empty statistics实际测量了DRAM访问pattern的locality。Mess paper详细分析了row-buffer状态分布：在Intel Skylake上，100%-read低带宽时hit/empty/miss=84/13/3%→高带宽时降至约60/20/20%；当write traffic增加时hit率恶化。DRAMsim3模拟器在此处存在显著误差——其row-buffer hit率异常偏高(84-93%)且与实际分布趋势不一致（intermediate read/write ratio反而hit率最低），导致模拟的bandwidth-latency曲线严重偏离实际。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
Row-buffer的物理实现是CMOS sense amplifier（交叉耦合反相器latch），面积小但在每bank中重复数千次。在芯片设计层面，row-buffer的影响通过以下方式被管理：(1) **DRAM地址映射**：memory controller将物理地址的某些bits映射为bank/row/column——好的映射使顺序访问落在同一row上最大化row-buffer hit（open-page policy）或分散到不同bank实现bank-level parallelism；(2) **内存控制器调度**：FR-FCFS（First-Ready, First-Come-First-Serve）等调度策略优先服务row-buffer hit请求以最大化吞吐——但也可能引入公平性问题；(3) **Row-buffer管理策略**：open-page policy（保持row激活等待后续访问）vs closed-page policy（每次访问后立即precharge）——前者适合locality好的workload，后者适合随机访问。测量row-buffer状态需要硬件计数器支持（Intel Skylake提供perf事件 unc_m_rpq_inserts/occupancy等），Mess paper利用这些计数器深入分析了模拟器row-buffer模型的误差，揭示了DRAMsim3/Ramulator模拟器不准确的根本原因之一。

## JEDEC DRAM Timing Constraints (JEDEC DRAM时序约束)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
JEDEC (Joint Electron Device Engineering Council) 是全球半导体行业标准化组织，其JC-42委员会制定的DRAM时序标准定义了DDRx/LPDDRx/HBM/GDDRx等各代内存的电气和时序参数。关键的时序约束分为：(1) **同bank时序**：tRCD（RAS-to-CAS delay, row activate→column read）、tCL（CAS latency, column read→data valid）、tRP（row precharge time, precharge→bank idle）、tRAS（row active time, activate→precharge最小间隔）、tWR（write recovery time, 最后一次write→precharge最小间隔）、tWTR（write-to-read turnaround, 同bank write→read最小间隔）；(2) **同rank跨bank时序**：tFAW（four active window, 短时间内最多4次bank activate）；(3) **同channel/rank时序**：tRRD（row-to-row delay, 不同bank连续activate最小间隔）、tCCD（CAS-to-CAS delay, 连续column命令最小间隔）。Mess paper指出写操作相关的时序约束（tWR、tWTR）是造成mixed read/write流量性能低于纯读的根本原因——写操作完成后需额外等待时间才能precharge或read同bank的另一row。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
JEDEC时序约束根植于DRAM cell的物理特性（电容充放电时间、bitline settling time、sense amplifier恢复时间等）。以DDR4-2666 (tCK=0.75ns) 为例的关键时序链路：(1) **同bank读**：PRE (tRP=13.75ns) → ACT (tRCD=13.75ns) → RD (tCL=14ns) → data burst → 可选下一个RD (tCCD_L=5.6ns)；(2) **同bank写**：PRE→ACT→WR (tCWL=11ns) → data burst → 必须等tWR (15ns) 后才能precharge；(3) **read-to-write**：需等tRTW (read-to-write turnaround)；(4) **write-to-read**：需等tWTR_L (≈7.5ns, 即10×tCK)。这些约束在芯片设计时通过state machine（时序控制器）硬编码到DRAM device中。DRAMsim3/Ramulator/Ramulator2等cycle-accurate simulartor的核心任务就是正确建模这些时序——它们经过Verilog model验证，保证不违反JEDEC时序。然而Mess paper发现，**时序正确≠性能正确**：模拟器虽然不违反JEDEC timing，但无法正确复现row-buffer hit/miss率→影响平均延迟和带宽→产生与真实系统数十个百分点的偏差。这说明JEDEC时序是必要但非充分的建模要素，更高层的访问pattern-driven behavior（如row-buffer usage）同样关键。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
JEDEC标准文档为各代内存定义完整的时序参数表格（如JESD79-4 for DDR4, JESD79-5 for DDR5），制造商（Samsung/SK Hynix/Micron）的DRAM datasheet列出具体芯片的参数值（通常比JEDEC标准更优，如更低的tCL和更高的频率等级）。在模拟器中使用：(1) DRAMsim3通过配置文件（.ini）指定全部时序参数；(2) Ramulator通过C++代码中的struct定义DRAM standard（如DDR4, `tRCD=13.75ns`等）；(3) 时序参数也可通过SPD（Serial Presence Detect）EEPROM直接从DIMM上读取。对benchmarking，理解JEDEC时序能解释bandwidth saturation行为：write traffic加重时tWR/tWTR使bank忙碌时间更长→bank conflict概率增加→row-buffer hit率下降→有效带宽降低且延迟上升。这是Mess paper观测到的"write traffic使饱和点提前"的物理根源。CXL memory expander和传统DDRx在时序上有本质不同——CXL是全双工互联（独立读写link），其带宽-延迟曲线在balanced read/write时最优（而非pure read），因此需要独立的性能模型。

## Hierarchical Binary Tree Chip Organization for PIM Communication

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Hierarchical Binary Tree Chip Organization（分层二叉树芯片组织）是 CoCoTree 提出的 DIMM PIM 系统中用于集体通信的分层芯片互连结构。该组织匹配 DIMM 的内在物理层次（bank → chip → rank → DIMM），在每个层级通过完美二叉树连接 PEs：(1) Chip-level：每个 PIM chip 内含 N 个 Co-Leaf（嵌入各 bank PE，N=8）和 N-1 个 Co-Node（通过片上连线形成芯片内完美二叉树）。(2) Rank-level：M 个 chip（M=8）通过双向 SerDes 链接（GRS 单端串行 [Poulton et al., JSSC 2019]，25Gb/s/pin，reach 80mm on PCB）连接到含 M-1 个 Co-Node 的 rank-level CoCoTree chip。(3) DIMM-level：多个 rank 进一步通过 Ribbon Cable（16Gb/s/pin，reach 500mm [Gao et al., PACT 2015]）组成 DIMM-level 二叉树。这种 LEGO-like composable 和 self-similar hierarchical organization 使 CoCoTree 能从小规模（8 PE/chip）扩展到大规模（2048+ PE，20 DIMMs）。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
CoCoTree 的分层芯片组织如何将 DIMM 层次映射为通信树：

1. **Intra-Chip Tree**（芯片内）：UPMEM 每个 PIM chip 含 8 个独立 bank（8 PE），CoCoTree 在芯片内布置 8 Co-Leaf + 7 Co-Node 形成 perfect binary tree。PE 的 local data 通过 Co-Leaf 进入 tree → 经 3 层 Co-Node 处理（log₂8=3）到达 chip-level root Co-Node。此树通过片上连线实现，无需额外芯片。

2. **Inter-Chip Tree**（芯片间/rank 内）：一个 rank 含 8 个 PIM chip，8 个 chip-level root Co-Node 通过双向 SerDes 连接到 rank-level CoCoTree chip（含 7 个 Co-Node 形成 rank 级 tree）。每个 chip root Co-Node 的 parent 接口连接 SerDes → PCB trace → rank-level CoCoTree chip 的 child 接口。SerDes 使用 GRS 技术，25Gb/s/pin，支持 80mm PCB 走线。

3. **Inter-Rank Tree**（rank 间/DIMM 内）：一个 DIMM 含 2 ranks，rank-level root 通过 Ribbon Cable (500mm, 16Gb/s/pin) 连接 DIMM-level CoCoTree root。

4. **跨 DIMM Tree**：服务器最多 20 UPMEM DIMM，进一步通过 Ribbon Cable 或 host-level 连接实现 system-level tree。

CoCoTree 芯片设计的 modularity：(1) Functional Modularity：类 RISC-V 模块化 ISA，Co-Node FU 功能按需配置（B/T/RB/RSU/RSF/RCU/RCF 模块），控制面积/功耗 trade-off。(2) Structural Modularity：同构 Co-Node/Co-Leaf 模块像 LEGO 积木一样按 DIMM 层次堆叠，每层 tree 结构 self-similar——N 个 child + N-1 个 Co-Node 形成完美二叉树。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
分层芯片组织的实现：(1) Co-Node 与 Co-Leaf 使用 Chisel RTL 编写，编译为 Verilog 后进行 Yosys 综合 + iEDA 物理实现（NanGate45 45nm 工艺）。(2) Co-Node 部署位置：chip 内 Co-Node 集成在 buffer chip 或 memory controller 中，rank-level CoCoTree chip 为独立芯片通过 SerDes 连接各 PIM chip。(3) SerDes 链接参数：rank-level 用 GRS 单端串行 (25Gb/s/pin, 1.17pJ/b)；DIMM-level 用 Ribbon Cable (16Gb/s/pin)。(4) 硬件开销：8-PE chip tree（7 Co-Node + 8 Co-Leaf）总开销 0.20mm²/9.76mW（45nm），相对 PIM bank 面积功耗可忽略。结构适用于任何按 bank-chip-rank-DIMM 层次组织的 DIMM PIM 系统。

## Fine-Grained DRAM Activation for Processing-Using-DRAM

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Fine-Grained DRAM (FG-DRAM)是一种DRAM访问技术，允许以小于完整DRAM row的粒度进行行激活和数据访问。传统DRAM的每次ACT操作同时激活一个subarray内所有mats的所有columns（如128 mats × 512 columns = 65536 bit）。FG-DRAM利用DRAM的层级结构——每个subarray由多个独立的DRAM mat（2D cell阵列，512-1024 rows × 512-1024 columns）组成，每mat有自己的local row decoder和local sense amplifier——通过修改DRAM访问电路，仅激活选定mat(s)的rows和columns。MIMDRAM将FG-DRAM引入PUD计算：mat isolation transistor切断global wordline到特定mat local row decoder的连接，row decoder latch存储local row地址，mat selector根据logical mat range [mat_begin, mat_end]断言对应mat的isolation transistor。这使得每次PUD操作的SIMD宽度从"全部columns"变为"选定mat(s) × columns_per_mat"，实现可变粒度计算。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
FG-DRAM for PUD的芯片级运转流程（以执行mat0上的TRA为例，参考MIMDRAM图4）：
1. **MAT Range编码**：memory controller发出TRA命令+logical mat range [mat_begin=0, mat_end=0]→14-bit编码（7-bit mat_begin + 7-bit mat_end），其中3 MSB用于chip select（识别8个chip中的目标chip），4 LSB用于mat addressing。
2. **Chip Select Logic**：每DRAM chip内4个comparators+2 AND gates+2 2:1 MUX+3-bit chip ID register→判断本chip是否在mat range内→若是则转发physical mat range到mat queue。
3. **Mat Queue Buffering**：因DDR4 C/A bus仅27 pins且ACT期间多数被占用，mat信息通过ACT-enqueue/PRE-enqueue与已有ACT/PRE命令重叠传输（ACT-ACT overlap或PRE-ACT overlap），暂存在每chip的mat queue中。
4. **Mat Selector + Isolation Transistor**：mat selector收到physical mat range→断言mat0的matline→mat0的isolation transistor打通global wordline→local row decoder latch存储行地址→mat0的local wordline仅驱动mat0内部→仅mat0的rows参与TRA。其余mats不参与→可并发执行其他操作。
5. **缩放**：1 mat = 512 columns SIMD；2 mats = 1024；128 mats = 65536（传统全row模式）。编译器根据循环的max vectorization factor选择最优mat数。

FG-DRAM for PUD的实现要素：

## Inter-Mat and Intra-Mat Data Movement Interconnects in DRAM (GB-MOV / LC-MOV)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
MIMDRAM在DRAM subarray内实现两种低成本的data movement interconnect，解决传统PUD无法在DRAM内部做列间数据搬移的问题。(1) Inter-mat interconnect（全局I/O搬移）：利用global row buffer中SA驱动能力高于local SA的物理特性，在每4个1-bit SA组间加入2:1 MUX连接邻接SA组，实现跨mat的4-bit列数据搬移，对应GB-MOV命令。(2) Intra-mat interconnect（局部I/O搬移）：复用DRAM mat已有的HFFs (Helper Flip-Flops)和column select logic (CSL)通路——通过保持HFF enable信号为high使HFF不重新latch数据而是drive已存数据回写local SA，实现同mat内4-bit列数据搬移，对应LC-MOV命令。

Inter-mat GB-MOV的芯片级执行流程（从mat_M-2搬移4-bit到mat_M-1）：
1. **Step 1**：memory controller向mat_M-2的rowsrc发出ACT→rowsrc数据load到mat_M-2的local SA。同时向mat_M-1的rowdst发出ACT→rowdst load到mat_M-1的local SA。
2. **Step 2**：向mat_M-2发出RD(columnsrc)→CSL选通4-bit columnsrc→HFFs latch并amplify this 4-bit→drive到global row buffer的SA group SA_M-2。
3. **Step 3**：向mat_M-1发出WR(columndst)（标记为GB-MOV命令）→2:1 MUX选择来自SA_M-2的datapath（而非I/O bus）→SA_M-2的data load到SA_M-1→drive mat_M-1的HFFs→HFFs drive mat_M-1的local SA→local SA restore到rowdst的bitlines→完成写回。
4. **延迟**：tRAS + tRELOC + tWR + tRP（最差case，源和目标行不同时）。
5. **Step 4**：重复step 1-3，每次搬移4-bit（因每mat有4 HFFs），直到全量数据搬完。

Intra-mat LC-MOV的芯片级执行流程（mat_M内从rowsrc搬移到rowdst）：
1. **Step 1**：ACT rowsrc→RD columnsrc→CSL选通→HFFs latch 4-bit data→PRE rowsrc。不降低HFF enable信号→数据保持在HFFs中。
2. **Step 2**：ACT rowdst→WR columndst→CSL选通columndst→HFF enable保持high→HFFs overwrite（而非latch）local SA data→新data propagate到rowdst的DRAM cells→PRE rowdst。
3. **延迟**：2×(tRAS + tRP) + tRELOC + tWR（最差case）。

## ReRAM In-situ Computing (ReRAM存内计算: 1T1M阵列, Analog vs Digital)

## NPU-PIM Unified Memory System (NPU-PIM统一内存系统)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
NPU-PIM Unified Memory System是IANUS提出的芯片级内存组织方案，指PIM（Processing-in-Memory）内存芯片同时充当NPU主存和PIM计算内存的物理统一编址架构。与传统的Partitioned Memory System（host NPU专用DRAM + PIM专用内存分离，shared data需duplicate在两处）不同，Unified Memory将全部物理内存容量赋予PIM chips，NPU通过NoC直接访问PIM内存获取weight/activation数据，同时PIM computation也在同一PIM芯片内执行。这种组织使内存容量利用率翻倍（LLM中FC参数占91%，unified vs partitioned ≈ 2× 内存节省），并完全消除NPU-PIM间的数据重复和搬移。代价是normal memory access和PIM computation不能并发——PIM bank同一时刻只能服务一种操作——需PIM Access Scheduling协调。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
IANUS的Unified Memory物理组织：每NPU连接8个PIM channel（基于GDDR6-AiM），每个PIM channel含16 banks。通过全互联NoC（all-to-all connection），NPU的任意core可访问任意PIM channel。PIM内存地址采用PIM-aware mapping：(MSB)Row-Channel-Bank-Column(LSB)，使同一tile内数据共享row address避免row conflict，同时分布到不同channel/bank以利用all-bank和all-channel并行。运转流程：(1) NPU MU/VU需要weight→通过DMA发出normal memory read→PIM MC将地址映射到对应的channel/bank→按标准GDDR6时序读取数据→NoC传回NPU scratch-pad；(2) PIM computation时→macro PIM cmd指定channel/bank→PCU解码→PIM MC按GDDR6时序+新增PIM states发出bank-level compute commands→所有指定banks并行执行MAC→结果存回PIM memory；(3) Scheduling关键：当macro PIM cmd ready时，command scheduler强制未issue的DMA commands进入wait状态→PIM bank dedicately服务compute→完成后释放DMA→恢复normal access。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
Unified Memory System的实现需要：(1) PIM芯片硬件同时支持普通内存读写和PIM compute双模操作——GDDR6-AiM在原GDDR6 state machine基础上增加PIM compute states（如MAC active state），PIM MC需track每个bank的两种状态；(2) 地址映射需为PIM操作优化——IANUS的Row-Channel-Bank-Column映射将tile内row分布到不同bank/channel实现bank-level parallelism，但不同于传统mapping（channel在低地址位以支持streaming bandwidth），可能导致非PIM workload时的bandwidth下降——不过IANUS每NPU core主要负责2个PIM channel使多channel并行访问弥补；(3) 需要scheduling硬件（command scheduler + PCU + PIM MC）协调normal和PIM操作的分时复用。这种unified方案特别适用于LLM场景（大量FC参数需在NPU和PIM间共享），但对general-purpose workload可能有performance penalty。类似概念：Chopim [Cho et al., ISCA 2020]也提出host与near-data accelerator间的unified memory，但面向homogeneous kernels而非LLM的heterogeneous kernels。

## PIM-aware DRAM Address Mapping (存内计算感知的DRAM地址映射)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
PIM-aware DRAM Address Mapping是IANUS为最大化PIM计算吞吐量而设计的一种非传统DRAM地址映射方案：(MSB)Row-Channel-Bank-Column(LSB)，与传统的(MSB)Row-Bank-Col_M-Channel-Col_L-Offset形成关键差异。传统mapping将相邻数据交错分布到不同channel（channel地址位置低），以维持streaming bandwidth处理diverse server-class workload。PIM-aware mapping将Row地址放MSB（对应tile index）、Channel和Bank放中间、Column放LSB（对应tile内column index），使同一tile的数据共享相同row地址、分布到不同channel/bank以激活全bank全channel并行PIM计算。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
PIM操作中data allocation和mapping协同设计：(1) Weight matrix（如FC layer）按tile分割——每tile = 16 rows（=banks per channel）× 8 columns（=channels for IANUS），每row最多1024 BF16 elements（=2KB per DRAM row）；(2) 每tile内的row映射到各bank各channel的同一DRAM row address→PIM可同时激活所有16 banks×8 channels=128 rows并行执行MAC；(3) Row address bit作为MSB对应tile index→不同tile使用不同row address→无row conflict；(4) Column bit作为LSB→同一row内1024元素由同一bank的单个PE顺序处理（1 MAC per cycle per PE）。在GPT-2 XL weight矩阵上，该mapping使PIM internal bandwidth达1024 GB/s（8 channels × 128 banks × 1 GHz × 1 MAC × 2B BF16），约为external GDDR6 bandwidth（256 GB/s）的4×。代价是非PIM操作时的bandwidth degradation：传统mapping将spatial locality的连续数据分布在多个channel实现interleaving，PIM-aware mapping则将channel bit上移。IANUS每core主要负责2个PIM channel的访问来补偿这一缺陷。

实现需修改DRAM controller的address decoder和bank/row/column command生成逻辑。对每个physical address：(1) extract tile index = Row bits；(2) extract channel = Channel bits；(3) extract bank = Bank bits；(4) extract column offset = Column bits。GDDR6 command生成时根据mapping定位目标bank→发出ACTIVATE(Row)→READ/WRITE(Column)或PIM_COMPUTE(Column)。与JEDEC标准mapping不兼容，需定制memory controller。IANUS将PIM MC设计为同时支持PIM commands和normal memory commands的双模controller，内部维护各bank状态机（含新增PIM states）并按GDDR6 timing约束（tCK=0.5ns, tCCDS/tCCDL=1ns, tRAS=21ns等）发出命令。

## CXL Memory Expansion for Near-Data Processing

## SRAM + LPDDR Memory Hierarchy for AI Accelerators

术语解释
SRAM + LPDDR Memory Hierarchy是MTIA 2i采用的非传统内存层次设计——以256MB大容量on-chip SRAM (2.7 TB/s) 作为主工作内存，配合64-128GB off-chip LPDDR5 (204.8 GB/s) 作为容量扩展层，避免使用高成本的HBM。

术语是什么？
传统GPU和大多数AI加速器使用HBM（High Bandwidth Memory，如A100用HBM2e 80GB @ 2 TB/s，H100用HBM3 80GB @ 3.35 TB/s）作为主内存，HBM通过硅中介层（silicon interposer）与计算die紧密集成，提供极高带宽（1-3 TB/s）但成本和功耗极高。MTIA 2i的设计反向而行：(1) 大容量SRAM——256MB on-chip SRAM提供2.7 TB/s带宽，满足推荐模型inference的低延迟需求；(2) LPDDR5——64-128GB off-chip LPDDR5提供204.8 GB/s带宽，作为容量扩展，成本远低于HBM；(3) SRAM:LPDDR带宽比约13:1，意味着当working set超出SRAM容量时性能急剧下降；(4) 设计依据——Meta的推荐模型（DLRM/DHEN）exhibit显著data locality：activation buffer可在模型执行期间复用同一块SRAM区域，weights为read-only可被LLC evict而无需writeback→>95% SRAM hit rate achievable。此设计使MTIA 2i在推荐模型推理上的TCO相比GPU降低44%。

从芯片设计角度拆解术语：
MTIA 2i的SRAM+LPDDR memory hierarchy工作流程（以执行一个FC layer为例）：
1. **Data placement决策**（autotuning 离线完成）：activation buffer pin在LLS（software-managed scratch，避免被硬件自动evict），weights放在LLC（hardware-managed cache，允许自动evict无需writeback），LPDDR存放完整权重和embedding tables。
2. **FC layer执行**：DPE需要weight tile→LLC检查→(a) hit：从SRAM LLC直接读取（2.7 TB/s带宽）；(b) miss：从LPDDR读取weight tile（204.8 GB/s带宽），利用DMA prefetch feature提前将后续weight tile从LPDDR预取到SRAM LLC。
3. **Activation管理**：activation在LPDDR中有backing copy但primary working copy保留在LLS——因为activation在多个operators之间复用（lifetime贯穿模型执行），pin在LLS避免writeback→DRAM的延迟惩罚。
4. **Batch size权衡**：更大batch→activation buffer更大→可能超出LLS容量→must spill到LLC或LPDDR→性能下降。因此autotuning选择activation fit LLS的最大batch size。
5. **模型复杂度上限**：当模型超过特定compute intensity时，GEMM本身变为DRAM bandwidth-bound（weight tile miss rate上升），LPDDR的204.8 GB/s成为硬瓶颈→MTIA 2i无法高效运行>2 GFLOPS/sample的模型或LLM decode。

## LLS/LLC Partitioning for On-Chip SRAM

术语解释
LLS (Software-managed Scratch) / LLC (Hardware-managed Cache) Partitioning是MTIA 2i的256MB on-chip SRAM管理机制，将SRAM以32MB粒度分区为两种不同管理策略的区域：LLS保存activation buffer（软件管理，不被自动evict），LLC缓存weights等短期数据（硬件管理，允许自动evict无需writeback）。

术语是什么？
MTIA 2i的全局SRAM不采用统一的管理策略，而是分区混合管理：
- **LLS (Local-Like Scratchpad)**：software-managed scratch memory，数据由软件显式管理其生命周期，硬件不会自动evict。适合存放activation buffer——activation在模型执行期间持续复用（多个operators的输入/输出交替使用同一内存区域），若被硬件自动evict需writeback到LPDDR，代价极高。
- **LLC (Last-Level Cache)**：hardware-managed cache，类似传统CPU/GPU的L2/L3缓存，硬件自动管理eviction。适合存放weights——weights为read-only（推理中不变），被LLC evict时无需writeback，仅下次需要时重新从LPDDR fetch。LLC还接收hardware hints（如标记tensor不会被复用→skip writeback）。
- 分区粒度：32MB per region。Autotuning框架自动决定LLS大小——通常设置为恰好容纳整个activation buffer（同一块内存可复用为不同时刻的不同activation tensors，因它们的lifetime不重叠）。

从芯片设计角度拆解术语：
一个推荐模型的LLS/LLC分区决策流程：
1. **Profiling阶段**：offline测量模型的activation buffer峰值大小（考虑所有operators activation的liveness range）和weight total size。
2. **Autotuning决策**：尝试将LLS设为activation buffer大小→若LLS+LLC=256MB能同时容纳activation和热weights→性能最优（activation pin + weight cache hit）。若activation buffer > 256MB→尝试降batch size使activation fit LLS→对比大batch+activation in LLC vs 小batch+activation in LLS的throughput→选较优者。
3. **运行时**：activation tensors在LLS中分配（软件通过MTIA memory allocator API显式allocate/free），weight tiles通过LLC自动缓存（硬件管理，utilizing LRU或类似替换策略）。
4. **Writeback优化**：当activation无法完全fit LLS而部分spill到LLC时，利用memory hints告知硬件某些tensor的eviction不需要writeback→减少LPDDR写带宽消耗。

术语一般如何实现？如何使用？
LLS/LLC partitioning的实现：(1) SRAM controller——在物理上将256MB SRAM按32MB partition划分为region，每个region配置为LLS或LLC模式（通过memory-mapped control register）；(2) LLS寻址——PE通过physical address直接访问LLS区域，地址映射在FI DMA descriptor中指定；(3) LLC寻址——PE通过caching protocol访问LLC（类似L2 cache的tag lookup + data access），miss时自动触发LPDDR fill；(4) Autotuning集成——MTIA autotuning framework自动选择partitioning方案（论文描述为简单启发式：LLS=activation buffer size，余量为LLC），未来可探索更精细的per-tensor placement tuning。类似概念：(a) GPU shared memory vs L1 cache partitioning（NVIDIA支持可配置的SMEM/L1 split）；(b) Intel Sapphire Rapids的shared L3 cache部分可配置为pseudo-L4 cache；MTIA 2i的特色在于partitioning主要用于区分activation（pin）和weight（cache）两种语义不同的数据类型，使得在大SRAM中精准控制数据驻留。

## Rank-level Processing Module (RPM) for DIMM-based PIM

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Rank-level Processing Module (RPM) 是HeterRAG在AccelDIMM中提出的diMM rank级in-memory processing单元，集成在DDR4 DIMM的buffer chip上，每个rank配备一个独立的RPM。RPM的核心功能是解码PIM指令并在DRAM rank本地执行vector distance computation，消除跨DIMM interface的数据搬运。单RPM含：PIM-Inst Queue (128-entry) → PIM-Inst Decoder → Vertex Cache (128KB SRAM, LRU, 分local/global) + DDR-C/A Generator (生成DDR ACT/RD/PRE命令) + Distance Computation Unit (32 FP32并行乘法器 + 32 FP32加法器，支持inner product距离)。选择rank级而非bank级deployment的原因：ANNS的sparse/irregular memory access pattern在bank级部署会导致大量idle compute resources、更高面积开销和有限性能收益。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
RPM在PIM-enabled DIMM芯片中的运转流程：
1. **指令接收**：DPM（DIMM-level Processing Module）按Rank ID将PIM-Inst分发到对应rank的RPM→存入128-entry PIM-Inst Queue→Inst Decoder依次解码。
2. **地址解析**：Inst Decoder从PIM-Inst中提取DRAM addr字段（作为Vertex Cache tag）、DDR command info字段（含bank/row/column地址）。
3. **Cache Lookup**：DRAM addr tag与Vertex Cache (128KB) 比较：
   - **Cache hit**→缓存的vertex vector data直接送入Distance Computation Unit。
   - **Cache miss**→DDR-C/A Generator根据DDR command info生成标准DDR ACT（激活目标row）→RD（读取column data）→PRE（precharge row）命令序列→DRAM返回vector data→同时送入Distance Computation Unit和更新Vertex Cache (LRU)。
4. **Distance Computation**：Distance Computation Unit (32 FP32 Mult + 32 FP32 Adder) 并行计算query vector（存储在Query Reg）与vertex vector的内积距离。内积距离是RAG中最常用的相似度度量；通过minor modifications可扩展支持L2距离等其他度量。
5. **结果输出**：计算得到的(Qid, Vid, Distance)写入DPM中对应rank的Distance Buffer (32KB)。DPM按round-robin方式将各rank buffer结果通过MUX传回TPM Functional Block。
6. **Cache合并**：查询完成后，local cache entries合并到global cache（更新LRU），local cache清空。

RPM的面积开销：3.368mm² per RPM (22nm工艺，从65nm @500MHz综合后scale)。采用Synopsys Design Compiler综合Verilog RTL→65nm面积/power→scale到22nm (+50%面积补偿logic vs DRAM工艺差异)。使用方式：每个DDR4 DIMM rank配备一个RPM，2-rank DIMM共2个RPM。DPM (0.842mm²) + 2 RPM (2×3.368=6.736mm²) ≈ 7.817mm² total per DIMM，相对buffer chip (>100mm²) 可接受。RPM的rank级部署策略是AccelDIMM区别于其他DIMM-based PIM（如TRiM的bank级tensor reduction、RecNMP的bank级sparse gathering）的关键设计选择。rank级避免了bank级在sparse access下的资源碎片化问题。

## Bank-level Processing Module (BPM) for HBM-based PIM

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Bank-level Processing Module (BPM) 是HeterRAG在AccelHBM中提出的HBM bank级in-memory processing单元，集成在HBM2 DRAM die上，每两个bank共享一个BPM。BPM专门执行LLM decode阶段的GEMV操作（低算术强度、memory-bound），利用HBM bank级内部带宽消除数据搬运瓶颈。单BPM含：2个向量内积计算单元（各32 FP16乘法器 + 32 FP16加法器）+ 8Kb buffer。输入数据来源：(1) bank-level row buffer（weight data，从DRAM row中读取）；(2) channel-level global buffer（activation data，从HBM channel的全局buffer中获取）。BPM设计灵感来自state-of-the-art GEMV-optimized PIM accelerators如Newton [MICRO'20]。

BPM在PIM-enabled HBM芯片中的运转流程（以单次GEMV offload执行为例）：
1. **PIM请求广播**：AccelHBM TPM的Request Generator发出PIM请求→broadcast到所有HBM Controller→各Controller将PIM请求下发到其管理的所有BPM。
2. **数据获取**：BPM从bank row buffer读取weight vector（已在之前ACT命令中加载到row buffer）+ 从channel global buffer获取activation vector→数据送入BPM的8Kb input buffer。
3. **向量内积计算**：2个向量内积计算单元并行工作——每个单元32 FP16 Mult同时计算weight[i] × activation[i]→32 FP16 Adder将乘积累加→产生一个部分和（partial sum）。
4. **部分和聚合**：各BPM的部分和沿HBM的hierarchy聚合——bank内BPM结果→channel内聚合→跨channel聚合→最终GEMV结果返回TPM Functional Block。
5. **与GEMM的协同**：GEMM（prefill阶段，算术强度较高）不在BPM执行而在TPM的Matrix Unit (128×128 Systolic array ×8) 执行。选择逻辑：按算术强度划分——GEMV的FLOP/Byte比远低于roofline knee point→适合in-memory execution；GEMM高于knee point→适合TPM systolic array执行。

## CXL-based PIM Device Interconnect (基于CXL的PIM设备互联)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
CXL-based PIM Device Interconnect 是HeterRAG提出的基于Compute Express Link (CXL) 标准的异构PIM设备互联方案。AccelDIMM和AccelHBM作为CXL Type 2 devices（类似GPU/FPGA），通过PCIe slots连接到CXL switch，实现host与device、device与device之间的高速数据传输。三种CXL子协议同时使用：cxl.io（设备发现/配置/MMIO）、cxl.cache（device cache host memory）、cxl.mem（host访问device memory）。单链路PCIe 5.0 ×8提供约32GB/s带宽。CXL switch网络可支持最多4096节点 (4K nodes)。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
CXL-based PIM互联在HeterRAG中的芯片级运转流程：
1. **拓扑结构**：Host CPU ↔ CXL Switch ↔ [AccelDIMM₀, AccelDIMM₁, ..., AccelDIMMₙ, AccelHBM₀, AccelHBM₁, ..., AccelHBMₘ]。所有PIM device通过PCIe 5.0 ×8 lanes连接到CXL switch。
2. **数据传输path**：
   - **Host→AccelDIMM**：query vectors通过cxl.io/cxl.mem广播到所有AccelDIMM device（少量数据，BW需求低）。
   - **AccelDIMM→Host**：各device返回top-k结果（每个device ~100 results × 8KB total），aggregation在host CPU完成（BW压力小）。
   - **Host→AccelHBM**：编码后的input tensors通过cxl.mem发送到AccelHBM（数据量取决于token数和模型dimension）。
   - **AccelHBM→Host**：生成的token sequences返回host（autoregressive逐个token，BW需求不高）。
3. **关键设计理由**：CXL的32GB/s带宽远低于HBM内部带宽(~256GB/s)和DDR4 DIMM内部带宽(~19.2GB/s)，因此HeterRAG的设计原则是——仅在CXL上传输query/result等少量数据，检索/生成的计算和数据访问完全在PIM device内部完成，避免通过CXL搬运大块数据。这区别于仅用DIMM做存储+通过CXL搬运到HBM-PIM处理的NaiveHBM方案（该方案CXL成为瓶颈，性能甚至不如CPU-GPU baseline）。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
CXL互联的实现基于CXL Consortium标准 (https://computeexpresslink.org/cxl-specification)。CXL Type 2 device需在device侧实现CXL controller IP（含cxl.io/cxl.cache/cxl.mem三种protocol处理逻辑）和PCIe 5.0 PHY。使用方式：(1) 设备枚举——启动时host通过cxl.io枚举CXL switch下所有device，分配MMIO地址空间；(2) 数据传输——host通过cxl.mem直接读写device attached memory（如将query写入AccelDIMM的buffer memory），device通过cxl.cache可选地cache host memory pages；(3) 扩展——通过增加CXL switch ports扩展device数量，线性增加系统总容量和计算能力。CXL-based PIM架构也见于其他PIM系统如NeuPIMs和IANUS，是实现可scale异构PIM的标准互联方案。

## Two-Bank Shared PIM Architecture with Access Interleaving

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Two-Bank Shared PIM Architecture 是 Pimba 的 PIM 芯片级架构设计，每个 State-update Processing Unit (SPU) 同时连接两个 DRAM bank，通过 access interleaving 技术交替从两个 bank 的 row buffer 读取和写入，在不超过 25% 面积约束的前提下实现与 per-bank 设计相同的计算吞吐。SPU 数量从 banks 减少到 banks/2（面积减半），同时通过 4-stage pipeline 和 access interleaving 维持 100% SPU 利用率。采用 all-bank PIM 设计——所有 bank 同时配备 SPU（banks/2 个），与 HBM-PIM、Newton 等产业标准一致。关键硬件模块：SPU-to-bank MUX/DEMUX（根据 iteration parity 切换读写目标 bank）、SPE（MX Multiplier + MX Adder + Dot Product Unit）、扩展 DRAM command decoder（支持 5 条 custom commands）、Quantization Unit（host 端 memory controller 中，fp16→MX8 转换）。

Pimba PIM 芯片在 HBM2E DRAM die 上的物理集成：
1. **芯片层次映射**：HBM stack 含 8 channels，每 channel 4 bank groups × 4 banks = 16 banks。Pimba 在每 2 banks 间部署 1 SPU → per channel 16/2=8 SPUs；per HBM stack 8×8=64 SPUs。
2. **面积预算（10nm 等效）**：Per bank DRAM ≈ 0.69 mm²。Per SPU = compute 0.053 mm² + buffer 0.039 mm² = 0.092 mm²。Per bank 有效开销 = 0.092/2 = 0.046 mm²。Area overhead = 13.4%（含 buffer area）。

6. **Attention 支持**：同一 SPU 也支持 attention 操作（score 用 dot product unit、attend 用 multiplier+adder），实现统一微架构服务于两类 LLM 操作。

## HBM3 Core Die / Logic Die Dual-Processor Integration (PIM+PNM共集成)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
HBM3 Core Die / Logic Die Dual-Processor Integration是在单个HBM3 stack内部同时在core die上集成PIM（Processing In Memory）处理单元、在logic (buffer) die上集成PNM（Processing Near Memory）加速器逻辑的芯片级设计。HBM3的3D堆叠结构天然包含多层core die（存储cell阵列，通常8-Hi即8层）和1层logic die（位于堆叠底部，含PHY、TSV接口、MBIST等I/O逻辑）。传统HBM3的logic die仅承担I/O功能，面积利用率约52.3%（即约47.7%的logic die面积空闲 [45]）。MNM利用这一空闲面积在logic die上放置PNM加速器，同时在core die的bank旁放置PIM MAC单元，形成单stack内的PIM+PNM双处理器异构集成。区别于将PIM和PNM放在不同DIMM/HBM device的分离式异构方案（如HeterRAG [60]），MNM的PIM和PNM共享同一个HBM stack的TSV带宽和散热路径，通过dual row buffer解决core die上PIM MAC和PNM读取的bank争用问题。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
在MNM中的芯片组织流程：
1. **Core die层（8层）**：每层core die含16个bank（per channel 4 bank groups × 4 banks = 16 banks per channel, 16 channels per stack = 256 banks total per stack）。每bank旁放置1个16-bit FP MAC单元（PIM logic），占用bank area的一部分（MAC 0.11mm² + dual row buffer 1.38mm² = per bank overhead ≈ 1.49mm² per core die × 16 banks = 14.4mm² MAC + 3.8mm² dual row buffer = 18.2mm² total per core die / 121mm² standard core die = 15.0% area overhead）。PIM logic用DRAM工艺（1z-DRAM节点），面积需从14nm logic合成结果×10缩放（account for FEOL large transistor + BEOL less metal layers/routing pitch [14]）。
2. **Logic die层（1层）**：位于HBM stack底部，含PHY（物理层接口，驱动external I/O @5.2Gb/s/pin）、TSV interface（连接上方core die的through-silicon vias）、MBIST/IEEE 1500测试电路、decoupling capacitor、interposer driver等标准组件。MNM在logic die上placement：16个PQ code scanner + 1个Top-k selector + 1个Softmax calculator + Precomputed LUT SRAM + PQ codebook SRAM + MNM Controller。Logic die用标准逻辑工艺（ASAP 7nm PDK合成），area overhead 2.0% of HBM3 logic die area。
3. **TSV互联**：core die的DRAM banks通过TSV向下连接到logic die的PHY→external memory bus。PNM的PQ code scanner通过TSV从core die读取PQ codewords（利用standard memory read path），PIM的MAC结果通过global bus→TSV送到logic die的Softmax calculator→结果再通过TSV送回core die的global vector buffer。
4. **Thermal-aware placement**：PNM逻辑远离PHY和TSV区域（主要热源，junction temperature ~101-102°C），通过1D layered-conduction model估算PNM logic自身温升0.38-0.47°C→避免在HBM3 105°C温度准则下触及thermal throttling。
5. **Dual row buffer的芯片级实现**：每bank布置2条row buffer（原本1条），1条用于PIM MAC操作（data from activated row→MAC unit），另1条独立用于PNM memory read（读取PQ codewords）。Buffer select信号由MNM Controller根据当前command类型（PIM vs PNM vs standard I/O）动态切换。Dual row buffer增加3.8mm² per core die area（比单row buffer多约4.2% core die area），但使PIM和PNM可真正并发执行。

## Data-Reuse-Driven Multi-FPU PIM Bank Design for LLM Decoding

术语解释
PAPI提出的利用LLM decoding parallelism（batching和speculative decoding）带来的DRAM data reuse来降低per-access DRAM energy，从而在HBM power budget内支持更多FPUs per DRAM bank的PIM chip设计方法。

术语是什么？
Data-Reuse-Driven Multi-FPU PIM Bank Design 是PAPI FC-PIM采用4P1B配置（4 FPUs per DRAM bank）而非传统1P1B的关键理论依据。传统PIM（如AttAcc 1P1B）中，DRAM access energy（activate+precharge DRAM row）占总energy的96.7%，无data reuse时1P1B的power consumption已接近HBM3 116W budget。LLM decoding的batching和speculative decoding使同一weight DRAM row被多个request的activation或同一request的多个speculated token复用——当data reuse level ≥ 4时，DRAM access energy从96.7%降至33.1%（因单次activation energy被多次computation均摊），整体power降至<116W，释出的power headroom允许增加FPU数量至4 per bank。

从芯片设计角度拆解：
PAPI的Data-Reuse-Driven Multi-FPU设计流程：
1. **Energy breakdown分析**（baseline 1P1B, no reuse）：
   - DRAM Access: 96.7% (activate + precharge DRAM row for weight read)
   - Transfer: ~2% (activation data via TSV + global controller + bank-group controller)
   - Computation: ~1.3% (FPU multiply-add operations)
   
2. **Data reuse effect**（1P1B, reuse level = 64）：
   - DRAM Access: 33.1% (一次DRAM row activate服务于64次computation)
   - Transfer + Computation: 66.9%
   
3. **Power constraint check**（HBM3 116W budget, IDD7 methodology）：
   ```
   配置 | data reuse=1 | data reuse=4 | data reuse=64
   4P1B | >150W (超标) | <116W (通过) | <<116W (充裕)
   2P1B | ~130W (超标) | <116W (通过) | <<116W
   1P1B | <116W (通过) | <<116W
   ```
   LLM decoding中，batch size ≥ 4或speculation length ≥ 4即产生≥4× data reuse → 4P1B满足power budget。
   
4. **面积约束**（CACTI-3DD, 22nm, FPU=0.1025mm², bank=0.83mm²）：
   ```
   m × (0.1025 × n + 0.83) ≤ 121 mm² (HBM die max)
   4P1B: m × 1.24 ≤ 121 → m ≤ 97 → PAPI选择96 banks (3 BGs × 32)
   1P1B: m × 0.9325 ≤ 121 → m ≤ 129 → 更多banks但更低compute
   ```

术语一般如何实现？如何使用？
实现：(1) 利用LLM parallelism自然产生的data reuse——batching使同一weight被B个request复用，speculative decoding使同一weight被S个token复用；(2) PIM hardware——FPU数量和bank数量通过面积/功耗公式trade off，PAPI选择4P1B + 96 banks (3 bank groups in 8-High HBM3)；(3) HBM3 power verification——严格按JEDEC HBM3 IDD7 methodology验证116W power budget。通用适用：任何PIM加速器设计可通过理解target workload的data reuse特性来配置optimal per-bank compute parallelism。

## Bank Bundle (DRAM Bank Bundle for PIM Concurrent Access)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Bank Bundle是Duplex论文提出的HBM3 DRAM die微架构修改方案，将每个pseudo channel内16个banks分为upper和lower两个bank bundle（各8 banks），通过在bank I/O和bank group I/O之间放置switches分离xPU（如GPU）和Logic-PIM的datapath，允许两者在不同bank bundle上同时进行数据访问。其核心目标是避免修改DRAM bank内部layout（如增加prefetch size和I/O datapath width会带来~77% bank size increase [37]），转而通过增加simultaneously operating banks数量实现4x有效内存带宽。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
Bank Bundle在HBM3 DRAM die中的芯片级组织流程（以8-hi HBM3, 4 DRAM dies per rank, 8 pseudo channels per die为例）：
1. **Pseudo Channel Bank组织**：标准HBM3每pseudo channel含4 bank groups × 4 banks = 16 banks per rank。banks共享external wires（bank I/O和bank group I/O），任一时刻仅1 bank可输出数据——因此标准HBM3 bandwidth = 256 bits per tCCD_S (different bank groups) 或 per tCCD_L (same bank group)。
2. **Bank Bundle划分**：16 banks被分为upper bank bundle（banks 0-7）和lower bank bundle（banks 8-15），每组8 banks。在bank I/O和bank group I/O汇聚点之间插入switches——upper bundle和lower bundle的I/O分别路由到不同的物理路径。
3. **Logic-PIM读取路径**：Logic-PIM发送相同C/A到目标bank bundle的全部8 banks→每bank group贡献2 banks × 256 bits (per bank I/O width) = 512 bits per bank group → 4 bank groups × 512 bits = 2048 bits per bundle per tCCD_L。由于读取来自同bank group，受tCCD_L=4约束（即读取间隔≥4 cycles = 6ns）。数据通过bank group I/O→Logic-PIM-path switch→dedicated TSVs→logic die。
4. **xPU读取路径**：xPU通过conventional HBM controller→xPU-path switch→标准data TSVs→读取其他bank bundle的数据。xPU和Logic-PIM使用不同bank bundle时完全无冲突。
5. **Bank Bundle Index管理**：每pseudo channel有4个bank bundles（2 ranks × 2 bundles per rank），编号1-4。Duplex将device memory space划分为4个section对应bank bundle index——expert FFNs round-robin分配在4 spaces上，KV cache交替分配在3 spaces，第4 space作prefill scratch。Co-processing时xPU和Logic-PIM分配在不同spaces处理以避免bank bundle冲突。
6. **TSV物理布局优化**：Logic-PIM dedicated TSVs放置在power TSV区域（而非data TSV区域）以缩短bank group到TSV的I/O路径长度，减少energy消耗 [37]。TSV pitch从50μm缩至22μm [49]，4x TSV数量仅增加9% area overhead（10.89mm² per Logic-PIM stack）。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
- **DRAM时序适配**：Bank bundle读取需遵守标准HBM3时序约束——tCCD_S=2（不同bank group最小读取间隔）、tCCD_L=4（同bank group最小读取间隔）、tFAW（4 activation window，限制短时间内可激活的bank数）。Duplex选择每bundle 8 banks使同bank group内banks能在tCCD_L约束下pipeline读取：bank 0 ready @cycle 0, bank 1 ready @cycle 4, bank 0再次ready @cycle 8... 通过交错2 banks per bank group实现无bubble的数据流。
- **Bank结构零修改**：与直接增加bank prefetch size相比（quadrupling prefetch→77% bank size increase [37]），bank bundle方案保持DRAM bank内部layout完全不变，仅增加bank I/O level switches（simple MUX/DEMUX逻辑，area overhead negligible）。
- **Memory Space-to-Bank Bundle映射**：Device driver/HBM controller在分配memory时根据bank bundle index放置数据——确保expert co-processing时分配到xPU的expert和分配到Logic-PIM的expert位于不同bank bundle的memory space。
- **与先前方案的对比**：传统PIM如Bank-PIM [28][29]每bank有独立处理单元可实现更高的internal parallelism（全bank同时运算），但处理单元受限（每bank仅少量MACs）且在DRAM die上area overhead大。TOP-PIM [60]也有GPU+PIM concurrent access但未给出详细的DRAM die bank bundle架构。Duplex的bank bundle首次系统性地设计了xPU和Logic-PIM的并发datapath分离。

## Heterogeneous Dataflow Multi-Chiplet Module (MCM) AI Accelerator

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Heterogeneous Dataflow Multi-Chiplet Module (MCM) AI Accelerator 是一种将具有不同数据流（dataflow）类型的AI加速器chiplet集成在同一2.5D硅中介层（silicon interposer）上的多芯片模块架构。与同构MCM（所有chiplet使用相同dataflow）不同，异构MCM在同一个封装内混合集成多种dataflow chiplet（如NVDLA-style compute-optimized dataflow + Shi-diannao-style memory-efficient dataflow），使不同ML operator类型（Conv、FC、Attention等）能够映射到最适合其计算/访存特征的chiplet上执行。每个chiplet是独立AI加速器（含PE array、on-chip memory、Network-on-Chip），拥有特定的loop ordering、parallelization、tiling策略（即dataflow）。chiplet间通过Network-on-Package (NoP)互联，chiplet位于MCM两侧具备off-chip DRAM接口。SCAR论文的MCM参考Simba [64]的架构设计（36-chiplet prototype，16nm FinFET，每chiplet 4 TOPS），但扩展为支持异构dataflow chiplet集成。异构MCM的核心优势在于workload adaptivity——多模型工作负载中不同类型的ML operator可被调度到最匹配的dataflow chiplet上，而非所有operator强制适应单一dataflow。

Heterogeneous Dataflow MCM的芯片设计运转流程（以SCAR的Het-Sides配置为例，6×6 chiplet MCM）：
1. **Chiplet类型定义**：MCM集成|DF|种dataflow类型的chiplet（SCAR中|DF|=2：NVDLA-style和Shi-diannao-style）。每种dataflow类型通过其loop ordering（如K→C→H→W→R→S for weight-stationary vs H→W→C→K→R→S for output-stationary）、spatial unrolling（沿哪些维度在PE array上并行展开）、tiling factors（各维度的tile大小）来区分。NVDLA chiplet偏Spatial Architecture——PE array上spatial unrolling多维度、高计算吞吐；Shi-diannao chiplet偏Temporal Architecture——更多data staging和reuse optimization。
2. **Chiplet物理布局**：Het-Sides配置将两类chiplet分别放置在MCM左右两侧——左侧18个NVDLA chiplet、右侧18个Shi-diannao chiplet。两侧chiplet分别连接各自的off-chip DRAM接口。NoP为2D mesh，所有chiplet通过router node连接mesh。跨侧（left↔right）chiplet间的数据通过mesh horizontal links传输，hop count from 1 to 5。
3. **数据流适配**：当Conv layer被分配到NVDLA chiplet，数据以NVDLA dataflow的方式在chiplet内流动——weight从off-chip DRAM预加载到chiplet local buffer（weight stationary），activation在PE array上spatial multicast。当FC layer被分配到Shi-diannao chiplet，activation在chiplet local memory中保留更久以减少off-chip traffic。异构MCM使每个layer获得最优dataflow匹配，消除同构MCM中"一个dataflow适配所有operator类型"的瓶颈。
4. **封装级互联**：异构chiplet间通过NoP交换中间数据。如ResNet-50的Conv layer（NVDLA chiplet）产出feature map → 通过NoP → 后续FC layer（Shi-diannao chiplet）消费。此cross-chiplet data transfer的延时由data_size / BW_NOP × n_hops建模。
5. **MCM配置空间**：除Het-Sides外，SCAR还探索了Het-CB（Checkerboard交错排列——不同类型chiplet相邻排列以支持fine-grained near-neighbor pipelining）、Grid（标准行列对齐排列）等异构配置。Chiplet总数从4×4到6×6可扩展。

Heterogeneous Dataflow MCM的实现基于现有chiplet技术：(1) 2.5D silicon interposer作为chiplet间的物理连接基板（如AMD MI300X [1]、Nvidia Grace-Blackwell [14]使用的interposer技术），提供高密度die-to-die interconnect（如UCIe标准，Universal Chiplet Interconnect Express）；(2) 每chiplet是独立设计的AI加速器（可能来自不同IP供应商或同一供应商的不同dataflow变体），通过standardized die-to-die PHY连接到interposer；(3) MCM package将interposer、chiplet、off-chip DRAM（如HBM或LPDDR）封装在一起。实际产品化案例：Nvidia Grace-Blackwell将1 CPU + 2 GPU chiplet集成在同一package，AMD MI300X集成多个compute die。SCAR在此趋势上提出将不同dataflow的AI加速器chiplet混合集成，通过调度器动态匹配workload heterogeneity。与同构MCM相比，异构MCM的工程挑战在于chiplet间dataflow compatibility（producer chiplet的output data layout需被consumer chiplet理解）和NoP bandwidth pressure（cross-dataflow chiplet的通信量可能更大）。

## Network-on-Package (NoP) for MCM AI Accelerators

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
Network-on-Package (NoP) 是Multi-Chiplet Module (MCM)中连接各chiplet的封装内互联网络。与SoC中的Network-on-Chip (NoC)不同，NoP跨越多个独立的chiplet die，在interposer上通过物理链路（如silicon interposer traces、EMIB bridge、或UCIe die-to-die links）连接各chiplet的router node。SCAR假设2D mesh NoP拓扑（类似Simba [64]），每个chiplet作为一个mesh node，chiplet间数据通过hop-by-hop routing在package内传输。NoP的关键性能参数为BW_nop（NoP链路带宽，通常远高于off-chip DRAM带宽），其高带宽特性使inter-chiplet pipelining——即在一个chiplet完成部分计算后立即将中间数据通过NoP转发到下一个chiplet继续处理（而非先写回off-chip DRAM再读出）——成为可能，大幅增强in-package data reuse并减少off-chip traffic。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
NoP的芯片设计运转流程（以SCAR的6×6 mesh NoP为例，chiplet间传输中间feature map）：
1. **NoP拓扑与routing**：2D mesh中每个chiplet通过5-port router（North/South/East/West/Local）连接相邻chiplet和本地PE array。chiplet (0,0)位于左下角，chiplet (5,5)位于右上角。chiplet (i,j)到chiplet (k,l)的数据传输需|i-k| + |j-l| hops（XY dimension-ordered routing）。
2. **数据打包与注入**：Producer chiplet完成layer segment部分计算后，将输出数据按目的地chiplet打包为NoP flit（flow control unit）。数据通过Local port注入chiplet的NoP router → router根据destination chiplet坐标决定输出port → flit逐hop前进。
3. **Inter-chiplet pipelining over NoP**：Producer chiplet在完成部分output（如output feature map的前几行）后即可通过NoP streaming发送给consumer chiplet（无需等全部output完成）。Consumer chiplet从NoP接收数据边计算，与producer形成producer-consumer pipeline——在时间上重叠两个chiplet的计算。
4. **Off-chip interface**：chiplet位于MCM左右两侧的具备off-chip DRAM接口。off-chip access需经过chiplet router→NoP edge → DRAM PHY。BW_offchip通常低于BW_nop（如64 GB/s vs 128 GB/s），因此inter-chiplet pipelining利用高BW_nop避免off-chip往返。
5. **NoP contention**：多个chiplet对同时通过NoP传输数据时可能产生link contention。SCAR在调度时考虑hop distance和link sharing，通过segment-to-chiplet mapping优化减少cross-traffic。

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
NoP的物理实现技术：(1) Silicon interposer traces——在passive silicon interposer上的金属走线，支持高密度并行链路（如Simba的GRS ground-referenced signaling，单端串行4Gb/s/pin）；(2) EMIB (Embedded Multi-die Interconnect Bridge)——Intel的硅桥技术，在package substrate中嵌入小硅片实现die-to-die高密度连接；(3) UCIe (Universal Chiplet Interconnect Express)——标准化chiplet互联协议，支持2D/2.5D封装，带宽密度可达1.3 TB/s/mm (advanced package) 或 50 GB/s/mm (standard package)；(4) 3D stacking with hybrid bonding——通过face-to-face bonding实现微米级pitch的直接die-to-die连接。在MCM AI加速器中，NoP路由通常采用simple XY dimension-ordered routing或source routing（静态确定路径），避免complex adaptive routing的硬件开销。NoP的BW_nop和latency特征直接影响inter-chiplet pipelining的收益——高BW_nop使得频繁的封装内数据交换成本可接受。

## SRAM-based In-Cache Computing Peripheral Circuits (SRAM缓存内计算外围电路)

术语解释
附加在SRAM阵列外围、使标准SRAM单元能执行逻辑和算术运算的定制电路层。MagiCache的fused array包含五层外围电路：Logic、Add、Shift、Register和Writeback。

术语是什么？
SRAM-based In-Cache Computing Peripheral Circuits是实现bit-line computation的物理电路基础。传统SRAM阵列仅支持读/写操作：激活一条word-line→bit cell数据流到bit-line→sense amplifier数字化读出。IMC外围电路在此基础上增加：(1) 同时激活两条word-line的能力（需降低word-line电压避免破坏bit cell数据，代价是操作频率略微降低）；(2) 处理bit-line上模拟信号的电路（AND/NOR → digitalization → arithmetic）。MagiCache的五层结构：
- **Logic Layer**：从sense amplifier输出的AND和NOR值通过组合逻辑生成所有logic outputs：AND, NAND, OR, NOR, XOR, XNOR（XOR = NAND AND OR, XNOR = AND NOR OR）。
- **Add Layer**：carry chain连接相邻bit-lines——BL[i]的carry out (Cout)连接到BL[i+1]的carry in (Cin)，实现ripple carry addition。全加器输出SUM和CARRY。ADD(S)信号选择加法模式。
- **Shift Layer**：左移（lshift）和右移（rshift）单元。左移：BL[i]输入来自BL[i-1]的输出；右移：BL[i]输入来自BL[i+1]的输出。结合carry chain和shift实现乘法（iterative shift-and-add）。
- **Register Layer**：每bit-line一个1-bit register（D flip-flop或latch），保存中间值（如shift operation的部分结果、multiplication的accumulator）。同时作为write buffer coalesce多个vector element的写操作。
- **Writeback Layer**：选择最终结果（可从logic/add/shift/register各层输出中选择）并写入destination word-line。

从芯片设计角度拆解术语
MagiCache fused array（256×256）的物理实现：
1. **Sub-array Partitioning**：256×256 array拆分为两个128×256 sub-arrays共享同一套外围电路（因sub-array内bit-line较短，信号完整性更好）。两个sub-arrays分时复用外围电路。
2. **Bit-line Pitch匹配**：外围电路需要匹配SRAM bit cell的bit-line pitch（即column间距）。在TSMC 40nm工艺下，6T SRAM bit cell高度约0.5μm，bit-line pitch约0.24μm——外围电路需在此pitch内完成logic/add/shift/register/writeback全部功能，对layout密度要求高。
3. **面积breakdown**：单个128×256 sub-array面积overhead 17.7%（vs vanilla SRAM）。当两个sub-arrays共享外围电路时，整体256×256 fused array overhead降至8.9%。Micro-code ROM 8 KB占额外1.6%面积。
4. **能耗分析**：Bit-line computation能耗比read/write高54%（1.6nJ vs 1.04nJ per operation @1.1V, 40nm）。考虑bit-line computation占fused array总operations的约17%（其余83%仍为常规read/write），整体功耗增加约9%。但关键优势：bit-line computation的source和destination rows在同一fused array内→数据不需经过H-tree网络传输。传统cache的H-tree占>80%总能耗[Balasubramonian 2011]→省去H-tree传输后，实际系统级能效显著提升。
5. **时序约束**：Vanilla SRAM read/write cycle time = 1.0ns (1GHz @TSMC 40nm)；bit-line computation cycle time = 1.6ns (625MHz @TSMC 40nm) → +60%延迟。设计选择：以1.6ns为fused array的统一cycle time，或采用双时钟域（read/write @1GHz + compute @625MHz）。

术语一般如何实现？如何使用？
设计与验证流程（基于MagiCache论文）：
1. **全定制电路设计**：Cadence Virtuoso schematic capture → 手动layout（因bit-line pitch匹配需手工优化）→ DRC/LVS检查。
2. **功能验证**：Cadence Spectre模拟（TSMC 40nm, 1.1V, TT corner, 25°C）→ inject多组随机输入测试向量 → 打印关键节点信号波形（word-line activation, bit-line voltage, sense amplifier output, logic/add/shift outputs）→ 与预期真值表比对。
3. **RTL综合**（virtual engine control logic）：Verilog RTL → Synopsys Design Compiler + TSMC 28nm 0.81V @1GHz → area/power reports。
4. **使用方式**：外围电路对上层RISC-V vector程序透明。程序员无需了解peripheral circuit细节，通过标准RVV intrinsics编程即可利用in-cache computation。
5. **可移植性**：Peripheral circuit design与technology node相关（40nm→28nm→7nm需重新设计/porting）。OpenRAM [Guthaus 2016]可能用于生成base SRAM array layout，外围电路需手工添加。

## DIMM-link (DIMM间直连通信链路)

术语是什么？回答尽量完整，回答逻辑链中每一步都解释出来。通过联网搜索让回答具体和精准。
DIMM-link是一种在DIMM模组之间建立direct point-to-point高速通信链路的硬件设计，最早由Zhou等人[68]在HPCA 2023提出。它通过在DIMM之间添加专用的bidirectional external data links，使data可以直接从一个DIMM传输到另一个DIMM而无需经过host CPU/memory controller中转。在Hermes系统中，DIMM-link用于支持NDP-DIMM间的cold neuron remapping——当window-based scheduling决定将某个cold neuron从高负载DIMM迁移到低负载DIMM时，neuron weight数据通过DIMM-link直接搬移。关键参数：25Gb/s per lane, 8 lanes per link = 25GB/s per link, 1.17 pJ/b energy efficiency。相比host CPU forwarding方案（DIMM→PCIe→CPU→PCIe→另一DIMM），DIMM-link提供62× speedup，使migration overhead从5.3%总时间降至<0.2%。

从芯片设计角度拆解术语，比如术语如何在芯片设计中发挥作用，给出术语在芯片设计中运转流程的具体例子。通过联网搜索让回答具体和精准。
DIMM-link芯片级组成和数据传输流程：
```
DIMM A (source)                            DIMM B (target)
┌──────────────────────┐         ┌──────────────────────┐
│ DRAM Banks           │         │ DRAM Banks           │
│   ↓ read             │         │   ↑ write            │
│ Center Buffer        │         │ Center Buffer        │
│   ↓                  │         │   ↑                  │
│ DIMM-link Controller │         │ DIMM-link Bridge     │
│   ├─Serializer       │         │   ├─Deserializer     │
│   ├─Lane Driver ×8   │────────→│   ├─Lane Receiver ×8 │
│   └─Flow Control     │ 25GB/s  │   └─Buffer/Arbiter   │
└──────────────────────┘         └──────────────────────┘
        bidirectional external data links (8 lanes × 25Gb/s)

数据搬移流程:
1. Source DIMM controller issue internal READ → DRAM bank activate → column read
2. Data enters center buffer → forwarded to DIMM-link controller
3. Serializer packs data into lane-width chunks → 8 lanes parallel transmission
4. 25Gb/s/lane → 200Gb/s raw → 25GB/s effective (after 8b/10b encoding overhead)
5. Target DIMM bridge receives → deserializer → buffer
6. Arbiter schedules DRAM WRITE (may wait for ongoing NDP compute or memory access)
7. Data written to target DRAM via local memory controller
```

术语一般如何实现？如何使用？通过联网搜索让回答具体和精准。
实现方式：(1) 物理层：基于高速SerDes技术（类似PCIe或JEDEC DDIMM标准的物理层），使用differential signaling on external PCB traces连接相邻DIMM slots。(2) 链路层：支持flow control (credit-based)和error detection (CRC)，保证数据完整性。(3) 拓扑：point-to-point pairwise连接——Hermes中采用固定配对方案（DIMM-id ↔ DIMM-(J-1-id)），避免全连接mesh的布线复杂度和成本。(4) 面积/功耗：1.17 pJ/b，对于DIMM form factor面积overhead negligible。(5) 与alternative方案的对比：CPU Forwarding——DIMM A→memory channel→CPU memory controller→CPU cache→memory channel→DIMM B，延迟高（multi-hop）且占用host memory bandwidth。ABC-DIMM[ISCA 2021]——multi-drop bus，支持broadcast但不支持concurrent pairwise通信。(6) DIMM-link不仅用于Hermes的neuron remapping，也是通用DIMM-PIM系统inter-PE通信的基础设施选项之一。

## Three-Tier Memory System for AI Accelerators (AI加速器三阶存储系统：SRAM-HBM-DDR)

术语是什么？
三阶存储系统是SambaNova SN40L RDU引入的新型AI加速器存储层次，由三级不同容量和带宽的memory组成：(1) **第一阶On-chip Distributed SRAM**：520MB，分布在1040个PMU中，提供hundreds of TBps的aggregate on-chip bandwidth；(2) **第二阶Co-packaged HBM**：64GB，1.8TB/s per socket，通过2.5D CoWoS封装集成；(3) **第三阶Off-package DDR DRAM**：最高1.5TB（可插拔DIMMs），200GB/s per socket，直接attached到accelerator（不经host PCIe）。三阶之间带宽递减约一个数量级：SRAM >> HBM (1.8TB/s) >> DDR (200GB/s)。DDR→HBM copy在8-socket SN40L Node上aggregate >1TB/s。设计目标：SRAM提供计算所需极高带宽（pipeline streaming），HBM利用temporal locality缓存active expert weights和KV cache，DDR提供大容量存储所有expert models（消除host memory spill）。

从芯片设计角度拆解术语：
在SN40L设计中，三阶存储在芯片级以以下方式运作：(1) **物理集成**：HBM通过CoWoS interposer与2个RDD (Reconfigurable Dataflow Dies) 在同一package内互联——HBM PHY直接连接RDD的TLN (Top Level Network)，提供高带宽低延迟访存路径。DDR PHY也直接接入TLN，但走package pins到external DIMM slots。(2) **数据流路径**：Compute (PCU) → RDN (on-chip mesh) → PMU (SRAM scratchpad, 第一阶) → AGCU (memory bridge) → TLN → HBM controller (第二阶) 或 DDR controller (第三阶)。DDR和HBM共享同一AGCU-TLN访问路径，但带宽分配由compiler static bandwidth model管理。(3) **容量vs带宽tradeoff物理实现**：HBM用1024-bit wide interface + TSV stacking实现>1TB/s，DDR用标准72-bit DIMM interface实现200GB/s。面积预算：HBM PHY + interposer area占package面积可观比例，DDR仅需PHY + pinout。(4) **Chiplet级集成**：2个RDD通过D2D Interface互联，每个RDD有独立的HBM和DDR channel——aggregate bandwidth across dies通过TLN路由。

术语一般如何实现？如何使用？
- **SN40L实现**：TSMC 5nm工艺，CoWoS封装。DDR tier使用标准DDR5 DIMMs（可插拔，用户可扩展至1.5TB）。Compiler自动管理三阶存储分配：默认优先HBM→不够时按symbol访问频率排序spill到DDR（LLM weights优先留HBM，activation/intermediates可能spill）。CoE runtime利用DDR的大容量存储所有expert weights（150+ experts, ~1T params total），HBM作为software-managed cache of active experts（LRU eviction）。
- **与其他架构对比**：NVIDIA A100/H100仅有HBM（无accelerator-attached DDR tier），expert spill需经PCIe到host DRAM（32-64GB/s vs SN40L DDR 200GB/s per socket）。NVIDIA GH200虽有更大aggregate memory但无三阶层次化设计。Google TPUv4/v5同样仅有HBM。
- **通用适用性**：三阶存储适用于需要serve大量独立模型（CoE, MoE with many experts, multi-tenant serving）的场景——DDR存全量模型，HBM缓存热模型，SRAM提供计算带宽。

## CoWoS Chiplet-based Dataflow Accelerator Package (CoWoS Chiplet数据流加速器封装)

术语是什么？
CoWoS (Chip-on-Wafer-on-Substrate) 是TSMC的2.5D先进封装技术，将多个芯片die并排集成在silicon interposer上，通过interposer中的high-density interconnect实现die间通信。SN40L RDU使用CoWoS封装集成2个Reconfigurable Dataflow Dies (RDDs) 和HBM stacks。每个RDD是TSMC 5nm工艺的compute die (<650mm²)，HBM通过interposer连接到RDD的memory controller。两个RDD之间通过Die-to-Die (D2D) Interface直接流式传输数据，不经interposer外的off-chip路径。总socket提供638 BF16 TFLOPS。CoWoS封装相比传统PCB-level multi-chip module的优势：更低的die-to-die通信延迟、更高的互连带宽密度、更低的功耗per bit传输。

从芯片设计角度拆解术语：
SN40L CoWoS封装的关键设计：(1) **Silicon Interposer**：作为passive interconnection substrate，承载RDD-to-RDD和RDD-to-HBM的micro-bump连接。Interposer中的RDL (Redistribution Layer) 提供high-density routing。(2) **D2D Interface**：两个RDD之间通过interposer上的parallel bus直连——数据直接在die间streaming而不经HBM（避免off-chip memory hop的开销），支持tensor parallelism跨die通信。(3) **HBM Integration**：HBM stacks通过interposer上的1024-bit wide interface连接RDD PHY——对比传统PCB-based HBM连接，CoWoS提供显著更高的signal integrity和带宽密度。(4) **热管理**：2个RDD + HBM stacks共封装的thermal density要求——CoWoS package的热设计需处理~数百瓦级别的功耗。

术语一般如何实现？如何使用？
- SN40L使用TSMC 5FF工艺 + CoWoS-S (Silicon Interposer) 或 CoWoS-L 变体。2.5D集成的通用流程：各die分别制造 (RDD用5nm, HBM base die用专用工艺) → 测试 → thinned → 通过micro-bump bonding到interposer → interposer通过C4 bumps连接到package substrate → substrate连接到PCB。
- 类似的CoWoS应用：NVIDIA A100/H100使用TSMC CoWoS集成GPU die + HBM。但SN40L的独特之处在于2个对等RDD共享同一CoWoS package（而非GPU+显存的non-symmetric集成），RDD之间通过D2D direct streaming实现low-overhead multi-die dataflow。
- 芯片设计考虑：CoWoS的interposer size限制（reticle limit）→ 多少RDD和HBM可集成；micro-bump pitch决定互连密度；thermal budget决定sustained performance。
