    1、论文工作：
        - 论文要解决的核心问题：FlashFuser 解决的是现代 GPU 上 compute-intensive operator chain 的 kernel fusion 被单个 SM 的寄存器 / shared memory 容量限制卡住的问题。论文关注的典型链路包括 GEMM->GEMM 的 FFN、Gated FFN 以及可转成 GEMM chain 的 convolution block。随着 H100 一类 GPU 的 FP16 算力相对 HBM 带宽增长更快，FFN 等层在 LLM 推理中会消耗 40%--60% 的执行时间，并呈现明显的 memory-bound 特征；如果两个 GEMM 之间的中间矩阵不能留在片上，就必须写回 global memory 再读回，形成低效 round trip。
        - 论文的主要贡献：论文提出 FlashFuser，一个利用 GPU inter-core connection / Distributed Shared Memory (DSM) 做 kernel fusion 的编译框架。它的贡献分为三层：第一，定义 `dsm_comm` 抽象，把 cluster 内 SM 划分、SM 间数据交换和 reduce / shuffle / multiply 等通信模式统一描述出来；第二，提出 Dataflow Analyzer，把 loop schedule、tile size、resource mapping 扩展到 register、SMEM、DSM、global memory 的层次化内存空间，并估算各层数据移动量；第三，提出 Fusion Search Engine，用分析代价模型和 DSM-aware pruning 在极大搜索空间中找到可行的融合执行计划，再生成高性能 CUDA / CUTLASS 风格 fused kernel。
        - 论文所处背景：传统 kernel fusion 已能处理 compute + activation / bias 这类较小中间数据，也能通过 Chimera、BOLT、Welder、MCFuser 等方法探索 GEMM chain 的局部数据复用。但这些系统主要把中间结果放在 register 或单 SM 的 SMEM 中。H100 开始提供 thread block cluster 和 DSM，允许 cluster 内多个 SM 的 shared memory 通过片上互连访问，形成类似 L1.5 的更大高带宽缓冲池；问题在于现有 DL compiler 基本没有系统地把 DSM 纳入 fusion 搜索、dataflow 分析和代码生成。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：cuBLAS、CUTLASS、PyTorch、TensorRT 等库一般执行独立算子或较有限的融合，两个 compute-intensive 算子之间仍会把中间结果落到 global memory。Relay 和 TASO 等图编译器可做 compute-activation 融合或图替换，但不支持把顺序 GEMM / convolution chain 系统地融合成一个大 kernel。BOLT 依赖 CUTLASS 模板和固定 block 执行顺序，搜索空间有限；Chimera 能对连续 GEMM 做分析式融合并探索 block order，但中间结果仍受单 SM SMEM 容量限制，论文给出的 H100 单 SM SMEM 上限约 227KB，大模型 FFN 中间矩阵超过该阈值时会 fusion failure。DSM 相关工作如 T10、WaferLLM 利用了 inter-core data sharing，但目标硬件不是主流 GPU，且没有解决自动 kernel fusion；ClusterFusion 使用 GPU DSM 做 fusion，但论文称其偏手写 kernel，缺少自动参数选择和代码生成。
        - 论文的设计方法：FlashFuser 把 fusion plan 表示为 cluster-level tiling + block-level tiling + loop schedule + resource mapping。对于两段 GEMM，执行分为 GEMM0、GEMM1 和 Store 三个阶段：GEMM0 产生中间矩阵 C 的局部 partial sum；`dsm_all_exchange` 在 cluster 内沿 K 维完成聚合，使 block 获得完整中间 tile；GEMM1 阶段通过 `dsm_shuffle` 在 shuffle group 内交换 C 的切片，让不同 block 计算输出 E 的不同 partial sum；Store 阶段用 `dsm_reduce_scatter` 做 cluster 内 scatter-reduce，再用基于 Hopper TMA `cp.reduce.async.bulk` 的 `inter_cluster_reduce` 聚合跨 cluster partial sum。对于 Gated FFN，`dsm_all_exchange` 可以从加法 all-reduce 变成执行 Mul，把 SiLU 分支和另一 GEMM 分支的结果合并。
        - 方法如何对冲 Baseline 缺陷：FlashFuser 的关键不是简单“多一层缓存”，而是把 DSM 变成可搜索、可建模、可代码生成的 fusion 层次。Dataflow Analyzer 会根据 loop schedule 和 tile footprint 贪心地把复用 tensor 优先放到高层缓存，容量不足时逐层 spill 到 SMEM、DSM 甚至 global memory，并计算 DSM traffic；Fusion Search Engine 再用 `C_l = V_l(T_l) / B_l` 的分层数据移动模型做 minimax 优化，目标是避免某一层内存搬运成为瓶颈。这样，当 Chimera / BOLT 因单 SM SMEM 放不下中间 tile 而放弃融合时，FlashFuser 可以把中间结果分布到 cluster 内 DSM，并通过 shuffle / reduce 原语完成生产者 GEMM 和消费者 GEMM 的数据衔接。
        - 关键 trade-off：FlashFuser 接受了更复杂的搜索空间、cluster 同步、DSM 通信和代码生成复杂度。DSM 的带宽和延迟随 cluster size 变化，cluster 越大不一定越好；`dsm_shuffle`、`dsm_reduce_scatter`、Mul 等通信会引入额外数据搬运和同步成本；搜索空间也从传统方法约 10^4 级别扩大到论文示例 GPT-6.7B 中约 10^6 个 pruning 后候选。因此论文需要额外的 pruning rule、cost model、top-K profiling 和 offline compilation；运行时则通过 binning / table lookup 针对动态变化的 M 维选择预编译 kernel。

    3、论文实现：
        - Baseline 如何实现：实验比较了库、推理框架和研究型编译器。库侧包括 PyTorch 2.6（GEMM 使用 cuBLAS，并启用 `torch.compile` 降低 launch overhead）和 TensorRT；编译器侧包括 TVM / Relay、TASO、BOLT、Chimera。论文说明 Relay 主要做 compute-activation 模式融合，TASO 通过等价子图替换优化图但不支持 compute-intensive sequential operator fusion，BOLT 基于 SMEM / register 和 CUTLASS 模板融合 consecutive GEMM，Chimera 也融合 consecutive GEMM 并探索不同 block execution order，但均没有使用 DSM。
        - 新设计如何实现：FlashFuser 是建立在 NVIDIA CUTLASS 上的代码生成框架。前端是 Python search engine，枚举 LoopSchedule、TilingSize 和 ResourceMapping，调用 Dataflow Analyzer 得到每个 reused tensor 在 register / SMEM / DSM / global memory 中的放置方案和数据移动量，再用代价模型筛选 top-K。后端把最佳 plan 翻译为 CUDA 代码：理论计算 register 使用量以避免 spill 到 global memory；当 SMEM 不足时把数据放到 DSM；`dsm_comm` 的 SHUFFLE、MUL、REDUCE 由 TMA 数据移动和 `mbarrier` many-to-many synchronization 实现。代码生成器扩展 CUTLASS kernel 的 prologue、mainloop、epilogue：prologue 初始化 DSM semaphore，mainloop 插入 DSM mul / shuffle，epilogue 执行 DSM reduce 后写回 global memory。
        - 实验 / 实现平台：实验平台是一台服务器级 H100 SXM GPU，主机为双路 Intel Xeon Platinum 8468，共 96 cores，2.10GHz。软件栈包括 CUDA 12.4、PyTorch 2.6、TVM 0.9、Triton 3.2、Nsight Compute 2025.2.0。benchmark 覆盖 GEMM chains、ResNet 中抽取的 convolution chains、Gated FFN，以及基于 SGLang 的 end-to-end LLM 推理；模型 / 子图配置包括 DLRM、GPT-2 Small、GPT-6.7B、OPT-1.3B、Performer、BERT、Llama、Qwen 等维度设置。大模型扩展实验还包含 Llama3-70B、Qwen2.5-14B 和 Qwen2.5-32B。
        - 关键实验设置与指标：论文主要测 kernel latency / normalized speedup、global memory access、cost model 搜索准确性、search time、DSM primitive bandwidth / utilization、ablation 和 end-to-end speedup。摘要和结论给出的核心结果是：在 H100 上，FlashFuser 相对 highly-tuned libraries 获得最高 3.3x kernel speedup，相对 SOTA compilers 获得最高 4.1x kernel speedup，平均减少 58% memory access，并带来 1.24x end-to-end speedup。更细分地，GEMM chain 上相对 BOLT、Chimera、Relay、TASO、TensorRT、PyTorch 的平均 speedup 分别为 5.4x、4.6x、4.7x、3.4x、2.4x、3.1x；convolution chain 上分别为 6.3x、6.4x、5.6x、4.3x、3.3x、3.9x。Nsight Compute 分析显示 PyTorch kernel 平均产生 FlashFuser 2.4x 的 global memory traffic。搜索方面，top-K 取 11 时接近最优；对 G3/G4/G5，search engine 相对 brute-force 的搜索时间加速分别为 12.25x、29.05x、68.26x。

    4、pipeline/kernel 解析：
        - 新 pipeline/kernel 是什么：论文引入的是一个 DSM-aware fused kernel generation pipeline，以及在 kernel 内运行的 cluster-level fused GEMM / FFN dataflow。其核心 pipeline 可概括为“DNN 子图输入 -> search engine 枚举 loop schedule / tile / cluster 配置 -> Dataflow Analyzer 生成层次化 tensor placement 和 data movement -> cost model + pruning 保留 top-K -> CUTLASS-based backend 生成 fused CUDA kernel -> H100 实测选择最佳 kernel -> runtime 用 binning / table lookup 选择预编译版本”。在 kernel 内，具体执行路径是“prologue 初始化 DSM 同步资源 -> GEMM0 生产中间 partial tile -> DSM all-exchange / Mul 聚合或合并中间结果 -> GEMM1 通过 DSM shuffle 消费中间 tile -> epilogue 用 DSM scatter-reduce / inter-cluster reduce 聚合输出 -> global memory store”。
        - 新 pipeline/kernel 的执行流例子：以标准 FFN 的两段 GEMM 为例，设输入 A 与权重 B 先计算中间矩阵 C，C 再与权重 D 计算输出 E。FlashFuser 先根据模型维度和 H100 约束选择 cluster size，例如论文图示中 `(clsm, clsn, clsk, clsl) = (2, 4, 2, 4)`，并决定每个 block 的 tile 大小。执行 GEMM0 时，同一 cluster 内多个 block 沿 K 维并行计算 C 的 partial sum，例如得到 `C0,0(0)` 和 `C0,0(1)`；随后 `dsm_all_exchange` 在 cluster 内做 all-reduce，使 block 得到完整 `C0,0`。进入 GEMM1 后，完整 C tile 不写回 global memory，而是通过 `dsm_shuffle` 在 shuffle group 中交换到需要它的 block，与 D 的不同 tile 相乘，产生 `E0,0(0)`、`E0,1(0)` 等 partial sum。最后 Store 阶段先在 cluster 内使用 `dsm_reduce_scatter` 把多个 partial E 聚合为负责写回的输出 tile；如果多个 cluster 都贡献同一输出，还通过 TMA `cp.reduce.async.bulk` 实现 inter-cluster reduce。对 Gated FFN，两个 Up-FFN 分支可以空间划分到不同 block group，SiLU 分支和另一分支的中间结果通过 `dsm_all_exchange` 执行 Mul 后再进入后续 GEMM；另一种策略则是在同一个 block 内顺序执行两个 GEMM，以减少 DSM 通信但牺牲部分并行性。
