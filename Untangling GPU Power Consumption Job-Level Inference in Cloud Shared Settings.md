    1、论文工作：
        - 论文要解决的核心问题：云数据中心里的 GPU 功耗监控通常默认两个前提：第一，可以直接通过 NVML / nvidia-smi / DCGM 读取设备功耗；第二，一个 GPU 同一时刻只属于一个 job，因此整卡功耗可以直接归给该 job。论文指出这两个前提在 hyperscale cloud 中都不稳固：GPU 可能被 temporal sharing、MIG / spatial sharing 或 VM passthrough 使用；在 shared / multi-tenant 场景中，整卡功耗不能直接按单作业归因；在 passthrough 场景中，宿主机甚至失去对 GPU driver-level 指标的可见性。
        - 论文的主要贡献：论文提出一组面向黑盒云环境的 job-level GPU power inference 方法，而不是单一模型。对 temporal sharing，论文验证用活跃容器数均分整卡功耗可作为近似归因模型；对 spatial sharing / MIG，论文提出按 GPU 静态功耗和 slice-specific 最大功耗分解的 slice 功耗模型，并给出带 unused-energy 修正项的精度版本；对 passthrough，论文用 IPMI 温度传感器作为非侵入式 proxy，通过相关性选择传感器、邻近 GPU 热干扰校正和回归模型推断 GPU usage / power。论文还用 OVHcloud 的 176 张 H100 IaaS GPU 集群数据说明 dedicated IaaS GPU 存在严重利用率不足。
        - 论文所处背景：该工作位于云计算、绿色数据中心、GPU resource sharing、carbon accounting 和 multi-tenant monitoring 交叉方向。它关心的不是 kernel 内部能耗建模，而是云服务商在无法看到客户 workload 内容、不能要求客户安装 agent、甚至无法访问 passthrough GPU driver 指标时，如何仍然为不同租户 / job 提供可解释、可集成到碳核算和调度系统中的 GPU 功耗估计。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：最直接的 baseline 是“整卡功耗直接归给当前作业”，这只适用于 bare-metal 或独占 GPU，不适用于多个容器 / VM 分享 GPU 的场景。另一个常见做法是只依赖 NVML / nvidia-smi / DCGM 读数，但 passthrough 后宿主机无法访问 GPU 设备指标，MIG 模式下经典 GPU utilization 指标也不足以反映每个 slice 的功耗。已有 GPU power modeling 框架如 GPUWattch / AccelWattch 更偏架构模拟，不适合云上 live job accounting；JoularJX 等工具可以做进程级 GPU 功耗归因，但论文认为它没有处理多进程共享同一 GPU 的情况，也不适配 hyperscale black-box 云场景。
        - 论文的设计方法：论文按云产品中的 GPU 分配语义拆成三条路径。Temporal sharing 中，Kubernetes GPU Operator / vGPU 通过 time-slicing 让多个进程轮流使用 GPU，论文通过 GPU-burn 与实际 benchmark 测量发现 total power 在共享后略降但 context-switch 开销不随复用度显著增加，因此用 Container_power = GPU_power / n 作为简单近似，其中 n 是并发容器数。Spatial sharing 中，MIG 将 GPU 切成 GI / CI，论文先用 GPU-burn 标定不同 MIG profile 的 worst-case power，再把 slice 功耗建模为 P_chip_static / N 加上 slice-specific 动态范围；若目标是更贴近实际而非 worst-case accounting，则增加经验因子 F 估计 oversized allocation 中未使用的能量。Passthrough 中，论文放弃改 VFIO 或客户 agent，转向宿主机仍可访问的 IPMI 温度传感器：先用 Pearson correlation 选出每张 GPU 最相关的温度传感器，再用邻近 GPU 温度 delta 的线性回归扣除热串扰，最后用密度边界或回归把校正温度映射到 GPU usage / power。
        - 方法如何对冲 Baseline 缺陷：相对整卡独占归因，temporal 模型至少把同一 GPU 的静态 / 动态功耗分摊到并发容器，避免所有功耗都归给单个 job。相对只看 GPU driver 指标，MIG 模型利用 allocation size 和离线标定参数，不要求每个 tenant 暴露 workload 细节。相对 passthrough 完全不可见，IPMI 温度路径保持 VM 隔离和隐私边界，只通过主板传感器恢复粗粒度 usage / power visibility。相对直接用原始温度，邻近 GPU 校正处理了多 GPU 服务器中 airflow 导致的热干扰，使 idle / non-idle 识别更可靠。
        - 关键 trade-off：Temporal 均分模型简单、可解释，但只能近似处理同质并发，无法区分同一时间片内不同 workload 的真实 SM / memory 压力。MIG worst-case 模型更 deterministic，适合云上账单 / 碳核算中的上界估计，但对 oversized allocation 会高估功耗；加入 F 因子后更准确，却降低了 deterministic accounting 的政策清晰度。Passthrough 温度模型非侵入、保护隐私，但需要按服务器型号和冷却系统校准，受采样频率、传感器精度、环境温度、风冷 / 水冷布局影响；论文也承认 spatial 模型目前与 NVIDIA MIG 特性绑定，向其他厂商 GPU 泛化仍需后续研究。

    3、论文实现：
        - Baseline 如何实现：Temporal 实验先用 GPU-burn 测最高功耗，再在 Kubernetes 集群中调整 NVIDIA GPU Operator replica 参数，探索 r=1、2、4、8 的 oversubscription，并部署 0 到 r 个容器，用 NVML / nvidia-smi 读整卡功耗。真实 workload baseline 包括 Blender 4.3.0 渲染 Monster 场景、NVIDIA HPCG、Llama-3.2-1B 推理和 YOLOv8 训练，各 workload 用 Python wrapper 循环执行并周期性输出 performance metric。Spatial 实验对 A100 / H100 / GH200 等支持 MIG 的 GPU 标定 GI / CI 配置下的功耗，并比较不同 compute slice 数带来的性能与功耗变化。Passthrough 实验用 GPU-burn 和 MIG 控制 0%、约 30%、约 60%、100% 等使用水平，同时采集 IPMI 温度与 nvidia-smi power 作为训练 / 校验信号。
        - 新设计如何实现：Temporal 模型实现为按并发容器数分摊整卡功耗的归因规则，并通过性能下降与每容器功耗下降的关系判断 energy efficiency。Spatial 模型实现为两类公式：Equation 1 将 slice 功耗写成静态芯片功耗在 N 个 slice 上均摊，加上该 slice profile 的 empirical 最大动态范围，偏 worst-case / deterministic；Equation 2 在动态范围项前乘以经验因子 F，用于表达 oversized allocation 未真正使用的能量，论文在测试平台上用 F_slice(i)=1-0.07*(i-1)。Passthrough 模型实现为传感器选择、邻居热干扰校正、baseline correction / min-max normalization 与线性或多项式回归组合；水冷 OVHcloud 环境中还使用 pybaselines 的 Automated Baseline Correction 的 imor 实现，用 24 小时滑动窗口消除温度基线偏移。
        - 实验 / 实现平台：Temporal sharing 覆盖 P100、V100、A100、H100，其中 P100 用于 Kubernetes oversubscription power 图，V100 / A100 / H100 用于 performance-power trade-off。Spatial sharing 覆盖 A100-PCIE-40GB、A100-PCIE-80GB、A100-SXM4-40GB、H100-PCIE-80GB、H100-NVL-94GB 和 GH200-NVL-96GB 等 MIG-capable 设备，并记录 driver 版本 570.86.15 / 570.124.06 与旧版 535.183.06 的差异。Passthrough 风冷实验包括 4xA100 Apollo 6500 Gen10 Plus 和 8xA100 Nvidia DGX；水冷实验来自 OVHcloud 虚拟化生产环境中的 A100 / H100。真实云分析使用 OVHcloud 一个包含 176 张 H100 的商业 IaaS GPU 集群，两个月覆盖 865 个 VM。
        - 关键实验设置与指标：Temporal 里关注每容器 power、performance degradation 和 performance / energy trade-off；结果显示 time-sharing 对 SM idle pattern 明显的 workload 更有意义，例如 A100 上的 HPCG 和 Llama 推理可因共享提升 energy efficiency，但 compute-bound 的 YOLO 训练和 Blender 收益有限。Spatial 里关注 MIG power scaling、driver / DCGM 对功耗的影响、RMSE / MAPE 和不同 slice 下的 performance-power 比；Equation 1 在 1-compute-slice right-sized 配置上 RMSE 为 13.4 W、MAPE 为 11.5%，oversized allocation 的 MAPE 上升到 26.5%；Equation 2 将全局 RMSE 从 53.0 W 降到 19.7 W，MAPE 从 26.5% 降到 14.2%。Passthrough 风冷 4xA100 的 4 级使用率识别 accuracy 为 76.7%，idle precision 为 98.6%，30% / 60% / 100% precision 分别为 82.0%、59.5%、68.6%；若只分 idle 和 non-idle，precision 为 98.6% 和 97.0%。8xA100 两级实验中温度区间不重叠，accuracy 达 100%。水冷环境中线性回归 RMSE 为 A100 5.0%、H100 13.3%，多项式回归降到 A100 3.7%、H100 11.2%。OVHcloud IaaS 数据显示 80% VM 的 median GPU compute usage 低于 25%，即使看 95th percentile，60% deployment 仍低于 50% GPU compute capacity。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：论文没有提出新的 GPU kernel、operator fusion 或 runtime kernel pipeline；最接近的新执行流是 cloud GPU job-level power inference pipeline。它把云平台中的 GPU 分配模式先分类为 temporal sharing、spatial sharing / MIG 和 passthrough，再分别走功耗分摊、slice 标定模型或温度 proxy 推断。该 pipeline 的关键不是优化计算路径，而是在 workload privacy 和 black-box cloud 约束下，把可观测信号转换为 job-level power / energy accounting。
        - 新pipeline/kernel的执行流例子：假设云平台需要估计一个客户 job 的 GPU 能耗。第一步，控制面根据产品类型判断该 GPU 是 time-sliced container、MIG instance 还是 passthrough VM。如果是 temporal sharing，平台在采样窗口读取整卡 GPU_power 和并发容器数 n，将该 job 的瞬时功耗近似记为 GPU_power / n，并随时间积分得到能耗。如果是 MIG，平台读取该 job 的 GI / CI profile，例如 1g、2g、3g 或 7g，用离线 GPU-burn 标定得到的 P_chip_static、N 和 P_slice_max 参数代入 Equation 1；若平台政策追求更贴近实际使用，则用 Equation 2 加入 F 因子修正 oversized allocation。如果是 passthrough VM，宿主机无法访问 nvidia-smi，因此平台改读 IPMI 温度传感器；它先选择与该 GPU 历史功耗最相关的传感器，再找到最可能产生热干扰的邻近 GPU，用邻居温度 delta 的线性系数扣除热串扰，随后对水冷 / 季节性温度偏移做 baseline correction 和 normalization，最后用回归模型输出 GPU usage 或功耗估计。得到每个采样窗口的 job-level power 后，平台可将其接入 carbon accounting、capacity planning 或 energy-aware scheduling。
