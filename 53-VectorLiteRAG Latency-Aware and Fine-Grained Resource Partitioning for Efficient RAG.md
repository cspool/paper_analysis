论文标题：VectorLiteRAG: Latency-Aware and Fine-Grained Resource Partitioning for Efficient RAG

RAG Serving对齐CPU+GPU系统。

1、论文工作：
    - 论文要解决的核心问题：RAG 服务把向量检索和 LLM 推理串在同一条请求路径上。传统 CPU-only vector search 在大规模、高维 IVF/PQ 索引上会变成 compute、memory bandwidth 和 I/O 受限的瓶颈；但把检索直接搬到 GPU 又会和 LLM generation 争抢 GPU memory、KV cache 空间、shared memory、kernel scheduling 资源和 GPU 计算资源。论文关注的是严格 TTFT / end-to-end SLO 下，检索延迟、检索尾部查询和 LLM batching 相互放大的系统问题，而不是单个 ANN 算法的离线吞吐。
    - 论文的主要贡献：VectorLiteRAG 提出一个 latency-aware RAG serving system：第一，基于 IVF cluster access skew 建模热 cluster 命中率，识别只需要放到 GPU 的热索引部分；第二，用搜索延迟模型、LLM throughput / KV cache 空间模型和 SLO 约束共同决定 CPU-GPU index partitioning point；第三，设计 hybrid CPU-GPU distributed retrieval pipeline，通过 mapping table 做 query- and shard-aware routing，避免 Faiss IndexIVFShards 的无效 shard probe 和 GPU thread block launch；第四，引入 dynamic dispatcher，让已经完成检索的 query 尽早进入 document retrieval / LLM 阶段，降低 batch 内慢查询造成的 head-of-line blocking；第五，提供 online index update，在访问分布漂移时重新 profiling、拆分和加载 GPU shard。
    - 论文所处背景：RAG pipeline 通常包含 query embedding、vector similarity search、document retrieval、LLM prefill / decode。LLM inference 需要 GPU 做大矩阵计算，并占用模型权重和 KV cache；向量数据库通常在 CPU memory 或 storage 中。随着 embedding 维度、数据规模和 nprobe 增大，CPU 检索会拖慢 GPU LLM。论文观测到，128M vector 数据库上 CPU retrieval 可比 LLM prefill 慢到约 2 倍，使总 TTFT 从无检索的 197ms 增至 606ms。另一方面，GPU IVF search 可比 CPU FastScan 近一个数量级更快，但 GPU index 会挤压 KV cache，降低 LLM throughput，尤其在 Qwen3-32B、Llama3-70B 等需要 tensor parallelism 的模型上更敏感。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：CPU-only baseline 使用 FAISS-CPU IVF FastScan，优点是不占 GPU，但 LUT construction / LUT scan 仍主导检索延迟，大库和高到达率下会持续 SLO violation。DED-GPU baseline 用专用 GPU 跑 FAISS-GPU IVF，能降低检索延迟，但减少可用于 LLM 的 GPU 数，模型并行配置被迫变差，Llama3-70B 这类模型尤其容易吞吐下降。ALL-GPU baseline 把 FAISS-GPU IVF shard 到所有 GPU，低负载或小库时可行，但索引占用 KV cache 空间，且检索 kernel 与 LLM kernel 共用 GPU compute/shared memory/scheduling，接近吞吐上限时 TTFT 和 end-to-end latency 快速发散。HedraRAG 也利用 cluster skew 做 GPU caching，但它按 stage throughput 平衡分配资源，不显式满足检索 SLO，因此在 latency-critical setting 中可能缓存过多或过少。
    - Baseline 瓶颈来自哪里：主要来自 memory capacity、memory bandwidth、GPU resource contention、batching tail amplification 和调度粒度。IVF-PQ 搜索中，coarse quantization 后需要为 query 和 PQ codeword 构建 LUT，再扫描候选 cluster 累加距离；FastScan 可用 SIMD 改善 CPU 端 LUT 操作，但 CPU 仍难满足严格延迟。GPU 端瓶颈不是检索本身算不动，而是 GPU memory 被模型参数和 KV cache 占用，检索 kernel 的 shared memory 和 thread block launch 还会干扰 LLM serving。批处理时，只要 batch 中存在低 hit-rate 的 long-tail query，整批检索完成时间就会被慢查询约束。
    - 论文的设计方法：VectorLiteRAG 把 IVF index 分成 CPU cold clusters 和 GPU hot clusters。离线阶段用 training queries profiling CPU search latency、CQ/LUT latency、cluster access distribution 和 bare LLM throughput；再用 Beta distribution 近似 batch 内 per-query hit-rate 分布，估计 batch 中最慢 query 的 minimum hit rate；最后用 latency-bounded partitioning algorithm 在 SLOsearch、KV cache 空间和 LLM throughput 之间二分搜索 GPU cache coverage ρ。运行时阶段保留 CQ 在 CPU 上，避免 CPU-GPU-CPU-GPU 多次设备切换；LUT scanning / distance computation 中命中 hot cluster 的部分派发到 GPU shard，miss 部分由 CPU 扫描；mapping table 记录 original cluster ID、shard ID 和 local cluster ID，支持按实际 resident cluster 精确路由。
    - 方法如何对冲 Baseline 缺陷：相对 CPU-only，它只把能显著降低 tail search latency 的 hot cluster 移入 GPU，降低 CPU LUT workload；相对 ALL-GPU，它不把整个索引搬上 GPU，而是用 SLO 决定最小或合适的 GPU resident region，保护 KV cache 和 generation throughput；相对 DED-GPU，它不牺牲整张 GPU 给检索，使 LLM tensor parallelism 更稳定；相对 Faiss IndexIVFShards，它避免每个 shard 固定 nprobe 和无效 kernel launch，只给持有相关 cluster 的 GPU worker 发送请求；相对 HedraRAG，它把“满足显式 latency SLO”作为 partitioning 目标，而不是只做 stage throughput matching。
    - 关键 trade-off：VectorLiteRAG 接受 CPU-GPU hybrid search 的复杂性，包括 profiling、mapping table、custom router、dispatcher callback 和在线 shard refresh，换取更细粒度的 GPU memory / compute 控制。它不追求检索绝对最低延迟；在 SLO 放宽时会减少 GPU index size，让表现接近 CPU-only，以保留更多 KV cache。它依赖访问分布存在 cluster-level skew，并假设少量 profiling queries 可代表一段时间内的请求分布；若 workload 分布频繁剧烈变化，必须依赖 runtime update。它也保留 CQ 在 CPU 上，因此 CQ 本身若成为瓶颈，当前设计不会把这部分 offload 到 GPU。

3、论文实现：
    - Baseline 如何实现：CPU-only 使用 vanilla FAISS-CPU IVF FastScan；DED-GPU 使用 FAISS-GPU IVF 并分配专用 GPU 给 retrieval；ALL-GPU 使用 sharded FAISS-GPU IVF index 分布到所有 GPU；HedraRAG 对比中复现其 GPU caching 思路，但因 HedraRAG 不支持 distributed retrieval，论文用 IndexIVFShard 承载其缓存方案。公平对比 HedraRAG 时，CPU-only 在 nprobe=256 下达到 35 RPS 和 0.94 NDCG@50，VectorLiteRAG 将 nprobe 提高到 6144 来匹配精度；HedraRAG 将 73% cluster 放入 GPU，而 VectorLiteRAG 在 400ms SLOsearch 下选择 31.5% partition point。
    - 新设计如何实现：检索系统基于 Faiss v1.9.0，并扩展 flexible nprobe setting 和 dispatcher callback；整体 profiler、latency-aware scheduler 和系统协调逻辑用 Python 实现。初始化时先给 index 分配 GPU memory，再给 LLM 分配 memory，以降低互相抢占；retriever 和 LLM 运行在不同 subprocess，使用不同 GPU streams 并发。Index splitter 根据 hot cluster 排名和目标 cache coverage 选择 GPU-resident clusters，再按 cluster size 排序并 round-robin 分配到 GPU shards，以平衡 shard memory。Router 根据 mapping table 把每个 query 的 hot probes 分派到对应 GPU worker，cold probes 留给 CPU SIMD path。Dynamic dispatcher 启动独立线程，监听 GPU worker completion flags 和 CPU scan callback；当某个 query 的 CPU/GPU scan 都完成时，立即 merge、rerank top-k 并送入后续 document retriever，而不是等待整个 batch 的最慢 query。
    - 实验 / 实现平台：数据集包括 Wiki-All、ORCAS 1K、ORCAS 2K。Wiki-All 含 88M 个 768 维向量，压缩 IVF index 约 18GB；ORCAS 1K 和 ORCAS 2K 使用 Wikipedia chunks 与 Stella embedding，维度为 1024 / 2048，索引约 40GB / 80GB；ORCAS 保留 Bing query duplicates 以反映真实访问偏斜。LLM 包括 Llama3-8B、Qwen3-32B、Llama3-70B，使用 vLLM v0.9.1。硬件包括 8×NVIDIA L40S 48GB + dual Xeon 6426Y 节点，以及 8×NVIDIA H100 80GB + Xeon Platinum 8462Y 节点；L40S 用于较小模型，H100 用于需要模型并行的大模型。
    - 关键实验设置与指标：请求到达服从 Poisson distribution；每个 query 检索 top-25 documents，构造 1024-token input，LLM 生成 256-token output；初始 nprobe=2048，对应平均 0.91 NDCG@50。SLO 分开定义为 retrieval SLO 和 generation SLO：Wiki-All / ORCAS 1K / ORCAS 2K 的 SLOsearch 分别为 150ms / 200ms / 300ms；Llama3-8B / Qwen3-32B / Llama3-70B 的 SLOLLM 分别为 217ms / 191ms / 311ms。指标包括 TTFT SLO attainment、end-to-end latency、P90/P95 tail latency、hit-rate estimation、throughput / arrival rate 范围、dispatcher ablation、input/output length sensitivity、SLO sensitivity 和 GPU 数量 sensitivity。
    - 主要实验结果：VectorLiteRAG 在所有测试组合中扩展 SLO-compliant request rate range，论文摘要和结论报告最高可比 baseline 提升 1.5× SLO-bound throughput，且不增加额外硬件。CPU-only 在轻负载下也可能因检索慢而 SLO violation；DED-GPU 因减少 LLM 可用 GPU 而吞吐上限低；ALL-GPU 在大库和高负载下因为 KV cache 受挤压和 kernel contention，ORCAS 2K + Llama3-8B / Qwen3-32B 中 end-to-end latency 可超过 2×。Dynamic dispatcher ablation 显示其通过 overlap early-query merge/rerank 与慢查询扫描，将 search latency 最高降低 16%。在线 index update 从 profiling、算法、splitting 到 loading 总体在 1 分钟内完成，单 shard 生成和加载小于 10 秒；刷新期间 router 临时把对应 cluster 的请求切回 CPU path 以保持服务连续。
    - 实验限制与不确定性：SLOsearch 是论文启发式设定，因为 retrieval 阶段没有通用标准；LLM throughput 与 KV cache 空间关系用插值近似，作者认为是保守下界，但仍会受具体 serving runtime、allocator 和模型配置影响。hit-rate variance 用 Beta distribution 近似，适合论文观测到的分布形态，但极端 workload drift 需要更频繁 reprofiling。Artifact 公开，但完整预处理需要 1.5TB 到 2TB 存储和 40-60 小时，完整评估约 10 小时，并依赖 H100 / L40S 多 GPU 环境、Intel MKL、NVCC 12.1、GCC 11.3、Python 3.10 等具体软件栈。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：论文没有提出新的 CUDA attention kernel 或新的 ANN 算法核心，而是提出 Distributed VectorLiteRAG Pipeline，即一个 latency-aware hybrid CPU-GPU retrieval pipeline。它的核心执行路径是：CPU coarse quantizer 先确定候选 clusters；Router 根据 mapping table 把 hot cluster scanning 发到对应 GPU shard，把 cold cluster scanning 留给 CPU FastScan；GPU worker 和 CPU worker 并行完成各自 probe；Dynamic dispatcher 以 query 粒度提前合并 CPU/GPU partial results，rerank 得到 final top-k vectors，并将完成的 query 推进到 document retrieval 和 LLM prefill / decode。
    - 为什么 CQ 留在 CPU：论文明确不给 CQ 做 GPU offload。原因一是 CQ 通常使用 HNSW 等 memory-intensive quantizer structure，搬到 GPU 会增加额外图结构内存；原因二是若 CQ 分布到 GPU shards，会产生 CPU -> GPU quantization、GPU -> CPU merge/routing、CPU -> GPU search、GPU -> CPU final merge 的多次设备往返和同步；论文目标是满足稳定 latency budget，而不是最小化单次 search absolute latency。因此 VectorLiteRAG 只 offload LUT scanning / distance computation 中命中 hot cluster 的部分。
    - 一个请求的执行流例子：用户 query 到达后先由 embedding model 生成 query vector。系统把一批 query 送到 CPU coarse quantizer，得到每个 query 需要 probe 的 IVF cluster 列表。假设某个 query 的 nprobe=2048，其中一部分 cluster 属于 GPU hot set：Router 查 mapping table，发现 cluster 17、91、208 属于 GPU 0 的 local IDs，cluster 45、132 属于 GPU 2，其余 cluster 是 cold clusters。Router 只向 GPU 0 / GPU 2 发送实际 resident probes，并给每个 shard 设置对应的 flexible nprobe；CPU 同时对 cold clusters 做 FastScan。GPU worker 完成自己的 cluster scan 后设置 completion flag；CPU scan loop 每处理完一组 cluster 通过 callback 检查该 query 是否已达到 expected nprobe。若该 query 的所有 CPU/GPU probes 都完成，dispatcher 从 thread-safe queue 取出 partial results，merge 并 rerank top-k vectors，映射到文档，随后把上下文送入 vLLM 的 continuous batching 队列执行 prefill / decode。这样高 hit-rate query 不必等待同 batch 中低 hit-rate query 扫描完冷 cluster。
    - partitioning pipeline 例子：离线 profiling 先测 CPU CQ 和 LUT 在不同 batch size 下的 piecewise linear latency，统计 cluster access frequency，并测 bare LLM throughput 与 KV cache 空间。给定 SLOsearch 后，系统用 τs = SLOsearch / (1 + ε) 设置检索延迟预算，ε=1 近似最坏排队延迟等于一个 batch latency。对某个候选 GPU cache coverage ρ，系统根据热 cluster 覆盖率得到 mean hit rate，再用 Beta distribution 和 order statistics 估计 batch 内 minimum hit rate ηmin。由于 CPU LUT workload 约与 miss rate (1 - η) 成正比，hybrid search latency 被建模为 TCQ_CPU(b) + (1 - η) * TLUT_CPU(b)。如果估计 latency 超 SLO，就提高 GPU cache coverage；如果满足 SLO 且 GPU memory 占用影响 LLM throughput，则继续二分寻找合适的 partition point。最终 hot clusters 被拆分成 GPU shards，并生成 routing mapping tables。
    - runtime update pipeline：服务运行时 Router 周期性记录 average hit rate 和 per-cluster access frequency。每隔几分钟或几千请求后重置计数器以检测 drift；当 SLO attainment 降低且 observed hit rate 偏离 expected value 时，触发 update cycle：重新 profiling 当前访问模式、运行 latency-bounded partitioning algorithm、生成新 shards、加载到 GPU。刷新某个 GPU shard 时，Router 暂时把该 shard 对应 cluster 的请求导向 CPU，加载完成后再切回 GPU。论文避免 per-cluster update，因为 cluster contiguous layout 有利于高带宽访问，单独更新不同大小 cluster 会造成 memory fragmentation；full-shard refresh 更简单且单 shard 迁移只需数秒。
    - pipeline 的边界和适用性：该 pipeline 适合 IVF-like clustered vector index、访问分布有 cluster-level skew、检索能受益于 GPU acceleration、LLM 与 retriever 共享 GPU 且存在严格 TTFT / SLO 的在线 RAG 服务。它不是面向所有 ANN index 的通用替代品；若索引结构不是 cluster-based，或访问无明显热 cluster，或者 GPU memory 极度充裕到无需精细 partition，收益会下降。论文未来工作提到可扩展到 prefill-decode disaggregation，因为 retrieval 是 bandwidth-bound，prefill 是 compute-intensive，但那需要同时建模 vector search、prefill 和 decode 的吞吐与干扰。
