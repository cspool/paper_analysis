    1、论文工作：
        - 论文要解决的核心问题：现有 FaaS / serverless 平台把一次函数调用看成“启动一段用户代码并按占用资源时间计费”，但函数真正需要的网络数据依赖通常藏在代码内部，例如函数启动后才通过 socket / HTTP / S3 API 去取对象。平台在放置和调度函数时不知道 𝑓 实际要计算 𝑓(x)，也不知道 x 的大小、位置、是否可由其他计算生成、下游是否马上消费输出。因此一旦 x 不在附近或网络 / 存储延迟较大，函数会在已分配 CPU / RAM 的容器里等待 I/O，造成 CPU starvation、内存占用和网络传输浪费。这个问题在短计算、长网络延迟、成千上万并行 serverless invocation 的工作负载中尤其突出。
        - 论文的主要贡献：论文提出 Fix，一种面向 serverless computing 的低层 ABI / 计算表示，把程序、用户和平台共享的“计算是什么”统一成 deterministic procedure + well-specified data / computation outputs。核心思想是 externalizing network I/O：用户函数不能在执行中任意发起网络 I/O，而是必须在 invocation 表示中声明可访问数据、不可访问引用、延迟计算、以及对 thunk 的求值请求。论文还实现了 Fixpoint runtime，用 packed binary Fix objects、Wasm-to-x86 trusted toolchain、共享地址空间 codelet、Flatware POSIX/WASI 适配层和分布式调度器展示这种表示可以以约 1.5 us 的 invocation overhead 运行，并能让平台先拉取依赖、再分配 CPU / RAM、按数据 locality 选择执行位置。
        - 论文所处背景：该工作处于 serverless computing、I/O-compute separation、cluster orchestration、workflow serverless、content-addressed storage 和轻量容器隔离的交叉位置。它不是在现有 Lambda / OpenWhisk 语义下继续优化 cache、container warmup 或 gateway，而是改变 outsourced computing 的接口：从“用户代码内部自己做 I/O，平台为占用时间收费”的 pay-for-effort 模型，转向“用户声明 deterministic computation，平台负责移动数据和调度执行”的 pay-for-results 模型。论文目标负载包括图像处理、编译、B+-tree / 文件系统式树遍历、map-reduce 字符串统计等短任务或 burst-parallel 任务，而不是依赖共享可变状态、数据库事务或任意 Web API 的应用。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：OpenWhisk + Kubernetes + MinIO 代表典型 FaaS：函数被调度到某台机器后才向对象存储读取输入，平台无法在资源分配前看到数据依赖，因此容易让 CPU / RAM 被等待 I/O 的容器占住。Ray 能表达动态 task graph 和 ObjectRef，但依赖解析绑定在语言 runtime 和 client / scheduler 交互里，Linux executable 只能通过 Popen 运行并从 MinIO 读写；Ray blocking-style 会在函数内部 ray.get 时占用 worker 等 I/O，Ray continuation-passing-style 虽接近 Fix 的分解方式，但函数调用 / orchestration overhead 高，细粒度拆分会得不偿失。Pheromone 可表达 serverless workflow 的函数依赖并降低 orchestration overhead，但依赖粒度主要是函数级和中间结果级，难以表达每个 invocation 对外部 durable storage 中数据子集的精确需求。Faasm 用 Wasm 做轻量隔离，但仍提供更通用 host interface，如文件 I/O 和共享状态，因此没有强制 I/O externalization，runtime overhead 也更重。
        - 论文的设计方法：Fix 用 Blob、Tree、Object、Ref、Thunk 和 Encode 建立统一可序列化表示。Blob 是字节区，Tree 是 Handle 集合；Object 表示下一次函数调用可直接读取的数据，Ref 表示可知道类型和长度但不可直接读取的数据；Thunk 表示延迟计算，包括 Application Thunk、Identification Thunk 和 Selection Thunk；Encode 表示对 Thunk 的求值请求，Strict Encode 会递归求完全值并变成 Object，Shallow Encode 只做足以继续推进的求值并返回 Ref。每个 Thunk 有 minimum repository，即运行前必须可用的数据和 RAM 等资源上界。函数执行期间不能扩大自己的 repository、不能阻塞等待新 I/O、不能直接调用另一个函数，只能构造新的 Thunk / Encode 让 runtime 在下一阶段执行。
        - 方法如何对冲 Baseline 缺陷：相对把 I/O 藏在用户进程里的 FaaS，Fix 让平台在 invocation 启动前就知道下一阶段真正需要的 code、data、resource limit 和可能的下游 dependency，因此可以并行传输代码和数据、等数据到达后再分配 CPU / RAM、按数据位置调度到最少数据移动的节点、把大输出放在下游附近，甚至在输入可由 deterministic computation 重算时选择重算而不是跨网传输。相对 Ray / Pheromone，Fix 把 data flow 和 control flow 绑定到一个 packed binary object 中随数据一起传递，不依赖中心 scheduler 维护额外 metadata，也不需要每次 invocation 都向 client / scheduler round-trip。相对粗粒度 workflow DAG，Fix 的 Selection Thunk 和 minimum repository 支持只取大型 Tree / Blob 中的必要子范围，让 B+-tree、目录树、嵌套数据结构遍历能以更小 memory footprint 运行。
        - 关键 trade-off：Fix 牺牲了通用性来换取可调度性和确定性。用户程序必须是 pure function，输入只能来自 content-addressed data 或其他 computation outputs；函数不能访问容器外数据，当前不支持 clock、true random、多用户数据库、消息总线、任意 Web API 等 nondeterministic services。现有 Linux executable 不能直接运行，程序需通过 Fix-targeting toolchain 重新编译；POSIX 程序可借 Flatware 适配，但 socket、thread 等未支持依赖需要修改。开发者还要把程序拆成 I/O 和 compute 阶段、选择每个阶段的 minimum repository，并在复杂数据依赖中以 continuation-passing style 暴露未来依赖，这会增加编程负担。

    3、论文实现：
        - Baseline 如何实现：论文把 Fixpoint 与 OpenWhisk + Kubernetes + MinIO、Ray、Pheromone 和 Faasm 比较。OpenWhisk 由 Kubernetes 负责 container factory / placement，函数二进制、输入和输出通过 MinIO 管理。Ray 有三种实现：Ray + MinIO 用 Linux executable 从 MinIO 读写并由 Ray Popen 启动；Ray blocking-style 用 Python Ray API，在函数内部 ray.get 阻塞等待 ObjectRef；Ray continuation-passing-style 避免 ray.get，遇到新依赖就发起新的 Ray remote function。Pheromone 部署在 Kubernetes 上，用函数依赖和 bucket 依赖表达 serverless orchestration。Faasm 本地用 Docker 部署，作为使用 Wasm 隔离但不 externalize I/O 的 serverless runtime baseline。
        - 新设计如何实现：Fixpoint 是 Linux application，可作为多节点 Fix runtime。用户 procedure 被编译成 x86-64 machine codelet，每个 codelet 是 ELF，入口为 _fix_apply，输入为 TreeObject Handle，返回 Fix object Handle。Fixpoint 当前使用受信任 Wasm toolchain 保证 codelet 的 memory safety 和 determinism：Wasm 先经 wasm2c 转成 C，再用 libclang 并行编译成 object files，最后由 liblld 链成 ELF；Fixpoint 内部有小型 in-memory ELF linker 将 codelet 与 Fixpoint API 链接。Fixpoint API 允许 codelet attach_blob / attach_tree、create_blob / create_tree、创建 application / identification / selection Thunk、创建 strict / shallow Encode，并查询 Handle 类型和大小。Handle 由截断 192-bit BLAKE3 hash、16-bit metadata/type 和 48-bit size 组成，小于等于 30 bytes 的 Blob 可直接 literal 存在 Handle 中；Handle 可用 AVX2 256-bit vector register 传递。Flatware 在 Fixpoint API 上实现 WASI / Unix-like filesystem 视图，使 CPython 3.12、clang 等 POSIX 风格程序能把 Thunk 参数当作文件系统依赖来访问。
        - 实验 / 实现平台：实验运行在 Amazon EC2 m5.8xlarge 实例上，每台 32 vCPU、128 GiB memory，存储为 EBS gp3 3000 IOPS。分布式实验使用 10 节点、总计 320 vCPU 的集群。Fixpoint worker thread 共享 job queue、runtime storage 和 program registry；每个 worker 内有 Scheduler，根据 Fix semantics 递归解析 Application Thunk Tree，收集 data dependency 和 computation dependency。多节点 Fixpoint 通过 Network Worker 交换 Fix values 和对象位置信息；当前实现没有中心调度器，每个本地 scheduler 根据“哪些对象在哪些节点”的 view 选择最少数据移动的节点执行，未来计划替换为分布式 object store。
        - 关键实验设置与指标：论文评估 invocation overhead、500 函数链 orchestration overhead、带 150 ms 远端存储响应的 one-off function、984 个 100 MiB Wikipedia shard 的字符串统计、6M Wikipedia title 的 B+-tree traversal、大规模 C 编译任务和 SeBS 函数移植。关键结果包括：add 两个 8-bit 整数的 Fixpoint invocation overhead 约 1.46 us，Linux vfork+exec 为 449 us，Pheromone 为 1.05 ms，Ray 为 1.29 ms，Faasm 为 10.6 ms，OpenWhisk 为 30.7 ms；500 function chain 中 Fixpoint 和 Pheromone 能一次表达控制流，Ray 需要 500 次 client round-trip。远端存储 one-off function 中，Fixpoint 因先取依赖后分配资源，执行 1024 个请求耗时约 268 ms、吞吐 3827 tasks/s，而 internal I/O ablation 为 2638 ms、388 tasks/s。Wikipedia 字符串统计中，Fixpoint 为 3.25 s，Fixpoint no-locality 为 31.43 s，OpenWhisk + MinIO + K8s 为 63.68 s，Ray continuation-passing 为 6.39 s，Ray blocking 为 42.29 s；Fixpoint CPU waiting 比例约 37%，internal I/O 和 OpenWhisk 约 92%。B+-tree arity 256 时，Fix 为 0.14 s，Ray blocking 为 2.8 s，Ray fine-grained continuation-passing 为 5.74 s。编译近 2000 个 C source file 时，Fixpoint 为 39.53 s，Ray + MinIO 为 76.87 s，OpenWhisk + MinIO + K8s 为 100.01 s。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：论文没有提出 GPU / CPU 算子 kernel；最核心的新执行路径是 Fix 的 externalized-I/O invocation pipeline，以及 Fixpoint 对这个 pipeline 的 runtime 实现。这个 pipeline 由 Fix packed binary ABI、content-addressed Handle、minimum repository、Object / Ref / Thunk / Encode、Fixpoint Scheduler、runtime storage、worker threads、Network Worker、trusted Wasm-to-codelet toolchain 和 Flatware POSIX adapter 组成。可以把它理解为一种“先声明计算和数据依赖，再由平台执行数据移动和函数放置”的 serverless runtime path：函数本身只在已知依赖齐备时运行，运行中不再阻塞网络 I/O；如果需要后续数据或子计算，函数返回新的 Thunk / Encode，让 runtime 接管下一阶段。
        - 新pipeline/kernel的执行流例子：以论文中的图像缩放 𝑓(x) 或 Wikipedia shard count-string 为例，用户不再上传一个启动后自己去 S3 取 x 的黑盒函数，而是构造一个 Application Thunk：Tree 中包含 resource limit、procedure handle、输入数据 handle 或对输入的 Encode。Fixpoint scheduler 收到这个 Thunk 后，先沿 Fix semantics 展开 Tree，识别哪些是 Object、哪些是 Ref、哪些 Encode 需要求值；对于尚未本地可用的数据，它让 Network Worker 从持有节点取回 Blob / Tree，或在另一个节点先计算相关 Thunk。若某个 shard 已在节点 A，而 codelet 和其他依赖迁移到 A 的成本最低，scheduler 选择在 A 执行；若输出 size hint 表明下游消费大结果，scheduler 可把上游计算放到更接近下游依赖的位置。只有当 minimum repository 中声明的输入、代码和 RAM 资源都准备好，worker thread 才直接跳转到 codelet 的 _fix_apply，而不是创建 Linux process 或容器。codelet 通过 attach_blob / attach_tree 零拷贝读取已授权 Object，不能 map Ref；如果它只需读取一个 Tree 的第三个子项，会创建 Selection Thunk 和 Shallow / Strict Encode，让 runtime 只拉取这个子项，而不是把整个 Tree 加进当前 repository。函数返回后，返回值可能是 Blob / Tree，也可能是表示下一步的 Application Thunk；Fixpoint 把结果写入 runtime storage，并唤醒依赖它的 Encodes。对 map-reduce 字符串统计而言，984 个 count-string Thunk 可在拥有 shard 的节点并行运行，merge-counts Thunk 逐层归约；由于数据依赖在启动前可见，CPU 不被等待远端 shard 的 invocation 占住，节点间只传必要输入和中间计数结果。
