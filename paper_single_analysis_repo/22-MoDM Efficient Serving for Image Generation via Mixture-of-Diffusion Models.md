论文标题：MoDM: Efficient Serving for Image Generation via Mixture-of-Diffusion Models

大小Model混合生成图片，text2text相似性（Cache hit）跳过去噪步骤时使用的中间数据，使用缓存的最终图像加噪（逆向生成到当前step）而不是中间latent，后续使用小Model继续去噪。

1、论文工作：
    - 论文要解决的核心问题：文本到图像 diffusion serving 存在天然的 latency-quality trade-off：大模型如 Stable Diffusion-3.5-Large、FLUX 质量高但推理慢，小模型如 SDXL、SANA 推理快但质量较低。传统 serving 若所有请求都用大模型，吞吐和尾延迟在请求突发时很快成为瓶颈；若直接用小模型或蒸馏模型，则图像分布质量、FID 或细节保真容易下降。MoDM 要解决的是在在线图像生成服务中，如何在满足 SLO 的同时保持接近大模型的生成质量。
    - 论文的主要贡献：第一，提出缓存最终生成图片而非中间 latent / activation / patch，使缓存内容跨模型、跨模型家族可复用。第二，用 text-to-image similarity 做缓存检索，而不是只比较 prompt 文本 embedding，从而提升检索到的图像与用户视觉意图的一致性。第三，提出 cache hit 请求的生成路径：取回相似图片、重新加噪到中间 timestep，再用小模型完成剩余 denoising steps；cache miss 请求仍由大模型完整生成。第四，提出 Request Scheduler 与 Global Monitor，按 cache hit/miss、请求率、denoising step 分布和 GPU worker 数动态分配大/小模型，在质量优先和吞吐优先两种模式之间调整。第五，在 DiffusionDB、MJHQ、SD3.5L、FLUX、SDXL、SANA 等设置上验证，报告相对大模型-only baseline 最高约 2.5x 到 3.2x throughput 提升，并在附录中报告能耗下降。
    - 论文所处背景：Diffusion 模型通过 50 步左右的 iterative denoising 生成图像，每一步都要运行完整模型，因此单次请求可能需要十几秒甚至更久。已有缓存方法如 Nirvana 能复用中间 latent 并跳过部分 denoising steps，但中间状态与具体模型结构、noise schedule 和内部 representation 绑定，难以跨模型共享；同时 Nirvana 论文中报告的计算下降约 20%，在高请求率时仍不足以避免 SLO violation。MoDM 的背景判断是：生产服务中用户常进行相似 prompt 的反复试探，存在时间局部性；最终图片本身是模型无关的视觉对象，可以成为更通用的缓存单位。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：Vanilla baseline 对每个请求都用大模型完整 denoise，质量可靠但 compute-bound，尾延迟和吞吐随请求率升高迅速恶化。Nirvana baseline 缓存中间 latent，用 text-to-text similarity 选择相似 prompt 并跳过部分 denoising steps，但 latent 需要按模型存储，论文以 SD3.5L 为例指出多个 latent 中间态约 2.5MB/image，高于最终图片约 1.4MB/image；并且 latent 不兼容不同模型，限制了 mixture-of-models serving。Pinecone 类 baseline 只按文本 embedding 找相似图片并直接返回或重新生成，没有 generative refinement，容易出现 image-text alignment 差。单独使用 SDXL、SANA 或 SD3.5L-Turbo 这类小/蒸馏模型虽快，但质量指标尤其 FID 与大模型差距明显。
    - 论文的设计方法：MoDM 把服务路径拆成缓存检索、加噪重入 diffusion、大小模型分工和资源全局调度。Request Scheduler 收到 prompt 后用 CLIP text embedding 到缓存中的 image embedding 做 cosine similarity 检索；若相似度超过阈值，则视为 cache hit，取回最终图片，根据相似度选择可跳过的 denoising step 数 k，再按 image-to-image diffusion 的方式给图片加噪并交给小模型完成 T-k 个步骤；若 cache miss，则放入 miss queue，由大模型从噪声开始完整生成 T steps。Global Monitor 周期性统计请求率、cache hit rate、k 分布和 worker throughput，通过启发式加 PID 控制在大模型 worker 与小模型 worker 之间分配 GPU。
    - 方法如何对冲 Baseline 缺陷：相对 Vanilla，MoDM 对 cache hit 请求不再完整运行大模型，而是让小模型只完成后半段 refinement，直接减少 denoising steps 和单步模型成本。相对 Nirvana，MoDM 缓存最终图片，因此同一缓存能被 SD3.5L、FLUX、SDXL、SANA 等不同模型使用，避免模型专属 latent cache 的碎片化；text-to-image retrieval 也能利用图像 embedding，论文用 CLIPScore 和 PickScore 显示其视觉对齐优于 text-to-text retrieval。相对 Pinecone，MoDM 不是简单返回旧图，而是重新加噪并按新 prompt denoise，使结果可以继承缓存图像的高层结构，同时调整颜色、纹理和细节。相对单独小模型，MoDM 的 cache miss 仍由大模型生成，cache hit 的小模型输出也受高质量缓存图片约束。
    - 关键 trade-off：缓存最终图片带来跨模型兼容性和较低存储开销，但它要求维护 image embedding 并执行相似度检索；论文报告 100000 个 image embedding 约 0.29GB，检索延迟约 0.05s，相比 10s 级 denoising 可忽略。k 的选择是质量-速度核心 trade-off：相似度越高可跳过越多早期 denoising steps，速度越快，但过大的 k 会让输出过于接近缓存图或质量下降；论文用 alpha=0.95 的质量约束离线选择阈值。缓存策略也有 trade-off：只缓存大模型输出质量更稳，缓存所有模型输出可提高 hit rate 和 throughput，但 FID 有轻微下降。系统模式上，quality-optimized mode 倾向保留更多大模型 worker，throughput-optimized mode 则让 hit 请求尽量走小模型。

3、论文实现：
    - Baseline 如何实现：Vanilla baseline 是每个请求都用大模型完整生成，不使用缓存。Nirvana baseline 复用近似缓存中间 latent，按 prompt 文本相似度找缓存项并跳过部分 denoising steps。Pinecone baseline 用 CLIP text embedding 做最相似 prompt/image 检索，命中时直接服务检索图片，不执行 refinement，未命中时重新生成。论文还对比 SDXL、SANA、SD3.5L-Turbo 等 standalone small/distilled 模型，以区分系统级缓存收益和模型压缩/蒸馏收益。
    - 新设计如何实现：MoDM 用 Python 和 PyTorch 实现，请求调度器、Global Monitor 和每个 worker 都运行在独立进程中，节点通信通过 PyTorch RPC 完成。缓存项包含最终图片及其 CLIP image embedding；scheduler 侧运行 CLIP 文本编码，按 cosine similarity 检索缓存。cache hit 时，系统按照 diffusion model 的 noise schedule 把缓存图片重噪到目标 timestep，再运行剩余 T-k denoising steps。论文把 k 限制在 {5, 10, 15, 20, 25, 30}，总 denoising steps T=50，并用离线 10000 个 prompt / 100000 张缓存图片的实验确定相似度阈值。Global Monitor 使用两类模式：质量优先模式先保证 miss workload 由大模型满足，再尽量最大化 large worker 数；吞吐优先模式根据 hit/miss workload 和大小模型吞吐比按比例分配 worker，并用 PID 参数 Kp=0.6、Ki=0.05、Kd=0.05 平滑调整。
    - 实验 / 实现平台：论文部署在两类硬件上：一台含 4 张 NVIDIA A40 48GB GPU 的服务器，以及 16 节点 AMD MI210 集群，每节点 4 张 MI210 64GB GPU。模型包括大模型 SD3.5L 8B、FLUX.1-dev 12B，小模型 SDXL 3B、SANA 1.6B，以及蒸馏 baseline SD3.5L-Turbo。除 SDXL 使用 FP16 外，其余模型按开发者建议使用 BF16；大多数模型使用 T=50 denoising steps，SD3.5L-Turbo 使用 10 steps。数据集包括 DiffusionDB 2M prompts/images 和 MJHQ-30k。arrival process 用 homogeneous Poisson process 建模，并按 DiffusionDB 原始到达顺序重放以模拟生产负载。
    - 关键实验设置与指标：系统指标包括 maximum throughput、SLO compliance、99th percentile tail latency、maximum sustainable load；质量指标包括 CLIPScore、FID、Inception Score 和 PickScore。吞吐实验用每个数据集 10000 个请求和 10000 张大模型生成缓存；latency/SLO 实验按不同 request rate 加 timestamp；质量实验先 warm up 生成 10000 张缓存图，再服务 10000 个请求。主要结果包括：DiffusionDB 上 MoDM-SANA 相对 Vanilla SD3.5L 达到 3.2x throughput，MoDM-SDXL 达到 2.5x；MJHQ 上因时间局部性弱，提升降为 2.4x 和 2.1x；以 FLUX 为大模型时，MoDM-SANA / MoDM-SDXL 仍达到约 2.9x / 2.4x。SLO 方面，A40 上 Vanilla 和 Nirvana 在请求率超过约 5 req/min 后明显违约，而 MoDM 可支撑到约 10 req/min；16 MI210 上 MoDM 在 2x latency SLO 下支撑到约 22 req/min，在 4x SLO 下支撑到约 26 req/min。质量方面，MoDM 的 CLIP、IS、Pick 接近大模型 baseline，FID 明显好于 standalone 小模型，但 MoDM-SANA 的 FID 比 MoDM-SDXL 差，显示更激进小模型带来质量代价。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：论文没有提出新的 CUDA kernel，而是提出一条 cache-aware mixture-of-diffusion-models serving pipeline。它的核心 runtime path 是“最终图片缓存检索 -> 重噪到中间 timestep -> 小模型 refinement / 大模型 full generation -> 动态 GPU worker 分配”。这里的创新点不在单个算子，而在把缓存粒度从模型内部 latent 提升到最终图片，并把 cache hit 请求改造成 image-to-image diffusion refinement 请求，从而让小模型处理绝大多数相似请求。
    - 新 pipeline/kernel 的执行流例子：假设用户提交 prompt “a garden that can grow from weeds in watercolor style”。Request Scheduler 先用 CLIP text encoder 得到 query embedding，然后和缓存中每张图片的 CLIP image embedding 做 cosine similarity。若最相似缓存图超过阈值，例如检索到一张“large garden with growing flowers and vegetables in the sun illustration style”的图片，scheduler 判定为 cache hit。相似度越高，系统选择越大的 k，例如从 {5,10,15,20,25,30} 中选一个值；较大的 k 表示跳过更多早期 denoising steps。随后 scheduler 按目标 timestep 的 sigma 把缓存图片和 Gaussian noise 混合，构造 noisy image latent，将该请求放入 cache-hit queue。小模型 worker 取走请求后，只执行剩余 T-k 个 denoising steps，按新 prompt 调整图像细节并输出最终图片。
    - cache miss 路径例子：如果新 prompt 与缓存没有足够接近的图片，scheduler 把请求放入 cache-miss queue。大模型 worker 运行 SD3.5L 或 FLUX 的完整 50-step generation，生成高质量新图。该新图会被加入 FIFO image cache，同时存储对应 image embedding，旧缓存按滑动窗口策略淘汰。论文观察到 DiffusionDB 中超过 90% 的 cache-hit 请求会命中 4 小时内缓存的图片，因此 FIFO 不仅简单，还能利用用户连续 prompt 修改的时间局部性。
    - Global Monitor 调度路径例子：系统每个 monitoring period 收集请求率 lambda、cache hit rate Hcache、cache hit 请求的 k 分布、大/小模型的单 GPU throughput。若 miss workload 升高，monitor 必须分配足够 large workers，因为 cache miss 只能由大模型完整生成。剩余资源再服务 hit workload；在 quality-optimized mode 中，它尽量增加 large worker 数以保持质量，在 throughput-optimized mode 中则按 hit/miss workload 和 small/large throughput 比例分配 worker。PID controller 平滑 num_large 的变化，避免请求率波动时频繁切换模型造成系统振荡。最终，worker 列表被更新为若干 large model worker 和若干 small model worker：large worker 优先处理 miss，也可处理 hit；small worker 专注处理 cache-hit refinement。
    - pipeline 的边界和限制：MoDM 的收益依赖请求流中的语义/视觉重复和时间局部性。MJHQ 因缺少强时间相关性，hit rate 和吞吐收益低于 DiffusionDB。最终图片缓存虽然跨模型，但如果检索相似度误判，refinement 仍可能继承错误结构；因此 text-to-image threshold 和 k 选择是质量安全阀。论文的质量指标主要基于 CLIPScore、FID、IS、PickScore，虽覆盖语义和分布质量，但不能完全替代真实用户偏好或安全审查。Artifact 要求至少 4 张 40GB 以上 GPU，并依赖 gated SD3.5L 权限、DiffusionDB/MJHQ 数据与预生成缓存，因此完整复现实验门槛较高。
