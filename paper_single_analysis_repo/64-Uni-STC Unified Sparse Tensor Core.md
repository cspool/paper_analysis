论文标题：Uni-STC: Unified Sparse Tensor Core

GPU Sparse TC的新设计，面向SpMM（A稀疏）、SpMV（A稀疏）、SpMSpV（A和V稀疏）、SpGEMM（A和B稀疏）算子。

1、论文工作：
    - 论文要解决的核心问题：现有 sparse tensor core 对通用稀疏计算的支持不完整，并且在非结构化稀疏矩阵上利用率和能效不足。已有 DS-STC 主要支持 SpGEMM 的 outer-product 路径，RM-STC 也是面向 SpGEMM 的 row-row 路径；它们难以统一覆盖 SpMV、SpMSpV、SpMM、SpGEMM 四类常见 sparse kernel。同时，这些架构的任务划分较粗，依赖固定形状的 T2/T3 子任务，遇到长行、长列、对角集中或双边稀疏时会产生大量低 MAC utilisation 周期，并通过较大的网络持续搬运中间乘积，带来能耗问题。
    - 论文的主要贡献：论文提出 Uni-STC，一个替代 GPU SM 中原始 tensor core 的统一稀疏张量核。它的贡献包括：第一，提出 Bitmap-Bitmap-CSR（BBC）格式，用 CSR 组织 4x4 sparse tile，用两级 bitmap 描述 tile 内非零位置，使四类 kernel 共享同一数据结构；第二，提出 TMS、DPG、SDPU 三个功能单元，把 16x16x16 的 T1 任务拆成 4x4x4 的 T3 任务，再拆成 1x1x4 的 T4 segmented dot-product 任务；第三，设计 UWMMA 指令序列和软件/硬件协同数据流，让 SM 通过 load、task generation、numeric execution 三类操作驱动 Uni-STC；第四，在 SuiteSparse、DLMC 和 AMG solver 上评估性能、能耗和面积。
    - 论文所处背景：tensor core 已经成为 GPU、CPU、TPU 中提升 GEMM 吞吐和能效的核心单元，但它们多面向 dense 或结构化稀疏。科学计算和 AI 工作负载中，GNN、AMG、BFS、稀疏 DNN 推理会同时调用不同 sparse kernel，且稀疏模式高度不规则。论文认为只扩展软件接口不足以解决问题，因为底层硬件仍会被固定任务形状、低数据复用和中间结果传输限制；因此需要 storage format、指令、调度和硬件 dataflow 的协同设计。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：论文主要对比 DS-STC、RM-STC、NV-DTC，并在部分实验中比较 GAMMA、SIGMA、Trapezoid。NV-DTC 作为 dense tensor core 只能有限利用稀疏性，真实矩阵上大量周期 MAC 利用率低于 25%。DS-STC 的 outer-product 任务以 A 的半列和 B/x 的半行为基本单位，RM-STC 的 row-row 任务以 scalar-vector 组合为基本单位；两者都把 sparse data gather 成固定形状任务，容易出现无效访存和任务负载不均。论文给出的量化观察是，DS-STC 和 RM-STC 有 61.68% 和 62.78% 的周期低于 50% utilisation，而 Uni-STC 的对应低利用率周期比例显著更低。它们还受 task concatenation 约束：DS-STC 难以沿 K 维拼接不同位置的任务，RM-STC 基本只沿 N 维拼接，无法把很多低负载任务灵活聚合到同一 MAC array。
    - 论文的设计方法：Uni-STC 的核心方法是从“gather data”转向“gather tasks”。软件侧用 BBC 格式把 sparse matrix 拆成 top-level CSR sparse tile 和 tile 内两级 bitmap；硬件侧用 TMS 读取 top-level bitmap 并生成 4x4x4 T3 task，用 8 个 DPG 并行读取底层 bitmap，把 T3 再分解为 1x1x4 T4 sparse dot-product task，并把这些任务以 Z-shaped 顺序放入 dot-product queue；最后 SDPU 将相邻 T4 任务合并为 segmented dot-product，在局部执行多个短向量点积并预合并 partial products。Uni-STC 还采用两层 network 和 dynamic DPG activation：TMS 根据 queue head 的 intermediate product prefix sum 决定需要开启多少 DPG，多余 DPG 和关联网络被 power-gate。
    - 方法如何对冲 Baseline 缺陷：BBC 避免了不同 sparse kernel 之间的在线格式转换，也避免 RM-STC 类硬件 decoder 的高面积成本；4x4x4 T3 任务在 M/N/K 三个维度上保持对称，有利于统一 bitmap 解码和多 kernel 支持；1x1x4 T4 任务则把原本形状僵硬的 outer-product/row-row 任务转成约束更少的 dot-product 片段，使多个低负载任务可以被拼接到 SDPU 上执行。TMS 的 task ordering 会在 outer-product 与 row-major 顺序之间选择，以提升 A/B tile 复用并降低 write conflict；SDPU 的 merge-forward 结构可以在写 C 前合并最多四个 partial products，减少 C 网络和写回流量。相比 DS-STC/RM-STC 持续移动大量中间乘积，Uni-STC 更倾向移动紧凑控制信息，数值数据通过 buffer/register 与本地网络按需转发。
    - 关键 trade-off：Uni-STC 接受了额外硬件和软件预处理成本。硬件上，TMS/DPG、SDPU 附加加法器、metadata buffer、A buffer、accumulator buffer 和 Benes/MUX networks 带来面积开销；论文估计在 A100 类 432 个 Uni-STC 单元部署下，额外面积约为 0.0425 mm2，对 826 mm2 die 约 2.12%。软件上，BBC 需要一次离线构建；论文报告在 64-core AMD EPYC 7702 CPU 上小于 1000 ms，在 NVIDIA A100 GPU 上小于 100 ms，并认为可被 GNN training、linear solver 等迭代应用摊销。另一个 trade-off 是 DPG 数量：DPG 从 4 增至 8 对 SpMM/SpGEMM 更有利，但对 SpMV/SpMSpV 的 EED 会略降；论文最终选择 8 DPG 作为平衡点。

3、论文实现：
    - Baseline 如何实现：论文在硬件比较中基于 Accel-Sim 扩展 STC simulator，并加入 asynchronous memory access 支持，用同一 T1 task 调用粒度比较 GAMMA、SIGMA、Trapezoid、NV-DTC、DS-STC、RM-STC 和 Uni-STC。为了公平比较不同架构，论文按理论计算吞吐对齐 MAC array，例如 64 MAC@FP64 或 128 MAC@FP32，并采用 SIGMA 的 PE 设计缩放不同 accelerator。能耗根据 register activity 和 Sparseloop 方法外推；面积使用 Yosys、FreePDK45 与 CACTI 7 建模。
    - 新设计如何实现：Uni-STC 被集成为 GPU SM 内的独立 coprocessor，需要更新 instruction decoder 解析 UWMMA opcode，并扩展 warp scheduler 分发指令。数据交互主要通过 register file 和 operand collector 完成；SM90+ 可利用高带宽 operand collector，而 Ampere 这类早期架构需要拓宽 register-file port，论文给出的需求是每线程每周期最多 16 个 FP64 source 和 4 个 FP64 destination operands。指令生命周期包括 stc.load 同步收集 metadata/values，stc.task 异步触发 TMS/DPG 生成 task queue，stc.numeric 检查 READY/BUSY 状态并驱动 SDPU 执行，完成后结果写回 register file。BBC 的 ValPtr_Lv2 直接提供给 Uni-STC，使 TMS 能控制 tile 内数据转发，避免复杂硬件解码。
    - 实验 / 实现平台：数据集方面，SpMV、SpMSpV、SpMM 使用 SuiteSparse 的 2893 个矩阵，SpGEMM 使用其中 2126 个方阵并计算 C=A2；DNN inference 使用 DLMC 的 302 个权重矩阵，模型包括 ResNet-50 和 Transformer，稀疏度设置为 70% 和 98%；SpMSpV 的输入向量随机生成 50% sparsity，SpMM 的 B 矩阵列数设为 64。应用级实验将已有 AMG solver 的 FP64 dense tensor core 计算替换为 STC 设计。Artifact Appendix 说明复现环境为 Docker 容器，包含 Python、Bash、C++ simulators，Ubuntu 22.04、GCC/OpenMP/OpenCV，fast verification 约 5 小时，complete verification 约 75 小时并需要约 500GB/600GB 级存储。
    - 关键实验设置与指标：论文评估 performance、energy、area 和 energy efficiency density。总体上，Uni-STC 在 kernel level 相对 DS-STC 和 RM-STC 的几何平均 speedup 为 3.35x 和 2.21x，energy reduction 为 1.97x 和 1.27x，energy efficiency gain 为 7.05x 和 2.96x；应用级在 DNN 上有 1.43x speedup，在 AMG solver 上有 1.92x speedup。随机 8192x8192 矩阵实验中，Uni-STC 对 GAMMA、SIGMA、Trapezoid 的平均 speedup 分别为 1.67x、1.73x、1.13x。SuiteSparse 全量结果中，相对 RM-STC，Uni-STC 在 SpMV、SpMSpV、SpMM、SpGEMM 上的平均性能提升分别约为 1.47x、3.39x、2.52x、1.45x（64 MAC@FP64 配置）；相对 DS-STC，提升分别为 3.76x、4.18x、3.07x、2.40x。面积分析显示 critical path 位于 Execution & Write C stage，并满足 1.5 GHz 目标频率。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：论文引入的是 Uni-STC 的 UWMMA sparse execution pipeline，而不是单个传统 GPU CUDA kernel。最接近“新 pipeline”的路径是 BBC + UWMMA + TMS/DPG/SDPU 三阶段内部流水线。Stage 1 是 Task Generation：TMS 从 Meta Buffer 读取 top-level bitmap，把 16x16x16 T1 任务拆成 4x4x4 T3 task 并写入 Tile queue。Stage 2 是 Task Concatenation：8 个 DPG 并行读取 bottom-level bitmap，把 T3 task 分解为 1x1x4 T4 task code、operand vector 和 network control signal，并写入 Dot-product queue。Stage 3 是 Execution & Write C：SDPU 弹出合并后的 T4 task，执行 segmented dot-product，累加到 1KB accumulator buffer，并更新寄存器或结果 tile。这个 pipeline 的关键点是 queue 中主要存控制信息而不是 A/B 数值，从而降低宽 datapath 面积，并允许 task generation 与 SM 后续操作异步重叠。
    - 新 pipeline/kernel 的执行流例子：以 SpGEMM C=A×B 的一个 warp 处理某一行 row 为例，软件首先根据 BBC 的 top-level CSR 找到 Arow_ptr[row] 到 Arow_ptr[row+1] 的非零 tile。warp 读取 A 的 16x16 tile value 和 tile 内 bitmap/index，把 A value 装入 Uni-STC 的 Matrix A Buffer；随后遍历 B 中与 Acol 对应的非零 tile，若 A16b×B16b 与 bloom/filter 类索引判断说明 C 中存在目标 tile，就读取 B 和 C 的 bitmap/index，发出 stc.load.meta mm。接着 stc.task gen.mm 触发 TMS：TMS 依据 4 个 K-layer 的 tile bitmap 产生 T3 task，并用 task ordering 尽量复用同一 A/B tile、减少写冲突。每个 DPG 再把一个 T3 task 的底层 bitmap 做 overlay，形成表示 sparse dot-product pattern 的 4-bit code，并组合 C tile 的结构信息生成 8-bit T4 task code；例如某个 T4 code 的高 4 bit 表示 C tile 中第几个非零元素作为累加目标，低 4 bit 表示 K 维哪些位置参与点积。最后 stc.numeric.mm 进入 SDPU：多个 T4 task 被拼接成 segmented dot-product，SDPU 对 A 和 B 的短向量做乘加，并在写 C 前尽量预合并相同目标的 partial sums。若 task queue 尚未准备好，numeric 指令因 BUSY stall；准备好后进入 READY 并执行，完成后回到 IDLE，结果通过 register file 写回并由软件更新 C 的 BBC value 区。对于 SpMV/SpMSpV，流程类似，只是 B 侧变成 dense/sparse vector x，最终每个线程的 ry 经 shfl_gather 聚合后写回输出向量。
