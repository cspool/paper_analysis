1、论文工作：
    - 论文要解决的核心问题：生产级 LLM serving 逐渐从单一应用走向共享 foundation model 支撑多类应用，例如聊天、代码补全、文档摘要等。这些应用有不同 SLO：聊天类通常要求较低 TTFT 以保证首 token 响应，但 TBT 可以接近人类阅读速度；代码补全则同时要求快速 prefill 和低于 50 ms 的 TBT。现有 serving 系统大多以 iteration 为调度粒度，也就是一次完整前向传播必须穿过所有 transformer layers 后才可切换或重排请求。这种粒度在 multi-SLO 工作负载下过粗，导致 prefill 阶段难以及时抢占长 prompt，decode 阶段又必须用统一 TBT 约束服务不同请求，最终牺牲 goodput 或造成 SLO violation。
    - 论文的主要贡献：第一，论文提出 layer-level scheduling，把一次 iteration 内的全层计算拆成逐层可调度操作，使系统能在 layer 边界切换、合并、分批执行请求。第二，针对 prefill 提出 layer-level chunked prefill，使长 prefill chunk 可以在层边界被 latency-critical 请求抢占，也可以在同一 iteration 内动态合并 relaxed 请求形成更大的 chunk。第三，针对 decode 提出 layer-level decode batching，用每个请求每轮执行的层数 L 和调度 offset O 控制不同 SLO 请求的执行频率，让 relaxed 请求用更少层/轮的方式释放 batch 容量。第四，提出 Laser 系统，将实例内 layer-level 调度、模块化 latency model、实例间 request dispatching、SLO-homogeneous decode group 管理结合起来。第五，在 4 台各 4 张 A100 80GB 的集群上，使用 Qwen2.5-14B、Qwen2.5-32B 和 LLaMA-3-70B 评估，报告相对 Sarathi-Serve 和 DistServe 最高 68.9% / 1.67x goodput 提升，在 99% attainment 目标下最高 1.85x。
    - 论文所处背景：LLM serving 通常分为 compute-bound 的 prefill 和 memory-bound 的 decode。Orca 的 continuous batching 将批处理粒度从 request 降为 iteration；Sarathi-Serve 在此基础上用 chunked prefill 避免长 prompt 阻塞 decode；DistServe 等 disaggregated serving 系统把 prefill 和 decode 放到专用实例上分别优化。这些方法已经是高效 LLM serving 的主流基础，但它们仍然把一次完整模型 forward 视为不可再细分的调度单元。Laser 的出发点是：当多个应用共享模型且 SLO 差异很大时，iteration-level 的统一执行无法利用 relaxed 请求的 slack，也无法足够快地响应 critical 请求。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：Sarathi-Serve 采用 prefill-decode aggregation 架构和 chunked prefill，能够减少长 prefill 对 decode 的阻塞，但它为了满足严格 50 ms TBT 会限制每个实例的 prefill chunk size，导致 prefill SLO violation 明显增加。DistServe 将 prefill 和 decode 分离到专用实例，相比 Sarathi-Serve 已能提升 goodput，但每个实例内部仍使用 iteration-level scheduling，面对 multi-SLO 请求时仍无法在 iteration 内抢占、合并或差异化执行。论文中指出，prefill chunk size 从 100 增到 3200 时，per-iteration latency 增加 16x，但 per-token latency 下降 45.4%，说明大 chunk 更高效却会阻塞紧急请求；decode 中 relaxed 请求可容忍最多约 7x 更多并发，但 iteration-level scheduler 为满足最严格 TBT 只能采用小 batch，导致 relaxed 请求容量被浪费。
    - Baseline 的瓶颈来源：prefill 瓶颈主要是调度粒度和 compute 资源占用之间的矛盾。大 chunk 有更好 token-level 效率，但完整 iteration 期间不可切换，会造成 head-of-line blocking；小 chunk 响应性更好，却降低 GPU utilization 并增加 relaxed 请求排队。decode 瓶颈主要是 memory-bound KV cache 访问和统一 batch 策略之间的矛盾。不同 SLO 请求的可接受 TBT 差异很大，但 iteration-level batching 要求同一批请求每轮都跑完整模型层数，因此所有请求被迫服从最严格 SLO。
    - 论文的设计方法：Laser 的核心机制是 layer-level scheduling。对 prefill，Scheduler 在 layer 边界保存中间状态，评估新请求的 slack，即 TTFT target 与预计 prefill latency 的差值；如果当前 chunk 剩余 iteration time 超过新请求 slack，就在层边界恢复/保存状态，并决定把新请求合并到当前 layer 后继续执行，或直接优先执行新请求。对 decode，Planner 为每个请求生成 execution plan，包含 L，即每个 iteration 执行多少层，以及 O，即下一次执行前延迟多少 iteration；系统据此在 layer 粒度交错不同请求，使 strict 请求更频繁跑完整层，relaxed 请求以较低频率推进，从而扩大实例可容纳 batch。
    - 方法如何对冲 Baseline 缺陷：对 prefill，Laser 不再等待一个完整 forward pass 结束才调度，而是在 transformer layer 边界进行切换；长 LongBench 类请求可以被 HumanEval 类紧急请求打断，后者及时完成 TTFT，前者之后从 intermediate cache 恢复。对 relaxed prefill 请求，Laser 可在同一 iteration 内补齐其它请求的部分 layer 后合并执行，避免小 chunk 造成 GPU 利用不足。对 decode，Laser 不再用统一 TBT 决定所有请求 batch size，而是把请求的 SLO slack 转化成不同 L/O 计划，允许 relaxed 请求被更多地 colocate 和 interleave。实例间调度上，prefill 选择最大 slack 的实例，decode 则通过 SLO group 和去中心化 performance evaluation，把新请求优先分到相近 TBT 组，同时允许跨组以避免负载不均。
    - 关键 trade-off：Laser 接受更复杂的系统状态管理和调度计算，以换取 multi-SLO 场景下更细粒度的执行控制。layer-level preemption 需要在 GPU 上维护 intermediate cache，并在 layer 边界缓存/恢复 hidden states；decode execution plan 需要 latency model 支持，否则 L/O 搜索空间会组合爆炸。论文通过事件触发更新、贪心调整 relaxed 请求、offset 负载均衡和并行 overlap 来降低开销，但这意味着系统实现更依赖 vLLM/Ray runtime、GPU memory 管理、CUDA fused state caching，以及对模型 layer latency 的离线 profiling。另一个边界是 Laser 主要解决 serving 调度，不改变模型结构、attention kernel 数学或 KV cache 压缩策略；如果 workload 的 SLO 差异不明显，或实例数量太少，decode group 的收益会下降。

3、论文实现：
    - Baseline 如何实现：实验对比 Sarathi-Serve 和 DistServe。Sarathi-Serve 作为 prefill-decode aggregation baseline，采用 chunked prefill 以减轻长 prefill 对 decode 的阻塞；DistServe 作为 disaggregated baseline，将 prefill 和 decode 放到专用实例，但各实例仍采用 iteration-level scheduling。为公平比较，两个 baseline 都采用 EDF，而不是 FCFS。论文在 end-to-end 结果中指出，Sarathi-Serve 偏向优先 decode，为满足 50 ms TBT 限制 prefill chunk size，造成 prefill SLO violation；DistServe 通过 phase disaggregation 改善阶段隔离，但仍难处理 heterogeneous SLO。
    - 新设计如何实现：Laser 原型集成 vLLM 作为默认 inference backend，Ray 作为 communication backend。系统由 Global Controller、prefill instances 和 decode instances 组成。Global Controller 在初始化时进行 offline profiling，建立 latency model，并在运行时选择 prefill instance、执行 decode assignment 和管理 SLO group。prefill instance 包含 Scheduler 和 Executor：Scheduler 评估 slack、EDF 队列、抢占与合并策略；Executor 支持 layer-level preemption/restoration。decode instance 包含 Planner 和 Executor：Planner 用 latency model 构建 execution plan，输出每个请求的 L 和 O；Executor 按计划在 layer 边界切换请求。
    - 模块化 latency model：Laser 把 transformer layer 拆为 stateless modules 和 stateful self-attention module 分别建模。QKV projection、attention output projection、FFN 等 stateless modules 对 token 独立，受 GPU tile quantization 影响，论文用分段线性函数刻画 token count 与 latency 的关系，segment width 取 32 token。self-attention 是 stateful module，其延迟主要由 token count 和总 context length 决定；论文报告这些参数与 module latency 的 Pearson 系数绝对值超过 0.78，因此用线性函数建模。最终每层 latency 是 stateless latency 与 attention latency 之和，离线 profiling 对不同 token count 和 context length 测量后拟合参数。
    - layer-level chunked prefill 实现细节：每个新请求到达时，prefill Scheduler 计算 TTFT slack。如果当前执行 chunk 的剩余 iteration time 会让新请求错过 SLO，Scheduler 在当前 layer 结束后保存正在执行请求的中间状态，然后判断是否可把新请求推进到同一 layer 并与旧请求合并；若合并会违反新请求 SLO，则直接优先执行新请求。若无需抢占且队列为空，Scheduler 会尝试把新请求合入当前 chunk；否则放入队列。队列按 arrival time + TTFT target 排序，体现 EDF 原则。完成一个 prefill computation 后，Scheduler 根据最紧 deadline 对 pending requests 动态 chunk，以兼顾吞吐与 SLO。
    - layer-level decode batching 实现细节：decode Planner 用 execution plan 控制每个请求的执行频率。若模型有 N 层，请求 r 的 L_r 表示每轮执行的层数，O_r 表示调度 offset；例如 L=N 表示每个 iteration 都完整执行，L=N/2 表示请求分多轮推进。Planner 对未来若干 iteration 做模拟，按每层 active request 数和 context length 估计 latency，并取最大 predicted latency 作为当前 plan 的 per-iteration latency。构建计划时，它先把新请求设为每轮执行所有 N 层，再以最严格 TBT 为目标；若估计 latency 超标，就选择 SLO 最 relaxed 的请求，把其 L 降到仍能满足自身 SLO 的最小值，并选择 offset 平衡同 L group 的 layer latency；若 latency 低于目标，则优先尝试把最 strict 请求恢复到全层执行。更新只在 request arrival/departure 或 latency 接近最严格 SLO 等 critical events 触发。
    - inter-instance dispatching 实现细节：prefill 侧，Global Controller 持续监控各实例的调度决策和请求统计，找出在 EDF 和 layer-level chunked prefill 下可接纳新请求且不会违反 TTFT 的实例，并选最小 slack 最大的实例；如果都不安全，则选 prefill token 最少的实例做 best-effort。decode 侧，Laser 维护虚拟 SLO-homogeneous instance groups，根据各 SLO group 的 arrival rate 与最大 SLO-compliant batch size 下的 per-instance throughput 估计所需实例数，并动态调整 group 大小。新 decode 请求到达时，候选实例先本地运行 ExecPlan 估计接纳后的 latency 与 memory 状态；若会违反 SLO 或内存耗尽则拒绝，否则返回 aggregated TBT increment。Global Scheduler 按请求目标与 group SLO 的差距排序 group，选择 TBT increment 最小且 SLO-compliant 的实例。
    - intermediate state 与 cache migration：为支持 layer 边界切换，每个实例在 GPU memory 中维护 intermediate cache 和 state manager，作用类似 KV cache 但存的是 layer-level intermediate states。论文设置 prefill instance 的 intermediate cache 为 16,384 tokens，decode instance 为 2,048 tokens；在 Llama-70B 上消耗少于 256 MB GPU memory。缓存满时，Laser evict 最 relaxed SLO 的请求，并在有空间后异步恢复。论文还实现 fused CUDA kernel，把 state caching 和 retrieval 合并为一次操作。prefill 到 decode 的 KV cache migration 也按 layer 粒度异步进行，并与 prefill computation overlap；若动态 context length 导致内存耗尽或 SLO violation，Laser 通过 live cache migration 重新分配 decode 请求，保证 transfer 期间继续生成 token。
    - 实验 / 实现平台：实验集群为 4 台物理主机，每台 4 张 NVIDIA A100 80GB GPU，主机间 100 Gbps LAN，机内 GPU 通过 NVLink 互连。模型包括 Qwen2.5-14B、Qwen2.5-32B、LLaMA-3-70B，分别配置 1-way、2-way、4-way tensor parallelism。工作负载来自三个应用：ShareGPT 用于 chatbot，HumanEval 用于 code completion，LongBench 用于 document summarization；请求到达服从 Poisson distribution。指标包括 SLO attainment 和 goodput，goodput 定义为保持 90% SLO attainment 时达到的 throughput；decode 请求若 P99 TBT 超过 SLO target 则视为 violation。Table 1 中平均长度和 baseline SLO 为：ShareGPT input 350.9 / output 187.9 / TTFT 0.25 s / TBT 100 ms；HumanEval input 201.5 / output 105.2 / TTFT 0.125 s / TBT 50 ms；LongBench input 2084.5 / output 95.7 / TTFT 2 s / TBT 200 ms。
    - 关键实验设置与指标：end-to-end 评估显示，Laser 相比 Sarathi-Serve 和 DistServe 显著降低 SLO violation，并最高提升 goodput 68.9% / 1.67x；当 goodput attainment target 提升到 99% 时，吞吐提升最高 1.85x。相对所有 baseline，Laser 在 Qwen-14B、Qwen-32B、Llama-70B 上分别提升 goodput 43.4%、68.9%、56.6%。请求分布实验中，当 relaxed 请求占比从 2:4:1 转到 2:1:4 时，Laser 改进从 19.4% 增至超过 86%。SLO scale 实验中，在 tight 0.8x SLO 下，Laser 相对 DistServe 和 Sarathi-Serve 的 goodput gain 最高为 1.08x 和 6.25x；即使在 relaxed 1.3x SLO 下仍有超过 34% goodput 提升。集群规模实验中，相对 DistServe 的提升从 16 GPU 的 43.4% 降到较小规模下的 31.3%，但当集群小于 8 GPU 时，DistServe 更易受长请求 head-of-line blocking 影响，Laser 仍取得 1.23x goodput gain。
    - 消融实验与开销：prefill 消融将 layer-level chunked prefill 与 iteration-level baseline 对比，负载越高收益越明显；论文表述为 violation rate 改善幅度从 25 Req/s 下的 4.4% 增至 50 Req/s 下的 21.6%，并且 flexible consolidation 让平均 TTFT 降低超过 10%。decode 消融中，layer-level batching 相比 iteration-level batching 将 TBT SLO violation rate 降低超过 6.7%，再加入 group-based decode assignment 后额外降低 10.5%；Laser 还将 TPOT SLO violation rate 最高降低 11.8%。Latency prediction 方面，stateful/stateless module 预测准确率为 94.8% 到 98.6%，聚合到 layer/iteration 后仍超过 94.6%，profiling 在 2 秒内完成。开销方面，request switching 在 prefill 和 decode 中都低于 1.5%，scheduling overhead 最高 3.8%，并通过关键事件触发和与 model execution overlap 掩盖；Global Controller 选择 prefill instance 开销小于 2 ms，decode dispatch 约 10 ms，且基本不随实例数量增长。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：论文没有提出新的模型 kernel 或 attention 算子，而是提出一条新的 layer-level serving pipeline。它把 traditional iteration-level “一次请求/批次跑完整 N 层后再调度” 改为 “每个 transformer layer 都可以成为调度边界”。在 prefill 中，这条 pipeline 体现为 layer-level chunked prefill：请求可在 layer 边界被抢占、恢复、合并。 在 decode 中，这条 pipeline 体现为 layer-level decode batching：请求不必每个 iteration 都执行所有 N 层，而是通过 L/O execution plan 在多个 iteration 间交错推进。系统层面，Laser 再用 Global Controller 做 phase-specific dispatching，将 prefill slack-aware selection 和 decode SLO-group assignment 与实例内 pipeline 结合。
    - prefill pipeline 执行流例子：假设 Req1 是一个长 LongBench 文档摘要请求，已经在 prefill instance 上以较大 chunk 执行到第 k 层；此时 Req2 是 HumanEval 代码补全请求，TTFT SLO 更严格且刚到达。Scheduler 计算 Req2 的 slack，并估计 Req1 当前 chunk 剩余 layer 的执行时间。如果继续跑完 Req1 的完整 iteration 会让 Req2 错过 TTFT，Executor 在当前 layer 结束后把 Req1 的 intermediate state 写入 GPU intermediate cache。随后 Scheduler 判断是否可以先把 Req2 推进到第 k 层再与 Req1 合并；如果合并仍能满足 Req2 SLO，则两者从第 k 层后共同执行，提高 batch size 和 GPU utilization；如果不安全，则 Req2 被单独优先执行。Req2 完成关键 prefill 后，Req1 从 intermediate cache 恢复并继续执行。
    - relaxed prefill 合并例子：假设 Req1 是 relaxed SLO 的 ShareGPT 请求，并以较小 chunk 启动；Req2 随后到达，但队列为空且合并不会造成 TTFT violation。iteration-level chunked prefill 只能等 Req1 完整 iteration 结束后才加入 Req2，导致 Req2 排队、GPU batch 偏小。Laser 则可以在某个 layer 边界先执行 Req2 的前几层，使其达到 Req1 当前 layer 位置，然后把 Req1 和 Req2 后续层合并执行。这样做把多个小 chunk 动态拼成更大的有效 chunk，在不破坏 SLO 的前提下降低 average token processing latency。
    - decode pipeline 执行流例子：假设模型有 N 层，decode instance 同时服务一个 50 ms TBT 的 HumanEval 请求和多个 100/200 ms TBT 的 ShareGPT/LongBench 请求。Planner 初始假设所有请求每轮执行 N 层，然后用 latency model 估计未来若干 iteration 的最大 per-iteration latency。如果该 latency 超过最严格 TBT，Planner 会选择 SLO 最 relaxed 的请求，把其 L 调低，例如从 N 降到 N/2 或 N/4，并选择 offset O 让不同 relaxed 请求错开 layer 执行。执行时，strict 请求每轮都走完全部层，relaxed 请求在不同 iteration 只走一部分层，其 hidden state 被 intermediate cache 保存，下一轮按 offset 恢复并继续。这样同一 decode instance 可以同时容纳更多 relaxed 请求，同时不让 strict 请求的 P99 TBT 超标。
    - group-based decode assignment 路径例子：当一个新的 LongBench decode 请求进入系统时，Global Scheduler 先根据其 200 ms TBT target 查找最接近的 SLO group。每个候选 decode instance 本地取当前 N、请求集合 R、L/O plan 和 latency 统计，把新请求临时加入后运行 ExecPlan，估计新的 per-iteration latency、TBT increment 和内存占用。如果实例会触发 SLO violation 或 memory exhaustion，它返回拒绝；否则返回接纳后的 aggregated TBT 增量。Global Scheduler 按 SLO 差异从小到大检查 group，并在可行实例中选择 TBT increment 最小者。若相近 group 都不合适，则允许跨 group 分配以平衡负载；若全部不可行，则把请求放到 least-loaded instance 做 best-effort。
    - 该 pipeline 的关键机制与边界：Laser 的关键不在于减少单个 layer 的算子延迟，而在于把 layer 作为调度原子，显式管理中间状态和未来 iteration latency。它适用于 shared model、多应用、多 SLO、prefill/decode disaggregation 已存在的 serving 环境，尤其对 long prompt 与 strict TBT 混合、relaxed 请求比例高的 workload 有利。其限制是系统复杂度较高，需要准确 latency model、GPU intermediate cache、layer-level state switch、request group 管理和跨实例控制器；同时论文未公开 artifact，无法通过源码验证其 vLLM/Ray 集成、fused CUDA kernel 和 live migration 的边界行为。对于 SLO 同质化、请求长度较短或调度开销难以 overlap 的场景，Laser 的相对收益可能低于论文中的 multi-SLO 实验场景。
