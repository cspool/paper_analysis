1、论文工作：
    - 论文要解决的核心问题：RoMeo 解决的是 4-bit weight-activation LLM serving 中，现有混合精度量化只处理 channel-wise outliers，无法处理 token-wise residual outliers，导致低比特量化准确率下降的问题。论文指出，8-bit 下保留少量异常 channel 往往足够，但当精度降到 INT4 后，即使剪除 top channel outliers，激活中仍存在按 token 集中的大值；这些值会扩大量化 scale，使大多数普通值在 4-bit 表示范围内分辨率不足，从而带来明显 perplexity 和 downstream accuracy 损失。
    - 论文的主要贡献：第一，论文通过 Qwen3-8B 等模型的激活分布观测，把 outlier 问题从单一 channel 维度扩展为 channel + token 双维度问题。第二，提出 RTMPQ，即 Rotated Token-wise Mixed Precision Quantization：先用 Hadamard rotation 抑制 channel-wise outliers 并把不规则性迁移到 token 维度，再对 outlier tokens 使用 INT8、普通 tokens 使用 INT4。第三，设计 RoMeo serving system，用 permutation-free mixed precision computation、cross-precision CUDA kernels、software pipelining、Triton fused quantization kernels 和多 CUDA stream asynchronous execution，使 token-wise mixed precision 能够在 GPU 上接近 uniform INT4 的效率运行。第四，在 Qwen3、Llama-3.1 多个模型上评估准确率和 serving 性能，展示比 QuaRot、MixQ 等 baseline 更好的准确率与最高 2.10x 的端到端加速。
    - 论文所处背景：LLM serving 同时受 GPU 显存容量、memory bandwidth、Tensor Core compute throughput 约束。W4A4 量化理论上能显著减少内存占用并利用 INT4 Tensor Cores，但 activation outliers 会破坏量化精度。已有路线包括 uniform precision quantization，例如 QuaRot 用旋转降低 outlier 影响；以及 mixed precision quantization，例如 MixQ、LLM.int8()、COMET、Atom 等把敏感区域保留在更高精度。问题是多数 mixed precision 方法沿 channel / reduction dimension 设计，适合离线标定和 GEMM 分解；RoMeo 的目标场景是高吞吐 LLM 推理，尤其是希望在现有 NVIDIA GPU 上把 W4A4 的效率和 mixed precision 的准确率同时保留下来。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：QuaRot 这类 rotation-based uniform INT4 量化能缓解 channel outliers，但它仍把所有 token 统一放进 4-bit 表示，无法单独保护由输入语义触发的 token-wise outliers。MixQ 这类 channel-wise mixed precision 方法保留 outlier channels 为高精度，但 residual outliers 不再集中在固定 channel，而是出现在特定 token 行上；这些 token-wise outliers 属于 GEMM 的 non-reduction dimension，不能像 channel-wise outliers 那样沿 reduction dimension 拆成几个 dense GEMM 分别执行。Atom 的 group-wise mixed precision 粒度更细，准确率在部分模型上有优势，但论文认为其额外计算和系统开销更高，不适合直接作为高效 serving 路径。朴素 INT4 round-to-nearest 更糟，在 WikiText2 上 perplexity 可达到极高值，说明低比特 activation quantization 对 outlier 极其敏感。
    - 论文的设计方法：RTMPQ 采用两阶段算法。第一阶段对 activation 做在线 Hadamard rotation，同时把 weight 离线乘以对应转置 Hadamard matrix，以保持线性层数学等价；Hadamard matrix 的 +/-1 正交结构把 channel 上的极端值平滑到多个 channel，从而降低 peak magnitude。第二阶段根据每个 token 的最大激活值做 top-k outlier detection，把 top outlier tokens 放入集合 O_A；普通 token 用 INT4，outlier token 用 INT8。weight 在经过离线旋转后也可能出现类似 token/column 方向的不均匀性，因此同样做混合精度量化。最终 GEMM 需要处理 W4A4、W4A8、W8A4、W8A8 四类精度组合，并用 INT32 accumulator 防止累加溢出，再按 per-token / per-channel scale 反量化回 FP16/BF16 输出。
    - 方法如何对冲 Baseline 缺陷：对 QuaRot，RoMeo 没有停留在 rotation 后的 uniform INT4，而是进一步把 rotation 后仍然显著的 token-wise outliers 以 INT8 保存，因此量化 scale 不再被少量异常 token 主导。对 MixQ，RoMeo 通过 Hadamard rotation 把 channel outlier 问题变成更单纯的 token-wise outlier 问题，避免同时做复杂的双维 sparse mixed precision；再用系统设计把 non-reduction dimension mixed precision 变成可密集执行的多组 uniform-precision dense kernels。对朴素 token permutation 方案，RoMeo 不对整块 activation / weight 做昂贵重排，而是预分配 outlier buffer，把 outlier tokens 额外拷贝并量化到 INT8 buffer，同时完整矩阵仍量化为 INT4；最后用高精度 outlier 结果覆盖对应输出位置，以少量冗余计算换取连续内存布局和 Tensor Core 友好的执行。
    - 关键 trade-off：第一，RoMeo 接受 outlier tokens 同时参与 INT4 和 INT8 计算的冗余，以避免 permutation index 计算、in-place swap 和非连续访问开销。第二，RTMPQ 的准确率依赖 outlier percentage；论文默认 activation 和 weight 的 token-wise outlier 比例为 5%，比例越高准确率越好，但 INT8 计算占比和 buffer 开销也越高。第三，RoMeo 选择 separate kernels 而不是单个 fused mixed-precision kernel，使每类精度组合可以按自身 shared memory / register / occupancy 需求调优，但需要用 asynchronous execution 和 CUDA events 隐藏额外 kernel launch 与小矩阵并行度不足。第四，Hadamard rotation 带来额外在线 FWT 成本，不过论文认为其 O(m n log n) 开销相对 dominant linear GEMM 较小。

3、论文实现：
    - Baseline 如何实现：准确率 baseline 包括 BF16 原模型、朴素 INT4、QuaRot、MixQ 和 Atom。QuaRot 使用 Hadamard transformation 做 outlier suppression，并以 per-token / per-channel 粒度做 uniform 4-bit weight-activation quantization。MixQ 沿 channel 选择 outlier channels，并以较高精度保护这些 channel；论文为公平比较把 MixQ 的在线 activation outlier 比例设置为 10%，以匹配 RoMeo 的稀疏/额外计算约束。Atom 使用 channel-level group-wise mixed precision，group size 为 128，准确率可能更高但计算开销更大。性能 baseline 包括 PyTorch BF16 GEMM、CUTLASS INT8 kernel、Atom mixed precision kernel、QuaRot INT4 multiplication + fused dequantization kernel，以及 RoMeo 内部的 unified single kernel / separate kernels / pipelined / asynchronous ablation 配置。
    - 新设计如何实现：RoMeo 被实现为 PyTorch extension，量化后的线性层封装为 PyTorch nn.Module，可以替换原始 linear layers。Hadamard transform 使用 HadaCore。在线 outlier identification、quantization 和 INT4 packing 由 fused Triton kernels 实现；核心 mixed precision GEMM 使用 CUTLASS 开发 CUDA kernels，并通过 compiled dynamic libraries 暴露给 Python。RoMeo 使用 JIT compilation，在首次运行时按具体模型维度编译 kernel，并缓存二进制供后续调用；同时用 runtime profiling 做 auto-tuning，调节 tiling sizes、pipeline stages 等关键参数。
    - 实验 / 实现平台：准确率评估覆盖 Qwen3-8B、Qwen3-14B、Qwen3-32B、Llama-3.1-8B、Llama-3.1-70B。任务包括 WikiText2 perplexity，以及 ARC-Challenge、ARC-Easy、HellaSwag、LAMBADA、PIQA、WinoGrande 六个 zero-shot downstream tasks；lm-eval batch size 为 32，perplexity 评估 batch size 为 2、sequence length 为 2048。效率评估在 NVIDIA GeForce RTX 4090 24GB GPU 上进行，使用 Python 3.12、PyTorch 2.8.0、CUDA 12.8。端到端 layer-level 测试集成到 Transformers，serving prefill throughput 测试集成到 SGLang v0.5.5；Qwen3-14B 使用 TP=2，Qwen3-32B 使用 TP=4。
    - 关键实验设置与指标：准确率方面，RoMeo 默认 token-wise outlier percentage 为 5%。在 WikiText2 perplexity 上，RoMeo 在所有列出的 Qwen3 与 Llama-3.1 模型上优于 QuaRot，并显著优于朴素 INT4 和多数 MixQ 配置；例如 Qwen3-8B perplexity 从 BF16 9.72 到 RoMeo 10.97，而 QuaRot 为 11.53、MixQ 为 14.76。zero-shot 平均准确率上，RoMeo 在 Qwen3 系列取得最高平均值；在 Llama-3.1 上 Atom 因更细粒度量化略强，但 RoMeo 以更低计算开销保持接近准确率。性能方面，RoMeo 在 Qwen3 layer-level latency 上最高达到 2.10x over BF16，并与 QuaRot 接近；SGLang prefill throughput 中，Qwen3-8B batch 64 达到 20073.60 tokens/s，对 BF16 的 10545.13 tokens/s 为 1.90x。kernel microbenchmark 中，RoMeo mixed precision kernel 几何平均相对 BF16 为 4.68x，相对 INT8 为 3.39x，并接近 QuaRot 的 4.55x。消融显示，Hadamard、outlier identification / quantization、post-multiplication overwrite 约占 BF16 baseline latency 的 12%，而 software pipelining 与 asynchronous separate kernels 是抵消这些开销的关键。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：论文的新执行路径可以概括为 RoMeo RTMPQ serving pipeline：offline weight rotation and mixed precision quantization -> online activation Hadamard rotation -> online row-max + top-k token outlier detection -> normal activation INT4 quantization and outlier activation INT8 quantization -> four dense cross-precision GEMM kernels -> outlier-result overwrite -> scaling/dequantization and output writeback。它的核心 kernel 不是单个普通 GEMM，而是一组面向 token-wise mixed precision 的 non-reduction dimension cross-precision GEMM kernels，配合 fused Triton quantization kernels 和 CUDA-stream task graph。RoMeo 的关键创新在于把原本稀疏、动态、跨精度的 token-wise outlier 计算重构为多个 dense uniform-buffer kernels，从而仍能使用 Tensor Core 友好的连续 tile 访问。
    - 新 pipeline/kernel 的执行流例子：假设一个 batch 的 activation 矩阵 A 进入某个 rotated linear layer，shape 为 M x K，weight 矩阵 W 为 K x N。离线阶段，RoMeo 已经把 W 乘以转置 Hadamard matrix，并根据 weight 的 outlier column / token-like 维度准备好 INT4 主权重 buffer 和 INT8 outlier weight buffer。在线阶段，activation A 先经过 Fast Walsh-Hadamard Transform，channel 维度的大 outlier 被平滑；随后 fused Triton kernel 对每一行 token 计算 max(abs(A_i))，做 top-k 选出 5% outlier tokens。整块 A 被量化并 pack 成 INT4 buffer；被选中的 outlier token rows 额外拷贝到预分配 outlier buffer，并按 INT8 量化，scale 也一并保存。
    - 接下来 RoMeo 构造四类乘法任务：普通 activation 与普通 weight 走 W4A4 kernel，outlier activation 与普通 weight 走 W4A8 或 A8W4 路径，普通 activation 与 outlier weight 走另一类 cross-precision kernel，outlier activation 与 outlier weight 走 W8A8 kernel。每个 kernel 内部按 M/N/K tile 切分，先用 `cp.async` 把若干 K tiles 从 global memory 预取到 shared memory 填满 pipeline；steady state 中等待最旧异步拷贝完成，用 Tensor Core `mma` 执行当前 tile，再发起下一 tile 的异步拷贝；涉及 INT4/INT8 混合时，在 shared memory 中用低成本 binary arithmetic 把 INT4 转成 INT8 参与 `mma`，避免昂贵类型转换。计算完成后在寄存器中乘以对应 activation / weight scale，写回 output buffer。
    - RoMeo 的并发调度把上述任务拆成细粒度 DAG。row-max + top-k 之后，普通 token 的 INT4 quantization 可以先启动 W4A4 GEMM；outlier token 的 INT8 quantization 完成后，再触发 W8A4 / W4A8 / W8A8 等依赖它的 kernels。四类 GEMM 之间没有严格串行依赖，RoMeo 用多个 CUDA streams 并发执行，用 CUDA events 只约束真实依赖。最后，高精度 outlier multiplication 的输出覆盖 W4A4 主结果中对应 outlier token / weight 位置。这样，一个动态输入 token 触发的 outlier 不需要重排整块矩阵，只是在 outlier buffer 中多走一条高精度旁路，最终以 overwrite 方式修正输出。
    - pipeline 的边界和限制：RoMeo 主要优化 linear module 中的 weight-activation quantized GEMM，不解决 attention KV cache 管理、跨节点通信、调度策略或模型结构搜索。小 batch 下，CPU/framework overhead 和 kernel launch overhead 可能压过量化 GEMM 收益，论文在 Qwen3-8B batch 8 的 SGLang prefill 中也观察到 RoMeo 低于 BF16；作者指出可通过 prefill 阶段启用 CUDA Graph 缓解。分布式 tensor parallel 场景下，通信开销会部分抵消 quantization compute gain，因此 Qwen3-14B / 32B 的 serving speedup 低于单卡 Qwen3-8B。最后，RoMeo 的 accuracy-efficiency 平衡受 outlier percentage、模型激活分布、Hadamard transform 位置和 kernel auto-tuning 影响；如果未来模型的 token-wise outlier 分布较弱或更复杂，默认 5% outlier 设置未必总是最优。
