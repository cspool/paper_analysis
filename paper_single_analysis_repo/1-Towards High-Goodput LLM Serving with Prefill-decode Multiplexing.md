1、论文工作：
    - 论文要解决的核心问题：LLM 在线服务需要同时满足 prefill 阶段的 TTFT 和 decode 阶段的 TBT SLO，但两阶段资源特征不同：prefill 更偏计算密集，decode 更偏访存和 KV cache 访问；同时真实服务中请求到达、输入长度、输出长度、多轮复用上下文长度都高度动态。已有系统要么将 prefill / decode 分离，带来 GPU 空闲、KV cache pool 缩小、KV 迁移或重算；要么采用 chunked-prefill，把 prefill chunk 与 decode iteration 绑定在同一个 token budget 下执行，导致 decode SLO 与 GPU 高利用率之间互相牵制。
    - 论文的主要贡献：论文提出 MuxWise，一种 intra-GPU prefill-decode multiplexing 的 LLM serving 框架。它在同一 GPU、同一进程内将 SM 动态划分给 prefill 和 decode，使两阶段空间并发执行，同时共享同一显存空间和 KV cache pool。系统由 bubble-less multiplex engine、contention-tolerant estimator 和 SLO-aware dispatcher 组成，在优先保证 decode TBT SLO 的前提下，把剩余 SM 分配给 prefill，从而提升 goodput。
    - 论文所处背景：该工作属于 LLM serving 系统、GPU 资源管理和在线推理调度方向，面向 Model-as-a-Service 中的高并发、严格尾延迟 SLO、多轮会话、长上下文、agent / reasoning 工作负载。论文发表于 ASPLOS 2026，基于 SGLang 0.4.10post2 实现，并与 chunked-prefill in SGLang、NanoFlow、LoongServe、SGLang-PD 等系统比较。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：Splitwise / DistServe 类静态 disaggregated serving 将 prefill 和 decode 放在不同实例上，GPU 分配在初始化后固定；当请求负载和阶段比例波动时，某些 GPU 容易空闲。同时每个实例维护独立模型权重和 KV cache pool，在相同 GPU 总量下有效 KV cache 容量下降，论文给出的例子中 cache hit rate 可从 36.6% 降到 4.2%，进而引入重算并损害 goodput。LoongServe 虽然支持按请求长度和阶段动态调整 GPU 资源，但会释放原 GPU 上的 KV cache，难以支持跨请求 KV cache reuse，多轮请求仍可能重算。Chunked-prefill / SARATHI-Serve 类方法通过 token budget 限制 decode 延迟，但 token budget 同时控制 GPU 饱和与 SLO：论文例子中 Llama3-70B 在 8 张 A100 上需要约 4K token budget 才能饱和 GPU，对应 505ms TBT，远高于 100ms TBT SLO；而 SLO 合规预算约 256。长 reused context 还会使 prefill chunk 反复访问历史 KV cache，使 TBT 随复用上下文增长而恶化。
    - 论文的设计方法：MuxWise 将 compute partition 与 memory / KV cache management 解耦。它使用 NVIDIA GreenContext 做 intra-process spatial sharing，把 CUDA streams 绑定到不同 SM 集合，并以微秒级 stream synchronization 代价调整分区；prefill 与 decode 共享同一进程地址空间，因此共享同一 KV cache pool。系统层面，bubble-less multiplex engine 将 prefill 拆成 layer-wise execution，并配合 query-based synchronization；contention-tolerant estimator 通过 offline profiling 建立 solo-run latency predictor，并叠加 contention guard 得到 worst-case decode latency；SLO-aware dispatcher 在线选择 multiplexing plan，为 decode 分配 best-fit SM，其余 SM 给 prefill，同时允许短 prefill 非递归地抢占长 prefill。
    - 方法如何对冲 Baseline 缺陷：相对 disaggregation，MuxWise 只在 SM 计算资源上做动态分区，不拆分显存和 KV cache pool，因此减少固定实例带来的 GPU 空闲和 cache pool 缩小。相对 LoongServe，MuxWise 不需要为了动态 compute allocation 释放原位置 KV cache，因此更适合多轮上下文复用。相对 chunked-prefill，MuxWise 不再把 prefill chunk 和 decode iteration 强行塞入同一个 token budget，而是让二者在空间上并发、调度上独立推进，因此能在 decode 小延迟预算下继续利用剩余 SM 推进 prefill。
    - 关键 trade-off：MuxWise 增加了运行时调度复杂度、offline profiling 成本以及对 GPU 细粒度空间分区机制的依赖。contention guard 采用 worst-case 估计，会为了 SLO 安全牺牲一部分资源分配激进性。layer-wise scheduling 带来更多细粒度 kernel launch 和 CUDA event polling，但论文报告 finest-grained layer-wise prefill 的总运行时开销在 1.5% 内。它更适合严格在线 SLO 场景；若是无严格 TBT / TTFT 约束的离线吞吐任务，MuxWise 相对 chunked-prefill 或 NanoFlow 的优势会下降。

3、论文实现：
    - Baseline 如何实现：论文将 MuxWise 与四类 baseline 比较。Chunked-prefill in SGLang 采用 SARATHI-Serve 风格 token budget，并按 workload 和 TBT SLO offline tuning；SGLang 使用 FlashInfer 融合 prefill / decode attention kernel。NanoFlow 使用与 chunked-prefill 相同的 token budget，代表 operator-level intra-GPU multiplexing。LoongServe 按其模型并行配置运行：Llama-70B 使用 sequence parallelism 2、tensor parallelism 4，Llama-8B 使用 sequence parallelism 4、tensor parallelism 2。SGLang-PD 代表静态 disaggregation，P:D ratio 为 1:1，每个 instance 的 tensor parallelism 为 4，并支持跨 phase / request 的 KV-cache sharing。
    - 新设计如何实现：MuxWise 基于 SGLang 0.4.10post2 扩展 PD multiplexing。SM 空间分区依赖 GreenContext，在同一进程中把 prefill 和 decode 的 CUDA streams 绑定到不同 SM 集合。Decode 侧采用 graph-level scheduling，因为 decode batch size 变化相对有限，适合 CUDA graph；prefill 侧 batch size 与 input length 变化更大，不适合大量静态 graph 捕获，因此被拆成 transformer layer 粒度执行。Estimator 使用 reused length、new input length、output length、decode batch size、partition configuration 等变量做 offline profiling，训练 prefill / decode solo-run predictor；decode 再叠加 contention guard，估计共享带宽干扰下的 worst-case latency。Dispatcher 在每次 prefill batch 完成或 decode iteration 结束后重新选择分区，并根据估计结果决定发射多少 prefill layers。
    - 实验 / 实现平台：主实验平台为 8 张 A100-80GB GPU，NVLink 带宽 600 GB/s；另在 8 张 H100-SMX5-80GB 和 8 张 H200-SMX5-141GB 上测试泛化性。软件环境包括 PyTorch 2.6.0、SGLang 0.4.10post2、GPU driver 570.124.06、CUDA 12.8。模型包括 Llama-8B、Llama-70B 和 Qwen3-235B MoE（22B activated）。workload 包括 Conversation、Tool&Agent、ShareGPT、LooGLE、OpenThoughts，覆盖短输入、长上下文、多轮复用和 reasoning 场景。Artifact 附录提供 CodeLlama-34B-Instruct-hf、ShareGPT、LooGLE 与 H200 NVL 复现实验说明。
    - 关键实验设置与指标：论文以 goodput 为核心指标，用 99%-ile / P99 尾延迟衡量 SLO attainment；decode 侧使用 TBT 而不是 TPOT，因为 TPOT 是平均指标，会掩盖单 token 的差尾延迟。TBT SLO 设置为 Llama3-8B 的 50ms 和 Llama3-70B 的 100ms。总体结果显示，MuxWise 在 SLO 保证下平均 goodput 提升 2.20x，最高 3.06x。真实 workload 中，MuxWise 在 TTFT 上相对 chunked-prefill、NanoFlow、LoongServe、SGLang-PD 分别取得平均 3.57x、5.98x、4.65x、1.66x speedup；在 Tool&Agent 负载升高实验中，Llama-8B goodput 相对四个 baseline 分别提升 2.6x、5.2x、2.0x、1.3x，Llama-70B 相对 chunked-prefill、LoongServe、SGLang-PD 分别提升 3.06x、2.62x、1.62x。H100 / H200 实验中，相比 chunked-prefill 平均 99%-ile TTFT 提升 2.28x，99%-ile TBT 提升 1.81x。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：论文的核心新执行路径可以概括为 MuxWise 的 intra-GPU PD multiplexing pipeline，而不是单个新的数学 kernel。该 pipeline 将 prefill 和 decode 从传统的时间串行、chunk 融合或跨实例分离，改为同一 GPU 内的 SM 空间并发：decode iteration 在一组 SM 上运行，prefill layers 在另一组 SM 上运行，二者共享模型权重地址空间、KV cache pool 和显存管理。具体机制包括三层：第一，GreenContext-based spatial partition 将 CUDA streams 绑定到不同 SM 集合；第二，layer-wise prefill execution 把一个长 prefill phase 拆成多个 transformer layer 粒度的 prefill layer units，缩短可调度单元；第三，SLO-aware dispatcher 使用 worst-case latency estimation 选择 decode SM 数、prefill SM 数和本轮要发射的 prefill layer 数。
    - 新 pipeline/kernel 的执行流例子：假设当前 GPU 上已有一个 decode batch 正在服务 16 个请求，同时请求队列中来了一个新的 prefill batch P0。MuxWise 首先用 estimator 预测在不同 SM 分区下 decode iteration 的 worst-case TBT；如果 60% SM 足以满足 TBT SLO，就把 60% SM 分给 decode，把剩余 40% SM 分给 P0。Dispatcher 优先发射 decode iteration，因为 decode launch latency 很低且 TBT 更严格；随后根据公式估计需要发射的 prefill layer 数，使这些 prefill layers 的执行时间大致覆盖一个 decode iteration 的时间窗口。Decode stream 和 prefill stream 分别在 GreenContext 绑定的 SM 集合上并发执行，但共享同一 KV cache pool。若 P0 的最后一个 prefill layer 完成，query-based synchronization 通过 CUDA event polling 发现完成事件，并把 P0 合并进后续 decode batch，而不长时间阻塞 decode。若此时来了一个短 prefill P2，而当前长 prefill P1 继续执行会让 P2 错过 TTFT SLO，dispatcher 可以让 P2 非递归抢占 P1；P1 的后续 layers 稍后继续执行。
    - 与 baseline pipeline 的差异：在 chunked-prefill 中，prefill chunk 和 decode iteration 被融合在同一个 token budget 下，decode 需要等待 chunk 中对历史 KV cache 的访问，长 reused context 会放大 TBT；MuxWise 则让 prefill layer 和 decode iteration 独立推进，decode 不被 prefill token budget 绑定。在 disaggregated serving 中，prefill / decode 是跨实例 pipeline，compute 与 memory 管理耦合，KV cache pool 被拆小或需要迁移；MuxWise 则只拆 compute SM，不拆 memory pool。相比普通 intra-GPU overlap，MuxWise 的 layer-wise prefill、query-based synchronization 和 contention guard 明确面向 GPU bubbles 与 memory bandwidth contention 两个风险。
    - pipeline 的局限性：该 pipeline 依赖 GPU 支持低开销 intra-process spatial sharing。论文指出 NVIDIA Pascal 以后可用 GreenContext，AMD 可用类似 CUMask 接口，但不同硬件、驱动和 runtime 成熟度会影响部署。它主要控制 SM，不能精确控制 HBM / L2 / NVLink 等共享带宽，因此仍需要 conservative contention guard。Offline profiling 对每个 LLM-machine pair 都要重新建立；模型、量化格式、kernel 或硬件变化频繁时，维护成本可能上升。大规模集群中，MuxWise 仍需要与全局路由、跨节点 KV placement、负载预测和 disaggregated deployment 协同。
