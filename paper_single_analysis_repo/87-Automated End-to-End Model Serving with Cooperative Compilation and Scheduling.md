1、论文工作：
    - 论文要解决的核心问题：现有 DNN model serving 系统通常把 operator 当作编译和调度的基本单位，导致两类瓶颈。第一，GPU 硬件利用率低：单个 operator kernel 要么整体执行、要么不执行，无法利用部分依赖 operator 之间的 tile 级并行；kernel 内 instruction pattern 单调，容易只压某一类 GPU pipeline unit，如 FP unit 或 global memory bandwidth，而其他单元空闲。第二，服务端调度粒度粗：CUDA runtime / stream / MPS / Triton / Paella 等方案难以精确控制 GPU 内部 kernel 空间共享、任务公平性和实时优先级，面对异构模型和突发请求时会出现 tail effect、cold start、队列阻塞和不稳定 tail latency。
    - 论文的主要贡献：论文提出 Infera，一个通过 cooperative compilation and scheduling 做端到端 DNN inference serving 的系统。核心思想是在编译期把大 operator 切成 micro operators / tiles，并为每个 tile 静态生成多个 micro-kernel 版本；在推理期根据任务状态、kernel 元数据、GPU 状态和优先级动态选择、融合和发射 kernel。贡献包括：对 NVIDIA GPU pipeline 的 ILP / TLP / arithmetic intensity trade-off 分析；基于 TVM 的 tile-based zero-tuning compiler；SASS 级 cut-and-patch instruction scheduling 与 warp specialization；支持 deadline / real-time / normal 任务的 inference server；以及 warp-level binary kernel fuser、device-side daemon kernel launch 和快速 preemption。
    - 论文所处背景：该工作位于深度学习编译器、GPU kernel generation、模型服务系统、GPU spatial sharing 和实时推理调度交叉处。它主要面向传统 DNN / CNN / Transformer encoder 类 inference，而不是 LLM 自回归 serving；评测模型包括 BERT、ViT、Inception、DenseNet、EfficientNet 和 LSTM。论文关注的是单 GPU 上多模型、多请求、异构计算/访存负载的 end-to-end p99 latency 与吞吐，而非 KV cache、prefill/decode 或分布式 serving。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：编译侧 baseline 包括 Ansor、MetaSchedule、Roller 和 cuDNN。Ansor / MetaSchedule 依赖长时间搜索，编译成本高；Roller 减少搜索空间但仍要用性能模型评估程序；cuDNN 性能强但依赖手工优化库。它们大多以完整 operator kernel 为目标，不能为推理期动态调度提供足够细粒度和多版本选择。服务侧 baseline 包括 CUDA Stream、MPS、Triton 和 Paella。Stream / MPS 允许并发但不能保证 GPU 内部空间共享；Triton 是通用 serving 平台但调度仍受 kernel launch / queue 机制约束；Paella 改善 kernel 执行顺序控制，但仍以较粗的 kernel / operator 为调度对象。共同问题是 compilation 与 scheduling 没有协同：编译器不知道推理期的并发上下文，调度器又缺少足够多、足够细、带元数据的 kernel 版本。
    - 论文的设计方法：Infera 把 compilation 和 scheduling 从“动态耦合”改成“静态提供调度空间、运行时动态选择”。编译期，Infera 将 ONNX model 转成 TVM Relay，再用定制 TVM compiler 生成 tile-based TensorIR；对大 operator 生成 micro operators，对小 operator 合成 shepherd operator 以避免过细调度开销；随后生成不同 register / shared memory / tile size / pipeline stage 配置的 micro kernels，提供 ILP、TLP 和 intensity 的多种 trade-off。推理期，Infera server 由 Job Dispatch Unit、Task Schedule Unit 和 Task Execution Unit 组成：JDU 把 job 分发到有足够显存且预计剩余时间最少的 GPU；TSU 用 deadline、real-time 和 GCFS normal policies 生成带 instruction budget 的 Virtual Task；TEU 每个调度周期选择数据块与 kernel，做 warp-level binary fusion，再通过 daemon kernel 在 device side 发射 fused kernel。
    - 方法如何对冲 Baseline 缺陷：相对 operator-based compilation，tile 化让下游 operator 的某些 tile 可以在上游 operator 完整结束前推进，扩大 data asynchrony；多版本 micro kernels 让调度器可以按当前 GPU 占用和 kernel hazard 选择更合适的 ILP/TLP/intensity 配置。相对单纯 CUDA stream 或 MPS，Infera 先把多个 primitive kernels 在 SASS / CUDA binary 层做 warp-level horizontal fusion，再以一个 fused kernel 形式发射，使不同任务的 warp 能在 SM 内空间共存，而不是把空间共享交给不可控的 GPU inner scheduler。相对 tuning-heavy compiler，zero-tuning 通过静态资源约束、instruction scheduling、warp specialization 和固定数值化配置直接生成候选 kernel，降低编译成本。相对粗粒度服务调度，Virtual Task 和 instruction budget 让 normal 任务能按 priority 公平共享调度周期，同时 deadline / FIFO 任务可以抢占低优先级任务。
    - 关键 trade-off：Infera 用很高的系统复杂度换取更细粒度的 GPU 控制：它需要 SASS 级代码改写、driver-level placeholder kernel slot 覆盖、GDRCopy、CUDA Dynamic Parallelism daemon kernel、device-side queue 和自研调度器。daemon kernel 独占一个 SM，带来小于 1/#SM 的设备侧开销；host 侧 kernel fusion、queue 管理和 fused kernel 缓存带来 CPU 与内存开销。tile 级调度也有粒度 trade-off：过细会增加调度/fusion/launch 元数据开销，过粗又退化回 operator-based scheduling。另一个限制是实现强依赖 NVIDIA GPU 执行模型、SASS 格式和底层 driver 行为，跨 GPU 代际和跨厂商可移植性需要额外验证。

3、论文实现：
    - Baseline 如何实现：kernel generation 实验比较 Ansor、MetaSchedule、Roller、cuDNN 和 Infera compiler。搜索型编译器被允许运行到收敛，cuDNN 选最优配置，Infera 从自己生成的候选 kernels 中选择最佳程序。serving 实验比较 Stream、MPS、Triton、Paella 和 Infera，其中 Stream / MPS 使用 TVM 作为 inference backend。preemption 实验比较 EffiSha、REEF-N、Infera-P 和 Infera-R。
    - 新设计如何实现：Infera compiler 基于 TVM 0.16.0，输入 ONNX model，输出 CUDA binary 静态库和模型结构。compiler 先做 tile-based graph/operator partition，再生成 micro kernels：mainloop warps 与 shared-to-register copy warps 各 4 个；global-to-shared 使用 asynchronous copy 和 warp specialization；register 使用上限 64/96/128，shared memory 使用上限 48/80/112/144 KiB；global-level grid size 固定为 64，reduction axis tile size 作为 kernel argument。instruction scheduling 通过 cut-and-patch 完成：先关闭 nvcc 优化生成 CUDA binary，反汇编 SASS，切出 mainloop computation segment，用 list scheduling 降低 stall cycles，再插回优化片段。Inference server 从头实现，核心是约 17k LoC 的 C++ kernel-space module；latency-sensitive CPU 线程使用实时调度或绑定 isolated CPU core，GPU daemon kernel 独占一个 SM。
    - 实验 / 实现平台：评测机器为 Intel Xeon Gold 6330 CPU、512 GB RAM、NVIDIA A100-PCIE-40GB GPU，系统为 Linux 6.1.0 和 CUDA 12.0。kernel generation 评测 operator 包括 Gemm、Conv2D 及其变体、Transpose 和 AveragePool，覆盖 compute-bound 与 memory-bound。serving 评测模型包括 BERT、ViT、Inception、DenseNet、EfficientNet 和 LSTM，参数规模从 20.2M 到 428M；请求到达分布包括 uniform、lognormal sigma=1.5 和 lognormal sigma=2，模型请求分布也有 uniform 与 lognormal 两类，分别测试 single-model 与 mixed multi-model serving。
    - 关键实验设置与指标：kernel 侧指标包括 operator speedup 和 compilation/search time。Infera 生成的 kernel 平均性能至少比其他编译器/库高 5%；相对 Ansor / MetaSchedule，编译搜索时间低 2 到 3 个数量级；相对 Roller，CPU 编译时间节省 66% 到 86%。serving 侧主要指标是 normalized p99 latency 随 request rate 的变化、GPU stall cycles、preemption latency、CPU load 和 host memory usage。single-model inference 中，Infera speedup 为 1.14x 到 1.40x，平均 1.28x；multi-model inference 中，在 uniform requests/uniform models 下至少快 1.6x，在 lognormal requests/lognormal models 下最高快 3.5x。preemption 中，Infera-P 约 10 us，Infera-R 约 5 us；Infera-P 比 REEF-N 快约 2.5x，比 EffiSha 快超过一个数量级。host overhead 随请求率近似线性增长，在最大吞吐前约为 13% CPU 和 600 MiB host memory。

4、pipeline/kernel解析：
    - 新pipeline/kernel是什么：论文最关键的新执行路径是 Infera 的 tile-based cooperative compile-schedule serving pipeline，不是单个新的数学 kernel。它包含两条紧密配合的路径。离线路径是 ONNX -> TVM Relay -> tile-based TensorIR -> CUDA C++ -> PTX/SASS/CUDA binary multi-level optimization -> static library。在线路径是 inference job queue -> JDU GPU dispatch -> per-GPU TSU task scheduling -> TEU kernel selection -> warp-level binary fusion -> HKQ/DKQ/daemon kernel launch -> fused micro-kernel execution。pipeline 的核心变化是把“operator kernel”拆成可独立调度的 tile/micro-kernel，并让 runtime 每个调度周期按 Virtual Task、data dependency DAG、kernel hazard 和 priority 动态组织这些 micro-kernels。
    - 新pipeline/kernel的执行流例子：以一个 mixed-model inference 请求为例。用户先离线调用 Infera compiler，把 ONNX 模型编译成 runtime model；server 注册模型结构、CUDA binary static libraries 和权重，并把相关 host memory 标成 cudaHostAllocWriteCombined / cudaHostAllocPortable 以便快速传输和多设备一致性。在线请求到达后，JDU 将 job 放入 inference job queue，并根据 GPU 可用显存和由指令数/IPC 估算的剩余时间选择目标 GPU。job 在该 GPU 上变成 task，先进入 New 状态，检查显存需求后进入 Blocked 或 Ready；如果权重、输入 tensor 或中间 tensor 不在 GPU global memory，task 会在 mm_wq 中等待异步 swap-in，GPU 显存中的旧模型按 LRU 后台回收。进入调度周期后，TSU 按 runqueue 优先级选择任务：deadline task 用 EDF 且 VTB 通常只含一个 unlimited-budget task，real-time task 用 FIFO，normal task 用 GCFS 按 nice value 分配 instruction budget。TEU 对 VTB 执行三阶段。第一，SelectKernels 在数据依赖 DAG 中选择 zero in-degree 且能最大化 future asynchronous wavefront 的 data blocks，再在可用 micro-kernel 版本中按 #inst / IPC、TLP>=4、结构/数据 hazard 估计和在线轻量回归模型选择 kernel。第二，FuseKernels 把多个 CUDA binary primitive kernels 横向融合到一个统一签名的 kernel 中：为每个线程生成 prologue 恢复 special registers 和 arguments，给各原 kernel 的 shared memory 访问加 offset，重组 barrier，插入 preemption、locking 和 progress flags。第三，LaunchKernel 将 fused kernel 放入 host-side HKQ，再通过 GDRCopy 把代码和参数传到 device-side DKQ，并覆盖预留 placeholder kernel slots；常驻 daemon kernel 用 CUDA Dynamic Parallelism 的 fire-and-forget launch 发射 fused kernel。若高优先级任务到达，TSU 发送 preemption signal，host launcher 暂停 HKQ->DKQ，daemon 保存 DKQ 和 shared-memory queue 上下文，in-flight kernels 通过 flag 尽快终止；由于 DNN kernel execution 被设计为幂等，即使少数 kernel 未及时停下也不会破坏系统状态。
