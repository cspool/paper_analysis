论文标题：PAT: Accelerating LLM Decoding via Prefix-Aware Attention with Resource Efficient Multi-Tile Kernel

    开源仓库确认：

        - 链接：https://github.com/flashserve/PAT；https://zenodo.org/records/18217189；https://doi.org/10.1145/3779212.3790200
        - 说明：论文摘要和 Artifact Appendix 明确给出 PAT 的 GitHub、Zenodo DOI、Docker 镜像和 MIT license。公开仓库 flashserve/PAT 目前是 Public，README 写明该仓库提供 PAT 的 CUTLASS-based decode attention 实现，包含 benchmark、csrc、plugin/vllm、prefix_attn、test 等目录；Zenodo 记录显示该 artifact 于 2026-01-12 发布，文件为 PAT-main.zip，并标注 Repository URL 指向同一 GitHub 仓库。因此可确认论文实现和复现实验材料已公开。需要注意的是，论文 §8.1 写启用方式为 VLLM_ATTENTION_BACKEND=PAT，而当前 GitHub README 示例写 VLLM_ATTENTION_BACKEND="PREFIX_ATTN"，说明开源版本的接口名可能与论文正文存在小幅演进。

    1、论文工作：
        - 论文要解决的核心问题：LLM online serving 的 decode 阶段越来越被 decode attention 主导，而 decode attention 的关键瓶颈不是计算量，而是每步生成 token 时从 global memory 反复加载不断增长的 KV cache 到片上 memory。现实负载中还普遍存在多层级共享 prefix，例如 system prompt、tool/template、RAG 文档和 agent workflow；这些 prefix 在 vLLM / SGLang 等系统中可以复用物理 KV cache 来省显存，但传统 attention kernel 仍会在同一个 decode batch 内重复把共享 KV prefix 从 global memory 读入片上 memory，因此没有真正减少 decode attention 的带宽压力。
        - 论文的主要贡献：第一，论文系统刻画共享 prefix 负载，报告四个真实 trace 的 prefix ratio 为 51.9%-75.0%，ToolAgent / Conversation trace 下 intra-batch shared prefixes 可覆盖 2.8%-82.6% 的 KV cache，并且每个 batch 平均有 2.72 个 distinct shared prefixes。第二，提出 PAT，一个面向 decode attention 的 prefix-aware attention kernel implementation，用 pack-forward-merge paradigm 把共享 prefix 的查询打包到同一个 CTA，从而让 KV 在 shared memory 内复用。第三，设计 memory-centric pack scheduler、resource-efficient multi-tile kernel、runtime tile selector、multi-stream forward、long-KV split 和 lightweight merge kernel。第四，把 PAT 作为 vLLM 插件实现，并在 A100 / H100、synthetic batch、真实 ToolAgent / Conversation trace、TP/PP 和 MoE 场景中评估。
        - 论文所处背景：背景是长上下文、长输出和连续批处理推高 decode attention 的占比。论文在 A100、vLLM v0.9.0、batch size 64 上分析 Llama-3-8B 与 Qwen3-8B，指出 decode attention 可占总延迟最高 53%。已有 FlashAttention / FlashInfer 等 fused attention kernel 通过 tiling、online softmax 和 pipelined memory transfer 降低中间结果和延迟，但仍以 one-query-per-CTA 为主；KV-centric 方法如 FastTree、RelayAttention、DeFT、Cascade Inference 尝试复用共享 prefix，却常用固定 tile 或简单 packing，不能同时适配共享 prefix 的多层结构、动态 batch、不同 query 数量和不同 KV 长度。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：Query-centric kernels（FlashAttention、FlashInfer）把每个 query 和它的 KV cache 独立映射到一个 CTA。这个设计简单，但在 batch 内多个 query 共享 prefix 时会重复加载同一批 KV blocks；论文用 ncu profiling 指出 FlashAttention 的 KV cache traffic 比理论最小值高 4.3-8.7x，也比 PAT 高 4.1-9.5x。KV-centric kernels 虽把共享 KV 放进同一 CTA，但常采用 one-size-fits-all tile size，例如固定 m=64、n=32 或其他固定配置；当共享该 prefix 的 query 数少于 tile query size 时需要 padding，浪费 shared memory / registers；当 CTA 的 KV length 差异很大时，固定 tile 又造成 tail stage 的 SM 空转和 execution bubble。FastTree 的 cost model 更偏 compute-oriented，论文认为这与 decode attention 的 memory-bound 本质不匹配。
        - 论文的设计方法：PAT 的核心是 pack-forward-merge。Pack 阶段把 vLLM block table 转换为 prefix tree，每个 internal node 表示一段 shared prefix，记录 shared KV length 和共享它的 query 数；pack scheduler 用 memory-centric profit model 比较 split / merge 策略，决定哪些 prefix node 和 query 应被放进同一个 CTA。Forward 阶段为不同 CTA 选择不同的 Q tile size m 和 KV tile size n，用多组 CUTLASS/CuTe kernel 适配 query size、KV length、shared memory、register、bandwidth 和 Tensor Core 约束。Merge 阶段把一个 query 被多个 CTA 计算出的 partial max score、log-sum-exp accumulator 和 partial value-weighted sum 用 online softmax 合并为最终输出。
        - 方法如何对冲 Baseline 缺陷：对 query-centric baseline，PAT 把共享 prefix 的多个 query 放入同一个 CTA，使共享 KV block 只从 global memory 加载一次，直接减少 memory-bound decode attention 的主开销。对 KV-centric fixed-tile baseline，PAT 不再使用单一 tile 配置，而是离线求解可行 tile set，运行时根据 CTA query 数和 KV length 做 constant-time tile selection：m 选能覆盖当前 query 数的最小可行 tile，减少 query 维 padding；n 根据 KV length 和 execution bubble trade-off 选择，长 KV 偏大 n 以降低 CTA concurrency 和 tail bubble，短 KV 偏小 n 以避免最后 tile 的 compute bubble。Multi-stream forward 让不同 tile configuration 对应的 kernel 在多个 CUDA streams 上并行，long-KV split 把过长 KV 的 CTA 沿 KV 维拆分，进一步缩短最后完成的 CTA。
        - 关键 trade-off：PAT 接受更复杂的 runtime scheduling、更多 kernel variants、partial results 写回/读回和 merge kernel 的额外开销，以换取共享 prefix 下显著更少的 global KV reads 和更好的片上资源利用。其收益依赖 batch 中 prefix sharing 的比例和并发度；论文也承认没有共享 prefix 或小 batch 下，PAT 的 KV 读复用优势会变小，只剩 multi-tile / multi-stream 带来的较小收益。PAT 还依赖 GPU kernel 工程和特定硬件配置的离线 profiling；移植到新 GPU 需要重新根据 shared memory、register、bandwidth 和 CUTLASS 约束推导等价 tile set。由于 GPU scheduling 不完全可控，multi-stream 仍会留下 residual bubbles，距离理论最优仍有差距。

    3、论文实现：
        - Baseline 如何实现：论文比较七类 attention implementations。FlashAttention v2.5.9 是 query-centric baseline，固定 tile config 为 (64,128)；FlashInfer v0.2.5 也是 query-centric，通过 dynamic CTA partitioning 改善 SM load balance，decode tile config 为 (16,128)；FastTree 是 KV-centric baseline，用 compute-oriented cost model 打包并减少重复 KV 读，使用 (64,32) 和 (16,32) 两个 tile configs；RelayAttention 打包 first-level shared prefixes 并使用 FlashAttention kernel；RelayAttention++ 是作者扩展版本，用 vLLM-style KV-cache reuse 让 non-first-level prefix 的共享 KV block 也能在物理空间复用并受益于 L2 cache；DeFT 聚合共享 KV query 并调整每个 CTA 的 KV length 做负载均衡，固定 tile config 为 (32,16)；Cascade Inference 用固定设置把 prefix 打包进 CTA。
        - 新设计如何实现：PAT 的 decode full attention kernel 由约 3k 行 Cutlass/CuTe 和 C++ 实现，multi-tile kernel、multi-stream forward 和 merge kernel 都基于 Cutlass/CuTe；asynchronous pack scheduler 和 API wrappers 用 C++ 实现；Python 侧通过 pybind11 暴露 kernel API，并用约 1.2k 行 Python 集成为 vLLM v0.9.0 的插件。全局到 shared memory 的数据搬运使用 cp_async 和 double buffering。PAT 复用 vLLM paged KV cache：KV entries 仍以 fixed-size blocks 和 block table 管理，pack scheduler 只处理 vLLM 产生的 logical block IDs，不重写 KV paging 机制。
        - 实验 / 实现平台：Kernel benchmark 在 NVIDIA A100 80GB 和 H100 上评估；端到端 online serving 使用单张 A100，模型为 Qwen3-8B 和 Llama-3-8B；分布式和 MoE 扩展实验使用 Qwen2.5-72B-Instruct，在 4 张 A100 上 TP=2、PP=2，以及 Qwen3-30B-A3B 在单张 A100 上运行。软件环境为 CUDA 12.4、PyTorch 2.7.0、vLLM v0.9.0。Artifact 要求 x86-64 Linux、至少 64GB RAM、200GB 空闲磁盘和 A100 80GB；官方提供 flashserve/pat:ae Docker 镜像，kernel benchmark 约 1.5 小时，端到端完整实验超过 60 GPU-hours。
        - 关键实验设置与指标：Synthetic kernel workloads 用 B 和 L 描述 prefix tree 结构、leaf 数量和各层 KV length，例如 B=[1,4,16] 表示两级 shared prefix 和 16 个 leaves，L=[128,256,1024] 表示两级共享 prefix 长度和每请求 non-shared 长度。head configs 选取 Llama、Qwen、Gemma 常见的 (#heads,#kv_heads)=(64,8),(32,8),(16,8),(32,32)，head dimension 为 128，数据类型为 FP16。真实 workload 使用 ToolAgent trace 和 Conversation trace；Conversation 构造三层 prefix，Llama3 tokenizer 下长度为 46、348、2123，Qwen3 下为 45、351、2126。指标包括 attention latency、mean TTFT、mean TPOT、P99 TPOT，kernel 配置重复 20 次取平均。核心结果是：synthetic shared-prefix 配置下，PAT 相对 FlashAttention、FlashInfer、FastTree、RelayAttention、RelayAttention++ 的最高加速分别为 21.5x、11.7x、3.2x、11.9x、5.7x；对 query-centric FlashAttention / FlashInfer 平均降低 attention latency 67.8% / 52.1%；对 DeFT / Cascade Inference 平均降低 attention latency 76.6% / 41.2%。真实端到端下，PAT 相比 RelayAttention++、FlashAttention、FlashInfer 的 mean TPOT 分别降低 17.2%-68.1%、17.0%-89.5%、32.2%-93.1%，TTFT 分别降低 9.3%-98.6%、10.1%-99.6%、22.5%-99.8%。分布式 / MoE 下，相比三种 baseline，PAT 在 Qwen2.5-72B-Instruct 上降低平均 TPOT 14.3%-26.7%，在 Qwen3-30B-A3B 上降低 5.53%-16.9%。Ablation 显示，固定 tile 的 PAT-fixed 比 PAT 慢 39%，serial multi-kernel 的 PAT-serial 比 PAT 慢 4.8%，compute-oriented / naive pack scheduler 的 memory read/write 分别比 PAT 高 10.9% / 16.7%。

    4、pipeline/kernel 解析：
        - 新 pipeline/kernel 是什么：PAT 提供的不是单个传统 CUDA kernel，而是一条 prefix-aware decode attention execution pipeline，即 pack-forward-merge pipeline。它包含三类关键执行单元：第一，pack scheduler，把 decode batch 的 block table 转为 prefix tree，并用 memory-centric profit model 生成 CTA partition；第二，multi-tile forward kernels，为每个 CTA 选择合适的 (m,n) tile 配置，并用 Cutlass/CuTe kernel 执行 QK 和 PV；第三，merge kernel，用 online softmax 合并每个 query 被拆分到多个 CTA 后产生的 partial results。论文把这一设计作为 vLLM 后端插件，而不是改变上层 serving system 的 continuous batching 或 KV paging 语义。
        - 新 pipeline/kernel 的执行流例子：一次 decode step 开始时，vLLM 已经为 batch 中每个请求维护 block table，表中每一行是该 query 的 KV block IDs。PAT 的 pack scheduler 先读取这些 logical block IDs，把相同 leading block IDs 抽象成 prefix tree：root 到 leaf 的路径表示某个 query 的完整 KV blocks，internal node 表示多个 query 共享的一段 prefix。随后 scheduler 对每个 internal node 估算把 query 打进同一 CTA 能节省多少 global KV reads，以及 split / merge 会带来多少 FP32 partial intermediate 写回和读回；当 child node 的 query 数足够多、共享 prefix 足够短时，可能把 parent 和 child merge 到同一个 CTA，否则 split 为独立 CTA。lazy update 让该调度结果在 block table 未变化时跨 continuous-batching iterations 复用，并与 QKV projection / metadata preparation 等 pre-attention task 异步重叠。
        - 新 pipeline/kernel 的执行流例子：进入 forward 阶段后，每个 packed CTA 包含一组共享某些 KV blocks 的 queries 和对应 KV range。runtime tile selector 先根据该 CTA 的 query 数 q 选择最小可行 Q tile size m，避免 m 过大造成 padding；再根据 KV length、A100/H100 的可行 tile set、带宽饱和约束和 tail bubble profiling 选择 KV tile size n。具有相同 (m,n) 的 CTAs 被分组放进同一个 CUDA stream，不同 tile configuration 的 streams 并行提交；若某个 CTA 的 KV length 超过 batch 中 CTA KV length 均值，则沿 KV 维拆成多个部分，以免长尾 CTA 拖住整个 kernel。每个 forward kernel 内部用 cp_async 和 double buffering 把 K/V tile 从 global memory 搬到 shared memory，同时执行 QK^T、softmax 相关统计和 PV 累加；由于多个 query 共享 prefix KV，同一个 shared-memory KV tile 可被 CTA 内多个 query 复用，不再重复从 global memory 加载。
        - 新 pipeline/kernel 的执行流例子：当一个 query 的 KV 被多个 CTA 或多个 KV splits 处理后，每个 CTA 输出 per-query/per-head 的 partial max score、log-sum-exp accumulator 和 partial value-weighted sum。这些 intermediate 会写到 global memory。最后 lightweight merge kernel 读取同一 query 的所有 partial intermediates，用 online softmax 做跨 CTA 的 max / sum 归并，再归一化 partial value-weighted sum，拼接所有 heads，写回最终 query output。整体来看，PAT 把传统“每 query 一个 CTA、每个 CTA 重读完整 KV”的路径改成“按 prefix 打包 CTA、共享 KV 读、按 CTA 选择 tile、多 stream forward、online merge”的路径，直接围绕 decode attention 的 global-memory bandwidth 瓶颈进行优化。
