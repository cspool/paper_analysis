# Paper Experiment Idea

## Overview

根据所给论文标题，在 `papers/paper_2026/` 目录中找到原文PDF。阅读全文，提取复现论文实验所需的软硬件信息，按照实现和实验分层次输出到 `experiment_repo/`，并提取设计思路输出到 `idea_repo/`。

## Workflow

### 第一步：定位论文PDF

1. 根据用户提供的论文标题、编号、文件名或标识符，在 `paper_2026/` 目录中搜索原始PDF。
2. 搜索策略：使用 `rg --files`、标题片段、论文编号和可能的文件名变体进行匹配。
3. 如果用户只给出编号，先通过 `paper_titles.md` 等本地标题列表进行映射。
4. 优先使用可读全文，顺序为：配套提取的 `.txt`/`.md` > PDF文本提取 > 官方论文HTML > 其他本地笔记。不要仅依赖摘要或二手总结。

### 第二步：阅读论文实现与实验章节

阅读以下最有助于实验提取的章节：

- Implementation / Experiment / Evaluation / Experimental Setup
- Method / Design / Architecture / Approach
- Introduction / Motivation / Background

基于以上章节总结：

- 论文如何实现 baseline
- 论文如何实现所提出的创新设计
- 使用了哪些硬件平台、模拟器、框架、模型、数据集或 benchmark
- 为了复现，从联网搜索或从原文查找是否开源和开原链接

### 第三步：实验分层与输出

按实现和实验对论文的实验和实现分层次，输出到 `experiment_repo/` 目录下的对应层次 md 文件。论文实现可以包含多个层次，对应多个层次 md 文件中写入要求内容。不完全匹配的按照最相似/最接近来分层并反馈。

### 第四步：设计思路提取

基于论文实现归属的层次，和论文原文中的 Introduction、Motivation、Background 和 Method 章节，说明论文通过什么具体设计解决 Baseline 的痛点，输出到 `idea_repo/idea库.md`。

## 证据规则

- 信息以论文原文为准。若细节缺失，写 `论文未明确说明` 而非自行编造。
- 保留模型、数据集、框架、模拟器、benchmark、芯片、GPU、RTL工具、kernel、指标等具体名称。
- 若论文给出模拟器或工具名称但未给链接，先在论文中搜索URL；必要时用联网搜索，无法确认则写明无法确认。
- 将"baseline 实现"理解为论文实际评估的具体版本，而非仅是一般性的先前工作概念。
- 保持文字简洁、技术化、便于后续复现使用。

## 实验层次分类

将每篇论文的实现和实验按以下层次分类。一篇论文可属于多个层次，每个匹配的层次对应一个追加条目。

### 算法pipeline

适用条件：实现是通过稀疏、压缩、量化、蒸馏、共享或近似权重/激活等方式加速算法推理，同时保证准确率，或者提出新的算法模型。

追加到 `experiment_repo/实验_算法pipeline.md`：

```md
## <论文标题>

- 属于算法pipeline的实现是什么？实验比较什么？
- 硬件平台是什么，配置是什么。
- 模型是什么。数据集和bench分别是什么。
- 开源情况。基于开源文档和论文，使用例子解释，解释算法pipeline。
```

适用条件：实现是修改开源Serving框架来优化多请求调度，满足SLO要求或提高吞吐量等。

## 设计思路条目

在实验分层完成后，追加一条设计思路到 `idea_repo/idea库.md`。使用论文的 Introduction、Motivation、Background 和 Method/Design 章节，说明论文通过什么具体设计解决 Baseline 的痛点。

```md
## <论文标题>

- baseline方法是什么？
  <说明baseline方法，并给出baseline在模型推理算法-系统框架-编译框架-kernel调度-硬件架构全栈的执行例子。缺失层次写"论文未明确说明"。>

- 论文方法是什么？如何对应解决Baseline的缺陷？
  <说明论文方法，并对比baseline给出论文方法在模型推理算法-系统框架-编译框架-kernel调度-硬件架构全栈的执行例子。缺失层次写"论文未明确说明"。>
```

全栈执行例子应具体而非穷举。好的回答沿着一个请求、token、tensor、tile、kernel、内存传输或芯片级数据路径，穿过论文支持的尽可能多的层次。

## 质量检查

完成前确认：

- 验证每个追加的文件路径存在且包含新条目。
- 验证目标论文尽可能通过 `paper_2026/` 解析，若非 `paper_2026/` 来源则说明。
- 确保同一论文在同一目标文件中不重复（除非用户明确要求再次处理）。
- 确认每个实验层次条目包含比较目标/baseline、提出的实现、评估的硬件/模拟器/框架。
- 确认设计思路条目直接将 baseline 缺陷映射到方法设计选择。
- 在最终回复中列出更新的文件、任何缺失证据、以及近似层次匹配情况。
