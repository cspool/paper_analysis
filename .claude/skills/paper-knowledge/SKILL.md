# Paper Knowledge Base

## Parameters

调用本 skill 时，用户**必须**提供以下两个路径参数：

| 参数 | 说明 | 示例 |
|------|------|------|
| `{{paper_dir}}` | 论文 PDF 所在目录 | `/data3/paper_analysis/papers/paper_2026` |
| `{{knowledge_dir}}` | 知识库 Markdown 文件输出目录 | `/data3/paper_analysis/repo_2026/knowledge_repo` |

## Overview

根据所给论文标题，在用户提供的 `{{paper_dir}}` 目录中找到原文PDF。阅读论文背景、Preliminary、动机（baseline）和方法章节，提取关键学术术语，判断关键术语所处层次，并根据论文描述或联网搜索解释，输出到对应层次的知识库。

某层次知识库中已有相同术语，则根据新描述，补充原有术语的知识。

对于无法精准归属层次的术语，自行判断最可能/最接近的归属层次，并反馈报告。

## Workflow

### 第一步：定位论文PDF

1. 根据用户提供的论文标题、编号、文件名或标识符，在 `{{paper_dir}}` 目录中搜索原始PDF。
2. 搜索策略：使用 `rg --files`、标题片段、论文编号和可能的文件名变体进行匹配。
3. 如果用户只给出编号，先通过 `paper_titles.md` 等本地标题列表进行映射。
4. 优先使用可读全文，顺序为：配套提取的 `.txt`/`.md` > PDF文本提取 > 官方论文HTML > 其他本地笔记。不要仅依赖摘要或二手总结。

### 第二步：阅读论文关键章节

阅读以下最有助于术语提取的章节：

- Background / Preliminary
- Motivation
- Baseline
- Introduction
- Method / Design / Architecture / Approach
- Implementation / Evaluation（仅在需要理解术语具体使用时阅读）

### 第三步：提取关键学术术语

提取可跨论文复用的关键学术术语：机制、模块、调度概念、kernel、框架、算法、模型组件、硬件组件、内存/互联概念、系统策略等。

对每个术语，捕获：

- 术语本身的定义和解释
- 从归属层次角度的拆解（结合论文描述的具体流程或计算过程）
- 术语的一般实现方式和使用方式

结合论文描述和联网搜索，让解释具体、精准。

### 第四步：术语分层

将每个术语归类到以下六个层次中最接近的一个。无法精准归属的术语，选择最可能/最接近的层次，并在最终报告中标注。

### 第五步：更新知识库

更新 `{{knowledge_dir}}` 目录下对应的 Markdown 文件。如果文件不存在则创建。不覆盖无关内容。

如果某层次知识库中已有相同术语或明确别名，将新论文的解释**合并**到已有条目中，而非创建重复条目。将论文标题追加到 `涉及论文标题`。

### 第六步：输出报告

在最终响应中列出：

- 更新的文件
- 解析的论文来源
- 分类或解释存在不确定性的术语

## Evidence Rules

- 以论文文本为基础。如果术语被命名但未解释，仅从论文近邻上下文推断，并明确标注信息缺口。
- 使用联网搜索帮助使术语解释、层次拆解和实现/使用说明具体和精准，特别是当论文文本简洁时。优先使用官方文档、论文、项目页面或权威参考资料，并保持条目以目标论文的使用为基础。
- 如果证据缺失，写 `论文未明确说明` 而非编造细节。
- 保留模型、框架、kernel、模拟器、芯片、互联、内存系统、调度策略和指标的具体名称。
- 每个术语保持简洁但对后续研究复用有用。当论文提供更具体的用法时，避免通用的一行字典定义。
- 对于 `从<层次>角度拆解术语...`，解释术语在分配层次内部如何工作，而非给出通用定义。优先给出操作流程。对于 `kernel调度` 和 `算法pipeline`，当论文给出足够细节时包含伪代码或具体计算过程。
- 对于 `术语一般如何实现？如何使用？`，结合论文的具体描述和必要的可靠背景知识。优先使用实现/使用机制，如芯片互联组织、硬件模块、kernel执行、编译器优化、服务/运行时行为或模型pipeline步骤。
- 对于不精确的层次匹配，在最终响应中简要解释最接近层次的理由；仅在有助于未来复用时将该理由添加到条目本身。
- 谨慎处理别名。合并明显的变体，如 `KV Cache`、`KVCache` 和 `Key-Value Cache`；将最标准的可见术语作为标题，并在解释中提及别名。

## Merging Rules

更新已有术语时：

- 保留有用的旧解释，追加新论文特定的细微差别。
- 当新论文提供更清晰或不同的证据时，添加或修订各字段。
- 仅当论文标题未列出时才追加。
- 避免在同一层次文件中为同一术语创建重复标题。

## Quality Checklist

完成前验证：

- 每个更新的文件路径存在且包含新条目或合并条目。
- 目标论文尽可能通过 `{{paper_dir}}` 解析。
- 每个术语包含其目标层次所需的字段。
- 已有术语被合并而非重复。
- 每个 `涉及论文标题` 包含当前论文标题。
- 在最终响应中记录所有近似的层次选择、最接近层次的理由和缺失的论文证据。
