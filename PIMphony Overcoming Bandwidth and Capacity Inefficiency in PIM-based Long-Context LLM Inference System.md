    1、论文工作：
        - 论文要解决的核心问题：长上下文 LLM decoding 让 Attention 的 KV cache 访问成为主要瓶颈。随着 context length 从 4K、32K 扩展到 128K 甚至 1M tokens，Attention 从较高计算强度的 GEMM 转向围绕 KV cache 的低计算强度 GEMV；系统同时受 memory bandwidth 和 memory capacity 限制。PIM 理论上适合这类 memory-bound 操作，但既有 PIM LLM 加速器在长上下文下暴露出三个系统性低效：PIM channel underutilization、WR-INP/MAC/RD-OUT 固定流水造成的 I/O bottleneck、按最大上下文长度静态预留 KV cache 造成的大量容量浪费。
        - 论文的主要贡献：PIMphony 是一个面向多节点 PIM 的 orchestrator，围绕长上下文 decoding 的 Attention 路径提出三项协同技术。第一，Token-Centric PIM Partitioning（TCP）把并行维度从 head/batch 转到 token 维度，在单个 PIM module 内把 token 分布到所有 channels，从而减少小 batch 和请求长度差异带来的闲置。第二，Dynamic PIM Command Scheduling（DCS）在 PIM controller 中跟踪 GBuf/OBuf entry 级依赖，让 WR-INP、MAC、RD-OUT 在真实依赖满足时乱序/提前发射，重叠数据搬运和计算。第三，Dynamic PIM Access（DPA）用动态 loop、动态 operand modification 和 on-module dispatcher 做运行时 VA-to-PA 翻译，使 KV cache 可以按需分配，而不是按 Tmax 静态预留。
        - 论文所处背景：既有 GPU serving 系统如 vLLM、TensorRT-LLM、SGLang 通过 paged attention、continuous batching、flash-decoding 等优化 GPU 侧吞吐，但在超长上下文 decoding 中仍受 GPU HBM 带宽和容量约束。既有 PIM 系统如 NeuPIMs、CENT、PAPI 等利用 DRAM-PIM 内部带宽加速 LLM decoding，但多数聚焦较短上下文或没有系统性处理长上下文下的 channel mapping、command scheduling 和 KV cache memory management。PIMphony 的目标不是改变 Transformer 数学，而是让 PIM 在长上下文 Attention 的真实访问形态下持续吃满内部带宽，并让内存容量随请求真实长度使用。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：第一类 baseline 是 head-first / batch-head partitioning 的 PIM 映射方式。它把 head-batch pair 分配到 channel，默认 batch size 或 head 数足够多；但长上下文下单个请求的 KV cache 足以占满一个 channel，batch size 被容量压低。Tensor Parallelism 下，不同请求 token length 不同会造成 channel 执行时间不一致，短请求所在 channel 早早空闲；Pipeline Parallelism 下，每个 stage 只激活与当前请求相关的少数 channel，形成 sparse channel activation。论文在 32K context 的 CENT 分析中观察到 MAC utilization 可下降 48%。
        - Baseline 的具体问题：第二类 baseline 是静态 PIM command scheduler。PIM primitive 采用 WR-INP -> MAC -> RD-OUT 序列，WR-INP 把 32B tile 写入 GBuf，MAC 读取 GBuf 做 dot-product 并累加到 output register，RD-OUT 再读出结果。传统 scheduler 只按固定命令顺序和保守时间间隔发射，不跟踪 GBuf/OutReg entry 级真实依赖；即使命令之间没有 hazard，也会等待固定 tWR-INP、tMAC、tRD-OUT 间隔。Attention 的 QK^T 和 SV 因 dh/dout 小、数据复用低，I/O transfer 更频繁，静态调度导致 MAC pipeline 大量 idle；论文示例中小维度 Attention 的 MAC utilization 可低至 14.7%。
        - Baseline 的具体问题：第三类 baseline 是静态 KV cache 管理。传统 PIM 指令的 loop count 和 operand address 在编译期固定，无法根据当前 token length 调整 row/col address；因此系统必须按最大上下文 Tmax 为每个请求预留 KV cache。真实 workload 如 LongBench、LV-Eval 的请求长度差异很大，静态预留导致平均容量利用率只有约 36.2%，论文后续 ablation 中静态方案在不同任务上只有 31.0%-40.5% utilization。
        - 论文的设计方法：TCP 在 PIM module 内沿 token 维度切分 Attention 的 QK^T 和 SV。对于 QK^T，每个 channel 处理一段 token 的 Key cache，与同一个 query 做部分 dot-product；结果在 module 内拼接或经 EPU/Softmax 继续处理。对于 SV，每个 channel 处理一段 score/value 的 partial context，再通过 PIM HUB/GPR 做 inter-channel reduction。TCP 只在 module 内切 token，避免跨 module 同步，论文报告 SV 的单次 module 内 reduction 开销在 LLM-7B、16K tokens 下低于 attention latency 的 0.2%。
        - 论文的设计方法：DCS 增加 I/O-aware buffering 和 dependency-aware scheduling。它把原有 multi-entry GBuf 作为 input buffer，并把 output register 扩展成更大的 output buffer（OBuf），使用 dual-port memory 支持 MAC 消费当前数据的同时写入下一批输入或读出已完成输出。controller 维护 Dependency Table（D-Table）记录每个 GBuf/OBuf entry 最近访问命令，维护 Status Table（S-Table）记录命令 ID、完成时间和 OBuf 的 is-MAC flag。新命令到达时只等待它真正依赖的前序命令完成；无关的 WR-INP、MAC、RD-OUT 可以乱序穿插。
        - 论文的设计方法：DPA 引入 Dyn-Loop 和 Dyn-Modi 两类动态 PIM 指令。Dyn-Loop 的 loop bound 来自请求当前 token length，而不是编译期最大值；Dyn-Modi 在 loop 内按 stride 修改 row/col 等 operand field，形成逻辑 virtual address。on-module dispatcher 保存 instruction buffer、configuration buffer 和 VA2PA table，在 PIM HUB 内把虚拟地址翻译到实际分配的物理 KV cache chunk。host 只在新请求进入、请求需要更多 chunk、请求结束释放时更新映射，不在每个 decoding step 介入。
        - 方法如何对冲 Baseline 缺陷：TCP 用充足的 token 维度替代稀缺且波动的 batch/head 并行，长上下文越长越容易让每个 channel 都有工作；论文指出在 16-channel、16-bank 的 commercial PIM module 配置下，QK^T token length 超过 256、SV token length 超过 32 即可 full channel activation。DCS 用 entry-level dependency 替代固定 timing gap，把 Attention 中不可避免的输入/输出搬运和 MAC 重叠，尤其让 GQA row-reuse mapping 的 KV reuse 不再被额外 input transfer 抵消。DPA 则让 KV cache 按真实 token length 以 1MB chunk lazy allocation，容量浪费只限制在每个请求最后一个 chunk，而不是按 Tmax 整段浪费。
        - 关键 trade-off：TCP 接受 module 内 partial result 聚合和 reduction 的开销，换取 channel utilization；它适合长上下文，因为 token 数足够大，短上下文收益会降低但论文仍报告 256 tokens 下有 2.1x speedup。DCS 接受更复杂的 PIM controller、D-Table/S-Table、dual-port/expanded output buffer 和 dependency check logic，换取更高 MAC utilization；论文估计 OBuf area 为 MAC unit area 的 0.47%/bank，DCS control blocks 带来 0.5% area 和 1.3% power 增加。DPA 接受 on-module dispatcher、VA2PA table 和 host-runtime memory mapping 复杂度，换取容量利用率和 batch size；dispatcher 内部 buffer 小于 200KB，但会带来 4% area overhead。

    3、论文实现：
        - Baseline 如何实现：评估使用两个 PIM-based baseline。CENT 是 PIM-only 系统，每个 module 16GB，内部带宽 16TB/s，所有 LLM 计算都由 PIM/PNM 处理；NeuPIMs 是 xPU+PIM heterogenous 系统，每个 module 32GB，内部带宽 32TB/s，compute-intensive kernel 如 GEMM/FC 交给 xPU/NPU，memory-bound Attention offload 到 PIM。论文还把 GPU system 作为对照，采用 A100-80GB，结合 flash-decoding（FD）和 paged-attention（PA），并按内存容量匹配 PIMphony：7B 使用两张 A100-80GB，72B 使用八张 A100-80GB。
        - 新设计如何实现：作者在 MLIR 上实现 PIMphony compiler/runtime。compiler 扩展 MLIR dialect，识别 Transformer decoder pattern，针对 QK^T、SV、FFN 等 PIM-amenable kernels 生成 PIM-specific code，并嵌入 token-centric partitioning 和 dynamic memory allocation metadata。runtime 基于 IREE runtime stack 和 HAL 扩展，与 commercial PIM SDK 抽象对接；编译离线完成，不计入 inference latency。
        - 新设计如何实现：系统集成上，作者把 compiler 生成的 PIM instruction sequences 接入 NeuPIMs 和 CENT simulator。在 NeuPIMs 中，PIMphony offload bandwidth-bound Attention 到 PIM modules；在 CENT 中，所有计算都在 PIM-only 后端执行。硬件建模包含 on-module dispatch logic、I/O buffering、DCS dependency table/status table、expanded output buffer，并用 AiMX PIM specification 校准 DRAM command timing 和 resource contention。
        - 实验 / 实现平台：评估使用 validated Ramulator-based simulator，结合 AiMX 架构参数。模型覆盖 LLM-7B 和 LLM-72B，包含 non-GQA 的 32K context 模型以及 GQA-enabled 的 128K context 模型；Table I 中 LLM-7B 有 32 layers、32 heads、dh=128，LLM-72B 有 80 layers、64 heads、dh=128。系统容量设置遵循 prior PIM studies：7B 用 128GB，72B 用 512GB；PIM-only CENT 配置中 7B 为 8 modules、72B 为 32 modules，xPU+PIM NeuPIMs 配置中 7B 为 4 modules、72B 为 16 modules。
        - 关键实验设置与指标：benchmark 使用 LongBench 和 LV-Eval。non-GQA 的 LLM-7B/72B-32K 在 LongBench 上评估，GQA-enabled 的 LLM-7B/72B-128K 在 LV-Eval 上评估；任务包括 QMSum、Musique、multifieldqa、Loogle-SD 等，真实输入长度均值从约 13.9K 到 60.8K，最大可超过 100K。指标包括 end-to-end throughput、Attention latency breakdown、MAC utilization、energy breakdown、capacity utilization、不同 TP/PP 组合、系统容量 128GB-1024GB scalability、context length 4K-1M scalability，以及相对 GPU baseline 的 throughput。
        - 关键实验结果：整体吞吐方面，PIMphony 在 PIM-only 系统上最高 11.3x，在 xPU+PIM 系统上最高 8.4x；non-GQA 32K 模型在 LongBench 上获得约 2.1-4.5x speedup，GQA 128K 模型在 LV-Eval 上因更长上下文和 row-reuse/DCS 协同而收益更大。可扩展性方面，在固定 512GB、context length 扩到 1M tokens 时，CENT baseline 因 pipeline bubbles 退化到 2% utilization，而 PIMphony 达到 46.6x speedup；NeuPIMs baseline 更稳定，但 PIMphony 仍有 5.0x speedup。容量方面，DPA 将平均 capacity utilization 提升到 75.6%，显著高于静态方案的 31.0%-40.5%。能耗方面，baseline Attention 中 background energy 占 71.5%，PIMphony 通过最高 19x Attention acceleration 将 background share 降到 13.0%，Attention energy 最高降低 3.46x。
        - 实验限制与假设：PIMphony 的性能结论主要来自 simulator 而非真实多节点 PIM prototype；硬件 overhead 通过 CACTI 和已有 AiMX/PIM 参数估计。论文评估覆盖 7B/72B 与最长 1M context 的可扩展性，但实现细节依赖作者修改的 CENT/NeuPIMs simulator 和 MLIR/IREE runtime，未公开 artifact 时外部复现难度较高。DPA 的 lazy allocation 采用 1MB chunk，容量碎片和 host-PIM 映射更新开销在论文 workload 中较小，但不同请求分布和更细粒度分配策略下可能需要重新评估。

    4、pipeline/kernel 解析：
        - 新 pipeline/kernel 是什么：论文没有提出传统 GPU kernel，而是提出 PIMphony 的 long-context decoding PIM execution pipeline。它由三个相互嵌套的路径组成：第一，TCP 改变 Attention 在 PIM module 内的 token-level dataflow；第二，DCS 改变 PIM controller 的 WR-INP/MAC/RD-OUT command issue path；第三，DPA 改变 KV cache address generation 和 memory allocation path。三者共同让一次 decode step 的 Attention 请求从 “静态地址 + head/batch 映射 + 固定命令序列” 变成 “动态地址 + token-centric 映射 + dependency-aware command scheduling”。
        - TCP 执行流例子：以一个长上下文请求的某层某个 attention head 为例。baseline HFP 会把该 head-batch pair 固定放到少数 channel 上，其他 channel 可能空闲。TCP 则把 token sequence 切成多个 token segment 分给同一 module 内所有 channel。执行 QK^T 时，query 被广播/写入 GBuf，每个 channel 读取自己负责的 Key cache token 段，计算部分 score；各 channel 的 score segment 在 PIM HUB/EPU 处拼接并进入 Softmax。执行 SV 时，score segment 与对应 Value cache 段在各 channel 上做 partial context，随后通过 module 内 reduction 得到完整 context vector。由于 token partition 不跨 module，避免了跨 module synchronization。
        - DCS 执行流例子：以一次 FP16 GEMV 的 PIM command stack 为例。compiler 生成 WR-INP W0/W1/W2，把 32B input tiles 写入 GBuf 0/1/2；随后 MAC M3/M4/M5 读取对应 GBuf entry 和 DRAM row/col，累加到 OBuf；RD-OUT R6 读出 output。静态 scheduler 会按固定顺序等待，哪怕 W2 与 M3 无依赖、R6 与后续 M7 无依赖也会阻塞。DCS 中，M3 到达时从 D-Table 查到它只依赖 GBuf 0 的 W0；S-Table 显示 W0 完成后，M3 可立即发射，不必等待 W2。类似地，M7 与 R6 不冲突时可以先于 R6 发射。论文示例中同一命令序列从 34 cycles 缩短到 22 cycles。
        - DCS 在 GQA/row-reuse 下的执行流：GQA 让多个 query heads 共享同一组 K/V，因此优先在当前打开 DRAM row 上处理所有共享该 KV row 的 query/score，可减少 ACT/PRE overhead；但这也会更频繁地把 query/score 写入 GBuf，扩大 WR-INP 压力。静态 scheduler 下，row-reuse 的收益会被 input transfer stalls 抵消。DCS 利用 dual-port GBuf/OBuf 和 entry-level dependency，在 MAC 消费当前 GBuf entry 时预取下一批 query/score，或在 MAC 继续写 OBuf 其他 entry 时读出已完成结果，从而把 row-reuse mapping 的 KV reuse 转化为真实吞吐收益。
        - DPA 执行流例子：一个请求进入系统时，host 给 on-module dispatcher 初始化 request ID、当前 token index Tcur、初始 VA2PA table。编译器生成的 DPA instruction 不枚举 Tmax 个物理地址，而用 Dyn-Loop 表示 “按当前 token length 循环”，用 Dyn-Modi 表示 “每轮按 stride 修改 row/col operand”。例如 MAC row 可以由 Tcur/(nCH * nBank) 推导成逻辑 row；dispatcher 在 decode instruction 时根据 request ID 查询 VA2PA table，把 virtual row/col 映射到已分配的物理 chunk。若 decode 过程中 KV cache 增长超过当前 chunk，host 分配新的 1MB chunk 并更新 VA2PA；已完成请求释放 chunk，后续请求可复用容量。
        - 一个完整 decode step 如何流过 PIMphony：对某个 batch 中的请求，xPU/PIM 或 PIM-only 后端先完成当前 token 的 projection/FFN 和 Q/K/V 生成；新的 K/V append 到 DPA 管理的动态 KV cache。进入 Attention 时，DPA dispatcher 按当前 Tcur 生成对应 token 范围的 PIM commands，并把虚拟地址翻译到实际 chunk。TCP 把 QK^T 的 token range 分散到 module 内所有 channels；DCS 在每个 channel 的 controller 内把 WR-INP、MAC、RD-OUT 按真实依赖重叠执行。QK^T 的 partial scores 拼接并 Softmax 后，SV 再沿 token 段并行读取 V cache，产生 partial context 并在 module 内 reduce。最终 context 返回后继续后续 decoder 计算，并在本 step 结束时更新 token length 和必要的 VA2PA 映射。
