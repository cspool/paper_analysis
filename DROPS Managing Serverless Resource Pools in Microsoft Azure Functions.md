论文标题：DROPS: Managing Serverless Resource Pools in Microsoft Azure Functions

    1、论文工作：
        - 论文要解决的核心问题：Azure Functions 为避免高 container-allocation latency，会为不同 runtime / container size 维护预热 container pool，并维护 ready VM pool 来快速补充 container pool。pool 太小会导致 allocation failure，请求需要临时创建 container 或甚至新 VM，延迟从几百毫秒到数秒、严重时到分钟级；pool 太大则产生大量 idle container / idle VM 成本。论文指出生产平台中预热资源池成本占总体平台成本的 76%，因此核心问题是如何在满足 allocation success-rate SLO 的同时减少预热资源池成本。
        - 论文的主要贡献：第一，论文基于 Azure Functions 生产 container-allocation trace 做了大规模表征，强调该 trace 不同于 function invocation trace，因为不是每次函数调用都会触发 container allocation。第二，论文系统评估传统 forecasting 和 reactive pool management，发现负载高度 bursty、缺少周期性、创建延迟长尾，导致预测和滞后反馈都难以同时满足 SLO 与低成本。第三，提出 DROPS，一个 statistical、data-driven 的资源池优化方法，用历史 demand trace、recycling trace 和 resource creation latency distribution 构建 pool-size-to-success-rate mapping，从而为任意目标 SLO 选择最小 pool size。论文称 DROPS 已部署到 Azure Functions 所有 region / data center。
        - 论文所处背景：该工作属于 serverless computing、云平台资源管理、resource buffering / pool sizing 和生产系统 cost optimization。Azure Functions 的平台结构是 VM 上运行多个 nested VM，每个 nested VM 承载一个特定 runtime 的 container；平台使用 nested virtualization 做租户隔离，不做资源 oversubscription，因此每个预热 container 会独占 VM 上的 core 和 memory。该设置让 container pool 过大直接转化为真实资源成本，而不是可通过超卖稀释的账面成本。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：静态方法由专家手工设置每个 container pool size，生产中为了保证 SLO 倾向 overprovision，带来高 idle container 成本；同时静态 container pool 和 reactive VM pool 的组合可能造成资源错配，即 VM pool 已耗尽但某些 container pool 仍有大量 idle container。Reactive 方法使用 allocation failure 作为 lagging signal，只有观察到 pool 被打空后才扩容，而 container / VM 创建延迟相对 burst 持续时间过长，新资源到位时 burst 中多数请求已经失败。Predictive 方法用 ARIMA、Theta、ETS、DeepAR、TFT、PatchTST、Chronos、Weighted Ensemble 等预测未来负载，但短时间粒度下平均误差很高，且单个预测值无法表达 burst 在预测窗口内的到达时序；即使用 perfect predictor 给出窗口内真实总量，如果映射到 pool size 时假设 constant / Poisson / concentrated load，也仍不能保证 100% success rate。
        - 论文的设计方法：DROPS 不预测未来某个窗口的请求数，而是从历史 trace 中直接统计“一个请求到达时，为了立即满足它，过去 creation-latency window 内还未补回来的资源消耗有多少”。具体地，对每个 consumption request，DROPS 取其前面一个最大 resource creation latency 长度的滑动窗口；对窗口内每个历史 consumption request，从 creation latency distribution 采样补货延迟，如果补货完成时间晚于当前请求，就说明该历史请求仍会占用 pool capacity，需要增加当前所需 pool size；同时统计窗口内 recycling events 并扣减所需 pool size。对所有请求重复后得到 pool size 样本分布，再计算 empirical CDF，形成 pool-size-to-success-rate mapping。
        - 方法如何对冲 Baseline 缺陷：相对 static 方法，DROPS 给出每个 pool 对应目标 SLO 的最小统计容量，避免人为保守放大 container pool。相对 reactive 方法，DROPS 离线或周期性地把历史 burst 和 creation latency 的组合风险纳入容量设置，因此在 burst 到达前已经准备好足够资源，而不是等失败后扩容。相对 predictive 方法，DROPS 不需要准确预测 burst 的具体时间和强度，也不需要把“预测窗口总量”转换成请求到达形状；它直接分析历史中 demand 与 supply latency 的相对关系，保留了 burst 时序对 pool size 的影响。
        - 关键 trade-off：DROPS 依赖历史 trace 与未来 workload 具有相近统计性质；如果 workload distribution 突变、新 runtime pool 上线、creation latency 分布改变，mapping 可能失效，因此系统每 4 小时用前一周 trace 重跑，并在 failure rate 或 resource utilization 异常时 on-demand 触发。DROPS 的输出是 success-rate SLO 下的容量选择，若目标接近 100%，尾部 burst 会显著推高成本；论文显示 99.9% success rate 所需 VM pool / .NET 8.0 pool size 分别约为 95% success rate 的 2.2 倍和 2.5 倍。另一个 trade-off 是 VM pool 与 container pool 的资源分配方式：DROPS 倾向保持更多 idle core 未绑定到具体 container pool，以便按需补充不同 runtime，但这要求平台能快速做 core-to-container allocation。

    3、论文实现：
        - Baseline 如何实现：静态 baseline 对 container pool 使用生产平台先前由专家配置的固定 pool size，对 VM pool 使用基于 idle CPU core 的 reactive scaling；当 idle core 低于 10% low watermark 时扩大 VM pool，高于 40% high watermark 时缩小 VM pool。Predictive baseline 为每个 container pool 选择表现最好的 forecasting model，再用 constant、Poisson 或 concentrated-load 三种方式把预测负载映射成 pool size。Reactive baseline 使用 multiplicative increase / additive decrease，即每次 container-allocation failure 后指数增加 pool size，在无失败期间线性下降。Predictive-reactive baseline 在每个预测区间开始时按预测负载调整 pool size，并在区间内部继续用 reactive 方法响应失败。
        - 新设计如何实现：DROPS 用 C# 实现，部署在 Azure Functions 全部 region 和 data center。系统每 4 小时运行一次 DROPS，输入为过去一周 trace；如果监控发现 allocation failure 超过目标 SLO 或资源利用异常，也会按需重跑。对 container pool，DROPS 直接使用该 pool 的 container-allocation trace 作为 demand trace，使用 container-creation latency distribution 作为 supply distribution，因为 Azure Functions 中 container 删除后不会回到 container pool，所以 recycling trace 为空。对 VM pool，DROPS 根据 container-allocation trace 和 container lifecycle distribution 生成 core-allocation trace 与 core-recycling trace：每个 container allocation 按 container size 映射为 core demand，并把 core-allocation event 向前平移一个采样的 container-creation latency；每个 container 的删除时间则由 creation、allocation、user workload、deletion 等 lifecycle stage 采样得到，删除释放的 core 构成 recycling trace。
        - 实验 / 实现平台：论文使用 Azure Functions 生产环境中一个 region 的 14 天 trace，时间为 2024 年 11 月 1 日到 11 月 14 日，包含 2.2M+ container-allocation requests，覆盖 16 个 runtime environments。trace 包括 container-allocation trace 和 container-lifecycle trace，后者记录 creation、allocation、user workload、deletion 等阶段耗时。预测模型训练使用一台包含两颗 Intel E5-2630 CPU、128GB RAM、480GB SSD 的机器，并通过 AutoGluon 统一训练 ARIMA、ETS、Theta、DeepAR、TFT、PatchTST、Chronos 等模型。系统评估使用自建 C# simulator 重放生产 trace，模拟 container 和 VM 生命周期，阶段延迟从真实经验分布采样；论文说明 simulator 已验证能忠实重放平台行为。
        - 关键实验设置与指标：主要指标包括 failure rate、cost 和 fulfillment latency。failure rate 指 allocation request 不能被 pre-warmed pool 立即满足的比例；这些请求不会被拒绝，但会产生额外创建延迟。cost 用 core-hours 计量，并拆分为 container pool、VM pool、platform overhead 和 user workload cost。评估采用 rolling 方式：用前 7 天优化当天 pool size，第二周用于评估。目标 SLO 为 100% 时，DROPS 和 static baseline 都达到 0% failure rate，但 DROPS 总成本比 static 低 41%；其中 idle container 成本下降 79%，idle core 成本上升 13%，反映其把资源更多保留在未绑定的 VM core pool 中。Reactive 方法因滞后扩容 failure rate 最高，且成本超过 DROPS 4.2 倍；predictive concentrated-load 映射虽然保守但仍不能满足 SLO，成本约为 DROPS 的 3.3 倍。跨 EastUS、EastUS2、WestUS2、NorthEurope 四个 Azure region 的一个月 trace 上，DROPS 与 static 都达到 0% failure rate，但 DROPS 成本降低 57% 到 80%。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：论文没有提出新的 GPU kernel、container runtime kernel 或函数执行内核；最接近的新 pipeline 是 DROPS 的 trace-driven resource-pool sizing pipeline。它不是在线请求处理路径中的调度器，而是周期性 / 触发式控制面流程：采集 container-allocation trace 和 lifecycle trace，构造 container pool 与 VM pool 的 demand / recycling / supply latency 输入，运行滑动窗口统计，输出每个 pool 的 pool-size-to-success-rate mapping，再把目标 SLO 对应的最小 pool size 下发到生产平台。论文还提出 aggressive container creation optimization：当 VM pool 中存在 idle core 时，用这些 core 额外创建 container 扩展各 container pool，但不触发新 VM 创建，从而不增加成本；idle core 按各 pool 达到 100% success rate 所需额外 container 数量成比例分配，并周期性重分配以避免某个 pool 缺资源而其他 pool 闲置。
        - 新pipeline/kernel的执行流例子：以 Python 3.11/1 container pool 为例，DROPS 先取过去一周该 pool 的 container-allocation requests，并读取该 runtime / size 的 container-creation latency distribution。对于某个历史请求 r，DROPS 回看 r 到达前一个最大创建延迟窗口内的所有 allocation requests；每个历史 request 都触发过一次 replenishment，DROPS 为它采样一个 creation latency，如果该 replenishment 在 r 到达时还没完成，就把 r 所需的 pool size 加一。由于 container 删除后不会回到同一个 container pool，该 pool 的 recycling trace 为空。所有请求处理完后，DROPS 得到一组“每个请求要立即成功所需的 pool size”样本，计算 CDF；如果目标 SLO 是 99.9%，就取 99.9 percentile 作为该 Python 3.11/1 pool 的新 size。对 VM pool，流程多一步：系统先把所有 container allocations 转为 core-allocation events，并根据 lifecycle 采样生成 core-recycling events；然后同样运行 DROPS 滑动窗口算法，得到 VM pool 的 size。生产平台应用这些 size 后，普通用户请求到来时仍按 Azure Functions 原路径从预热 container pool 取 container；若 pool 被消费，平台触发 replenishment。DROPS 的作用是在控制面预先设定 pool 的统计安全容量，使 burst 到达时大多数或全部请求能直接命中预热资源。
