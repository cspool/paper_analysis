    1、论文工作：
        - 论文要解决的核心问题：serverless inference workflow 由 GPU functions 和 CPU functions 共同组成，函数之间需要传递视频帧、检测结果、裁剪图像、KV cache 等中间数据。现有 serverless 数据平面主要通过 host-side storage 传递数据，gFn-gFn 也要走 GPU-host-GPU 的 PCIe 路径，论文在 INFless + host shared memory 的实验中测到 data passing 可占端到端延迟的 92%，其中 gFn-gFn 为 63%，gFn-host 为 29%。把存储搬到 GPU 上也不够，因为 NCCL / UCX / NVSHMEM 这类 GPU 通信库面向长生命周期 serverful 进程和 collective / point-to-point 通信，不理解 serverless 函数放置、容器 GPU 虚拟 ID、临时数据生命周期和动态 GPU 空闲显存，导致额外拷贝、链路带宽利用不足和 GPU 临时存储低效。
        - 论文的主要贡献：论文提出 GRouter，一个 GPU-centric serverless data plane。它的贡献包括：第一，统一 Put / Get 数据传递接口，抽象 gFn-gFn、cross-node gFn-gFn 和 gFn-host，并用函数物理位置和 GPU topology 消除不必要 relay copy；第二，跨 PCIe、NVLink 和 NIC 做 fine-grained bandwidth harvesting，把数据切成 chunk 后通过多路径并行传输，同时按函数 SLO 做带宽隔离；第三，在非对称 NVLink topology 中做 topology-aware transfer scheduling，为弱连接 GPU pair 选择 direct / indirect parallel NVLink paths；第四，实现 elastic GPU data storage，按请求间隔、数据大小和并发度动态扩缩 GPU memory pool，并根据下游函数队列进行 proactive data migration。
        - 论文所处背景：该工作处在 serverless computing、GPU-enabled serverless inference、serverless workflow data plane 和 GPU cluster communication 的交叉位置。目标应用不是单个模型 kernel 加速，而是云端多阶段推理工作流，例如 traffic monitoring、auto-driving image segmentation、video face recognition、image classification 和 mixture-of-agent LLM workflow。论文特别关注 serverless 的两个系统特征：函数以容器隔离并按需调度，函数位置动态变化；GPU 显存和互连链路在不同请求阶段存在可被临时利用的空闲资源。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：INFless+ 代表 host-centric 设计，用 host-side shared memory 做函数间数据交换，GPU 数据必须经 PCIe 到 host 再回 GPU，数据路径长且 PCIe-bound。NVSHMEM+ 代表 GPU-side storage baseline，能绕过 host memory，但存储服务和函数执行解耦，无法感知函数实际物理 GPU；受 GPU virtualization 和 CUDA IPC address mapping 限制，storage container 看到的是自己的本地 GPU 地址语义，不能正确按生产者 / 消费者位置放置数据，导致中间数据被随机放在远端 GPU 上并产生多次 GPU-GPU 拷贝。DeepPlan+ 增加 parallel PCIe transfer，但只优化 gFn-host，不处理 gFn-gFn locality，也缺少 topology-aware route selection，在 DGX-V100 这类非对称 NVLink 机器上可能选到连接弱的 route GPU。通用 GPU 通信库还通常只用本地单条 PCIe / NIC / NVLink，无法聚合 peer GPU 的空闲链路；GPU storage 方面，静态预留 memory pool 会造成论文实验中约 4x 的显存浪费，LRU eviction 又可能把即将被下游函数读取的数据赶到 host memory。
        - 论文的设计方法：GRouter 把数据平面放到 GPU 侧，但让控制面显式维护函数放置、数据 ID、数据位置、GPU topology 和链路带宽使用矩阵。Put() 时，系统识别调用函数所在 GPU，在本地 GPU memory pool 中存储输出并返回全局 data ID；Get() 时，根据消费者位置和数据位置选择零拷贝、本机 NVLink、cross-node GDR 或 parallel PCIe / NIC 路径。传输层将数据划分为默认 2 MB chunk，用多个 CUDA streams 按预先规划的 pipeline paths 并行传输；对于 PCIe / NIC，计算每个函数满足 SLO 所需的最低传输速率 Rate_least，并把剩余带宽优先给 SLO 最紧的函数；对于 NVLink，根据带宽使用矩阵先找空闲路径，再在必要时对 busy path 做 bandwidth balancing。存储层用 99th percentile 的 request interval、intermediate data size 和 concurrency 估计 memory reservation，空闲期保留最小 memory pool，并在显存压力下优先迁移队列尾部下游函数才会访问的数据。
        - 方法如何对冲 Baseline 缺陷：相对 INFless+，GRouter 避免 host memory 作为 gFn-gFn 的必经中转，same-GPU 数据可通过 CUDA IPC 映射直接访问，不同 GPU 数据按 NVLink / GDR 直接写入目标 GPU 地址。相对 NVSHMEM+，GRouter 的 storage 不是盲目选 GPU，而是按生产者和消费者 placement 维护 locality，因此每个数据对象在 GPU functions 间只需必要的一次跨 GPU 传输。相对 DeepPlan+，GRouter 不只借 PCIe，还把 NIC 和 NVLink 都纳入统一路径规划，并避免选择共享 PCIe switch 或弱 NVLink 连接造成的 contention。相对 LRU / 静态 GPU storage，GRouter 用 request queue-aware migration 保留即将访问的数据，并在显存恢复时主动 restore，减少 host-memory fallback。
        - 关键 trade-off：GRouter 用更复杂的控制面和数据平面换取数据传递延迟下降：系统需要维护全局 / 本地 data mapping、GPU topology、实时 bandwidth matrix、per-GPU I/O thread、daemon transfer thread 和 elastic memory pool。多路径传输引入 chunk 切分、stream 同步和批量传输调度；为了避免单 chunk 不可抢占造成新函数长时间等待，论文默认把 5 个 chunk 组成一个 transfer batch，在公平性和连接 / 调度开销之间折中。GPU storage 把中间数据留在 GPU 侧会占用显存，因此论文把存储使用限制在 free memory 的 50% 阈值内，并在显存紧张时迁移到 host memory 或 remote idle GPU；这能保护函数执行显存，但在极低可用显存下仍需要额外搬运。

    3、论文实现：
        - Baseline 如何实现：论文在 INFless 上扩展三类 baseline。INFless+ 在 host 侧加入 shared-memory storage，代表当前 serverless 系统中更高效的 host-centric 数据传递。NVSHMEM+ 使用 NVSHMEM 实现 GPU-side shared memory space，每个 data object 随机分配到某个 GPU storage 上，GPU functions 可直接 store / retrieve，但缺少函数位置感知。DeepPlan+ 在 NVSHMEM+ 基础上集成 DeepPlan 的 parallel PCIe 优化，能使用 GPU node 中多个 PCIe links 加速 host-gFn 数据传输，但其 parallel PCIe 由 storage service 处理，函数本身不可见其他 GPU 的 PCIe 资源。LLM MoA 场景中，论文还把 Mooncake 作为 KV cache-centric baseline 接到 serverless 系统中，称为 Mooncake+。
        - 新设计如何实现：GRouter 基于 INFless 实现，包含约 5K 行 C++ 代码，每个函数运行在按需分配 CPU / GPU 的容器中。host 侧为每个函数挂载 host volume，并用 shared memory region 做数据和消息交换；GPU 侧在每张 GPU 上维护 elastic memory pool。函数存取数据时，GRouter 从 local pool 分配内存，并通过 CUDA IPC 映射到函数地址空间。每张 GPU 运行 I/O thread 负责回收未使用内存，并根据可用空间在 GPU 与 host memory 间迁移数据；每张 GPU 还运行 daemon transfer thread，用多个 GPU streams 管理不同方向的并行传输，并按预生成的 PCIe / NIC / NVLink pipeline path 与其他 GPU thread 协作。调度层采用 hierarchical control plane，大多数传输和调度决策在节点内完成，跨节点只做较少的全局协调；节点内函数调度采用 MAPA 策略，并沿用 SHEPHERD 式 function / model pre-warming 来缓解 cold start。
        - 实验 / 实现平台：实验使用两个 AWS GPU testbed。DGX-V100 testbed 为 p3.16xlarge，每台 8 张 NVIDIA V100，NVLink 互连，Xeon E5-2686 v4 CPU，244 GB memory，4 个 100 Gbps NIC；DGX-A100 testbed 为 p4d.24xlarge，每台 8 张 NVIDIA A100，NVSwitch，Xeon Platinum 8275CL CPU，1152 GB memory，8 个 200 Gbps NIC。无 NVLink 适用性测试使用 4x A10 GPU server。真实 workflow 包括 Traffic、Driving、Video、Image 和 Mixture-of-Agent，前后处理用 NVIDIA CV-CUDA，输入数据来自 Adainf；请求到达用 Azure Function production trace，覆盖 sporadic、periodic 和 bursty 模式。LLM workflow 使用不同 stage 部署在分离的 8x H800 GPU nodes 上，节点内 200 GB/s NVLink，节点间 200 Gbps network。
        - 关键实验设置与指标：主要指标包括 pairwise data passing latency、P99 end-to-end latency、maximum throughput、ablation latency、SLO compliance、limited-memory tail latency、LLM receiver TTFT / TFTT、CPU overhead 和 GPU memory overhead。pairwise data passing 中，GRouter 在 intra-node gFn-gFn 上相比 INFless+、NVSHMEM+、DeepPlan+ 分别降低 95%、75%、75% latency；host-gFn 上分别降低 63%、63%、75%；inter-node gFn-gFn 上分别降低 91%、87%、87%。真实 workflow 上，DGX-V100 中 GRouter 相比 INFless+、NVSHMEM+、DeepPlan+ 的 P99 end-to-end latency 分别降低 61%、48%、54%，DGX-A100 中分别降低 53%、36%、30%。吞吐方面，同节点部署相比三类 baseline 分别提升 2.1x、1.74x、1.37x，跨节点分别提升 2.73x、1.55x、1.39x。component 消融显示，DGX-V100 上禁用全部优化会让 latency 增加 1.57x-1.82x，DGX-A100 上增加 1.30x-1.61x。显存受限实验中，在 10% GPU memory 可用于 storage 时，GRouter 相比 INFless+、LRU、RQ 分别降低 tail latency 46%、27%、7%；即使只有 1% 可用显存，仍分别降低端到端延迟 24%、14%、9%。MoA LLM workflow 中，4K input 下 GRouter 相比 INFless+ 和 Mooncake+ 分别降低 receiver TTFT / TFTT 66% 和 57%；不同模型与 tensor parallelism 设置下还分别报告 36% 和 28% 的 TTFT / TFTT 降低。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：论文没有提出新的 DNN operator kernel 或 attention kernel；最核心的新执行路径是 GRouter 的 GPU-centric serverless data passing pipeline。它由 Put / Get API、location-aware GPU storage、CUDA IPC / GPUDirect RDMA 直接映射和写入、parallel PCIe / NIC / NVLink transfer pipeline、SLO-aware bandwidth partitioning、topology-aware NVLink path selection，以及 elastic GPU storage migration 共同组成。可以把它理解为一个面向 serverless inference workflow 的“数据路由器”：它不是让函数自己调用 NCCL / NVSHMEM，而是由数据平面根据函数放置、数据位置、拓扑和 SLO 自动决定数据应当留在哪张 GPU、走哪几条链路、何时迁移。
        - 新pipeline/kernel的执行流例子：以 Traffic workflow 中 GPU 上的 YOLO 检测结果传给后续识别模型为例。上游 gFn 完成推理后调用 Put()，GRouter 识别该函数所在物理 GPU，把检测输出写入该 GPU 的 elastic memory pool，并生成全局 data ID；控制面在本地 mapping table 中记录 data ID、GPU device ID、memory address 和 workflow metadata，必要时同步到 global table。下游识别 gFn 启动并调用 Get(data ID) 后，GRouter 查询数据位置和消费者 GPU 位置：如果二者在同一 GPU，直接通过 CUDA IPC 映射地址实现 zero-copy；如果在同节点不同 GPU，优先选择 direct NVLink，若直连弱或繁忙，则根据 bandwidth matrix 选择多条空闲 / 可平衡的 indirect NVLink paths，把数据按路径带宽比例切成 chunk 并用多个 CUDA streams pipeline 传输到目标 GPU；如果跨节点，则先经本节点 route GPUs 和多 NIC 用 GDR 并行传到远端，再经远端 NVLink 聚合到目标 GPU；如果目标是 host 或数据从 host 来，则借助多个不共享瓶颈 PCIe switch 的 route GPUs 并行搬运。传输过程中，GRouter 按每个函数的 data size、offline-profiled inference latency 和 SLO 计算最低带宽需求，先保障各函数 Rate_least，再把 idle bandwidth 给 SLO 更紧的函数。若 GPU storage 触发 memory pressure，GRouter 检查下游函数队列，优先把队列尾部才会读取的数据迁到 host 或 remote idle GPU，保留马上要被 Get() 的数据；当空间恢复时再主动 restore，从而让即将执行的函数尽可能本地读取。
