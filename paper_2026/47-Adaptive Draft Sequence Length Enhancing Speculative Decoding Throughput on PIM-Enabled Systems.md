    1、论文工作：
        - 论文要解决的核心问题：LLM 解码阶段是自回归过程，常规 decoding 每轮只生成一个 token，吞吐受限。Speculative decoding 用轻量 Draft Language Model（DLM）先生成多个 draft tokens，再由 Target Language Model（TLM）并行验证，可以提高 token-level parallelism。但已有 PIM-enabled heterogeneous speculative decoding 系统通常使用固定 draft sequence length，在 batch size 增大时会产生大量被 TLM 拒绝的 draft tokens，使 DLM 生成和 TLM 验证都在无效 token 上浪费计算与访存，甚至在 batch size=64 时吞吐低于普通 autoregressive decoding baseline。
        - 论文的主要贡献：提出 SADDLE，一个面向 PIM+GPU 异构系统的 adaptive draft sequence length speculative decoding 系统。它包含三部分：基于累计接受概率的运行时 draft length 调整、DLM prediction 与 TLM verification 解耦的异步 pipeline、以及 arithmetic intensity-aware operator scheduler。论文报告 SADDLE 相比 GPU-only speculative decoding 平均吞吐提升 2.88x，相比最佳 GPU+PIM baseline 平均吞吐提升 1.71x。
        - 论文所处背景：DLM prediction 仍是逐 token 自回归，主要体现为 memory-intensive GEMV；TLM verification 可以一次验证多个 draft tokens，更多体现为 compute-intensive GEMM。已有 PIM+GPU 系统利用 PIM 承担 memory-bound operators、GPU/xPU 承担 compute-bound operators，但 SpecPIM 等系统依赖固定 draft length 和离线 operator mapping，难以适应 batch 内请求差异、不同任务语义、不同模型以及 decoding iteration 变化。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：论文重点对比 SpecPIM 类 PIM-enabled heterogeneous speculative decoding 系统以及 GPU-AD、GPU-SD、PIM-AD、PIM-SD。Baseline 问题有三类。第一，固定 draft length 与真实最优 draft length 不匹配：draft length 过长会降低 token acceptance rate，产生大量被丢弃 token；过短则增加 TLM verification 频率，降低 TLM 并行度。论文在 OPT-66B+OPT-1.3B、Dolly、batch size=64 上观察到 SpecPIM 在 d=4 时吞吐最高、比 autoregressive baseline 高 1.41x，但 d=8 时反而低于 baseline。第二，简单地对每个请求使用自适应 draft length 会让短请求等待长请求完成 DLM prediction，造成 micro-batch 内同步气泡。第三，draft length 和 effective batch size 改变 operator arithmetic intensity，使离线静态映射失效；例如 TLM attention 可能随 draft length 增大转向更适合 GPU，而 DLM FC 可能因 effective batch size 降低转向更适合 PIM。
        - 论文的设计方法：SADDLE 运行在由 host、SADDLE Manager 和多个 SADDLE PIM devices 组成的系统中。SADDLE Manager 负责 draft length 控制、Shared Pool / Eager Pool 管理和 operator scheduling；SADDLE PIM device 包含 centralized processor（论文实验中用 A100 GPU）、router 和多个 HBM-PIM chips。每个 HBM bank 附近配置 PE，buffer die 上集成 SFU 支持 softmax、layer norm、activation 等非矩阵操作。模型层按 pipeline stages 划分到 PIM device groups，batch 划分为 micro-batches 以占满 pipeline。
        - 方法如何对冲 Baseline 缺陷：自适应 draft length 通过 DLM 采样 token 的概率 p_i 维护累计概率 H_t=prod_i p_i，当 H_t 低于阈值 theta 时停止该请求继续 drafting，从而少生成低置信度、可能被 TLM 拒绝的 token。Shared Pool 跨 micro-batch 聚合 draft tokens，达到 GPU verification capacity C 或 GPU 空闲时触发 TLM 并行验证，避免每个 micro-batch 因最长 draft 请求而同步等待。Eager Pool 则允许 TLM 正在验证 Shared Pool 中 token 时，DLM 继续基于“当前 token 将被接受”的乐观假设生成后续 draft tokens；若旧 token 通过验证，Eager Pool token 迁入 Shared Pool，否则丢弃并用 TLM 修正 token 重新开始。Scheduler 根据 effective micro-batch size 和 Shared Pool 中 draft token 数估算 arithmetic intensity，把 DLM attention、TLM FC 等较稳定算子固定映射到适合设备，并动态决定 DLM FC 与 TLM attention 运行在 PIM 还是 GPU。
        - 关键 trade-off：SADDLE 接受了额外硬件控制逻辑、Shared Pool / Eager Pool / SRAM、SFU 和 PIM PE 面积开销，换取更高吞吐与更低无效计算。Eager Pool 的乐观执行可能在 TLM 拒绝旧 draft token 时丢弃新生成 token，但论文认为这部分可由更少 pipeline idle time 抵消。阈值 theta 需要离线用验证集校准，运行时可根据负载调节；这意味着效果依赖 workload 分布和 DLM/TLM 组合。论文评估主要基于 cycle-accurate simulation，而不是实机完整系统原型。

    3、论文实现：
        - Baseline 如何实现：GPU-AD 是在 GPU 上用 TLM 做 autoregressive decoding；GPU-SD 是在 GPU 上做 speculative decoding；两者基于 DeepSpeed Inference，在 8 个 NVIDIA A100 80GB GPU 的 DGX 系统上评估，总显存 640GB、聚合带宽 16TB/s。PIM-AD 使用 AttAcc 风格 HBM-PIM 架构，在 PIM 上执行 attention、GPU 上执行 FC。PIM-SD 采用 SpecPIM 的映射策略，在执行前根据初始 batch size 和 maximum sequence length 做 design-space exploration，得到静态 operator mapping。
        - 新设计如何实现：SADDLE 采用 8 个 SADDLE PIM devices；每个 device 有一个 A100 作为 centralized processor，并连接 5 个 HBM3 stacks，每 stack 16GB，总容量与 A100 80GB 对齐。PIM PE 沿用 PIM-AD / PIM-SD 中每个 HBM bank 附近一个 PE 的设计。Manager 配置 1KB Shared Pool、1KB Eager Pool，后者按 micro-batch 划分，每个 pool 最多存 512 tokens；另有 1KB SRAM 存储 logits 和累计接受概率。Controller 中包含 softmax unit、multipliers 和 comparators，用于低延迟更新 H_t 并比较阈值 theta。
        - 实验 / 实现平台：论文构建 cycle-accurate simulator，修改 Ramulator2 和 ATTACC，用于评估 GPU systems 与 SADDLE 的执行时间和能耗。面积和能耗开销通过 Synopsys Design Compiler 在 28nm technology node、1GHz 下综合 PE，并把面积缩放到 DRAM process；HBM energy 参考已有工作。实验使用 FP16。TLM/DLM 组合包括 Llama-3.1-70B-Instruct + Llama-3.2-1B-Instruct、OPT-66B + OPT-1.3B、OPT-175B + OPT-6.7B。数据集使用 Dolly instruction-following dataset。batch size 从 16 到 128，默认最大 sequence length 多数实验为 1024，OPT-175B 因内存限制为 512。
        - 关键实验设置与指标：主要指标为 normalized throughput、energy efficiency、latency breakdown、communication cost、GPU/PIM utilization、area overhead。整体结果显示，SADDLE 相比 GPU-AD、GPU-SD、PIM-AD、PIM-SD 平均吞吐分别提升 3.36x、2.88x、1.94x、1.71x；平均能效分别提升 6.81x、5.96x、2.32x、1.45x。消融显示，仅使用 adaptive draft length 的 SADDLE-d 反而比 PIM-SD 平均吞吐低 1.22x，说明自适应长度如果没有 pipeline 和调度配套会被气泡与映射失效吞掉收益；加入 Shared Pool 的 SADDLE-p 比 SADDLE-d 高 1.52x、比 PIM-SD 高 1.25x；加入 Eager Pool 再带来 1.24x；动态 operator scheduling 进一步带来 1.13x。论文还报告 SADDLE monitoring/decision-making 只占端到端 latency 的 0.83%，相对 PIM-SD 将 prediction 和 verification latency 分别降低 1.18x、1.23x，并通过异步重叠获得 1.73x 端到端 latency reduction。

    4、pipeline/kernel 解析：
        - 新 pipeline/kernel 是什么：本文的核心新执行路径是 SADDLE 的 prediction-verification decoupled asynchronous speculative decoding pipeline。它不是单个 GPU kernel，而是一个跨 DLM prediction、TLM verification、Shared Pool、Eager Pool、PIM/GPU operator scheduling 的系统级 pipeline。它同时改变 token 生成时机、verification batching 方式和 operator-to-device mapping。
        - 新 pipeline/kernel 的执行流例子：假设一个 batch 被切成多个 micro-batches，每个 micro-batch 有自己的 Draft Generator。某个请求进入 DLM prediction 后，Controller 每生成一个 draft token 就读取该 token 概率 p_t，并更新累计概率 H_t。如果 H_t 仍高于阈值 theta，请求继续生成 draft token；若低于 theta，则停止该请求 drafting。生成出的 draft tokens 先进入 Shared Pool。当 Shared Pool 中 token 数达到 GPU verification capacity C，或 GPU 空闲时，TLM 对 Shared Pool 中跨 micro-batch 聚合的 tokens 并行 verification。与此同时，DLM 不必等待 TLM 完成；对那些 H_t 仍高、且旧 draft tokens 正在验证的请求，DLM 继续生成后续 tokens，并暂存到 Eager Pool。TLM 返回结果后，如果某请求旧 draft tokens 全部被接受，Eager Pool 中该请求的新 tokens 被迁入 Shared Pool，进入下一轮 verification；如果某个旧 draft token 被拒绝，则该请求后续 Eager Pool tokens 全部作废，并以 TLM 修正 token 作为新的上下文继续 drafting。整个过程中 Scheduler 在 prediction 后根据仍活跃请求数估计 DLM FC arithmetic intensity，在 verification 前根据 Shared Pool 中每请求 token 数估计 TLM attention arithmetic intensity，并与预先刻画的 PIM/GPU 阈值比较，动态选择 PIM 或 GPU 执行。这样，一个请求不再被固定 draft length 和 micro-batch barrier 限死，PIM 与 GPU 也不再受离线静态映射限制。
