1、论文工作：
    - 论文要解决的核心问题：论文试图回答 serverless 为什么贵。作者指出，serverless 相比 VM 和容器托管服务不仅单位资源价格更高，还会因为 wall-clock allocation-based billing、初始化时间计费、固定 invocation fee、资源 / 时间 rounding up、keep-alive 资源占用和 OS 调度粒度等机制，让用户支付的 billable resources 显著高于真实消耗。论文在 Huawei serverless trace 上量化发现，当前模型下 billable vCPU time 平均可达到真实 CPU usage 的 1.01× 到 3.63×，billable memory 平均可达到真实 memory usage 的 1.57× 到 4.35×。
    - 论文的主要贡献：第一，系统整理并建模多个主流 public serverless 平台的计费模型，将成本分解为 billable wall-clock time、allocated / consumed resources、billing granularity / minimum cutoff 和 fixed invocation fee。第二，基于 558.74M Huawei serverless 请求、388,955 个可追踪 cold starts，以及跨 AWS Lambda、Google Cloud Run Functions、Azure Functions、IBM Cloud Code Engine、Cloudflare Workers 等平台的实测，揭示计费膨胀、并发模型、request serving architecture、keep-alive 和 OS scheduling 对成本的影响。第三，首次将 serverless 用户侧成本和 Linux CPU bandwidth control / cgroup 调度粒度联系起来，解释 fractional vCPU 场景下的 CPU overallocation、性能量化跳变和 billable cost 偏差。
    - 论文所处背景：serverless 被宣传为 fine-grained pay-per-use、自动扩缩容和免运维的云计算模型，但其底层仍需要 sandbox provisioning、isolation、request dispatch、runtime / control plane、keep-alive、CPU / memory 隔离和调度。早期 serverless characterization 工作关注架构、性能或资源管理，但论文认为这些工作没有覆盖最新平台的计费实践、serving architecture、keep-alive 行为和 OS scheduling granularity，因此无法解释用户账单中隐藏的系统性成本来源。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：论文中的 Baseline 不是单一系统，而是当前 public serverless 平台及既有 characterization / rightsizing 方法。其问题包括：资源控制旋钮与 billable resources 强绑定，CPU 常被嵌入 memory pricing 或通过固定 CPU-memory combo 间接计费；wall-clock allocation-based billing 会对 idle、I/O blocking 和低利用率阶段收费；cold start 初始化阶段消耗 provider 资源，但如果只按 execution time 计费会产生收入缺口；短函数在 invocation fee 和 100 ms / 1 ms 等 rounding granularity 下被 disproportionally 计费；multi-concurrency sandbox 如果未调优会因为 CPU、cache、memory 竞争让执行时间和账单同时上升；HTTP server 型 serving path 相比 API polling 或 code/binary execution 有额外 routing、queue、proxy、HTTP parsing 和 sidecar 开销；OS bandwidth control period、quota 和 timer tick 粒度较粗，会给短 serverless function 带来 CPU overallocation 和性能跳变。
    - 论文的设计方法：作者采用 top-down characterization 方法，从用户可见的 billing model 出发，向下分析 request serving architecture 和 keep-alive，再进一步进入 OS scheduling。具体方法包括：抽象统一成本公式；用 Huawei production trace 将同一批请求映射到 AWS、GCP、Azure、Huawei、Cloudflare 等代表性计费模型；用 cold start trace 比较初始化阶段资源消耗和后续 request execution 的 billable resources；在公有云上部署 minimal function、FunctionBench PyAES 等微基准测量 serving overhead、concurrency slowdown、keep-alive duration 和资源分配；设计用户态 scheduler profiling 函数，通过 CLOCK_MONOTONIC 时间跳变推断 throttle interval、throttle duration 和 obtained CPU time，并用本地 VM 的 CFS / EEVDF、不同 timer frequency 和 runC 容器实验对齐云平台行为。
    - 方法如何对冲 Baseline 缺陷：该方法不直接替换现有 serverless 平台，而是把账单膨胀拆到可解释的系统层级。对于计费层，它证明高成本不是单一 provider 定价策略，而是跨平台普遍存在的 allocation、rounding、invocation fee 与 cold-start compensation 组合。对于 serving 层，它区分 single-concurrency、multi-concurrency、API long polling、HTTP server、code/binary execution 等架构差异，解释为什么某些平台的短函数延迟和计费更高。对于 keep-alive，它指出 AWS / GCP 会冻结或缩减资源，而 Azure Consumption 可能保持完整资源，使 provider 成本、用户 latency 和潜在 background work 之间产生不同 trade-off。对于 OS 层，它把 fractional vCPU 的性能跳变归因到 cgroup CPU bandwidth control period / quota / tick frequency，并指出当前 invocation fee 和 rounding granularity 实际上也在吸收 under-accounted CPU overallocation。
    - 关键 trade-off：论文揭示的核心 trade-off 是 provider 成本回收、公平计费、低延迟和资源利用率之间的张力。按真实 usage 计费更公平，但 memory 等资源难以安全 overcommit，且 cold start、keep-alive、control-plane overhead 难以直接归属到单次请求。turnaround-time billing 能补偿初始化成本，但会增加冷启动用户账单。multi-concurrency 提高 sandbox 复用率，但在突发负载和慢扩容下会导致性能与成本双重惩罚。缩短 bandwidth control period 或提高 timer frequency 能减少 overallocation / overrun，却会带来更高调度中断和 accounting overhead。保持 keep-alive 能减少冷启动，但可能占用资源或保留容量。

3、论文实现：
    - Baseline 如何实现：计费 Baseline 来自各 public serverless 平台公开文档和价格模型，论文按 request-based billing、instance-based billing、execution-time billing、turnaround-time billing、allocation-based billing、usage-based billing、fixed invocation fee、billing granularity / cutoff 等维度建模。trace Baseline 使用 Huawei Public request tables，排除零 CPU usage 或缺失有效 pod ID / flavor 的请求后用于计费映射。serving Baseline 在 AWS Lambda、GCP、Azure、Cloudflare 等平台部署返回空字符串和 status code 的 minimal function，并对比 API long polling、HTTP server、code/binary execution 三类架构。调度 Baseline 则使用云平台的实际 fractional CPU 配置，以及本地 VM 中 CFS / EEVDF + cgroup CPU bandwidth control 的可控实验来反推平台参数。
    - 新设计如何实现：论文没有实现一个新的 serverless runtime 或调度器，主要实现的是测量与复现 artifact。官方仓库按 README 划分为 `billing`、`serving-architecture`、`os-scheduling` 和 `dataset` 目录，包含用于分析 billing models、serving architecture hidden cost、OS scheduling impact 的 notebooks、脚本和 released profiling datasets。论文的方法实现包括：统一成本公式；trace replay / billing mapping 脚本；concurrency benchmark；minimal function latency benchmark；keep-alive cold-start probability 测试；CPU profiling workload；以及用户态 scheduler profiler。调度 profiler 运行固定时长，通过监控 monotonic clock 中大于 500 微秒的跳变记录 throttle 事件，并计算 throttle interval、throttle duration 和 throttle 前获得的 CPU time。
    - 实验 / 实现平台：实验覆盖主流 public serverless 平台，包括 AWS Lambda、Google Cloud Run Functions、Azure Functions、IBM Cloud Code Engine、Cloudflare Workers，也分析 Huawei serverless traces。计费实验使用 558.74M 请求和 527.05M 个执行时间至少 1 ms 的请求；cold start 实验使用 388,955 个 traceable cold starts。concurrency 实验在 AWS Lambda 和 GCP 上部署 FunctionBench 的 PyAES compute-intensive function，按 1 vCPU 配置发送 120 s 短突发和 20 min 稳定流量。OS scheduling 实验在 AWS Lambda、GCP、IBM 上采样，并在本地 10 vCPU Intel Xeon E5-2673 v4 VM 上使用 Linux 6.2 CFS、Linux 6.8 EEVDF、250 Hz / 1000 Hz timer frequency、runC 容器进行对照。
    - 关键实验设置与指标：关键指标包括 billable vCPU time、billable memory GB-seconds、actual CPU / memory usage、resource utilization rate、cold start 初始化资源与后续请求资源差值、equivalent billable wall-clock time、rounded-up billable time / memory、execution duration、p95 / mean latency、cold start probability、keep-alive duration、CPU allocation during keep-alive、throttle interval、throttle duration、obtained CPU time、CPU bandwidth control period 和 scheduler tick frequency。重要结果包括：GCP 模型下 billable memory 平均可到真实 memory 的 4.35×；AWS 1 ms 粒度下 billable vCPU / memory 仍平均为真实消耗的 2.49× / 2.72×；42.1% cold starts 的初始化资源消耗大于或等于后续请求执行资源；AWS Lambda 128 MB 的 fixed invocation fee 等价于约 96 ms billable wall-clock time；GCP multi-concurrency 在短突发中 execution time 最多上升 9.65×；HTTP server 架构平均额外 latency 可到 5.93 ms；AWS keep-alive 约 300-360 s，GCP 常见约 900 s，Azure 约 120-360 s 且可能更长；推断出的 AWS / GCP / IBM bandwidth control period 分别约为 20 ms / 100 ms / 10 ms，scheduler tick frequency 分别约为 250 / 1000 / 250 Hz。

4、pipeline/kernel解析：
    - 新pipeline/kernel是什么：论文没有提出命名的新 pipeline、GPU kernel、operator 或 runtime datapath。最接近的新 pipeline 是一条“serverless cost demystification pipeline”：从 billing model 建模和 trace-based cost replay 开始，进入 request serving architecture benchmark 与 keep-alive resource allocation profiling，最后进入 OS scheduling granularity profiling 和本地 scheduler 对照验证。该 pipeline 的目标不是加速某个 kernel，而是把用户账单中的成本膨胀逐层归因到计费规则、服务架构和 OS 调度机制。
    - 新pipeline/kernel的执行流例子：以一个短 CPU-bound 函数请求为例，pipeline 首先读取平台价格文档和 billing rule，把该请求的 wall-clock execution / turnaround time、allocated memory、隐含或显式 vCPU、billing granularity、minimum cutoff 和 invocation fee 映射为 billable resources。然后在 trace replay 中比较 billable vCPU / memory 与真实 CPU / memory usage，判断 allocation-based billing 和 rounding 造成的膨胀。接着把同类函数部署到 AWS / GCP 等平台：如果是 AWS API long polling，请求进入 runtime API，被 sandbox 内 runtime blocking poll 到，再调用 handler 并返回；如果是 GCP / Azure HTTP server，请求经过 ingress / queue / proxy 后到用户 HTTP handler，额外 HTTP path 造成更高短函数 overhead；如果是 Cloudflare code/binary execution，则由语言 runtime / V8 或 Wasm 路径加载执行。随后 pipeline 在不同 idle interval 下检测 sandbox 是否 cold start，估计 keep-alive duration 和 keep-alive 中 CPU / memory 是否冻结、缩减或保持。最后，对于 fractional vCPU 配置，scheduler profiler 在函数内忙等并记录 monotonic clock 跳变；一次短任务如果 CPU time 小于 bandwidth control period 或在 tick accounting 延迟下 overrun，就可能获得超过 quota 的 CPU time，导致执行时间低于理想 reciprocal scaling，但也产生资源公平性和账单补偿问题。整条路径把同一个请求从“用户看到的账单”追踪到“serving path”和“Linux 调度粒度”。
