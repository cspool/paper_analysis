1、论文工作：
    - 论文要解决的核心问题：移动端 LLM 能保护用户隐私、降低云端延迟和成本，但模型权重被放在用户设备上，越狱用户、恶意应用或被攻陷的 REE OS 都可能窃取专有模型。仅加密模型文件不够，因为 inference 过程中明文参数、activation 和 KV cache 会出现在内存中。直觉方案是把 LLM inference 放进 Arm TrustZone 的 TEE，但传统 TEE 面向轻量 TA，难以高效提供 LLM 所需的大容量内存和 NPU 资源。论文聚焦两个瓶颈：第一，TEE 中静态预留大 secure memory 会浪费移动设备内存，而按需扩展 secure memory 又会显著拉长 TTFT；第二，REE 应用也需要 NPU，TEE 和 REE 缺少低开销、低 TCB、可验证的 NPU time-sharing 机制。
    - 论文的主要贡献：TZ-LLM 是一个用 Arm TrustZone 保护 on-device LLM 机密性的系统。它提出 pipelined restoration，利用 LLM DAG inference 的确定性参数访问顺序，在 prefill 计算期间并行做 secure memory allocation、模型参数 I/O 和解密，把 restoration latency 尽量隐藏到 computation latency 下；并提出 co-driver NPU 设计，把完整 NPU driver 的 control plane 留在不可信 REE，把很小的 data plane driver 放进 TEE，用 shadow job、TrustZone 硬件配置和序号校验完成安全 NPU job 执行。论文还实现了 partial parameter caching、pipeline-aware secure memory management、TA 多线程 shadow thread、framework initialized-state checkpoint 等工程机制。
    - 论文所处背景：该工作位于 mobile LLM inference、TEE / Arm TrustZone、模型机密性保护、NPU accelerator sharing 和系统安全的交叉点。它不同于 TEE-shielded model partition 只保护部分参数或把部分参数 offload 到 REE accelerator，也不同于 SecDeep 这类把 accelerator 静态配置给 TEE 的方案；TZ-LLM 的目标是端到端保护 LLM 参数，同时保持移动系统中 REE 应用继续使用 NPU 的兼容性，并避免长期占用数 GB 级 secure memory。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：TEE strawman 的 cold start 路径需要启动 llama.cpp、初始化 metadata 和 tokenizer、用 CMA 分配连续物理内存、从 flash 加载加密模型、解密参数，然后用 CPU 做 inference。论文在 8-bit Llama-3-8B、512-token prompt 上给出的例子中，参数 CMA allocation 约 4.182s，parameter load 约 4.054s，decryption 约 0.892s，相关 overhead 可使 TTFT 增加 11.6s；CPU prefill 更达到 164s 量级。另一类 warm start / static partitioning 能降低启动时间，但要长期预留约 8GB 参数内存，在 24GB 以内的移动设备上内存效率差。NPU baseline 方面，直接在 TEE 和 REE 各放一个完整 NPU driver 会导致 Rockchip NPU detach-attach 约 32ms world-switch overhead，并把约 60K LoC、依赖 Linux device / memory / interrupt / power management 子系统的驱动代码纳入 TEE TCB。S2PT page-grained secure memory 虽然避免连续内存分配，但会给 REE 应用带来持续 stage-2 translation overhead，且不能单独防 DMA 攻击。
    - 论文的设计方法：对内存问题，TZ-LLM 把 LLM computation graph 扩展为 computation operator 前插入 allocation、loading、decryption 三类 restoration operator 的 pipeline。I/O 和 NPU 按拓扑顺序调度，CPU 上用 priority-based greedy policy：若 CPU computation operator ready 则优先执行，否则执行与最早 computation operator 相关的 restoration operator。为减少 operator 时间错配引起的 pipeline bubble，allocation 和 decryption 被切成 micro-operator，允许 preemptive scheduling。为消除 pipeline 起始 bubble，系统在 REE memory pressure 允许时缓存早期 prefill operator 的参数，并在 inference 后按 reverse topological order 逐步释放。对 NPU 问题，TZ-LLM 使用 co-driver：REE driver 负责 scheduling、power/frequency 等 control plane；TEE user-mode data plane driver 只负责 secure job context、MMIO launch、interrupt completion，并通过 TZPC/TZASC/GIC 切换 NPU secure mode。
    - 方法如何对冲 Baseline 缺陷：pipelined restoration 把原本串行的 CMA allocation、flash I/O、decryption 和 prefill 计算改为可重叠路径，使按需 secure memory scaling 不再完全暴露在 TTFT 上；partial parameter caching 只缓存最早会用到的参数，保留冷启动优化收益，又不会像 static partitioning 那样长期锁住完整模型内存。pipeline-aware extend/shrink 接口利用 LLM 参数按拓扑顺序加载、逆序释放的 first-in-last-out 模式，保证 TZASC 所需的连续 secure memory。co-driver 让 REE 继续掌握复杂控制逻辑和统一调度，TEE 只验证并执行最小 data plane，从而避免完整驱动进入 TCB，也避免每次切换都重建 control plane。shadow job 使 REE scheduler 能把 secure job 放入统一队列，而 TEE 通过 job initialized / not issued 状态和 monotonic sequence number 防止任意启动、重放和重排序攻击。
    - 关键 trade-off：TZ-LLM 牺牲了一定泛化性和工程简单性换取安全与性能并存。pipelined restoration 依赖 LLM inference 参数访问顺序足够确定；对 MoE 或 early-exit 这类非确定 workload，系统可能预取本次未使用的专家或层。TrustZone TZASC 要求连续物理内存，所以仍会受到 CMA allocation 和内存碎片影响，只是开销变成 transient 并可与 I/O/compute 重叠。co-driver 依赖具体 NPU driver 可分离出小 data plane，论文在 Rockchip 上实现，并只调查了 Qualcomm 开源 NPU driver 的可行性，未做跨平台完整验证。安全模型不覆盖物理 DRAM 攻击、side channel、cryptographic attack 和 DoS；参数 tensor size 与 secure NPU job execution time 也会向 REE 泄漏模型结构级信息。

3、论文实现：
    - Baseline 如何实现：REE-LLM-Memory 是未修改 llama.cpp 在 REE 中运行，全部参数预加载在内存里，代表无保护、内存低效但接近理论最佳性能的 baseline。REE-LLM-Flash 是未修改 llama.cpp 在 REE 中运行，在 inference start 时使用 pipelined restoration 加载模型参数，采用 buddy system allocation 且不解密，代表实用但无保护的 REE baseline。Strawman 是论文第 2.3 节的 TEE cold start：每次请求都在 TEE 中冷启动、动态扩展 secure memory、加载/解密参数，并用 CPU 做计算，提供安全和内存效率，但缺少 pipelined restoration 和 TEE NPU support。
    - 新设计如何实现：TZ-LLM 原型基于 OpenHarmony OS 及其 TEE 系统，REE OS 为 OpenHarmony v4.1 / Linux v5.10，LLM TA 基于 llama.cpp，NPU driver 基于 Rockchip NPU driver v0.9.8。TEE OS 原本约 17K LoC，论文只扩展约 62 LoC 支持 CMA page memory mapping，约 50 LoC 支持动态 TZASC/TZPC 配置。llama.cpp 扩展约 1.2K LoC 实现 pipelined restoration，约 1K LoC 集成 NPU data plane driver，并用 OpenSSL 做参数解密；computation graph 通过 llama.cpp 内部接口提取。REE Linux kernel 侧新增约 364 LoC，其中 NPU driver 增加 167 LoC 做 shadow job scheduling，TZ driver 增加 197 LoC 做 CMA allocation/deallocation。TEE NPU driver 被放在 user mode，TEE OS 只映射 NPU MMIO，并用 TZASC 限制 NPU 只能访问 secure job execution context 相关区域。
    - 实验 / 实现平台：测试平台为 Orange Pi 5 Plus，SoC 为 RK3588，包含 4 个 Cortex-A76 2.4GHz、4 个 Cortex-A55 1.8GHz、16GB LPDDR4X、1TB NVMe SSD 和 3-core NPU，NPU 峰值约 6 TOPS。模型包括 TinyLlama-1.1B、Qwen2.5-3B、Phi-3-3.8B 和 Llama-3-8B，均为 8-bit quantized，参数大小约为 1.0GB、3.3GB、3.7GB、7.9GB。LLM TA 使用 4 个 Cortex-A76 CPU core 和全部 3 个 NPU core。为触发 CMA memory migration，实验用 stress-ng 模拟 REE memory pressure，四个模型对应压力分别为 13GB、11GB、10GB、6GB，并将 stress threads 与 LLM threads pin 到不同 CPU core。benchmark 使用 UltraChat、PersonaChat 和 DroidTask，另用 YOLOv5 / MobileNet 评估 NPU time-sharing 对 REE NN 应用的影响。
    - 关键实验设置与指标：核心指标包括 TTFT、decoding speed、pipeline critical path overhead、不同 cache proportion 下 TTFT、NPU sharing throughput，以及 CMA allocation 对 REE Geekbench 应用的干扰。相比 strawman，TZ-LLM 在固定 prompt length 下 TTFT 降低 77.1%-91.1%，在 real-world benchmark 上降低 76.1%-90.9%；相比 REE-LLM-Flash，TZ-LLM 在 TTFT 上平均有 5.2%-28.3% overhead，主要来自 CMA allocation 和 parameter decryption；相比 REE-LLM-Memory，短 prompt 的 overhead 可达 2.5x-3.7x，但 PersonaChat / DroidTask 等较长 prompt 下 overhead 为 8.1%-21.2%。decoding 阶段相比 strawman 提升 0.9%-23.2%，相比 REE-LLM 降低 1.3%-4.9%，overhead 来自 TEE/REE NPU driver 通信。pipeline scheduling 在 memory stress 下距理论 critical path lower bound 只有 0.01%-9.9% overhead；preemptive scheduling 进一步降低 TTFT 最多 16.2%；partial parameter caching 在达到阈值前近似线性降低 TTFT。NPU time-sharing 相比 REE 内部 sharing 对 NN 应用和 LLM 额外 slowdown 分别最高 3.8% 和 3.0%；CMA 对并发 Geekbench 的性能下降最高约 6.7%。

4、pipeline/kernel解析：
    - 新pipeline/kernel是什么：论文没有提出新的矩阵乘 kernel 或 attention kernel；它的新机制是 TEE 内 LLM inference 的 pipelined parameter restoration pipeline，以及 TEE-REE co-driver NPU job execution path。前者把模型参数从 flash 到 secure memory 的过程拆成 allocation / loading / decryption / computation 的可重叠流水线，配合 priority-based + preemptive CPU scheduling 和 partial parameter caching；后者把一次 NPU job 拆成 REE control plane scheduling 与 TEE data plane secure launch，通过 shadow job 触发 NPU world switch，并用 TZPC/TZASC/GIC 建立 secure execution environment。
    - 新pipeline/kernel的执行流例子：以 Llama-3-8B 在 TEE 中处理 512-token prompt 为例。系统启动请求后，LLM TA 不再一次性等待 7.9GB 参数全部分配、读取和解密完成，而是根据 llama.cpp 提取的 computation graph 拓扑顺序，先为最早 prefill operator 所需参数调用 extend_allocated，从 REE Linux CMA 中拿到连续物理内存；REE file system 直接把加密参数异步读入这段尚未 TZASC-protected 的 allocated memory，避免 bounce buffer；随后 TA 调用 extend_protected，TEE OS 扩展 TZASC region，把该内存映射进 TA，再解密参数。只要第一个 computation operator 的参数 ready，就开始 CPU/NPU prefill，同时后续 operator 的 allocation、I/O、decryption 在后台继续推进。如果 CPU computation ready，scheduler 抢占长 allocation/decryption micro-operator 先跑 computation，减少 NPU/CPU 等待。若上一轮 inference 后内存压力不高，早期参数仍缓存于 secure memory，本轮可直接从这些 operator 开始计算，起始 bubble 被消除。
    - 新pipeline/kernel的执行流例子：对于一个 secure NPU matrix computation job，LLM TA 在 TEE 中准备 secure execution context，包括 command/register sequence、I/O page table、input/output buffer，并向 REE driver 提交 paired shadow job。REE NPU driver 仍用自己的统一队列调度 REE NN job 与 TEE shadow job；当 shadow job 被选中时，REE driver 通过 smc 把控制交给 TEE data plane driver。TEE driver 先用 TZPC 阻止 REE 访问 NPU MMIO，并配置 GIC 将 NPU interrupt 路由到 TEE；然后等待已有 non-secure NPU job 完成，再通过 TZASC 允许 NPU 访问 secure memory。TEE driver 校验该 secure job 已初始化、未发行，且 sequence number 符合当前顺序，随后写 MMIO launch NPU job。secure interrupt 到达后，TEE driver 收尾并把 NPU 切回 non-secure mode，REE driver 标记 shadow job 完成并继续调度下一个 job。这个路径的关键是：REE 可以调度但不能伪造或窥探 secure job，TEE 可以执行和验证但不承担完整 NPU control plane。
