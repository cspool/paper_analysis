论文标题：AdaServe: Accelerating Multi-SLO LLM Serving with SLO-Customized Speculative Decoding

LLM的不同应用SLO需求的Serving框架优化。

1、论文工作：
    - 论文要解决的核心问题：现代 LLM serving 同时承载 coding copilot、chatbot、summarization 等不同应用，这些应用的 TPOT / per-token latency SLO 不同。现有 serving 系统大多用 uniform batching / continuous batching，让同一 batch 中所有请求经历近似统一的 per-token latency，无法对严格 SLO 请求分配更快的解码速度，也无法在满足严格请求的同时保持系统 goodput。
    - 论文的主要贡献：AdaServe 把 multi-SLO serving 形式化为带预算约束的 token tree 构造问题：在每次 decoding iteration 中，给定硬件可验证 token budget，为不同请求构造 speculation token tree，使每个请求的期望 accepted tokens 满足其 TPOT 约束，同时最大化总体 accepted tokens。论文提出 SLO-customized speculative decoding，并实现 speculate-select-verify pipeline：先用小 draft model 生成候选 token tree，再按 SLO 需求和吞吐目标选择要提交给大模型验证的节点，最后用 LLM 并行验证。系统还根据活跃请求数动态调节 speculation depth 和 beam width。
    - 论文所处背景：LLM decoding 的传统自回归路径每轮只产出一个 token，加载大模型权重但计算利用率不足；continuous batching 用更大 batch 提高 GPU 利用率，但会拉高 per-token latency。Speculative decoding 用小模型提前猜测多个 token、再由大模型一次性验证，能把 under-utilized compute 转化为多 token 进展。AdaServe 的切入点不是单请求最快，而是在多 SLO 混部场景下把 speculative decoding 变成细粒度资源调度机制。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：vLLM、TensorRT-LLM、Sarathi-Serve 等 continuous batching 系统把请求同质化处理，iteration-level scheduling 造成 batch 内请求 per-token latency 近似一致；当严格 SLO 请求增多时，要么缩小 batch 牺牲吞吐，要么保持大 batch 导致 latency violation。vLLM + Priority 这类优先级策略通过限制 batch 或抢占非紧急请求照顾严格请求，但论文指出这会进一步损害整体 SLO attainment。vLLM-Spec 和 SpecInfer 虽然使用 speculative decoding，但策略静态、缺少 per-request SLO aware allocation，不能随请求分布和系统负载动态调节。
    - 论文的设计方法：AdaServe 用 token budget 表示当前硬件在一次 verification 中可承载的 token 数，先定义每个请求当前至少需要接受多少 token 才能追上 TPOT SLO，再用 path probability 近似一个 token tree 节点被 LLM 接受的概率。理论算法分两步：先为每个请求贪心选择最高 path probability 节点直到满足 SLO 约束；再把剩余 budget 分配给全局 path probability 最高的节点以提升吞吐。实际系统中，由于真实 path probability 无法预先知道，AdaServe 用 draft model logits 估计；由于逐节点 GetTop 会导致高 speculation overhead，AdaServe 把构造拆成 speculation phase 和 selection phase，用 beam search 先生成有限候选树，再在候选树上选择节点。
    - 方法如何对冲 Baseline 缺陷：continuous batching 的问题是“所有请求同速前进”；AdaServe 则允许同一 batch 中不同请求在一次 SD iteration 中接受不同数量的 token。严格 SLO 请求会在 SLO-customized selection 中优先获得足够高概率节点，宽松 SLO 请求则不必浪费验证预算。相对静态 speculative decoding，AdaServe 根据请求当前 latency、已生成 token 数、SLO 阈值和活跃请求数动态调节 tree depth / width 与 token allocation，使 speculation aggressiveness 随负载变化。
    - 关键 trade-off：更深、更宽的候选树提高命中多个 token 的机会，但会增加 draft model decoding 开销，并产生更多最终被丢弃的候选 token；更保守的 speculation 降低开销但限制加速空间。AdaServe 接受一个期望约束近似：用 draft logits 近似 path probability，用 expected accepted tokens 代替真实接受 token 数；这让调度可在线执行，但效果依赖 draft model 与 verifier 分布的一致性。系统还设置 per-request token limit，避免某个严格请求吞掉过多低概率 budget，但这也意味着极端紧 SLO 可能只能被尽量推进而非完全满足。

3、论文实现：
    - Baseline 如何实现：实验比较 vLLM、Sarathi-Serve、vLLM-Spec(n) 和 SpecInfer。vLLM 代表 PagedAttention + continuous batching；Sarathi-Serve 代表 chunked prefill 与 decode/prefill co-batching；vLLM-Spec(n) 是基于 vLLM 的固定 speculation length 序列式 speculative decoding，n 取 4、6、8；SpecInfer 是原生集成 speculative decoding 的推理引擎。论文还在动机图中讨论 vLLM + Priority、vLLM + Chunked prefill、FastServe、VTC 等策略，作为现有多 SLO serving 难点的对照。
    - 新设计如何实现：AdaServe 构建在 FlexFlow Serve 之上，request manager 维护 active request pool，并由 SLO-customized scheduler 执行 speculation、SLO-customized selection、throughput-optimized selection 与 verification。execution engine 在 GPU 上运行 draft model 和 target LLM。系统集成 FlashInfer 的 batched prefill kernel，并将其改造用于 speculation steps 和 LLM verification。AdaServe 还利用 CUDA Graph 优化 draft model decoding：从第二个 speculation step 到第 d 个 step，若活跃请求数相同，每步形状一致，可以复用预捕获 CUDA graph，减少 kernel launch overhead。
    - 实验 / 实现平台：评测节点为 4 张 NVIDIA A100 80GB GPU，通过 NVLink 互连；CPU 为 AMD EPYC 7763，64 cores / 128 threads，256GB DRAM。模型包括 Llama3.1-70B-Instruct，使用 4-way tensor parallelism 和 4×A100 80G；Qwen2.5-32B-Instruct，使用 2-way tensor parallelism 和 2×A100 80G。draft model 与 base model colocate 在其中一张 GPU 上，Llama 使用 Llama-3.2-1B-Instruct，Qwen 使用 Qwen2.5-0.5B-Instruct，且不做任务特定定制。Artifact Appendix 说明复现实验推荐 8×A100-SXM4-40GB 或 AWS p4de.24xlarge，CUDA 12.4 与 Docker + NVIDIA container runtime。
    - 关键实验设置与指标：工作负载由三类请求组成：coding copilot 使用 HumanEval，SLO 为 1.2× baseline latency；chatbot 使用 Alpaca，SLO 为 50ms/token；summarization 使用 CNN/DailyMail，SLO 为 150ms/token。请求到达时间来自真实 trace，经截断与缩放形成不同 RPS。主要指标是 SLO attainment，即满足 TPOT SLO 的请求比例；goodput，即成功满足 SLO 的请求产生的 tokens/s。论文明确排除 TTFT，因为 AdaServe 的目标是 decoding speed SLO 而非 prefill latency。
    - 主要结果依据：在不同 RPS 下，AdaServe 相对最佳 baseline 的 SLO attainment 提升可达 2.1× / 1.6×，未满足 SLO 请求数最多减少 4.3× / 3.2×，goodput 最高提升 1.9× / 1.7×。在改变严格请求比例时，AdaServe violation 最多减少 4.3× / 3.7×，goodput 最高提升 30% / 64%。在改变 SLO scale 时，AdaServe violation 最多降低 4.61× / 3.05×，goodput 最高提升 1.38×。Ablation 显示 SLO-customized token selection 是主要增益来源之一，throughput-optimized selection 和 CUDA Graph 进一步提升 SLO attainment；selection CPU overhead 只占总体 serving time 的 0.41% / 0.31%。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：论文的核心新执行路径是 SLO-customized speculative decoding 的 speculate-select-verify pipeline。它不是提出一个新的单一 GPU kernel，而是提出一种 serving runtime pipeline：用 draft model 的 beam-search speculation 生成候选 token tree，用 CPU scheduler 按 SLO 与 throughput 选择 token tree 节点，再用 target LLM 的 tree-based verification 并行验证。底层实现上，AdaServe 复用并改造 FlashInfer batched prefill kernel 支撑 speculation 与 verification，并用 CUDA Graph 降低重复 draft decoding steps 的 launch overhead。
    - 新 pipeline/kernel 的执行流例子：假设 batch 中有两个请求，一个 coding copilot SLO 很紧，另一个 summarization SLO 较宽。一次 decoding iteration 开始时，scheduler 从 request pool 取出活跃请求，根据每个请求当前已花费 latency、已输出 token 数和 TPOT SLO 计算它在本轮至少需要的 expected accepted tokens。随后 execution engine 用 draft model 做 d 步 beam search；每个请求的 root 是它最后生成的 token 或 prompt，第一步从 root 生成候选 child tokens，后续每步对上一层选出的 w 个 token 并行扩展，形成有限 candidate token tree，并记录由 draft logits 近似得到的 path probability。
    - SLO-customized selection 过程：scheduler 先处理更“落后”或需求更高的请求，在该请求候选树中按 path probability 从高到低加入节点，直到累计概率达到该请求本轮 SLO 所需阈值，或达到 per-request token limit / 当前可达上限。严格 SLO 请求通常获得更多高概率节点，因此一次 verification 后有机会接受多个 token；宽松 SLO 请求可能只获得少量节点，避免浪费预算。
    - Throughput-optimized selection 过程：如果满足各请求 SLO 推进目标后仍有剩余 token budget，AdaServe 把所有请求剩余候选节点放在一起按 path probability 全局排序，选择最高的若干节点补入各自 draft token tree，以最大化总 expected accepted tokens。这一步是吞吐优化，区别于前一步的 SLO 优先。
    - Verification 与状态回写：最终，所有请求的 selected draft token trees 被提交给 target LLM 做 tree-based verification。LLM 并行验证共享前缀的多条 speculative path，接受与目标分布一致的 token，拒绝错误分支并给出 correction token。verification 后返回 logits / verified tokens 给 scheduler；accepted tokens 被写回 request pool，完成的请求组装输出，未完成请求进入下一轮。这样，同一个 batch 中不同请求可以在同一次大模型验证中前进不同 token 数，从而打破 continuous batching 的统一 per-token latency 限制。
