论文标题：Scaling LLM Test-Time Compute with Mobile NPU on Smartphones

TTC在decoding时需要计算多个branch，多个branch的GEMV能充满NPU的GEMM Core。

1、论文工作：
    - 论文要解决的核心问题：移动端 LLM 面临“小模型能力不足、大模型资源太贵”的两难。传统思路是增大模型参数，但手机端内存容量、带宽、功耗和 NPU 地址空间都会快速成为瓶颈。论文观察到，Qualcomm Hexagon NPU 这类移动 NPU 在 decode 阶段存在被浪费的矩阵计算能力：自回归生成时每步通常只处理 1 个 token，线性层 GEMM 退化成 GEMV，而 HMX matrix unit 的 FP16 基本 tile 是 32×32，单 token 只填充了激活 tile 的 1 行，其余 31 行没有有效计算。并行 test-time scaling 正好需要在 decode 时生成多个候选路径，扩大 batch size，因此可以把原本空置的 HMX tile 行利用起来，提高小模型生成质量，而不线性增加矩阵乘法延迟。
    - 论文的主要贡献：论文提出一个面向 Qualcomm Snapdragon / Hexagon NPU 的端到端 LLM inference system，用移动 NPU 支持 Best-of-N 和 Beam Search 等 parallel test-time scaling。核心贡献包括：第一，分析 Hexagon NPU 的 “HVX vector + HMX matrix” 架构，指出 decode 阶段 HMX underutilization 是移动端 test-time scaling 的机会；第二，提出 hardware-aware fine-grained tile quantization，在 HMX 期望的 tile layout 上做 4-bit group quantization，并通过 quantization group coalescing 提高 HVX 宽向量寄存器利用率；第三，用 LUT-based computation 替换 Softmax 中的 exp 和 mixed-precision GEMM 中的 dequantization / scale broadcast；第四，将这些算子集成到 llama.cpp 的 Hexagon NPU backend，实机证明小模型 + test-time scaling 可以达到或超过大模型 base decoding 的 accuracy-latency / accuracy-energy Pareto frontier。
    - 论文所处背景：该工作位于 on-device LLM inference、mobile NPU programming、quantized LLM、test-time scaling 和 heterogeneous edge runtime 的交叉方向。它不同于传统 NPU LLM 工作只加速 prefill 或单路径 decode，而是把 test-time scaling 的“多候选生成”看成一种可利用 NPU tile 空洞的 workload 形态。应用场景主要是手机上运行 1B-3B 级 Qwen / Llama 小模型，在数学推理等适合 Best-of-N、Self-Consistency、Beam Search、PRM/ORM 验证的任务上提升输出质量。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：模型扩容 baseline 直接运行 3B/7B 模型，但在手机上带来更高显存/内存占用、带宽需求、功耗和延迟；小模型单路径 baseline 成本低但数学推理准确率不足。系统 baseline 方面，QNN 是闭源固定图框架，低比特主要支持 per-tensor / per-channel 等粗粒度量化，论文中 Llama3.2-1B-Instruct 的 QNN W4A16 在 MATH500 / GSM8K 上准确率从 AutoAWQ 的 15.9 / 32.6 降到 2.1 / 3.4，无法支撑 test-time scaling；llm.npu 不在 decode 阶段利用 NPU；PowerServe 等 QNN-based 系统受精度或闭源图限制；llama.cpp OpenCL backend 可用 Adreno GPU，但在大 batch test-time scaling 下不能利用 Hexagon HMX 的矩阵 tile 空洞。
    - 论文的设计方法：论文将权重量化和算子实现都围绕 Hexagon NPU 的物理数据路径重排。权重先转换到 HMX FP16 tile 所需的层次化 layout：tile 级 column-major，tile 内每两行做特殊 permutation；随后在这个 memory order 上做 group size 32 的 4-bit 量化，相当于以 2×16 tile 片段为量化组。为适配 HVX 的 128-byte / 1024-bit vector register，系统把 8 个 32 元素的 INT4 quantization groups 合成一个 super-group，使 256 个 INT4 值正好占满一个 vector register，并将 scale 聚集到连续位置。对复杂操作，系统用 HVX LUT 指令替代 exp、INT4-to-FP16 conversion 和 scale broadcast：Softmax 使用 64KiB FP16 exp LUT，只存 x<=0 的 safe-softmax 输入范围；dequantization 用 vlut16 直接把 4-bit 值映射到 FP16 表示，并用 LUT 广播多个 group 的 scale。
    - 方法如何对冲 Baseline 缺陷：相对直接增大模型，论文把“更多 test-time compute”转化为 decode batch parallelism，用空闲 HMX tile 行做多候选生成，从而让 1B/1.5B 小模型在数学任务上接近或超过 3B/7B base 模型，同时保持更低延迟/能耗。相对 QNN 粗粒度量化，tile-group quantization 保留 fine-grained group quantization 的精度，又把 layout 改成 HMX/HVX 可高效访问的形式。相对常规 group layout，HMX layout 消除 runtime scatter 到 TCM 的大量开销；group coalescing 让 HVX 一次向量操作处理足够多 INT4 数据。相对 polynomial exp 和常规 mask-unpack-convert，LUT-based Softmax / dequantization 减少 VLIW 顺序依赖和指令数，让弱通用计算单元不再拖慢 HMX。
    - 关键 trade-off：论文用更强的硬件相关性换取移动端可用性能：实现依赖 Qualcomm Hexagon SDK、FastRPC、dmabuf/rpcmem、HMX/HVX 指令和部分 reverse-engineered FP16 HMX 行为，移植到其他 NPU 需要重新适配 tile layout 和指令。使用 FP16 HMX 做 dequantized GEMM 避开了低精度 HMX 对 fine-grained quantization 的限制，但仍有 runtime dequantization overhead，论文也承认当前 decode speed 受此限制。系统为了规避 Hexagon NPU 32-bit virtual address space，将 lm_head / logits 放在 CPU 上，batch size 到 16 时 CPU logits 计算可接近或超过 50% 时间，削弱 scaling。当前评估集中在数学推理和 batch-parallel TTS；对不适合并行采样验证的任务，收益不一定成立。

3、论文实现：
    - Baseline 如何实现：主要系统 baseline 是 llama.cpp 的 OpenCL backend，使用 Snapdragon Adreno GPU 上的优化 Q4_0 matrix multiplication kernel；QNN FP16 / QNN-based 数据作为参考而非主 baseline，因为 QNN 粗粒度量化会显著损害推理准确率，且固定图不适合本文自定义 decode/TTS workload。论文没有把 llm.npu、Powerinfer-2、HeteroLLM 作为主要 baseline：llm.npu decode 不用 NPU，PowerServe/QNN-based 有精度问题，Powerinfer-2 和 HeteroLLM 未开源。operator-level baseline 包括 conventional memory layout 的 dequantization GEMM、只应用 HMX layout 的版本、无 dequantization 上界，以及 F32/F16 polynomial exp 的 Softmax。
    - 新设计如何实现：系统基于 llama.cpp 修改，约 7K 行 C/C++ 和 inline assembly。实现分两部分：Hexagon NPU operator library 编译成独立 Hexagon DSP shared object，负责计算 kernel、power management、hardware resource management 和 computation thread pool；CPU 侧在 llama.cpp 中新增 Hexagon NPU backend，用 rpcmem / dmabuf 共享物理内存作为 buffer type。系统通过 FastRPC 启动 remote NPU session，在共享内存中提交计算请求，NPU 侧线程轮询请求；由于 Snapdragon SoC 上 CPU 到 NPU 只有单向一致性，CPU 写入共享内存后需要手动 cache maintenance。模型转换脚本会把 HuggingFace 权重转成 HMX layout 的 GGUF，再通过 llama-quantize 生成 IQ4_NL / Q4_0 / Q8_0 / F16 等混合量化格式。
    - 实验 / 实现平台：NPU 性能实验使用三台 Android 手机：OnePlus Ace3 / Snapdragon 8 Gen 2 / Hexagon V73，OnePlus 12 / Snapdragon 8 Gen 3 / Hexagon V75，OnePlus Ace5 Pro / Snapdragon 8 Elite / Hexagon V79。部分准确率实验在 NVIDIA RTX3090 服务器上完成。模型为 Qwen2.5-1.5B、Qwen2.5-3B、Llama3.2-1B、Llama3.2-3B，并在 accuracy-cost trade-off 中额外使用 Qwen2.5-7B；数学推理使用 Instruct 版本。Best-of-N 和 step-level Beam Search 使用 Skywork-1.5B-PRM 作为 outcome/process reward scorer。数据集包括 MATH500、GSM8K、WinoGrande、MMLU、Wikitext-2。
    - 关键实验设置与指标：TTS 任务使用 pass@1 accuracy 和 on-device per-token decode latency / energy 衡量 Pareto frontier，统一 0-shot CoT prompt。operator-level GEMM 选取 Qwen2.5 和 Llama3.2 中 Attention 的 Wq/Wo 以及 FFN 的 Wgate/Wup/Wdown；多数矩阵使用 Q4_0 约 4.5 BPW，FFN down matrix 使用 Q8_0 约 8.5 BPW 降低误差。总体结果显示，在 Best-of-N 中，Qwen2.5-1.5B 和 3B 的 TTS 结果分别超过 3B 和 7B base accuracy；Beam Search 中 Qwen2.5-1.5B 和 Llama3.2-1B 可达到与各自 3B 版本相当或略优的效率。算子消融中，LUT Softmax 相比 F32 exp 快 1.26-2.19 倍，相比 F16 exp 最高快 1.60 倍；dequantization GEMM 相比 conventional layout 快 9.65-19.04 倍，相比仅 HMX layout 版本快 1.82-3.45 倍，平均只比 no-dequantization 上界慢 27%。内存方面，在 context budget 4096 下，Qwen2.5-1.5B / 3B 的 dmabuf 约 1056 MiB / 2090 MiB，总内存约 1.3 GiB / 2.4 GiB。

4、pipeline/kernel解析：
    - 新pipeline/kernel是什么：论文的新执行路径可以概括为 mobile-NPU test-time scaling inference pipeline，以及两个核心 kernel/dataflow：hardware-aware tile-quantized dequantized GEMM 和 LUT-based FP16 FlashAttention / Softmax。pipeline 从离线权重转换开始：HuggingFace weight -> HMX tile layout GGUF -> group/tile quantization -> HVX-friendly repack；在线 decode 时，llama.cpp CPU backend 调度 NPU operator library，经 shared memory / FastRPC 把 activation、KV cache 和 operator request 交给 Hexagon NPU；NPU 上 HVX 负责 dequantization、LUT Softmax、scale broadcast 等通用向量操作，HMX 负责 FP16 matrix tile multiplication。TTS 层面则把 Best-of-N / Beam Search 的多个候选路径组织成 decode batch，填充 HMX 的 32×32 tile 行，利用原本单 token decode 时空置的矩阵计算能力。
    - 新pipeline/kernel的执行流例子：以 Qwen2.5-1.5B 在手机上做 Best-of-N、batch size 为 8 的 decode 为例。离线阶段，模型权重先按 HMX 需要的 FP16 tile 顺序重排，每个 32 元素 group 在 tile memory order 中量化为 INT4，并把 8 个 group 合并为一个 128-byte super-group，同时把 scale 连续存放。在线阶段，CPU 侧 llama.cpp 维护采样、候选路径和未迁移到 NPU 的 logits/lm_head；当一个 transformer layer 的线性层开始执行时，CPU 将请求写入 rpcmem 共享区域并做 cache flush，NPU 轮询到请求后，DMA/l2fetch 将权重 tile、activation tile 搬入 TCM。HVX 用 vlut16 把 INT4 权重值直接查表成 FP16，并查表/广播 scale，形成 HMX 可消费的 FP16 tile；HMX 对 8 条候选路径对应的 activation rows 与 weight tiles 做 tile-level inner product，虽然 batch size 只有 8 仍未填满 32 行，但比单路径 decode 显著提高 tile 利用率。进入 attention 时，FlashAttention 按 Q tile 和 KV tile 流式处理，HMX 计算 QK 和 PV，HVX 做 rowmax、rowsum 和 LUT_Exp；safe softmax 保证 exp 输入非正，因此只需 64KiB 负半区 FP16 LUT。每步生成后，Best-of-N 保留多个候选继续采样，最终由 reward/verifier 选择最优输出；Beam Search 则在中间步骤用 PRM 打分并剪枝低质量路径。整个执行流的关键不是减少 token 数，而是把 test-time scaling 带来的多路径 batch 映射到 NPU 原本低利用率的 matrix tile 上，同时用 tile quantization 和 LUT 降低 HVX/TCM 侧的附加成本。
