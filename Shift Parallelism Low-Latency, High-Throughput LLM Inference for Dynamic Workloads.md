论文标题：Shift Parallelism: Low-Latency, High-Throughput LLM Inference for Dynamic Workloads

    开源仓库确认：

    1、论文工作：
        - 论文要解决的核心问题：生产级 LLM inference 同时面对 interactive workload 和 batch workload。interactive 请求重视 TTFT 和 TPOT，需要低延迟；batch 请求重视 combined throughput，需要低成本高吞吐。现有并行策略通常只能优化其中一端：Tensor Parallelism（TP）能降低单请求延迟，但 layer 间 all-reduce 通信降低总体 tokens/sec；Data Parallelism（DP）跨请求复制模型，吞吐高，但不能加速单个请求，低并发交互场景响应慢。真实企业流量又是 bursty 和 dynamic 的，同一个 Llama-70B 之类部署会同时承载 chatbot、agentic coding、RAG、批量摘要/翻译等多种请求，因此静态选择 TP 或 DP 会产生明显 latency-throughput-cost trade-off。
        - 论文的主要贡献：论文把训练中使用的 Ulysses Sequence Parallelism（SP）改造成可用于 inference 的并行方式，并发现 SP 与 TP 可以共享相同的 attention/KV cache layout，也就是论文称为 KV cache invariance 的性质。基于这个性质，论文提出 Shift Parallelism：在同一个部署里保留 base configuration（SP 或 SP+TP，偏吞吐和 TTFT）与 shift configuration（full TP，偏 TPOT），运行时根据每轮 batch token 数是否超过阈值动态切换。它还补齐 inference 场景缺失的 GQA 支持、KV cache replication、小 batch load balancing、任意 `(SP, TP)` 组合、KV head ordering 对齐、双模型权重加载和 vLLM 插件集成。
        - 论文所处背景：该工作位于 LLM serving 系统、多 GPU inference parallelism、KV cache 管理和生产流量自适应调度的交叉点。它不是提出新的模型结构或 attention 稀疏算法，而是在 vLLM 一类 continuous batching serving runtime 里改变多 GPU 并行策略的选择路径。与 chunked prefill / Sarathi-Serve 主要解决 prefill-decode 混批不同，Shift Parallelism 关注的是同一批次大小随流量变化时，GPU 间并行通信模式和 KV cache 布局如何在低延迟与高吞吐之间切换。与 disaggregated prefill/decode 不同，它不把 prefill worker 和 decode worker 拆到不同 GPU 池，也避免每个请求跨 worker 传 KV cache。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：TP baseline 将每层权重和计算切到多 GPU 上，适合降低 TTFT/TPOT，但 row-parallel linear 等路径需要 all-reduce。论文用复杂度和实验说明，在固定序列长度下，TP degree 增加会提高 communication-to-compute ratio，导致高流量下 combined throughput 低。DP baseline 跨请求复制完整模型，通信少、吞吐高，但单请求内部没有被并行化，interactive 低并发或长 prompt 时 TTFT/TPOT 不理想。更关键的是，TP 与 DP 的 KV cache memory layout 不兼容，不能在同一个请求生命周期中低成本切换；如果分别部署 TP 节点和 DP 节点再路由请求，会复制部署成本并增加系统复杂度。原始 Ulysses SP baseline 虽然有接近 DP 的吞吐性质并能并行 prefill，但它来自训练场景，缺少 inference 常见的 GQA 支持；当 KV head 数小于 GPU 数时不能自然扩展；低流量 decode batch 很小时会因为 sequence partition 不均衡造成严重 load imbalance；并且 SP 不能像 TP 那样并行 decode step，所以 TPOT 可能变差。
        - 论文的设计方法：Shift Parallelism 使用两个配置。Base configuration 使用 full SP 或 mixed `(SP, TP)`，满足 `SP × TP = GPU 数`，用来处理大 batch，以降低 prefill TTFT 并提高 combined throughput；shift configuration 固定为 `(SP=1, TP=GPU 数)` 的 full TP，用来处理小 batch，以降低 decode TPOT。运行时算法非常直接：如果当前 batch token 数大于 shift threshold，就执行 base config；否则执行 shift config。为了让切换不搬移 KV cache，论文要求 base 与 shift 在 attention head shard 和 ordering 上保持一致。对于任意 `(SP, TP)` 组合，它构造 `SP_TP` group，使 shift model 按 base config 的 SP group order 加载权重，从而维持 KV cache coherence。
        - 方法如何对冲 Baseline 缺陷：相对 TP，SP/base config 避免了 TP 在高流量下大量 all-reduce 带来的吞吐损失，尤其适合大 batch prefill 和 batch workload；相对 DP，SP 能在单请求的 sequence 维度上并行 prefill，因此 TTFT 更低。相对“TP 与 DP 双部署”，Shift Parallelism 只在同一个 serving deployment 内切换 SP 与 TP，不需要复制整套服务池。相对原始训练版 Ulysses，论文加入 GQA extension、KV cache replication、fused all-to-all、padding-based load balancing 和 mixed `(SP, TP)` forward path，使 SP 能覆盖 Llama/Qwen dense model 与 MoE model 等 inference workload。KV cache invariance 是核心对冲点：它避免了 TP/DP 那种 KV cache layout mismatch，使请求不需要在切换时重排或搬迁已有 KV cache。
        - 关键 trade-off：Shift Parallelism 用更高实现复杂度和额外内存换取动态性能。Separate models 实现会同时加载 base model 与 shift model，两者共享 KV cache 但不共享权重；论文给出 shift model 额外权重开销约为 `1 / SP`，例如 `SP=8` 时约 12.5%。如果改用 on-the-fly weight slicing 可以省内存，但 Hopper FP8 tensor core 的限制会导致切片需要矩阵转置，性能不如双模型加载。SP 在小 batch decode 上仍然有 load imbalance 和 TPOT 劣势，因此需要阈值切换；阈值选择会影响 latency/throughput 偏好。高 traffic 下 Shift Parallelism 的吞吐仍低于纯 DP 的理论上限，因为 parallel attention 仍有 GPU communication。长上下文场景中 attention time 和 KV cache bandwidth 仍会随 context length 增长，论文明确把 sparse attention 等进一步优化留在本工作之外。MoE 模型上，SP 与 expert parallelism 的组合尚未解决，是 future work。

    3、论文实现：
        - Baseline 如何实现：实验比较 vLLM 内置的 TP 与 DP，并额外实现 SP/Ulysses inference 作为候选并行方式。论文还在 production 对比中使用 SGLang v0.4.6、TensorRT-LLM v0.18.2，以及 vLLM 的 latency-optimized 和 throughput-optimized 配置。对于真实 trace，Azure LLM Code Trace 使用 Llama-70B-FP8；Mooncake Conversation Trace 使用 Qwen-32B-FP8，并在初始配置无法承载流量后开启 FP8 KV cache 以增加 KV cache 容量。synthetic burst benchmark 则修改 vLLM serving benchmark，构造低频 steady stream 加高频 burst 的请求到达模式。
        - 新设计如何实现：论文把 Shift Parallelism 集成进 vLLM 插件系统，也就是 ArcticInference。实现上，base model 和 shift model 分别编译和 CUDA graph capture；运行时根据 batch size 选择执行哪一个。base model 使用 Ulysses SP 或 mixed `(SP, TP)`；shift model 把 `tensor_parallel_size` 扩展为 `SP × TP`，并把 `ulysses_sequence_parallel_size` 设为 1，在 `SP_TP` group 上执行 full TP。两套模型共享同一个 attention/KV cache。SP for inference 的实现补充了 GQA：QKV 投影中 Q head 与 KV head 数不一致时，KV cache replication 通过 all-to-all 的 send/receive buffer 完成；Q、K、V 相关通信被融合到单次 all-to-all 中。低 traffic 小 batch 下，SP 使用 padding 到 SP degree 倍数的方式缓解负载不均，但这会增加 redundant token 并拉长 TPOT，所以小 batch 会切到 TP。
        - 实验 / 实现平台：主要硬件是 AWS p5en.48xlarge 单节点 8×H200 GPU，每卡 141GB HBM、4.8TB/s 带宽、Hopper FP8 tensor core，GPU 间通过 NVSwitch 互联，标称 900GB/s。论文默认使用 vLLM v0.9.2 插件实现；Artifact Appendix 的复现依赖写为 vLLM v0.10.1 和 ArcticInference 插件 commit `5e08f0f`。模型包括 FP8 量化的 Llama-70B、Qwen-32B、Llama-17B-16E、Qwen-30B-A3B；前两者是 dense model，后两者是 MoE/sparse model。数据集包括 synthetic burst pattern、Azure LLM Code Trace、Mooncake Conversation Trace、参数化随机请求，以及 HumanEval / CodeAct against SWEBench 混合生成的 speculative decoding 数据。
        - 关键实验设置与指标：指标包括 TTFT、TPOT、completion time、combined throughput tokens/sec、arrival rate 下的 latency-throughput 曲线，以及不同 input sequence length 下的峰值吞吐和最低延迟。Bursty synthetic workload 中，Llama-70B、8×H200、4k input/250 output 场景下，Shift Parallelism 的 median TTFT 为 148ms，median TPOT 为 51ms，peak throughput 为 69,147 tok/s；vLLM throughput-opt DP 为 1,355ms / 83ms / 75,535 tok/s，vLLM latency-opt TP 为 3,930ms / 85ms / 51,162 tok/s。参数化基准中，Llama-70B 的 Shift Parallelism TTFT 约 102ms、TPOT 约 10.1ms、throughput 约 37.4k tok/s；相对 TP 吞吐损失显著更小，相对 DP 延迟显著更低。论文总结 Shift Parallelism 在不同 context size 和 arrival rate 上可提供最高约 1.51× 吞吐提升、最高约 1.67× 到 6.97× response 加速、最高约 2.45× generation 加速。production 组合实验把 Shift Parallelism 与 SwiftKV、speculative decoding 叠加，报告相对最佳 throughput-optimized 部署有 3.4× 更低 completion time 和 1.06× 更高 throughput。

    4、pipeline/kernel 解析：
        - 新 pipeline/kernel 是什么：论文没有提出一个新的单独 CUDA kernel 名称；最接近的新执行路径是“Shift Parallelism dual-configuration inference pipeline”。它由三个关键运行时路径组成：第一，base path 使用 SP 或 `(SP, TP)` 执行大 batch，sequence 维度被切分到多个 GPU，attention 前后通过 all-to-all 在 sequence parallel 与 head parallel layout 之间转换；第二，shift path 使用 full TP 执行小 batch，避免 SP 小 batch 的 padding/load imbalance，并降低 TPOT；第三，shared-KV switching path 通过一致的 attention head layout/order 和共享 KV cache，在同一请求生命周期内按 batch token 阈值在两条路径之间切换。实现层面还包括双模型加载、`SP_TP` parallel group、双 CUDA graph capture，以及 vLLM/ArcticInference 的 runtime model selection。
        - 新 pipeline/kernel 的执行流例子：以 8×H200、base config 为 `(SP=4, TP=2)`、shift config 为 `(SP=1, TP=8)` 为例，一个请求到达后先进入 vLLM continuous batching 队列。如果当前轮 batch token 很多，例如 burst 期间大量 prompt token 等待 prefill，runtime 发现 batch size 大于 threshold，于是选择 base model。输入 token 先按 SP degree 在 sequence 维度切到不同 SP rank；QKV projection 后，系统在 SP group 内执行 fused all-to-all，把 attention 所需的 head shard 聚合到对应 GPU；每个 GPU 对本地 head/KV shard 运行 attention，再经过必要的 all-to-all / all-reduce 和 MLP 路径得到输出，并写入共享 KV cache。由于 base config 的 head ordering 被固定，KV cache 的 head 布局与 shift config 兼容。

          当 burst 结束或某个 decode iteration 中 batch token 很少时，runtime 发现 batch size 小于等于 threshold，于是切换到 shift model。此时同一个请求已有的 KV cache 不需要拷贝或重排；shift model 在 `SP_TP` group 上按 full TP 方式读取同一份 KV cache，并用 TP 并行 attention/linear 路径生成下一个 token，以降低 TPOT。下一轮如果又来了大量 prefill token，系统可以再回到 base model。对一个具体 token 来说，它在大 batch prefill 阶段通过 SP 得到更高 input-token throughput，在小 batch decode 阶段通过 TP 得到更低 per-token latency；贯穿其中的关键是 KV cache invariance，使 token 的历史 K/V 状态不会因为并行策略切换而被搬家。
