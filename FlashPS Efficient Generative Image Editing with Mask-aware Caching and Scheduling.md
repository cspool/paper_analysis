论文标题：FlashPS: Efficient Generative Image Editing with Mask-aware Caching and Scheduling

    1、论文工作：
        - 论文要解决的核心问题：生产级 generative image editing 服务中，请求通常带有 mask，只修改图像模板中的局部区域，但现有 diffusion serving 系统仍近似按整图重新生成执行，导致无 mask 区域的大量 token 在 transformer attention / feed-forward 中被重复计算。与此同时，mask-aware 加速后单请求计算量下降，传统 static batching 造成排队延迟，且不同请求 mask ratio 差异很大，会让 worker 之间出现计算和 cache loading 负载不均。
        - 论文的主要贡献：提出 FlashPS，一个面向图像编辑服务的端到端系统。核心贡献包括：第一，利用 mask 指导复用 unmasked regions 的中间 activation，只对 masked tokens 做主要计算；第二，用 bubble-free pipeline 在 GPU computation stream 与 cache load stream 间重叠计算和 activation 加载，并用 DP 决定哪些 transformer block 使用 cache；第三，把 continuous batching 迁移到 diffusion model 的 denoising step 级别，并把 CPU preprocessing / postprocessing 与 GPU denoising 拆到不同进程；第四，用 mask-aware load balancing 根据 mask ratio 估算计算和加载负载来路由请求。
        - 论文所处背景：论文面向云端图像编辑服务，动机来自两个 workload 观察：公开 diffusion serving trace 中约 70% 请求涉及 image editing；作者在 2025 年 1 月收集的 14 天生产 trace 覆盖 20k GPU、3400 万张生成图像，模板复用很高，仅 970 个 templates 平均各被复用约 35k 次。mask 区域通常较小，作者 trace 的平均 mask ratio 为 0.11，公开 trace 为 0.19，虚拟试穿 benchmark 约 0.35，这使 unmasked activation reuse 具备系统收益。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：Diffusers 等标准系统 mask-agnostic，使用 static batching 并按整图执行，因此推理延迟高、队列等待长。DistriFusion 等多 GPU 并行可以降低单次推理延迟，但使用 8 倍 GPU 仅报告 2.8x speedup，吞吐成本不划算。TeaCache 复用 denoising 过程中的中间 activation 来跳过计算，但面向通用生成任务，存在 latency-quality trade-off，不能利用 mask 精确区分应保持不变的区域。FISEdit 利用 mask sparsity 做图像编辑加速，但论文指出其实现只适配 SD2.1，不支持 SDXL / Flux，且不能把不同 mask ratio 的请求组成 batch，服务吞吐与排队表现受限。请求调度方面，request-count 或 token-count 负载均衡忽略 cache loading 与 mask ratio 差异，在高负载时会放大 tail latency。
        - 论文的设计方法：FlashPS 把图像 latent reshape 后的空间位置视作 transformer tokens，并按 mask 划分 masked / unmasked tokens。对 token-wise 计算，如 linear projection、feed-forward、LayerNorm、activation 等，unmasked token 的计算可以直接由 cached activation 替代；对 attention，FlashPS 观察到 unmasked token 在输出 Y 上跨请求高度相似，且 masked / unmasked token 之间 attention 较弱，因此选择缓存 transformer block 输出 Y 中 unmasked tokens 的 activation，而不是缓存 K/V。系统将 cached activations 存在 host memory / disk / distributed storage，GPU 上通过 CUDA stream 异步加载；调度器根据 request mask ratio、当前 running batch 和 worker 状态做路由。
        - 方法如何对冲 Baseline 缺陷：相对整图 Diffusers，FlashPS 的 mask-aware computation 把 feed-forward、linear projection 和 attention score 的主计算复杂度从全量 token 降到与 mask ratio m 成正比，理论 speedup 约为 1/m，同时保留 cached unmasked activation 作为全局上下文，避免简单裁剪 unmasked 区域导致图像失真。相对 naive cache loading，FlashPS 不把所有 block 都强制 cache，而是比较“加载 cache 后计算 masked tokens”的完成时间与“直接全量计算”的完成时间，用 O(N) DP 选择每个 transformer block 是否使用 cache，从而消除 pipeline bubble。相对 static batching，FlashPS 允许已完成请求在每个 denoising step 后退出，新请求在一个 denoising step 后加入 running batch，避免等待整个 batch 完成。相对 LLM 式 naive continuous batching，FlashPS 将 CPU 密集的 image preprocessing / postprocessing 拆到独立进程，避免它们打断 GPU denoising。相对请求数均衡，mask-aware scheduler 用离线拟合的线性模型估算 computation latency 与 cache loading latency，再选择预计 pipeline latency 最低的 worker。
        - 关键 trade-off：FlashPS 用 host memory / secondary storage 容纳 GiB 级 template activations，换取 GPU HBM 容量压力降低，但引入 PCIe / storage 到 HBM 的加载开销，需要 pipeline 重叠来隐藏。选择缓存 Y 而不是 K/V 减少 cache footprint，但相比 K/V cache 会牺牲少量单请求延迟优势；论文给出的例子中，mask ratio 20% 时缓存 K/V 可把 latency 从 2.27s 降到 2.06s，但 cache size 翻倍。FlashPS 的收益依赖 mask 区域较小、模板复用高、编辑目标确实保持 unmasked region 不变；对 style transfer 这类全局改变图像外观的任务，mask-aware computation 与 mask-aware load balance 收益会下降。系统还增加了 scheduler、cache engine、异步 batch 组织和跨进程通信，但论文测得这些开销为毫秒级。

    3、论文实现：
        - Baseline 如何实现：Diffusers 作为标准 full-image generation baseline，使用 static batching 且无 load balance policy。FISEdit 作为 mask sparsity baseline，只用于 SD2.1，因为其不兼容 Hopper GPU 且不支持不同 mask ratio 请求 batching。TeaCache 作为 activation reuse baseline，被配置为在可接受图像质量下尽量降低推理延迟。论文还为这些 baselines 实现 static batching 和 request-level load balancing，以便与 FlashPS 的 continuous batching / mask-aware load balance 做系统级对比。
        - 新设计如何实现：FlashPS 是一个 end-to-end serving system，前端使用 FastAPI 接收 image template、mask 和输入条件；后端基于 HuggingFace Diffusers / PyTorch，并保留 FlashAttn 等已有优化。mask-aware image editing 通过改造 attention operator 实现，cache loading 使用 CUDA streams 异步加载 cached activations。continuous batching 的 request queues 和 load balance scheduler 用 asyncio 实现，scheduler 与 workers 之间使用 ZeroMQ 通信。FlashPS 的 worker 包含 model executor、cache engine 和 continuous batching 逻辑；cluster 侧由 scheduler 将请求路由到多个 worker replicas。
        - 实验 / 实现平台：评测模型包括 SD2.1、SDXL 和 Flux。SD2.1 在 NVIDIA A10 GPU 上运行，SDXL 和 Flux 在 NVIDIA H800 GPU 上运行。在线服务评测使用一台 8 GPU 机器，每个 worker 分配 1 张 GPU；SD2.1 最大 batch size 为 4，SDXL / Flux 最大 batch size 为 8。请求到达按 Poisson process 生成，不同 RPS 下测试 end-to-end latency；每个请求的 mask ratio 按生产 trace 分布采样。图像质量评测使用 InstructPix2Pix、VITON-HD 和 PIE-Bench，并用 CLIP、FID、SSIM 及 50 人用户研究评估质量。
        - 关键实验设置与指标：主要指标包括 end-to-end request latency、queuing time、serving engine throughput、P95 tail latency、kernel / image latency 随 mask ratio 的变化、图像质量，以及 scheduler / batching / serialization overhead。整体结果显示，FlashPS 相比 Diffusers 平均请求延迟最高降低 14.7x，相比 FISEdit 最高降低 4x，相比 TeaCache 最高降低 6x；在 RPS=3 时，P95 request latency 相比 Diffusers、FISEdit、TeaCache 分别降低 88%、71%、60%。serving engine 方面，batch size 大于等于 2 时 FlashPS 相比 baselines 最高提升 3x throughput。质量方面，FlashPS 的 SSIM 最高达到 0.99，并在用户研究中相比 FISEdit 和 TeaCache 分别获得 2.0x 和 1.63x 更高 acceptance rate。消融显示，mask ratio 为 0.2 时，FlashPS 对 SD2.1 / SDXL / Flux 的 image-level inference speedup 分别为 1.3x / 2.2x / 1.9x；FlashPS 的 disaggregated continuous batching 相比 static batching 和 naive continuous batching 可把 P95 latency 分别降低约 35% 和 40%；mask-aware load balance 在高请求流量下相比 request-granularity / token-granularity baselines 可把 tail latency 降低最多 26%。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：本文最核心的新执行路径是 FlashPS 的 mask-aware cached activation serving pipeline。它不是单个 GPU kernel，而是把 transformer block 内 mask-aware operator、block 粒度 activation cache、CUDA stream cache loading、DP bubble-free block selection、denoising-step continuous batching 和 mask-aware request routing 串起来的系统级 pipeline。论文也改造了 attention operator：只对 masked tokens 执行必要计算，并用 cached Y activations 补回 unmasked tokens 的输出。
        - 新pipeline/kernel的执行流例子：一个请求到达后，scheduler 读取其 image template、mask 和输入条件，并根据 mask ratio 估算把该请求放入每个 candidate worker 后的 computation latency 与 cache loading latency。估算函数会调用扩展版 pipeline DP，考虑当前 running batch、block 级 cached / uncached 选择和加载时间，最终把请求放到预计 latency 最低且有 batch slack 的 worker。进入 worker 后，preprocessing process 将图像和 mask 编码到 latent；主进程在每个 denoising step 组织 running batch。对某个 transformer block，如果 DP 决定使用 cache，cache load stream 先异步把该 block 的 unmasked token activations 从 host memory 加到 HBM，同时 computation stream 对 masked tokens 执行 attention / feed-forward 等计算；block 输出时，masked token 结果与 cached unmasked activations 合并成完整 Y，送入下一个 block。如果 DP 发现加载 cache 会产生 bubble 或比直接全量计算慢，则该 block 直接计算 masked 与 unmasked 全部 tokens。一个请求完成全部 denoising steps 后立即退出 running batch，并交给 postprocessing process 解码成输出图像；新请求只需等到下一个 denoising step 边界即可加入，而不必等待整批图像全部完成。
