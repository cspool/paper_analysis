1、论文工作：
    - 论文要解决的核心问题：长上下文 LLM decoding 中，attention 需要反复访问持续增长的 KV cache，GPU 受外部内存带宽限制，PIM 虽然有高内部带宽和近数据 GEMV 能力，但现有 PIM attention 设计多按 dense/full KV cache 访问组织数据。动态稀疏 attention 会选择少量重要 token，访问模式细粒度、动态且不规则，和 PIM 的 DRAM row 级访问/计算粒度不匹配，导致即使算法上选择了少量 token，硬件上仍会激活很多包含无关 token 的行，无法真正跳过计算和访存。
    - 论文的主要贡献：STARC 提出一种面向 PIM 的稀疏 KV cache 数据映射方案：用 cosine K-means 将语义相似的 key/value pair 聚类，并把同一 cluster 的 KV 连续放在 PIM bank/row 友好的物理位置；decoding 时 query 不再逐 token 检索，而是先和预计算 cluster centroid 打分，再按 cluster 粒度取回 KV，直到满足 KV budget。论文还把 K-means 的 normalization、assignment、update 映射到 AttAcc 风格 HBM-PIM 的现有命令和近 bank GEMV 单元上，只增加轻量 VNORM 融合命令，避免把 KV 拉回 GPU 做聚类。
    - 论文所处背景：该工作处于 long-context LLM inference、KV cache sparsity 和 GPU-PIM heterogeneous acceleration 的交叉点。现有 AttAcc/NeuPIMs/PAPI/IANUS 等 PIM 系统主要解决 dense attention 的内存带宽瓶颈；SparQ、InfiniGen、Quest 等 sparse attention 方法减少 token 检索量，但没有充分处理 PIM row 级访问约束。Quest 这类 page-wise 方法更贴近硬件 row/page 粒度，却会把语义无关 token 一起取回，影响有效 token recall 和精度。STARC 的切入点是同时保持 sparse attention 的语义选择能力和 PIM 的规则行级执行效率。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：Full KV retrieval 在长上下文 decoding 中访问全部历史 KV，attention 层访存和 GEMV 成本随上下文增长，PIM 也会被大量 dense row activation 占满。Token-wise sparsity 如 SparQ/InfiniGen 可以保留较高精度，但选中的 token 散布在许多 DRAM row 中；PIM 一次行激活会带出整行 KV，最终出现频繁 row switching、无关数据 over-fetch 和 bank/GEMV 利用率下降。Page-wise sparsity 如 Quest 把访问粒度变成固定 page，和 PIM row 更匹配，但 page 内常只有一两个真正重要 token，剩余 token 被无效读取和计算，导致模型精度和语义相关性不足。
    - 论文的设计方法：STARC 用 clustering-based remapping 重排 KV cache。每 64 个 token 作为一个 block，设 K=4 做 cosine K-means，使每个 cluster 约 16 个 token，对齐论文假设下 PIM row across bank group 可容纳的 16 个 key 或 value vector。聚类只基于 key，value 继承 key 的 cluster label；cluster 形成后固定，不在后续 decoding 中反复更新。查询阶段对所有 centroid 计算 q^T mu 分数，按分数排序选择 top clusters，直到达到 KV cache budget，最后一个 cluster 可截断；尚未累计到 64 个的新生成 token 作为 non-clustered tokens 保留参与 attention。
    - 方法如何对冲 Baseline 缺陷：相对 Full KV，STARC 仍只访问 budget 内的 KV，减少 attention 访存和计算；相对 token-wise sparsity，它把可能被同一 query 命中的语义相近 token 预先放到连续 row 中，让一次 row activation 携带更多有用 token；相对 page-wise sparsity，它的 page/row 不是按原始时间顺序硬切，而是由 key 语义聚类形成，因此减少 page 内无关 token。硬件侧，STARC 把 sparse attention 的不规则 token mask 变成更规则的 cluster-level row access，让 PIM 可以实际跳过未选 cluster 对应的行。
    - 关键 trade-off：STARC 接受 cluster 粒度访问带来的选择粗化，可能在最后一个 cluster 截断或 cluster 内混入少量低相关 token；K-means 引入额外聚类开销和 centroid 存储/打分路径；cluster 固定后不随后续 query 重新优化，牺牲部分动态适应性。但论文用 append-only/incremental clustering 控制开销，使每个 token 只聚类一次，并用 K=4、block size=64 将算法 arithmetic intensity 调到 HBM-PIM 计算/带宽平衡点附近。

3、论文实现：
    - Baseline 如何实现：精度比较包括 Full KV、SparQ、InfiniGen 和 Quest。论文在同一框架下复现这些 sparse attention 方法，并保留 Quest 的设置，即前两层使用 full KV cache，因为这些层通常 sparsity 较低。性能比较中，Quest 代表 page-wise sparsity，SparQ 代表 token-wise sparsity，Full KV retrieval 作为 dense attention 基准；PIM 性能仿真把各方法真实 inference 产生的 attention mask 映射到 DRAM row 粒度，统计每步需要激活多少行。
    - 新设计如何实现：STARC 在 HBM-PIM 内实现 cosine K-means 的三个阶段。Normalization 用 MAC_AB 做 self-dot，再用 MVSB 转移 norm，并引入 VNORM 做 reciprocal square-root 近似和缩放；Assignment 用 WRGB 写入 sample vector，MAC_AB 计算 sample 和 centroid 的点积，MVSB 收集分数，由 host 做 argmax 并返回 label；Update 用 MVGB 广播 sample，MAC_AB 累加/平均，WRGB 写回新 centroid。论文强调这些阶段主要由 AttAcc 已有 PIM primitives 组合完成，VNORM 和控制逻辑不引入明显额外面积开销。
    - 实验 / 实现平台：PIM 架构基于 AttAcc，AttAcc 扩展 Ramulator 建模 heterogeneous GPU-PIM 系统。系统配置为 DGX+AttAcc：DGX 侧 8 个 NVIDIA H100 GPU、40 个 HBM3 stack，总 1.28 TB；AttAcc 侧额外 40 个 HBM3 stack，总 1.28 TB；每个 DRAM bank 集成一个 GEMV unit，使用 1P1B 配置。论文假设每个 pCH 有 32 个 GEMV unit，每个 GEMV unit 有 16 个 FP16 FMA pair、666 MHz，系统峰值约 873 TFLOPs/s、内部带宽约 242 TB/s，计算/访存平衡点约 4 FLOPs/Byte。精度评测在 LongChat-7B-v1.5-32K、LLaMA-3.1-8B-Instruct、Mistral-7B-Instruct-v0.3 上进行；性能评测用 LLaMA-7B、Mistral-7B、GPT-13B，FP16，batch size 16，prefill/decoding 长度为 2K/16K、2K/24K、2K/32K。
    - 关键实验设置与指标：默认 KV cache budget 为 1024 token，附录还给出 256、512、2048 budget。精度 benchmark 包括 LongBench 16 个数据集、RULER 32K context、PG-19 perplexity，以及 important token recall。性能指标包括 normalized end-to-end decoding latency/energy、attention-layer latency/energy、clustering overhead。主要结果是：在 attention 层，相比 token-wise sparsity，STARC 最高降低 78% latency 和 65% energy；相比 full KV retrieval，最高降低 93% attention latency 和 92% attention energy。端到端 decoding 中，STARC 相比 full KV 提供约 25%-48% speedup 和 34%-56% energy reduction，并比 token-wise 方法进一步快约 13%-21%、能耗低约 11%-18%。RULER 32K 上 STARC 平均 0.8727，接近 Full KV 0.8812 和 SparQ 0.8831，高于 InfiniGen 0.8419 与 Quest 0.7848；LongBench 上 STARC 通常优于 Quest，并接近 token-wise sparsity 精度。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：论文没有提出一个传统 GPU fused kernel，而是提出 STARC cluster-level selective token access pipeline：一个将 KV cache 聚类、物理 remapping、centroid 检索和 PIM row-aligned attention execution 连接起来的 decoding 数据流。其核心执行单元是 PIM 上的 cosine K-means command sequence，以及 decoding 时按 centroid 分数选择 cluster 后触发的行级 attention 计算路径。
    - 新 pipeline/kernel 的执行流例子：假设 prefill 后已有一段 KV cache。首先，系统把 prefill KV 按 64-token block 分块；对每个 block，只拿 key 在 HBM-PIM 内做 K=4 的 cosine K-means，最多 16 次迭代；每个 cluster 约 16 个 token，于是 key/value pair 被按 cluster label 连续写入 PIM bank/row 对齐的位置，cluster centroid 留作后续检索。进入 decoding 后，GPU/xPU 生成当前 token 的 query/key/value，attention 仍 offload 到 PIM，FFN/QKV generation 留在 GPU。当前 query q_t 与所有 cluster centroid 计算 q_t^T mu_j，cluster 按分数排序；系统选择 top clusters 直到达到 KV budget B，必要时截断最后一个 cluster，同时把尚未聚满 64 个的新 token 也加入候选。PIM 只激活被选 cluster 所在的连续 rows，执行 attention score 和 value aggregation；未选 cluster 的 rows 被跳过。每当新生成 token 积累到 64 个，STARC 再对这 64 个新 key 做一次本地 K-means，将新 KV append 到已有 cluster layout 后面，旧 cluster 不重排。这样，一个 token 在生命周期中只参与一次 clustering，但在后续许多 query 中可以被 row-aligned cluster retrieval 重复利用。
