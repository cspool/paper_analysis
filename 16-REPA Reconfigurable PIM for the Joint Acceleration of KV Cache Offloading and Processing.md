1、论文工作：
    - 论文要解决的核心问题：LLM inference 中 KV cache 同时带来两个系统瓶颈。第一，KV cache 占据大量 GPU memory，并随 sequence length 增长，迫使系统把 KV cache offload 到 SSD/CPU/外部存储，但 offloading 会引入明显持久化、迁移和 reload 开销；论文在 Azure23 trace 上观察到 SSD-based KV cache offloading 在触发 1-2 次 eviction 时可造成约 0.3-2.0x slowdown。第二，decoding 阶段的 scoring 和 context 操作本质是围绕 KV cache 的 GEMV，arithmetic intensity 很低，且不同请求的 KV cache 不能共享，因此 GPU 即使增大 batch size 也难以充分利用。
    - 论文的主要贡献：REPA 把“KV cache offloading”和“KV cache processing”合并为一个硬件/系统问题，提出 GPU-PIM hybrid system。GPU 继续执行 prefill 和 decoding 中可 batch 的 qkv generation、projection、FFN；REPA-PIM 承担 decoding 中不可 batch 的 scoring、context 以及 KV cache persistence。论文选择 reconfigurable ReRAM PIM 作为 REPA-PIM，以获得非易失性、高容量和足够的并行计算能力；再通过 bulk-wise memory setting instruction、多级 controller、locality-aware data mapping、sub-batch pipelining 和 transfer overlapping 来弥补 reconfigurable PIM 单次操作较慢的问题。
    - 论文所处背景：传统 KV cache offloading 系统主要研究何时 evict/reload 或如何组织 offloading hierarchy，但 offloaded KV cache 通常只作为被动数据保存，等请求重新调度后才处理。Stage-split inference 则把 prefill/decoding 放到不同 GPU 上改善高端 GPU 利用率，但 decoding 中 scoring/context 的低算术强度并没有消失，只是转移到较弱设备上。PIM 的特殊性在于它既是存储又是近数据计算硬件，正好适合把 KV cache 存在 PIM 中并直接处理非 batchable attention 路径。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：GPU-only baseline 的问题来自 memory bandwidth 和低 batchability。KV cache 大量驻留或迁移，占用 30%-80% 级别的 GPU memory；decoding 中 q×K^T 和 S×V 对每个请求的 KV cache 独立执行，batch size 增大时 scoring/context 的 GPU utilization 提升很有限。已有 offloading baseline 如 FlexGen 主要优化外存/显存之间的数据调度，但不能在 offload-time 处理 KV cache。已有 DRAM PIM baseline 如 AttAcc、PAPI、TransPIM 类方案依赖 near-bank CMOS logic，单次操作快但并行粒度和容量/数据布局受限；reconfigurable DRAM PIM 如 DRISA 面积低但 DRAM cell array 不支持多 wordline 并行，限制了 parallelism。Analog ReRAM PIM 速度快但 ADC 和模拟外设面积/功耗大，在同等面积预算下容量不足，不适合保存 memory-hungry KV cache。
    - 论文的设计方法：REPA 采用 GPU-PIM split。GPU 处理 prefill 全阶段以及 decoding 中可 batch 的 qkv generation、projection、FFN；REPA-PIM 处理不可 batch 的 scoring、context 和 KV cache offloading/persistence。REPA-PIM 是 3D-stacked ReRAM PIM，包含 buffer die 和 8 个 PIM dies，每个 die 有 16 tiles，每个 tile 有 8 个 PU，每个 PU 有 128 个 1024×2560 cell arrays，划分为 4 个 array groups。每个 cell array 中 2048 columns 用于 PIM region，512 columns 用于 temp region。
    - 方法如何对冲 Baseline 缺陷：针对 reconfigurable PIM 操作次数多的问题，REPA 提出 BLK_SET bulk-wise memory setting instruction，在 64 条相邻 wordlines 上并行执行 NOR-based addition/multiplication 所需的 memory setting，指令只需 opcode、block address、Input1、Input2 四类字段，并由固定 NOR type 推断输出 bitline。针对并行控制问题，REPA 在 tile group、tile、PU 层放置 multi-level controllers，使 tile group controller 分发操作、tile controller 并行 PU 并累加 partial results、PU controller 操纵 ReRAM cells。针对传统 PIM interleaved mapping 带来的 long-range transfer，REPA 按 attention head 组织 KV cache group，把 per-head K/V matrix 切为较大 slice 并放入邻近 cell arrays，让 q×K^T 的 partial result 能在同一 PU 或 tile 内被后续 S×V 使用。
    - 关键 trade-off：REPA 接受了新的 PIM 硬件、controller 面积、runtime memory management 和 mapping 复杂度，换取 KV cache persistence 与 decoding GEMV 的合并加速。Reconfigurable ReRAM PIM 的单次操作速度不如 DRAM PIM，需要更强的 wordline/array/tile 并行来摊薄开销；4 controllers/PU 带来 5.76mm2 per-die 额外面积，但换得 3.91x q×K^T speedup。REPA 的收益更依赖长序列、大 batch、较大模型和足够并行度；在 one-shot inference 或较短 sequence 下，传统 DRAM PIM 的单操作速度优势可能缩小甚至超过 REPA。ReRAM endurance 也是潜在风险，论文通过估算每 cell 每年 memset 次数低于 2.8×10^10，并引用高 endurance ReRAM 大于 10^12 的报告来说明可接受，但这仍依赖器件工艺假设。

3、论文实现：
    - Baseline 如何实现：评测中使用 NVIDIA A100 GPU-only baseline；AttAcc 作为 state-of-the-art DRAM PIM attention acceleration baseline；PAPI 作为根据 compute-bound/memory-bound 动态调度任务到合适设备的 DRAM PIM baseline；DRISA 作为 reconfigurable DRAM PIM baseline；AiF 作为 in-flash processing baseline。端到端 offloading 评测使用 FlexGen 作为 testbed，并构造 Flex-REPA，即保持 FlexGen offloading strategy 不变，把 offloaded weights 和 KV cache 放到 REPA-PIM 中处理。设计消融包含 REPA-SPL、REPA-NOP、REPA-NOL、Dual-PIM，以及 data mapping baseline Interleaved mapping 和 RingBroadcast。
    - 新设计如何实现：作者实现了 REPA prototype，其中 Python 约 1.3K LoC，C++ 约 7.0K LoC。系统扩展 PyTorch，提供 Python wrapper 供 inference task 使用 REPA-PIM；同时实现 runtime system 管理 task status 和 PIM memory。PIM memory 以 byte addressing 管理，分配粒度为 64×2560 blocks，其中 2048 cells 保存 KV cache 和执行 PIM computation，512 cells 用作 intermediate temporary storage。REPA 使用 FP16 数据格式，reconfigurable addition/multiplication 采用 FloatPIM 的过程；softmax 中 maximum 使用 ReSQM 的 fast in-situ maximum，exp 被改写为 2^(x log2 e)，再分解为 2^x 和 multiplication，其中 2^x 用 left-shifting 完成。
    - 实验 / 实现平台：实验机器为 Ubuntu 22.04 server，40 CPU cores 和 8 张 NVIDIA A100 GPUs。REPA-PIM 通过 in-house simulator 实现，基于 NeuroSim-3D，即 3D-stacked version of cycle-accurate NeuroSim。模拟器设定 REPA-PIM 运行在 1GHz，使用 14nm bipolar resistive technology node，参数来自 VTEAM model；ReRAM switching delay 为 1ns，SET/RESET voltage pulse 分别为 1V 和 2V。instruction dispatch latency 由 time-to-flight latency 与 on-wire transfer latency 估计，TGC->TC、TC->PUC、PUC->DRV on-wire latency 分别为 4ns、2ns、2ns；per-TG、per-Tile、per-PU bandwidth 分别为 256GB/s、32GB/s、4GB/s。
    - 关键实验设置与指标：benchmark 使用 Llama2-7B、13B、70B，sequence length 覆盖 1024、2048、4096，并测试不同 batch size 和 input length。指标包括 token generation performance、end-to-end latency、throughput、offloading speed、TTFT、TBOT、energy per token、scoring/context energy、memory fragmentation、area 和 power overhead。核心结果是：相对 A100，REPA 在 2048 sequence length 下生成 1.8-2.8x tokens，在 4096 sequence length 下生成 2.1-5.5x tokens；相对 AttAcc，4096 sequence length 下提升 0.4-2.4x，但短序列或 one-shot 场景优势缩小。集成到 FlexGen 后，REPA/Flex-REPA 在 E2E latency、throughput、offloading speed 上分别带来 2.4-5.2x、1.0-2.4x、0.9-2.0x 改善。设计消融显示，REPA 相对 REPA-SPL 的 E2E/TBOT 提升说明 batchable decoding projection/FFN 仍应交给 GPU；相对 REPA-NOP 的 0.5-0.8x 改善说明 sub-batch pipelining 必要；相对 REPA-NOL 的 0.2-0.3x 改善说明 transfer overlapping 有效；相对 Dual-PIM 的 1.6-4.0x 改善说明不应让 PIM 承担全部 compute-intensive operations。REPA-PIM 每 die area 为 73.3mm2，其中 ReRAM cells 约 26.2mm2，controller 约 7.92mm2；满并行功耗约 68.92W。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：论文没有提出传统 GPU kernel，而是提出一条 GPU-REPA-PIM hybrid inference execution pipeline。它的核心路径可以概括为“GPU 执行 batchable path，REPA-PIM 执行 non-batchable KV path”。在 prefill 中，GPU 生成 K/V 矩阵并执行主要模型计算，REPA-PIM 同步接收并持久化 KV cache；在 decoding 中，GPU 生成 batched q/k/v，并执行 projection/FFN，REPA-PIM 对已持久化的 K/V 执行 q×K^T scoring、softmax 和 S×V context。硬件层面的 closest kernel 是 REPA-PIM 上的 BLK_SET-driven reconfigurable PIM computation：通过 bulk-wise memory setting、array-group parallelism 和 locality-aware KV placement 执行 partial scoring/context。
    - 新 pipeline/kernel 的执行流例子：以一个 decode batch 中某个请求的一个 attention head 为例。prefill 时，GPU 对每个 decoder block 生成 K 和 V。REPA 将该请求的 KV cache 按 head 拆为 KV cache group；每个 per-head K/V matrix 最多 1MiB，被切成四份，分别放到同一 PU 的四个 array groups 中。K matrix 按 rows 切分并映射到 cell array rows；V matrix 使用转置形式并按 slices 映射，使后续 S×V 能复用 q×K^T 的局部 partial result。多个 head 的 KV matrices 由不同 tile groups 并行持久化，减少 offload-time persistence 延迟。
    - 新 pipeline/kernel 的执行流例子：进入 decoding step 后，GPU 对 batch 执行 q/k/v generation。REPA-PIM 接收 batched q 和 k/v vectors；q/k transfer 与 GPU 生成 v 的计算重叠，v transfer 与 REPA-PIM 上的 q×K^T computation 重叠。对某个 head 的 K slice，REPA-PIM 将 q vector 复制到对应 cell array rows，使用 BLK_SET 指令在 64 adjacent wordlines 上批量执行 NOR-based multiplication/addition，完成 q0×K.0、q1×K.1 等 element-wise multiplication 和 row-wise reduction。四个 array groups 各自产生 partial S，再通过局部累加和 softmax 构造该 head 的 scoring vector。
    - 新 pipeline/kernel 的执行流例子：随后 context 操作使用同一 head 的 V slices。REPA-PIM 将 scoring vector S 切成 64-element slices 并按行复制，与 V^T slices 并行做 dot-products；array groups 内产生 partial context C 的分量，最后 reduce 成 per-head context vector。因为 K/V slices of the same decoder block 被放在相同或邻近 array group/PU 中，q×K^T 的 partial result 到 S×V 的传播大多停留在 tile 内，论文报告超过 92% computation 在 single tile 内完成，相比 Interleaved/RingBroadcast mapping 显著减少 long-range transfer。最后 REPA-PIM 把 per-head context slices 聚合，广播或返回给 GPU 继续 projection/FFN，完成本轮 token decoding。
    - 新 pipeline/kernel 的执行流例子：sub-batch pipelining 让一个 batch 被拆成两个 sub-batches，并按执行阶段在 GPU 与 REPA-PIM 之间交错。例如 sub-batch A 的 q×K^T/scoring 在 REPA-PIM 上执行时，GPU 可处理 sub-batch B 的 qkv/projection/FFN；当 sub-batch A 进入 context 或返回 GPU 时，sub-batch B 的 KV path 又可进入 REPA-PIM。sub-batch size 根据最近一段时间内设备对当前 workload 的处理能力自适应决定，并结合 iteration-level scheduling 支持新请求加入和完成请求移出。这个 pipeline 的目标不是改变 Transformer 数学，而是让“KV cache 存储位置”和“KV cache 计算位置”一致，从而同时减少 offloading overhead、GPU memory pressure 和 decoding 低利用率。
