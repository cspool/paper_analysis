    1、论文工作：
        - 论文要解决的核心问题：现有模型 serving 系统主要面向 unimodal model，通常把模型看成一个 monolithic computation graph，用统一 batch size 顺序执行所有层 / 模块。这套策略迁移到多模态模型时会忽略模块间异构性：视觉 encoder、文本 encoder、文本 decoder 或 LLM 的结构、计算强度、输入模式和预处理开销差异很大，导致某些阶段 GPU compute 不足、某些阶段 GPU 严重空闲。论文在 BLIP / BLIP-2 上观察到，用 Triton serving VQA 时 GPU 在超过一半时间里 active SM 低于 10%，核心瓶颈是 intra-GPU utilization 低，而不是单个算子本身不够快。
        - 论文的主要贡献：论文提出 EEVEE，一个基于 module multiplexing 的 multimodal serving system。它把多模态模型中的 modality-specific modules 当作可独立调度单元，而不是整模型统一执行；为不同模块配置独立 batch size 和 SM allocation，并允许它们在同一 GPU 上并发执行。为解决模块乱序执行带来的中间结果管理问题，EEVEE 设计了 modal cache controller，用于识别、缓存、复用和压缩模块输出；为解决模块既竞争 GPU 又需要协作完成同一请求的问题，论文提出 constrained-search 的 synergistic scheduling algorithm，在离线 profiling 基础上选择 batch size 和 SM 划分策略。实验覆盖 CLIP、BLIP、BLIP-2、LLaVA-1.5 和 InternVL2.5-8B，报告相对最好 baseline 平均 capacity 提升 157%，高负载下 LLaVA 平均 latency 相比 Triton 降低约 90%，BLIP 场景 GPU active SM 接近 90%。
        - 论文所处背景：该工作位于 multimodal model serving、GPU spatio-temporal multiplexing、batch scheduling、cache management 和 vLLM-based inference runtime 的交叉方向。它关注的多模态模型既包括传统 Multimodal Transformer Models，例如 BLIP，也包括 MLLMs，例如 BLIP-2、LLaVA、InternVL。论文的关键观察是：多模态模型天然由多个模块组成，模块之间存在 directed dependency，但各模块对 batch size、SM 数、CPU preprocessing、modal input reuse 的敏感度完全不同，因此应把 serving 单位从“模型”下沉到“模块 / 阶段”。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：Triton / TF-Serving / Clipper / TurboTransformers 等传统 serving 系统把模型整体顺序执行，用统一 batch size 和完整 GPU 资源处理每个模块。这会造成三类浪费。第一，inference heterogeneity：BLIP 中 visual encoder latency 可达 text encoder 的 8 倍，text encoder 在 batch size 约 8 后趋于饱和，而 text decoder 到 batch size 32 以后仍能受益；统一 batch size 无法同时匹配这些模块。第二，preprocessing heterogeneity：图像需要 loading、resize、tensor conversion、normalization 和 H2D transfer，预处理时间可达 visual encoder 计算时间的 40%，串行预处理会让 GPU 等 CPU。第三，input imbalance / reuse：VQA、VideoQA 等任务中，一个图像或视频会对应多个问题，VQA2 中单张图至少有 3 个问题，最多可达 246 个；每个请求都重算视觉模块会浪费 compute 和 latency。Gpulet 类 spatio-temporal sharing baseline 虽然能让模块并发，但它把模块当成彼此独立竞争资源的模型，没有建模多模态模块之间的依赖、stage-level pipeline 和 cache reuse。
        - 论文的设计方法：EEVEE 的核心方法是 module multiplexing：每个模块由单独进程运行，拥有自己的 batch size 和 SM percentage；多个模块可在同一 GPU 上通过 NVIDIA MPS 并行。系统离线根据模型、任务 SLO、GPU 硬件和 workload pattern 生成若干 multiplexing strategies，在线根据当前部署配置选择策略。controller 维护 active cache 和 historical cache：active cache 服务当前乱序执行中的模块间传递，historical cache 支持多轮请求复用；modal cache 对 encoder-decoder 模型存储 cross-attention 消费的视觉 token KV pairs，对 decoder-only MLLM 存储 decoder 消费的视觉 token KV pairs。scheduler 使用 greedy search：从传统顺序执行的可行策略初始化，逐步增加各模块 batch size，把增量放到对 batch latency 负面影响最小的 stage，并用 balanced SM allocation 将 SM 从短 latency 模块转给长 latency 模块，直到不再满足 SLO。
        - 方法如何对冲 Baseline 缺陷：相对 monolithic serving，EEVEE 让高计算模块、小计算模块、预处理阶段和 cache loading 阶段可以错开并发执行，把原本串行 pipeline 中的 GPU bubble 用其他模块工作填充。相对统一 batch size，EEVEE 可以让 text decoder 使用更大的 batch、visual encoder 使用更小或更适合输入复用的 batch，使模块处在更接近饱和的区域。相对每次重算视觉输入，modal cache 允许后续问题直接加载视觉 token 或视觉 KV cache，跳过 visual encoder 和部分 decoder prefill。相对普通 GPU multiplexing，synergistic scheduler 把模块依赖、egress module batch multiplier、SLO latency 和 SM 总量约束写进搜索问题，避免只追求局部并发而把瓶颈转移到其他模块。论文的 strawman 实验也说明，随意并发可能让 BLIP VQA throughput 从 baseline 的 22.2 requests/s 降到 14.1 requests/s，而合适策略可提升到 34.4 requests/s。
        - 关键 trade-off：EEVEE 用更复杂的 runtime、离线 profiling、额外进程和 cache 管理开销换取在线吞吐与 GPU 利用率。module multiplexing 会打破严格顺序执行，必须缓存上游模块输出，因此引入 GPU memory、host memory、I/O 和 hash lookup / eviction 管理成本。modal cache compression 用 attention score 剪掉低重要性视觉 token，可显著减小缓存，但存在轻微 score 下降；系统需要在 full modal cache 和 critical modal cache 间按资源状态取舍。调度策略依赖离线 profiling，并要求 CUDA 初始化前配置 MPS active thread percentage，因此在线频繁重新 profiling 不现实；动态负载主要靠预计算策略切换和 conservative default 兜底。多进程 vLLM 还会带来参数副本和 inter-module transfer 问题，论文通过 GPU shared memory、CUDA IPC 和只读参数共享缓解，但这仍是系统复杂度的一部分。

    3、论文实现：
        - Baseline 如何实现：实验 baseline 包括 Triton 和 Gpulet。Triton 代表主流 unimodal serving 范式，把多模态模型当成 monolithic graph，所有模块顺序执行并使用统一 batch size。Gpulet 代表 spatio-temporal GPU sharing；为支持多模态推理，论文把每个模块当成一个独立模型，并强制所有模块使用固定共享 batch size。为公平比较，论文将 baselines 集成到相同底层 inference engine vLLM 上，主要比较调度和资源管理策略，而不是底层 kernel 差异。motivation 阶段的 GPU utilization 初步实验使用 NVIDIA Triton 在 GeForce RTX 3090 上部署 BLIP / BLIP-2 的 VQA，batch size 为 8。
        - 新设计如何实现：EEVEE 原型约 5,000 行 Python，使用 vLLM 作为后端 inference engine。每个多模态模块启动一个 vLLM process，并配置不同 batch size 与 SM allocation；GPU 资源隔离和并发依赖 NVIDIA MPS。单机内模块间通信使用 shared memory；每个 GPU 上有 controller process 管理共享 GPU memory space，并让 module processes 和 scheduler 通过 CUDA IPC handler 访问。为减少多个进程使用同一模块时的显存占用，controller 在 GPU shared memory 中保存模块参数的 read-only copy。每个待缓存输出用 64-bit hash 标记，缓存条目为 <hash, tensor, last-access>，按模块维护 hashmap，并用 global LRU queue 在超过 GPU memory quota 时 eviction；过期 tensor 会 spill 到 host memory 或由后台线程释放，尽量与 kernel launches overlap，避免阻塞计算。
        - 实验 / 实现平台：实验使用两类服务器。高端服务器配 Intel Xeon Gold 6230R CPU、104 cores、503GB 内存和 8 张 NVIDIA A100 40GB GPU；低端服务器配 Intel Xeon E5-2698 v4 CPU、80 cores、377GB 内存和 8 张 NVIDIA GeForce RTX 3090 GPU。GPU 与 CPU 通过 PCIe 3.0 连接，峰值带宽 32GB/s。软件环境为 Ubuntu 20.04.5、Docker 23.0.1、CUDA 11.8、Python 3.8、PyTorch 2.0.1。模型包括 CLIP、BLIP、BLIP-2、LLaVA-1.5、InternVL2.5-8B；数据集包括 VQA2、NVLR2、COCO；任务包括 VQA、NVLR 和 image-text matching。请求到达模式使用 Poisson distribution，并从 Azure trace 采样以模拟生产负载；modal cache compression ratio 设为 30%。
        - 关键实验设置与指标：主要指标包括 max capacity，即满足 average SLO / P99 SLO 时系统能承载的最大 queries per second；average latency 与 P99 latency；以及 Nsight 的 SMs Active 指标。VQA 的 SLO 定义为 BLIP batch size 2 推理 latency 的 4x、6x、8x；NLVR 的 SLO 定义为 BLIP batch size 1 推理 latency 的 4x、6x、8x，实际运行 batch size 由到达率、SLO 和 multiplexing strategy 决定。capacity 实验显示，在 3090 server 的 Azure workload 上，EEVEE 相对最好 baseline 平均 max capacity 提升 157%；更严格 SLO 下，EEVEE 性能可达到 Triton 的 2.54x、Gpulet 的 1.9x。latency 实验在 A100 server、Poisson workload 和五种模型上进行，load level 设为 EEVEE peak throughput 的低 / 中 / 高比例；高负载 LLaVA 中，EEVEE 平均 latency 相比 Triton 降低约 90%。micro-benchmark 显示 BLIP VQA / NVLR 中 EEVEE 平均 active SM 接近 90%；调度算法在相同步数下比 simulated annealing、genetic algorithm 和随机 greedy 等 heuristic 最多提升 30.10%；多 GPU 下 BLIP-2 和 LLaVA capacity 近似线性扩展；ablation 显示 stage-level parallelism、module-level scheduling、request-aware reuse 分别把 capacity 提升 146%、152%、116%。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：论文没有提出新的 attention kernel 或 CUDA 微内核；最接近的新执行路径是 EEVEE module multiplexing serving pipeline。它把传统“preprocess -> module1 -> module2 -> ... -> output”的整模型串行路径，改成“离线 profile / strategy search + 在线 module-level concurrent execution + modal cache control + dynamic strategy switching”的系统 pipeline。核心执行单元从完整模型变成模块和阶段：模块可拥有不同 batch size 与 SM percentage，预处理、encoder compute、decoder compute、modal cache loading / compression 可以并发或流水化。对多 GPU，EEVEE 先区分适合 intra-operator model parallelism 的大模块和适合 data parallel replication 的小模块，再在单 GPU 策略基础上复制到多 GPU。
        - 新pipeline/kernel的执行流例子：以 BLIP VQA 为例，用户先提交一张图像和问题 Q1。front-end 将请求送入 scheduler；visual encoder process 处理图像并产生 visual tokens / visual KV cache，controller 用 64-bit hash 为该图像的 modal cache 建索引，优先放入 GPU memory，超额时按 LRU spill 到 host memory。与此同时，text encoder 或 text decoder process 可以在同一 GPU 上处理上一批请求，MPS 按离线策略给 visual encoder 例如 70% SM、给 decoder 分配剩余 SM，使视觉计算与文本解码重叠。随后 text encoder / decoder 消费 visual tokens 与文本 token，生成答案 A1。若同一图像的后续问题 Q2 到达，scheduler 不再重跑 visual encoder；controller 根据 hash 从 GPU 或 host memory 取回 modal cache，必要时加载 full modal cache，资源紧张时加载 compressed critical modal cache。cache loading 可以与新图像的 encoder compute overlap；decoder 使用更适合自身吞吐的 batch size 继续生成 Q2 的答案。整个请求流中，传统 baseline 每个问题都按固定 batch 顺序经过 visual encoder、text encoder、text decoder；EEVEE 则把可复用视觉部分缓存起来，并把不同请求、不同模块、不同阶段交错执行，从而减少 GPU idle time 和重复计算。
