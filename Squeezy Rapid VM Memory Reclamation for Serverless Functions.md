    1、论文工作：
        - 论文要解决的核心问题：N:1 serverless isolation 模型会把同一用户的多个函数实例作为 containers 部署在同一个 VM 中，从而降低 1:1 microVM 模型的 cold start 和 per-instance memory tax；但当负载下降、函数实例被回收后，VM 在 host 侧仍然占着为高并发预留过的内存。现有 VM memory elasticity 机制，如 ballooning、vDIMM hotplugging 和 virtio-mem，在 scale down 时需要在 guest OS 中寻找可 offline 的 memory blocks；由于 Linux 按需、非连续分配页面，不同函数实例的 footprint 会在 ZONE_MOVABLE 中交错，回收 terminated instance 的内存时往往要迁移其他仍在运行实例的页面，导致 reclamation latency 高、CPU 资源被占用，并干扰 co-located functions。
        - 论文的主要贡献：提出 Squeezy，一个面向 VM-sandboxed serverless functions 的 guest OS memory manager 扩展。核心贡献包括：第一，将 hotplugged memory 按函数实例的用户配置内存上限切成 fixed-size Squeezy partitions，让每个实例的 anonymous allocations 只来自自己的 partition，从根源上避免不同实例 footprint 交错；第二，扩展 virtio-mem hot(un)plug 路径，使 unplug 操作可以直接识别已空的 partition 并 offline/remove，无需 page migration；第三，引入 shared Squeezy partition 保存 runtime / library / file-backed mappings，以保留 N:1 模型跨实例共享依赖的内存优势；第四，将机制集成到 OpenWhisk-based FaaS runtime 中，使 runtime 在 scale up/down 函数实例时同步触发 partition plug/unplug。
        - 论文所处背景：该工作处于 serverless computing、microVM / VM sandboxing、Linux virtual memory、memory hotplug 与 cloud resource harvesting 的交叉位置。论文关注的是 FaaS 中 N:1 模型的内存弹性：1:1 模型可以通过关闭 microVM 立即释放每个实例资源，但 cold start 和内存重复开销较高；N:1 模型冷启动更快、共享状态更多，却会在 bursty workload 后留下 host 侧 idle memory。Azure production traces 中热门函数每分钟可能有上千个实例创建和回收，因此内存回收需要达到亚秒级并且不能拖慢正在运行的请求。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：Ballooning 以 page 粒度回收内存，guest balloon driver 需要分配并上报大量 guest physical pages，VM exit 开销随回收大小爆炸。Vanilla virtio-mem 以 128 MiB memory block 为粒度，优于 ballooning，但仍依赖 Linux 原生 offline 路径；当目标 block 中混有其他进程页面时，需要 page migration，论文测得 page migration 平均占 virtio-mem unplug latency 的 61.5%。此外，Linux 在 init_on_alloc 配置下可能对即将 unplug 的 pages 做不必要 zeroing，平均占 unplug latency 的 24%。HarvestVM 类优化通过 proactive reclamation 和 buffering 缓解慢回收对后续 scale up 的影响，但它没有消除 page migration，本质上以额外 reserved memory、激进回收和可能的 tail-latency 偏置换取可用缓冲。
        - 论文的设计方法：Squeezy 利用 FaaS 函数内存需求可预先知道这一约束，在 VM boot 时根据 concurrency factor N 创建 N 个空的 Squeezy zone structures，每个 partition 对应一个函数实例可用的 private anonymous memory。函数实例通过 Squeezy syscall 请求一个 populated 且 free 的 partition，kernel 在 mm_struct 中记录 partition id；后续 page fault 走 Squeezy-aware handler，只从该 partition 分配 anonymous pages。对 file mappings，Squeezy 不放进 private partition，而是放入 VM 级 shared partition，使 container root filesystem、language runtime 和 library pages 仍可被多个实例共享。scale down 时，runtime 先销毁 container，partition_users reference counter 归零后，virtio-mem driver 可直接把对应 free partition offline/remove，并通知 hypervisor 用 madvise(MADV_DONTNEED) 释放 host pages。
        - 方法如何对冲 Baseline 缺陷：相对 vanilla Linux / virtio-mem，Squeezy 不是在回收时再扫描 guest physical memory 寻找可移除 block，而是在分配阶段就把每个函数实例的 footprint 约束在独立 partition 内；因此函数退出后，该 partition 的生命周期也结束，unplug 路径天然知道哪些 blocks 可安全移除。相对 ballooning，Squeezy 仍利用 virtio-mem 的 block-level hotplug 粒度，避免 page-level VM exit 放大。相对 HarvestVM-opts，Squeezy 不依赖预留 buffer 或提前回收别的 idle instances 来隐藏慢路径，而是让同步回收本身足够快，减少 memory waste 和 tail latency 抖动。相对 1:1 microVM，Squeezy 保留 N:1 的共享 OS/runtime 状态，scale up 时只需 plug memory 并创建 container，不需要重新 boot 每个 microVM。
        - 关键 trade-off：Squeezy 接受更强的 workload 假设：每个 N:1 VM 内的函数实例具有相同或相近的 memory requirement，partition size 和 concurrency factor N 需要在 VM boot 时由 runtime 设置；对长生命周期、内存需求不可预测或 NUMA 拓扑复杂的 workloads，固定大小 partition 可能不够灵活。Squeezy 还需要修改 Linux memory manager、virtio-mem driver 和 FaaS runtime，工程侵入性高于纯 hypervisor 侧策略。shared partition 保留了 file-backed sharing，但匿名内存被强隔离，若 workload 中存在跨实例可共享的匿名状态，Squeezy 不会自动复用。plug 阶段仍有 35-45 ms 开销，freshly plugged memory 首次访问还会触发 nested page faults，因此动态 N:1 VM 的 cold start 仍比静态 over-provisioned VM 慢 3-35%，只是显著快于新建 1:1 microVM。

    3、论文实现：
        - Baseline 如何实现：论文比较 balloon driver、vanilla virtio-mem、HarvestVM-style optimized virtio-mem、static over-provisioned N:1 VM，以及 1:1 microVM。Ballooning 通过 guest driver inflation 回收 pages；vanilla virtio-mem 使用 Cloud Hypervisor v38.0 中的 virtio-mem 热插拔实现和 Linux 原生 memory online/offline 路径；HarvestVM-opts 不是复现原系统，因为原论文面向 Windows / Hyper-V 且没有可用实现，而是在作者环境中把 proactive reclamation 和 slack memory buffering 思路应用到 virtio-mem 上；static N:1 作为没有内存回收压力的理想化上界；1:1 microVM 则为每个函数实例启动单独 microVM。
        - 新设计如何实现：Squeezy 实现在 Linux Kernel 6.6.30 中，约 800 LOC，扩展 guest OS memory manager 与 virtio-mem hot(un)plug guest driver。内核侧用类似 ZONE_MOVABLE 的 zone structs 表示 Squeezy partitions，并存入 per-NUMA node zonelists；每个 partition 有唯一 partition id、per-partition lock 和 partition_users reference counter。Squeezy syscall 为进程分配 partition，并在 mm_struct 中加入 partition id 字段。page fault handler 根据 anonymous/file mapping 类型分别从 private partition 或 shared partition 分配页面。fork() 时子进程继承父进程 partition，以保证一个函数实例内的多进程/线程位于同一 partition。unplug 路径修改 virtio-mem，使其选择 free Squeezy partition 并跳过要被 unplug 的 pages 的即时 zeroing。
        - 实验 / 实现平台：实验 host 为 dual-socket 40-core Intel Xeon E5-2630，每个 socket 128 GiB memory；动态 N:1 VM 使用单 NUMA node，vCPU pin 到物理 core，core frequency 设为 2.2 GHz，并启用 host THP。VMM 使用 Cloud Hypervisor v38.0。FaaS runtime 基于 OpenWhisk-based runtime，新增在 VM 中创建 containers、按请求负载扩缩实例、并按实例数量 hotplug/unplug VM memory 的逻辑。workloads 包括 memhog microbenchmark，以及 FunctionBench 的 CNN JPEG classification 和来自 prior work 的 Bert ML inference、BFS、HTML web service；资源限制分别为 CNN 1 vCPU/768 MiB、Bert 1 vCPU/1536 MiB、BFS 1 vCPU/768 MiB、HTML 0.25 vCPU/768 MiB。真实负载使用 Azure Functions Trace 2021 中 bursty traces。
        - 关键实验设置与指标：主要指标包括 unplug latency、reclamation throughput、guest/host CPU utilization、function P99 latency、scale-down interference、limited-memory end-to-end latency、memory utilization GiB·s、1:1 vs N:1 cold start 和 memory footprint。microbenchmark 中，vanilla virtio-mem 回收 512 MiB 需要 617 ms，回收 2 GiB 近 2.5 s；Squeezy 平均快 10.9x，回收 2 GiB 只需 127 ms，并且在 64 GiB VM 中回收 2 GiB 时约 125 ms，基本不受 VM memory utilization 影响。真实 FaaS trace 下，Squeezy 的 memory reclamation throughput 平均比 virtio-mem 高 7x；当 HTML scale down 与 CNN 运行重叠时，vanilla virtio-mem 的 page migration 会让 CNN latency 超过 2x，而 Squeezy 几乎无干扰。host memory 受限时，virtio-mem 因慢回收使 P99 latency 平均变为 3.15x，HarvestVM-opts 为 1.36x，Squeezy 约 1.1x；Squeezy 的整体 memory footprint GiB·s 相比 HarvestVM-opts 和 virtio-mem 分别降低 45% 和 42.5%。与 1:1 microVM 相比，Squeezy 动态 N:1 scale up 的 cold start 平均快 1.6x、最高快 2.35x，且 1:1 实例平均占用 2.53x 内存。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：论文没有提出新的计算 kernel；最核心的新执行路径是 Squeezy 的 partition-aware VM memory hot(un)plug pipeline。它由 FaaS runtime 的 scale up/down 决策、hypervisor/virtio-mem plug/unplug 请求、guest OS 中的 Squeezy partitions、Squeezy syscall、page-fault-time partitioned allocation、shared partition file cache、reference-counter-based free partition tracking、以及 MADV_DONTNEED host release 组成。可以把它理解为把“函数实例生命周期”和“hotplugged memory block 生命周期”绑定起来，让内存回收不再依赖事后迁移零散页面，而是直接移除已经空掉的实例 partition。
        - 新pipeline/kernel的执行流例子：以一个 Bert 函数在 N:1 VM 中因请求 burst 扩容为例。runtime 发现没有 idle container 可服务新请求后，一方面通知 VM 内 Agent 创建新实例，另一方面通过 VMM 向 guest virtio-mem 发出 plug 请求，大小等于 Bert 的 1536 MiB memory limit。guest 收到 plug 后，Squeezy onlining path 把这段 hotplugged guest physical memory 关联到某个空 Squeezy partition，并唤醒等待 partition 的进程。Agent 创建 container 时调用 Squeezy syscall，请求把该进程标记为 Squeezy process；内核找到 populated free partition，把 partition id 写入 mm_struct。函数执行时，匿名页缺页进入 Squeezy page fault handler，只从该 private partition 分配；若触碰 runtime library 或 container root filesystem 的 file-backed mapping，则从 shared Squeezy partition fault in，并可被其他 Bert 实例复用。负载下降后，keep-alive 超时，Agent 销毁 idle Bert container，partition_users 降为 0，runtime 再通过 hypervisor 发出 unplug 请求。virtio-mem driver 根据 reference counter 找到 free partition，不扫描普通 ZONE_MOVABLE，也不迁移其他函数的 pages，直接 offline/remove 该 partition 对应的 memory blocks；hypervisor 收到完成通知后对对应 host virtual memory range 调用 madvise(MADV_DONTNEED)，将物理页返还给 host，供其他 VM 或后续 scale up 使用。
