## Prefill-decode Multiplexing（PD Multiplexing）

术语解释：
Prefill-decode multiplexing 是一种 LLM serving 范式，把同一 GPU 内的计算资源在 prefill 和 decode 两个阶段之间动态划分，而不是把两个阶段固定拆到不同实例，或把 prefill chunk 与 decode iteration 绑定到同一个 token budget。论文中的 MuxWise 将 prefill 和 decode 放在同一进程、同一显存地址空间中执行，只动态切分 SM 计算资源，因此可以同时适配负载变化并保留共享 KV cache pool。

术语关联术语的使用例子：
MuxWise 会先用 contention-tolerant estimator 判断 decode batch 在多少 SM 上可以满足 TBT SLO，例如 60% SM 给 decode，剩余 40% SM 给 prefill layers。两个阶段通过 GreenContext 绑定到不同 SM 集合并发运行，同时共享 KV cache pool，避免 disaggregated serving 中的 KV 迁移或重算。

涉及论文标题：
- Towards High-Goodput LLM Serving with Prefill-decode Multiplexing

## Goodput

术语解释：
Goodput 指在满足服务级目标约束下的有效吞吐量，而不是单纯 token throughput。论文把 high goodput 定义为在 TTFT / TBT SLO 保证下尽可能提升峰值服务能力，因此一个系统即使 GPU 利用率高，如果频繁违反 decode TBT，也不能算高 goodput。

术语关联术语的使用例子：
Chunked-prefill 需要较大 token budget 才能让 GPU 饱和，但该预算会让 TBT 超过 SLO；MuxWise 则通过 PD multiplexing 在保证 decode TBT 的同时用剩余 SM 推进 prefill，因此在论文实验中相对 baselines 平均提升 2.20x goodput。

- Towards High-Goodput LLM Serving with Prefill-decode Multiplexing

## Service Level Objective（SLO）

术语解释：
SLO 是在线 LLM serving 必须满足的延迟目标。论文区分 prefill 阶段的 Time-To-First-Token（TTFT）和 decode 阶段的 Time-Between-Tokens（TBT），并强调 TBT 比 TPOT 更严格，因为 TBT 按每个输出 token 计量尾延迟，不会被平均值掩盖。

术语关联术语的使用例子：
MuxWise 的 SLO-aware dispatcher 优先保证 decode TBT，例如 Llama3-8B 使用 50ms TBT SLO，Llama3-70B 使用 100ms TBT SLO；prefill 侧则尽早执行，并在短 prefill 请求可能错过 TTFT 时允许抢占长 prefill。

## Inflight Batching

术语解释：
Inflight batching 是 LLM serving 中把不同时间到达的请求合并到后续 decode batch 的系统机制。它可以提升 decode 阶段的计算利用率，但当新请求 prefill 很长时，原有 decode 可能被阻塞，从而造成 TBT SLO 违约。

术语关联术语的使用例子：
论文用典型服务流程说明：新请求到达后，系统可能暂停当前 decode 来执行 prefill，再把新请求并入 decode batch。MuxWise 通过 layer-wise prefill execution 和 query-based synchronization 降低这种合并过程造成的 GPU bubble。

## Disaggregated Serving

术语解释：
Disaggregated serving 将 prefill 和 decode 拆到不同实例或 GPU 组上，以便分别满足两个阶段的 SLO。论文指出这种方式容易把 compute management 和 memory management 绑定在一起：GPU 资源静态或半动态分配时容易空闲，KV cache pool 也会被拆小或需要迁移。

术语关联术语的使用例子：
Splitwise 式静态 disaggregation 为 prefill instance 和 decode instance 分配固定 GPU，负载波动时会出现空闲；LoongServe 可以动态调整 GPU 资源，但会释放原 GPU 上的 KV cache，导致多轮请求的 cross-request KV cache reuse 失效。MuxWise 用 intra-GPU PD multiplexing 避免这些问题。

- Towards High-Goodput LLM Serving with Prefill-decode Multiplexing

## Chunked-prefill

术语解释：
Chunked-prefill 将 prefill 阶段切成多个 chunk，并把每个 chunk 与 decode iteration 融合执行，用 token budget 限制 decode 延迟。论文认为该方法在 SLO compliance 和 GPU utilization 之间存在内在矛盾：预算太小无法饱和 GPU，预算太大又会拉高 TBT。

术语关联术语的使用例子：
论文给出 Llama3-70B 在 8 张 A100 上的例子：约需 4K token budget 才能饱和 GPU，但 100ms TBT SLO 下可接受的预算约为 256。长 reused context 还会让 prefill chunk 重复访问历史 KV cache，使 TBT 继续恶化。

- Towards High-Goodput LLM Serving with Prefill-decode Multiplexing

## KV Cache Pool

术语解释：
KV cache pool 是 serving 系统中存放 attention key/value 中间结果的共享显存池，用于避免重复计算并支持多轮请求复用上下文。论文强调高 goodput 不只依赖计算资源，还依赖能否在 prefill、decode 以及跨请求之间保持高效 KV cache reuse。

术语关联术语的使用例子：
在 disaggregated serving 中，每个实例维护独立模型权重和 KV cache pool，有效 cache 容量变小，论文示例中 cache hit rate 可从 36.6% 降到 4.2%。MuxWise 让 prefill 和 decode 在同一进程共享显存空间，因此只切 compute partition，不切 KV cache pool。

- Towards High-Goodput LLM Serving with Prefill-decode Multiplexing

## Bubble-less Multiplex Engine

术语解释：
Bubble-less multiplex engine 是 MuxWise 的执行协调模块，目标是在 prefill 与 decode 空间并发时减少 GPU bubbles。它把 prefill 拆成 transformer layer 粒度执行，并配合 query-based synchronization，让 prefill 的可调度粒度更接近 decode iteration 的时间尺度。

术语关联术语的使用例子：
当 decode iteration 很快结束而完整 prefill 仍在运行时，普通 multiplexing 会留下 decode SM 空闲。MuxWise 只发射若干 prefill layers，使这些 layers 的执行时间覆盖一个 decode 窗口；最后一层完成后通过 CUDA event polling 合并到 inflight decode batch。

## Contention-tolerant Estimator

术语解释：
Contention-tolerant estimator 是 MuxWise 用于 SLO 安全调度的延迟估计模块。它先通过 offline profiling 建立 prefill / decode solo-run latency predictor，再对 decode 叠加 contention guard，得到共享带宽干扰下的 worst-case latency estimate。

术语关联术语的使用例子：
Estimator 使用 reused length、new input length、output length、decode batch size 和 SM partition configuration 等变量估计延迟。SLO-aware dispatcher 根据 worst-case decode latency 选择 best-fit SM 数，避免空间并发时 memory bandwidth contention 造成 TBT SLO 违约。

## SLO-aware Dispatcher

术语解释：
SLO-aware dispatcher 是 MuxWise 的在线请求调度器，在每个 prefill batch 完成或 decode iteration 结束后重新选择 multiplexing plan。它优先为 decode 保留刚好满足 TBT SLO 的 SM，再把剩余 SM 分配给 prefill，并决定本轮发射多少 prefill layers。

术语关联术语的使用例子：
论文示例中，dispatcher 先给 decode batch 分配 60% SM、prefill batch P0 分配 40% SM；P0 完成并入 decode 后，decode batch size 变大，dispatcher 可能改为 80% SM 给 decode。若短 prefill P2 即将错过 TTFT，dispatcher 可让 P2 非递归抢占长 prefill P1。

涉及论文标题：
- Towards High-Goodput LLM Serving with Prefill-decode Multiplexing
