论文标题：Bullet: Boosting GPU Utilization for LLM Serving via Dynamic Spatial-Temporal Orchestration

    1、论文工作：
        - 论文要解决的核心问题：现代 LLM serving 中，prefill 阶段通常是 compute-intensive，decode 阶段通常是 memory-bound，二者对 GPU SM、Tensor Core、HBM 带宽的需求互补，但现有系统没有把这种互补性稳定转化为 GPU 利用率。生产系统常用的 chunked prefill / hybrid batching 用固定 token budget 把 prefill chunk 与 decode token 绑在同一个 lock-step batch 中，虽然可以控制 TPOT，却牺牲 TTFT、prefill 吞吐和 GPU 利用率；在长上下文、突发请求、多样化输入长度下，会造成 prefill 排队、decode batch 变大、尾延迟放大。
        - 论文的主要贡献：提出 Bullet，一个面向 LLM serving 的 spatial-temporal orchestration 系统。它在同一 GPU 内把 prefill 与 decode 拆成可并发执行的两个 engine，通过 layer-wise prefill 调度、step-wise decode 调度、动态 SM partition 和 SLO-aware scheduling，在满足 TTFT / TPOT 约束的同时提升 GPU 利用率。论文贡献可以概括为：识别 chunked prefill 的硬件利用率瓶颈；提出 SM-scaling roofline model（SRM）和轻量 calibration 来预测不同 SM budget 下的 latency；设计 SLO-aware task scheduler 与低开销 resource manager；在 SGLang + PyTorch 上实现系统，并在真实 workload 上验证 throughput / goodput / latency 改善。
        - 论文所处背景：该工作属于 LLM inference serving、GPU resource management、prefill-decode co-location / disaggregation 的交叉方向。它直接对比或讨论 vLLM、SGLang、NanoFlow、MuxServe、Semi-PD、Drift，以及 Mooncake 支持的 disaggregated prefill。与把 prefill 和 decode 放到不同 GPU / 节点的 disaggregation 不同，Bullet 关注 intra-GPU prefill-decode disaggregation：在同一个 GPU 上用 SM 级分区和运行时调度让两个阶段并发。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：第一，prefill 本身并没有充分利用 GPU。论文指出 wave quantization 与 attention bottleneck 会让完整 Transformer layer 的 compute utilization 只有约 70%-76%，短序列或小 chunk 下 O-proj / attention 更容易出现 SM idle。第二，chunked prefill 用固定 token budget 先填 decode，再把剩余预算分给 prefill，长 prompt 会被拆成多轮 chunk；每个新 chunk 的 attention 又要访问先前 chunk 的 KV cache，形成近似 N(N+1)/2 次 KV reload。论文在 16k-token prefill 上观察到 1k chunk 的 compute efficiency 从 71% 降到 61%，最后一个 chunk 的处理时间达到第一个 chunk 的 1.9x，总 prefill latency 比 unchunked 高 1.13x；2k chunk 虽缓解利用率下降，但平均 per-chunk latency 增加 1.86x。第三，已有系统各有偏置：NanoFlow 靠固定 overlap pipeline，MuxServe 靠固定 SM quota，disaggregated serving 需要跨实例 KV / state 迁移和手动资源调参，难以同时适应动态 workload、TTFT / TPOT SLO 和快速重分区。
        - 论文的设计方法：Bullet 用四个组件组成闭环。Performance estimator 先用 SRM 建模 compute、memory、network bandwidth 随 SM 数变化的性能上界，再用少量 offline concurrent samples 校准 prefill/decode interference，并用在线统计持续修正。SLO-aware scheduler 周期性读取全局状态，预测 TTFT / TPOT，重排 waiting requests，并为下一批 prefill layer 或 decode step 搜索合适的 SM 分区。Resource manager 通过 SM masking / `libsmctrl_set_stream_mask` 修改 CUDA stream metadata，使后续 kernel 限制在指定 SM 子集执行。Concurrent execution engine 把 prefill 和 decode 放在独立进程 / worker 中，使用共享 CPU metadata buffer、统一 GPU memory pool、CUDA IPC、ZeroMQ metadata 传递来协同执行。
        - 方法如何对冲 Baseline 缺陷：相对 chunked prefill，Bullet 不把 prefill 进度绑定到固定 decode token budget，而是在不同 SM 子集上并发运行 prefill layer 和 decode step，从机制上减少 chunk 切分引入的重复 KV reload 和 lock-step 等待。相对固定 SM sharing，Bullet 按实时队列、执行进度和 SLO 压力动态调节 SM：prefill 队列堆积时可以把 prefill SM 提高到接近全 GPU 以快速清队列；decode TPOT 接近 SLO 边界时减少 prefill SM 或增加 decode SM。相对纯 profiling，SRM 把 SM 数、算术强度、memory / network bandwidth saturation 关系纳入模型，只需少量校准样本，论文声称 profiling overhead 低于 1 小时，预测/更新开销为微秒级。
        - 关键 trade-off：Bullet 的收益来自更复杂的 runtime control plane 与更依赖硬件/驱动能力的 SM partition 机制。它依赖 MPS、CUDA stream mask、libsmctrl 这类非普通 LLM serving 框架默认抽象，未来驱动兼容性和平台可移植性是风险。prefill/decode 解耦后可能带来额外 memory traffic，例如独立 weight loading，但论文认为相比 chunked prefill 中 Tensor Core 因重复 KV reload 被闲置，这个 trade-off 是值得的。系统还需要针对模型、kernel 库、硬件平台做 profiling / calibration；新 attention 变体、LoRA serving、量化 kernel、MoE routing 变化都可能引入新的误差。对于 DeepSeek-MLA 这类天然更适合 disaggregation 的特殊架构，Bullet 的 co-located 方案未必达到专用解耦系统的峰值性能。

    3、论文实现：
        - Baseline 如何实现：单 GPU 实验比较 vLLM v0.8.5（1024 chunk）、SGLang v0.4.6 + FlashInfer v0.2.7（1024 / 2048 chunk）、NanoFlow（1024 chunk）和 Bullet。多 GPU 实验还比较基于 SGLang + Mooncake 的 xP-yD disaggregated-prefill 配置，例如 1P1D、3P1D、6P1D、30P6D。chunked prefill baseline 通过固定 chunk size / token budget 形成 hybrid batch；NanoFlow 在 chunked prefill 基础上做静态 kernel overlap；disaggregated baseline 把 prefill 与 decode 放到不同实例或节点，依赖跨阶段状态 / KV 管理。
        - 新设计如何实现：Bullet 实现在 SGLang v0.4.6 和 PyTorch 2.6.0 之上，约 4100 行 Python 代码，并集成修改版 libsmctrl 做 GPU resource allocation。prefill / decode engine 是两个 SGLang workers，MPS 用于 spatial sharing。GPU memory 由初始化进程预先分配模型权重和 KV cache，再通过 CUDA IPC 在两个 engine 间共享；CPU 侧用 OS-managed shared memory 保存全局状态与 metadata；请求 metadata 用 ZeroMQ 异步传递，避免 KV cache transfer。调度层面，prefill 以若干 layer 为粒度运行并同步回 CPU 决策，decode 使用 CUDA Graph 发射一个 step 的小 kernel，降低 launch overhead。资源层面，scheduler 下发 repartition command 后，resource manager 立即修改相应 CUDA stream 的 SM mask，使后续 kernel 只在新分配的 SM 子集运行。
        - 实验 / 实现平台：实验使用三类服务器：8xA100-80GB（108 SM/GPU，intra-node bandwidth 20 GB/s）、8xH100（132 SM/GPU，600 GB/s）、8xH20（78 SM/GPU，intra-node 400 GB/s，inter-node 200 GB/s），CUDA 12.4。模型包括 Llama3.1-8B、Llama3.1-70B，以及 Qwen3-235B-A22B-FP8 MoE。workload 包括 ShareGPT、Azure-Code、arXiv-Summary，并补充生产 RAG trace。Artifact Appendix 的复现实验聚焦单张 NVIDIA A100 80GB、Llama3.1-8B、ShareGPT，要求 Debian 5.10、CUDA 12.4、Python 3.12.9、PyTorch 2.6.0、CMake 3.17、GCC 10.2.1、约 20GB 磁盘，运行 `artifact_evaluation/run_all.sh` 生成日志、JSON 结果和图。
        - 关键实验设置与指标：指标包括 TTFT、TPOT、throughput、P90 SLO attainment、normalized input/generation latency，以及 Nsight Systems 采集的 SM / Tensor Core / memory bandwidth utilization。论文摘要声称 Bullet 在真实 workload 上平均 throughput gain 为 1.26x，最高 1.55x。单 GPU A100 + Llama3.1-8B 上，相比 SGLang-1024，Bullet 平均 throughput 提升 1.09x，最高 1.20x，平均端到端 speedup 为 1.86x；ShareGPT 20 req/s 下 mean TTFT 为 0.16s、P90 为 0.31s，相比 SGLang-1024 分别好 54.9x 和 78.5x，SLO compliance 提升 1.49x。多 GPU A100 + Llama3.1-70B 上，ShareGPT 3.0 req/s 时 Bullet mean latency 173 ms/token、P99 223 ms/token，低于 vLLM 的 207 ms/token 和 SGLang 的 319 ms/token；Azure-Code 中 Bullet 在 0.5 req/s 仍保持 203 ms/token，而 vLLM / SGLang 在 0.3 req/s 已达 213 / 291 ms/token。H20 + Qwen3-235B-A22B 上，Bullet 在 4.0 req/s 达到 110 ms/token、1.4s TTFT、45ms TPOT，相比 vLLM 分别低 17.7x、16.2x、4.2x。开销方面，metadata send/recv 平均 0.21 ms，performance prediction 平均 10.2 us，resource re-config 平均 4.1 us；并发时间片中 Bullet 平均 active SM cycles 为 86.2%，比 SGLang 高 11.2%，Tensor Core utilization 高 11.8%，memory-bandwidth utilization 高 19.3%。

    4、pipeline/kernel解析：
        - 新 pipeline/kernel 是什么：论文没有提出单个全新 CUDA kernel，而是提出一个新的 serving runtime pipeline：intra-GPU prefill-decode disaggregation + layer-wise spatial-temporal orchestration。这个 pipeline 的核心不是改写 attention / GEMM kernel，而是用现有高性能 kernel 库，在运行时把 prefill 与 decode 的 kernel stream 分配到不同 SM mask 上，并以 SLO-aware scheduler 控制每个 prefill layer group 与 decode CUDA Graph step 的发射时机。可以把它理解为一个“动态 SM 分区的 prefill/decode 并发执行 pipeline”。
        - 新 pipeline/kernel 的执行流例子：假设一个新请求到达。首先，frontend 将请求放入 waiting queue，prefill scheduler 从共享 metadata buffer 读取当前 decode batch size、TPOT、已有 prefill 进度和队列状态，并用 estimator 预测不同 request ordering、layer group、SM partition 下的 TTFT / TPOT。若当前 prefill 队列压力较大且 decode TPOT 仍满足 SLO，scheduler 会给 prefill 分配更多 SM，resource manager 调用 libsmctrl 修改 prefill CUDA stream mask；prefill engine 随后在这些 SM 上执行若干 Transformer layer 的 QKV、attention、O-proj、MLP 等 kernel。与此同时，decode engine 将当前 decode batch 封装为 CUDA Graph step，在 decode stream 对应的 SM 子集上生成下一 token。每个 scheduler step 后，prefill / decode engine 把进度、latency、batch 状态写回共享 CPU buffer；如果 decode P90 TPOT 接近阈值，下一轮调度会减少 prefill SM 或提高 decode SM。某请求完成 prefill 后，不复制 KV cache，只通过共享 GPU address space 与 metadata 把它转交给 decode engine；decode engine 将其合入后续 running batch，继续 step-wise 生成 token。这个流程通过空间上的 SM mask 分区和时间上的 layer/step 调度，把原来 chunked prefill 的 lock-step hybrid batch 改成可动态重分区的并发 pipeline。
