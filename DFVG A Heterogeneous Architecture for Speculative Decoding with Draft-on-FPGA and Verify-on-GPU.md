论文标题：DFVG: A Heterogeneous Architecture for Speculative Decoding with Draft-on-FPGA and Verify-on-GPU

    1、论文工作：
        - 论文要解决的核心问题：传统 LLM 自回归解码每个 token 都要一次完整 forward，延迟高且硬件利用率低；speculative decoding 通过小 draft model 先生成候选 token，再由大 target model 并行验证，可以保持输出分布不变并提高吞吐。但现有 speculative decoding 系统仍有三个关键瓶颈：draft 与 verify 模型的资源需求严重不匹配，固定 token tree 生成大量低质量候选，draft/verify 两阶段异步切换导致等待、通信和 rollback 开销。
        - 论文的主要贡献：DFVG 提出一种 Draft-on-FPGA、Verify-on-GPU 的异构 speculative decoding 架构。它把轻量、低延迟、分支并行的 draft 生成放到 FPGA，把大模型 verification 放到 GPU；用 ADAPT 动态生成硬件感知的 token tree；用 TreeSort-Verify 把不规则 tree attention 重排为更适合 GPU block-parallel 执行的结构；再用跨设备 compact pipeline 让 FPGA drafting 与 GPU verification/forwarding 重叠。论文报告端到端 latency speedup 为 2.44x 到 3.26x，energy efficiency 提升 4.33x 到 5.79x，FPGA runtime power 约 75W。
        - 论文所处背景：论文面向 LLM serving 的解码阶段，尤其是需要连续生成 token 的在线推理。Speculative decoding 的理论收益取决于 draft/target 模型速度比、draft 长度以及 acceptance rate；当 draft model 足够快且 token 被 target model 接受概率高时，大模型一次 forward 可以验证多个候选 token。但实际系统中，小模型 draft 通常 autoregressive、序列长度近似为 1、数据复用低，更偏带宽/低延迟；大模型 verify 则更像 prefill，对矩阵计算吞吐要求高。单一 GPU 或 CPU 很难同时匹配这两种行为，因此论文主张用 FPGA 的细粒度流水和低功耗并行去承接 draft，用 GPU 的 Tensor/矩阵算力承接 verify。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：第一类 baseline 是普通 autoregressive decoding，如 vLLM、LLaMA.cpp、GPT-Fast 等高效框架，虽然通过 PagedAttention、C/C++ 实现或 kernel/operator fusion 优化单步解码，但本质仍是逐 token 生成，难以摊薄大模型 forward 成本。第二类 baseline 是传统 speculative sampling 或 SpS，draft 之后立即 verify，两个阶段相互等待，论文 profiling 显示 LLaMA-7B 在 RTX 4090 上 GPU utilization 只有 51.72%，有时低于 10%，同时 draft 和 verify 模型显存长期占用。第三类 baseline 是 tree-based 或分布式 speculative decoding，如 SpecInfer、DuoDecoding、Dovetail、SpecPIM 等；它们要么使用固定树形分支，不能根据 token confidence 和硬件并行预算动态调整，要么采用 CPU/GPU、多 GPU 或 PIM 等异构方式但仍存在阶段解耦、通信、rollback 和资源不匹配问题。
        - 论文的设计方法：DFVG 的核心方法分为算法、GPU verification、FPGA architecture 和 runtime pipeline 四层。算法层用 ADAPT 建模 budget-constrained token tree selection，在计算预算、每层最大分支数、pipeline 深度下最大化期望通过验证的 token 数，并用 temperature-controlled probabilistic sampling / Gumbel sampling 近似求解。GPU 层用 TreeSort-Verify 将 token tree path-packing，重排为 block-diagonal lower triangular mask，使多个 block 可独立调用优化过的 GEMM/attention 路径。FPGA 层用 multi compute core overlay processor 承接 draft model，包含 HBM 数据读取、systolic PE array、parallel adder tree、SFU、KV-cache management、dynamic token management 和 branch management。系统层用 CPU host、PCIe Gen4 x16、共享 host buffer、BAR 状态寄存器、中断和 DMA 连接 FPGA 与 GPU，让两端异步但紧密协同。
        - 方法如何对冲 Baseline 缺陷：相对普通 AR，DFVG 通过一次 GPU verification 接受多个 draft token，减少大模型 forward 次数。相对固定 speculative tree，ADAPT 根据 draft confidence 和 FPGA 并行上限动态调整分支数和深度，高置信位置生成更多可用候选，低置信位置减少无效分支，避免把硬件预算花在低 acceptance token 上。相对不规则 tree verification，TreeSort-Verify 把拓扑相关的稀疏 causal mask 转为连续 block，提升 GPU memory locality 和 cuBLAS/GEMM 利用率。相对 CPU/GPU 或单 GPU 混部，FPGA 负责轻量 draft 可减少 GPU 显存/算力竞争；compact pipeline 让 GPU 在等待 draft 时继续从已验证 prefix forward，让 FPGA 在 GPU verification 时继续生成下一批候选，减少 pipeline bubble。
        - 关键 trade-off：第一，DFVG 依赖额外 FPGA 资源和 PCIe/host runtime，部署门槛高于纯 GPU 框架。第二，ADAPT 的收益依赖 draft model 与 target model 的 token 分布一致性；当 temperature 很高、输出随机性强或模型 pair 不匹配时，acceptance rate 会下降，rollback 频率增加。第三，FPGA overlay 占用资源很高，论文 V80 实现约占 89.6% LUT、90.9% FF、8192 DSP，并使用大量 BRAM/URAM 存 KV cache，后续扩展到更大 draft model 会受片上资源约束。第四，系统主要证明 single-round speculative decoding，论文 future direction 才讨论 multi-round speculative decoding、多 FPGA/GPU 和更多模型族支持。第五，PCIe Gen4 x16 在论文 workload 下不是瓶颈，但跨设备方案仍受互连延迟、DMA 调度和 host 控制路径影响。

    3、论文实现：
        - Baseline 如何实现：论文 baseline 包括 AR token-by-token decoding、SpS 原始 speculative sampling、DuoDecoding、SpecInfer，以及 vLLM、LLaMA.cpp、GPT-Fast 等推理框架。端到端实验中，AR 与 SpS 作为基础生成策略，DuoDecoding 代表 CPU+GPU 异构 speculative decoding，SpecInfer 代表多 GPU tree-based speculative verification；vLLM、LLaMA.cpp、GPT-Fast 用于和成熟软件框架的 speedup 对比。论文说明为公平起见使用足够 GPU，Fig. 14 中 speedup 和 energy efficiency 均相对 AR baseline 报告。
        - 新设计如何实现：FPGA 端实现为 V80 FPGA 上的 custom micro-architecture，Verilog HDL 编写并综合到 300MHz，包含多核 overlay PE array、HBM/AXI NoC 数据通路、SFU、parallel adder tree、输出 buffer、KV-cache pruning/contiguous allocation、token/branch management。PE 微架构包含 branch concatenation 支持多分支矩阵拼接，以及利用 BF16 7-bit mantissa 特性的 DSP packing，使一个 DSP 支持两个 BF16 x BF16 操作以提高并行度。GPU 端实现 TreeSort-Verify framework，包含统一 KV-cache 管理和 NCCL 多 GPU 同步，verification 前先 PathPacking 和 GroupSiblings，再进行 ParallelBlockAttention 与 VerifyTokens。CPU runtime 负责 FPGA/GPU 之间的 task orchestration、BAR 状态寄存器轮转/中断处理、共享 buffer 和 DMA 传输。
        - 实验 / 实现平台：主要平台为 Intel Xeon 4310 CPU、NVIDIA RTX 4090 GPU、AMD/Xilinx V80 FPGA；论文也列出 A100 GPU 与 U200 FPGA 对比。表 2 中 RTX 4090 有 24GB DRAM、约 1008GB/s 带宽、PCIe 64GB/s；A100 有 80GB DRAM、约 1935GB/s 带宽；V80 FPGA 包含 10848 个 DSP58、32GB + 32GB DRAM/HBM、约 51GB/s 与 820GB/s 带宽、PCIe 64GB/s，TDP 190W，但优化后推理 runtime power 约 75W。artifact 附录写明运行环境为 Ubuntu 20.04、CUDA 12.1、Xilinx Runtime 2024.1，编译包含 Xilinx Vivado 2024.1、GCC 9.4.0、C++/Verilog/GPU kernels/host controller。
        - 关键实验设置与指标：模型包括 Vicuna-7B/Vicuna-160M、LLaMA-2-7B/LLaMA-160M、OPT-13B/OPT-125M、Qwen3-8B/Qwen3-0.6B 等 target/draft pair；数据集按 Spec-Bench 使用 MT-Bench、Translation、Summarization、Question Answering、Math Reasoning、RAG。指标包括 end-to-end speedup、energy efficiency ratio、tokens/s、acceptance rate、ablation speedup、通信占比和 FPGA resource utilization。每个实验在相同 thermal/scheduling 条件下重复 5 次取平均，能耗用 Xilinx `xbutil` 和 `nvidia-smi` 以 1ms 采样。Qwen3-0.6B/Qwen3-8B 的 token-level acceptance rate 稳定在 75%-85%；端到端 speedup 为 2.44x-3.26x，energy efficiency 为 4.33x-5.79x。ablation 显示 HW-Branch 达到 2.21x，加入 TreeSort-Verify 到 2.46x，加入 MCore-Acc 到 3.08x，最终 Pipe-Overlap 到 3.26x。通信开销在 1.08%-3.2% 范围内，论文认为 PCIe Gen4 x16 已满足该系统数据传输需求。

    4、pipeline/kernel 解析：
        - 新 pipeline/kernel 是什么：论文的核心新 pipeline 是 FPGA-GPU compact heterogeneous speculative decoding pipeline：FPGA 端负责硬件感知动态 draft tree 生成，GPU 端负责 TreeSort-Verify 并行验证，CPU/PCIe runtime 负责跨设备 token stream 和状态反馈。核心 kernel/执行路径包括三部分：ADAPT 生成动态 tree，TreeSort-Verify 将 tree verification 转成 block-parallel attention，FPGA multi compute core overlay processor 执行 draft model 的多分支 token generation。它不是单个 CUDA kernel，而是算法、FPGA 数据流、GPU attention verification 和跨设备调度组成的系统级执行流。
        - 新 pipeline/kernel 的执行流例子：以一次解码迭代为例，输入 prefix 先进入 FPGA draft model。FPGA 根据当前 token logits/confidence 运行 ADAPT，受计算预算、最大分支数和最小 pipeline 深度约束生成 token tree；branch management 模块决定每层选哪些候选，multi-core overlay 把多个分支映射到 PE array 并行计算，KV-cache management 把候选分支的 K/V 写入 temp buffer。随后 FPGA 将 token IDs、可选 logits/confidence 和元数据写入共享 host buffer，更新 BAR status 并触发中断；CPU 发起 DMA，使 GPU 获取候选 tree。GPU 先用 PathPacking 对 tree 节点重排，再用 GroupSiblings 形成连续 block，将原本不规则的 tree causal mask 转为 block-diagonal lower triangular mask；接着 ParallelBlockAttention 用优化矩阵算子验证候选 token，并根据 target/draft 概率比执行 acceptance 判断。若 token 被接受，GPU 返回 accepted prefix，FPGA 将对应 KV 从 temp buffer 合并到 prune/accepted buffer 并继续从新 prefix 生成；若某分支被拒绝，FPGA 依据返回 prefix length 回滚本地 token/KV 状态。整个过程中，GPU 在验证候选时 FPGA 可以继续 draft 下一批 token；如果 GPU 验证完成而 FPGA 尚未请求验证，GPU 可从已确认 prefix 继续 target forward，避免等待。最终墙钟时间不是 draft 时间与 verify 时间简单相加，而是由重叠后的最长关键路径、acceptance rate 和少量 PCIe/rollback 开销共同决定。
