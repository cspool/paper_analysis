论文标题：ARIADNE: Adaptive UVM Management for Efficient GPU Memory Oversubscription

    1、论文工作：
        - 论文要解决的核心问题：NVIDIA Unified Virtual Memory（UVM）虽然能让 GPU 在显存不足时继续通过主机内存运行，但在 GPU memory oversubscription 下会因为 page fault handling 延迟和 thrashing 出现严重性能退化。Baseline UVM 以 2 MB VABlock/chunk 粒度迁移数据，而实际 GPU fault 是 4 KB page 粒度；稀疏访问时只需要少量 page，却会占用整块 2 MB GPU chunk，导致 Working Chunk Set Size（WCSS）被放大，进而反复 eviction/refetch。
        - 论文的主要贡献：提出 ARIADNE，一个完全在 NVIDIA open-source UVM host driver（nvidia-uvm）内实现的 runtime UVM management framework。它的三个核心机制是：1）pipelined VABlock fault handling，将 Populate、Copy、Eviction 解耦并跨 VABlock 重叠执行；2）Sharing Degree，用最近 fault 的 source uTLB ID 数量作为每个 VABlock 的线程级访问局部性指标；3）基于 WCSS、Sharing Degree-aware eviction queue 和 transient Zero-copy 的动态 VABlock placement。
        - 论文所处背景：GPU 计算负载的模型和数据规模持续增长，但板载显存容量有限。UVM 通过 cudaMallocManaged 和 driver 级迁移降低编程负担，并允许 memory oversubscription；但高内存压力下，UVM 的长 fault handling 路径、粗粒度 chunk allocation、last-fault-time victim selection 和 speculative prefetcher 会共同放大 CPU-GPU 数据搬运和 fault 停顿。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：Baseline UVM 在 fault handling 中顺序执行 Populate、Eviction、Copy。论文测得 Populate 是主要延迟来源，接近 Copy/Eviction 的两倍，因为它包含 host-side unmap 等不可避免操作；处理一个 VABlock fault 需要几十微秒，一批最多 256 个 fault 可累积到毫秒级。Baseline 还按 last fault time 选择 eviction victim，容易淘汰仍活跃的 VABlock。Access Counter-based migration（AC）用固定阈值把 hot page 从 Zero-copy 迁入 GPU，在部分 oversubscription 条件下有效，但阈值不随访问模式和内存压力变化；SUV 等 compiler-assisted/static 方法需要代码或编译器介入，且静态估计难以跟随 runtime phase shift。
        - 论文的设计方法：ARIADNE 在 driver 中维护每个 VABlock 的 Sharing Degree，即最近 16 次 fault 的 source uTLB ID 去重数量。uTLB ID 被用作 SM/thread group 的近似代理，高 Sharing Degree 表示多个 SM 同时访问同一 VABlock，通常意味着 VABlock 内部 page 利用率高，适合迁入 GPU；低 Sharing Degree 表示稀疏访问，更适合留在 host memory 以 Zero-copy 的 128 B cache-line 粒度访问。ARIADNE 同时估计 WCSS，并在超出可用 GPU memory 时把低价值或刚被 evict 后又被访问的 VABlock 临时置为 Zero-copy。
        - 方法如何对冲 Baseline 缺陷：pipelined fault handling 将 Populate 与后续 Copy/Eviction 分离，使 VABlock A 的 Copy、VABlock X 的 Eviction 和 VABlock B 的 Populate 可以并发，隐藏 Populate 长延迟。Sharing Degree-aware eviction queue 把 last fault time 与 Sharing Degree 结合，高 Sharing Degree 的 VABlock 在 GPU 中保留更久，减少活跃数据被错杀。Dynamic Zero-copy 给刚被 evict 但又被访问的 VABlock 一个 100 ms 的 host-side second chance，避免马上 refetch 到 GPU 造成 eviction-fetch 循环。
        - 关键 trade-off：ARIADNE 保留 UVM 的透明性，不需要改硬件、编译器或应用代码，但它依赖 NVIDIA open kernel module driver 内部机制，属于 driver 原型而非通用产品。Zero-copy 能降低粗粒度迁移和 thrashing，但对稠密访问会比 GPU memory 慢，因此需要 Sharing Degree 判断。参数也有经验性：最近 fault history 长度为 16，低平均 Sharing Degree 阈值为 3，Zero-copy Pintime 为 100 ms，SD Weight 为 100 us；论文通过 sensitivity analysis 说明这些值较稳定，但跨硬件和 workload 仍可能需要重新评估。

    3、论文实现：
        - Baseline 如何实现：Baseline UVM 使用 NVIDIA open-source kernel module v535.86 的默认 UVM 系统。AC 使用 NVIDIA UVM driver 中的 Access Counter-based migration 机制。SUV 使用其提供的 API 修改 benchmark code，配置 GPU free memory size 和 memory footprint 等 compile-time 变量后重新编译；论文强调 SUV 对环境要求不同，因此 artifact 中将 SUV 对比作为可选部分。
        - 新设计如何实现：ARIADNE 通过修改 UVM host driver 约 1600 LOC 实现，不增加 GPU 硬件资源，也不要求应用重新编译。Sharing Degree 在 nvidia-uvm 中按 VABlock 维护：driver 为每个 VABlock 记录最近 16 个触发 fault 的 source uTLB ID，并计算 unique uTLB ID 数。WCSS 由 GPU-resident VABlock、Zero-copy VABlock、以及最近 evict 后高概率 re-access 的 VABlock 组成；若初始阶段平均 Sharing Degree 低于 3，ARIADNE 将最近 evict 的 VABlock 在 WCSS 中保留 500 ms，以缓解初始 thrashing。Dynamic Zero-copy 通过 UVM 内部 API 设置/取消目标 VABlock 的 Zero-copy 状态，相关 metadata 按 GPU 保存在 driver 中，不占用 GPU memory。
        - 实验 / 实现平台：主评估平台为 NVIDIA RTX A5000 GPU、AMD Ryzen 7700X CPU、PCIe 4.0 x16、64 GB DDR5 DRAM、Linux kernel 6.0、NVIDIA open-source kernel driver 535.86。artifact 附录列出 CUDA 12.1、GCC/NVCC、NVIDIA open-kernel modules driver、RTX A5000；SUV 实验需要 Linux 6.2、CUDA 11.8、driver v525。benchmark 来自 Polybench、Rodinia、HeCBench 和 XSBench，共 10 个：2DC、ATAX、BICG、GEMM、GEMV、MVT、XSB、BFS、HEL、NW，每个 memory footprint 设为 4 GB。
        - 关键实验设置与指标：oversubscription ratio 定义为 memory footprint / GPU memory size，通过 cudaMalloc 预留部分 GPU memory 来模拟 no oversubscription、130%、175%、300% 等内存压力。主要指标是 benchmark CUDA kernel 端到端执行时间，同时分析 VABlock fault handling latency、PCIe traffic、component breakdown、参数敏感性和跨 RTX 2060/3070/A5000 的可迁移性。ARIADNE 相对 AC 在 130%、175%、300% oversubscription 下分别取得 1.9x、2.3x、4.0x geomean speedup；相对 SUV 分别为 1.9x、5.0x、4.8x。相比 no-oversubscription，ARIADNE 在 130%、175%、300% 下 runtime 只增加 1.6x、1.8x、2.3x，呈近线性退化。Pipelined VABlock fault handling 平均降低 17% fault handling latency，最高在 BFS 上降低 48%；ARIADNE 的 PCIe traffic 平均约为 AC 的 51%。Llama3.1 70B inference 中，ARIADNE 在 Decode 和 Prefill 阶段分别比 AC 快 4.2x 和 1.6x。

    4、pipeline/kernel 解析：
        - 新 pipeline/kernel 是什么：论文没有提出新的 CUDA kernel 或应用算子，而是提出新的 UVM driver runtime path，可称为 ARIADNE pipelined VABlock fault handling + dynamic placement pipeline。这个 pipeline 把传统 UVM 单体式 fault handling 拆成 Populate、Copy、Eviction 三条可重叠路径，并在每次 VABlock fault 处理时更新 Sharing Degree 和 WCSS，随后决定该 VABlock 应继续迁入 GPU memory、被 evict、还是临时进入 Zero-copy。
        - 新 pipeline/kernel 的执行流例子：一个 CUDA kernel 访问 managed memory 中尚未驻留 GPU 的 page，GPU 以 4 KB 粒度产生 page fault，并把 fault 写入 fault buffer；UVM driver 一次最多取出 256 个 fault，并按 2 MB VABlock 聚合。对某个 VABlock A，ARIADNE 首先在 Populate stage unmap host pages、分配 GPU chunk、准备迁移 metadata，但不立即 Copy；随后 VABlock A 进入 Copy stage，由 copy engine 将 fault page 和 prefetcher 选择的 page 迁入 GPU，并更新 page table。如果 GPU free chunk 低于阈值，专用 Eviction thread 并行运行，按 last fault time + SD Weight * Sharing Degree / Nfault_history 的 priority key 选择 victim，例如低 Sharing Degree 的 VABlock X 更容易被 evict。与此同时，driver 可以开始处理 VABlock B 的 Populate，因此 A 的 Copy、X 的 Eviction、B 的 Populate 重叠执行。处理完一批 fault 后，ARIADNE 检查 WCSS 是否超过可用 GPU memory；若某个刚被 evict 的 VABlock 又被访问，且当前内存压力仍高，它不会马上被 refetch 到 GPU，而是进入 100 ms transient Zero-copy，GPU 后续以 cache-line 粒度从 host memory 读它。如果这段时间过后仍被访问，它才被重新提升回 GPU memory；如果不再活跃，则释放 Zero-copy 状态，避免无效占用管理资源。
