论文标题：Towards Compute-Aware In-Switch Computing for LLMs Tensor-Parallelism on Multi-GPU Systems

1、论文工作：
    - 论文要解决的核心问题：大规模 LLM 的 Tensor Parallelism 在推理和训练中频繁引入 AllReduce、AllGather、Reduce-Scatter 等 collective 操作，通信位于 GEMM 等计算 kernel 的关键路径上。NVLink / NVSwitch 和 NVLink SHARP (NVLS) 能加速 collective 本身，但现有 NVLS 的 communication-centric 设计只关心把 collective 做快，没有匹配 LLM 计算 kernel 的内存语义，导致计算与通信仍被全局 barrier 分成独立阶段，SM 在通信等待期间利用率不足。论文给出的 LLaMA-7B TP profiling 中，扩展到 8 GPU 后平均通信时间约为计算时间的 1.6x；TP 的通信开销在已有研究中可占端到端延迟的 40% 到 60%。
    - 论文的主要贡献：提出 CAIS，即 Compute-Aware In-Switch computing framework。CAIS 的核心是让计算 kernel 按自身读写语义直接发起可合并的远程 load / reduction 请求，并由 switch 在数据路径上动态合并这些请求。系统包含三部分：一是扩展 GPU ISA 和 NVSwitch 微结构，引入 `ld.cais` 与 `red.cais` 以及 switch 侧 merge unit；二是 merging-aware TB coordination，通过编译器分组和轻量同步让多 GPU 上访问相同数据的 TB 在时间上对齐，提高 request merge 成功率；三是 graph-level dataflow optimizer，利用 TB 级依赖把 GEMM-RS、LayerNorm、AG-GEMM 等通信密集子图更细粒度地 overlap，并通过 asymmetric kernel overlapping 平衡 GPU-to-switch 与 switch-to-GPU 双向链路流量。
    - 论文所处背景：LLM 分布式训练/推理通常结合 DP、PP、TP，其中 TP 负责把大矩阵运算沿 hidden/head/sequence 维度切分到多个 GPU 上，因此每层都可能产生同步点。DP 的通信主要出现在梯度同步，PP 的通信主要是 stage 间 activation 交换，而 TP 的 collective 随层数和宽度频繁出现。NVIDIA Hopper 以后支持 `multimem.st`、`multimem.ld_reduce`、`multimem.red` 等 NVLS PTX 指令，可在 NVSwitch 内做 multicast/reduction，但论文指出这些 primitive 的 push/pull 模式与 TP kernel 的实际 memory access semantics 不一致，使 NVLS 潜力无法直接转化为细粒度计算通信重叠。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：第一类 baseline 是 TP-NVLS / SP-NVLS，它们使用 NVLS 加速 collective，但仍按 collective phase 与 GEMM phase 分离执行。具体 mismatch 包括：AG-GEMM 需要计算 kernel 以 read 语义按需读取远端数据，但 NVLS AllGather 主要通过 `multimem.st` push 数据；GEMM-RS 需要 producer 以 write / reduction 语义把结果写向归属位置，但 NVLS Reduce-Scatter 使用 `multimem.ld_reduce` pull 模式；Basic TP 的 AR-GEMM / GEMM-AR 需要读写混合语义，而 NVLS 的 `multimem.red` 仍是 push 模式。第二类 baseline 是 CoCoNet、FuseLib、T3 等 compute-communication overlap 方案，它们能做软件或硬件辅助 overlap，但没有利用 NVLS 的 in-switch reduction/multicast。第三类 NVLS-enhanced overlap baseline 如 CoCoNet-NVLS、FuseLib-NVLS、T3-NVLS 虽引入 NVLS，但仍受通信中心语义、粗粒度依赖、resource contention 或 asymmetric traffic 限制。LADM 这类 locality-aware TB scheduling 主要优化 GPU 内/跨 die 远程访问局部性，不面向 TP collective 的 in-switch merging。
    - 论文的设计方法：CAIS 把 switch 从被动转发/collective 加速器变成 compute-aware merging agent。ISA 上，`ld.cais` 标记可合并的远程 load，`red.cais` 标记可合并的 reduction 写入，二者在 request 中带 1-bit CAIS flag。switch 侧增加 CAM Lookup Table 和 Merging Table：CAM 按地址和请求类型匹配 session，Merging Table 保存 load cached data、reduction partial sums、状态、计数和请求元数据。对于 load merging，switch 对同一地址只向目标 GPU 发起一次读取，然后把返回数据复制给多个 requester；对于 reduction merging，switch 在内部累加多个 GPU 对同一地址的贡献，待参与者到齐后只把最终和写回目标 GPU。为了让这些请求在同一 switch 收敛，CAIS 使用基于地址的 deterministic routing。
    - 方法如何对冲 Baseline 缺陷：CAIS 不再把 collective 当作独立通信阶段，而是让 GEMM 等计算 TB 在执行过程中直接发起符合自身语义的远程访问。AG-GEMM 中，计算 TB 用 `ld.cais` pull 所需远端 operand；GEMM-RS 中，计算 TB 用 `red.cais` 把 partial result 送入 switch 侧合并。这样 producer-consumer 依赖从全局 kernel/collective barrier 降到 TB 级局部 barrier，不同 SM 和不同 TB 可以自然交错通信与计算。为解决多 GPU TB 独立调度导致的 request stagger，编译器静态分析 CUDA-to-PTX 阶段的地址表达式，如果访问 index 不含 GPU ID 且相同 `blockIdx` 会访问同一数据区域，就把这些跨 GPU TB 编为 TB Group，并在 JIT 中替换为 CAIS 指令；GPU synchronizer 和 switch Group Sync Table 提供 pre-launch 与 pre-access 同步，使平均同地址请求等待差从 35 us 降到 3 us 以下。graph-level optimizer 则利用 TB 级依赖，让下游 TB 在上游对应 tile/row 就绪后提前启动，并把 GEMM-RS 与 AG-GEMM 这类方向相反的 asymmetric traffic 交错执行，提升双向链路利用率。
    - 关键 trade-off：CAIS 接受了硬件、ISA、编译器和调度复杂度，换取更细粒度的 overlap 和更低冗余通信。switch 需要 merge unit、CAM/Merging Table、Content Array、LRU/timeout eviction、deterministic routing 和虚拟通道；GPU 侧需要 synchronizer、TB group metadata 和 pre-launch/pre-access 同步。同步虽然只交换空包，论文设置下往返约 0.5 us，但它仍可能 throttle 领先 TB。merge table 太小会造成 eviction 或 bypass，因此 CAIS 依赖 TB coordination 把 per-port 需求控制在约 40 KB。graph-level optimizer 还要求能识别细粒度 producer-consumer 依赖，并在 asymmetric overlapping 时控制 load/reduction traffic 争用。

3、论文实现：
    - Baseline 如何实现：论文在仿真框架中实现或扩展 9 个 baseline。TP with NVLS 包括 Basic TP-NVLS 和 SP-NVLS；Overlap Solutions 包括 CoCoNet、FuseLib 和 T3，其中 T3 在评测中扩展支持 AG-GEMM overlap；Overlap Solutions with NVLS 包括 CoCoNet-NVLS、FuseLib-NVLS 和 T3-NVLS，其中 CoCoNet/FuseLib 变体使用扩展 `multimem` 指令，T3-NVLS 使用 NVIDIA 提出的 DMA-based NVLS 设计；Locality-aware TB schedule 采用 LADM。论文还设置 CAIS-Base，关闭 merging-aware TB coordination 和 graph-level dataflow optimizer，用于消融分析。
    - 新设计如何实现：CAIS 在 Accel-Sim 中加入 Hopper 相关 GPU 特性，在 BookSim2 中扩展 NVSwitch/NVLink 网络，并增加 CAIS 的 switch merge datapath、NVLS `multimem` 支持、load/reduction request merging、merge table 管理、group synchronization 和 traffic control。GEMM kernel 使用 CUTLASS 实现。编译器/运行时侧通过静态 index analysis 识别 GPU-invariant 地址访问，把跨 GPU 同 `blockIdx` 的 TB 归入 logical TB Group，并在 JIT 编译时将相关 memory access 改写为 `ld.cais` 或 `red.cais`，同时把 TB Group metadata 附到 kernel launch 配置中。switch 侧维护 Group Sync Table，GPU 侧 synchronizer 接入 TB/warp scheduler，支持 pre-launch 和 pre-access 同步。
    - 实验 / 实现平台：论文模拟一个 8-GPU 系统，通过 4 个 NVSwitch 互连，拓扑参考 NVIDIA DGX-H100。GPU 参数按 NVIDIA H100 配置建模，NVLink/NVSwitch 使用真实设备参数：NVLink 为 16B flit、single-flit header、双向传输；NVSwitch 每端口配置 40 KB Merge Table，约 320 entries，每个 input port 有 8 个深度 256 的 virtual channel；GPU 到 switch 或 switch 到 GPU 的单程链路延迟设为 250 ns，round-trip 约 1 us。硬件开销在 TSMC 12nm 下评估：switch 侧修改约 0.50 mm2，小于 NVIDIA NVSwitch die 的 1%；GPU 侧 TB-group synchronization 逻辑约 0.019 mm2 / die，小于 H100 GPU 面积的 0.01%。
    - 关键实验设置与指标：评测使用 Mega-GPT-4B、Mega-GPT-8B 和 LLaMA-7B 的缩放版本，覆盖 training 和 inference，inference 聚焦通信密集的 prefill 阶段。因 Accel-Sim 内存与仿真时间限制，论文把 hidden size、FFN hidden size 等关键矩阵维度相对 SOTA 大模型缩小 50%，并相应减少 50% SM 数量；Table II 显示 half-scale 与 full-scale 在 CAIS speedup 上接近，例如 Full 为 1.43、Half 为 1.40。端到端指标上，CAIS 对 TP-NVLS、SP-NVLS、CoCoNet、FuseLib、T3、CoCoNet-NVLS、FuseLib-NVLS、T3-NVLS、LADM 的 inference geomean speedup 分别为 1.38x、1.89x、1.98x、1.90x、1.61x、1.25x、1.21x、1.45x、7.60x；training geomean 分别为 1.37x、1.89x、1.96x、1.89x、1.60x、1.23x、1.20x、1.45x、7.59x。sub-layer 上针对 Output projection / FFN / LayerNorm / projection 等通信密集组合，CAIS 对各 baseline geomean speedup 最高到 7.90x。消融中，TB coordination 将 merge 所需最小 table size 最高从 250 KB / port 降到 40 KB 以下，减少 87%；graph-level optimizer 将平均链路带宽利用率从 CAIS-Base 的 62.4% 提升到 CAIS-Partial 的 84.7%，完整 CAIS 达到 90.2%。NVLS simulator 用 NCCL AllReduce 在 1 到 16 GB message size 上和真实硬件对比，平均误差为 3.87%。

4、pipeline/kernel解析：
    - 新pipeline/kernel是什么：论文没有提出一个单独的 CUDA kernel，而是提出一个面向 TP 通信密集子图的 compute-aware in-switch execution pipeline。这个 pipeline 由 CAIS ISA (`ld.cais` / `red.cais`)、switch-side load/reduction merge micro-functions、merging-aware TB Group coordination、以及 graph-level asymmetric kernel overlapping 组成。它把传统的 “GEMM kernel 完成 -> collective kernel / NVLS primitive 完成 -> 下游 GEMM kernel 启动” 改成 “计算 TB 在执行中直接触发可合并远程访问 -> switch 按地址和类型合并 -> 下游 TB 按 TB 级依赖提前启动”。
    - 新pipeline/kernel的执行流例子：以 Transformer 子层中的 GEMM-RS + LayerNorm + AG-GEMM 为例。第一步，编译器在 CUDA-to-PTX 阶段分析 GEMM 的远程访问地址，发现跨 GPU 同 `blockIdx` 的 TB 会访问相同数据区域，就把这些 TB 编为同一个 TB Group，并把相关 load/reduction 指令替换为 `ld.cais` 或 `red.cais`。第二步，kernel launch 时 TB Group metadata 传给运行时和 switch；每个 GPU 的 synchronizer 在 TB 调度前向 switch 发送 pre-launch sync 空包，switch Group Sync Table 等待所有 GPU 的对应 TB 到齐后广播 release。第三步，GEMM-RS 中每个 GPU 的 TB 产生 partial result，并在写出阶段用 `red.cais` 发起 reduction request；deterministic routing 让同地址请求到同一 switch merge unit，CAM Lookup Table 命中同一 session 后，Merging Table 在 switch 内累加 partial sum，待参与 GPU 到齐后只把最终结果写回目标 GPU。第四步，对应 LayerNorm TB 不必等待整个 GEMM-RS kernel 结束，只要所需 row/tile 的输入就绪即可启动。第五步，下游 AG-GEMM 的 TB 通过 `ld.cais` 按需读取远端 operand；switch 对同地址 load 只向 home GPU 取一次数据，response 返回后缓存到 Content Array，并复制给所有 requester。第六步，graph-level optimizer 同时调度 GEMM-RS 和 AG-GEMM 这类方向互补的 kernel：GEMM-RS 主要产生 GPU-to-switch 流量，AG-GEMM 主要产生 switch-to-GPU 流量，CAIS 将 SM 分成不同执行组并用独立 virtual channel 和 round-robin arbitration 控制 load/reduction traffic，避免单方向链路长期拥塞。最终，CAIS 在保持 TP 计算语义不变的情况下，减少重复数据传输，把全局通信 barrier 缩小为 TB 级同步，并提升 interconnect 双向带宽利用率。
