    1、论文工作：
        - 论文要解决的核心问题：MoE LLM 推理只激活少量 expert，但为了未来可能被 gate network 选中，传统 serving 仍要把大量 inactive experts 常驻 GPU，造成严重显存低效。Expert offloading 试图把不活跃 expert 放到 CPU 内存、按需或提前加载到 GPU，但现有方法难以同时做到低延迟和低显存占用：保留太多 expert 会接近 no-offload 的大显存 footprint，保留太少则 cache miss 频繁，on-demand CPU-to-GPU expert loading 拉高 TTFT / TPOT。
        - 论文的主要贡献：论文提出 FineMoE，一个细粒度 expert offloading 系统。核心贡献是 expert map 数据结构：它记录每个 inference iteration、每个 MoE layer 上 gate network 对所有 experts 的概率分布，而不是只记录 request-level 的 expert 激活计数。FineMoE 结合 prompt semantic embedding 与已观测 expert trajectory，在 Expert Map Store 中检索最相似的历史 expert map，用于指导 expert prefetching、caching 和 offloading。论文还提出 similarity-aware expert selection、异步 map search / prefetch、expert map 去重，以及结合概率和访问频率的 cache eviction。
        - 论文所处背景：该工作属于 MoE-based LLM serving、lossless expert offloading、GPU memory management 和 inference runtime scheduling 方向。论文面向 Mixtral-8x7B、Qwen1.5-MoE、Phi-3.5-MoE 这类 decoder-only MoE LLM；这些模型由于 sparse activation 会产生 72% 到 84% inactive parameters，对应数十 GB 的潜在显存浪费。同时现代 MoE 模型常用 load-balancing loss，使 expert routing 更均匀，降低了粗粒度热 expert 统计的可预测性。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：DeepSpeed-Inference 类参数卸载不具备 expert awareness，按层或按参数 on-demand loading，缺少 expert prefetch/cache，延迟高。Mixtral-Offloading 使用 layer-wise speculative expert prefetching 和 LRU cache，但预测粒度仍较粗，并且 LRU 与 MoE layer-wise 顺序执行的访问特征不匹配。MoE-Infinity 使用 request-level Expert Activation Matrix 和同步 expert prediction / prefetching，粗粒度聚合会把 iteration-level 清晰模式冲淡，导致 expert hit rate 下降，并且同步 prefetch 会阻塞 forward。ProMoE 用 stride-based speculative prefetching 和 per-layer predictor，但论文指出 NN predictor 需要训练、重训练和额外 GPU memory，且 ProMoE 代码未开源，作者只能 best-effort 复现。共同根因是 baseline 多依赖 request-level 或固定规则，忽略了不同 iteration、不同 layer、不同 prompt 语义之间的细粒度差异。
        - 论文的设计方法：FineMoE 把 MoE expert offloading 表述为在 GPU cache 容量约束下最小化 on-demand expert loading latency 的问题；由于精确 ILP 求解和完美预测不可行，系统采用 heuristic。对于当前 iteration，FineMoE 对前 d 层使用 semantic-based expert map search：从模型 embedding layer 获取 prompt semantic embedding，与历史 embedding 做 cosine similarity，取最相似历史 iteration 的前 d 层 expert probability map。对于 d 之后的层，FineMoE 使用 trajectory-based expert map search：收集当前 iteration 已经过的层的 gate probability trajectory，与历史 expert maps 做 cosine similarity，取对应目标层概率分布。随后系统根据 similarity score 动态设置阈值 delta=clip(1-score,0,1)，按概率从高到低选择最少 expert，使累计概率超过 delta，且至少选择 top-K 所需数量。
        - 方法如何对冲 Baseline 缺陷：相对 request-level hit count，expert map 保留 iteration-level probability distribution，因此不会把早期 decode iteration 的清晰模式和后续 iteration 混成高熵统计；论文的 entropy 分析显示 coarse-grained patterns 比 fine-grained patterns 更不可预测。相对只看 expert 是否被选中，概率分布还能表达 gate network 对 expert 的置信度，支持 FineMoE 在低相似度时多预取以降低 miss，在高相似度时少预取以节省显存。相对同步 prefetch，FineMoE 用 Publisher-Subscriber 异步路径把 context collection、map search、prefetch 和 map update 与 inference forward 解耦，降低额外延迟。相对 LRU/LFU，FineMoE 的 prefetch priority 结合 expert probability 和距离当前 layer 的距离，eviction priority 结合 probability 与 cache visit frequency，更直接服务于“未来哪些 expert 更可能且更快被用到”。
        - 关键 trade-off：FineMoE 用 CPU 侧 Expert Map Store、相似度计算、异步线程和 cache 管理复杂度换取更高 expert hit rate 与更低 GPU expert footprint。低相似度时系统会增加预取数量来降低 miss 风险，这会短期占用更多 GPU cache；高相似度时更激进地少预取，则依赖历史 map 与当前 prompt/trajectory 的相关性。Expert Map Store 容量越大越容易找到相似历史 map，但 CPU memory 与搜索成本也增加；论文实验中 1K maps 已接近收益拐点，32K maps 仍低于 200MB CPU 内存。该方法也依赖 prefetch distance：太小无法隐藏 search/prefetch 开销，太大则预测 hit rate 下降。

    3、论文实现：
        - Baseline 如何实现：论文比较四个 SOTA MoE serving baselines。MoE-Infinity 使用 request-level expert activation matrix，作者在评测前准备对应历史矩阵。ProMoE 因未开源且需要针对每个 MoE 模型训练 predictor，作者基于 MoE-Infinity 做 best-effort prototype。Mixtral-Offloading 使用 layer-wise speculative prefetching 与 LRU expert cache。DeepSpeed-Inference 被实现为 expert-agnostic layer-wise parameter offloading；作者在 MoE-Infinity codebase 中实现其 offloading 逻辑并加入 expert cache，以便公平比较。所有 baseline 都被改造成能够服务 HuggingFace Transformers 中的 MoE 模型。
        - 新设计如何实现：FineMoE 原型基于 HuggingFace Transformers，并复用 / 修改 MoE-Infinity codebase。Expert Map Store 用 Python、PyTorch 和 NumPy 实现，用 ndarray 存储 semantic embeddings 和 expert maps，并转换为 tensor 做相似度计算。Expert Map Searcher 用 PyTorch native operations 实现 pairwise semantic / trajectory similarity 和 redundancy score。Expert Cache 用 C++ 和 CUDA Runtime APIs 在 MoE-Infinity 基础上实现 prefetching、caching、offloading 和 on-demand loading。多 GPU 推理采用 expert parallelism，expert 通过 hash map 和 round-robin 分配到不同 GPU，GPU 侧 task pool 与异步线程调度 expert prefetching 和 on-demand loading。
        - 实验 / 实现平台：主实验平台是 6 张 NVIDIA GeForce RTX 3090，每张 24GB GPU memory，GPU 间 pairwise NVLinks，CPU-GPU 通过 PCIe 4.0 连接，带宽 32GB/s；CPU 为 AMD Ryzen Threadripper PRO 3955WX，32 cores，480GB CPU memory。补充实验使用 NVIDIA A100，80GB HBM2e，2TB/s 峰值带宽。模型包括 Mixtral-8x7B、Qwen1.5-MoE 和 Phi-3.5-MoE；数据集包括 LMSYS-Chat-1M 和 ShareGPT。在线 serving 实验使用 Microsoft Azure 发布的 LLM inference traces 派生输入与生成长度，随机采样 256 个请求，负载约 2.91 requests/s。
        - 关键实验设置与指标：离线实验对 LMSYS-Chat-1M 和 ShareGPT 采用 7:3 split，70% prompt 的 semantic embedding 和 expert map 放入 FineMoE 的 Expert Map Store，30% 用于测试；每个数据集随机采样 64 个 prompts。论文分别报告 prefill 的 TTFT、decode 的 TPOT，以及 expert hit rate、cache limit sensitivity、online end-to-end latency、overhead 等系统指标。整体结果中，FineMoE 相比 DeepSpeed-Inference、Mixtral-Offloading、ProMoE 和 MoE-Infinity，平均 TTFT 分别降低 74%、67%、56%、53%，平均 TPOT 分别降低 46%、38%、27%、22%。Expert hit rate 相比 Mixtral-Offloading、ProMoE 和 MoE-Infinity 分别提升 14%、37%、68%。在 6GB expert cache limit 下，FineMoE 相比 DeepSpeed-Inference、Mixtral-Offloading、ProMoE 和 MoE-Infinity 的 TPOT 分别降低 36%、25%、16%、29%。论文总结 FineMoE 相比 SOTA 平均降低 inference latency 47%，提升 expert hit rate 39%。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：论文没有提出新的 GPU kernel、attention kernel 或算子融合内核；它提出的是 FineMoE 的 fine-grained expert offloading runtime pipeline。这个 pipeline 由 inference context collection、Expert Map Store、Expert Map Searcher、similarity-aware expert prefetching、Expert Cache eviction/offloading、on-demand loading 和 expert map update 组成。关键变化是：每个 decode iteration 都把 gate probability distribution 与 semantic/trajectory similarity 变成可执行的 expert prefetch/cache 决策，而不是用 request-level 热度或固定 stride 规则。
        - 新pipeline/kernel的执行流例子：假设一个请求 prompt “Summarize this paper” 到达。Prefill 或某个 decode iteration 开始前，FineMoE 先收集上下文：对初始层，读取 embedding layer 输出的 semantic embedding；对后续层，收集已执行层的 gate probability distribution 作为 expert trajectory。Expert Map Searcher 订阅这些 context，在 Expert Map Store 中计算 cosine similarity：前 d 层按 semantic similarity 找历史 map，d 之后按 trajectory similarity 找历史 map。对目标 layer，系统取检索到的 expert probability distribution，并按 similarity score 计算 delta；如果相似度低，delta 高，系统会选择更多高概率 experts 预取以降低 misprediction；如果相似度高，delta 低，则只预取较少 experts 来节省 GPU cache。Expert Cache 按 p/(l-l_now) 排序执行预取，把 CPU 内存中的 expert weights 异步搬到对应 GPU；当 cache 超限时，根据 1/(p*freq) 优先驱逐低概率、低访问频率 expert。随后 MoE layer 真正执行 gate network top-K 选择：如果所需 expert 已在 GPU cache 中，直接计算；如果 miss，FineMoE 暂停普通 prefetch task，立即从 CPU 到 GPU on-demand loading 缺失 expert，再继续 forward。该 iteration 结束后，新产生的 expert map 和 semantic/trajectory context 被写回 Expert Map Store；若 store 达到容量限制，FineMoE 用 semantic 与 trajectory redundancy score 去重，保留更有覆盖性的 maps，供后续请求继续使用。
