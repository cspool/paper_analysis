    1、论文工作：
        - 论文要解决的核心问题：LLM serving 的 decode 阶段同时受模型权重和 KV cache 显存占用限制，但 GPU SM 计算资源常常没有被充分利用。现有系统通常在显存不足时把整请求 KV cache swap 到 host memory，或丢弃整请求 KV cache 后再整体 recompute；这些方案粒度太粗，仍假设正在 decode 的请求需要完整保留所有历史 token / layer 的 KV 状态，导致并发 batch size 被显存卡住，TTFT 和 throughput 下降。论文指出，在 Llama2-13B / A100 / ShareGPT 场景中，显存占用可长期接近饱和，而计算利用率大多处在很低区间；即便 DeepSeek-V2-Lite 这类 MLA 模型压缩了 KV cache，也仍出现 memory occupancy 高、GPU utilization 相对不足的问题。
        - 论文的主要贡献：论文提出 eLLM，一个面向高吞吐和低延迟 LLM serving 的 adaptive KV caching 系统。核心思想是在 decode 期间只缓存部分 token 的 KV cache，对未缓存 token 在需要时动态重算，并让这些重算工作与当前 token decode 并行执行。系统贡献包括：第一，token-wise / layer-wise 的 KV cache 管理机制，把缓存决策从整请求粒度细化到 token 与 layer；第二，请求级优化，在线调整 batch size 和 uncached token ratio，在满足 TPOT SLO 的前提下提高并发；第三，层级优化，通过 communication-computation overlap 和 layer-wise kernel fusion 减少重算、换入和 decode 的额外延迟；第四，closed-loop adaptation，把 layer-level 得到的额外显存开销反馈给 request-level optimizer，联合优化缓存比例、batch size 和 fused kernel thread allocation。
        - 论文所处背景：该工作属于 LLM serving、KV cache memory management、GPU runtime scheduling 和 kernel fusion 的交叉方向，面向在线推理中请求到达动态、上下文长度变化大、TPOT / TTFT SLO 严格的生产式负载。论文发表于 EuroSys 2026，基于 vLLM 和 PagedAttention 生态实现，并与 vLLM-Recompute、vLLM-Swap 和 HCache 比较。

    2、相对 Baseline 解决的问题与设计方法：
        - Baseline 的具体问题：vLLM-Recompute 在显存不足时丢弃 preempted requests 的 KV cache，等请求重新获得显存后再整体重算，问题是长序列下重算开销高，并且粒度仍是整请求。vLLM-Swap 把 preempted requests 的 KV cache 搬到 host memory，再通过 PCIe 恢复，问题是 host-GPU 带宽和传输延迟成为瓶颈。HCache 通过缓存 hidden states 加速状态恢复，但恢复仍主要发生在 request-level 或 layer subset 粒度，并且 decode 阶段仍倾向保留当前并发请求的完整 KV cache。共同缺陷是 all-or-nothing 或粗粒度恢复策略没有利用“旧 token 可部分重算、SM 有空闲、显存更稀缺”这一资源不平衡。
        - 论文的设计方法：eLLM 把 KV cache 策略改成 adaptive token-wise caching：对每个请求的历史 token，较新的部分继续缓存 KV，较旧的一段按照 uncached token ratio r 释放显存；decode 到某层时，对未缓存 token 临时重算该层 KV，再与缓存 token 的 KV 拼接用于当前 token attention，使用完即释放。为避免小 batch 重算效率差，eLLM 把所有当前 running requests 的未缓存 token 聚合成 batch 统一重算。进一步地，eLLM 把 KV block 从 vLLM 的“所有 layer / 多 token 放在一个物理 block”重塑为按 F 个连续 layer 划分的更小单元，并维护 seq_id、token_id、layer_id、logical_block_id、physical_block_id、#filled 等 map table 元数据，使部分 token、部分 layer 的缓存、swap 和 recompute 都能定位。
        - 方法如何对冲 Baseline 缺陷：相对 vLLM-Swap，eLLM 不需要把整请求全部换出换入，而是把显存省在最有价值的位置：缓存一部分 token，另一部分用空闲 SM 重算，从而用 compute 换 memory。相对 vLLM-Recompute，eLLM 避免整请求全量重算，只重算被释放的 token / layer，并通过跨请求 batching 提高 GEMM / attention 操作效率。相对 HCache，eLLM 不只优化恢复速度，还直接降低 decode 阶段每个 active request 的 KV residency，从源头提高并发容量。layer-wise overlapping 让 host-GPU KV 传输与 GPU 重算并行；kernel fusion 将 compute-intensive 的重算 kernel K1 和 memory-intensive 的 decode kernel K2 合并执行，减少 launch overhead 并提升 SM utilization。
        - 关键 trade-off：eLLM 用额外计算和运行时复杂度换取显存释放与更高并发。uncached token ratio 太高会增加 recomputation FLOPs，使 TPOT 上升；cache ratio 太高则显存节省有限，batch size 提升受限。kernel fusion 可降低 TPOT，但需要临时存放 K1 产生的 KV，论文观察到这会占用额外一层显存资源并可能拉高部分请求 TTFT。新的 layer-granular block 与 map table 降低粗粒度 block 的内部碎片，但引入 map lookup 和更复杂的调度元数据；F 过小会增加映射访问开销，F 过大又增加内存碎片，论文最终默认 F=4。

    3、论文实现：
        - Baseline 如何实现：论文比较三个 baseline。vLLM-Recompute 和 vLLM-Swap 使用 vLLM 中已有的两类 preemption 策略：前者在 GPU memory 满时丢弃 KV cache 并在后续恢复时重算，后者把 preempted requests 的 KV cache swap 到 host memory 后再恢复。HCache baseline 由作者在 vLLM 中复现其核心功能，包括 StatePartitionAlgorithm；其 hidden state 快速恢复逻辑类似 eLLM 的 recomputation function，KV cache 在 GPU / host 间恢复类似 eLLM 的 Comm-Com Overlapping。为公平比较，论文把 eLLM 的 Eq. (5) 中 r=0 集成到 vLLM-Recompute、vLLM-Swap 和 HCache 中，使它们也能在满足 TPOT SLO 下更新估计 batch size。
        - 新设计如何实现：eLLM 作为 vLLM 扩展实现，保留 vLLM 核心 API 以兼容第三方模型。系统包含 request-level 和 layer-level 两级组件。request-level 侧，调度开始时收集实时 request metadata，并用 SciPy SLSQP 解一个受显存和 TPOT SLO 约束的优化问题，求出近似最优 batch size b 与 uncached token ratio r；每个 SequenceGroup 维护自己的 uncached-token ratio，用来控制 KV cache block 在 GPU 与 CPU memory 之间的划分。layer-level 侧，eLLM 用 torch.cuda.stream 启动异步 CUDA streams，使缓存 token 的数据传输与未缓存 token 的重算并行；同时预编译多组 CUDA shared libraries，对 recomputation 和 decoding kernels 生成 32 到 1024、步长 32 的线程配置，运行时根据层级计算需求选择合适 .so。
        - 实验 / 实现平台：主实验在一台 4 NVIDIA A100-80GB GPU 服务器上完成，GPU 通过 PCIe 4.0 x16 连接，无 NVLink；CPU 为 96-core Intel Xeon Gold 6342 @ 2.80GHz，host memory 256GB。软件栈运行在 Docker 中，使用 CUDA 12.4 与 NVIDIA Driver 550.107.02。模型包括 Llama2-13B 和 Llama2-70B，前者使用 MHA 并在 1 张 A100 上测试，后者使用 GQA 并在 4 张 A100 上以 tensor parallel 运行。数据集包括 ShareGPT 和 L-Eval paper assistant subset；ShareGPT 平均输入 / 输出长度约 222 / 1,346，L-Eval 平均输入 / 输出长度约 35,956 / 5,189。host memory 为 Llama2-13B 分配 40GB，为 Llama2-70B 分配 160GB 存放 swapped KV caches。
        - 关键实验设置与指标：请求到达基于 DynamoLLM 中的 Azure LLM invocation trace，原始平均请求间隔 22.15ms、标准差 37.49ms；ShareGPT 将间隔减半约 50 req/s，L-Eval 将间隔加倍约 12.5 req/s，每个 end-to-end 实验含 2 分钟 warm-up，并重复 5 次。核心指标为 output token throughput、TTFT、TPOT SLO attainment，以及运行时 running requests、SM utilization、energy efficiency、goodput 和系统 overhead。ShareGPT 上，eLLM 对 Llama2-13B 相比 vLLM-Recompute / vLLM-Swap / HCache 的 throughput 提升分别为 2.64x / 2.61x / 1.91x，对 Llama2-70B 提升为 2.0x / 2.0x / 1.6x；TTFT 最多降低 2.63x，SLO attainment 达 97.3% 和 98.6%。L-Eval 长上下文上，eLLM 平均只缓存约 0.53 的 prefix length，获得超过 47% memory saving，并最高实现 3.03x throughput 提升、1.79x TTFT 降低。Ablation 显示禁用 Comm-Com Overlapping 或 Kernel Fusion 都会退化，但保留 token-wise caching 后仍优于多数 baseline；系统开销方面，request-level optimizer 的 90% overhead 低于 10ms，map table layer lookup 平均低于 0.015ms。

    4、pipeline/kernel解析：
        - 新pipeline/kernel是什么：论文引入的核心执行路径可以称为 eLLM adaptive KV caching serving pipeline，而不是单个新的 attention 算子。它包含三条相互耦合的路径：第一，request-level batching / token-wise caching pipeline，在线选择 b 和 r；第二，layer-wise KV management pipeline，把 KV block 按 F 个 layer 重塑，并对 uncached tokens 在层级上选择 recompute、swap 或 decode 所需的临时存储；第三，layer-wise fused execution pipeline，把下一层未缓存 token 的 KV recomputation kernel K1 与当前层当前 token 的 decode kernel K2 融合，并用两个 CUDA streams 让 host-GPU KV transfer 与 compute 同步推进。
        - 新pipeline/kernel的执行流例子：假设当前等待队列有一批请求，每个请求已经完成 prefill，正在 decode，并且历史序列越来越长导致 GPU KV cache 接近满。eLLM 的 scheduler 先读取每个请求的 sequence length、队列大小、最大等待时间、模型参数和 GPU 显存 / FLOPS 信息，用 SLSQP 求出本轮可服务的 batch size b 与 uncached token ratio r。对于某个请求，若 r=0.4，则它较早的 40% 历史 token 不再长期保留 KV cache，较新的 60% token 仍保留在 GPU blocks 中。进入某个 transformer layer i 时，eLLM 通过 map table 找到 cached tokens 在该 layer 的物理 block；对 uncached tokens，系统要么从 host memory swap 少量层 KV，要么在 GPU 上重算这些 token 的 layer KV，并临时保存用于当前 attention。与此同时，eLLM 可在 Stream A 上传输下一层需要的 cached KV，在 Stream B 上执行 recomputation K1 和 decode K2 的 fused kernel：K1 负责为 layer i+1 的旧 uncached tokens 生成 KV，K2 负责用 layer i 已准备好的完整历史 KV 为新 token 做 decode attention。线程分配按 K1 / K2 的估计计算量比例确定，并调整为 32 的倍数以匹配 NVIDIA warp granularity。当前 token 完成该层 decode 后，临时重算得到的旧 token KV 被立即释放，新 token 的 KV 根据当前缓存策略写入对应 layer-granular block；layer-level optimizer 计算真实 overlap / fusion 额外显存 Mo 后反馈给 request-level optimizer，下一轮可能进一步增加 b 或降低 r，从而形成闭环。
