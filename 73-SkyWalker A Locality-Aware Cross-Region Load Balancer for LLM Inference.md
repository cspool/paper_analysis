1、论文工作：
    - 论文要解决的核心问题：多地域 LLM inference 部署中，服务商通常在每个地域用 reserved instances 或 on-premise GPU 集群做长期容量承诺。地域内流量有明显昼夜峰谷，且不同地域峰值时间错开；如果每个地域都按本地峰值独立预留 GPU，就会在离峰地域产生大量闲置容量。直接把请求跨地域集中到一个全局负载均衡器又会引入跨地域 RTT、中心瓶颈和单点故障。论文要解决的是：如何在不明显牺牲 TTFT 和吞吐的前提下，把多地域的容量按全局峰值而不是各地域峰值之和来使用。
    - 论文的主要贡献：SkyWalker 提出一个 locality-aware cross-region load balancer。它在每个地域部署入口 load balancer，通过 load balancer 间的两层路由转发过载地域的流量；同时用 prefix-aware routing 保持 KV cache locality，并用基于 pending requests 的 selective pushing 避免 LLM 输出长度不可预测导致的副本过载。论文给出两个前缀路由版本：SkyWalker-CH 使用 user/session key 的一致性哈希，SkyWalker 使用 load balancer 维护的局部 prefix trie snapshot。实验显示 SkyWalker 相比现有负载均衡器达到 1.12-2.06x throughput、1.74-6.30x lower latency，并在区域倾斜负载下通过减少副本数实现 25% serving cost reduction。
    - 论文所处背景：LLM serving 由 prefill 和 decode 构成，prefill 生成 KV cache，decode 自回归逐 token 生成。在线服务关注 TTFT，地域本地请求的 TTFT 主要由 prefill 和排队决定；跨地域请求还会增加最高约 200 ms 的网络延迟。不同于网页或搜索这类几十毫秒级任务，LLM 请求通常持续数秒到数十秒，因此适度跨地域转发可能是可接受的。但 LLM 请求的输入/输出长度、KV cache 占用、连续批处理容量都强烈变化，使传统 round robin、least load、中心化 prefix-aware router 很难直接迁移到多地域生产部署。

2、相对 Baseline 解决的问题与设计方法：
    - Baseline 的具体问题：Region-local deployment 的问题是成本和 GPU 利用率：每个地域按本地峰值预留容量，而昼夜模式导致离峰地域 GPU 闲置。论文用 WildChat trace 说明，单地域负载方差可达 2.88x 到 32.64x，而聚合后的全局负载方差降到 1.29x；按聚合全局峰值预留比各地域独立峰值预留成本低 40.5%。Naive centralized load balancer 的问题是性能和可用性：用户可能经历两次跨地域 RTT，且所有请求依赖一个中心 load balancer。传统 RR / least load / SGLang Router / GKE Gateway 的问题分别是：不感知 KV cache、把 outstanding request count 当作负载但无法反映 LLM 内存占用、中心化 prefix tree 跨地域通信代价高、通用网关没有 LLM-specific prefix/cache 与 selective pushing 机制。
    - 论文的设计方法：SkyWalker 采用两层跨地域路由。客户端通过统一域名访问最近的地域 load balancer；该 load balancer 优先把请求发给本地可用 replica，本地满载时再把请求转发给远端可用 load balancer，由远端 load balancer 在本地 replica 内做最终放置。KV cache locality 通过两类策略维护：SkyWalker-CH 在 load balancer 层和 replica 层都按 session/user key 做一致性哈希；完整 SkyWalker 则在每个 load balancer 维护两个 prefix trie，一个记录本地 replicas 的前缀状态，一个记录本地域历史转发到远端 load balancer 的 snapshot，用最长前缀匹配选择 cache hit 可能性最高的目标。负载控制使用 selective pushing by pending requests：请求先排在 load balancer，只有当 replica 的 continuous batch 没有 pending request、还能接纳新请求时才推送。
    - 方法如何对冲 Baseline 缺陷：对 region-local 的成本缺陷，SkyWalker 允许过载地域把请求转发到离峰地域，从而用全局容量池吸收地域峰谷差。对中心化路由的延迟和单点问题，SkyWalker 让客户端先到最近 load balancer，并只在本地资源不可用时跨地域转发。对 prefix-aware baseline 的单地域假设，SkyWalker 用 regional snapshot 近似全局 prefix 状态，避免每个请求都跨地域同步 prefix tree。对 LLM 请求长度不可预测的问题，它不用 outstanding request 数量硬设阈值，而是读取 replica 是否出现 pending queue；pending request 直接表示当前 continuous batch 已受 GPU memory 等资源限制，不能再接纳更多请求。
    - 关键 trade-off：SkyWalker 接受更复杂的 control plane、load balancer 间 heartbeat/probe、prefix trie snapshot 维护和远端转发逻辑，以换取更低成本和更高 GPU 利用率。SkyWalker 的 prefix snapshot 是近似的，不保证拥有远端所有 replica 的完整 prefix 状态；这种设计牺牲一部分全局最优性来避免跨地域同步开销。跨地域转发仍会增加网络延迟，因此策略明确优先本地可用 replica。基于 pending requests 的 selective pushing 需要 100 ms 级探测，过短会增加探测开销，过长会降低响应负载变化的能力。

3、论文实现：
    - Baseline 如何实现：宏基准对比包括 GKE Gateway、Round Robin、Least Load、Consistent Hashing、SGLang Router、SkyWalker-CH 和 SkyWalker。RR、LL、CH、SGL 作为 single global coordinator baseline，通过修改 SkyServe 支持，单个 load balancer 部署在美国；GKE Gateway、SkyWalker-CH、SkyWalker 在每个地域部署 load balancer。SGLang Router 是 prefix-aware baseline，但其全局 prefix-aware router 仍属于中心化思路，不具备 SkyWalker 的 load balancer 间两层地域路由。microbenchmark 中，论文扩展 SGLang Router 比较 blind pushing、基于 outstanding request 阈值的 selective pushing，以及 SkyWalker 使用的基于 pending request 的 selective pushing。
    - 新设计如何实现：SkyWalker 基于 SkyServe 实现，扩展约 3000 行 Python 代码，兼容 OpenAI API 形式的推理引擎，例如 vLLM 和 SGLang。每个 load balancer 维护本地 replica 连接和远端 load balancer 连接，周期性 probe 本地 replica 的 pending queue，并通过 heartbeat 同步每个远端 load balancer 是否有可用 replica 以及其 load balancer queue 是否超过小 buffer。系统通过 AWS Route53 生成统一域名，DNS 根据客户端源 IP 解析到最近 load balancer。controller 负责添加/删除 replicas、重配 load balancers、健康探测和 failure recovery；load balancer 失败后，controller 会把该地域 replicas 临时交给地理上最近的 load balancer 管理。
    - 实验 / 实现平台：主要端到端实验在 AWS 上运行，GKE Gateway baseline 在 GCP 上运行。所有 replica 使用一张 NVIDIA L4 GPU，通过 SGLang 运行 meta-llama/Llama-3.1-8B-Instruct。replicas 和 clients 分布在美国、欧洲、亚洲三个地域。workload 包括 ChatBot Arena 多轮对话、WildChat 多轮对话、Tree of Thoughts on GSM、Mixed Tree。ChatBot Arena 每地域 80 个并行 conversation；WildChat 使用美国 40、欧洲 30、亚洲 30 个 client；ToT 使用 12 个 replica 均匀分布在三地域，树深为 4；Mixed Tree 让美国运行更大的 4-branch trees，其他地域仍运行 2-branch trees，用来模拟异构用户请求模式。
    - 关键实验设置与指标：主要指标是 service throughput、TTFT latency、end-to-end latency、KV cache hit rate、replica load imbalance 和 cost reduction。宏基准显示 SkyWalker 在多轮对话上相比 single-load-balancer 方案提升 1.12-1.2x throughput，相比 GKE Gateway 提升 1.43-1.62x throughput；在所有实验中，SkyWalker 的 P50/mean TTFT 为 baseline 的 15.87%-57.63%，P90 TTFT 为 baseline 的 10.08%-23.38%。ToT 场景中 CH 在完全均匀树 workload 下可略高于 SkyWalker 约 2%，但在 Mixed Tree 这类异构 burst 场景下会因为同一 key 持续打到同一 replica 而过载。microbenchmark 中，基于 pending request 的 selective pushing 相比 blind pushing 提升 1.27x throughput，并把 P90 TTFT 降低 18.47x；相比基于 outstanding request 阈值的方法，throughput 提升 1.4x。区域倾斜实验中，美国 120 clients、亚洲和欧洲各 40 clients，SkyWalker 用 9 个 replicas 达到 region-local 12 个 replicas 类似吞吐，对应 25% 成本节省。

4、pipeline/kernel 解析：
    - 新 pipeline/kernel 是什么：SkyWalker 没有提出新的 GPU kernel；它提出的是一个跨地域 LLM serving routing pipeline。核心路径是“最近地域入口解析 -> 本地/远端可用性判断 -> prefix-aware candidate selection -> selective pushing 到 replica -> continuous batching 执行 -> heartbeat/probe 更新状态”。这个 pipeline 的新意在于把多地域容量调度、KV cache locality、LLM continuous batching 的可接纳状态放在同一个负载均衡决策里，而不是只按地域、请求数或中心化 prefix tree 做路由。
    - 新 pipeline/kernel 的执行流例子：假设亚洲地域某一时刻进入流量高峰，一个用户发来带长上下文的 chat 请求。客户端先解析统一域名，DNS 把它指向亚洲最近的 SkyWalker load balancer。请求进入该 load balancer 的 FCFS queue。队首请求被处理时，load balancer 先检查 LocalAvail：如果亚洲本地至少一个 replica 没有 pending request，说明其 continuous batch 还能接纳新请求，于是优先在本地 replica 中选择目标；若启用 SkyWalker-CH，则按 session/user key 在本地 replica ring 上找可用虚拟节点；若启用完整 SkyWalker，则用输入 prompt 在本地 replica prefix trie 中做最长前缀匹配，选 prefix hit ratio 最高且可用的 replica。
    - 跨地域转发路径：如果亚洲本地 replicas 都已满，load balancer 查询 RemoteAvail，选择仍有可用 replica 且远端 load balancer queue 没超过 buffer 的地域，例如欧洲。选择远端目标时，SkyWalker-CH 在 load balancer ring 上按 session/user key 找远端 load balancer；完整 SkyWalker 则用本地域维护的远端 prefix trie snapshot 估算哪个远端地域最可能复用该 prompt 的 KV cache。请求被转发到欧洲 load balancer 后，欧洲 load balancer 再在自己的本地 replicas 中重复 prefix-aware selection，把请求推入具体 replica。
    - selective pushing 如何工作：SkyWalker 不在请求到达时盲目压入 replica 队列，而是周期性 probe replica 的 pending queue。pending request 指还没有进入 continuous batch 的请求；只要出现 pending queue，就说明当前 replica 的 batch 已受资源限制，继续推送会让请求卡在 replica 内部队列里，并可能让其他 replica 空闲。SkyWalker 因此把后续请求留在 load balancer queue，等待本地 replica 释放容量，或者在本地全满时转发到远端可用地域。这避免了 blind pushing 下“一个长输出请求堵住某个 replica，而其他 replica 还有空闲”的负载不均。
    - prefix trie snapshot 路径：完整 SkyWalker 的每个 load balancer 维护本地 prefix tree 和远端 snapshot tree。每当一个请求被转发或服务，输入 prompt 会沿 trie 插入路径，并在每个节点记录目标 replica 或远端 load balancer。为了限制内存，tree 有最大大小，超限时淘汰最早插入记录。查询时，SkyWalker 沿 prompt 做最长前缀匹配，并只考虑当前可用目标；如果某个节点没有可用目标，遍历可提前终止，因为子节点目标集合只会是父节点的子集。这个机制用近似局部视图保留 KV cache locality，同时避免每个请求跨地域访问全局 prefix tree。
    - 运行时回写与恢复：load balancer 通过 100 ms probe interval 获得 replica pending 状态，通过 heartbeat 获得远端 load balancer 的可用 replica 数和 queue 状态。controller 监控 replica 和 load balancer 健康；若某个 load balancer 失败，关联 replicas 会被临时转给最近的健康 load balancer，恢复后再迁回。这个路径使 SkyWalker 不依赖单个中心调度器，降低了中心化 baseline 的瓶颈和单点故障风险。
